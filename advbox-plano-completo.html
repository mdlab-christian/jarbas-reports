🔍 Deep Review — Correções e Melhorias
ℹ️ O que o Deep Review revelou Depois de inspecionar os payloads reais dos endpoints, encontrei 6 correções no plano original + 8 campos novos que não tinham sido mapeados + 2 limitações confirmadas. Documentado abaixo.

✅ Confirmado correto

CPF/CNPJ como chave de dedup — campo identification confirmado, com validação de algoritmo real na Advbox.
Número CNJ como chave de processos — campo process_number validado contra base do Poder Judiciário na criação.
Rate limit 30 GET/min — confirmado. Estratégia de 2.4s delay correta.
GET /settings como primeiro passo — retorna todos os IDs de referência necessários.
Endpoint /last_movements para bulk — mais eficiente que iterar por processo.

⚠️ Correções necessárias

birthdate retorna DD/MM/YYYY — a resposta real usa formato brasileiro, não YYYY-MM-DD. Precisa converter antes de salvar no Supabase (tipo date).
folder = max 30 chars — confirmado como label de texto livre. NÃO é caminho de arquivo. Documentos não são acessíveis por esta API.
Mapeamento cellphone → WhatsApp — no payload real existe cellphone e phone. cellphone vai para whatsapp, phone vai para telefone.

🆕 Campos novos descobertos nos payloads reais (não estavam no plano anterior)

👤 Clientes — campos adicionais
  • gender → sexo (campo existe no MdFlow)
  • street → endereco_logradouro (endereço completo!)
  • region → endereco_bairro
  • country → nacionalidade (inferir)
  • number_ctps, number_pis, number_cid → documentos JSONB
  • origin (label textual) → origem direto sem precisar mapear ID
  • lawsuits: [] → array de IDs de processos do cliente ← muito útil para linking
💰 Transações — campos confirmados
  • type: income/expense → tipo da transação
  • date_due + date_payment → datas de vencimento e pagamento
  • competence → mês de competência (MM/YYYY)
  • category → categoria (HONORÁRIOS, ALVARÁS, ALUGUEL, etc.)
  • lawsuit_id + process_number → vínculo com processo
  • debit_bank / credit_bank → conta bancária
  • cost_center → centro de custo
⚠️ Confirmado: documentos NÃO acessíveis via API pública O campo folder nos processos aceita máximo 30 caracteres — é claramente um label, não um path. Os 19 endpoints públicos não incluem nenhum endpoint de arquivo. Ao receber o token, vamos testar endpoints não documentados (/files, /cloud, /documents), mas o Plano B (Playwright para download manual) deve estar pronto.

Arquitetura da solução
🏛️ Os 3 Pilares do Projeto
Pilar 1 · Migração
📦 Extração Histórica Completa

Fazer uma única extração completa de tudo que a Advbox tem e enriquecer o MdFlow. Executa uma vez, depois vira backup.

  • Pull completo de clientes com todos os campos
  • Enriquecimento de campos vazios no MdFlow
  • Pull completo de processos + detalhes
  • Movimentações históricas (TRIBUNAL + MANUAL)
  • Publicações históricas do Diário da Justiça
  • Histórico de tarefas por processo
  • Financeiro histórico (honorários, custas)
  • Dump completo como backup permanente
Pilar 2 · Monitoramento
🔍 Validação Contínua de Cobertura de Intimações

Enquanto a Advbox estiver ativa (1+ mês), comparar diariamente o que a Advbox capturou vs o que o JARBAS capturou. Este pilar é o que garante que podemos cancelar com segurança.

  • Skill de comparação diária (skill_advbox_monitor)
  • Buscar publicações/movimentações das últimas 24h na Advbox
  • Comparar com tabela jarbas_tarefas + movimentacoes
  • Classificar divergências: gap real vs filtro intencional (João)
  • Relatório diário de cobertura → Telegram às 7h
  • Score: % do que JARBAS captura vs Advbox
  • Meta: 100% antes de cancelar
  • Fallback ativo: processo sem captura no JARBAS → puxa da Advbox
Pilar 3 · Auto-Aprendizado
🧠 Loop de Melhoria Contínua do JARBAS

Quando a Advbox capturou algo que o JARBAS não capturou e NÃO foi por filtro do João → o JARBAS precisa aprender. Loop de retroalimentação com supervisão humana.

  • Para cada gap real detectado → criar entrada em advbox_divergencias
  • Categorizar automaticamente o tipo de gap
  • Gerar "regra candidata" baseada no padrão da divergência
  • Notificar João para validar a regra
  • João aprova → regra vira filtro oficial no JARBAS
  • João rejeita → registrar como "exclusão intencional"
  • Histórico de regras aprovadas → base de conhecimento do João
  • Score de divergências cai progressivamente → JARBAS fica melhor
🎯 Relação entre os pilares Pilar 1 faz uma vez. Pilares 2 e 3 rodam em paralelo por 1 mês. Quando o score de cobertura atingir 100% por 7 dias consecutivos e não houver gap real não-explicado → cancelar a Advbox com confiança.
