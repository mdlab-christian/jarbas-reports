Problema 1 — Mais urgente
Processos legados sem advbox_id e sem CNJ

1.521 processos do legado (migração antiga) que têm cliente e empresa mas nunca foram localizados na AdvBox. Sem o link, movimentações, intimações e prazos na AdvBox não chegam até o MdFlow — risco real de perder prazos.

Problema 2
Processos novos no MdFlow não chegam na AdvBox

Quando um processo é cadastrado no MdFlow (origem jarbas ou auto_cadastro_jarbas), ele deveria automaticamente ser criado também na AdvBox e receber o advbox_id de volta. Esse loop não está funcionando corretamente — há 195 processos recentes sem vínculo.

Novo processo MdFlow
→
POST /processes AdvBox
→
Recebe advbox_id
→
UPDATE processos SET advbox_id
Problema 3
Processos AdvBox novos não chegam no MdFlow

Processos cadastrados diretamente na AdvBox (pelo operador) não estão sendo importados para o MdFlow. O fluxo de distribuição da AdvBox → MdFlow depende de o processo já existir no MdFlow com o advbox_id correto.

Recomendação JARBAS
📋 Plano em 3 Fases
ℹ️ Premissas do plano Usar a API AdvBox existente (rate limit: 30 req/min = delay 2.4s entre GETs). Chave de dedup: CPF/CNPJ do cliente para pessoas, número CNJ para processos já distribuídos. Execução incremental com checkpoint para não perder progresso em caso de falha.
⚡ Fase 1 — Vincular os 195 com CNJ Mais fácil · ~1h
  • Buscar na AdvBox por numero_cnj para cada um dos 195 processos
  • Se encontrado → UPDATE processos SET advbox_id = X, advbox_sincronizado_em = now()
  • Se não encontrado → marcar para criar na AdvBox (Fase 2)
  • Resultado esperado: ~180-190 vinculados (alguns podem não existir na AdvBox ainda)
-- Processos alvo
SELECT id, numero_cnj, cliente_id, empresa_id 
FROM processos 
WHERE advbox_id IS NULL AND ativo = true AND deleted_at IS NULL
  AND coalesce(numero_cnj, '') != ''
ORDER BY created_at DESC;
-- ~195 registros
🔍 Fase 2 — Recuperar legado por CPF+Empresa Trabalhoso · ~4h execução
  • Para cada um dos 1.521 processos legado sem CNJ:
  • GET /clients?cpf=XXX na AdvBox → localiza o cliente pelo CPF
  • GET /processes?client_id=Y → lista processos do cliente
  • Filtra pelo nome da empresa ré (normalizado) → match com ±85% de confiança
  • Se match único → grava advbox_id + numero_cnj no MdFlow
  • Se múltiplos candidatos → salva em tabela advbox_vincular_revisar para revisão humana
  • Ambíguos (match não único) → NÃO vincular automaticamente, salvar para revisão
-- Breakdown dos 1.521
-- Estratégia: processar 200/dia para não exceder rate limit
-- 1521 / 200 = ~8 dias de processamento incremental
-- Checkpoint por advbox_sincronizado_em = NULL
🔄 Fase 3 — Loop automático bidirecional Definitivo · n8n workflow
  • MdFlow → AdvBox: Trigger em processos INSERT → criar na AdvBox → gravar advbox_id
  • AdvBox → MdFlow: Cron a cada 30min → GET /processes?updated_since=X → importar novos
  • Webhook AdvBox (se disponível) para receber atualizações em tempo real
  • Reconciliação semanal: varrer todos processos sem advbox_id e tentar vincular
  • Pendente confirmar: AdvBox tem webhook de eventos ou só polling?
⚠️ Riscos identificados • Rate limit AdvBox (30/min) torna Fase 2 lenta — rodar em background
• Normalização de empresa ré pode falhar (variações de nome)
• ~1% dos processos legado pode não ter correspondente na AdvBox
✅ Impacto esperado • Fase 1: +195 processos vinculados (imediato)
• Fase 2: +1.200-1.400 processos recuperados (8 dias)
• Fase 3: zero novos desvinculados (permanente)
