01 · O Fluxo por página

8 etapas. Uma direção só.

0
Handoff — Christian investiga a página
Duração: 5–15 min de áudio · Custo: R$0
Christian acessa a página no sistema real e grava áudio descrevendo: o que funciona, o que não funciona, o que incomoda visualmente, o que quer mudar no UX, funcionalidades que deveriam existir mas não existem
Manda o áudio para o JARBAS no Telegram. JARBAS transcreve via Whisper e salva como context_PAGINA_YYYY-MM-DD.md
Opcionalmente: jarbas handoff /rota para extrair contexto da conversa Claude Web sobre a página
1
Investigação completa + HTML #1
Duração: ~20 min · Custo: ~R$25
Lê spec da página em ~/workspace/mdflow/docs/_resumido___PAGINA.txt — identifica o que deveria existir vs o que existe
Screenshot-First: 4-8 screenshots (todas as abas, modais abertos, estados de loading, empty state)
Análise offline: código TSX, TypeScript errors, console.log, imports mortos, staleTime ausente, arquivos > 500 linhas
Todos os filtros, search boxes e modais da página — lista completa para testar
n8n: mapeia todos os workflows da página → chama webhook → verifica execução → confirma response no Supabase → confirma atualização no frontend
Consulta 21.st Magic: componentes disponíveis relevantes para a página
Gera HTML #1 com: diagnóstico completo, plano de correção, sugestões de melhoria, ideias de integração JARBAS, se vale redesign do frontend, perguntas para Christian
2
Validação com Christian → PRD definitivo
Duração: 10–20 min (Christian) · Custo: ~R$5
Christian revisa HTML #1, responde perguntas, aprova/ajusta o plano proposto
JARBAS incorpora o feedback e gera o PRD definitivo: PAGINA_YYYY-MM-DD_HHmm.md
PRD contém: checklist atômico por camada · plano de correção por item · critério de aceite · ordem de execução · arquivos envolvidos
Christian aprova o PRD. A partir daí, nada pode ser pulado.
3
Execução — Claude Code segue o PRD
Duração: 30–90 min · Custo: R$80–R$300
JARBAS dispara Claude Code com o PRD completo. Auto-approve ativo. Um C-prompt por item do checklist
Após cada batch: verifica 0 TypeScript errors antes de avançar
n8n: toda alteração → teste imediato do ciclo completo (webhook → execução → Supabase → frontend). Zero tolerância para workflow quebrado
Para testes: cria dados → testa funcionalidade → limpa dados. Nenhum dado de teste fica no banco
Filtra e testa: todos os filtros, search boxes, modais, abas da página
4
Limpeza + Refatoração
Duração: ~20 min · Custo: ~R$50
Identifica e remove arquivos .tsx não referenciados pela página
Quebra componentes > 500 linhas em sub-componentes atômicos
Remove: console.log, comentários mortos, imports não usados, dead code acumulado
n8n: remove workflows obsoletos da página (0 execuções nos últimos 30 dias → candidatos a exclusão após confirmar com Christian)
5
UI/UX Redesign — 21.st Magic
Duração: ~30 min · Custo: ~R$80 · Preview antes de commitar
Claude Code consulta 21.st: /ui dark [componente], bg #12100e, primary #20c997
Substitui componentes customizados por biblioteca 21.st quando superiores
Aplica DS v5: Manrope títulos, Inter corpo, JetBrains Mono código/CNJ, Lucide icons, glassmorphism cards/modals
Adiciona estados ausentes: loading skeleton, empty state, error feedback, animações de transição
JARBAS gera HTML preview dos componentes redesenhados → envia para Christian aprovar → só então commita
6
Revisão final — 3 agentes + HTML #3
Duração: ~20 min · Custo: ~R$40 · Nota mínima 8/10
3 agentes em paralelo: Frontend Auditor (DS v5, TS, performance) + Backend Specialist (RLS, RPCs, segurança) + E2E Tester (navega como usuário, testa tudo)
E2E testa: clica em cards (verifica se modais abrem), preenche todos os forms, usa todos os filtros, verifica respostas n8n no frontend
Todos ≥ 8/10 → commit + HTML #3 com checklist completo, nota final, screenshots antes/depois, custo total da página
Qualquer < 8 → loop volta para Etapa 3 com C-prompts cirúrgicos para o que falhou
7
Pós-aprovação — Documentação
Duração: ~10 min · Custo: ~R$10
JARBAS gera spec atualizada da página em Markdown (section × section, layer × layer, componente × componente)
Compara com spec antiga — valida que nova supera (não substitui se nova for incompleta)
Substitui ~/workspace/mdflow/docs/_resumido___PAGINA.txt com nova spec
Cria doc no Notion: pasta "Mdflow - spec atualizada lancamento"

🎯 Recomendação final — o que eu faria diferente
Nada radicalmente diferente. Seu plano está correto. O que acrescento:

1. Sempre o handoff do Claude Web antes de cada página crítica. Você tem contexto valioso lá que não está nas specs.
2. PRD em duas versões: v1 (após investigação) → você valida → v2 definitivo (incorpora feedback). Mais claro que um único documento que vai mudando.
3. n8n obsoletos: listar ANTES de começar, não durante a revisão de cada página. Criar um inventário único de workflows inativos → você aprova quais deletar → aí durante a revisão é só executar.
4. Notion pode esperar. Specs locais em Markdown vão funcionar igualmente bem para o lançamento. Notion é um bônus, não um bloqueador.
5. JARBAS discorda? Fala. Essa é a regra que você pediu e que vai fazer diferença. Se vir algo que não faz sentido, eu aviso antes de executar.
