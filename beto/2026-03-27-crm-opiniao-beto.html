Revisão crítica como dev: o que está certo, o que está errado, e o que me preocupa

Modo: Opinião 100% Sincera 3 bugs críticos identificados Nomenclatura confusa 2026-03-27
⚖️ Veredicto Geral
⚠️ Minha avaliação honesta

O CRM tem uma base sólida — a estrutura de pipeline, realtime e drag-and-drop está bem construída. Mas existe um gap crítico no fluxo de distribuição: o lead não é arquivado automaticamente como você espera quando vai para a distribuição. O ModalDistribuirLead (que realmente arquiva) não está conectado ao CrmPage. Isso é um bug funcional sério, não cosmético.

🟢 O que está funcionando bem
  • ✅ Drag & drop com optimistic update + rollback
  • ✅ Realtime via Supabase channels
  • ✅ Lead card com informações densas e úteis
  • ✅ Filtros persistidos no localStorage
  • ✅ Validações ao mover (WhatsApp, advogado)
  • ✅ ModalTriagem com RPC atômica
  • ✅ Bulk actions (mover, arquivar, atribuir)
  • ✅ Deep links funcionais (?lead, ?modal, ?etapa)
🔴 O que me preocupa
  • 🔴 ModalDistribuirLead desconectado do CrmPage
  • 🔴 Lead não é arquivado após triagem aprovada
  • 🔴 Dois realtime conflitantes (useCrmKanban + useCrmRealtime)
  • 🟡 Nomenclatura: ModalDistribuicao ≠ distribuição
  • 🟡 Limite de 200 leads/coluna silencioso
  • 🟡 onSuccess da ModalTriagem não passado no CrmPage
  • 🔵 ModalDistribuicao diz "Prosseguir para Triagem" (lead já está em triagem)
  • 🔵 Dupla fonte de restrições no checklist (JSONB + tabela)
🔄 Fluxo Real vs Esperado
O que Gustavo espera
Lead em Triagem → Clica "Distribuir" → Aprovação Jurídica → Processos criados → Lead ARQUIVADO automaticamente → N8N gera petições
O que realmente acontece hoje
Lead em Triagem → Clica "Distribuir" → ModalDistribuicao (checklist) → Clica "Prosseguir para Triagem" (confuso) → ModalTriagem (aprovação) → rpc_crm_aprovar_triagem → cliente + processos → Lead NÃO é arquivado ❌ → N8N NÃO é disparado ❌

O ModalDistribuirLead (que chama distribuir_lead_completo + arquiva + dispara n8n) existe mas nunca é chamado a partir do CrmPage. Ele está órfão.

🔴 Bug crítico

A RPC rpc_crm_aprovar_triagem cria cliente + processos, mas não arquiva o lead. O lead continua aparecendo no Kanban mesmo após convertido. Isso gera confusão operacional e leva atendentes a acharem que o lead ainda está ativo.

🐛 Bugs Confirmados
BUG #1 — Lead não arquivado após distribuição
O ModalDistribuirLead (com a lógica de arquivamento via distribuir_lead_completo) não está conectado ao CrmPage. O fluxo de triagem atual termina no ModalTriagem que só converte o lead em cliente mas não arquiva. Lead fica "convertido" mas visível no Kanban.
Impacto: Alto Arquivo: CrmPage.tsx
BUG #2 — onSuccess do ModalTriagem não é passado no CrmPage
No CrmPage, o ModalTriagem é montado sem o prop onSuccess. Isso significa que após a conversão, não há nenhuma ação de acompanhamento (abrir drawer do cliente, navegar para distribuição, ou arquivar o lead). O usuário fica olhando para um Kanban que não refletiu a mudança.
Impacto: Alto CrmPage.tsx linha ~270
BUG #3 — Duplo realtime pode gerar race condition
O useCrmKanban.ts tem um listener interno na tabela leads que invalida queries. O useCrmRealtime (chamado no CrmPage) também assina a tabela leads. Dois canais Supabase assinando a mesma tabela para a mesma org = duplo disparo de invalidações, possível flickering e consumo desnecessário de conexões realtime.
Impacto: Médio useCrmKanban.ts + useCrmRealtime.ts
BUG #4 — Limite de 200 leads/coluna silencioso
O Kanban limita 200 leads por coluna (LEADS_PER_COLUMN_LIMIT = 200) mas não avisa o usuário. Se uma coluna tiver 250 leads, 50 são silenciosamente ocultados. Combinado com filtros, pode fazer um atendente achar que um lead não existe quando na verdade está além do limite.
Impacto: Médio CrmKanban.tsx
BUG #5 — ModalDistribuicao com texto errado
O botão principal do ModalDistribuicao diz "Prosseguir para Triagem" mas o lead JÁ está em triagem. Deveria dizer "Prosseguir para Aprovação Jurídica" ou simplesmente "Continuar". Isso confunde o atendente sobre em que etapa ele está.
Impacto: UX ModalDistribuicao.tsx

⚠️ Anti-pattern: inicialização de estado

No ModalTriagem, o init dos checkboxes está fora de useEffect, o que pode causar renders extras:

// ❌ Como está hoje (ModalTriagem.tsx) if (lead && restricoes.length > 0 && !initialized) { // Isso roda em cada render até initialized = true setSelectedIds(initial); setInitialized(true); } // ✅ Como deveria ser useEffect(() => { if (!lead || !restricoes.length || initialized) return; setSelectedIds(new Set(restricoes.filter(r => r.processavel).map(r => r.id))); setInitialized(true); }, [lead, restricoes, initialized]);

📡 Realtime — Problema de Duplicação
🔴 Race condition potencial

Dois hooks subscrevem eventos da tabela leads simultaneamente para a mesma organização.

useCrmKanban.ts

Assina leads via postgres_changes
Invalida: crm-leads-kanban + crm-colunas-contagem
⚠️ Toast quando awaiting_human vira true

useCrmRealtime.ts

Assina crm_channel_events + provavelmente leads
Invalida queries do kanban/lista/radar
⚠️ Pode disparar junto com useCrmKanban

Na prática: um update num lead pode disparar 2-4 invalidações seguidas de 2 sources diferentes. Isso não quebra mas é ineficiente e pode causar flickering no Kanban. Solução ideal: consolidar numa subscription única ou tirar o realtime do useCrmKanban.

🎨 UX / UI — O que me incomoda
Botão "Distribuir" leva para Triagem, não para distribuição
O botão no card quando coluna = triagem se chama "Distribuir" com ícone de balança (Scale). Mas ele abre o ModalDistribuicao (checklist) que leva ao ModalTriagem. É confuso — o usuário pensa que vai distribuir mas está aprovando juridicamente.
Sem feedback visual após conversão do lead
Quando a triagem é aprovada com sucesso, só aparece um toast e confetti. O lead some do Kanban (via invalidação de query) mas não há transição suave nem confirmação de onde ele foi. O usuário pode ficar confuso.
Lead card muito denso
O card tem: nome, CPF, data nascimento, WhatsApp, advogado, atendente, badge estado, badge origem, score IA, contagem restrições, notas, MdZap status, MdZap send button, etiquetas, observações e botão distribuir. São ~14 elementos num card. Pode ser demais para visualização rápida.
Aba "Arquivados" sem indicador de contagem no tab
As abas Pipeline e Radar têm badges com contagem. Arquivados não tem — seria útil saber quantos arquivados existem sem clicar na aba.
🌟 Pontos Positivos — Crédito onde é devido
✅ Isso está muito bem feito

Antes de só criticar: a estrutura geral do CRM é sólida. O nível de detalhe técnico — optimistic updates com rollback, undo toast, realtime, bulk actions, filtros persistidos — é acima da média para um SaaS em estágio beta.

🏆 Destaques técnicos
  • 🥇 RPC rpc_crm_aprovar_triagem é atômica — não deixa lead em estado inconsistente

📋 Conclusão — O que eu faria primeiro
1. Conectar ModalDistribuirLead ao CrmPage
Esse é o fix mais importante. Após o ModalTriagem ter sucesso (onSuccess), abrir o ModalDistribuirLead para o cliente criado — ou integrar o arquivamento direto na RPC de triagem.
2. Arquivar lead automaticamente após aprovação jurídica
Confirmar: a RPC rpc_crm_aprovar_triagem arquiva o lead? Se não, adicionar UPDATE leads SET arquivado = true WHERE id = p_lead_id dentro da transação da RPC.
3. Consolidar o realtime em um único hook
Tirar a subscription de useCrmKanban e deixar só no useCrmRealtime. Ou vice-versa. Mas não os dois.
4. Corrigir texto "Prosseguir para Triagem" no ModalDistribuicao
Mudar para "Prosseguir para Aprovação Jurídica". É uma linha de código.
5. Mostrar aviso quando coluna está no limite (200 leads)
Banner discreto no topo da coluna: "Mostrando 200 de 347 leads. Use filtros para refinar."
