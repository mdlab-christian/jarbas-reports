🚨 Fluxo de Distribuição — O Core do Problema CRÍTICO
❌ O fluxo de distribuição está incompleto

Você descreveu: "quando um lead é enviado para distribuição, ele vira cliente, virou processo, e é arquivado automaticamente." Isso é o comportamento esperado. Mas hoje o código não faz isso. Veja:

Fluxo esperado (o que Gustavo quer)
Lead em Triagem → Clica "Distribuir" → Checklist pré-envio → Aprovação jurídica → Cliente criado → Processos criados → Lead ARQUIVADO ✅ → N8N dispara petição ✅
Fluxo atual (o que o código faz)
Lead em Triagem → Clica "Distribuir" → ModalDistribuicao (checklist) → ModalTriagem (aprovação) → rpc_crm_aprovar_triagem → Cliente criado ✅ → Lead fica no Kanban ❌ → N8N não é chamado ❌
O que existe mas não está conectado

O arquivo ModalDistribuirLead.tsx já tem tudo implementado corretamente:

  • ✅ Chama distribuir_lead_completo (RPC atômica)
  • ✅ Cria processos para cada restrição
  • ✅ Arquiva o lead automaticamente (lead_arquivado: true)
  • ✅ Dispara o N8N com as petições em lotes de 5
  • ✅ Toast de sucesso com contagem de restrições distribuídas
  • ✅ Invalida queries do CRM após sucesso

⚠️ Mas esse modal nunca é aberto pelo CrmPage. Ele é órfão.

📦 Arquivamento Automático — O que confirmar
⚠️ Pergunta que precisa de resposta antes de codar

Existem duas RPCs que criaram processos de formas diferentes. Antes de conectar o ModalDistribuirLead, precisamos confirmar o fluxo correto:

RPC O que faz Arquiva lead? Chama N8N? Usada onde
rpc_crm_aprovar_triagem Converte lead → cliente + processos NÃO NÃO ModalTriagem (ativo)
distribuir_lead_completo Cria processos por restrição + arquiva SIM SIM ModalDistribuirLead (órfão)
💡 Possíveis soluções (para decidir com Gustavo)

Opção A: Conectar o ModalDistribuirLead após o ModalTriagem ter sucesso (via onSuccess). Fluxo: Triagem aprova → cria cliente → ModalDistribuirLead distribui as restrições → arquiva lead.

Opção B: Modificar a RPC rpc_crm_aprovar_triagem para já arquivar o lead e disparar o n8n ao final. Mais simples, menos modais.

Opção C: Separar claramente: ModalTriagem só aprova juridicamente. ModalDistribuirLead é chamado logo depois ou manualmente. Lead é arquivado só quando distribuído.

📡 Realtime Duplicado — useCrmKanban vs useCrmRealtime
⚠️ Duas subscriptions para a mesma tabela

O useCrmKanban assina leads (UPDATE/INSERT) com toast de awaiting_human. O useCrmRealtime assina leads (INSERT) para invalidar o board. Resultado: um INSERT dispara duas invalidações quase simultâneas.

HookTabelaEventoAção
useCrmKanban leads * (UPDATE/INSERT/DELETE) Invalida kanban + contagens + toast awaiting_human
useCrmRealtime crm_channel_events * Invalida kanban + lista + radar
useCrmRealtime leads INSERT Invalida kanban + lista + radar
Solução recomendada
// Opção: tirar a subscription de leads do useCrmKanban // e centralizar tudo em useCrmRealtime // useCrmRealtime.ts — adicionar lógica do awaiting_human .on('postgres_changes', { event: 'UPDATE', schema: 'public', table: 'leads', filter: `organizacao_id=eq.${organizacaoId}`, }, (payload) => { if (payload.new.awaiting_human && !payload.old.awaiting_human) { toast(`Lead "${payload.new.nome}" aguardando atendimento`); } invalidateAll(); }) // E remover o useEffect de realtime do useCrmKanban.ts inteiramente
🟡 Performance useCrmKanban.ts useCrmRealtime.ts ~20 min
🔗 onSuccess Faltando — ModalTriagem no CrmPage
🔴 Blocker #2

O ModalTriagem tem um prop onSuccess que recebe o resultado da conversão (cliente_id, processos_ids, etc.). Mas no CrmPage ele é montado sem esse prop. Isso significa que após a aprovação jurídica, nada acontece além do toast e confetti.

Como está hoje (CrmPage.tsx ~linha 270)
// ❌ onSuccess não passado {triagemLeadId && ( <ModalTriagem open={!!triagemLeadId} onOpenChange={(open) => { if (!open) setTriagemLeadId(null); }} leadId={triagemLeadId} // onSuccess ausente! /> )}
O que precisa ser feito
// ✅ Com onSuccess que conecta ao próximo passo {triagemLeadId && ( <ModalTriagem open={!!triagemLeadId} onOpenChange={(open) => { if (!open) setTriagemLeadId(null); }} leadId={triagemLeadId} onSuccess={(result) => { // Aqui: decidir o próximo passo // Opção A: abrir ModalDistribuirLead setDistribuirLeadId(triagemLeadId); setTriagemLeadId(null); // Opção B: navegar para /distribuicao // navigate('/distribuicao'); // Opção C: só invalidar queries (se RPC já arquiva) }} /> )}
🔴 Blocker CrmPage.tsx ~10 min
🎨 Melhorias de UX — Não são blocker mas importam

🟡
Sem contagem na aba Arquivados
Pipeline e Radar têm badges com contagem. Arquivados não tem. Seria útil mostrar mesmo que de forma diferente (cinza, não pulsante).
🔵 Melhoria CrmPage.tsx 20 min
🔵
ModalTriagem: inicialização de state fora de useEffect
O init dos checkboxes está fora do useEffect (anti-pattern React). Mover para um useEffect com dependências corretas para evitar renders extras.
🔵 Qualidade ModalTriagem.tsx 10 min
🔵
Drawer do cliente não abre após aprovação jurídica
Após o lead ser convertido em cliente, seria natural oferecer "Ver perfil do cliente" ou abrir o ClienteDrawer automaticamente com o cliente_id retornado pelo onSuccess. Hoje o toast já tem um link "Ver Cliente" mas ele navega para uma rota nova, saindo do CRM.
🔵 UX CrmPage.tsx 30 min
🔵
Checklist do ModalDistribuicao não bloqueia mesmo quando crítico
O texto diz "Você pode prosseguir mesmo com pendências" — mas se o lead não tem CPF válido ou não tem restrição processável, ele não vai poder ser distribuído mesmo assim. Considerar bloquear o prosseguimento quando os 2 critérios críticos falham (CPF + restrição processável).
