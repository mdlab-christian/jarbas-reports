Como o loop vai funcionar
1
Leitura total — mapear todos os arquivos 47 arquivos / ~6000 linhas

CC lê todos os 47 arquivos do mdzap — componentes, hooks, types, pages, lib. Não uma visão geral: lê o código real, linha por linha. Para cada arquivo, monta uma lista de achados potenciais.

2
Triage — classifica todos os achados por severidade

Cada achado vai para um backlog interno: P0 (quebra funcionalidade), P1 (UX ruim / dados errados), P2 (código frágil / melhoria), P3 (nice-to-have). Ordena P0 primeiro.

3
Correção por arquivo — abre, corrige, salva, commit parcial

Para cada achado P0/P1: abre o arquivo, aplica a correção, salva. Faz um commit a cada 3–5 correções relacionadas para manter o histórico limpo. Não pede confirmação — só aplica e segue.

4
Re-leitura — relê os arquivos corrigidos em busca de regressões

Após corrigir um arquivo, relê ele e os arquivos que importa/é importado por ele. Correções podem introduzir novos problemas. Qualquer achado novo vai pro backlog e o loop continua.

5
TypeScript check a cada rodada

A cada 5 arquivos corrigidos: roda npx tsc --noEmit. Erros de tipo são P0 automáticos e entram no backlog imediatamente.

6
Critério de saída — zerar P0 e P1, documentar P2/P3

O loop termina quando: backlog de P0 e P1 está vazio E tsc --noEmit passa limpo E npm run build passa. P2/P3 pendentes vão pro MDZAP-QA-R2-RESULTADO.md como sugestões.

O que procurar em cada arquivo (guia de auditoria)
A
Componentes de UI — o que checar src/components/mdzap/*.tsx
  • Cores hardcoded: qualquer text-[#xxx], bg-[#xxx], border-[#xxx], fill-[#xxx] — todos substituir por tokens DS
  • Classes Tailwind deprecated: violet-* → chart-2, green-* → success, yellow-* → warning, red-* → destructive
  • Acessibilidade: botões sem aria-label, imagens sem alt, dialogs sem foco correto
  • Estados de loading/empty: componentes que podem ficar em branco sem mostrar estado vazio adequado
  • Condicionais nulas perigosas: conversa.campo sem checar se conversa é null, data.map() sem verificar se data existe
  • Console.log / TODO / FIXME: remover ou resolver
  • Imports não usados: limpar
  • Keys em listas: toda lista React com .map() deve ter key estável (não índice)
B
Hooks — o que checar src/hooks/mdzap/*.ts + src/hooks/useMdzap*.ts
  • Memory leaks: useEffect com subscription (supabase.channel, setTimeout, setInterval) deve ter cleanup no return
  • Infinite loops: dependências de useEffect com objetos/arrays criados inline — checar se useMemo ou useCallback está sendo usado corretamente
  • Stale closures: callbacks dentro de useEffect sem as deps corretas no array
  • Race conditions: múltiplos fetches sem cancelamento (AbortController ou isMounted flag)
  • Error handling: todo await supabase.from(...) deve verificar error e ter tratamento adequado
  • Query keys inconsistentes: verificar se ['mdzap', ...] é usado de forma consistente em todo o código para invalidação funcionar
C
Tipos — o que checar src/types/mdzap.types.ts
  • Campos as any: substituir por tipos corretos
  • Campos opcionais perigosos: se o banco sempre retorna um campo, não precisa ser ? — false optionals causam checagens desnecessárias
  • Metadata tipado como unknown: criar interface específica ou manter Record<string, unknown> com cast explícito
  • Sync com supabase.ts: verificar se os tipos locais batem com o que a RPC fn_mdzap_conversas e as tabelas retornam
D
Performance — o que checar
  • Re-renders desnecessários: componentes pesados (MdzapMsgThread, MdzapLeadPanel) sem React.memo ou com props instáveis
  • Funções inline em props: onClick={() => fn(id)} dentro de listas — usar useCallback
  • Arrays criados inline como props: queryKey={['a', 'b']} sem useMemo causa re-render a cada render
  • Realtime over-subscription: checar se há canais sendo criados/destruídos a cada render em vez de uma vez por conversa
