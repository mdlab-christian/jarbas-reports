B1 — Drag-and-drop do Kanban não funciona

Crítico ⏱ 30 min fix src/components/crm/CrmKanban.tsx
B1

@dnd-kit/core — DndContext, useDraggable, useDroppable

DraggableLeadCard em CrmKanban.tsx

Após um drag-and-drop bem-sucedido, o card abre o painel lateral do lead, sobrepondo a ação de mover. Visualmente parece que o DnD "não fez nada" pois o drawer abre imediatamente após soltar o card.

O dragStartedRef é declarado mas nunca setado para true. Portanto, o guard no handleClick nunca bloqueia o clique pós-drag.

// CrmKanban.tsx — DraggableLeadCard
const dragStartedRef = useRef(false);

// handleClick checa o ref, mas ele NUNCA é setado para true
const handleClick = useCallback(() => {
  if (dragStartedRef.current) {
    dragStartedRef.current = false;
    return; // este return NUNCA executa
  }
  onOpenPanel?.(lead.id!); // sempre abre o painel
}, [...]);

// Não há useEffect ou callback que seta dragStartedRef.current = true
+ useEffect(() => {
+   if (isDragging) {
+     dragStartedRef.current = true;
+   }
+ }, [isDragging]);

Quando isDragging vira true, o ref é marcado. Ao soltar o card, o handleClick lê o ref como true, bloqueia a abertura do painel e reseta para false.

B2 — Modal Distribuição exibe dados do lead ERRADO

Crítico ⏱ 15 min fix src/components/crm/modals/ModalDistribuicao.tsx
B2

Ao abrir o modal de checklist de distribuição para um lead, às vezes os dados exibidos são de outro lead. Aparece o CPF do lead anterior em vez do atual.

A query de dados do lead tem staleTime: 60 * 1000 (60 segundos). Nesse intervalo, o React Query retorna dados em cache sem refetch, mesmo que o leadId seja diferente.

// ModalDistribuicao.tsx
const { data: lead } = useQuery({
  queryKey: ['crm-lead-checklist', leadId],
  queryFn: async () => {
    return supabase.from('leads').select('...').eq('id', leadId)...
  },
  enabled: !!leadId && open,
  staleTime: 60 * 1000,  // ← BUG: dados ficam stale 60s
});

A query usa .eq('id', leadId) corretamente (não usa CPF), mas o cache de 60s faz com que ao fechar e reabrir o modal para outro lead, os dados antigos apareçam por uma fração de segundo ou até serem refrescados.

  staleTime: 60 * 1000,
+ staleTime: 0,  // sempre refetch quando modal abre

B3b — CPF Completo (ApiFull) sem resposta

Crítico ⏱ 2–4h investigação supabase/functions/ + rpc_disparar_auditoria
B3b

Ao clicar "Consultar CPF Completo" no painel de auditoria do CRM, a consulta fica em estado "processando" indefinidamente ou falha sem mensagem clara. O bureau apifull_115 não retorna dados.

O fluxo passa por:

  1. rpc_disparar_auditoria cria registro em tabela auditorias com status pendente
  2. Algum worker deve pegar esse registro e chamar a API ApiFull
  3. Worker atualiza o registro com resultado e muda status para concluida

Problema: Não há Edge Function de processamento de auditorias em supabase/functions/. As funções existentes são: audio-transcribe, encrypt-credentials, invite-user, olivia-chat, olivia-smart-blocks.

  • A Edge Function de auditoria pode não estar no repositório (deploy manual?) ou não estar deployada
  • O token da API ApiFull pode não estar configurado nas secrets do Supabase
  • A URL/endpoint da API ApiFull pode estar incorreta
  • Pode haver um pg_cron ou trigger que chama a função — não visível no repo
  1. Verificar em Supabase Dashboard → Edge Functions se existe função de auditoria deployada
  2. Verificar Supabase → Secrets se existe APIFULL_TOKEN ou similar
  3. Verificar logs da Edge Function para ver o erro exato
  4. Verificar se há pg_cron ou trigger no banco que processa auditorias
  5. Verificar n8n se existe workflow mdflow/auditoria

P1 — Botão "Triagem" no drawer não move o lead

Alto ⏱ 15 min fix src/pages/crm/CrmPage.tsx
P1

Ao clicar no botão "Triagem" na barra de ações do drawer de lead, o modal abre e fecha imediatamente com erro "Lead não está em triagem". O lead não é movido para a coluna Triagem.

O botão "Triagem" no header do drawer chama onCrmAction('triagem', id). Em CrmPage, isso abre ModalTriagem (aprovação jurídica), que valida se o lead já está na coluna triagem. Como o lead ainda não está em triagem, o modal fecha com erro imediatamente.

// ClienteDrawerHeader.tsx
onClick={() => onCrmAction?.('triagem', id)}  // botão "Triagem"

// CrmPage.tsx
case 'triagem': setTriagemLeadId(leadId); break;  ← ERRADO

// ModalTriagem.tsx — valida e fecha se não está em triagem:
if (d.coluna_crm !== 'triagem') {
  toast.error('Lead não está em triagem.');
  onOpenChange(false);
  return null;
}

O correto seria abrir o ModalDistribuicao (checklist de preparação), que ao clicar "Prosseguir" move o lead para a coluna triagem via coluna_crm: 'triagem'.

  case 'triagem': setTriagemLeadId(leadId); break;

P2 — Toggle "Aguardando resposta" ativo por padrão

Alto ⏱ 5 min fix src/hooks/crm/useLeadFilters.ts
P2

O toggle "Aguardando" no pipeline está ativo para novos usuários ou quando os filtros foram salvos com ele ativo. O Kanban fica filtrado mostrando apenas leads awaiting_human=true por padrão.

// useLeadFilters.ts
function loadFilters(userId: string | undefined): LeadFilters {
  try {
    const raw = localStorage.getItem(getStorageKey(userId));
    if (!raw) return { ...DEFAULT_FILTERS };
    // BUG: spread do localStorage sobrescreve DEFAULT_FILTERS
    return { ...DEFAULT_FILTERS, ...JSON.parse(raw) };
    // Se localStorage tem awaiting_human: true, mantém true
  } catch {
    return { ...DEFAULT_FILTERS };
  }
}
  return { ...DEFAULT_FILTERS, ...JSON.parse(raw) };

O toggle "Aguardando" não deve ser persistido entre sessões — é um filtro situacional.

P3 — WhatsApp salvo sem máscara de formatação

Alto ⏱ 30 min fix ModalEditarLead.tsx + ModalNovoLead.tsx
P3

O WhatsApp é salvo no banco sem formatação (51999990000). Ao editar o lead, o campo mostra o número sem máscara. Ao criar novo lead, o campo aceita input sem máscara se o usuário não formatar.

No save, o valor é normalizado para apenas dígitos antes de persistir. Ao carregar o form de edição, o valor vem como dígitos crus e a máscara só é aplicada no onChange, não no valor inicial.

// ModalNovoLead.tsx — ao salvar
whatsapp: normalizeDigits(data.whatsapp), // salva "51999990000"

// FormContato.tsx — ao editar
value={watch('whatsapp') ?? ''}  // mostra "51999990000" sem máscara
onChange={(e) => setValue('whatsapp', applyPhoneMask(e.target.value), ...)}
// Onde form.reset() é chamado com dados do lead (ModalEditarLead.tsx):
+ form.reset({
+   ...leadData,
+   whatsapp: leadData.whatsapp ? applyPhoneMask(leadData.whatsapp) : '',
+ });

applyPhoneMask já está importado em helpers.ts.
