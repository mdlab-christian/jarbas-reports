BUG #3 — P0 Aba: Restrições
Restrição principal ausente na aba Restrições
ProcessoTabRestricoes.tsx:22
useRestricoes.ts:44
Causa Raiz

Há dois mecanismos de FK entre processos e restricoes:

  • Forward: processos.restricao_id → restricoes.id (processo aponta para restrição principal)
  • Reverse: restricoes.processo_id → processos.id (restricao aponta para processo)

O fn_restricoes_listar filtra apenas pelo reverse link (restricoes.processo_id = p_processo_id). A restrição principal, criada durante o lead/triagem, só tem o forward link — restricoes.processo_id é null.

🔴 Resultado: a aba Restrições retorna vazio ("Nenhuma restrição vinculada") mesmo havendo restrição principal no processo.

Correção — Migração SQL:

supabase/migrations/XXXXXX_fix_fn_restricoes_listar.sql 
-- Atualizar fn_restricoes_listar para incluir restrição principal via forward FK
WHERE restricoes.processo_id = p_processo_id
   OR restricoes.id = (
     SELECT restricao_id
     FROM processos
     WHERE id = p_processo_id
   )

-- Alternativa: garantir que ao distribuir um processo,
-- restricoes.processo_id seja atualizado para processo.id
BUG #1B — P0 Aba: Dados
Cliente não encontrado na aba Dados — sem fallback para cliente_nome
ProcessoDadosCliente.tsx:17
Causa Raiz

ProcessoDadosCliente exibe "Nenhum cliente vinculado" quando processo.clientes === null. Isso ocorre quando:

  • processos.cliente_id é null (processo migrado com cliente_nome mas sem FK)
  • OU o nome da FK processos_cliente_id_fkey não corresponde ao nome real da constraint no banco

O ProcessoHeader exibe processo.cliente_nome (campo texto denormalizado) — nome aparece no topo mesmo sem o join funcionar, criando inconsistência visual.

Correção:

src/components/processo/sections/ProcessoDadosCliente.tsx 
// Antes (linha 17):
if (!processo.clientes) {
  return <EmptyState message="Nenhum cliente vinculado" />;
}

// Depois: fallback para cliente_nome quando clientes é null mas nome existe
if (!processo.clientes) {
  if (processo.cliente_nome) {
    return <ClienteNomeOnly nome={processo.cliente_nome} />;
  }
  return <EmptyState message="Nenhum cliente vinculado" />;
}

BUG #4 — P1 Página completa apenas
Data da restrição não exibida — COALESCE ausente no select direto
ProcessoRestricaoPrincipal.tsx:65
useProcessoBySlug.ts — select sem COALESCE
Causa Raiz

O ProcessoModalShell usa rpc_processo_get_completo que faz COALESCE(restricao_data_new, restricao_data) → data exibe corretamente no drawer.

O useProcessoBySlug (página completa) usa select('*') direto, sem COALESCE → processo.restricao_data pode ser null enquanto restricao_data_new tem o valor real.

O fallback em ProcessoRestricaoPrincipal só dispara quando !hasInlineData — se outros campos inline existem, dataRest = null e a seção "Data" não renderiza.

✅ Drawer (funciona)
ProcessoModalShell
  • usa rpc_processo_get_completo
  • RPC faz COALESCE
  • restricao_data correto
❌ Página completa (quebrado)
useProcessoBySlug
  • usa select('*') direto
  • sem COALESCE
  • restricao_data = null

Correção recomendada:

src/hooks/useProcessoBySlug.ts 
// Migrar useProcessoBySlug para usar rpc_processo_get_completo
// em vez de select('*') direto — elimina TODOS os problemas de COALESCE
const { data } = useQuery({
  queryKey: ['processo', numeroCnj, organizacaoId],
  queryFn: async () => {
    const { data, error } = await supabase
      .rpc('rpc_processo_get_completo', {
        p_numero_cnj: numeroCnj,
        p_organizacao_id: organizacaoId,
      });
    if (error) throw error;
    return data;
  },
  staleTime: 30_000,
});

BUG #5 — P1 NOVO Cache / Invalidação
Invalidação de cache incorreta após restaurar processo no drawer
ProcessoHeader.tsx:43
ProcessoModalShell.tsx:40
Causa Raiz

O ProcessoModalShell usa queryKey ['processo-modal', processoId].

O ProcessoHeader.handleRestore() invalida apenas:

  • ['processo', processo.numero_cnj]
  • ['processos']

A queryKey ['processo-modal', processoId] nunca é invalidada → drawer continua exibindo o banner de "excluído" mesmo após restauração, até o cache expirar (5 min).

Correção:

src/components/processo/ProcessoHeader.tsx — handleRestore() 
const handleRestore = async () => {
  // ... lógica de restauração ...

  // Invalidações existentes:
  queryClient.invalidateQueries({ queryKey: ['processo', processo.numero_cnj] });
  queryClient.invalidateQueries({ queryKey: ['processos'] });

  // ✅ Adicionar invalidação da queryKey do modal:
  queryClient.invalidateQueries({ queryKey: ['processo-modal', processo.id] });
};

Inconsistências de Arquitetura

Problema A
Dois data sources para o mesmo dado

A mesma aba Dados renderiza corretamente no drawer mas incorretamente na página completa porque cada contexto usa uma fonte diferente:

✅ Drawer — Correto
ProcessoModalShell
  • queryKey: ['processo-modal', id]
  • fonte: rpc_processo_get_completo
  • COALESCE aplicado
  • staleTime: 5 min
  • restricao_data ✅
  • todos os campos ✅
❌ Página completa — Inconsistente
useProcessoBySlug
  • queryKey: ['processo', cnj, orgId]
  • fonte: select('*') com joins
  • sem COALESCE
  • staleTime: 30s
  • restricao_data pode ser null ❌
  • whatsapp / data_nascimento / sexo ❌
💡 Recomendação: Unificar ambos para usar rpc_processo_get_completo — elimina todos os problemas de COALESCE e de campos faltantes de uma vez.
Problema B
QueryKeys inconsistentes — invalidação pode falhar
Contexto QueryKey Invalidado por Status
Modal / Drawer ['processo-modal', processoId] Nada (BUG #5) Quebrado
Página completa ['processo', numeroCnj, organizacaoId] ['processo', numeroCnj] (sem orgId) Pode falhar multi-org

A invalidação em ProcessoHeader omite organizacaoId da queryKey → pode não invalidar em cenários multi-tenant.
