Achados

❌ P0-001 — empresa_nome sempre null nas RPCs

Root cause: fn_acordos_listar e fn_acordos_exportar fazem JOIN com e.razao_social, mas a tabela empresas tem os dados em e.nome. A coluna razao_social esta vazia em todos os registros testados.

Impacto: Empresa nunca aparece no frontend. Rankings mostram "Empresa desconhecida".

-- FIX fn_acordos_listar (linha do SELECT):
-- DE: e.razao_social AS empresa_nome,
-- PARA: COALESCE(e.razao_social, e.nome, e.nome_fantasia) AS empresa_nome,

-- FIX fn_acordos_exportar (mesma mudanca):
-- DE: em.razao_social AS empresa_nome,
-- PARA: COALESCE(em.razao_social, em.nome, em.nome_fantasia) AS empresa_nome,

-- FIX vw_acordos_aguardando (usa e.nome, mas tambem deveria usar COALESCE):
-- DE: e.nome AS empresa_nome,
-- PARA: COALESCE(e.razao_social, e.nome, e.nome_fantasia) AS empresa_nome,

❌ P0-002 — fn_acordo_marcar_pago overload sem p_organizacao_id ainda existe

Root cause: A versao antiga (3 args) da RPC ainda existe no banco. Nao filtra por organizacao_id — qualquer usuario autenticado pode marcar qualquer acordo como pago se souber o UUID.

Impacto: Violacao de isolamento multi-tenant na versao antiga.

-- FIX: dropar a versao antiga
DROP FUNCTION IF EXISTS fn_acordo_marcar_pago(uuid, date, text);

⚠️ P1-001 — acordos_plataformas sem deleted_at

Tabela usa ativo boolean ao inves de deleted_at TIMESTAMPTZ. Padrão MdFlow exige soft delete. Frontend ja filtra por .is('deleted_at', null) em useAcordoFormData — que falha silenciosamente pois a coluna nao existe.

-- FIX:
ALTER TABLE acordos_plataformas ADD COLUMN deleted_at TIMESTAMPTZ;
CREATE INDEX idx_acordos_plataformas_deleted ON acordos_plataformas(organizacao_id) WHERE deleted_at IS NULL;

⚠️ P1-002 — fn_expirar_propostas_vencidas nao filtra deleted_at

O cron atualiza propostas soft-deleted para 'expirada', poluindo dados. Deveria filtrar AND deleted_at IS NULL.

-- FIX: adicionar ao WHERE:
AND deleted_at IS NULL

⚠️ P1-003 — fn_acordos_stats marcada como VOLATILE (deveria ser STABLE)

A funcao so faz leitura (SELECT). Marcar como STABLE permite ao PostgreSQL otimizar execucoes repetidas na mesma transacao. Impacto direto no tempo de resposta (atual: 181ms media).

-- FIX: recriar com STABLE
-- Na definicao: LANGUAGE plpgsql STABLE SECURITY DEFINER

💡 P2-001 — valor_proposta/valor_aceito sao NUMERIC, padrao MdFlow e BIGINT (centavos)

Funciona corretamente com NUMERIC mas diverge do padrao. Frontend usa formatBRLReais() que espera reais, nao centavos. A mudanca seria alto risco (precisa converter todos os valores existentes). Recomendacao: manter como esta, documentar excecao.

💡 P2-002 — fn_acordos_listar usa subquery correlata para total_propostas

(SELECT COUNT(*) FROM acordos_propostas ap2 WHERE ap2.acordo_id = a.id) — executa N queries. Com 7 registros nao importa, mas com 500+ vai degradar. Solucao: LEFT JOIN com GROUP BY ou window function.

💡 P2-003 — Indice parcial em acordos(organizacao_id) WHERE deleted_at IS NULL ausente

Existe idx_acordos_org sem filtro parcial. Adicionar indice parcial otimizaria todas as queries que filtram por deleted_at IS NULL (que sao todas).

CREATE INDEX idx_acordos_org_active ON acordos(organizacao_id) WHERE deleted_at IS NULL;

Diagnostico Geral

Backend dos acordos esta solidamente construido — RLS perfeito, RPCs com SECURITY DEFINER, CHECK constraints em tudo, triggers de audit e sync financeiro. O ponto critico e o BUG da empresa_nome: as RPCs fazem JOIN com razao_social mas os dados estao em nome. Isso e um fix de 1 linha em 3 funcoes. A versao antiga de fn_acordo_marcar_pago (sem organizacao_id) deve ser dropada — e um risco de seguranca residual. O resto sao melhorias incrementais (STABLE em stats, indice parcial, subquery correlata). Nota: 78% — bom fundamento, 2 fixes urgentes.
