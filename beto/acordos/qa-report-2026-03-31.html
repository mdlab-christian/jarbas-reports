Bug Cards Detalhados

P0 BUG-008: Colunas Propostas desalinhadas com headers

Arquivo: src/pages/acordos/tabs/PropostasTab.tsx linhas 199-210

Arquivo: src/pages/acordos/tabs/propostas/PropostaRow.tsx linhas 72-243

Causa: O header da tabela define a ordem: Checkbox, Cliente/Processo, Empresa, Status, Valor, Contraproposta, Canal, Validade, Agendado, Interesse, Visualizacao, Respondente, Acoes. Porem o PropostaRowComponent renderiza as celulas na ordem: ...Valor, Canal, Validade, Agendado, Interesse, Visualizacao, Contraproposta, Respondente, Acoes. A coluna Contraproposta no header esta na posicao 6 mas os dados aparecem na posicao 11.

Impacto: Os dados de cada proposta sao exibidos sob o header errado. Canal aparece embaixo de "Contraproposta", Validade embaixo de "Canal", etc. Todos os dados a partir da 6a coluna estao deslocados.

Solucao: Mover o <TableCell> de Contraproposta no PropostaRow.tsx (atualmente linhas 179-191) para logo apos a celula de Valor (apos linha 121), OU mover o <TableHead> de "Contraproposta" para depois de "Visualizacao" no header.

P0 BUG-001: acordos_plataformas sem deleted_at

Tabela: acordos_plataformas (Supabase)

Causa: A tabela nao possui coluna deleted_at TIMESTAMPTZ, violando o padrao obrigatorio de soft delete do MdFlow (CLAUDE.md secao 4).

Evidencia: Query SELECT deleted_at FROM acordos_plataformas LIMIT 1 retorna erro 42703: "column acordos_plataformas.deleted_at does not exist".

Impacto: Plataformas desativadas nao podem ser soft-deleted, apenas desativadas via flag ativo. O ModalGerenciarPlataformas nao filtra por deleted_at porque a coluna nao existe. Nao ha hard delete implementado, entao o risco atual e baixo, mas viola o contrato arquitetural.

Solucao: Migration: ALTER TABLE acordos_plataformas ADD COLUMN deleted_at TIMESTAMPTZ; + index + atualizar query em ModalGerenciarPlataformas.tsx para filtrar .is('deleted_at', null).

P1 BUG-007: DashboardRankings usa fn_acordos_listar com limit 200

Arquivo: src/pages/acordos/tabs/dashboard/DashboardRankings.tsx linha 41

Causa: Busca 200 acordos completos apenas para extrair empresa_id e contar ocorrencias client-side. Desperdicio de bandwidth/processamento.

Impacto: Com volume alto de acordos (>200), o ranking fica incompleto. Transfere dados desnecessarios (valores, datas, CNJ) quando so precisa de empresa_id + count.

Solucao: Criar RPC fn_acordos_top_reus(p_organizacao_id, p_limit) que retorne GROUP BY empresa_id com count server-side.

P1 BUG-009: fn_acordo_marcar_pago sem p_organizacao_id no frontend

Arquivo: src/pages/acordos/modals/ModalMarcarPago.tsx linha 95

Causa: A chamada supabase.rpc('fn_acordo_marcar_pago', ...) nao envia p_organizacao_id. A spec da RPC aceita esse parametro (vide checklist), mas o frontend omite. Se a RPC depende de public.get_org_id() via JWT, funciona com autenticacao normal. Mas se a RPC usa o parametro, a omissao pode causar erro.

Impacto: Potencial falha ao marcar pago se a RPC espera o parametro. A resposta com ID inexistente retorna corretamente {success:false, message:"Acordo nao encontrado"}, confirmando que a RPC valida.

Solucao: Verificar assinatura da RPC. Se aceita p_organizacao_id, adicionar ao call. Se usa get_org_id(), OK como esta.

P1 BUG-010: ModalGerenciarPlataformas sem filtro deleted_at nem soft delete

Arquivo: src/pages/acordos/modals/ModalGerenciarPlataformas.tsx linhas 68-76

Causa: Consequencia do BUG-001 -- como a coluna nao existe, o modal lista TODAS as plataformas sem filtro de soft delete. O modal tambem nao oferece opcao de "excluir" plataforma, apenas ativar/desativar.

Impacto: Nao e possivel remover plataformas permanentemente (nem soft delete). Plataformas ficam acumulando indefinidamente.

Solucao: Apos migration BUG-001, adicionar botao de exclusao (soft delete) e filtro .is('deleted_at', null).

P2 BUG-011: DashboardTab staleTime inconsistente

Arquivo: src/pages/acordos/tabs/DashboardTab.tsx linhas 34 e 52

Causa: Os hooks locais do DashboardTab usam staleTime: 5 * 60 * 1000 (5 min), enquanto o padrao MdFlow e staleTime: 30_000 (30s). O useAcordosStats no hook dedicado usa 30s corretamente.

Impacto: Dados do dashboard podem ficar stale por 5 minutos em vez de 30 segundos. Pode causar confusao quando o usuario atualiza dados em outra aba e volta ao dashboard.

Solucao: Alinhar para staleTime: 30_000 ou documentar por que 5 min e aceitavel para KPIs.

P2 BUG-012: useAcordoFormData nao filtra deleted_at em plataformas

Arquivo: src/pages/acordos/hooks/useAcordoFormData.ts linha 9

Causa: Query filtra por .eq('ativo', true) mas nao por .is('deleted_at', null). Atualmente irrelevante porque deleted_at nao existe na tabela (BUG-001), mas apos a migration, este query precisa ser atualizado.

Impacto: Baixo atualmente. Potencial apos migration.

Solucao: Apos migration BUG-001, adicionar .is('deleted_at', null).

P2 BUG-013: empresa_nome retorna null em acordos reais

Tabela: acordos via fn_acordos_listar

Causa: Dos 6 acordos aguardando pagamento, apenas 2 tem empresa_id preenchido, e mesmo esses retornam empresa_nome: null na RPC. O JOIN na RPC pode estar falhando ou os registros de empresas nao existem.

Impacto: Cards de acordos e drawer nao exibem nome da empresa, dificultando identificacao. Rankings de "Top Reus" ficam vazios para esses acordos.

Solucao: Verificar a RPC fn_acordos_listar -- provavel LEFT JOIN incorreto ou empresas_id referenciando registros inexistentes. Tambem verificar seed data.
