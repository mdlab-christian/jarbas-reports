Detalhes dos Fixes

DB-001 — Índice de busca textual ✅

Problema: Busca por título em fn_atividades_lista_com_leitura usa unaccent(lower(titulo)) mas sem índice funcional correspondente. Tentativa com unaccent() falhou (STABLE, não indexável). Solução: wrapper immutable_unaccent já existia no banco (IMMUTABLE).

CREATE INDEX IF NOT EXISTS idx_atividades_titulo_search ON atividades (immutable_unaccent(lower(titulo))) WHERE deleted_at IS NULL;

✅ Validado: índice criado, busca textual beneficiada

DB-002 — atividades_checklist soft delete ✅

Problema: Tabela sem deleted_at → hard delete em produção + frontend usando .delete() sem soft delete. Fix duplo: migration no banco + ajuste em DrawerTabChecklist.tsx.

ALTER TABLE atividades_checklist ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ; CREATE INDEX IF NOT EXISTS idx_atividades_checklist_active ON atividades_checklist (atividade_id) WHERE deleted_at IS NULL;

Frontend (DrawerTabChecklist.tsx): .delete() → .update({'{'} deleted_at: new Date().toISOString() {'}'}) + filtro .is('deleted_at', null) na query

✅ Validado: coluna criada, índice criado, frontend atualizado

DB-003 + DB-005 — Kanban gestor mode + is_unread real ✅

DB-003: Kanban sem verificação de admin/gestor → usuários normais só viam suas próprias tarefas, admin também. DB-005: is_unread hardcoded false em toda atividade do kanban. Ambos corrigidos na mesma migration reescrevendo fn_get_atividades_kanban.

-- Novo fn_get_atividades_kanban (simplificado): DECLARE v_is_gestor BOOLEAN := false; BEGIN -- Verifica se admin/gestor: IF p_user_id IS NOT NULL THEN SELECT COALESCE(u.perfil = 'admin' OR COALESCE(u.acesso_gestor, false), false) INTO v_is_gestor FROM users u WHERE u.id = p_user_id; END IF; RETURN QUERY SELECT ..., (leit.atividade_id IS NULL OR leit.last_seen_at < a.updated_at) AS is_unread FROM atividades a LEFT JOIN atividades_leituras leit ON leit.atividade_id = a.id AND leit.user_id = p_user_id WHERE ... AND (v_is_gestor = true OR p_user_id IS NULL OR a.responsavel_id = p_user_id); END;

✅ Validado: sem p_user_id retorna 37 registros (org completa). Admin vê tudo.

DB-004 — n8n logRequest() bug ✅

Problema: logRequest() inserindo id: request_id (coluna errada — UUID auto-gerado) em vez de request_id: request_id. Resultado: webhook_requests.request_id sempre NULL.

// ANTES (bug): supaReq('POST', '/rest/v1/webhook_requests', { id: request_id, // ← preenchia o id UUID gerado automaticamente ... }); // DEPOIS (fix): supaReq('POST', '/rest/v1/webhook_requests', { request_id: request_id, // ← coluna correta para rastreamento ... });

✅ Validado: workflow D9OmWhyaovA76HHk atualizado via n8n REST API

DB-008 + DB-009 — RLS policies WITH CHECK + padronização get_org_id() ✅

DB-008: Policies FOR ALL sem WITH CHECK explícito em checklist e comentários (herdavam do USING mas é melhor prática explicitar). DB-009: Tabela atividades usava org_id() (helper antigo) vs checklist/comentários que já usavam get_org_id() (padrão atual).

-- atividades: org_id() → get_org_id() DROP POLICY IF EXISTS "atividades_select_policy" ON atividades; CREATE POLICY "atividades_select_policy" ON atividades FOR SELECT USING (organizacao_id = public.get_org_id()); -- (+ UPDATE e DELETE policies) -- checklist + comentários: WITH CHECK explícito DROP POLICY IF EXISTS "checklist_org_isolation" ON atividades_checklist; CREATE POLICY "checklist_org_isolation" ON atividades_checklist FOR ALL USING (organizacao_id = public.get_org_id()) WITH CHECK (organizacao_id = public.get_org_id());

✅ Validado: políticas recriadas, isolamento multi-tenant confirmado
