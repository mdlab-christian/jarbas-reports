3. Auditoria de RPCs

fn_atividades_lista_com_leitura

SECURITY DEFINER search_path fixado paginação LIMIT/OFFSET unaccent para busca is_unread pipeline filtra deleted_at
Esta é a RPC mais completa do sistema
Retorna 30 campos, integra vw_atividades_lista com atividades_leituras para calcular is_unread. Ordenação inteligente (atrasadas primeiro, urgentes depois). Sem problemas críticos.
P2 — Sem filtro por responsavel_id no server
Os filtros responsavel_id, prioridade e processo_id são aplicados client-side no frontend após retorno. Com carga alta, isso fará a RPC retornar dados desnecessários antes de filtrar. Adicionar p_responsavel_id e p_prioridade como parâmetros opcionais.

fn_get_atividades_kanban

SECURITY DEFINER search_path fixado p_user_id DEFAULT NULL
P1 — Kanban filtra por responsavel_id = p_user_id quando user_id fornecido
Frontend sempre passa p_user_id = user.user_id. Isso significa que o kanban mostra apenas atividades do usuário logado como responsável, excluindo atividades onde ele é participante ou watcher. Um gestor que quer ver o kanban da equipe não consegue — a RPC não tem modo "ver tudo" para admins, ao contrário da fn_atividades_kpis.
P2 — is_unread hardcoded como false
O campo is_unread no kanban está fixo como false, sem consultar atividades_leituras. O kanban nunca mostrará badges de "novo" corretamente.

fn_atividades_kpis

SECURITY DEFINER admin ver tudo não-admin vê só suas
P1 — KPIs não refletem visão de equipe para não-admins
Um usuário juridico sem acesso_gestor verá KPIs (pendentes, atrasadas) apenas das suas próprias atividades. Ao mesmo tempo, fn_atividades_lista_com_leitura retorna atividades de toda a org sem filtro por user. Inconsistência: lista mostra tudo, KPIs mostram só as suas.
Lógica de gestores implementada corretamente
A flag acesso_gestor é consultada via JOIN na tabela users e determina visibilidade. Adminss e gestores veem KPIs de toda a organização.

fn_alterar_status_atividade

SECURITY DEFINER validação de status verifica existência filtra por organizacao_id retorna success boolean
RPC bem construída — sem problemas críticos
Valida status, verifica existência + org_id, atualiza timestamps corretos (concluida_em, cancelada_em, em_andamento_em), retorna JSONB com success. Teste com UUID inválido retornou corretamente success: false, error: "Atividade não encontrada".

fn_trigger_atividade_eventos

P2 — Trigger de eventos só captura criação e mudança de status
Ao criar: registra "Atividade criada". Ao mudar status: registra "Status alterado". Mas mudanças em titulo, descricao, responsavel_id, prazo_fatal não geram eventos. A aba "Histórico" no drawer fica incompleta para edições de campos.

Inconsistência de função helper RLS

P2 — Mistura de org_id() e get_org_id() nas policies
atividades usa org_id() nas policies. atividades_checklist e atividades_comentarios usam get_org_id(). get_org_id() é um wrapper que chama org_id() — funcionalmente idêntico. Mas a inconsistência é confusa para manutenção futura. Padronizar para public.get_org_id() conforme especificado no CLAUDE.md.

Policy INSERT sem WITH CHECK

P2 — INSERT policies sem WITH CHECK
As policies de INSERT em atividades, atividades_checklist, etc. têm qual = null (sem WITH CHECK). Isso significa que qualquer usuário autenticado pode inserir dados sem validação de organizacao_id via RLS. O controle é feito via SECURITY DEFINER nas RPCs, mas acesso direto à tabela não valida org. Risco baixo pois o anon_key não deveria ter INSERT direto, mas é boa prática adicionar WITH CHECK.

P1 — request_id sempre NULL em webhook_requests
O workflow n8n faz INSERT em webhook_requests usando id: request_id (campo id da tabela, não request_id). Mas webhook_requests provavelmente tem request_id como coluna separada de id. Os 3 últimos registros têm request_id = NULL. Isso quebra rastreabilidade de webhooks — não é possível correlacionar frontend request com execução n8n.
P2 — Ação "editar" não implementada no n8n
O frontend edita campos diretamente via supabase.from('atividades').update() (DrawerTabDetalhes.tsx). Não existe action "editar" no workflow n8n. Isso viola o princípio n8n é o Motor do CLAUDE.md para operações multi-step, mas é aceitável para edições simples de campos (descricao, urgente, importante) que não requerem orquestração.

9. Parecer do Dev

Saúde: 78% — Funcional com tech debt acumulado

O backend de /atividades está funcional e seguro. Sem falhas de segurança críticas — RLS está ativo em 10 das 11 tabelas, todas RPCs são SECURITY DEFINER com search_path fixado, e o isolamento multi-tenant foi testado e confirmado.

O maior problema real é a inconsistência de escopo entre as RPCs: fn_atividades_lista_com_leitura retorna tudo da org, fn_get_atividades_kanban retorna só do usuário logado, e fn_atividades_kpis retorna só do usuário logado (exceto admins). Isso gera uma experiência incoerente: um gestor vê na lista atividades de toda a equipe, mas os KPIs e o Kanban mostram só as suas.

O segundo problema é o hard delete em atividades_checklist — viola o padrão soft delete do projeto. Com 2 linhas agora é inofensivo, mas vai criar surpresas quando o histórico de checklists for solicitado.

O webhook request_id = NULL é um bug de auditoria silencioso — não afeta o funcionamento mas impede rastreabilidade completa de operações em produção.

FIX-3 (P1): Adicionar acesso gestor ao kanban

-- Modificar fn_get_atividades_kanban para respeitar acesso_gestor
CREATE OR REPLACE FUNCTION public.fn_get_atividades_kanban(
  p_organizacao_id uuid,
  p_user_id uuid DEFAULT NULL::uuid
)
RETURNS TABLE(...)
LANGUAGE plpgsql SECURITY DEFINER SET search_path TO 'public'
AS $$
DECLARE
  v_is_admin BOOLEAN := false;
BEGIN
  -- Verificar se usuário é admin/gestor
  IF p_user_id IS NOT NULL THEN
    SELECT (COALESCE(perfil,'') = 'admin' OR COALESCE(acesso_gestor, false))
    INTO v_is_admin
    FROM users WHERE id = p_user_id;
  END IF;

  RETURN QUERY
  SELECT ...
  FROM atividades a
  ...
  WHERE a.organizacao_id = p_organizacao_id
    AND a.deleted_at IS NULL
    AND a.status IN ('pendente', 'em_andamento', 'concluida')
    AND (
      v_is_admin = true  -- admin vê tudo
      OR p_user_id IS NULL  -- sem filtro de user
      OR a.responsavel_id = p_user_id  -- apenas as suas
    )
  ORDER BY ...;
END;
$$;
