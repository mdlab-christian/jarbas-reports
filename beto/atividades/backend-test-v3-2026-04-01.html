Diagnóstico Geral — Ciclo 3

O backend da página /atividades atingiu 91.5% de saúde nesta terceira auditoria, superando a meta de 90%. O fix DB2-001 (INSERT policy) foi confirmado e todas as 4 policies da tabela atividades agora utilizam o padrão canônico public.get_org_id().

  • Pontos fortes: RLS perfeito (10/10), 12 RPCs todas SECURITY DEFINER, 27 índices bem distribuídos com partial indexes otimizados, 8 triggers cobrindo consistency + audit trail, webhook com envelope canônico completo.
  • Tech debt aceitável: 3 campos date (DB2-002) são adequados para casos de uso de prazo jurídico sem timezone. Migração para timestamptz tem risco de regressão sem benefício prático imediato.
  • By design (DB2-003): Filtragem via parâmetros nas RPCs SECURITY DEFINER é o padrão correto quando a função precisa bypass de RLS para servir dados multi-contexto (admin vs user). O isolamento é garantido pelo WHERE explícito.
  • Performance: Todos os tempos <250ms com o conjunto de dados atual. Os partial indexes garantem escalabilidade. Nenhuma query N+1 identificada nas definições.

Findings P2 Restantes — Tech Debt
2 findings · Aceitáveis · Não bloqueantes
P2 DB2-002 Campos de data em tipo date em vez de timestamptz
Os campos data_vencimento, data_atividade e prazo_fatal utilizam tipo date (sem informação de timezone). Para contexto jurídico de prazos processuais, date é semanticamente correto — prazos são datas calendário, não instantes. Porém diverge do padrão TIMESTAMPTZ do CLAUDE.md. Risco de regressão em migração > benefício.
Impacto: Baixo Esforço: Médio Decisão CTO necessária
-- SQL sugerido (SOMENTE após aprovação CTO, com backup): ALTER TABLE atividades ALTER COLUMN data_vencimento TYPE TIMESTAMPTZ USING data_vencimento::timestamptz, ALTER COLUMN data_atividade TYPE TIMESTAMPTZ USING data_atividade::timestamptz, ALTER COLUMN prazo_fatal TYPE TIMESTAMPTZ USING prazo_fatal::timestamptz; -- ATENÇÃO: requer atualização de todos os índices que referenciam esses campos -- e ajuste nas RPCs que comparam com CURRENT_DATE
P2 DB2-003 RPCs filtram via parâmetros em vez de RLS puro
fn_atividades_lista_com_leitura e fn_get_atividades_kanban são SECURITY DEFINER e filtram por organizacao_id via WHERE v.organizacao_id = p_organizacao_id em vez de depender exclusivamente do RLS. Este é o padrão correto e intencional para funções que precisam de controle granular (admin vê todos, user vê só os seus). O risco é que um bug de parâmetro pode vazar dados — mas o teste com org fictícia confirmou isolamento correto.
Impacto: Teórico Status: By design Nenhuma ação necessária
-- Recomendação: adicionar assertion de segurança nas RPCs críticas: -- Verificar que p_organizacao_id não é NULL antes de retornar dados IF p_organizacao_id IS NULL THEN RAISE EXCEPTION 'organizacao_id obrigatório'; END IF; -- (baixa prioridade — não há caminho de código que passe NULL)
