🔴 Bugs Críticos — P0

🔴 BUG-001 P0 — CRASH Error Boundary após criar atividade — TypeError: Cannot read properties of undefined (reading 'toLocaleString')
📁 src/pages/atividades/kpis/AtividadesKpiGrid.tsx — linha 51
Snippet de código
// AtividadesKpiGrid.tsx — linhas 47-53
function NumberTicker({ value }: { value: number }) {
  return (
    <span className="font-display text-3xl font-bold tracking-tight tabular-nums">
      {value.toLocaleString('pt-BR')}  // ← CRASH: value pode ser undefined
    </span>
  );
}
Causa Raiz Técnica

Após criar uma nova atividade, queryClient.invalidateQueries({ queryKey: ['atividades'] }) é chamado no NovaAtividadeModal. Isso força o React Query a re-executar a query de KPIs. Durante o intervalo entre invalidação e refetch, o campo data[k.key] retorna undefined. O componente NumberTicker não tem guard para esse caso e tenta chamar undefined.toLocaleString('pt-BR'), disparando TypeError não capturado.

O error boundary da aplicação captura o erro e renderiza a tela "Algo deu errado", interrompendo completamente o fluxo do usuário.

Console Error Confirmado
TypeError: Cannot read properties of undefined (reading 'toLocaleString')
    at St (index-DjtZWUzg.js:1:10216)
    at xy (index-BP4yqHGB.js:39:16998)
    Timestamp: 2026-03-30T22:48:03.452Z
Impacto

🚨 BLOQUEANTE — A página crasheia após criar qualquer atividade. Usuário precisa recarregar manualmente. UX completamente quebrada no fluxo principal de criação.

Solução Recomendada
// Adicionar guard na função NumberTicker:
function NumberTicker({ value }: { value: number | undefined }) {
  if (value === undefined || value === null) return <span className="...">—</span>;
  return (
    <span className="...">
      {value.toLocaleString('pt-BR')}
    </span>
  );
}
🔴 BUG-002 P0 — KPIs ZERADOS KPIs todos = 0 para usuários admin — fn_atividades_kpis filtra apenas por responsável
📁 src/pages/atividades/kpis/AtividadesKpiGrid.tsx — linhas 23-36 | Supabase: RPC fn_atividades_kpis
Snippet de código
// AtividadesKpiGrid.tsx — linhas 23-36
const { data, error } = await supabase.rpc('fn_atividades_kpis', {
  p_organizacao_id: organizacao_id,
  p_user_id: user.user_id,  // ← Filtra APENAS atividades onde responsavel_id = user.user_id
});
// Mas a lista mostra TODAS as atividades (admin vê tudo)
// Resultado: Lista exibe 36 atividades, KPIs exibem 0
Evidência no Banco
-- fn_atividades_kpis com p_user_id de Gustavo:
{ total: 0, pendentes: 0, em_andamento: 0, atrasadas: 0 }

-- Atividades no banco da organização:
Content-Range: 0-36/37 (37 atividades totais)

-- Nenhuma atividade tem Gustavo como responsavel_id
Causa Raiz

A RPC calcula KPIs apenas para atividades onde responsavel_id = p_user_id. Para usuários com perfil admin que visualizam todas as atividades da organização, os KPIs ficam zerados pois nenhuma atividade foi atribuída a eles como responsável. A lista usa fn_atividades_lista_com_leitura que retorna todas as atividades da organização (sem filtrar por responsável), criando inconsistência visual grave.

Impacto

⚠️ ALTO — Usuários admin/gestor veem KPIs = 0 enquanto a lista mostra 36 atividades. Dashboard sem valor informativo para administradores.

Solução Recomendada

Adicionar parâmetro p_is_admin boolean na RPC para retornar KPIs da organização inteira quando admin=true. Ou criar RPC separada fn_atividades_kpis_org para administradores.
