Bugs P0 — Críticos (bloqueiam funcionalidade core)
BUG-P0-001 KPIs mostram contagem incorreta para perfil admin/gestor P0 Persiste do R1 como BUG-002
Arquivo
src/pages/atividades/kpis/AtividadesKpiGrid.tsx (linha 24)
banco: fn_atividades_kpis
Impacto
Admin/gestor vê KPI = 1 pendente, mas org tem 24 pendentes. KPI inutilizável para supervisores.
Causa Raiz

A RPC fn_atividades_kpis é user-scoped: filtra por responsavel_id = p_user_id. O perfil admin Gustavo tem apenas 1 atividade atribuída a si, logo KPI retorna pendentes=1. Já a RPC fn_atividades_lista_com_leitura é org-scoped para admin (retorna 37 atividades). A inconsistência cria uma ilusão: a lista mostra dezenas de itens, mas o KPI diz "total 1".

Evidência Supabase
-- fn_atividades_kpis para Gustavo (admin): { pendentes: 1, em_andamento: 0, atrasadas: 0 } -- Realidade no banco: SELECT status, count(*) FROM atividades WHERE deleted_at IS NULL GROUP BY status; → pendente: 24 → em_andamento: 5 → concluida: 7 -- fn_atividades_lista_com_leitura total_count para Gustavo: → 37 atividades (org-scoped)
Solução Proposta

Na RPC fn_atividades_kpis: verificar se o usuário tem perfil = 'admin' ou acesso_gestor = true. Se sim, contar TODAS as atividades da org (sem filtro por responsavel_id). Se não, contar apenas as atividades onde responsavel_id = p_user_id.

-- Lógica proposta na RPC: IF (SELECT perfil = 'admin' OR acesso_gestor FROM users WHERE id = p_user_id) THEN -- Contar todas da org SELECT count(*) FILTER (WHERE status = 'pendente') ... FROM atividades WHERE organizacao_id = p_organizacao_id AND deleted_at IS NULL ELSE -- Contar apenas as do responsavel SELECT count(*) FILTER (WHERE status = 'pendente') ... FROM atividades WHERE responsavel_id = p_user_id AND deleted_at IS NULL END IF
BUG-P0-002 Kanban completamente quebrado — fn_get_atividades_kanban erro tipo data (42804) P0 BUG-003 R1 em nova forma
Arquivo
banco: fn_get_atividades_kanban
src/pages/atividades/hooks/useAtividadesKanban.ts (linha 57)
Impacto
View Kanban sempre carrega em estado de erro (EmptyState com "Recarregar"). Kanban inutilizável 100%.
Causa Raiz

A RPC fn_get_atividades_kanban foi atualizada mas seu RETURN TYPE declarations não foram sincronizadas com a estrutura real da tabela. O PostgreSQL detecta incompatibilidade no column 7 da função: a RPC declara data_atividade TIMESTAMPTZ mas a coluna atividades.data_atividade é do tipo DATE.

Evidência Supabase
-- Chamada à RPC: POST /rest/v1/rpc/fn_get_atividades_kanban { p_organizacao_id: "...", p_user_id: "..." } -- Resposta de erro: { "code": "42804", "details": "Returned type date does not match expected type timestamp with time zone in column 7.", "hint": null, "message": "structure of query does not match function result type" } -- Confirmação do tipo real no banco: SELECT data_atividade, prazo_fatal FROM atividades LIMIT 1; → data_atividade: null (DATE type) → prazo_fatal: "2026-03-18" (DATE type — confirma que campo é DATE não TIMESTAMPTZ)
Contexto Adicional

A RPC fn_atividades_lista_com_leitura retorna data_atividade: null (DATE) sem erro. O problema é específico à declaração de RETURN TYPE em fn_get_atividades_kanban.

Solução Proposta

Recriar a função fn_get_atividades_kanban com o RETURN TYPE correto. Opção A: alterar declaração de data_atividade TIMESTAMPTZ para data_atividade DATE. Opção B (mais robusto): converter o campo para TEXT no SELECT usando data_atividade::TEXT, consistente com prazo_fatal.

-- Migration sugerida: CREATE OR REPLACE FUNCTION fn_get_atividades_kanban(...) RETURNS TABLE ( ... data_atividade DATE, -- era TIMESTAMPTZ, deve ser DATE prazo_fatal DATE, prazo_interno TIMESTAMPTZ, ... ) ...

BUG-P1-001 View Feed sempre vazia — nenhum evento de atividade registrado no banco P1
Arquivo
views/AtividadesFeed.tsx (linha 63)
banco: eventos_operacionais
Impacto
Feed exibe "Nenhuma atividade no feed" mesmo com 37 atividades existentes. Funcionalidade de auditoria inoperante.
Causa Raiz

A tabela eventos_operacionais não contém nenhum registro com entity_type = 'atividade'. Nenhum workflow n8n ou trigger PostgreSQL está gerando eventos quando atividades são criadas, atualizadas ou têm status alterado.

Evidência Supabase
-- Query do AtividadesFeed: SELECT * FROM eventos_operacionais WHERE organizacao_id = '...' AND entity_type = 'atividade' ORDER BY created_at DESC; → [] (0 registros) -- Total na tabela (sem filtro): SELECT * FROM eventos_operacionais LIMIT 1; → [] (tabela completamente vazia no projeto)
Contexto: DrawerTabEventos Funciona

O DrawerTabEventos permite inserir comentários no histórico. O INSERT é estruturalmente correto e foi validado via curl (sucesso com todos os campos obrigatórios: tipo, entidade_tipo, entidade_id, entity_type, etc.). O problema é que nunca houve eventos criados automaticamente pelo fluxo de negócio.

Solução Proposta

Duas frentes: (A) Trigger PostgreSQL: criar trigger AFTER INSERT OR UPDATE ON atividades que insere em eventos_operacionais para eventos de criação e mudança de status. (B) Workflow n8n: garantir que o webhook atividades/crud insere evento ao criar/concluir/cancelar atividade.

BUG-P1-002 ActivityCard (view lista) não mostra cliente nem empresa — AtividadeRow sem esses campos P1
Arquivo
cards/AtividadesActivityCard.tsx
atividades.types.ts (interface AtividadeRow)
Impacto
Na view Lista, cards mostram apenas título, CNJ, responsável e prazo. Sem contexto de cliente/empresa, difícil identificar rapidamente de qual processo/cliente trata a atividade.
Causa Raiz

O tipo AtividadeRow (usado na view Lista) contém cliente_id e empresa_id mas NÃO cliente_nome ou empresa_nome. A RPC fn_atividades_lista_com_leitura retorna esses campos em AtividadeListaRpc, mas a função mapRpcToAtividadeRow() descarta cliente_nome e empresa_nome ao mapear para AtividadeRow. O AtividadesActivityCard usa AtividadeRow e portanto não tem acesso aos nomes.

Comparação: KanbanCard vs ActivityCard
-- KanbanCard usa KanbanItem que TEM cliente_nome: interface KanbanItem { cliente_nome: string | null; ✓ exibido no card responsavel_nome: string | null; processo_numero: string | null; } -- ActivityCard usa AtividadeRow que NÃO TEM cliente_nome: interface AtividadeRow { cliente_id: string | null; ✗ apenas ID, sem nome empresa_id: string | null; ✗ apenas ID, sem nome }
Solução Proposta

Adicionar cliente_nome: string | null e empresa_nome: string | null ao tipo AtividadeRow. Atualizar mapRpcToAtividadeRow() para incluir esses campos da AtividadeListaRpc. Atualizar AtividadesActivityCard para renderizar atividade.cliente_nome abaixo do título (similar ao KanbanCard).

// Em atividades.types.ts — mapRpcToAtividadeRow, adicionar: cliente_nome: r.cliente_nome ?? null, empresa_nome: r.empresa_nome ?? null, // Em AtividadesActivityCard.tsx, adicionar abaixo do título: {(atividade.cliente_nome || atividade.empresa_nome) && ( <p className="text-xs text-muted-foreground truncate mt-0.5"> {atividade.cliente_nome ?? atividade.empresa_nome} </p> )}
