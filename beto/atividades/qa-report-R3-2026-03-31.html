2. Bugs por Prioridade

P0 — Bloqueadores Críticos (1)

P0 BUG-R3-P0-001
fn_alterar_status_atividade — erro 42804: CASE types UUID e TEXT incompatíveis
Backend

supabase/migrations/20260306_C5_rpcs_p1.sql linha 414

Qualquer chamada a fn_alterar_status_atividade retorna erro PostgreSQL 42804. Isso bloqueia: (1) mudar status para "pendente" ou "em_andamento" via drawer; (2) concluir via webhook n8n (que chama a mesma RPC internamente); (3) alterar status no Kanban para pendente/em_andamento via drag & drop. Resultado: mudança de status TOTALMENTE quebrada — apenas UPDATE direto funciona.

CASE expression na linha 414 mistura tipos incompatíveis: THEN retorna p_user_id (tipo UUID), ELSE retorna concluida_por_id (tipo TEXT no banco). Comparar com linha 416 que usa corretamente p_user_id::TEXT para cancelada_por. 

-- ERRO (linha 414 da migration):
concluida_por_id = CASE WHEN p_novo_status = 'concluida' THEN p_user_id ELSE concluida_por_id END,

-- Resposta do banco:
{"code": "42804", "message": "CASE types uuid and text cannot be matched"}

-- Linha 416 (correto, como referência):
cancelada_por = CASE WHEN p_novo_status = 'cancelada' THEN p_user_id::TEXT ELSE cancelada_por END,
-- FIX: adicionar cast ::TEXT na linha 414
concluida_por_id = CASE WHEN p_novo_status = 'concluida' THEN p_user_id::TEXT ELSE concluida_por_id END,

-- Migration a criar:
CREATE OR REPLACE FUNCTION fn_alterar_status_atividade(
  p_atividade_id   UUID,
  p_organizacao_id UUID,
  p_user_id        UUID,
  p_novo_status    TEXT,
  p_motivo         TEXT DEFAULT NULL
)
RETURNS JSONB ...
-- Apenas alterar a linha 414 adicionando ::TEXT

P1 — Importantes (2)

P1 BUG-R3-P1-001
fn_get_atividades_kanban não retorna empresa_nome — kanban mudo para atividades de empresa
Backend

RPC fn_get_atividades_kanban (banco) + src/pages/atividades/hooks/useAtividadesKanban.ts

AtividadesKanbanCard.tsx renderiza {item.cliente_nome ?? item.empresa_nome}. A RPC retorna cliente_nome mas não inclui empresa_nome. Atividades vinculadas apenas a empresa (sem cliente) mostrarão linha em branco onde deveria aparecer o nome da empresa. Bug silencioso — TypeScript não acusa pois KanbanItem declara empresa_nome: string | null mas a RPC simplesmente não inclui o campo (retorna undefined, tratado como null pelo fallback). 

-- Campos retornados pela RPC (confirmado via curl):
['id', 'titulo', 'status', 'urgente', 'importante', 'prioridade',
 'prazo_fatal', 'tipo_nome', 'tipo_cor', 'cliente_nome',
 'responsavel_nome', 'processo_numero', 'posicao', 'is_atrasada', 'is_unread']

-- 'empresa_nome' AUSENTE do resultado

-- KanbanItem type (useAtividadesKanban.ts) declara:
empresa_nome: string | null;  // declarado mas RPC não retorna

Adicionar empresa_nome ao SELECT da fn_get_atividades_kanban via LEFT JOIN em empresas.

P1 BUG-R3-P1-002
Webhook atividades/crud (criar) não registra em webhook_requests
Backend/n8n

Chamada ao webhook atividades/crud com contexto atividades.criar retornou success: true e criou a atividade no banco (ID: 9609b2fd), mas nenhum registro foi criado em webhook_requests. Pesquisa com contexto=eq.atividades.criar retorna array vazio.

Auditoria e rastreabilidade de criações de atividades perdidas. Não é possível monitorar falhas/retries do workflow n8n para criar atividade. LGPD compliance comprometida (log de ações obrigatório).

No cenário n8n atividades/crud, adicionar nó de INSERT em webhook_requests com request_id, contexto, organizacao_id, actor_user_id e status=completed ao final do fluxo de criação.
