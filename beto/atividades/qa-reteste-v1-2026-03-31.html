FIX-1 Kanban — erro 42804 resolvido (prazo_fatal DATE) ✅ PASSOU

Contexto

A função fn_get_atividades_kanban declarava prazo_fatal TIMESTAMPTZ no RETURNS TABLE, mas a coluna real na tabela atividades é do tipo DATE. O PostgreSQL levantava erro 42804 (incompatibilidade de tipo) e o kanban não carregava nenhuma atividade.

Migração aplicada

20260331000004 + 20260331000007 (v2 — garantia após possível re-aplicação de migration antiga). Ambas fazem DROP + CREATE com prazo_fatal DATE correto.

Evidência

QUERY
POST fn_get_atividades_kanban → sem erro
curl .../rpc/fn_get_atividades_kanban -d '{"p_organizacao_id":"55a0c7ba..."}' → [ { id, titulo, status, prazo_fatal: "2026-03-14", ... }, ... ] → length: 36 (sem erro 42804)
Critério atendido: RPC retorna 36 atividades sem qualquer erro. Campo prazo_fatal retorna string de data ("2026-03-14") compatível com tipo DATE. Kanban funcional.
FIX-2 KPIs admin mostrando toda a org (não só as próprias) ✅ PASSOU

Contexto

A RPC fn_atividades_kpis sempre filtrava por responsavel_id = p_user_id, fazendo admin/gestor ver apenas as próprias atividades (geralmente 0 ou poucas) em vez de toda a organização.

Lógica do fix

-- Migration 20260331000005 SELECT (perfil = 'admin' OR acesso_gestor = true) INTO v_is_admin FROM users WHERE id = p_user_id; WHERE a.deleted_at IS NULL AND (v_is_admin = true OR responsavel_id = p_user_id)

FIX-3 Feed de eventos — trigger trg_atividade_eventos criado ✅ PASSOU

O trigger trg_atividade_eventos estava ausente, fazendo a aba "Eventos" do drawer sempre aparecer vazia. A tabela eventos_operacionais nunca recebia registros automáticos de criação ou mudança de status de atividades.

Migração aplicada

FIX-4 ActivityCard mostra cliente_nome — empresa_nome ausente da RPC ⚠️ PARCIAL

O que foi corrigido

A função fn_get_atividades_kanban passou a incluir LEFT JOIN clientes c ON c.id = a.cliente_id retornando c.nome AS cliente_nome. O componente AtividadesActivityCard já renderiza o campo com fallback: cliente_nome ?? empresa_nome.

Resultado cliente_nome

-- 21 de 36 atividades têm cliente_nome preenchido { "titulo": "Verificar prazo recursal TJSC", "cliente_nome": "EDERSON RENAN DA SILVA", "responsavel_nome": "Heloísa Miglioranza" } { "titulo": "Elaborar petição inicial...", "cliente_nome": "EMERSON ALMEIDA MACHADO", "responsavel_nome": "Christian Paz" }
cliente_nome funciona. 21/36 atividades têm o campo preenchido (as demais são atividades sem cliente vinculado, que é válido).

Limitação: empresa_nome

-- fn_get_atividades_kanban RETURNS TABLE: id, titulo, status, urgente, importante, prioridade, prazo_fatal (DATE), tipo_nome, tipo_cor, cliente_nome, responsavel_nome, processo_numero, posicao, is_atrasada, is_unread -- empresa_nome NÃO está no RETURNS TABLE da RPC -- Todas as 36 atividades retornam empresa_nome como null implícito (campo não existe na RPC)
O tipo AtividadeListaRpc define empresa_nome e o componente já faz o fallback correto, mas a RPC não retorna esse campo. Cards com atividades vinculadas a empresas mostrarão célula vazia onde deveria aparecer o nome.

Impacto real: baixo — atividades de negação indevida são sempre vinculadas a clientes (pessoas físicas), não a empresas. O campo empresa_id não aparece preenchido nos dados atuais.

Critério parcialmente atendido

cliente_nome: ✅ funcionando, renderizado corretamente nos cards.
empresa_nome: ⚠️ ausente da RPC kanban — não é retornado. Impacto prático mínimo no contexto atual do Midas.
