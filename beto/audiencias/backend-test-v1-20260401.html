Achados — Parecer do Dev

⚠️ BT-001 (P1) — get_audiencias_semana: overload ambiguity via PostgREST
Existem 2 overloads: (p_organizacao_id uuid) e (p_organizacao_id uuid, p_data_inicio date, p_data_fim date). PostgREST retorna PGRST203 quando chamada com apenas p_organizacao_id porque não consegue desambiguar. Frontend funciona porque passa os 3 params, mas chamada via REST API direta falha.
-- FIX: Dropar o overload de 1 param (o frontend usa o de 3 params)
DROP FUNCTION IF EXISTS public.get_audiencias_semana(uuid);
-- Manter apenas:
-- get_audiencias_semana(p_organizacao_id uuid, p_data_inicio date, p_data_fim date)
⚠️ BT-002 (P1) — advogados_correspondentes: sem deleted_at (soft delete ausente)
Tabela não tem coluna deleted_at. Regra MdFlow: soft delete obrigatório. Se um correspondente for deletado com hard delete, audiências que referenciam ele ficam com correspondente_id apontando para registro inexistente (FK SET NULL resolve, mas perde histórico).
-- FIX:
ALTER TABLE advogados_correspondentes ADD COLUMN deleted_at TIMESTAMPTZ;
CREATE INDEX idx_correspondentes_org_active ON advogados_correspondentes(organizacao_id) WHERE deleted_at IS NULL;
-- Frontend: adicionar .is('deleted_at', null) nas queries
⚠️ BT-003 (P1) — audiencias_participantes: sem updated_at, sem deleted_at, sem trigger
Tabela faltando updated_at (coluna + trigger) e deleted_at. Viola padrão obrigatório MdFlow. Frontend faz DELETE+INSERT para atualizar participantes, mascarando o problema. Mas não há rastreabilidade de quando participantes foram alterados.
-- FIX:
ALTER TABLE audiencias_participantes
  ADD COLUMN updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  ADD COLUMN deleted_at TIMESTAMPTZ;

CREATE TRIGGER set_updated_at BEFORE UPDATE ON audiencias_participantes
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
💡 BT-004 (P2) — valor_correspondente e valor_audiencia: NUMERIC em vez de BIGINT (centavos)
Padrão MdFlow exige BIGINT armazenando centavos para valores monetários. As tabelas audiencias.valor_correspondente e advogados_correspondentes.valor_audiencia usam NUMERIC. Funciona mas é inconsistente com o resto do sistema. Frontend já usa formatBRLReais que espera centavos.
-- FIX (avaliar impacto — verificar se frontend e RPCs tratam como reais ou centavos):
-- Se reais: migrar dados multiplicando por 100, alterar tipo para BIGINT
-- Se centavos em NUMERIC: alterar tipo para BIGINT
-- ATENÇÃO: registrar_pagamento_audiencia insere valor direto — verificar
💡 BT-005 (P2) — Índice duplicado em audiencias (org_id)
Existem 2 índices idênticos: idx_audiencias_org e idx_audiencias_org_id — ambos em (organizacao_id). Desperdício de espaço e I/O desnecessário em INSERTs/UPDATEs.
-- FIX:
DROP INDEX IF EXISTS idx_audiencias_org;
-- Manter idx_audiencias_org_id
💡 BT-006 (P2) — get_audiencias_semana (3-param) marcada como VOLATILE, deveria ser STABLE
A função só faz SELECT (leitura), mas está marcada como VOLATILE. PostgreSQL não pode otimizar chamadas repetidas com mesmos params. A versão de 1 param está corretamente como STABLE.
-- FIX (após dropar o overload de 1 param):
CREATE OR REPLACE FUNCTION public.get_audiencias_semana(...)
  ... STABLE SECURITY DEFINER ...
💡 BT-007 (P2) — audiencias_ia_resultados.updated_at: NULLABLE sem default
updated_at é NULLABLE e sem default now(). Registros inseridos ficam com updated_at=NULL. O trigger existe mas só atua no UPDATE. Registros novos nunca terão updated_at preenchido na criação.
-- FIX:
ALTER TABLE audiencias_ia_resultados
  ALTER COLUMN updated_at SET NOT NULL,
  ALTER COLUMN updated_at SET DEFAULT now();
💡 BT-008 (P2) — reverter_pagamento_audiencia: não remove financeiro_refs
A RPC reverter_pagamento_audiencia reverte o flag pago=false na audiência e registra evento, mas NÃO remove ou soft-deleta o registro em financeiro_refs e financeiro. O lançamento financeiro de saída permanece ativo mesmo após reversão. Inconsistência entre registrar (que cria financeiro) e reverter (que não desfaz).
-- FIX sugerido (dentro da RPC):
-- Após UPDATE audiencias SET pago=false:
DELETE FROM financeiro_refs
WHERE organizacao_id = p_organizacao_id
  AND entidade_tipo = 'audiencia'
  AND entidade_id = p_audiencia_id;

-- E soft-delete do financeiro:
UPDATE financeiro SET deleted_at = now()
WHERE id = (SELECT financeiro_id FROM financeiro_refs
            WHERE entidade_tipo='audiencia' AND entidade_id=p_audiencia_id);
💡 BT-009 (P2) — WH audiencias/exportar: marcado INACTIVE no core mas funciona
O workflow de exportação não aparece na listagem de workflows com nome "audiencia" (provável nome diferente no n8n). Mas o webhook responde corretamente e gera CSV. O core HTML e docs/revisoes estão desatualizados marcando como INACTIVE/ghost route — mas ele funciona.
-- FIX: Atualizar documentação. O webhook está ATIVO e funcional.
💡 BT-010 (P2) — WH audiencias/chat retorna 404 (webhook não registrado)
O webhook chat retorna 404, não 200 com stub. Workflow foi desativado ou removido do n8n. Diferente de processar_ata que é STUB mas responde.

Diagnóstico Geral

O backend de audiências está bem implementado. RLS, SECURITY DEFINER, CHECK constraints e índices seguem o padrão MdFlow. As RPCs de pagamento são robustas (idempotência, validação de existência, eventos operacionais).

Os problemas são de consistência e completude, não de segurança: soft delete faltando em correspondentes e participantes, overload PostgREST, reversão de pagamento não desfaz o lançamento financeiro, e NUMERIC em vez de BIGINT para valores monetários. Nenhum é bloqueante para operação, mas são dívida técnica que cresce com o uso.

Surpresa positiva: o webhook de exportação funciona perfeitamente — a documentação que dizia "INACTIVE/ghost route" está errada. O webhook gera CSV e retorna signed URL para download.

P2 — BT-006: Marcar overload como STABLE

-- Após dropar o overload de 1 param, recriar o de 3 params como STABLE:
CREATE OR REPLACE FUNCTION public.get_audiencias_semana(
  p_organizacao_id uuid,
  p_data_inicio date DEFAULT NULL,
  p_data_fim date DEFAULT NULL
) RETURNS integer
LANGUAGE plpgsql
STABLE SECURITY DEFINER
SET search_path TO 'public'
AS $function$
DECLARE
  v_inicio TIMESTAMPTZ;
  v_fim    TIMESTAMPTZ;
  v_count  INTEGER;
BEGIN
  v_inicio := COALESCE(p_data_inicio::TIMESTAMPTZ,
    date_trunc('week', now() AT TIME ZONE 'America/Sao_Paulo'));
  v_fim := COALESCE(p_data_fim::TIMESTAMPTZ,
    date_trunc('week', now() AT TIME ZONE 'America/Sao_Paulo') + INTERVAL '7 days');

  SELECT COUNT(*) INTO v_count
  FROM audiencias
  WHERE organizacao_id = p_organizacao_id
    AND deleted_at IS NULL
    AND status = 'agendada'
    AND data_audiencia >= v_inicio
    AND data_audiencia < v_fim;

  RETURN COALESCE(v_count, 0);
END;
$function$;
