Bugs Encontrados

P0 staleTime = 5 * 60 * 1000 em todos os hooks (deveria ser 30_000)
Arquivos
useAudiencias.ts:70 — staleTime: 5 * 60 * 1000
useAudienciasKPIs.ts:81 — staleTime: 5 * 60 * 1000
useAudienciaCalendar.ts:46 — staleTime: 5 * 60 * 1000
useCorrespondentes.ts:77 — staleTime: 5 * 60 * 1000
useCorrespondenteSelect.ts:31 — staleTime: 5 * 60 * 1000
Causa Raiz
Todos os hooks usam 300.000ms (5 minutos) de staleTime. O CLAUDE.md do projeto exige staleTime: 30_000 em TODOS os hooks React Query. Violacao direta da regra P-08 do core.
Impacto
Dados ficam stale por 5 minutos. Se outro usuario cria/edita audiencia, o usuario atual nao ve a mudanca por ate 5 minutos, mesmo com realtime habilitado (realtime faz invalidateQueries, mas staleTime impede re-fetch se dados nao estao "stale").
Solucao
Trocar staleTime: 5 * 60 * 1000 por staleTime: 30_000 em todos os 5 hooks.
P1 reverter_pagamento_audiencia RPC executa sem validacao de existencia
Arquivo
RPC reverter_pagamento_audiencia (banco PostgreSQL)
Causa Raiz
Ao chamar a RPC com um audiencia_id inexistente (UUID zerado), retorna {"action":"reverted","message":"Pagamento revertido com sucesso","success":true} em vez de um erro. A RPC registrar_pagamento_audiencia corretamente retorna erro ("Audiencia nao encontrada"), mas a reverter nao valida.
Impacto
Silently succeeds on invalid data. Pode mascarar bugs no frontend onde um ID errado e enviado. Nao causa dano direto porque o frontend valida antes, mas e um hole de seguranca logica.
Solucao
Adicionar verificacao de existencia no inicio da RPC: IF NOT FOUND THEN RETURN jsonb_build_object('success', false, 'action', 'error', 'message', 'Audiencia nao encontrada');

index.tsx:46-69 — PageHeader actions

P2 Cegueira financeira parcialmente aplicada nos Correspondentes
Arquivo
CorrespondentesTab.tsx:57-59
Causa Raiz
No card do correspondente, quando acesso_financeiro=false, o valor exibe "---" (tres hifens) em vez do padrao "R$ •••••" usado em todo o sistema. Inconsistencia visual.
Impacto
Baixo — funcional, mas inconsistente com o padrao DS. O RN-10 diz que valor_correspondente nao tem cegueira financeira, porem o codigo aplica parcialmente.
Solucao
Alinhar com RN-10: correspondentes nao devem ter cegueira financeira, ou se aplicar, usar o padrao "R$ •••••".
P2 audiencias_participantes usa cast "as never" (TypeScript unsafe)
Arquivo
ModalAudiencia.tsx:113,135,145
Causa Raiz
A tabela audiencias_participantes nao esta tipada no schema Supabase gerado. O codigo usa supabase.from('audiencias_participantes' as never) para bypassar TypeScript. Isso indica que a tabela existe no banco mas nao foi gerada no types.
Impacto
Zero type safety nas operacoes de participantes. Se a tabela mudar schema, o TypeScript nao detecta.
Solucao
Regenerar types do Supabase: supabase gen types typescript e atualizar o arquivo de tipos.
P2 useAudienciaIA usa useState+useEffect em vez de useQuery (sem staleTime, sem retry)
Arquivo
useAudienciaIA.ts:15-77
Causa Raiz
O hook de IA usa useState+useEffect manual em vez de useQuery. Isso significa: sem cache, sem retry, sem staleTime, sem dedup de requests, sem suspense support. Todos os outros hooks da pagina usam useQuery corretamente.
Impacto
Baixo (feature pouco usada). Mas se o modal for aberto/fechado rapidamente, pode causar requests duplicados.
Solucao
Refatorar para useQuery com queryKey: ['audiencia-ia', audienciaId]. Manter o realtime subscription separado.
