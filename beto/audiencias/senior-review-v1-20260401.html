Achados — P0 Bloqueantes (2)

SR-001 P0 FRONTEND
KPI invalidation BROKEN — queryKey mismatch impede refresh
useAudienciasKPIs.ts:26 define queryKey ['audiencias', 'kpis', organizacao_id] mas TODOS os 5 locais que invalidam usam ['audiencias-kpis'] (com hifen). Keys diferentes = invalidacao nunca acontece.
Impacto: Apos criar, editar, pagar ou reverter audiencia, os 4 KPIs continuam exibindo dados antigos. Usuario ve info errada ate reload manual.
Afeta: ModalConfirmacaoPagamento:48 | ModalReverterPagamento:50 | AudienciasTabContent:100 | useAudienciaForm:188,217
FIX: Mudar queryKey em useAudienciasKPIs para ['audiencias-kpis', organizacao_id] (alinhar com os chamadores)
SR-002 P0 FRONTEND
CSV import envia apenas 5 linhas de preview, nao o arquivo completo
ModalImportAudiencias.tsx:55 faz setRows(parsed.slice(0, 5)) guardando apenas 5 linhas. Linha 70 envia registros: rows ao webhook — ou seja, 5 linhas.
Impacto: Importacao CSV de 100+ audiencias silenciosamente importa apenas 5. Perda de dados sem aviso.
Evidencia: Codigo fonte confirmado.
FIX: Separar allRows (completo) de previewRows (slice 0,5). Enviar allRows ao webhook.

Achados — P1 Importantes (7)

SR-003 P1 FRONTEND
Correspondentes queries sem filtro deleted_at — soft-deleted aparecem
3 hooks consultam advogados_correspondentes sem .is('deleted_at', null):
useCorrespondentes.ts:51 | useCorrespondenteSelect.ts:22 | useCorrespondenteForm.ts:61
Impacto: Correspondentes excluidos via soft delete continuam aparecendo na lista e no dropdown de selecao.
FIX: Adicionar .is('deleted_at', null) nas 3 queries.
SR-004 P1 FRONTEND
Participantes SELECT e DELETE sem filtro organizacao_id (defense-in-depth)
ModalAudiencia.tsx:112-116 (SELECT) e :135-138 (DELETE) usam apenas audiencia_id sem organizacao_id.
RLS protege via policy, mas regra do projeto exige organizacao_id em toda query como defense-in-depth.
Risco: Se RLS for desabilitado por erro ou via service_role, dados de outra org podem vazar.
FIX: Adicionar .eq('organizacao_id', organizacao_id) em ambas queries.
SR-005 P1 FRONTEND
CSV parser naive — nao trata campos entre aspas (RFC 4180)
ModalImportAudiencias.tsx:29 usa line.split(',') sem respeitar delimitadores entre aspas.
Impacto: CSV exportado do Excel/Google Sheets com campos como "Sao Paulo, SP" corrompera dados silenciosamente.
FIX: Usar regex RFC 4180 ou lib como papaparse para parsing CSV.
SR-006 P1 FRONTEND
window.location.reload() para recuperar de erro — destroi estado React
AudienciasTabContent.tsx:167 — botao de retry no error state faz hard reload.
Impacto: Perde filtros, posicao de scroll, estado de modais. Anti-pattern React.
FIX: Substituir por queryClient.invalidateQueries({ queryKey: ['audiencias'] }).
SR-007 P1 FRONTEND
DRY: parseBRL e toBRLInput duplicados em 2 hooks
useAudienciaForm.ts:10-19 e useCorrespondenteForm.ts:8-18 — mesmas funcoes copiadas.
Violacao: P-09 (DRY) do CLAUDE.md.
FIX: Extrair para src/lib/formatters.ts e importar em ambos hooks.
SR-008 P1 FRONTEND
Missing aria-label em 2 icon buttons (acessibilidade)
UploadAta.tsx:122 — botao X (remover arquivo) sem aria-label.
AudienciaCard.tsx:176 — botao MoreVertical (acoes dropdown) sem aria-label.
Impacto: Screen readers nao conseguem identificar a acao do botao.
FIX: Adicionar aria-label="Remover arquivo" e aria-label="Acoes da audiencia".
SR-009 P1 N8N
webhook_requests nao registra contexto para audiencias
Query em webhook_requests WHERE contexto ILIKE '%audiencia%' retorna zero registros. O workflow N8N-AUDIENCIAS-FINANCEIRO roda (2 execucoes success hoje) mas nao grava na tabela de tracking.
Impacto: Sem auditoria de webhooks de audiencias. Impossivel debugar problemas pos-facto.
FIX: Adicionar no de registro em webhook_requests no workflow n8n (request_id, contexto, status, payload).
