🚨 Achados P0 — Bloqueante

P0 BT-001 — fn_calculo_salvar_completo perde juros_data_inicial/final

O INSERT para calculo_periodos dentro da RPC não inclui as colunas juros_data_inicial e juros_data_final que foram adicionadas na migration v2.

Impacto: Ao salvar um cálculo com datas de juros independentes, os valores são silenciosamente descartados. Ao recarregar, as datas voltam como NULL (fallback para datas da correção).

Frontend afetado: useCalculoSave.ts:87 envia state.periodos que inclui juros_data_inicial e juros_data_final, mas a RPC ignora.

-- Adicionar juros_data_inicial e juros_data_final ao INSERT de calculo_periodos
-- dentro de fn_calculo_salvar_completo:
CREATE OR REPLACE FUNCTION fn_calculo_salvar_completo(...)
-- No INSERT INTO calculo_periodos, adicionar:
--   juros_data_inicial, juros_data_final
-- No VALUES, adicionar:
--   (v_periodo->>'juros_data_inicial')::date,
--   (v_periodo->>'juros_data_final')::date
P0 BT-002 — fn_calculo_buscar_completo retorna estrutura aninhada vs frontend espera flat

A RPC retorna jsonb_build_object('calculo', v_calculo, 'parcelas', [...], 'periodos', [...]) — ou seja, uma estrutura aninhada com calculo como sub-objeto.

O frontend em index.tsx:63-79 faz d.id as string, d.parcelas as ... — espera tudo no nível raiz.

Impacto: Deep link /calculo/:id carrega mas todos os campos ficam undefined ou []. O cálculo aparece vazio.

-- Opção A (recomendada): Achatar retorno na RPC
RETURN v_calculo || jsonb_build_object(
  'parcelas',   (SELECT jsonb_agg(...) FROM calculo_parcelas ...),
  'periodos',   (SELECT jsonb_agg(...) FROM calculo_periodos ...),
  'eventos',    (SELECT jsonb_agg(...) FROM calculo_eventos ...),
  'adicionais', (SELECT jsonb_agg(...) FROM calculo_adicionais ...),
  'ultimo_snapshot', (SELECT row_to_json(...) FROM calculo_snapshots ...)
);

-- Opção B: Adaptar frontend para ler d.calculo.id, d.parcelas, etc.
-- (menos ideal — RPC deveria ser a fonte de verdade)
P0 BT-003 — useIndiceSerie.ts:20 envia p_data_final mas RPC espera p_data_fim

A função fetchSerie() chama a RPC com parâmetro p_data_final, mas fn_indice_buscar_serie espera p_data_fim.

Testado: Com p_data_final (errado): 4 resultados. Com p_data_fim (correto): 6 resultados. PostgREST não rejeita parâmetros desconhecidos — p_data_fim recebe NULL, causando retorno incompleto.

Impacto: Séries de índices carregam com menos dados do que deveriam. Cálculos de correção monetária podem usar fator incompleto.

-- Fix no FRONTEND (src/hooks/calculos/useIndiceSerie.ts:17-21):
const { data, error } = await supabase.rpc('fn_indice_buscar_serie', {
    p_indice: indiceDb,
    p_data_inicio: dataInicial,
-   p_data_final: dataFinal,   // ❌ parâmetro não existe na RPC
+   p_data_fim: dataFinal,     // ✅ nome correto
  });

⚠️ Achados P1 — Importante

P1 BT-004 — calculo_consistencia sem trigger set_updated_at

Tabela de consistência não tem trigger de updated_at. Menor impacto pois registros são write-once, mas viola padrão obrigatório.

P1 BT-005 — indice_valores sem índice em (indice, referencia_data)

A RPC fn_indice_buscar_serie filtra por indice = p_indice AND referencia_data BETWEEN ... mas não há índice composto. Com 1615 registros o impacto é mínimo, mas crescerá com novos dados mensais (12 índices × 12 meses/ano = 144 registros/ano).

CREATE INDEX idx_indice_valores_indice_data
  ON indice_valores (indice, referencia_data);

💬 Parecer do Dev

O que está BOM:

  • Schema bem estruturado: UUID PKs, TIMESTAMPTZ, TEXT CHECK (nunca ENUM), BIGINT/NUMERIC para valores monetários
  • RLS completo em todas as tabelas. Isolamento multi-tenant testado e funcional
  • Todas RPCs com SECURITY DEFINER + SET search_path = public
  • Dados BCB SGS populados (1615 registros, 12 índices, jan/2015 — mar/2026)
  • Adapter pattern INDICE_PARA_SERIE para traduzir chaves motor↔banco é elegante
  • Snapshot auditável com versionamento — bom design para cálculos judiciais

O que está ERRADO (3 bugs P0):

  • BT-001: RPC de save perde dados de juros independentes — feature implementada no frontend e banco mas "desconectada" na RPC
  • BT-002: Deep link quebrado — RPC retorna nested, frontend espera flat. Qualquer cálculo salvo volta vazio ao recarregar
  • BT-003: Parâmetro errado na busca de séries — cálculos usam dados incompletos de índices

Diagnóstico:

Os 3 bugs são do tipo "última milha" — cada peça (banco, frontend, motor) funciona isoladamente, mas a cola entre elas (RPCs e chamadas RPC) tem erros de interface. Isso é típico de desenvolvimento que evolui rápido (v2 adicionou juros independentes, 12 índices) sem atualizar as RPCs existentes.

Prioridade de fix:

  1. BT-003 (1 min) — trocar p_data_final → p_data_fim no frontend
  2. BT-001 (5 min) — adicionar 2 colunas no INSERT da RPC
  3. BT-002 (10 min) — achatar retorno da RPC com v_calculo || jsonb_build_object(...)
