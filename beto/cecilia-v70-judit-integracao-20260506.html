O Problema que Resolvemos

Antes — Worker A com Playwright EPROC

  • 68.2% das intimacoes classificadas como movimento_em_duvida
  • Apenas 31.8% de intimacoes uteis detectadas
  • 0% de deteccao de polo correto em 68.2% dos casos
  • Navegacao via browser automation — lenta e fragil
  • Sujeito a CAPTCHA, mudancas de layout, sessao expirada
  • Dados recebidos sem contexto semantico ou estrutura

Depois — API Judit com push notification

  • Worker A (Playwright) substituido completamente pela Judit
  • Judit monitora os processos e envia dados via webhook
  • Cecilia recebe texto semantico, estruturado e rico
  • Juiz, fase processual, polo, valor — entregues diretamente
  • Sem polling, sem timeout, sem scraping — push real-time
  • Deduplicacao via judit_last_step_id — zero duplicatas
Causa raiz resolvida

A Cecilia navegava o EPROC via Playwright — um sistema lento, fragil e sem contexto semantico. A integracao com a API Judit resolve isso na raiz: agora recebemos push notifications com dados estruturados quando ha qualquer movimentacao nos processos monitorados. A Cecilia nunca mais precisa abrir um browser para descobrir o que aconteceu no processo.

Capacidades da API Judit

✅
O que antes exigia navegacao manual no EPROC via browser, agora chega automaticamente via push notification com dados estruturados, semanticos e deduplificados. A Cecilia nunca mais precisa fazer scraping.
📄 Dados do Evento (last_step)
last_step.content
Texto completo do ato processual
Sentenca, despacho, decisao, certidao — texto ja extraido pelo Judit, pronto para analise da Cecilia. Este e o campo mais importante.
last_step.step_type
Codigo EPROC do tipo de ato
Ex: "92" = Sentenca, "187" = Despacho, "226" = Certidao de Publicacao. Joao precisa criar o mapa completo (T-JU-01).
last_step.step_date
Data exata do ato processual
Data em que o ato foi registrado no EPROC. Usado para calcular prazos processuais.
last_step.step_id
ID unico do passo (deduplicacao)
Armazenado em judit_last_step_id com UNIQUE index — garante que o mesmo evento nunca seja processado duas vezes.
⚖️ Dados do Processo
judge
Nome completo do juiz responsavel
Permite cruzar com historico do juiz (favoravel/neutro/complicador) nos dados da Cecilia.
phase
Fase processual estruturada
INICIAL · RECURSAL · EXECUCAO — muda completamente a urgencia e o tipo de acao recomendada.
amount
Valor da causa em reais
Usado para priorizacao — processos com valor alto tem urgencia maior na analise.
status
ATIVO ou FINALIZADO
Quando FINALIZADO — sistema auto-encerra o processo no MdFlow. Evita analise de processos ja extintos.
tribunal_acronym
Tribunal de origem
TJRS · TJSC · TJSP. Permite aplicar regras diferentes por tribunal nos playbooks.
👥 Dados das Partes — Polo Automatico
parties[].side
Active (autor) ou Passive (reu)
Identifica qual polo e cada parte do processo.
parties[].lawyers[].oab
OAB dos advogados de cada parte
Cecilia cruza com nossas OABs cadastradas (RS116571, RS131163, SC074025, SC075466, SP535101) para detectar automaticamente qual polo somos — autor ou reu.
evento_polo resultante
autor · reu · desconhecido
Campo gerado pela Cecilia apos cruzar os dados. Meta: desconhecido menor que 20% (estava 68.2%).
📎 Attachments (em implementacao)
attachments[]
Lista de PDFs vinculados ao ato
PDF da sentenca, do despacho, da certidao. A Judit indica a existencia do PDF e disponibiliza para download quando solicitado.
🔮
Decisao D6=A aprovada: download automatico de PDFs em implementacao. Quando disponivel, o Worker C tera acesso ao conteudo completo do PDF, nao apenas ao last_step.content.

PRIORITARIO — Antes de continuar
T-JU-01 Mapa de step_types EPROC Alta prioridade
O que e A Judit entrega step_type como codigo numerico (ex: "92", "187", "226"). A Cecilia precisa saber que step_type 92 = Sentenca, 187 = Despacho, 226 = Certidao de Publicacao, etc.
Por que precisa Sem o mapa, a Cecilia nao consegue distinguir Sentenca de Despacho de Certidao — tudo vira movimento_em_duvida. E a causa raiz do problema de classificacao.
Formato sugerido Tabela com os 20-30 step_types mais comuns do TJRS/TJSC/TJSP: codigo, nome semantico, relevancia (ruido/relevante/urgente)
Como levantar Verificar nos primeiros callbacks recebidos quais step_types aparecem. Christian pode exportar os dados de cecilia_movimentacoes_brutas onde fonte='judit'.
T-JU-02 Tratamento de Content Vazio Alta prioridade
O que e Casos em que last_step.content esta vazio ou muito curto (menos de 50 chars). A Judit as vezes entrega eventos sem texto narrativo (ex: "Conclusao ao juiz").
Por que precisa Sem instrucao clara, a Cecilia pode tratar content vazio como movimento_em_duvida ou descartar incorretamente um evento importante.
Formato sugerido 5-10 exemplos reais de content vazio + decisao para cada: ruido (ignorar) ou movimento_em_duvida (sinalizar para humano com motivo "content indisponivel").
Regra base sugerida Se content vazio E ha attachment — sinalizar "PDF disponivel, aguardando download". Se content vazio E sem attachment — verificar step_type para decidir.
T-JU-03 Polo Desconhecido — Identificacao Manual Alta prioridade
O que e Casos em que parties[] nao tem nossas OABs cadastradas — processos antigos, partes sem advogado registrado na Judit, cadastros incompletos.
Por que precisa 68.2% dos casos tinham polo desconhecido. Parte deles pode ser inferida pelo texto da intimacao (ex: "autor condenado" — somos reu).
Formato sugerido 10 exemplos com texto real de intimacao onde o polo pode ser inferido do conteudo + a regra linguistica para identificacao.
Resultado esperado Reduzir evento_polo = 'desconhecido' de 68.2% para menos de 20%.
IMPORTANTE — Proximas semanas
T-JU-04 Processos Finalizados — Reconhecimento Semantico Medio prazo
O que e Quando status=FINALIZADO, o sistema auto-encerra. Mas as vezes a sentenca finalizadora chega antes de o status mudar — a Cecilia precisa reconhecer pelo texto.
Formato sugerido 15-20 exemplos de sentencas finalizadoras reais (transito em julgado, arquivamento, extincao) com classificacao correta e os termos indicadores.
T-JU-05 Fase Processual — RECURSAL vs INICIAL Medio prazo
O que e A Judit entrega phase (INICIAL, RECURSAL, EXECUCAO). A Cecilia usa isso no prompt mas nao tem playbook especifico para mudar o comportamento por fase.
Formato sugerido Matriz: fase x tipo_intimacao x acao esperada x prazo tipico. Ex: fase RECURSAL + Acordao — urgencia maxima, prazo embargos 15 dias.
T-JU-06 Attachments PDF — Quando Ficar de Olho Medio prazo
O que e A Judit indica quando ha PDF vinculado via attachments[]. Por enquanto nao baixamos automaticamente, mas e importante sinalizar quando ha PDF disponivel.
Formato sugerido 5 exemplos de eventos com attachments + comportamento esperado: quando content esta vazio mas ha attachment — nao descartar, sinalizar "PDF disponivel, aguardando download".
T-JU-07 Deduplicacao — step_id vs Novos Eventos Medio prazo
O que e A Judit usa judit_last_step_id para garantir que nao processamos o mesmo evento duas vezes. A Cecilia pode receber o mesmo callback duas vezes (falha de rede, retry).
Formato sugerido Explicacao tecnica do mecanismo + 3 exemplos de tentativa de duplicata + log do que acontece (Edge Function retorna 200 sem processar novamente).

Query 2 — Polo desconhecido (meta: desconhecido menos que 20%)
SELECT evento_polo, COUNT(*) as total, ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 1) as pct FROM cecilia_movimentacoes_brutas WHERE fonte = 'judit' AND created_at > NOW() - INTERVAL '7 days' AND deleted_at IS NULL GROUP BY evento_polo ORDER BY total DESC;
Query 3 — Taxa de sucesso dos jobs (meta: failed menos que 5%)
SELECT status, COUNT(*) as total FROM cecilia_jobs WHERE created_at > NOW() - INTERVAL '7 days' AND deleted_at IS NULL GROUP BY status;
