🔴 Bugs Críticos

BUG-1 CRÍTICO `.select()` chained ilegalmente em chamada RPC — advogado_responsavel nunca populado
Arquivo
src/hooks/useCliente.ts — linhas 13–16
const { data, error } = await supabase
  .rpc('get_cliente_by_cpf', { p_cpf: cpfSlug! })
  .select('*, advogado_responsavel:advogados!advogado_responsavel_id(id, nome, oab, oab_uf)')
  .maybeSingle();
Causa Raiz

O Supabase JS SDK não suporta encadeamento de .select() com sintaxe de join PostgREST em chamadas RPC. A sintaxe advogados!advogado_responsavel_id só é válida para queries em tabelas via PostgREST. Em chamadas RPC, o retorno é JSONB — os joins devem ser feitos dentro do SQL da RPC. O .select() encadeado é silenciosamente ignorado ou retorna resultado parcial.

Impacto

No contexto context=clientes, o drawer sempre exibe "Sem responsável" ou cai no fallback do campo advogado_responsavel_nome da query paginada. O formulário de edição na aba Dados também não pré-popula os dados do advogado.

BUG-3 CRÍTICO Restrições de clientes via lead_id invisíveis — migration pendente em produção
Arquivo
supabase/migrations/20260330000001_fix_fn_restricoes_listar_cliente_via_lead.sql
-- Cláusula adicionada na migration local (NÃO aplicada em prod):
OR (p_cliente_id IS NOT NULL AND r.lead_id IN (
  SELECT id FROM leads WHERE cliente_id = p_cliente_id AND deleted_at IS NULL
))
Causa Raiz

As restrições salvas durante o processo de enriquecimento AdvBox foram persistidas com lead_id (e não cliente_id). A função fn_restricoes_listar em produção não possui a cláusula de lookup "via lead". A migration 20260330000001 corrige isso mas ainda não foi aplicada ao banco de produção.

Impacto

A aba Restrições no drawer aparece vazia para a maioria dos clientes migrados do AdvBox, mesmo que existam restrições cadastradas. Este é provavelmente o bug de maior impacto funcional da página.

BUG-5 CRÍTICO Filtro CNPJ client-side após paginação server-side — total e páginas descompassados
Arquivo
src/hooks/useClientes.ts — linhas 27–33
const result = data as unknown as ClientesPaginados;
if (result?.data) {
  result.data = result.data.filter((c: any) => {
    const digits = (c.cpf_cnpj ?? '').replace(/\D/g, '');
    return digits.length !== 14;
  });
}
return result;
Causa Raiz

A RPC get_clientes_paginados retorna total e total_pages calculados server-side incluindo CNPJs. O filtro client-side remove alguns registros da página mas não recalcula o total. Por exemplo, se 5 dos 50 registros da página 1 são CNPJs, a UI exibe "Mostrando 1–45 de 2000 clientes" mas pagina como se ainda houvesse 2000/50 = 40 páginas, podendo exibir páginas vazias.

Solução Recomendada

Passar parâmetro p_excluir_cnpj BOOLEAN DEFAULT true para a RPC e filtrar server-side antes do COUNT.

Observação: Realtime channel name estático

O canal Realtime está nomeado 'clientes-realtime' (nome estático). Se duas abas do browser abrirem /clientes para a mesma organização, ambas tentam se inscrever no mesmo canal. Comportamento inócuo para Postgres Changes subscriptions mas pode gerar aviso no Supabase client.

Observação: queryKey com objeto filters

O objeto filters é passado diretamente como elemento do queryKey. O React Query v5 usa comparação profunda, então funciona corretamente — mas é um padrão que pode causar bugs sutis se o objeto for mutável.
