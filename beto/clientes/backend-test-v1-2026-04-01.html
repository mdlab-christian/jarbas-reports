Achados

❌ BT-P0-001 — get_cliente_by_cpf VAZA DADOS entre organizações

Gravidade: P0 — Bloqueante multi-tenant

Evidência: Chamada com SET request.jwt.claims = '{"organizacao_id": "ffffffff-..."}' retornou cliente real da org Midas (nome: ABRELINO OLIVEIRA, org: 55a0c7ba...). A RPC não filtra por organizacao_id — retorna o primeiro match global por CPF.

Impacto: Qualquer usuário autenticado de outra org pode consultar dados completos de clientes Midas via CPF (to_jsonb retorna TODAS as colunas incluindo endereço, telefone, email, observações).

Fix:

-- Adicionar filtro por org:
CREATE OR REPLACE FUNCTION public.get_cliente_by_cpf(p_cpf text)
 RETURNS jsonb LANGUAGE plpgsql STABLE SECURITY DEFINER
 SET search_path TO 'public'
AS $$
DECLARE v_result jsonb;
BEGIN
  SELECT to_jsonb(c) || jsonb_build_object(
    'advogado_responsavel', CASE WHEN adv.id IS NOT NULL THEN
      jsonb_build_object('id', adv.id, 'nome', adv.nome, 'oab', adv.oab, 'oab_uf', adv.oab_uf)
    ELSE NULL END
  ) INTO v_result
  FROM clientes c
  LEFT JOIN advogados adv ON adv.id = c.advogado_responsavel_id AND c.deleted_at IS NULL
  WHERE regexp_replace(c.cpf_cnpj, '[^0-9]', '', 'g') = regexp_replace(p_cpf, '[^0-9]', '', 'g')
    AND c.organizacao_id = public.get_org_id()  -- ← ADICIONAR ESTA LINHA
    AND c.deleted_at IS NULL
  LIMIT 1;
  RETURN v_result;
END; $$;

⚠️ BT-P1-001 — update_cliente_dados usa COALESCE — impossível limpar campos

Gravidade: P1 — Bug funcional

Todos os campos no UPDATE usam COALESCE(p_dados->>'campo', campo_atual). Se o usuário envia {"email": null} ou omite o campo, o valor existente é preservado. Não é possível limpar um email, telefone, ou observação uma vez preenchido.

Fix: Usar padrão "key exists" em vez de COALESCE:

email = CASE WHEN p_dados ? 'email' THEN (p_dados->>'email') ELSE email END

⚠️ BT-P1-002 — update_cliente_dados não atualiza campos bancários/extras

Gravidade: P1 — Schema drift

O array v_allowed NÃO inclui: naturalidade, banco, pix, renda_mensal. O drawer do frontend exibe e permite editar esses campos, mas a RPC os ignora silenciosamente.

Fix: Adicionar ao v_allowed + ao bloco SET:

v_allowed TEXT[] := ARRAY[... 'naturalidade','banco','pix','renda_mensal' ...];

⚠️ BT-P1-003 — Webhooks CLIENTES-CRUD retornam envelope incompleto

Gravidade: P1 — Contrato quebrado

Os webhooks clientes/criar e clientes/excluir retornaram {"message": "Workflow execution failed", "success": null, "action": null}. O envelope canônico MdFlow exige success (boolean), action (string), message (string). Mesmo em erro, success deve ser false, não null.

Além disso, nenhum registro foi criado em webhook_requests — rastreabilidade zero.

Fix: Revisar workflow n8n ID 6Z_tuZgOH5JuZr8gGcVyX — garantir nó de error handler com envelope canônico e INSERT em webhook_requests.

💡 BT-P2-001 — Campos monetários usam NUMERIC em vez de BIGINT (centavos)

Gravidade: P2 — Tech debt / inconsistência

valor_total_estimado_recuperavel e renda_mensal são NUMERIC. Convenção MdFlow pede BIGINT em centavos. NUMERIC funciona, mas diverge do padrão usado em financeiro, acordos, execucoes.

restricoes.valor também é NUMERIC — mesmo caso.

Risco: Baixo, NUMERIC é seguro para money. Mas inconsistência pode gerar confusão (é centavos ou reais?).

💡 BT-P2-002 — get_clientes_paginados: correlated subquery para processos_count

Gravidade: P2 — Performance

A RPC faz COALESCE((SELECT COUNT(*) FROM processos p WHERE p.cliente_id = c.id), 0) como correlated subquery para CADA row. Com 4.000+ clientes, isso gera ~4.000 subqueries por página (50 rows × 1 subquery, mas o COUNT total também itera).

Fix sugerido: Substituir por LEFT JOIN com subquery agrupada ou materializar o count em campo denormalizado com trigger.

💡 BT-P2-003 — Campos JSONB legados: documentos, restricoes, historico

Gravidade: P2 — Tech debt

A tabela clientes ainda carrega 3 campos JSONB legados: documentos (default []), restricoes (default []), historico (default {}). Esses dados já foram migrados para tabelas dedicadas (restricoes, clientes_historico, documentos_arquivos). Os campos JSONB são redundantes e podem confundir futuros devs.

Fix: Quando estabilizar, considerar remover esses campos (requer aprovação CTO para alterar tabela core).

Parecer Técnico

🔍 Diagnóstico Geral

O backend de /clientes é sólido na estrutura — schema bem desenhado, 20 índices, RLS granular, CHECK constraints corretos, triggers de auditoria. É um dos backends mais maduros do MdFlow.

O problema grave é único mas crítico: get_cliente_by_cpf vaza dados entre orgs. É o tipo de bug que um pentest encontra em 5 minutos. Fix é trivial (1 linha de SQL) mas o impacto é P0 porque viola multi-tenancy, que é princípio inviolável #4.

A issue com update_cliente_dados (COALESCE impede limpar campos) é funcional — usuário não consegue apagar email/telefone/observação. Os campos bancários faltantes no update são schema drift silencioso.

Webhooks n8n estão instáveis — ambos falharam na execução. Como o frontend usa callWebhook para criar e excluir clientes, se o workflow está com problema, a criação de clientes via modal pode estar falhando em produção sem log.

CREATE OR REPLACE FUNCTION public.get_cliente_by_cpf(p_cpf text)
 RETURNS jsonb LANGUAGE plpgsql STABLE SECURITY DEFINER

AS $$
DECLARE v_result jsonb;
BEGIN
  SELECT to_jsonb(c) || jsonb_build_object(
    'advogado_responsavel', CASE WHEN adv.id IS NOT NULL THEN
      jsonb_build_object('id', adv.id, 'nome', adv.nome, 'oab', adv.oab, 'oab_uf', adv.oab_uf)
