Achados

BT-001 P0 Webhook excluir roteado para nó Criar Cliente (wiring invertido no n8n)
No workflow [GUSTAVO] CLIENTES-CRUD (id 6Z_tuZgOH5JuZr8gGcVyX), o nó Switch Acao tem outputs invertidos. Payload com acao = 'excluir' é roteado para o nó Criar Cliente e retorna INSERT_FALHOU: null value in column "nome". Resultado: deleção de clientes completamente quebrada em produção. Confirmado via execução real (#202639).

Fix: Corrigir wiring no n8n editor — swapear conexões output 0/1 do Switch Acao, ou verificar condições do switch.
BT-002 P0 65 CPFs duplicados em produção — unique index não normaliza formato
Campo cpf_cnpj armazena formatos mistos: 3.285 com máscara (046.961.519-21) e 204 sem (04696151921). O unique index idx_clientes_cpf_org_unique é UNIQUE(organizacao_id, cpf_cnpj) sem normalização. 65 CPFs já existem duplicados com formatos diferentes para o mesmo cliente na mesma org.

Fix: Trigger BEFORE INSERT/UPDATE para normalizar cpf_cnpj para dígitos-only + recriar unique como expression index UNIQUE(organizacao_id, regexp_replace(cpf_cnpj, '\D', '', 'g')) + data cleanup dos 65 duplicados.
BT-003 P0 n8n Criar Cliente não persiste captador_id (silent drop)
O frontend envia captador_id no payload de criação (campo no formulário NovoClienteModal). O nó Criar Cliente no n8n monta insertData sem incluir captador_id. Campo silenciosamente descartado — clientes criados via modal sempre têm captador_id = NULL.

Fix: Adicionar if (data.captador_id) insertData.captador_id = data.captador_id no nó Criar Cliente do n8n.
BT-004 P1 KPI total_clientes diverge da lista por 343 CNPJs
get_clientes_kpis retorna total_clientes: 4.350 (todos sem deleted_at), mas get_clientes_paginados com p_excluir_cnpj: true (default) lista 4.007. Diferença: 343 CNPJs. KPI no topo mostra número que não corresponde ao que o usuário vê na tabela.

Fix: Adicionar filtro p_excluir_cnpj no get_clientes_kpis ou usar contagem consistente.
BT-005 P1 Busca por nome usa Seq Scan — índice GIN não aproveitado
Índice é GIN(nome gin_trgm_ops) sobre campo cru. RPC usa unaccent(lower(nome)) ILIKE '%term%' — expressão diferente. PostgreSQL faz Seq Scan em toda a tabela a cada busca. Degradação com crescimento da base.

Fix: Criar CREATE INDEX idx_clientes_nome_search ON clientes USING GIN (unaccent(lower(nome)) gin_trgm_ops).
BT-006 P1 LEFT JOIN advogados com condição errada
Em get_clientes_paginados: LEFT JOIN advogados adv ON adv.id = c.advogado_responsavel_id AND c.deleted_at IS NULL. A condição c.deleted_at IS NULL está no JOIN em vez do WHERE (já existe no WHERE). Bug latente — funciona acidentalmente, mas semanticamente incorreto.

Fix: Remover AND c.deleted_at IS NULL do JOIN condition.
BT-007 P1 Toast duplicado na criação de cliente
n8n retorna action: 'close_modal'. O handleWebhookAction mostra toast da message. Mas o NovoClienteModal já dispara toast.success('Cliente criado com sucesso') no callback de sucesso. Resultado: 2 toasts aparecem simultaneamente.

Fix: Remover um dos dois toasts (preferencialmente o do modal, mantendo o do action handler).
BT-008 P2 Exportação CSV busca sem unaccent — inconsistente com lista
Export usa .or('nome.ilike.%search%') sem normalização de acentos. Lista usa unaccent(lower(nome)) na RPC. Buscar "João" na lista e exportar com mesmo filtro pode retornar conjuntos diferentes.
BT-009 P2 update_cliente_dados não aceita cpf_cnpj nem origem_id
Whitelist da RPC inclui origem (text) mas não origem_id (uuid FK) nem cpf_cnpj. Clientes com CPF errado não podem ser corrigidos via RPC de edição.
