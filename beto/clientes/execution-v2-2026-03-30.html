EXEC-V2-01 P1 FRONTEND BUG-9: Erro CLIENTE_NOT_FOUND sem toast — falha silenciosa no drawer
Contexto

Quando get_cliente_by_cpf não encontra o cliente (slug inválido, deletado), o hook lança Error('CLIENTE_NOT_FOUND'), o drawer exibe "Cliente não encontrado", mas nenhum toast é disparado. Inconsistente com o padrão da plataforma.

Arquivo
~/beta-mdflow/src/hooks/useCliente.ts — linha ~18
~/beta-mdflow/src/components/shared/cliente_drawer/ClienteDrawer.tsx — onde isError é tratado
Dependência
⚡ Independente
Prompt para Claude Code
Projeto: ~/beta-mdflow/

Tarefa: exibir toast de erro quando cliente não é encontrado no drawer.

CONTEXTO:
Em ~/beta-mdflow/src/hooks/useCliente.ts, quando get_cliente_by_cpf retorna null:
  if (!data) throw new Error('CLIENTE_NOT_FOUND');

O drawer recebe isError=true e exibe "Cliente não encontrado", mas nenhum toast é disparado.
Padrão da plataforma: erros de carregamento devem mostrar toast.error().

SOLUÇÃO:

Opção A (preferida) — tratar no ClienteDrawer.tsx:
1. Ler ~/beta-mdflow/src/components/shared/cliente_drawer/ClienteDrawer.tsx
2. Localizar onde isError do useCliente é tratado (exibição do estado de erro)
3. Adicionar useEffect que dispara toast quando isError muda para true:
   useEffect(() => {
     if (isError && error?.message === 'CLIENTE_NOT_FOUND') {
       toast.error('Cliente não encontrado')
     }
   }, [isError, error])
4. Garantir que toast (sonner) está importado no arquivo

Opção B — tratar no hook (se o drawer não tiver acesso fácil):
Em useCliente.ts, na queryFn, disparar toast antes de throw:
  if (!data) {
    toast.error('Cliente não encontrado')
    throw new Error('CLIENTE_NOT_FOUND')
  }
(Importar toast de 'sonner')

Usar a opção que causar menos impacto no código existente.

Critério de aceite: navegar para /clientes/cpf-invalido (slug inexistente) → toast.error aparece.

Dependência

Prompt para Claude Code
Projeto: ~/beta-mdflow/
Supabase projeto: qdivfairxhdihaqqypgb

Tarefa: investigar por que KPI "Engajamento Alto" = 0.0% e corrigir.

INVESTIGAÇÃO:

1. Verificar quantos clientes têm engajamento_nivel = 'alto' em produção:
   Execute via supabase (ou psql se disponível):
   SELECT engajamento_nivel, COUNT(*) 
   FROM clientes 
   WHERE deleted_at IS NULL 
   GROUP BY engajamento_nivel;

   Para executar: cd ~/beta-mdflow && supabase db remote changes --schema public
   Ou criar um arquivo SQL e rodar via: supabase db execute --file /tmp/check.sql

2. Verificar a definição atual de get_clientes_kpis:
   supabase db dump --schema public --linked | grep -A 80 "get_clientes_kpis"
   
   Confirmar como engajamento_alto_pct é calculado.

DIAGNÓSTICO ESPERADO:
- Se COUNT de 'alto' = 0 → campo não está populado (problema de dados, não de código)
- Se COUNT de 'alto' > 0 → bug no cálculo da RPC

CORREÇÃO (se bug na RPC):
Criar migration ~/beta-mdflow/supabase/migrations/20260330000005_fix_get_clientes_kpis_engajamento.sql
Corrigir o cálculo de engajamento_alto_pct:
  ROUND(
    (COUNT(*) FILTER (WHERE engajamento_nivel = 'alto')::numeric / NULLIF(COUNT(*), 0)) * 100, 
    1

Aplicar: supabase db push --linked

RELATÓRIO:

- Qual era o cálculo antigo vs novo?
- Valor do KPI antes e depois
