Achados Detalhados 12 achados

F01 text-[hsl(var(--success))] verboso -- usar text-success
CAT-B P1 CSS only

O token success esta configurado no Tailwind como hsl(var(--success)), portanto text-success funciona nativamente. O padrao text-[hsl(var(--success))] e redundante e quebra a consistencia com outros usos de text-destructive.

Ocorrencias: ClienteList.tsx:231, ClienteKPIDashboard.tsx:47, ImportacaoModal.tsx:421,482

-- className="text-[hsl(var(--success))]"
++ className="text-success"

-- className="text-[hsl(var(--chart-5))]"
++ className="text-chart-5" (se token existe) ou manter hsl var
F02 Zero Tooltips em icon buttons (3 botoes sem tooltip)
CAT-I P2 JSX

Tres botoes icon-only possuem aria-label mas nenhum <Tooltip> visual: o botao "Mais acoes" (MoreHorizontal, mobile), "Novo Cliente" (mobile, icon-only), e "Acoes do cliente" (MoreVertical, cada row).

O aria-label e bom para leitores de tela, mas usuarios videntes precisam de tooltip on hover para descobrir a funcao.

-- <Button aria-label="Mais acoes"><MoreHorizontal /></Button>
++ <Tooltip><TooltipTrigger asChild><Button aria-label="Mais acoes">...</Button></TooltipTrigger><TooltipContent>Mais acoes</TooltipContent></Tooltip>
F03 Import morto: XCircle em ClienteFilters.tsx
CAT-B P2 JSX

O icone XCircle e importado de lucide-react na linha 2 de ClienteFilters.tsx mas nunca e utilizado no componente. Dead import aumenta bundle size marginalmente.

-- import { Search, SlidersHorizontal, XCircle, X, ArrowUpDown } from 'lucide-react';
++ import { Search, SlidersHorizontal, X, ArrowUpDown } from 'lucide-react';
F04 KPIs: apenas 3 cards -- falta KPI de processos ativos
CAT-A P1 Hook/Query

O dashboard KPI tem apenas 3 cards: Total de Clientes, Novos este Mes, Engajamento Alto. Para uma pagina de gestao de clientes, seria esperado um 4o KPI (ex: "Clientes com Processo Ativo" ou "Sem Contato >30 dias") para alinhar com o grid md:grid-cols-3 e dar mais valor analitico.

Em 1440px, o grid 3-cols funciona bem, mas em tablet (768px) 2-cols deixa 1 card orphan. Com 4 KPIs, o grid 2x2 ficaria simetrico.

F05 Tabela com min-w-[1000px] causa scroll horizontal em mobile/tablet
CAT-H P1 CSS only

A tabela principal em ClienteList.tsx:175 tem min-w-[1000px] com overflow-x-auto no container. Isso garante que todas as colunas sejam visiveis, mas em 375px e 640px o usuario precisa fazer scroll horizontal extenso.

Em 375px: a tabela inteira e scrollable, o que e funcional mas suboptimal. Uma alternativa seria trocar para cards em mobile ou esconder colunas menos importantes com hidden md:table-cell.

Atenuante: "Cadastrado em" ja e hidden em <xl. Porem colunas como "Advogado" e "Cidade/UF" poderiam ser hidden em <md tambem.

-- <Table className="min-w-[1000px]">
++ Considerar: ocultar colunas Advogado + Cidade em <md, ou card layout mobile
F06 staleTime inconsistente: hooks de lookup usam 60s-300s em vez de 30s
CAT-B P2 Hook/Query

O padrao MdFlow define staleTime: 30_000 para todos os hooks. Os hooks primarios (useClientes, useClienteKPIs) estao corretos. Porem 4 queries auxiliares usam staleTime maior:

  • ClienteFilters advogados: 5 * 60 * 1000 = 300s
  • NovoClienteModal origens: 5 * 60 * 1000 = 300s
  • NovoClienteModal captadores: 60_000
  • NovoClienteModal advogados: 60_000

Justificativa: dados de lookup (advogados, origens) mudam raramente. staleTime maior e defensavel. Porem o padrao diz 30_000 sem excecao. Classificado como P2.

F07 Paginacao sem indicacao de loading ao trocar pagina
CAT-G P2 JSX

Ao clicar em "Proxima" ou "Anterior", o React Query faz refetch e a tabela fica no estado anterior ate carregar. Nao ha indicador visual de loading na paginacao (ex: spinner nos botoes, ou opacity na tabela durante fetch). O usuario pode achar que o clique nao funcionou.

O hook useClientes retorna isFetching que poderia ser usado para mostrar transicao.

F09 Exportacao CSV client-side sem limite pratico (10.000 rows)
CAT-E P1 Hook/Query

A funcao handleExportar faz .limit(10000) no Supabase, mas para organizacoes grandes isso pode travar o browser. Nao ha confirmacao pre-export nem indicacao de quantos registros serao exportados.

Alem disso, a exportacao roda client-side com Papa.unparse, o que e mais lento que server-side para grandes volumes.

Recomendacao: Para <1000 registros, client-side e OK. Para >1000, considerar webhook n8n com download via URL (WH-CLIENTES-005).

F10 KPI "Engajamento Alto" sem delta (inconsistente com "Novos este Mes")
CAT-C P2 Hook/Query

O card "Novos este Mes" mostra delta vs mes anterior (+N ou -N). O card "Engajamento Alto" nao mostra delta, mesmo que a informacao seria valiosa (engajamento subiu ou caiu). O card "Total" tambem nao mostra delta.

Visualmente, a ausencia de delta nesses 2 cards cria assimetria: 1 card tem sub-texto, 2 nao tem. A altura visual fica inconsistente.

F11 Popover de filtros sem scroll indicator em mobile
CAT-H P2 CSS only

O popover de filtros usa w-[min(420px,95vw)] e max-h-[70vh] com overflow-y-auto. Em 375px, o popover ocupa quase toda a tela e o usuario pode nao perceber que ha conteudo abaixo (especialmente a secao de Periodo).

Sugestao: Considerar usar Sheet (bottom drawer) em mobile em vez de Popover, que e mais natural para toque.

F12 Search/Sort/Filtros em linha unica -- overflow em 640px
CAT-H P2 CSS only

Em 640px, os 3 elementos (search input max-w-sm, sort select w-[170px], botao Filtros) competem por espaco horizontal. O layout flex items-center gap-3 nao quebra em linhas. Em viewport estreito, o input de busca fica comprimido.

Sugestao: Adicionar flex-wrap ao container ou mover Sort para dentro do popover de filtros em <md.

-- <div className="flex items-center gap-3">
++ <div className="flex items-center gap-3 flex-wrap">
