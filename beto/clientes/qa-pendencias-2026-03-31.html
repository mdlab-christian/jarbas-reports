🔴 Pendência 1 — Restrições não aparecem no ClienteDrawer (context=clientes)

BUG-P1 CRÍTICO Aba "Restrições" vazia para clientes no contexto /clientes

Arquivos Envolvidos

src/hooks/useRestricoes.ts src/hooks/useClienteRestricoes.ts src/components/shared/cliente_drawer/ClienteDrawer.tsx src/components/shared/cliente_drawer/tabs/ClienteDrawerTabRestricoes.tsx

🔍 Diagnóstico do Banco

Query executada: GET /rest/v1/restricoes?select=id,cliente_id,lead_id,organizacao_id&limit=5&order=created_at.desc

Registros retornados: 5
Campo cliente_id: null em TODOS os 5 registros
Campo lead_id: preenchido em todos os registros

Exemplo:
{ id: "2abe38c9...", cliente_id: null, lead_id: "140eec64-4ef6-4097-ac07-e51cd4ccffec" }
{ id: "9be2dda8...", cliente_id: null, lead_id: "bf897aee-3604-4d4d-ac8c-c33465894005" }

🔍 Análise do Fluxo de Dados

1. ClienteDrawer.tsx é aberto com clienteId (UUID da tabela clientes)
2. useRestricoes() é chamado com clienteId mas sem leadId (quando contexto=clientes e não há lead associado)
3. A RPC fn_restricoes_listar recebe p_cliente_id mas todos os registros têm cliente_id = NULL
4. RESULTADO: 0 registros retornados — lista vazia na aba Restrições

🐛 Causa Raiz

O problema tem duas camadas:

Camada 1 — Campo cliente_id NULL no banco (linha 27, useClienteRestricoes.ts)

// useClienteRestricoes.ts — usa RPC diferente (get_cliente_restricoes_completo)
// Mas ClienteDrawerTabRestricoes.tsx usa useRestricoes (não useClienteRestricoes!)

// ClienteDrawerTabRestricoes.tsx linha 155:
const { data: restricoesAll = [], isLoading } = useRestricoes({
  organizacaoId, clienteId, leadId,
  apenasSemProcesso: !showProcessoStatus,
})

Camada 2 — ClienteDrawer.tsx linha 165-169: leadId pode ser undefined no contexto clientes

// ClienteDrawer.tsx — linhas 165-169
const { data: restricoes = [] } = useRestricoes({
  organizacaoId,
  leadId: lead?.id,              // ← undefined quando context='clientes' sem lead
  clienteId: clienteByCpf?.id ?? clienteById?.id,  // ← tem valor
  enabled: open,
})

Camada 3 — Banco: campo cliente_id nunca é populado nas restrições

Resultado da query: cliente_id = null em TODOS os registros As restrições são criadas APENAS com lead_id (via CRM/lead flow) Quando o lead vira cliente, cliente_id NÃO é propagado para a tabela restricoes

💡 Impacto

Aba "Restrições" aparece sempre vazia para clientes no contexto /clientes — mesmo que o cliente tenha restrições cadastradas via CRM. Afeta 100% dos clientes com restrições.

✅ Solução Recomendada

Solução A (banco): Criar trigger/migration que popula restricoes.cliente_id ao criar/atualizar restrições de leads que já viraram clientes. Fazer backfill dos registros existentes com JOIN leads → clientes via cpf.

Solução B (front + RPC): No ClienteDrawer.tsx, quando context='clientes', buscar o lead_id correspondente ao cliente (via campo clientes.lead_id) e passar para useRestricoes. A tabela clientes tem campo lead_id (confirmado no banco).

clienteById?.lead_id → passar como leadId para useRestricoes
A tabela clientes possui campo lead_id (confirmado: cliente "LEAD TEST DISTRIBUICAO" tem lead_id "91491d4d-d064-48bd-a890-e46b73f7c9d9"). A Solução B é mais rápida e não exige migration.

🔴 Pendência 2 — Empresas (CNPJ) aparecem na ClienteList

BUG-P2 CRÍTICO (RPC) RPC get_clientes_paginados com overloading — p_excluir_cnpj pode não ter efeito

Arquivos Envolvidos

src/hooks/useClientes.ts src/pages/clientes/components/ClienteList.tsx [Supabase] public.get_clientes_paginados

🔍 Diagnóstico do Banco — Dados

Query: GET /rest/v1/clientes?select=id,nome,cpf_cnpj&limit=50&order=created_at.desc
Amostra de 50 registros analisada:
- CPF (11 dígitos): 35 registros - CNPJ (14 dígitos): 0 registros - cpf_cnpj NULL: 15 registros - CPF formatado (xxx.xxx.xxx-xx): presente em alguns (dados seed)

Exemplo de registro sem máscara: { nome: "MICHELE BUENO", cpf_cnpj: "27770831857" } Registros com / (CNPJ format): 0 na amostra atual

🐛 Causa Raiz — CRÍTICA: RPC com Overloading

A causa raiz NÃO é filtro client-side, mas sim um problema crítico na RPC do Supabase:

// ERRO retornado pelo Supabase ao chamar get_clientes_paginados SEM p_excluir_cnpj:
{
  "code": "PGRST203",
  "message": "Could not choose the best candidate function between: 
    public.get_clientes_paginados(p_search, p_estados, p_advogado_ids, 
      p_periodo_inicio, p_periodo_fim, p_carteira, p_engajamento_niveis, 
      p_page, p_page_size),
    public.get_clientes_paginados(p_search, p_estados, p_advogado_ids, 
      p_periodo_inicio, p_periodo_fim, p_carteira, p_engajamento_niveis, 
      p_page, p_page_size, p_excluir_cnpj => boolean)"
}

// ERRO ao chamar COM p_excluir_cnpj=true:
{
  "code": "42703",
  "message": "column adv.deleted_at does not exist",
  "hint": "Perhaps you meant to reference the column c.deleted_at"
}
  
Código do hook — useClientes.ts linha 22

  
// src/hooks/useClientes.ts — linha 22
const { data, error } = await supabase.rpc('get_clientes_paginados' as any, {
  p_search: filters.search || null,
  p_estados: filters.estados.length ? filters.estados : null,
  p_advogado_ids: filters.advogado_ids.length ? filters.advogado_ids : null,
  p_periodo_inicio: filters.periodo_inicio || null,
  p_periodo_fim: filters.periodo_fim || null,
  p_carteira: filters.carteira || null,
  p_engajamento_niveis: filters.engajamento_niveis.length ? filters.engajamento_niveis : null,
  p_page: page,
  p_page_size: 50,
  p_excluir_cnpj: true,   // ← parâmetro adicionado na nova versão da RPC
});


  
🔍 Análise Detalhada do Overloading

  

    

      1. Existem 2 versões da RPC no banco: versão antiga (9 params) e nova (10 params com p_excluir_cnpj)
    

    

      2. PostgREST não consegue resolver qual usar → erro PGRST203
    

    

      3. Quando consegue usar a versão nova: erro 42703 "adv.deleted_at does not exist" dentro da RPC
    

    

      4. RESULTADO: Lista de clientes falha silenciosamente ou retorna vazia
    

  


  
💡 Impacto

  
A página /clientes pode exibir tela em branco ou lista vazia para todos os usuários. O filtro de CNPJ nunca chega a ser aplicado pois a RPC falha antes.


  
✅ Solução Recomendada

  

    P0 — Correção urgente no banco:

    1. Remover a versão antiga da RPC get_clientes_paginados (sem p_excluir_cnpj)

    2. Corrigir o bug interno da nova versão: substituir adv.deleted_at por c.deleted_at na query interna

    3. Garantir que o filtro WHERE length(regexp_replace(c.cpf_cnpj, '\D', '', 'g')) = 11 (ou similar) está implementado quando p_excluir_cnpj = true

// helpers.ts — linha 17
export function applyPhoneMask(v: string, maxLen = 11) {
  const d = normalizeDigits(v).slice(0, maxLen);
  if (d.length <= 2) return d;
  if (d.length <= 7) return `(${d.slice(0, 2)}) ${d.slice(2)}`;
  if (maxLen === 11) return `(${d.slice(0, 2)}) ${d.slice(2, 7)}-${d.slice(7)}`;
  return `(${d.slice(0, 2)}) ${d.slice(2, 6)}-${d.slice(6)}`;
}

No edit mode CLIENTE: CORRETO — linha 222 aplica a máscara no initialState.
No edit mode LEAD: BUG — linha 321 não aplica a máscara no initialState.

💡 Impacto

Baixo — o campo mostra o número sem máscara somente no momento de abertura do modo de edição para contexto CRM/leads. Ao digitar qualquer caractere, a máscara é aplicada. No contexto /clientes (view + edit mode), o comportamento está correto.

✅ Solução Recomendada

Correção simples (1 linha): Na função handleStartEdit(), linha ~321:
// DE:

// PARA:
