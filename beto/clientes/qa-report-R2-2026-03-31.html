P0-001
RPC get_clientes_paginados ignora LIMIT/OFFSET e p_search
backend RPC paginação busca
FALHOU
Arquivo:Supabase: função PostgreSQL get_clientes_paginados Impacto:100% da funcionalidade de busca e paginação — inutilizável Rota afetada:/clientes

Evidência — HAR capturado

POST https://qdivfairxhdihaqqypgb.supabase.co/rest/v1/rpc/get_clientes_paginados
Request body: {"p_search":"Michele","p_page":1,"p_page_size":50,"p_excluir_cnpj":true,...}
Response: HTTP 200 {"data":[ALL 4342 records],"total":4342,"page":1,"page_size":50,"total_pages":87}
Confirmação banco (user JWT):
# p_page_size=50 mas retornou 4342 registros:
Total (Michele search): 4342  ← deveria ser ~14
Page size recebido: 50 (metadado correto, mas data ignorou)
Received data count: 4342 ← deveria ser ≤ 50

# Page 2 retorna VAZIO:
p_page=2, p_page_size=50 → count: 0  ← impossível se total=4342

# CNPJ exclusion funciona:
p_excluir_cnpj=true → 4347-5 = 4342 ✓ (único filtro que funciona)

Causa Raiz

O SQL da função get_clientes_paginados em PostgreSQL não aplica corretamente LIMIT p_page_size OFFSET (p_page-1)*p_page_size nem o filtro WHERE unaccent(lower(nome)) ILIKE '%' || unaccent(lower(p_search)) || '%'. Todos os dados são retornados em uma única página.

Solução Proposta

-- Verificar e corrigir a função no Supabase:
CREATE OR REPLACE FUNCTION get_clientes_paginados(
  p_search TEXT DEFAULT NULL,
  p_page INT DEFAULT 1,
  p_page_size INT DEFAULT 50,
  p_excluir_cnpj BOOLEAN DEFAULT true,
  -- outros parâmetros...
) RETURNS JSONB AS $$
  ...
  -- VERIFICAR: WHERE e LIMIT/OFFSET estão sendo aplicados?
  LIMIT p_page_size
  OFFSET (p_page - 1) * p_page_size
  ...
$$ LANGUAGE sql SECURITY DEFINER;
P0-002
ClienteDrawer sempre mostra "Cliente não encontrado" — HTTP 400 por injeção de join via SetofOptions
frontend supabase-js types drawer
FALHOU
Arquivo:src/integrations/supabase/types.ts linha 32723 + src/hooks/useCliente.ts Impacto:100% dos clientes — drawer completamente inacessível Rota afetada:/clientes (drawer), /pagina_cliente

Evidência — HAR capturado

# Quando usuário clica em MICHELE BUENO (CPF: 277.708.318-57):
POST https://qdivfairxhdihaqqypgb.supabase.co/rest/v1/rpc/get_cliente_by_cpf
?select=*,advogado_responsavel:advogados!advogado_responsavel_id(id,nome,oab,oab_uf)
Body: {"p_cpf":"27770831857"}
Response: HTTP 400 {"code":"PGRST200","message":"Could not find a relationship between 'get_cliente_by_cpf' and 'advogados'"}

Causa Raiz

O arquivo types.ts gerado pelo Supabase CLI define SetofOptions para a função: 

// src/integrations/supabase/types.ts, linha 32723
SetofOptions: {
  from: "*"
  to: "clientes"      // ← PROBLEMA: diz que o RPC retorna SETOF clientes
  isOneToOne: false
  isSetofReturn: true
}

Isso faz o supabase-js v2.97.0 injetar automaticamente o ?select=*,advogado_responsavel:advogados!... na URL quando o código chama .rpc('get_cliente_by_cpf').maybeSingle(). PostgREST retorna HTTP 400 porque o RPC não é uma tabela e não tem FK com advogados.

Solução Proposta (2 opções)

// OPÇÃO A — Remover SetofOptions no types.ts (ou regenerar via CLI sem essa option):
// Linha 32723: deletar o bloco SetofOptions do get_cliente_by_cpf

// OPÇÃO B — No hook, usar cast 'as any' para ignorar tipos e evitar injeção:
// src/hooks/useCliente.ts
const { data, error } = await supabase
  .rpc('get_cliente_by_cpf' as any, { p_cpf: cpfSlug! })
  .maybeSingle();

// OPÇÃO C — Usar .from('clientes') diretamente em vez do RPC:
const { data, error } = await supabase
  .from('clientes')
  .select('*, advogado_responsavel:advogados!advogado_responsavel_id(id,nome,oab,oab_uf)')
  .eq('cpf_cnpj', cpfSlug!)
  .maybeSingle();
// Recomendação: Opção C elimina dependência do RPC e resolve o join nativo.
