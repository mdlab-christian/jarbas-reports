Achados Detalhados

P1 SR-001 -- ClienteIdRedirect sem organizacao_id

O componente ClienteIdRedirect.tsx (linha 13-22) faz query direta a tabela clientes usando apenas .eq('id', id) sem filtrar por organizacao_id. Embora o RLS da tabela filtre por org_id() no SELECT, este componente nao usa useAuth() e portanto nao verifica explicitamente a organizacao.

Impacto: Se o RLS estiver desabilitado temporariamente ou se a query usar service_role, poderia expor cpf_cnpj de outro tenant. Na pratica, o SDK usa anon key com RLS ativo, entao o risco imediato e baixo. Contudo, viola o principio P-04 (multi-tenant absoluto) e deve ser corrigido.

Arquivo: src/pages/clientes/ClienteIdRedirect.tsx linhas 13-22
Fix: Adicionar const {'{'} organizacao_id {'}'} = useAuth(); e .eq('organizacao_id', organizacao_id) na query.

P1 SR-002 -- Erros no workflow CLIENTES-CRUD (n8n)

O workflow [GUSTAVO] CLIENTES-CRUD (id: 6Z_tuZgOH5JuZr8gGcVyX) esta ativo e as ultimas 2 execucoes (2026-04-02) foram bem-sucedidas. Porem, em 2026-04-01 houve 2 execucoes com status: error (ids: 183507, 183506, ambas as 19:17).

Impacto: Falha silenciosa na exclusao de cliente via webhook (confirmado pelo DEEP-004 do ciclo anterior). O workflow esta recuperado agora, mas os 2 erros nao tiveram toast de erro no frontend (o catch generic mostra "Erro ao excluir cliente").

Evidencia: n8n API /executions?workflowId=6Z_tuZgOH5JuZr8gGcVyX retornou 2 execucoes error em 2026-04-01 19:17.
Observacao: O workflow CLIENTES-009 (Engajamento) tambem esta ativo e sem execucoes recentes (nao ha problema, e cron).

P2 SR-003 -- staleTime inconsistente nos lookups

O CLAUDE.md define staleTime: 30_000 como padrao para todos os hooks React Query. Os hooks principais (useClientes e useClienteKPIs) seguem corretamente com 30s. Porem, as queries de lookup divergem:

QuerystaleTimeEsperado
advogados-filter-list (ClienteFilters)5 * 60 * 1000 (5 min)30_000
leads-origens (NovoClienteModal)5 * 60 * 1000 (5 min)30_000
captadores (NovoClienteModal)60_000 (1 min)30_000
advogados (NovoClienteModal)60_000 (1 min)30_000

Nota: Lookups estao carregados apenas quando modal/popover abre (enabled: open), entao o staleTime alto faz sentido pratico. Padronizar e uma questao de consistencia, nao de funcionalidade.

P2 SR-004 -- Export client-side com limite 10k

A funcao handleExportar (index.tsx, linha 168) usa .limit(10000). Com 4.350 clientes ativos na org Midas, isso e suficiente hoje. Porem, nao ha aviso ao usuario de que a exportacao foi truncada caso atinja 10k no futuro.

Fix sugerido: Adicionar toast de aviso se data.length === 10000: "Exportacao limitada a 10.000 registros. Para exportacao completa, entre em contato."

P2 SR-005 -- KPI dashboard com 3 cards (spec previa 4)

O ClienteKPIDashboard exibe 3 KPIs: Total de Clientes, Novos este Mes, Engajamento Alto. O ciclo anterior (F04) identificou que o grid seria visualmente mais equilibrado com 4 cards (ex: "Com Processos Ativos"). A RPC get_clientes_kpis retorna apenas 4 campos, nao inclui contagem de processos.

Decisao: O grid sm:grid-cols-2 md:grid-cols-3 funciona bem com 3 cards. Um 4o card exigiria mudanca na RPC. Nao e bloqueante.

P2 SR-006 -- UF buttons sem aria-label no popover de filtros

No popover de filtros (ClienteFilters.tsx linhas 260-275), os 27 botoes de UF sao <button> com texto curto ("AC", "AL", etc.) mas sem aria-label. O texto curto e suficiente para leitores de tela na maioria dos casos, mas poderia ser melhorado com "Filtrar por estado AC", "Filtrar por estado AL", etc.

Impacto: Baixo. Os botoes tem texto visivel que coincide com o label semantico. E um refinamento, nao um bug.
