Bugs por Prioridade

P1 Bugs Importantes (5)

P1-01 config_paginas_sistema: registros duplicados corrompem PermissionGrid
components/PermissionGrid.tsx:48-58 | Tabela: config_paginas_sistema

Evidencia: Query Supabase retorna duplicatas: dois registros para /homepage (IDs 60fc11c1 e d7b0f5de), dois para /dashboard (IDs 287d3de9 e 6694c707).

Causa raiz: Tabela config_paginas_sistema nao tem constraint UNIQUE em (rota). Inserts duplicados geraram linhas fantasmas.

Impacto: PermissionGrid mostra paginas duplicadas na grid. Admin pode atribuir permissoes inconsistentes (salva em uma linha mas nao na outra).

Solucao: (1) DELETE duplicatas mantendo a de menor created_at. (2) Adicionar UNIQUE constraint em (rota) ou (rota, organizacao_id).

P1-02 PermissionGrid nao filtra por organizacao_id
components/PermissionGrid.tsx:48-58
const { data: paginas } = useQuery({
  queryFn: async () => {
    const { data, error } = await supabase
      .from('config_paginas_sistema')
      .select('id, rota, titulo, icone, ordem')
      .eq('ativo', true)  // <-- FALTA: .eq('organizacao_id', organizacaoId)
      .order('ordem');
  ...

Causa raiz: A query de paginas nao aplica filtro organizacao_id. Se a tabela tiver RLS isso e mitigado pelo JWT, mas com service_role ou em contextos futuros multi-tenant, mostraria paginas de outras orgs.

Impacto: Risco de vazamento multi-tenant em cenario multi-org. Atualmente mitigado parcialmente por RLS.

Solucao: Adicionar .eq('organizacao_id', organizacaoId) na query. Verificar se tabela tem coluna organizacao_id.

P1-03 TabContas: contas bancarias filtram ativo=true no SELECT mas exibem saldo como reais em vez de centavos
tabs/TabContas.tsx:34-36 | modals/ModalContaBancaria.tsx:152
// TabContas query:
.from('contas_bancarias').select('*').eq('ativo', true)

// ModalContaBancaria save:
saldo_atual: saldoAtualCentavos != null ? saldoAtualCentavos / 100 : 0

Causa raiz: O modal salva saldoAtualCentavos / 100 (converte centavos para reais antes de salvar). Isso conflita com a convencao do CLAUDE.md que diz "Banco: INT8 centavos". Se o banco armazena em reais (float), formatBRLReais na exibicao (TabContas:145) pode estar fazendo dupla divisao ou nenhuma.

Impacto: Saldo pode ser exibido errado (1500 centavos = R$15 mas se banco ja recebe R$15, display mostra R$0,15).

Solucao: Verificar schema da coluna saldo_atual (NUMERIC? INT8?). Se for reais no banco, nao dividir. Se centavos, nao dividir no save.

P1-04 TabContas: Nao mostra contas/cartoes inativos — sem opcao de reativar
tabs/TabContas.tsx:35-36 + :68-69
// Contas bancarias query:
.eq('ativo', true)

// Cartoes credito query:
.eq('ativo', true)

Causa raiz: Ambas as queries filtram ativo=true. Apos desativar uma conta/cartao, nao ha como ve-lo novamente na UI para reativar.

Impacto: Dados no banco (contas: 1 inativa, cartoes: 1 inativo) ficam inacessiveis. Confirmado: conta "a" e cartao "21321412312312312" estao ambos com ativo=false e invisíveis na listagem.

Solucao: Remover .eq('ativo', true) e adicionar badge de status + opcao de reativar, como feito em TabAdvogados e TabUsuarios.

P1-05 CrmFollowupConfig: query templates usa 'canal' mas tabela config_templates tem coluna 'tipo'
tabs/crm/CrmFollowupConfig.tsx:54-62
const { data: templates } = useQuery({
  queryFn: async () => {
    const { data } = await supabase
      .from('config_templates' as never)  // 'as never' sugere tipo incorreto
      .select('id, nome, canal')          // coluna 'canal' NAO EXISTE
      .eq('organizacao_id', organizacaoId)
      .eq('ativo', true)

Causa raiz: A tabela config_templates tem coluna tipo (whatsapp/email/sms/documento), nao canal. O cast as never mascara o erro de tipo. O select provavelmente retorna null para canal.

Impacto: Dropdown de templates no dialog de follow-up mostra nomes sem indicador de canal. Funciona parcialmente mas informacao de tipo e perdida.

Solucao: Trocar .select('id, nome, canal') por .select('id, nome, tipo'). Remover as never. Ajustar tipo da interface.

P2 Melhorias e Inconsistencias (6)

P2-01 7 queries usam .select('*') em vez de campos explícitos
TabContas(2), TabEtiquetas, TabTemplates, CrmOrigens, CrmFollowup, CrmEstagios

Impacto: Baixo. Traz dados desnecessarios (ex: created_at, updated_at). Viola convencao CLAUDE.md: "supabase.from('x').select('*') → .eq('organizacao_id', orgId)".

Solucao: Substituir por campos explícitos em cada query.

P2-02 ModalConvidarUsuario nao inclui perfil 'admin' no Select
modals/ModalConvidarUsuario.tsx:141-149

Impacto: Baixo. Correto por design (admin e perfil elevado, nao deve ser atribuido casualmente). Mas ModalEditarUsuario tambem nao permite alterar de/para admin. Coerente.

Status: By design. Nao e bug.

P2-03 TabAdvogados: advogado.oab armazena apenas digitos mas exibicao tenta reconstruir formato
tabs/TabAdvogados.tsx:145-149 | Banco: advogados.oab="RS116571"

Evidencia: Banco mostra oab="RS116571" (com UF prefixada), mas o campo oab_uf ja tem "RS" separado. O renderOab faz maskOAB(uf + oab) resultando em "RSRS116571".

Impacto: Medio. OAB pode ser exibida como "RS/RS116571" em vez de "RS/116571".

Solucao: Verificar se oab no banco ja inclui UF. Se sim, nao concatenar oab_uf. Ou normalizar dados: oab deve ser apenas digitos.

P2-04 HealthValidatorBanner: Drive e WhatsApp retornam 'warning' fixo
components/HealthValidatorBanner.tsx:83-91

Causa: Funcoes checkDrive() e checkMdZap() sao stubs que retornam status warning fixo ("Verificacao nao configurada").

Impacto: Baixo. Banner de saude sempre mostra 2 warnings mesmo que tudo esteja OK. Pode confundir usuario.

Solucao: Implementar health checks reais ou remover items do INITIAL_ITEMS ate implementacao.

P2-05 TabUsuarios: delete faz hard delete em vez de soft delete
tabs/TabUsuarios.tsx:153-155
const { error } = await supabase
  .from('users')
  .delete()          // HARD DELETE
  .eq('id', userId)

Impacto: Viola convencao CLAUDE.md "Soft delete obrigatorio (deleted_at). Hard delete proibido com FK". A tabela users provavelmente tem FKs (processos.responsavel_id, etc.). O codigo verifica processos vinculados antes, mas outros FKs podem falhar silenciosamente.

Solucao: Substituir por .update({ deleted_at: new Date().toISOString() }) ou manter desativacao como unica opcao.

P2-06 TabEstados: tabela 'estados_distribuicao' vs 'estados' do checklist original
tabs/TabEstados.tsx:67

Nota: O checklist original menciona tabela "setores" para Tab Estados, mas a tab real e sobre estados de atuacao (UFs), nao setores. Setores sao gerenciados via ModalGerenciarSetores dentro de TabUsuarios. Nomenclatura no checklist e confusa mas codigo esta correto.
