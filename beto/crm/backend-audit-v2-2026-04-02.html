Bugs Encontrados
P0 rpc_crm_bulk_action aceita coluna 'documentos_pendentes' que viola CHECK constraint
Causa Raiz
O CHECK constraint de leads.coluna_crm permite apenas 7 valores: novos, em_atendimento, aguardando_documentos, aguardando_assinatura, triagem, arquivados, convertido.

A RPC rpc_crm_bulk_action (ação mover_coluna) valida internamente 8 valores aceitando também documentos_pendentes. A validação da RPC passa, mas o UPDATE subsequente na tabela leads levanta ERROR: new row for relation "leads" violates check constraint "leads_coluna_crm_check" — o erro é capturado pelo bloco EXCEPTION e retorna success: false silenciosamente.
Impacto
Qualquer ação bulk "mover para Documentos Pendentes" via CRM falha silenciosamente — 0 leads movidos, usuário não sabe por quê.
Solução — Opção A: adicionar 'documentos_pendentes' ao CHECK (recomendada)
ALTER TABLE leads DROP CONSTRAINT leads_coluna_crm_check; ALTER TABLE leads ADD CONSTRAINT leads_coluna_crm_check CHECK (coluna_crm IN ( 'novos','em_atendimento','aguardando_documentos', 'aguardando_assinatura','triagem','arquivados', 'convertido','documentos_pendentes' ));
Solução — Opção B: remover 'documentos_pendentes' da lista de colunas válidas na RPC
-- No corpo de rpc_crm_bulk_action, substituir: IF v_nova_coluna NOT IN ('novos','em_atendimento','aguardando_documentos', 'aguardando_assinatura','triagem','arquivados','convertido','documentos_pendentes') -- por: IF v_nova_coluna NOT IN ('novos','em_atendimento','aguardando_documentos', 'aguardando_assinatura','triagem','arquivados','convertido')
P1 rpc_crm_kanban_counts não existe no banco
Causa Raiz
A spec do /crm cita rpc_crm_kanban_counts como RPC esperada. A função não foi encontrada na busca no pg_proc. O frontend usa queryKey crm-colunas-contagem (invalidado pelo useCrmKanban), mas sem RPC dedicada — possivelmente a contagem é feita via query direta ou não está implementada.
Impacto
Se o frontend chamar essa RPC, receberá erro 404 / PGRST202. Contadores de colunas do kanban podem não carregar.
Solução
-- Verificar se frontend chama essa RPC diretamente. -- Se sim, criar: CREATE OR REPLACE FUNCTION rpc_crm_kanban_counts(p_organizacao_id UUID) RETURNS JSONB LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$ BEGIN RETURN ( SELECT jsonb_object_agg(coluna_crm, cnt) FROM ( SELECT coluna_crm, COUNT(*) as cnt FROM leads WHERE organizacao_id = p_organizacao_id AND arquivado = false AND deleted_at IS NULL GROUP BY coluna_crm ) sub ); END; $$;

P2 leads_arquivamento_motivo sem campo updated_at e sem trigger
Causa Raiz
Schema: id, organizacao_id, nome, ativo, created_at. Faltam updated_at e deleted_at. Sem deleted_at, itens arquivados são hard-delete ou desativados via flag ativo=false.
Impacto
Auditoria de motivos de arquivamento incompleta. Sem updated_at, impossível rastrear quando o nome/ativo foi alterado.
Solução
ALTER TABLE leads_arquivamento_motivo ADD COLUMN updated_at TIMESTAMPTZ DEFAULT now() NOT NULL, ADD COLUMN deleted_at TIMESTAMPTZ; CREATE TRIGGER set_updated_at BEFORE UPDATE ON leads_arquivamento_motivo FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
