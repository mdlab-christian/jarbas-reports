3. Auditoria de RPCs

❌ P1 — fn_crm_kanban QUEBRADA (schema drift)
Referencia l.empresa e l.vendedor_id que não existem mais na tabela leads. Retorna erro: "column l.empresa does not exist". Esta RPC provavelmente era usada em versão anterior e foi abandonada — porém ainda existe no banco e quebra se chamada. 
-- Erro retornado:
{"code":"42703","message":"column l.empresa does not exist","details":"..."}

-- Fix: DROP or UPDATE the function
-- Se não é mais usada pelo frontend, remover:
DROP FUNCTION IF EXISTS fn_crm_kanban(uuid);
⚠️ P2 — fn_crm_stats usa p_org_id em vez de p_organizacao_id
Convenção MdFlow: parâmetros devem usar p_organizacao_id. A função fn_crm_stats usa p_org_id. O frontend não chama esta função diretamente (não encontrada nos hooks), mas há inconsistência de nomenclatura.
✅ rpc_crm_dashboard_kpis — Bem implementada
SECURITY DEFINER confirmado, parâmetros com prefixo p_, retorna JSONB com envelope success/data, RLS funcionando (org_ficticia retorna zeros).
✅ fn_restricoes_listar — Corrigida em sessão anterior
Bug crítico de JOIN (AND o.deleted_at IS NULL em tabela sem essa coluna) foi corrigido. Agora retorna 9.919 restrições corretamente. RLS isolamento verificado: org_ficticia retorna 0.
✅ fn_restricao_salvar — Funcional
Assinatura completa verificada: p_id, p_organizacao_id, p_lead_id, p_empresa_nome, p_orgao_nome, p_empresa_id, p_valor, p_tipo, p_processavel, p_motivo_nao_processavel, p_data_restricao, p_codigo_contrato, p_cliente_id. Retorna UUID da restrição. CHECK constraint 23514 falhou corretamente para dados inválidos (sem orgao_id quando requerido).
✅ rpc_disparar_auditoria — Funcional
Assinatura: p_organizacao_id, p_lead_id, p_cpf, p_bureaus (array), p_user_id. Retorna envelope canônico: {"success":true,"action":"open_modal","data":{"modal":"status_auditoria","auditoria_id":"..."}}. Resposta canônica correta.
✅ distribuir_lead_completo — Funcional
Validação de regra de negócio funcionando: retorna {"success":false,"message":"Lead não está na coluna triagem"} para lead fora da coluna correta. Comportamento esperado.

Chamadas do Frontend que NÃO existem como RPC

O frontend usa principalmente views + queries diretas em leads — não RPCs para listagem. Isso é aceitável para a view pattern mas reduz a capacidade de otimização server-side centralizada.

  • useCrmKanban.ts → acessa leads diretamente (view seria mais seguro)
  • CrmPage.tsx → acessa view_crm_colunas_contagem diretamente ✅
  • useLeadFollowup.ts → acessa leads_atividades diretamente (sem RPC de negócio)

Fix P1 — Adicionar USING clause em auditorias RLS

-- Verificar policies atuais:
SELECT policyname, cmd, qual, with_check FROM pg_policies WHERE tablename = 'auditorias';

-- Se houver apenas INSERT sem SELECT policy:

  FOR SELECT USING (organizacao_id = public.get_org_id());
