ETAPA_COLORS usa orange-500/orange-600 — fora do Design System
src/types/lead.types.ts:52
CSS only
A entrada documentos_pendentes no objeto ETAPA_COLORS usa orange-500 e orange-600, que são cores hardcoded fora dos tokens do Design System Warm Charcoal v5. O DS não define --orange como token. O correto é usar --warning (amber) ou criar um token --chart-3 (amber) que já existe no DS.
ANTES — Problema
Docs Pendentes orange-600 (HARDCODE)
documentos_pendentes: 'bg-orange-500/10 text-orange-600 border-orange-500/20'
DEPOIS — Correto
Docs Pendentes warning token (DS)
documentos_pendentes: 'bg-warning/10 text-warning border-warning/20'

CrmLeadCard usa amber-500/amber-600/dark:amber-400 — fora dos tokens DS
src/components/crm/CrmLeadCard.tsx:250
CSS only
O badge de notas do lead usa border-amber-500/30 text-amber-600 dark:text-amber-400. Amber não é um token do DS. O equivalente correto é --warning (hsl 38 92% 50% = #f59e0b). Adicionalmente, o uso de dark: prefix é desnecessário — o projeto é sempre dark (Warm Charcoal).

// tokens DS — sem dark: prefix

P1
CrmKanban.staleTime = 0 — aggressivo, sem fallback inteligente
CrmKanban.tsx:157
JSX
O kanban usa staleTime: 0 + refetchInterval: 60_000. O staleTime=0 significa que toda navegação entre tabs causa refetch imediato, degradando UX com flicker de opacity 50% a cada troca de aba. O realtime já está configurado via useCrmRealtime na CrmPage. O correto é staleTime: 30_000 + manter o refetchInterval como fallback para Realtime caído.
ANTES — UX degradada
staleTime: 0,
// refetch IMEDIATO ao voltar para pipeline tab
// kanban fica em opacity 50% → flickering
DEPOIS — Sem flickering
staleTime: 30_000,
// 30s de cache — realtime cobre o rest
// refetchInterval: 60_000 permanece como fallback
