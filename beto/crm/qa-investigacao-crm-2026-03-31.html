Área 1 — Aba Documentos do Drawer

ClienteDrawerTabDocumentosLead.tsx · RestricaoPastaCard.tsx · DocumentoItem.tsx · documentos_lead_hooks.ts

🏗️ Arquitetura atual — 3 tiers

Todos os documentos (lead, restrição, processo) ficam na mesma tabela processos_documentos, diferenciados por categoria + relacionamentos. Storage: bucket leads-docs.

Tier 1 — Documentos Centrais do Lead

  • Categoria: documento_lead, restricao_id = null
  • Upload com drag-drop na área central
  • Lista de arquivos com DocumentoItem (Abrir, Baixar, Renomear, Excluir)
  • Sem seleção múltipla — apenas ações individuais por arquivo

Tier 2 — Pastas por Restrição Processável (RestricaoPastaCard)

  • Categoria: documento_restricao — uma pasta accordion por restrição processável
  • Upload com drag-drop por pasta
  • Cada arquivo tem ações individuais via DocumentoItem (Abrir, Baixar, Renomear, Excluir)
  • Tier 1 docs aparecem dentro de cada pasta também (por design — visibilidade)
  • Sem checkboxes de seleção múltipla — não tem bulk select
  • Sem botão de apagar a PASTA (restrição) no header do card

Tier 3 — Contratos Gerados (ProcessosDocs)

  • Docs vinculados a processos gerados após distribuição
  • Botão "Gerar Petição" / "Gerar Procuração" por processo
  • Tem seleção múltipla — mas apenas para Download (não Rename/Delete)
  • Label muda com isDistribuicao: "Gerar Petição" vs "Gerar Procuração"

BUG-000 — DocumentoItem: tabela hardcoded 'processos_documentos' mas sem google_drive_id na interface
DocumentoItem.tsx:41-48

Problema crítico de resolução de URL: DocumentoItem tenta resolver a URL do doc verificando doc.google_drive_id com isStoragePath(), mas a interface DocumentoItemDoc define esse campo como google_drive_id?: string | null. O problema real é que o campo no banco é google_drive_id, mas os hooks useLeadDocs e useRestricaoDocs NÃO incluem esse campo no SELECT.

// documentos_lead_hooks.ts — useLeadDocs (linha 26)
supabase
  .from('processos_documentos')
  .select('id, nome, arquivo_url, status, created_at')  // ← google_drive_id AUSENTE

// DocumentoItem.tsx — resolveUrl (linha 41-48)
const resolveUrl = async (): Promise<string | null> => {
  if (isStoragePath(doc.google_drive_id ?? null)) {  // ← google_drive_id sempre undefined/null
    return getSignedUrl(doc.google_drive_id!, 'leads-docs')
  }
  if (doc.arquivo_url?.startsWith('http')) return doc.arquivo_url
  if (doc.arquivo_url) return getSignedUrl(doc.arquivo_url, 'leads-docs')  // ← este path funciona
  return null
}

⚠️ Impacto real: Quando o arquivo está em Storage (path relativo em arquivo_url como org-id/leads/uuid/file.pdf), a resolução funciona via o último fallback (getSignedUrl(doc.arquivo_url, 'leads-docs')). Porém se o arquivo foi salvo com google_drive_id populado, o signed URL correto não seria gerado. Verificar se uploads recentes retornam google_drive_id ou apenas arquivo_url.

Proposta de correção
  • Adicionar google_drive_id ao SELECT em useLeadDocs e useRestricaoDocs
  • Verificar se processos_documentos tem a coluna google_drive_id no schema real
  • Se a coluna não existir no schema: remover a verificação de google_drive_id em DocumentoItem
