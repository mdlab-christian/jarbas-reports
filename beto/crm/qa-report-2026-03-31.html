P1-01 — Botão "Triagem" no drawer abre ModalDistribuirLead em vez de ModalTriagem
P1
CrmPage.tsx:129 ClienteDrawerHeader.tsx:202-217 UX Quebrado
Descrição
O botão "Triagem" (tooltip: "Aprovar para jurídico") na action bar do drawer, que aparece para leads fora da coluna triagem, deveria abrir ModalTriagem (checklist de aprovação) ou no mínimo mover o lead para triagem. Em vez disso, o routing do handlePanelOpenModal direciona o case 'triagem' para setDistribuirLeadId, abrindo ModalDistribuirLead.
Fluxo atual (incorreto)
ClienteDrawerHeader: onClick → onCrmAction('triagem', id)
  ↓
CrmPage.handlePanelOpenModal case 'triagem':
  setDistribuirLeadId(leadId)  ← ERRADO: abre ModalDistribuirLead

ModalDistribuirLead para lead NÃO em triagem:
  - Mostra restrições sem processo (se houver)
  - Ao clicar "Distribuir" → RPC retorna: "Lead não está na coluna triagem"
  - Usuário vê erro confuso sem entender o que aconteceu
Fluxo esperado
ClienteDrawerHeader: onClick → onCrmAction('triagem', id)
  ↓
CrmPage.handlePanelOpenModal case 'triagem':
  setTriagemLeadId(leadId)  ← CORRETO: abre ModalTriagem (checklist)
  OU
  mover diretamente para coluna 'triagem' via update
Correção proposta
Em CrmPage.tsx linha 129:
Mudar case 'triagem': setDistribuirLeadId(leadId)
Para case 'triagem': setTriagemLeadId(leadId)
Impacto
Médio — o fluxo alternativo "Distribuição" (botão Send) ainda funciona para não-triagem. O bug só afeta o botão "Triagem" específico no header do drawer. A RPC impede dano real (retorna erro se lead não está em triagem), mas o UX está incorreto.
P1-02 — ModalDistribuirLead mostra restrições sem filtrar processavel=true
P1
ModalDistribuirLead.tsx:42-53 Inconsistência UI/RPC
Descrição
O modal busca todas as restrições sem processo vinculado (processo_id=is.null AND deleted_at=is.null), mas a RPC só distribui restrições com processavel=true. Isso pode criar uma divergência: o modal mostra "Distribuir 3 restrições" mas a RPC distribui apenas 1 ou 2.
Evidência do banco
-- Modal query (atual):
SELECT * FROM restricoes
WHERE lead_id = $1 AND processo_id IS NULL AND deleted_at IS NULL
-- Resultado LEAD FIX3: 2 restrições mostradas

-- RPC distribuir_lead_completo (linha 77):
WHERE lead_id = p_lead_id
  AND processavel = true   ← filtro adicional
  AND processo_id IS NULL
  AND deleted_at IS NULL
-- Resultado LEAD FIX3: 2 processáveis (coincidiu neste caso)
Impacto
Baixo/médio — no banco atual as restrições não-processáveis são raras, mas pode confundir o usuário sobre quantas restrições serão de fato distribuídas.
Correção proposta
Adicionar .eq('processavel', true as any) na query do modal para alinhar com o comportamento da RPC.
