P0 BUG-001 — Função RPC duplicada: rpc_crm_aprovar_triagem
Confirmado via

Supabase Management API: SELECT proname, pg_get_function_arguments(oid) FROM pg_proc WHERE proname = 'rpc_crm_aprovar_triagem' — retornou 2 registros com OIDs distintos.

Assinaturas conflitantes
OID 531435 (novo): p_lead_id uuid, p_advogado_id uuid, p_restricoes_selecionadas text[], p_restricoes_optout text[], p_observacoes text
OID 63799 (antigo): p_lead_id uuid, p_organizacao_id uuid, p_user_id uuid, p_observacoes text, p_advogado_id uuid, p_restricoes_selecionadas jsonb, p_restricoes_optout jsonb, p_tipo_acao text, p_dados_extras jsonb
Impacto

ModalTriagem chama com p_restricoes_selecionadas: string[] (array de UUIDs em texto). O PostgreSQL resolve para a função com assinatura text[] (OID 531435). O conflito causa ambiguidade de overload — em certas condições o Postgres pode resolver para a função errada (jsonb overload), resultando em erro de tipo silencioso ou falha na conversão do lead.

Localização

src/components/crm/modals/ModalTriagem.tsx linha 195 — supabase.rpc('rpc_crm_aprovar_triagem', {...})

Causa raiz

Migration incompleta: a versão nova foi criada (text[]) mas a versão antiga (jsonb) não foi removida. Ambas coexistem no banco.

Solução

Executar: DROP FUNCTION IF EXISTS rpc_crm_aprovar_triagem(uuid, uuid, uuid, text, uuid, jsonb, jsonb, text, jsonb); — remove apenas o overload antigo (OID 63799), preservando o novo.

P1 BUG-002 — view_crm_colunas_contagem não filtra deleted_at IS NULL
Confirmado via

Supabase Management API: SELECT definition FROM pg_views WHERE viewname = 'view_crm_colunas_contagem'

Definição da view
WHERE ((deleted_at IS NULL) AND (arquivado = false))
GROUP BY coluna_crm, organizacao_id;
Bug

A view filtra corretamente arquivado = false e deleted_at IS NULL. Porém leads com coluna_crm = 'convertido' (arquivado=false no banco, pois convertido é estado diferente de arquivado) são contados e aparecem nas contagens de colunas do tab Pipeline.

O CrmPage usa essa view para calcular totalAtivos e exibe na badge do tab "Pipeline". O cálculo já filtra corretamente: contagens.filter(c => c.coluna_crm !== 'arquivados' && c.coluna_crm !== 'convertido'), então o badge está correto.

Impacto real

Reduzido — badge do Pipeline não conta convertidos. Mas a view ainda retorna a linha coluna_crm='convertido', que é lixo desnecessário. Leads confirmados no banco com coluna_crm='convertido' e arquivado=false aparecem nessa view.

Solução

Adicionar filtro na view: AND coluna_crm NOT IN ('convertido', 'arquivados') ou garantir que leads convertidos também tenham arquivado=true após triagem aprovada.

P1 BUG-003 — ModalDistribuirLead usa URL hardcoded em vez de callWebhook
Localização

src/components/crm/modals/ModalDistribuirLead.tsx linhas 22-24 e 98-113

Código problemático
const N8N_DIST_PETICAO_URL =
  'https://primary-production-e209.up.railway.app/webhook/mdflow/distribuicao/gerar_peticao'
// ...
await fetch(N8N_DIST_PETICAO_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'mdflow': 'MdL1501@' },
  body: JSON.stringify({...}),
})
Impacto

Viola P-05 (hardcode de credenciais) e P-11 (separação de responsabilidades). O header secret 'MdL1501@' está hardcoded no frontend — visível no bundle JS compilado. Além disso, se a URL do n8n mudar ou o secret rodar, TODOS os deploys em produção quebrarão sem atualização de código.

Solução

Substituir por callWebhook('mdflow/distribuicao/gerar_peticao', {...}) — lê URL e secret de VITE_N8N_WEBHOOK_URL e VITE_N8N_WEBHOOK_SECRET, já configurados em .env.

Nota

webhook_requests log mostra entrada recente com contexto: 'distribuicao.gerar_peticao', status: 'failed' (2026-04-02T14:30), sugerindo que o webhook n8n está com problemas ativos.

const leadsFiltered = (allLeads ?? []).filter((l) => {
