BUG-P0-001
Webhook documentos_pendentes/cobrar com falha total no n8n — "Cobrar Todos" não funciona
P0
Arquivo + Linha
hooks/useDocPendentesBulkActions.ts (linha 75 confirmarCobrarTodos, linha 127 bulkCobrar) · components/DocumentosPendentesPage.tsx (handleCobrarTodos)
Evidência de Teste
POST https://primary-production-e209.up.railway.app/webhook/mdflow/documentos_pendentes/cobrar Headers: mdflow: MdL1501@ Payload: { "organizacao_id": "55a0c7ba-...", "user_id": "test", "data": { "modo": "todos" } } Response: {"message": "Workflow execution failed"} # Testado com payload mínimo e completo — mesmo resultado
Causa Raiz

O workflow n8n documentos_pendentes/cobrar falha internamente — não é validação de payload (outros webhooks retornam VALIDATION_ERROR quando faltam campos). Possíveis causas: credencial MDZap/WhatsApp expirada, nó com referência quebrada, função JavaScript com erro de sintaxe.

Impacto

100% dos fluxos de cobrança via WhatsApp por lote não funcionam. Afeta: botão "Cobrar Todos" no header, bulk cobrar selecionados no BulkBar. O botão "Cobrar WA" (AlertDialogBulkDocuments) usa endpoint diferente e não foi testado via curl diretamente.

Solução

Acessar n8n → workflow documentos_pendentes/cobrar → verificar último nó com erro → checar credencial MDZap/WhatsApp → corrigir → testar com payload de exemplo.

BUG-P1-001
Importação duplicada de differenceInCalendarDays e parseISO em NovoDocumentoModal
P1
Arquivo + Linha
modals/NovoDocumentoModal.tsx — linha 7 (import original) e linha 34 (duplicata)
Snippet
// linha 7 — correto: import { format, addDays, differenceInCalendarDays, parseISO } from 'date-fns'; // linha 34 — DUPLICATA (deve ser removida): import { differenceInCalendarDays, parseISO } from 'date-fns';
Causa Raiz

Adição incremental de imports durante desenvolvimento sem verificar imports existentes. O bundler pode aceitar silenciosamente, mas TS strict mode pode rejeitar com "Duplicate identifier".

Impacto

Warning no build em modo padrão. Possível erro de compilação em TypeScript strict. Afeta o componente de criação/edição de pendências — o mais crítico da página.

Solução

Remover completamente a linha 34 — os imports já existem na linha 7 com todos os símbolos necessários.

BUG-P1-002
ModoSequencialSheet: link_url sobrescreve arquivo_url no payload do Supabase
P1
Arquivo + Linha
sheets/ModoSequencialSheet.tsx — linhas 226–228
Snippet do Bug
if (ri.arquivo_url) updatePayload.arquivo_url = ri.arquivo_url; // linha 226 if (ri.link_url) updatePayload.arquivo_url = ri.link_url; // linha 227 ← SOBRESCREVE!
Causa Raiz

Ambos os ramos de código atribuem para updatePayload.arquivo_url. Se o objeto do batch contém arquivo_url (caminho Storage) E link_url não vazio, o link_url sobrescreve o caminho do arquivo, tornando o arquivo inacessível.

Solução

Usar else-if: if (ri.arquivo_url) updatePayload.arquivo_url = ri.arquivo_url; else if (ri.link_url) updatePayload.arquivo_url = ri.link_url;

BUG-P1-003
ModoSequencialSheet atualiza Supabase diretamente (bypass do n8n) — violação de P-11
P1
Arquivo + Linhas
sheets/ModoSequencialSheet.tsx — linhas 221–235 (update direto) e linha 250 (fire-and-forget)
Snippet
// Update DIRETO no Supabase (bypassando n8n): const { error } = await supabase .from('documentos_pendentes') .update({ status: 'recebido', updated_at: ... }) .eq('id', ri.pendencia_id); // n8n recebe SOMENTE fire-and-forget (sem await, sem verificar resultado): callWebhook('documentos_pendentes/resolver_lote', { ... }).catch(() => {});
Causa Raiz

Princípio P-11: lógica de negócio no frontend via Supabase direto em vez de webhook. O status é forçado para 'recebido' sem passar pela lógica n8n (notificações, auditoria, side-effects).

Impacto

Side-effects do n8n (WhatsApp, notificações, webhook_requests log) são pulados. Auditoria de eventos_operacionais pode ficar incompleta. Status é alterado mesmo se o n8n falhar silenciosamente.

Solução

Refatorar: usar SOMENTE callWebhook('documentos_pendentes/resolver_lote', ...) com await e verificar o response.action. Remover o loop de update direto no Supabase.

BUG-P1-004
UX: Ambos os botões do accordion header ("Cobrar" e "Ver Cockpit") chamam a mesma função openCockpit

BUG-P1-005
ReabrirPendenciaModal: UI diz "Opcional" mas webhook exige motivo mínimo 10 chars
P1
Arquivo + Linha
modals/ReabrirPendenciaModal.tsx — linha 92 (hint "Opcional") vs webhook validation
Evidência do Webhook
POST /webhook/mdflow/documentos_pendentes/reabrir Response: { "success": false, "message": "Payload invalido: ... data.motivo obrigatorio (min 10 chars)", "error_code": "VALIDATION_ERROR" }
Causa Raiz

O componente React mostra "Opcional" no hint do campo motivo e não valida tamanho mínimo, mas o webhook n8n exige motivo com pelo menos 10 caracteres. O usuário pode submeter sem motivo e receber um erro genérico de "Serviço indisponível".

Solução

Opção A (recomendada): tornar o campo obrigatório no frontend com validação mínimo 10 chars e remover o hint "Opcional". Opção B: alterar o webhook n8n para aceitar motivo vazio. O campo de motivo existe por boa razão de auditoria — tornar obrigatório é preferível.

BUG-P2-002
Type mismatch: ProcessoGroup.numero_cnj tipado como string mas banco permite NULL
P2
Arquivo + Linha
types/documentos-pendentes.types.ts — linha 30 · Confirmado: 3 processos com numero_cnj=null no banco
Evidência
// types: export interface ProcessoGroup { processo_id: string; numero_cnj: string; // ← tipado como string, mas DB retorna null pendencias: DocumentoPendente[]; } // Banco: GET /processos?numero_cnj=is.null → 3 registros com numero_cnj=null
Causa Raiz

O tipo não reflete a realidade do banco. Em runtime não quebra porque PendenciaAccordion já protege com proc.numero_cnj ? ..., mas o TypeScript perde a capacidade de detectar null-unsafe usos.

Solução

Alterar tipo para numero_cnj: string | null. O TypeScript então forçará checks em todos os usos, garantindo segurança de tipo.

BUG-P2-004
ClienteCockpitSheet busca campo whatsapp em clientes mas maioria tem NULL — fallback para telefone OK mas sem tratamento de formato
P2
Arquivo + Linha
sheets/ClienteCockpitSheet.tsx — linha 106
Evidência
// Banco: clientes com whatsapp: {"telefone": "54999344570", "whatsapp": null} // 90% dos casos {"telefone": "48988745170", "whatsapp": "(48) 98874-5170"} // whatsapp formatado // Código: const whatsappPhone = (clienteContato?.whatsapp || clienteContato?.telefone || '') .replace(/\D/g, ''); // remove formatação → OK // URL gerada: https://wa.me/5554999344570 → correto // Mas se whatsapp = "(48) 98874-5170" → 5548988745170 → correto

Adicionar validação de comprimento: whatsappPhone.length >= 10 && whatsappPhone.length <= 13 antes de renderizar o link wa.me.

BUG-P2-005
ConfirmarResolverModal: atualização direta no Supabase antes de chamar webhook — inconsistência se webhook falhar
P2
Arquivo + Linhas
modals/ConfirmarResolverModal.tsx — linhas 106–127 (update Supabase) vs linha 130 (webhook)
Causa Raiz

Quando há arquivo/link: primeiro atualiza arquivo_url no Supabase diretamente, depois chama webhook de resolver. Se o webhook falhar (HTTP 404), o arquivo já foi salvo no banco mas o status não foi alterado para "resolvido". O documento fica com arquivo_url preenchido mas status "solicitado".

Solução

Passar arquivo_url no payload do webhook e deixar o n8n fazer o update atômico. Ou: chamar webhook primeiro, verificar sucesso, depois fazer update Storage. Ambas as abordagens evitam estado parcialmente inconsistente.
