Achados — Ordenados por Severidade

P0 — Críticos (bloqueiam em produção)

❌ BK-001: rpc_decisoes_criar_execucao insere etapa='fila' — CHECK constraint rejeita

A RPC rpc_decisoes_criar_execucao faz INSERT INTO execucoes (..., etapa, ...) VALUES (..., 'fila', ...).

Porém o CHECK constraint execucoes_etapa_check só aceita: cadastrada, ajuizada, intimacao, penhora, alvara_pedido, alvara_expedido, pago, encerrada.

Impacto: Criar execução via /decisoes FALHA com erro de constraint. Fluxo decisão→execução quebrado.

Evidência: Hoje só existem 325 rows com etapa='cadastrada' — provavelmente criadas pelo ModalNovaExecucao (que usa 'cadastrada'). A RPC nunca completou um INSERT.

-- FIX: Atualizar CHECK ou RPC
-- Opção A: adicionar 'fila' ao CHECK (se fizer sentido no fluxo)
ALTER TABLE execucoes DROP CONSTRAINT execucoes_etapa_check;
ALTER TABLE execucoes ADD CONSTRAINT execucoes_etapa_check
  CHECK (etapa IN ('fila', 'cadastrada', 'ajuizada', 'intimacao', 'penhora', 'alvara_pedido', 'alvara_expedido', 'pago', 'encerrada'));

-- Opção B: corrigir a RPC para usar 'cadastrada' (mais conservador)
-- Na RPC: trocar 'fila' por 'cadastrada'
❌ BK-002: execucoes_eventos tipo CHECK diverge completamente do frontend

CHECK no banco: CRIADA, ALVARA_SOLICITADO, ALVARA_RECEBIDO, DEPOSITO_REALIZADO, DEPOSITO_CONFIRMADO, HONORARIOS_PAGOS, CONCLUIDA, CANCELADA, REABERTA

Frontend (EventoExecucao): CRIACAO, DEPOSITO, ALVARA_PEDIDO, ALVARA_EXPEDIDO, LEVANTAMENTO, PAGAMENTO_FINAL, ARQUIVO, ETAPA_ALTERADA, SITUACAO_ALTERADA, AVANCO_ETAPA, OBSERVACAO

RPC fn_append_evento_execucao usa: DEPOSITO, LEVANTAMENTO, PAGAMENTO_FINAL

Intersecção: ZERO. Nenhum tipo do frontend existe no CHECK.

Impacto mitigado: Os eventos vão no JSONB historico da tabela execucoes (não na tabela execucoes_eventos). A tabela execucoes_eventos tem 0 linhas — está órfã. O CHECK está lá mas nunca é exercitado pelo fluxo atual.

  CHECK (tipo IN ('CRIACAO', 'DEPOSITO', 'ALVARA_PEDIDO', 'ALVARA_EXPEDIDO',
    'LEVANTAMENTO', 'PAGAMENTO_FINAL', 'ARQUIVO', 'ETAPA_ALTERADA',

P1 — Importantes

⚠️ BK-003: Valores monetários em NUMERIC(15,2) — deveria ser INT8 centavos

6 colunas monetárias (valor_execucao, valor_deposito, valor_levantado, valor_condenacao, valor_pago, valor_pendente) usam NUMERIC(15,2).

Padrão MdFlow: BIGINT em centavos. Frontend já usa formatBRLReais() que assume reais. Há uma nota no revisoes.md: "Valores monetarios em REAIS no banco (nao centavos INT8) — inconsistencia com padrao do projeto".

Impacto: NUMERIC(15,2) funciona corretamente para reais. A inconsistência é com o padrão, não um bug funcional. Migration de conversão seria de alto risco com 390 registros.

Recomendação: Manter NUMERIC(15,2) e documentar como exceção aceita. Não migrar agora.

⚠️ BK-004: View usa subqueries JSONB pesados em cada linha

view_execucoes_list calcula data_ultima_movimentacao, total_depositado, total_levantado, tem_deposito, tem_alvara_expedido, total_arquivos via jsonb_array_elements(historico) para CADA linha.

Com 390 registros: ~178ms médio. Com 5.000+: pode escalar para 500ms+.

Recomendação: Materializar campos calculados (total_depositado, total_levantado) via trigger no UPDATE, em vez de calcular na view. Ou usar materialized view com refresh periódico.

⚠️ BK-005: tabela execucoes_eventos está órfã (0 linhas)

A tabela existe, tem RLS, tem CHECK constraint, mas nunca recebeu dados. Todos os eventos vão como JSONB no campo historico da tabela execucoes.

Problema: Duas implementações paralelas de audit trail — uma funcional (JSONB), uma vazia (tabela). Nenhum código referencia a tabela execucoes_eventos diretamente.

Recomendação: Decidir: (A) migrar eventos do JSONB para a tabela (melhor para queries/relatórios) ou (B) deprecar a tabela (aceitar JSONB).

⚠️ BK-006: Campos denormalizados sem trigger de sync

As colunas cliente_nome, empresa_nome, advogado_nome_oab, processo_numero na tabela execucoes são cópias denormalizadas. Se o nome do cliente mudar em clientes, o campo na execução fica stale.

Impacto: Baixo — nomes raramente mudam. Mas não há trigger para manter consistência.

Recomendação: Criar trigger ON UPDATE clientes que atualiza execucoes.cliente_nome quando clientes.nome muda. Idem para empresa e advogado.

P2 — Tech Debt / Melhorias

💡 BK-007: execucoes_eventos e execucoes_arquivos sem trigger updated_at

Padrão MdFlow: toda tabela com updated_at precisa do trigger set_updated_at. Essas tabelas não têm (e execucoes_eventos nem tem coluna updated_at — é append-only, OK). execucoes_arquivos também não tem updated_at — aceitável se é imutável.

💡 BK-008: Falta índice de busca textual (unaccent) para busca no frontend

O frontend faz busca em view_execucoes_list por cliente_nome, empresa_nome, processo_numero. A view faz o filtro, mas sem índice funcional unaccent(lower(...)) na tabela base.

Com 390 registros não impacta. Com 5.000+, sequential scan será lento.

CREATE INDEX idx_execucoes_cliente_nome_search ON execucoes (unaccent(lower(cliente_nome)));
CREATE INDEX idx_execucoes_empresa_nome_search ON execucoes (unaccent(lower(empresa_nome)));
CREATE INDEX idx_execucoes_processo_numero ON execucoes (processo_numero);
💡 BK-009: fn_sync_execucao_para_financeiro não valida organizacao_id

A função busca a execução sem filtrar por organizacao_id: WHERE e.id = p_execucao_id. Em teoria um service_role poderia sync uma execução de outra org. Na prática, só é chamada via trigger (que já filtra a execução correta). Risco baixo mas viola princípio de defesa em profundidade.

O QUE ESTÁ BEM

✅ RLS excelente: Todas 6 tabelas com RLS habilitado. Policies corretas. DELETE requer admin. Isolamento confirmado em testes.

✅ RPCs bem implementadas: Todas com SECURITY DEFINER + SET search_path. fn_append_evento_execucao é particularmente boa — 2 overloads, validação JWT, atualização atômica de valor_pago.

✅ Índices completos: 16 índices incluindo parciais com deleted_at IS NULL, composto etapa+pago, GIN para JSONB. Cobertura acima da média.

✅ Triggers financeiros: Sync automático para tabela financeiro quando etapa chega em alvara_expedido. Error handling com RAISE WARNING (não bloqueia).

✅ Webhooks n8n: Auth validada, envelope canônico com error_code, validação de payload.

✅ CHECK constraints: Todos os campos status/tipo usam TEXT + CHECK (não ENUM nativo). Correto.

DIAGNÓSTICO GERAL

Backend sólido no geral — RLS impecável, RPCs bem estruturadas, índices completos. O problema mais grave é o BK-001: a RPC de criação via decisões está fundamentalmente quebrada por um valor de etapa que não existe no CHECK constraint. Segundo problema grave é a confusão arquitetural entre JSONB historico (usado) e tabela execucoes_eventos (morta) — precisa de decisão do CTO. Os valores monetários em NUMERIC(15,2) divergem do padrão INT8 mas funcionam corretamente; não recomendo migrar agora. Performance aceitável para 390 registros mas a view JSONB vai sofrer com escala. Nota: 75% — puxada para baixo pelos 2 P0.
