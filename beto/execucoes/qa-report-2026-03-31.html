2. Bugs Encontrados

P0 — Bloqueantes

BUG-001: 173 execucoes "parcial" invisiveis nas tabs de lista

P0

Arquivo: useExecucoesList.ts linhas 36-44 + ExecucoesTabs.tsx

Descricao: Das 325 execucoes ativas, 173 tem tipo_decisao='parcial'. As tabs filtram: Favoraveis (favoravel+!pago), Desfavoraveis (desfavoravel+!pago), Ag. Alvara (etapa=alvara_pedido), Pagos (pago=true). Nao existe tab que mostre tipo "parcial" — 53% dos registros sao inacessiveis pela UI de lista.

36if (tab === 'favoraveis') { 37 q = q.eq('tipo_decisao', 'favoravel').eq('pago', false); 38} else if (tab === 'desfavoraveis') { 39 q = q.eq('tipo_decisao', 'desfavoravel').eq('pago', false); 40}

Dados Supabase: favoravel=152, parcial=173, desfavoravel=0, total=325

Impacto: Mais da metade dos registros nao aparecem em nenhuma tab de listagem. Usuario so ve no Dashboard KPI.

Solucao proposta: (A) Adicionar tab "Parciais" com filtro tipo_decisao='parcial' + badge count. (B) Ou incluir "parcial" na tab "Favoraveis" renomeando para "Favoraveis / Parciais".

BUG-002: useExecucaoDetail NAO filtra por organizacao_id

P0

Arquivo: hooks/useExecucaoDetail.ts linhas 9-15

Descricao: O hook que carrega o detalhe de uma execucao (usado pelo ModalExecucao) faz query na view_execucoes_list filtrando apenas por id e deleted_at, sem incluir organizacao_id. Isso depende inteiramente do RLS para isolamento multi-tenant.

9const { data, error } = await supabase 10 .from('view_execucoes_list') 11 .select('*') 12 .eq('id', execucaoId) 13 .is('deleted_at', null) 14 // MISSING: .eq('organizacao_id', organizacao_id) 15 .maybeSingle();

Impacto: Se RLS falhar ou for desabilitado, qualquer usuario poderia ver execucoes de outra organizacao via deep link /execucoes?id=UUID. Viola P-04 (Multi-tenant Absoluto).

Solucao: Adicionar .eq('organizacao_id', organizacao_id) e receber organizacao_id via useAuth(). Padrão defensivo ja usado nos demais hooks.

BUG-003: Exclusao nao implementada — stub "em desenvolvimento"

P0

Arquivo: ExecucoesTable.tsx linhas 321-325

Descricao: O botao "Excluir" no dropdown de acoes exibe apenas um toast informativo: toast.info('Exclusao sera implementada no UPDATE de Automacoes.'). Nao existe fluxo de exclusao funcional.

321<DropdownMenuItem 322 className="text-destructive focus:text-destructive" 323 onClick={() => toast.info('Exclusao sera implementada no UPDATE de Automacoes.')} 324> 325 <Trash2 className="mr-2 h-4 w-4" /> Excluir

Impacto: Checklist itens G1-G3 todos N/A. Funcionalidade ausente impede o usuario de remover execucoes incorretas.

Solucao: Implementar webhook execucoes/delete com confirmacao + soft delete (set deleted_at). Ou conectar a execucoes/update com action: 'soft_delete'.

P1 — Importantes

BUG-004: DrawerPagamentoFinal vaza valor financeiro sem cegueira

P1

Arquivo: DrawerPagamentoFinal.tsx linhas 54, 197-199

Descricao: A "Diferenca" (totalLevantado - valorCondenacao) e exibida via formatBRLReais(Math.abs(diferenca)) sem verificar acesso_financeiro. O componente BlindedVal e usado nos outros campos, mas a Diferenca e renderizada diretamente.

54const diferenca = (totalLevantado ?? 0) - (valorCondenacao ?? 0); ... 197<span className={`font-mono ${diferenca >= 0 ? 'text-primary' : 'text-destructive'}`}> 198 {formatBRLReais(Math.abs(diferenca))} 199 {diferenca < 0 ? ' (deficit)' : ''}

Impacto: Usuario sem acesso_financeiro ve o valor exato da diferenca mesmo quando os campos individuais mostram "*****". Viola cegueira financeira.

Solucao: Envolver a linha da Diferenca com BlindedVal ou adicionar condicao acesso_financeiro.

BUG-005: useExecucaoModalHandlers envia envelope inconsistente no avancar etapa

P1

Arquivo: useExecucaoModalHandlers.ts linhas 65-76

Descricao: O webhook execucoes/evento chamado por handleAvancarEtapa usa user_id em vez de actor_user_id no envelope. Todos os outros webhooks da pagina usam actor_user_id conforme o envelope canonico.

65const res = await callWebhook('execucoes/evento', { 66 request_id: crypto.randomUUID(), 67 contexto: 'execucoes.evento', 68 organizacao_id, 69 user_id: user?.user_id, // DEVERIA SER actor_user_id 70 origem_rota: '/execucoes',

Impacto: O n8n pode nao identificar o ator corretamente no audit trail. Se o workflow valida actor_user_id, o evento pode falhar silenciosamente.

Solucao: Trocar user_id por actor_user_id.

BUG-006: Tab "Ag. Alvara" nao inclui etapa alvara_expedido

P1

Arquivo: useExecucoesList.ts linha 42 + useExecucoesTabCounts.ts linha 23

Descricao: A tab "Ag. Alvara" filtra apenas etapa='alvara_pedido'. Nao inclui alvara_expedido. Execucoes onde o alvara ja foi expedido mas o pagamento ainda nao ocorreu ficam ocultas nesta tab — o usuario esperaria ver ambas as etapas nesta aba de "aguardando".

41} else if (tab === 'aguardando_alvara') { 42 q = q.eq('etapa', 'alvara_pedido'); 43 // Deveria tambem incluir 'alvara_expedido'?

Impacto: Execucoes com alvara expedido mas nao pagas nao aparecem nesta tab. Potencial perda de rastreabilidade.

Solucao: Usar .in('etapa', ['alvara_pedido', 'alvara_expedido']) ou renomear a tab para refletir apenas "Pedido".

BUG-007: ModalExportarExecucoes usa user?.acesso_financeiro em vez de useAuth()

P1

Arquivo: ModalExportarExecucoes.tsx linha 60

Descricao: O flag de cegueira financeira e obtido via user?.acesso_financeiro em vez do acesso_financeiro direto do useAuth(). Se o shape do user object mudar, a exportacao pode vazar dados financeiros.

59const { organizacao_id, user } = useAuth(); 60const temAcessoFinanceiro = user?.acesso_financeiro ?? false;

Impacto: Inconsistencia com os demais componentes que usam const { acesso_financeiro } = useAuth(). Risco baixo mas viola padrao.

Solucao: Usar const { acesso_financeiro } = useAuth() diretamente.

BUG-008: Inconsistencia no formato CPF/CNPJ no banco

P1

Tabela: view_execucoes_list.cliente_cpf_cnpj

Descricao: Alguns CPFs estao armazenados com mascara (381.398.078-26) e outros sem (41116730820). Isso pode causar falhas na busca/filtro e exibicao inconsistente no CopyableCPF.

Dados: Dos 10 registros amostrados: 5 com mascara, 5 sem mascara.

Impacto: Busca por CPF pode nao encontrar registros se o usuario digitar num formato diferente do armazenado.

Solucao: Normalizar CPFs na base para formato sem mascara (somente digitos). O frontend ja deve aplicar mascara na exibicao.

BUG-011: AVANCO_ETAPA nao e tipo valido em EventoExecucao

P2

Arquivo: useExecucaoModalHandlers.ts linha 74 vs execucoes.types.ts linhas 17-27

Descricao: O handler envia tipo: 'AVANCO_ETAPA' mas o type EventoExecucao define tipo: 'ETAPA_ALTERADA'. Tambem nao existe 'AVANCO_ETAPA' no eventoIconMap de execucao.shared.tsx.

74data: { 75 tipo: 'AVANCO_ETAPA', // Type nao existe em EventoExecucao.tipo 76 descricao: `Etapa avancada para: ${etapaMap[proximaEtapa]?.label ?? proximaEtapa}`,

Impacto: Se o n8n/backend salvar com este tipo, o icone no historico sera MessageSquare (fallback). Nao bloqueia mas gera icone incorreto.

Solucao: Usar tipo: 'ETAPA_ALTERADA' ou adicionar 'AVANCO_ETAPA' ao type + eventoIconMap.

BUG-012: resetTabFilters definido mas nunca chamado

P2

Arquivo: useExecucoesFilterParams.ts linhas 81-91

Descricao: A funcao resetTabFilters e definida e exportada pelo hook, mas nao e utilizada em nenhum componente. Quando o usuario troca de tab, os filtros antigos podem permanecer ativos e gerar confusao.

Impacto: Filtros de uma tab podem interferir em outra. Ex: usuario filtra etapa "penhora" na tab Favoraveis, depois vai para Desfavoraveis — o filtro persiste.

Solucao: Chamar resetTabFilters no handleTabChange de ExecucoesPage.tsx.
