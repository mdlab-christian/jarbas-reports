P0 Spec oficial da página ausente — risco de drift silencioso
O arquivo ~/beta-mdflow/docs/_resumido___homepage.txt não existe. A documentação de referência atual é apenas a revisão manual em docs/revisoes/homepage.md, que é produzida pelo QA e não pelo CTO. Sem spec oficial, toda auditoria futura depende de engenharia reversa do código. Qualquer divergência entre o que o CTO espera e o que está implementado não pode ser detectada automaticamente.
Ação: Criar _resumido___homepage.txt com as seções padrão (views por perfil, RPCs, KPIs esperados, regras de negócio).
P1 get_homepage_pessoal não trata perfil 'operacional' — KPIs retornam vazio
A RPC possui branches para 'juridico', 'admin' e 'comercial'. O perfil 'operacional' (existente no banco — 1 usuário confirmado) cai no ELSE implícito e retorna kpis: {}. O frontend roteia corretamente operacional → ViewJuridico (linha 225 de index.tsx), mas o hook passa user.perfil diretamente ao RPC sem normalização, portanto o usuário operacional vê uma homepage com todos os KPIs zerados.

Teste confirmatório: get_homepage_pessoal('66f80311...', org, 'operacional', '7d') → kpis: {}
Fix opção A (RPC): adicionar ELSIF p_perfil = 'operacional' THEN usando mesmo bloco do jurídico.
Fix opção B (hook): normalizar no frontend — p_perfil: user.perfil === 'operacional' ? 'juridico' : user.perfil.
Recomendado: opção A (fix no backend, mais robusto).
P1 get_homepage_gestor_juridico usa prazo_fatal ao invés de data_vencimento para atividades atrasadas
O campo atividades_atrasadas na RPC verifica atividades.prazo_fatal < now(). No entanto, o fluxo de atividades da plataforma utiliza data_vencimento como referência primária de prazo para o usuário (confirmado no hook useHomepagePessoal que conta prazos_abertos_7d via data_vencimento). prazo_fatal é um campo complementar para prazos legais fatais — não é preenchido em todas as atividades.

Impacto: o contador de "atividades atrasadas" no bloco executivo gestor pode subestimar significativamente o real backlog da equipe. Valor atual retornado: 5. Valor real usando data_vencimento pode ser muito maior.
Fix: alterar get_homepage_gestor_juridico para usar data_vencimento < now() em vez de prazo_fatal < now() no sub-select de atividades_atrasadas.
P2 avisos_escritorio_lidos sem índice isolado em user_id
O hook useHomepageAvisos consulta avisos_escritorio_lidos filtrando por user_id e organizacao_id. Existe um índice composto idx_avisos_lidos_unique(aviso_id, user_id) e outro em organizacao_id, mas nenhum índice com user_id como coluna líder. Para organizações com muitos usuários e avisos, esta query pode crescer linearmente.
Fix: CREATE INDEX idx_avisos_lidos_user ON avisos_escritorio_lidos(user_id, organizacao_id);
