Bugs Encontrados

P2 BUG-HP-L01 Header mobile overflow residual a 375px

A 375px de viewport, o header transborda 17px para fora da tela (documentScrollWidth=401px). O bug N01 adicionou overflow-x:hidden no body que mascara visualmente o problema, mas a causa raiz persiste: o logo mobile (h-10 w-auto) tem aspect ratio 4:1, renderizando com 160px de largura. Com a zona esquerda (36px menu + 8px gap + 160px logo = 204px) e a zona direita (4 botões visíveis = ~160px), o total ultrapassa os 375px disponíveis.


Medições:
  • Logo mobile: h-10 = 40px alto × aspect ratio 4:1 = 160px de largura
  • Zona esquerda: 36(menu) + 8(gap) + 160(logo) = 204px
  • Zona direita: Buscar(36) + OlivIA(36) + Notif(36) + GP(40) + 3×gap(12) = 160px
  • Total (sem spacer): 204 + 160 = 364px + padding px-3 (12px) = 376px > 375px
Fix sugerido: No app_header.tsx linha 265, trocar h-10 w-auto md:hidden por h-8 w-auto max-w-[120px] md:hidden. Alternativamente, reduzir o padding do header para px-2 em mobile ou esconder o logo completamente em <375px.
P2 BUG-HP-E01 Taxa conversão exibida sem "%" no GestorExecutiveBlock

O componente HomepageGestorExecutiveBlock formata todos os valores como .toLocaleString('pt-BR') sem distinção de tipo. A chave taxa_conversao representa uma porcentagem (0–100) mas é exibida como número puro:


Comportamento atual:
  • 7 dias → mostra 75 (deveria ser 75%)
  • 30 dias → mostra 69,2 (deveria ser 69,2%)
  • hoje → mostra 0 (deveria ser 0%)

Causa: Linha 53 do componente usa val.toLocaleString('pt-BR') sem verificar se o campo é percentual. O componente não tem schema de tipos por campo.
Fix sugerido: No homepage_gestor_executive_block.tsx, adicionar um Set de chaves percentuais e formatá-las com sufixo %:
const PERCENT_KEYS = new Set(['taxa_conversao', 'taxa_sucesso']); // na renderização: const formatted = PERCENT_KEYS.has(key) ? `${val.toLocaleString('pt-BR', {maximumFractionDigits:1})}%` : val.toLocaleString('pt-BR');
INFO BUG-HP-D01 Ranking todos com score 0 — problema de dados, não de código

O RankingCard mostra 10 usuários mas todos com score "0". A RPC get_homepage_ranking calcula o score corretamente via JOIN processos ON processos.advogado_id = users.id, porém os processos no banco de desenvolvimento usam UUIDs de seed (a1000001-0000-4000-8000-000000000001) que não correspondem a nenhum UUID real de usuário.


Verificação: Query direta confirma 7.487 processos com advogado_id IS NOT NULL, mas o top advogado_id é a1000001-0000-4000-8000-000000000001 (fictício). Quando os processos reais forem importados com advogado_id correto, o ranking funcionará.
Ação recomendada: Nenhuma correção de código necessária. Para testar o ranking em dev, rodar um seed que vincule alguns processos aos users.id reais da organização Midas.
