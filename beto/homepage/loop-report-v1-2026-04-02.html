Narrativa por Fase

FASE 1

Funcional — O que o QA revelou

O QA encontrou dois P0 bloqueantes de regressão: as migrations 20260331100001 e 20260331100002 foram criadas em 31/03 mas nunca aplicadas ao Supabase. Resultado: get_homepage_ranking falhava com erro 42703 (coluna inexistente) e taxa_conversao retornava 200%+. Ambas aplicadas via Management API. Adicionalmente, BUG-HP-N01 corrigiu overflow-x em mobile com overflow-x: hidden no body, e BUG-HP-N02 adicionou KPI_LABELS com traduções semânticas em PT-BR.

Lição aprendida: migrations devem ser aplicadas via supabase db push imediatamente após o commit.

FASE 2

Profundidade — O que o deep-explorer descobriu

O deep-explorer identificou dois achados funcionais e responsivos não cobertos pelo QA: (1) taxa_conversao sem sufixo "%" no GestorExecutiveBlock — adicionado PERCENT_KEYS Set com lógica condicional. (2) Logo mobile header overflow (causa raiz) — o fix N01 de overflow-x:hidden era safety net; a causa real era o logo com h-10 w-auto excedendo 375px → corrigido para h-8 max-w-[120px] w-auto.

FASE 4

Cobertura Total — Frontend 90%, Backend 72%→90%

Front-tester: 18/20 (90%) — zero falhas. 2 N/A por design (charts e hover só em ViewJuridico/ViewComercial, não ViewAdmin). Achado colateral: /usuario com "useBlocker must be used within a data router" — fora do escopo desta sprint.

Backend-tester + executor: saúde inicial 72% → 90% após 3 migrations. Fix principal: get_homepage_gestor_juridico estava usando prazo_fatal (nullable, semanticamente errado) em vez de data_vencimento para atividades_atrasadas. Também: branch para perfil operacional e índice user_id em avisos_escritorio_lidos.

FASE 5

Palavra Final — Bug sistêmico descoberto e resolvido

Descoberta crítica: processos.advogado_id é FK para advogados.id (UUIDs tipo a1000001-...), mas 3 RPCs/componentes faziam JOIN/filtro diretamente com users.id — campos diferentes. Resultado: ranking com score=0 para todos, processos_ativos=0, audiências nunca visíveis para o usuário correto.

Fixes: get_homepage_ranking reescrito com JOIN users → advogados(user_id) → processos(advogado_id). get_homepage_pessoal usa subquery advogados.id WHERE user_id = p_user_id. HomepageAudienciasDia corrigido no hook para buscar advogados.id antes de filtrar audiencias.

Resultado após fix: scores reais no ranking (6454, 1033), processos_ativos=8595, audiências visíveis para o advogado correto. Nota: 6.4 → 8.5/10.
