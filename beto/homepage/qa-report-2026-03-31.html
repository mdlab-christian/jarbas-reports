Bugs P0 - Criticos
P0 BUG-HP-01: RPC get_homepage_ranking quebrada -- coluna inexistente
RPC PostgreSQL • Impacto: Ranking + Comercial + Juridico (cascata)
Evidencia:
HTTP 400 ao chamar get_homepage_ranking:
{
  "code": "42703",
  "hint": "Perhaps you meant to reference the column \"a.responsavel_id\".",
  "message": "column p.responsavel_id does not exist"
}
Causa raiz: A RPC get_homepage_ranking referencia p.responsavel_id na tabela processos, mas essa coluna nao existe (provavelmente foi renomeada ou movida para atividades). O hint sugere que a.responsavel_id (alias "a" para outra tabela) seria o correto.
Impacto: Na view Admin E Gestor, o hook useHomepageGestor faz Promise.all das 3 RPCs. Quando get_homepage_ranking falha com erro 42703, o codigo faz throw err e toda a query falha. Isso causa:
  • Bloco "Comercial" mostra "Sem dados disponiveis" (dados existem mas sao descartados)
  • Bloco "Juridico" mostra "Sem dados disponiveis" (idem -- 7.488 processos ativos ignorados)
  • Bloco "Ranking" mostra "Sem dados de ranking"
  • Toast de erro: "Erro ao carregar dados da homepage: column p.responsavel_id does not exist"
Solucao:
-- Opcao A (corrigir a RPC): trocar p.responsavel_id pela coluna correta
-- Opcao B (isolar falha no hook): tratar ranking error separadamente
-- Recomendado: A + B (corrigir RPC e tornar ranking independente)
P0 BUG-HP-02: Erro cascata no useHomepageGestor -- uma RPC falha e derruba todas
hooks/useHomepageGestor.ts:L38-42 • Impacto: Dashboard gestor/admin completamente vazio
Codigo atual:
// useHomepageGestor.ts linhas 38-42
for (const err of errors) {
  if (err) throw err;  // <-- qualquer erro em 1 das 3 RPCs mata tudo
}
Causa raiz: As 3 RPCs (comercial, juridico, ranking) sao executadas em Promise.all mas o error handling nao e independente. Se ranking falha (e esta falhando -- BUG-HP-01), os dados de comercial e juridico que retornaram com sucesso sao descartados.
Impacto: O dashboard Admin e Gestor mostra todos os blocos vazios, mesmo que 2 das 3 RPCs tenham retornado dados validos. Isso afeta TODOS os usuarios com acesso_gestor=true ou perfil=admin.
Solucao:
// Tratar cada RPC independentemente
return {
  comercial: comercialRes.error ? {} : (comercialRes.data ?? {}),
  juridico: juridicoRes.error ? {} : (juridicoRes.data ?? {}),
  ranking: rankingRes.error ? [] : (rankingRes.data ?? []),
};
Bugs P1 - Importantes
P1 BUG-HP-03: staleTime 10x acima do padrao em todos os hooks
useHomepagePessoal.ts:L28, useHomepageGestor.ts:L47, useHomepageAvisos.ts:L50
Codigo:
staleTime: 5 * 60 * 1000,  // 300.000ms = 5 minutos
Spec pede:
staleTime: 30_000  // 30 segundos (CLAUDE.md seção 10)
Causa raiz: Os hooks foram implementados com staleTime de 5 minutos (possivelmente intencional para performance), mas o padrao do projeto define 30_000ms como mandatorio para todos os hooks React Query.
Impacto: Dados potencialmente desatualizados por ate 5 minutos. Em um escritorio com ~80-100 distribuicoes/dia, KPIs podem ficar significativamente defasados.
Solucao: Alterar para staleTime: 30_000 nos 3 hooks.
P1 BUG-HP-04: taxa_conversao retorna 185.7% -- valor absurdo
RPC get_homepage_gestor_comercial • Impacto: KPI comercial exibe dado incorreto
Evidencia (curl service_role, periodo=7d):
{
  "conversoes": 15,
  "leads_novos": 20,
  "taxa_conversao": 185.7,  // <-- percentual > 100%?
  "leads_em_atendimento": 4,
  ...
}
Causa raiz: A RPC provavelmente calcula conversoes / leads_novos * 100 mas pode estar dividindo por um subset ou usando um periodo diferente no denominador. 15 conversoes / 20 leads = 75%, nao 185.7%. Possivel bug na logica da RPC.
Impacto: No perfil comercial ou gestor, o card "Taxa de conversao" mostraria 186% com o format AnimatedNumber + %, um dado impossivel que compromete confianca no dashboard.
Solucao: Auditar a RPC get_homepage_gestor_comercial e corrigir o calculo de taxa_conversao. Adicionar cap de 100% no frontend como defesa.
P1 BUG-HP-05: Perfil "operacional" cai no fallback sem homepage
pages/homepage/index.tsx:L223-227 • Impacto: Usuario Mateus (MDZap Teste)
Gate logic:
if (user.acesso_gestor && user.perfil === 'admin') return <ViewAdmin />;
if (user.acesso_gestor) return <ViewGestor />;
if (user.perfil === 'juridico') return <ViewJuridico />;
if (user.perfil === 'comercial') return <ViewComercial />;
return <HomepagePerfilFallback />;
Causa raiz: O perfil operacional nao tem view dedicada. O usuario "Mateus (MDZap Teste)" com perfil=operacional vera apenas a tela "Perfil nao reconhecido" com instrucao para contatar o administrador.
Impacto: Baixo atualmente (1 usuario de teste), mas qualquer usuario com perfil nao previsto fica sem homepage funcional. Nao ha view para perfis: operacional, assinante, partner.
Solucao: Decidir se operacional deveria ver ViewJuridico, ViewComercial, ou uma view simplificada. Minimo: adicionar if (user.perfil === 'operacional') return <ViewJuridico />;

P2 BUG-HP-06: HomepageAudienciasDia nao filtra por user_id
components/homepage/homepage_audiencias_dia.tsx:L63
Codigo:
.eq('organizacao_id', user!.organizacao_id)
.eq('status', 'agendada')
// Falta: .eq('responsavel_id', user.user_id) ou similar
Causa raiz: A query busca TODAS as audiencias agendadas da organizacao, nao apenas as do usuario logado. Em um escritorio com 5+ advogados, cada um veria audiencias de todos os colegas na sua homepage pessoal.
Impacto: Atualmente sem audiencias no banco para manifestar. Quando houver, usuarios verao audiencias alheias.
Solucao: Filtrar por usuario responsavel: .eq('responsavel_id', user.user_id) ou via join com processos.advogado_id.
P2 BUG-HP-07: HomepageKpiGrid coluna dinamica nao cobre 6-7 cards
components/homepage/homepage_kpi_grid.tsx:L600-606
Codigo:
const lgColsMap: Record<number, string> = {
  2: 'lg:grid-cols-2',
  3: 'lg:grid-cols-3',
  4: 'lg:grid-cols-4',
  5: 'lg:grid-cols-5',
};
// Sem 6 nem 7 -- buildKpisJuridico retorna 6 cards, buildKpisComercial retorna 7
Causa raiz: O mapa de colunas vai ate 5, mas os builders criam 6 (juridico) e 7 (comercial) cards. O fallback lg:grid-cols-4 e aplicado, resultando em layout nao-otimo para 6 e 7 itens.
Impacto: Visual: 6 cards em grid de 4 = 2 colunas na ultima linha (OK mas nao ideal). 7 cards em grid de 4 = linha de 3 + linha de 4 desbalanceada.
Solucao: Adicionar 6: 'lg:grid-cols-3' e 7: 'lg:grid-cols-4' ao mapa, ou usar columns prop explicitamente.
P2 BUG-HP-08: AnimatedNumber exibe 0 para valores string
components/homepage/homepage_kpi_card.tsx:L514-543
Codigo:
// KpiCardData.value e number | string
const numericValue = typeof value === 'number' ? value : parseFloat(value) || 0;
// AnimatedNumber so aceita number -- se value for string nao-numerica, mostra 0
Causa raiz: O tipo KpiCardData.value aceita string, mas AnimatedNumber so anima numeros. Valores string passam por parseFloat que retorna NaN para texto, convertido a 0.
Impacto: Baixo atualmente (todos os builders passam numeros), mas o tipo permite string e nao ha tratamento visual para isso.
Solucao: Ou remover string do tipo KpiCardData.value, ou renderizar strings diretamente sem animacao.
