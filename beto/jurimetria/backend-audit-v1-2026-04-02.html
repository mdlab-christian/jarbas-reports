🚨 ACHADOS DETALHADOS — P0 / P1 / P2

P0 — Bloqueante (corrigir imediatamente)

P0 RLS ausente em 4 tabelas jurimetria — dados de multi-tenants expostos
As tabelas jurimetria_acordos, jurimetria_empresas, jurimetria_juizes e jurimetria_escritorios_reu têm RLS desabilitado. Qualquer usuário autenticado com ANON_KEY pode acessar todos os dados de todas as organizações via REST API direta.

Evidência: jurimetria_empresas contém 325 rows de 2 orgs distintas. jurimetria_juizes contém 486 rows multi-org. Embora estas tabelas sejam legado e possivelmente não usadas ativamente pelo frontend da página, elas estão no schema público acessível.

Fix: ALTER TABLE jurimetria_empresas ENABLE ROW LEVEL SECURITY; CREATE POLICY "tenant_isolation" ON jurimetria_empresas USING (organizacao_id = public.get_org_id()); (repetir para as 4 tabelas)
P0 julgadores.grau = NULL em 760/761 registros — filtro por grau completamente inoperante
O campo grau na tabela julgadores é NULL em 99,9% dos registros (760/761). A RPC get_jurimetria_ranking_julgadores aceita parâmetro p_grau e as tabs "Primeiro Grau" / "Segundo Grau" enviam grau=1 ou grau=2 respectivamente. Com todos os julgadores sem grau definido, essas tabs retornam zero resultados — funcionalidade completamente quebrada.

Evidência: search_julgadores retornou julgador com "grau": null. Ranking RPC retorna julgadores com "grau": null.

Fix: Backfill de dados — inferir grau a partir do nome (vara → grau 1, câmara/turma → grau 2). Ou criar migration de grau DEFAULT 1 e trigger que classifica pelo tipo.

P1 — Importante (corrigir neste ciclo)

P1 upsert_julgador duplicado — 2 assinaturas conflitantes no banco
Existem 2 versões da RPC upsert_julgador no banco (OIDs 37372 e 44273). A diferença é que a versão nova (44273) aceita o parâmetro p_ativo boolean DEFAULT true. O frontend usa a chamada com p_ativo, mas se o Supabase resolver para a versão antiga, o campo ativo nunca será persistido.

Fix: Remover a versão antiga: DROP FUNCTION upsert_julgador(uuid, text, text, integer, uuid, uuid, uuid, uuid, uuid);
P1 217 comarcas_varas com comarca ou vara NULL — BUG-CV-01
21% dos registros em comarcas_varas (217/1028) têm comarca ou vara como NULL. Isso causa exibição de "null" ou campos vazios no modal de cadastro de julgador e no filtro de localidade.

Fix: Adicionar constraint CHECK (comarca IS NOT NULL AND vara IS NOT NULL) em novos registros + cleanup dos existentes. Ou alternativamente: adicionar NOT NULL constraint e fazer backfill com placeholder.
P1 Webhook jurimetria/confirmar_import sem workflow n8n correspondente
O modal ModalImport/StepConfirmacao.tsx chama callWebhook('jurimetria/confirmar_import', ...), mas nenhum workflow n8n com esse path foi encontrado nos 100 workflows ativos. A funcionalidade de importação de dados via planilha está completamente quebrada — o step de confirmação vai falhar com erro de rede ou timeout.

Fix: Criar workflow n8n WH-JURIMETRIA — Confirmar Import com webhook path jurimetria/confirmar_import, processamento do arquivo CSV/XLSX do Storage Supabase e retorno com envelope canônico.
P1 julgadores sem soft delete (deleted_at ausente) — hard delete impossível com FKs
A tabela julgadores não tem coluna deleted_at, violando o padrão MdFlow de soft delete obrigatório. Qualquer operação de exclusão de julgador faria hard delete — dados de ranking histórico seriam perdidos.

Igualmente ausente em comarcas_varas.

Fix: ALTER TABLE julgadores ADD COLUMN deleted_at TIMESTAMPTZ; ALTER TABLE comarcas_varas ADD COLUMN deleted_at TIMESTAMPTZ; + atualizar RLS policies para filtrar WHERE deleted_at IS NULL.

P2 — Melhorias (próximo ciclo)

P2 761 julgadores sem tribunal_id — vínculo com catálogo global inexistente
100% dos julgadores cadastrados não têm tribunal_id definido. O filtro por tribunal em SubtabPrimeiroGrau e SubtabSegundoGrau está inoperante.

Fix: Backfill via trigger ou job n8n que infere tribunal a partir da UF do julgador ou da comarca vinculada.
P2 comarcas_varas.organizacao_id é NULLABLE no schema (mas todos preenchidos)
A coluna organizacao_id em comarcas_varas é nullable no schema, embora todos os 1028 registros tenham o campo preenchido. Sem constraint NOT NULL, inserts futuros podem criar registros sem tenant, causando vazamento de dados.

Fix: ALTER TABLE comarcas_varas ALTER COLUMN organizacao_id SET NOT NULL;
P2 Tabelas legado jurimetria_* sem índice em organizacao_id
As tabelas jurimetria_empresas, jurimetria_juizes e jurimetria_acordos não têm índice em organizacao_id. Se forem usadas ativamente (snapshot/import), queries sem índice serão seq scan.

Fix: CREATE INDEX IF NOT EXISTS idx_jurimetria_empresas_org ON jurimetria_empresas(organizacao_id); (repetir para as demais)
