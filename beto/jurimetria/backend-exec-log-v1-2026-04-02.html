BUG-BE-01 RLS OFF em tabelas legacy jurimetria P0 ✅ Resolvido
Diagnóstico
Tabelas afetadas jurimetria_empresas, jurimetria_juizes, jurimetria_escritorios_reu, jurimetria_acordos RLS antes OFF em todas as 4 tabelas organizacao_id Presente em 3 tabelas; jurimetria_acordos não tem (tabela global)
Solução aplicada
-- Tabelas com organizacao_id → modelo A (tenant_isolation)
ALTER TABLE jurimetria_empresas ENABLE ROW LEVEL SECURITY;
CREATE POLICY "tenant_isolation" ON jurimetria_empresas
  USING (organizacao_id = public.get_org_id());
-- (idem para jurimetria_juizes e jurimetria_escritorios_reu)

-- jurimetria_acordos → sem org_id → SELECT público
ALTER TABLE jurimetria_acordos ENABLE ROW LEVEL SECURITY;
CREATE POLICY "allow_select" ON jurimetria_acordos
  FOR SELECT USING (true);
Verificação

Consulta pg_class.relrowsecurity confirmou true em todas as 4 tabelas após aplicação.

BUG-BE-02 julgadores.grau = NULL em 760/761 registros P0 ✅ Resolvido
Diagnóstico
Distribuição grau=NULL → 760 registros | grau=2 → 1 registro Causa raiz Julgadores importados de dados legados sem comarca_vara_id. Trigger fn_valida_vinculo_julgador impede setar grau=1 sem comarca_vara_id. Impacto RPC com p_grau=1 excluía 760 julgadores — subtab "1° Grau" mostrava vazio. Frontend SubtabPrimeiroGrau e SubtabComparativo já tinham workaround client-side (j.grau == null = 1º grau).
Opções avaliadas
Opção A (descartada) UPDATE grau=1 — bloqueado pelo trigger (requer comarca_vara_id) Opção B (adotada) Corrigir RPC: OR (p_grau = 1 AND j.grau IS NULL) — alinha com lógica do frontend Opção C Dropar trigger + update em lote — invasivo demais para dados de produção
Fix na RPC
-- Antes:
AND (p_grau IS NULL OR j.grau = p_grau::smallint)

-- Depois:
AND (
  p_grau IS NULL
  OR j.grau = p_grau::smallint
  OR (p_grau = 1 AND j.grau IS NULL)  -- grau=NULL → 1º grau (dados legados)
)
