Bug Cards Detalhados

BUG-01
SubtabSegundoGrau sempre vazia — todos julgadores têm grau=null
P1
Arquivo: tabs/julgadores/SubtabSegundoGrau.tsx
Linha: 45-55
Impacto: Tab inteira inutilizável
Causa
SubtabSegundoGrau chama get_jurimetria_ranking_julgadores com p_grau: 2. Verificado no banco: todos os 760 julgadores registrados têm grau = null. A RPC filtra por grau exato, retornando 0 registros. A tab exibe eternamente "Nenhum desembargador/relator cadastrado".
Evidência Backend
SELECT grau FROM julgadores GROUP BY grau → [{grau: null, count: 760}] get_jurimetria_ranking_julgadores(p_grau: 2) → [] (0 items)
Solução Proposta
Opção A (conservadora): SubtabSegundoGrau usar p_grau=null (todos os julgadores) e filtrar client-side por grau IN [2, 3], mesmo que retorne vazio atualmente. Ao menos não engana o usuário.

Opção B (ideal): Cadastrar julgadores com grau preenchido (migration ou formulário padrão ao importar). O formulário ModalCadastroJulgador já tem campo Grau obrigatório — o problema é que os 760 atuais foram importados sem grau.

Opção C (UX): Ocultar SubtabSegundoGrau/Comparativo quando não há dados de grau classificado, com mensagem orientativa.
BUG-02
SubtabComparativo sempre mostra "Sem dados" — mesma raiz que BUG-01
P1
Arquivo: tabs/julgadores/SubtabComparativo.tsx
Linha: 38-41
Impacto: Widget de comparativo inútil
Causa
SubtabComparativo faz duas queries paralelas: p_grau: 1 (0 resultados) e p_grau: 2 (0 resultados). Ambas as colunas "1° Grau — Top 20 por Taxa" e "2° Grau — Top 20 por Taxa" mostram Scale icon + "Sem dados".
Solução
Mesma raiz que BUG-01. Prioritário: corrigir dados de grau dos julgadores ou adaptar query para p_grau=null + segmentar por outro critério (ex: tribunal tier).
BUG-03
DetalhesJulgador: badge "null° Grau" quando grau=null
P1
Arquivo: modals/ModalDetalhes/DetalhesJulgador.tsx

Similar fix em ModalRankingCompleto.tsx linha 221 (tabela Julgadores): {j.grau}° → {j.grau != null ? `${j.grau}°` : '?'}

BUG-05
Filtro Localidade: label "Comarca — null" para comarcas sem vara

BUG-06
DocsPendentesRegiao: query sem LIMIT busca todas as linhas client-side
P2
Arquivo: tabs/dashboard/DocsPendentesRegiao.tsx
Linha: 22-30
Impacto: Potencial lentidão em orgs maiores
Causa
Query sem .limit(): supabase.from('processos').select('id, estado_id').eq(...).is('deleted_at', null).eq('pendencias', true)
Atualmente: 465 rows (aceitável). Em orgs com milhares de pendências isso transfere muitos rows para o browser apenas para agrupar por UF.
Solução
Criar RPC get_jurimetria_docs_pendentes_regiao(p_org_id) que faz GROUP BY estado_id no banco e retorna [{uf, count}] diretamente. Alternativa: adicionar .limit(2000) como proteção mínima.

BUG-08
D1: KPI total_processos não responde ao filtro de período (por design, mas confuso)
P2
Arquivo: RPC get_jurimetria_dashboard
Tipo: UX / comportamento ambíguo
Causa
A RPC get_jurimetria_dashboard conta total_processos e processos_ativos SEM filtrar por período (filtro de data afeta apenas decisoes). Isso é tecnicamente correto (processos não têm data de criação filtrada), mas confunde: usuário seleciona "Últimos 30 dias" e vê "14.456 processos" — o mesmo valor de "Todos os períodos".
Solução Proposta
Adicionar tooltip no KPI "Total de Processos" explicando "Total histórico — não filtrado por período". Ou adicionar filtro por data_distribuicao no count de processos na RPC.
