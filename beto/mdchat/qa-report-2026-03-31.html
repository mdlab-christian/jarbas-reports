P0 — CRÍTICOS (3 bugs) — Bloqueiam uso ou corrompem dados
P0-01 Attachments armazenados como string JSON em vez de JSONB array P0 — Dados Corrompidos
Arquivo src/pages/mdchat/services/mdChatService.ts : linha 223
Evidência banco Ao inserir uma mensagem sem anexos, o banco retorna attachments: {} (JSONB object vazio) — não é um array. Confirmado via POST direto na API.
Causa raiz updateMessageAttachments usa .update({ attachments: JSON.stringify(attachments) }) — isso serializa o array para uma string "[...]" e grava como texto no campo JSONB. O Supabase aceita mas o dado fica como string, não como array JSONB.
Snippet 
// mdChatService.ts linha 223 — ERRADO
.update({ attachments: JSON.stringify(attachments) })

// CORRETO
.update({ attachments: attachments })
Impacto Qualquer mensagem enviada com arquivo terá attachments como string. A leitura usa Array.isArray(m.attachments) que retorna false para string, resultando em attachments sempre vazios. Arquivos enviados nunca aparecem nas mensagens após refresh.
Solução Remover o JSON.stringify() no update. Supabase JSONB aceita objeto JS diretamente. Além disso, verificar o default do campo no banco: atualmente é {} (objeto), deveria ser '[]' (array) para consistência.
P0-02 MdChatAboutTab — crash com created_at vazio P0 — Crash em Runtime
Arquivo src/pages/mdchat/services/mdChatService.ts : linha 26
src/pages/mdchat/components/MdChatAboutTab.tsx : linha 74
Causa raiz Em getChannelsForUser(), o campo created_at é hardcoded como string vazia ('') porque o RPC mdchat_get_channels_with_info não retorna created_at. Quando o usuário abre a tab "Sobre" no ReaderPanel, o componente chama format(new Date(''), ...), onde new Date('') retorna Invalid Date e date-fns format() lança exceção.
Snippet 
// mdChatService.ts linha 26 — causa
created_at: '',  // hardcoded — RPC não retorna este campo

// MdChatAboutTab.tsx linha 74 — crash
format(new Date(channel.created_at), "dd 'de' MMMM 'de' yyyy", ...)
// new Date('') = Invalid Date → format() throws
Evidência banco RPC mdchat_get_channels_with_info retorna estas chaves: archived_at, channel_id, descricao, is_favorite, is_muted, is_private, is_special, last_message_at, last_message_preview, last_message_user, member_role, nome, setor_nome, tipo, unread_count — sem created_at.
Impacto Toda abertura da tab "Sobre" no Reader Panel causa erro React não capturado. Se não houver Error Boundary, trava o Reader Panel inteiro.
Solução Opção A (rápida): Em MdChatAboutTab, adicionar guard: {channel.created_at ? format(new Date(channel.created_at), ...) : 'N/A'}
Opção B (correta): Adicionar created_at ao retorno do RPC mdchat_get_channels_with_info e mapear no service.
P0-03 Usuário sem membership — spinner eterno no provisioning P0 — Acesso Bloqueado
Arquivo src/pages/mdchat/hooks/useMdChatProvisioning.ts
Evidência banco Gustavo (user_id: bc557356-eecd-4da0-85b8-10681afe32f3) não está em nenhum registro de mdchat_channel_members. A tabela tem apenas 8 registros com 2 user IDs: 6dd16878... (Christian) e 5fe9f43f... (Christian dono). RPC mdchat_get_channels_with_info retorna [] para Gustavo.
Causa raiz O provisioning verifica count de mdchat_channels — como já existem 4 canais, count = 4 > 0, portanto não chama o RPC de provisionamento. O usuário fica sem membership mas o sistema assume que está tudo OK. isReady = true é setado mas a sidebar mostra zero canais.
Snippet 
// useMdChatProvisioning.ts — lógica falha
// Conta canais da org, não memberships do usuário
const { count } = await supabase
  .from('mdchat_channels')
  .select('id', { count: 'exact', head: true })
  .eq('organizacao_id', organizacaoId);

if ((count ?? 0) === 0) {  // 4 > 0, então nunca provisiona novo user
  setIsProvisioning(true);
  await supabase.rpc('mdchat_provision_default_channels', ...);
Impacto Todo usuário novo da organização (que se cadastrou depois do setup inicial) vê sidebar vazia e não consegue participar de nenhum canal. Afeta todos os novos usuários onboardados.
Solução Mudar a lógica de provisioning: checar memberships do usuário, não canais da organização. Se o usuário não tiver memberships, adicionar como membro nos canais públicos da org.

P1-03 MdChatThreadView — parentQuery sem staleTime (refetch desnecessário) P1 — Performance
Arquivo components/MdChatThreadView.tsx : linhas 26-33
Causa raiz O parentQuery não tem staleTime definido (padrão = 0). Isso causa refetch do pai em cada re-render. Agravante: para buscar a mensagem pai, ele faz getMessages() (que fetcha 50 mensagens do canal + user info) apenas para encontrar 1 mensagem. Extremamente ineficiente.
Snippet 
// ThreadView.tsx linhas 26-33 — sem staleTime + ineficiente
const parentQuery = useQuery({
  queryKey: ['mdchat-thread-parent', parentMessageId],
  queryFn: async () => {
    const msgs = await getMessages(orgId!, channelId, 50); // busca 50!
    return msgs.find(m => m.id === parentMessageId) || null;
  },
  enabled: !!orgId,
  // staleTime ausente — padrão 0 = refetch sempre
});
Solução Criar função getMessageById(msgId, orgId) no service. Adicionar staleTime: 30_000 na query.
P1-04 useMdChatUnreads — N+1 queries por canal (performance) P1 — Performance
Arquivo hooks/useMdChatUnreads.ts : linhas 35-54
Causa raiz Para contar não-lidos, o hook dispara 1 query por canal em paralelo (Promise.all(channelIds.map(...))). Com 4 canais = 4 queries. Com 20 canais = 20 queries simultâneas. A leitura de timestamps também faz 1 query inicial separada. Total: N+1 queries para um dado simples.
Solução Criar RPC mdchat_get_unreads_per_channel(p_user_id, p_org_id, p_channel_ids[]) que retorna {channel_id, unread_count}[] em uma única query com GROUP BY. Ou incluir unread_count no retorno de mdchat_get_channels_with_info (que já existe no RPC mas ignora no service mapeando member_count: 0).
P1-05 Sidebar não exibe grupo de canais arquivados P1 — Funcionalidade Ausente
Arquivo components/MdChatWorkspaceSidebar.tsx
Causa raiz Os filtros da sidebar são: favorites, regularChannels (não favorito + não direto), dms. Não existe filtro para canais arquivados (archived_at !== null). Canais arquivados aparecem misturados nos "Canais" ou ficam invisíveis — a spec menciona grupo "Arquivados".
Solução Adicionar grupo "Arquivados" colapsado por padrão. Filtrar regularChannels para excluir c.archived_at !== null. Adicionar archived = filtered.filter(c => !!c.archived_at).
P1-06 MdChatAboutTab — archiveChannel sem refetch da sidebar P1 — UX Quebrada
Arquivo components/MdChatAboutTab.tsx : linhas 29-41
Causa raiz A função handleArchive faz o UPDATE no banco mas não tem callback para refetch dos canais na sidebar. Após arquivar, o canal continua aparecendo como ativo na sidebar até o usuário recarregar a página.
Snippet 
// MdChatAboutTab.tsx — não dispara refetch
const handleArchive = async () => {
  const { error } = await supabase.from('mdchat_channels')
    .update({ archived_at: new Date().toISOString() })
    .eq('id', channel.id);
  if (error) throw error;
  toast.success('Canal arquivado.');
  // ⚠️ Sem callback para invalidar cache de canais!
};
Solução Adicionar prop onChannelArchived?: () => void propagada de MdChatPage via ReaderPanel → AboutTab. Ou usar o realtime já configurado para members que vai refetch automaticamente.
