Achados — P0 / P1 / P2

5 achados
P1
MDZAP-AI-COMPOSER retorna HTTP 500
Webhook mdzap/ai-composer falha com "Workflow execution failed". Workflow ativo com nós: Webhook → Montar Prompt → Claude API → Extrair Resultado → Respond to Webhook. Causa raiz: Claude API pode estar falhando por credencial expirada ou payload estruturado incorretamente (campos data.instrucao / data.historico_mensagens podem não corresponder ao que o nó espera). Impacto: Compositor de IA do MdZap não funciona — usuários não conseguem gerar respostas assistidas.
Fix: Verificar nó "Claude API" no n8n — checar ANTHROPIC_API_KEY nas env vars do Railway. Testar com payload real: data.instrucao + data.historico_mensagens formatados como o nó espera. Se o nó "Montar Prompt" não mapeia os campos do envelope canônico, corrigir o mapeamento.
P1
MDZAP-ENVIAR-MENSAGEM retorna HTTP 500
Webhook mdzap/enviar-mensagem falha. Workflow: Webhook → Parse → Fetch Numero Config → Merge Config → Meta API Send → Save Saida Supabase → Respond OK. Causa raiz: "Meta API Send" está falhando — provavelmente token Meta/WhatsApp expirado ou número_id de teste não encontrado em "Fetch Numero Config". Impacto: Envio de mensagens WhatsApp pode estar falhando em produção para números não conectados.
Fix: 1) Verificar META_ACCESS_TOKEN nas env vars do n8n Railway (pode ter expirado). 2) Verificar "Fetch Numero Config" — endpoint Supabase ou validação de numero_id. 3) Testar com numero_id real de whatsapp_numeros onde status='connected'.
P1
58 mensagens órfãs com conversa_id = NULL
58 registros em whatsapp_mensagens com conversa_id = NULL. Estes registros são invisíveis na UI (useMdzapMensagens filtra por conversa_id), mas ocupam espaço e indicam falha no fluxo de criação. Causa raiz: Mensagens criadas durante testes de desenvolvimento sem associar a uma conversa, ou webhook de recebimento de mensagens criando registro antes de criar a conversa.
Fix: 1) Limpeza imediata (P1): DELETE FROM whatsapp_mensagens WHERE conversa_id IS NULL; -- ou mover para soft delete: UPDATE whatsapp_mensagens SET deleted_at = now() WHERE conversa_id IS NULL; 2) Prevenção (P2): Adicionar NOT NULL constraint: ALTER TABLE whatsapp_mensagens ALTER COLUMN conversa_id SET NOT NULL;
P1
MDZAP-JULIA-PROMPT-SAVE retorna HTTP 500
Webhook mdzap/julia-prompt-save falha. Workflow: Webhook → Buscar Max Versao → Calcular Versao → Desativar Versao Atual → Salvar Nova Versao → Respond. Causa raiz: "Buscar Max Versao" faz chamada HTTP para Supabase — provavelmente tabela mdzap_julia_prompts existe mas query falha por estrutura de payload diferente (campos data.prompt / data.modulo não mapeados corretamente). Impacto: MdzapAdminIA não consegue salvar novos prompts da Julia.
Fix: Verificar nós "Buscar Max Versao" e "Salvar Nova Versao" — checar se as queries Supabase usam os campos corretos da tabela mdzap_julia_prompts. Verificar se o campo 'modulo' no payload corresponde ao nome da coluna no banco.
P2
useMdzapMensagens sem filtro organizacao_id (defense-in-depth)
Hook useMdzapMensagens.ts filtra mensagens apenas por conversa_id, sem adicionar .eq('organizacao_id', orgId). Risco: Se um usuário adivinhar o UUID de uma conversa de outra org (altamente improvável com UUIDs aleatórios), o RLS ainda bloqueia o acesso. Mas não segue o padrão de defense-in-depth do projeto.
Fix em src/hooks/mdzap/useMdzapMensagens.ts: .from('whatsapp_mensagens') .select('...') .eq('conversa_id', conversaId) .eq('organizacao_id', orgId) // ← adicionar .is('deleted_at', null)
P2
ViewMdzapTemplates UPDATE/DELETE sem filtro de org secundário
Em src/pages/admin/views/ViewMdzapTemplates.tsx, o UPDATE usa .eq('id', editando.id) sem .eq('organizacao_id', orgId). O DELETE usa .eq('id', deletandoId) sem filtro de org. Mitigado por RLS, mas não segue o padrão de defense-in-depth do projeto.
Fix: Adicionar .eq('organizacao_id', orgId) nas operações UPDATE e DELETE para garantir que um usuário não consiga modificar template de outra org mesmo se RLS falhar.
P2
mdzap_stage_mappings (spec) vs mdzap_stage_map (banco)
O audit spec referencia mdzap_stage_mappings mas a tabela real é mdzap_stage_map. O frontend usa o nome correto (mdzap_stage_map em MdzapGestaoPanel.tsx). Impacto operacional: zero. Impacto em documentação/spec: sim.
Fix: Atualizar spec do mdzap para referenciar mdzap_stage_map ao invés de mdzap_stage_mappings.
