P1 Importante (4)

P1 ArrowLeft buttons sem aria-label (mobile header)

Dois botões "Voltar" no header mobile de MdzapAppPage usam size="icon" sem aria-label. Afeta leitores de tela e conformidade a11y.

src/pages/MdzapAppPage.tsx:99 (view=thread) e src/pages/MdzapAppPage.tsx:118 (view=lead)

O close button do lead panel desktop MdzapAppPage.tsx:239 também não tem aria-label.

Fix: Adicionar aria-label="Voltar para conversas" / aria-label="Fechar painel" nos três botões.

P1 font-[Manrope] como valor hardcoded — não usa token font-display

10 ocorrências de font-[Manrope] nos componentes mdzap e na página. O Design System v5 define display: ['Manrope', 'sans-serif'] no Tailwind, mas o token de classe deveria ser font-display (ou font-heading), não o valor arbitrário hardcoded font-[Manrope]. Nenhuma ocorrência de font-heading encontrada no módulo.

Afetados: MdzapDashboard.tsx, MdzapGestaoPanel.tsx, MdzapChatPanel.tsx, MdzapAppPage.tsx

Fix: Verificar qual classe Tailwind mapeia para Manrope (ex: font-display) e substituir todas as ocorrências de font-[Manrope].

P1 Dois sistemas de layout paralelos: MdzapChatPanel vs MdzapAppPage

Existe duplicação de lógica de layout: MdzapChatPanel (usado em /mdzap desktop) e MdzapAppPage (usado em /mdzap/app) implementam a mesma estrutura 3-colunas e tabs de forma independente. A rota /mdzap redireciona mobile para /mdzap/app mas mantém dois componentes separados com lógica duplicada (hooks, estados, keyboard shortcuts apenas no ChatPanel).

src/pages/mdzap/index.tsx + src/pages/MdzapAppPage.tsx + src/components/mdzap/MdzapChatPanel.tsx

Fix: Avaliar unificação — MdzapAppPage já implementa os dois breakpoints (desktop 3-col + mobile views). MdzapChatPanel pode ser descontinuado.

P1 staleTime ausente em query do Dashboard dentro de componente

MdzapDashboard declara um useQuery diretamente no componente (linha 55) para dashboard-conversas. Este query tem staleTime: 30_000 — OK. Porém o hook useMdzapSmartReplies usa useQuery internamente. Verificação: useMdzapSmartReplies.ts não foi encontrado com staleTime configurado no hook principal (apenas nos hooks listados explicitamente).

src/hooks/mdzap/useMdzapSmartReplies.ts

Fix: Verificar e adicionar staleTime: 30_000 em todos os useQuery do hook useMdzapSmartReplies se ausente.

P2 Melhoria (5)

P2 Filtro de período no Dashboard desabilitado com tooltip — UX incompleta

O Select de período (Hoje/7d/30d) está desabilitado com opacity-50 cursor-not-allowed e tooltip "em breve". Isso sinaliza funcionalidade incompleta para o usuário. O Dashboard usa a RPC fn_mdzap_conversas sem filtro de data.

src/components/mdzap/MdzapDashboard.tsx:108-124

Fix: Implementar o filtro de período ou remover o Select até que a RPC suporte o parâmetro p_periodo.

P2 Botão "Devolver Julia" não tem aria-label

O botão Button size="sm" para devolver a conversa ao Julia no MdzapThreadHeader usa ícone + texto mas não tem aria-label explícito. Menor impacto que P1 pois tem texto visível, mas melhora consistência.

src/components/mdzap/MdzapThreadHeader.tsx:83-94

Fix: Adicionar aria-label="Devolver conversa para Julia".

P2 Empty state da lista de conversas sem ícone ou CTA

Quando a lista de conversas está vazia (ex: fila vazia, nenhuma sua), o componente exibe apenas um <p> com texto muted, sem ícone visual ou call-to-action. Outros empty states no módulo (JuliaLab, Memória, Dashboard) usam ícone + mensagem.

src/components/mdzap/MdzapConversaList.tsx:248-261

Fix: Adicionar ícone contextual (ex: Inbox para fila vazia) e mensagem mais descritiva com instrução ao usuário.

P2 backdrop-blur em MdzapSearchBar — uso em elemento que não é card/modal/popover

MdzapSearchBar aplica backdrop-blur-sm em uma barra de busca embutida no thread. O Design System define glassmorphism apenas para cards, modais e popovers — não para componentes inline/funcionais.

src/components/mdzap/MdzapSearchBar.tsx:30

Fix: Remover backdrop-blur-sm e usar bg-card sólido.

P2 MdzapMsgThread (506 linhas) acima do limite de 500 linhas

O arquivo MdzapMsgThread.tsx tem 506 linhas, levemente acima do limite de 500 do Design System. Já está muito próximo do limite e contém lógica de drag&drop, scroll, busca e envio de arquivos que poderiam ser sub-componentes.

src/components/mdzap/MdzapMsgThread.tsx — 506 linhas

Fix: Extrair lógica de drag&drop em hook useMdzapDragDrop, e lógica de scroll em useMdzapScroll para reduzir o componente.

Análise por Categoria

CAT-A — Tipografia e hierarquia 7.5/10
  • Manrope usado em títulos principais (H1/H2, KPIs) — porém via classe arbitrária font-[Manrope] (10x), não via token font-display ou font-heading definido no Design System.
  • Corpo usa Inter via herança da body — correto.
  • CNJ/números de telefone usam font-mono (JetBrains Mono) — correto em MdzapThreadHeader e SwipeableCard.
  • Hierarquia visual clara: título de contato (text-sm font-semibold) → preview (text-xs muted) → tempo (text-[10px] mono).
  • Timestamps internos usam text-[10px] que pode ser difícil de ler em alguns monitores.
  • Desconto: uso de classe arbitrária em vez de token oficial Tailwind.
CAT-B — Paleta de cores e tokens 9.5/10
  • Zero hardcode de cor hexadecimal em todo o módulo mdzap — excelente.
  • Primary hsl(var(--primary)) = #20c997 usado consistentemente em badges de ativo, CTAs, ícone de não lidas.
  • Background segue Warm Charcoal com bg-background, bg-card, bg-surface.
  • Status colors: success=verde, warning=âmbar, destructive=vermelho — mapeamento correto em todos os status badges.
  • Recharts usa hsl(var(--token)) em vez de hex — padrão correto e raro de encontrar.
  • Único ponto: constante SENTIMENT_COLORS no Dashboard usa hsl(var(...)) corretamente também.
  • Desconto mínimo: nenhum achado de cor hardcoded.
CAT-C — Layout e espaçamento 8.5/10
  • Layout 3 colunas desktop (280px sidebar | flex-1 thread | 280px lead) implementado corretamente com CSS flex e overflow controlado.
  • Mobile usa padrão de 3 views empilhadas (list → thread → lead) com transição contextual via estado — correto.
  • Gap e padding seguem escala Tailwind: px-3, py-2.5, gap-2, space-y-3 — consistente.
  • Dois sistemas de layout existem em paralelo (MdzapChatPanel vs MdzapAppPage) — duplicação arquitetural (P1) mas ambos funcionam.
  • MdzapChatPanel usa coluna lista de 320px vs MdzapAppPage de 280px — inconsistência menor entre rotas.
  • max-w não aplicado no container raiz — em telas muito largas (>1600px) o layout estica.
CAT-D — Componentes interativos 8.0/10
  • Botões usam shadcn-ui com variantes (default/ghost/outline) e estados hover/disabled nativos — correto.
  • Inputs têm focus ring via shadcn — correto.
  • Tab buttons personalizados (não shadcn) têm hover:text-foreground hover:bg-surface / hover:bg-muted/30 — adequado.
  • Dropdowns (Select, Popover, AlertDialog) abrem e fecham corretamente — sem loop visual identificado no código.
  • BottomNav tem item ativo destacado via text-primary vs text-muted-foreground — correto.
  • SwipeableCard implementa drag com framer-motion incluindo feedback visual e ação de arquivar — bem construído.
  • Desconto: ArrowLeft buttons mobile sem aria-label (P1) e botão devolver sem aria-label (P2).
CAT-E — Estados (loading, empty, error) 9.0/10
  • Loading skeleton em ConversaList (5 skeletons de conversa) — excelente implementação.
  • Loading skeleton na thread (4 bubbles) — correto.
  • Loading nos KPIs do Dashboard via prop loading no KpiCard.
  • Empty states em: thread sem mensagens (ícone + 2 textos), JuliaLab sem histórico (ícone + texto), Memória sem dados, Dashboard sem leads aguardando.
  • Error states: toast de erro em toda ação, com mensagem legível (não stack trace).
  • Estado de upload com progresso numérico em MdzapMsgInput.
  • Desconto: empty state de lista de conversas sem ícone (P2).
CAT-F — Acessibilidade básica 7.0/10
  • aria-label em icon buttons: 18/18 dentro dos componentes mdzap — perfeito.
  • aria-label ausente em 3 botões na página MdzapAppPage: 2x ArrowLeft (P1) e 1x X (close lead panel desktop).
  • Labels em inputs: Input de busca sem label explícita (tem placeholder). Textarea de nota sem label associada via htmlFor.
  • Imagens/avatares: gerados com letra inicial — sem alt porque são div, não img — OK.
  • Contraste: tokens do DS garantem contraste mínimo. text-muted-foreground (#9e9890) sobre #12100e = ~4.2:1 — levemente abaixo de 4.5:1 WCAG AA em alguns casos.
  • Tab order: seguido naturalmente pelo layout flex — sem elementos ocultos que quebrem o fluxo.
  • BottomNav: aria-label em cada botão — correto.
  • Desconto: 3 botões sem aria-label na página + inputs sem labels formais + contraste borderline em muted.
CAT-G — Responsividade 8.5/10
  • 375px mobile: layout de views empilhadas sem overflow horizontal — correto.
  • Touch targets mobile: ArrowLeft e Info no header têm min-w-[44px] min-h-[44px] — correto e explícito.
  • BottomNav com 5 abas, cada botão tem minHeight: 44 explícito + aria-label — excelente.
  • Threshold de breakpoint em 1024px (lg:) — cobre tablet portrait e landscape corretamente.
  • Thread mobile sem elementos cortados: overflow-hidden + min-h-0 em todos os wrappers.
  • Redirect automático: /mdzap detecta window.innerWidth < 1024 e redireciona para /mdzap/app.
  • Desconto menor: seletor de período no Dashboard (disabled) pode confundir em mobile. Coluna de 320px no ChatPanel vs 280px no AppPage.
CAT-H — Micro-interações e feedback 9.0/10
  • Hover em card de conversa: hover:bg-muted/20 transition-colors — correto.
  • Badge de não lidas: dot animado (animate-pulse) + badge numérico no card — excelente.
  • Smart replies aparecem como chips clicáveis com hover — implementado.
  • Toast de confirmação em toda ação (28+ instâncias) — coverage completo.
  • Lead panel desktop com transição animada via transition-all duration-200 em width.
  • Typing indicator: 3 dots com animação staggered (animationDelay 0/150/300ms) — touch profissional.
  • Swipe de card com framer-motion: drag constraint, elastic, threshold — bem implementado.
  • ScrollToBottom button aparece com nova mensagem (hasNewMsg) — comportamento inteligente.
  • Favicon badge com contagem de não lidas (useMdzapFaviconBadge) — detalhe de qualidade.
  • Desconto: sem animação de entrada para o lead panel mobile (apenas troca de view).
CAT-I — Coerência visual 8.5/10
  • 5 tabs (Conversas, Dashboard, JuliaLab, Templates, Gestão) mantêm mesmo padrão de header e estrutura de scroll.
  • Ícones: 100% lucide-react, tamanhos h-3.5/h-4/h-5 usados de forma consistente por contexto (header=h-5, ações=h-4, micro=h-3.5).
  • Bordas: border-border e border-border/50 em toda a página — sem elementos fora do sistema.
  • Dark mode: sem elementos brancos quebrados — todos os fundos usam tokens escuros.
  • Header/sidebar: MdzapChatPanel usa header próprio separado do AppLayout, MdzapAppPage usa fixed inset-0 — ambos consistentes internamente mas diferentes entre rotas.
  • Badges de status (Julia ativa / Aguardando / Atendido por) têm cores semanticamente corretas e bordas dedicadas.
  • Desconto menor: inconsistência entre largura da sidebar (280px vs 320px) nos dois sistemas de layout paralelos.

  • Zero cores hexadecimais hardcoded em todos os 34 componentes e páginas mdzap — excelência em tokens CSS.
  • 100% lucide-react para ícones — sem mistura de bibliotecas.
  • TypeScript sem erros no escopo mdzap — compilação limpa.

  • BottomNav mobile com aria-label em cada aba e minHeight: 44px explícito para touch.
  • SwipeableCard com framer-motion: drag nativo, threshold, ação de arquivar/restaurar com feedback visual.
