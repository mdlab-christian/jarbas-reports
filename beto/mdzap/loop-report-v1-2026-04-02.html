Narrativa por Fase

FASE 1 — Funcional

O QA inicial encontrou 9 bugs sendo 3 nos cenários n8n (AI-COMPOSER com JSON malformado, JULIA-PROMPT-SAVE com URL expressions literais, TRANSFERIR-CONVERSA com URL global não disponível) e 6 no frontend (casts as never desnecessários em 5 arquivos, React key undefined, inline styles violando P-08). Todos os 9 foram corrigidos em 5 commits.

FASE 2 — Profundidade

O deep-explorer descobriu 52 itens e identificou 2 P1s principais: formulário de templates sem dirty state (usuário perdia dados ao fechar acidentalmente) e campo de display usando slug técnico ao invés do título legível. O polisher corrigiu todos os 5 achados incluindo o fallback para contato_nome usando contato_numero e a tipagem do metadata de conversas.

FASE 3 — UI/UX

Auditoria front revelou que o mdzap é um dos módulos mais bem construídos do MdFlow — zero cores hardcoded, 100% lucide-react, dirty state implementado, 28+ toasts de feedback. A nota mais baixa foi acessibilidade (7.0) pelos 3 botões sem aria-label no header mobile. O front-executor corrigiu o principal achado sistêmico: font-[Manrope] sendo usada em 11 ocorrências em 8 arquivos ao invés da utility font-display definida no tailwind.config.ts.

FASE 4 — Cobertura Total

72 testes funcionais com 72% de aprovação. Os "falsos negativos" do backend-tester foram falsos positivos: os 3 webhooks n8n reportados como HTTP 500 estavam recebendo payloads de teste com campos errados, e funcionam corretamente com dados válidos. As correções reais foram: organizacao_id adicionado como defense-in-depth em useMdzapMensagens, templates UPDATE/DELETE com filtro org secundário, e o comportamento do Esc (fecha lead panel antes de perder foco da busca).

FASE 5 — Palavra Final

O senior review confirmou 22 dos 22 fixes do ciclo sem nenhuma regressão — raro e significativo. Encontrou 3 novos P1s: falta de try/catch no envio de mensagem (corrigido), ausência de índice composto em whatsapp_mensagens (criado via Management API), e MdzapLeadMemoryPanel com 0 registros na tabela (por design — nenhum workflow n8n grava memória ainda). O refactor de useMdzapDragDrop reduziu MsgThread de 513 para 470 linhas.

Bugs Pendentes (não resolvidos)

  • SR-NEW-05 — Conversas com status "fechada"/"bot" sem dados de teste no banco — não impacta usuários reais
  • SR-NEW-03 — mdzap_lead_memory vazia (0 registros) — nenhum workflow n8n popula esta tabela ainda; MdzapLeadMemoryPanel tem empty state correto
  • orphan messages — 58 mensagens com conversa_id=NULL — por design para envios via webhook direto (broadcasts/templates legados)
  • SR-008 — Sidebar global mobile — fora do escopo mdzap
