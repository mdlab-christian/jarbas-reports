P0
BUG-01 — invalidateQueries em sendText/sendFile sem orgId na key (cache miss potencial)
hooks/mdzap/useMdzapSendMessage.ts : linha 76 e linha 152
Código atual
// sendText linha 76: queryClient.invalidateQueries({ queryKey: ['mdzap', 'mensagens', conversa.id] }); // sendFile linha 152: queryClient.invalidateQueries({ queryKey: ['mdzap', 'mensagens', conversa.id] }); // useMdzapMensagens.ts linha 14 — queryKey real: const queryKey = ['mdzap', 'mensagens', orgId, conversaId];
Causa raiz

O invalidateQueries usa ['mdzap', 'mensagens', conversa.id] mas o queryKey real inclui orgId antes de conversaId. React Query v5 usa match parcial por prefixo — a key incorreta não bate exatamente. O React Query faz prefix match então ['mdzap','mensagens'] invalida TODAS as queries de mensagens (de qualquer conversa), mas falta o orgId intermediário. Na prática causa invalidação mais ampla que o necessário.

Impacto

Após envio otimista, o refetch invalida queries erradas — pode causar flicker ou não atualizar o cache correto em edge cases multi-org. Em multi-tenant strict este é P0.

✅ Solução: Passar orgId no invalidateQueries para exatidão: queryClient.invalidateQueries({ queryKey: ['mdzap', 'mensagens', organizacao_id ?? '', conversa.id] })
P0
BUG-02 — lead_awaiting_human retorna sempre false no RPC fn_mdzap_conversas (fila "Fila" fica vazia mesmo com conversas aguardando)
Supabase RPC fn_mdzap_conversas (definição SQL) + MdzapConversaList.tsx : linha 64-68
Evidência banco
-- Conversa 4d6287ba: -- metadata.awaiting_human = true -- lead_awaiting_human (campo RPC) = false ← BUG -- lead_id = null (sem lead vinculado) -- Todas as 6 conversas retornam lead_awaiting_human = false
Causa raiz

A RPC fn_mdzap_conversas provavelmente calcula lead_awaiting_human a partir do campo leads.awaiting_human via JOIN, mas a conversa 4d6287ba não tem lead_id (é NULL). O status awaiting_human=true está apenas em whatsapp_conversas.metadata. O RPC não lê de metadata — não integra o campo de metadata ao resultado.

Impacto

A tab "Fila" em MdzapConversaList usa c.lead_awaiting_human === true como filtro (linha 64). Como sempre é false, a Fila aparece sempre VAZIA mesmo quando há conversas aguardando humano. Isso quebra o fluxo de atendimento humano — agentes não veem as conversas que precisam atender.

✅ Solução: Atualizar fn_mdzap_conversas para retornar COALESCE(leads.awaiting_human, (wc.metadata->>'awaiting_human')::boolean, false) como lead_awaiting_human. Alternativamente, o frontend pode verificar também metadata.awaiting_human.

BUG-04 — 5 mensagens órfãs com conversa_id=null no banco (ausência de FK constraint)
Tabela whatsapp_mensagens (Supabase)
Evidência banco
-- Query: whatsapp_mensagens WHERE conversa_id IS NULL -- Resultado: 5 registros encontrados -- Exemplo: { "id": "6e8ba218", "conversa_id": null, "direcao": "entrada", "tipo": "document" } { "id": "b79c6bb9", "conversa_id": null, "direcao": "entrada", "tipo": "text" }
Causa raiz

Mensagens chegaram sem conversa_id mapeado (provavelmente webhook MDZap recebeu mensagem antes de criar a conversa correspondente, ou falha no webhook). A tabela permite NULL no campo conversa_id.

Impacto

Mensagens invisíveis na UI. Dados perdidos. Pode acumular ao longo do tempo se o webhook não mapear conversas corretamente.

✅ Solução: (1) Investigar e corrigir o webhook n8n que processa mensagens recebidas para garantir conversa_id sempre preenchido. (2) Opcional: adicionar NOT NULL constraint com conversa padrão ou trigger de criação automática. (3) Limpeza das 5 mensagens órfãs existentes.

BUG-05 — Arquivada com nao_lidas=2 — badge de total não conta, mas dado inconsistente no banco
Tabela whatsapp_conversas + useMdzapConversas.ts : linha 69
Evidência banco
-- whatsapp_conversas WHERE status = 'arquivada': { "id": "c1000000-0000-0000-0000-000000000002", "status": "arquivada", "nao_lidas": 2, "contato_nome": "Carlos Eduardo Figueiredo" } -- useMdzapConversas linha 69: const totalNaoLidas = rawConversas .filter((c) => c.status === 'ativa') // ← filtra apenas ativas .reduce((sum, c) => sum + c.nao_lidas, 0);
Causa raiz

Quando uma conversa é arquivada, o campo nao_lidas não é zerado. O totalNaoLidas não conta arquivadas (correto), mas o dado no banco fica sujo. Se a conversa for desarquivada no futuro, voltará com badge de 2 não lidas incorretas.

✅ Solução: Ao arquivar conversa (bulk ou individual), executar UPDATE SET nao_lidas=0. Adicionar trigger PostgreSQL: ao UPDATE status para 'arquivada', zerar nao_lidas automaticamente.

BUG-07 — Heartbeat null não detecta número offline (useMdzapStatus bypassa verificação)
hooks/useMdzapStatus.ts : linhas 30-40
Código atual
// Ambos números têm ultimo_heartbeat = null no banco numeros.map((n) => { if (n.status === 'connected' && n.ultimo_heartbeat) { // ↑ Nunca executa quando ultimo_heartbeat é null const diff = now - new Date(n.ultimo_heartbeat).getTime(); if (diff > HEARTBEAT_OFFLINE_MS) { return { ...n, status: 'disconnected' }; } } return n; // ← retorna como 'connected' mesmo sem heartbeat })
Evidência banco
{ "numero": "5551816855848", "status": "connected", "ultimo_heartbeat": null } { "numero": "5551999990000", "status": "disconnected", "ultimo_heartbeat": null }
Impacto

Um número pode estar marcado como "connected" no banco com heartbeat=null indefinidamente sem ser detectado como offline pelo frontend. O badge mostra "WA 1/2" verde quando pode estar offline. Usuário confia que WhatsApp está ativo quando não está. Envio de mensagens vai para primaryNumero desconectado.

✅ Solução: Tratar null como "nunca heartbeat" = offline se status é connected: if (n.status === 'connected' && !n.ultimo_heartbeat) return { ...n, status: 'disconnected' }. Ou configurar o sistema MDZap para sempre enviar heartbeat ao conectar.

Observações e Achados Adicionais

Positivo: O arquivo TemplatesManagerDialog.tsx em modo edição (linha 116) não inclui updated_at no UPDATE payload, mas a tabela tem trigger set_updated_at — portanto o campo é atualizado automaticamente via trigger. Sem bug real.
Atenção: useMdzapSmartReplies usa um Map de módulo (fora do React), o que significa que o cache persiste entre navegações de página. Em SPA single-session isso é aceitável, mas pode servir sugestões "stale" se o usuário navegar para outra página e voltar. O cleanup no useEffect deleta a entrada ao desmontar, o que mitiga parcialmente.
Design System: Verificação de tokens CSS — componentes analisados usam hsl(var(--primary)), text-success, text-warning, text-destructive corretamente. Nenhum hardcode de cor encontrado nos componentes principais. TONE_STYLES em MdzapSmartReplies usa hsl(var(--chart-2)_/_0.5) que é token válido do DS.
Bucket Storage: Confirmado: mdzap-media ✅ e mdzap-audios ✅ existem no Supabase. Upload de mídia funcionará corretamente.
Tab Templates — duplicidade de query: TemplatesManagerDialog.tsx faz sua própria query à whatsapp_templates (linha 60-74), e useMdzapTemplates.ts também tem sua própria query. A queryKey é idêntica ['mdzap', 'templates', orgId], então o React Query compartilha o cache. Sem duplicação de rede, mas há código duplicado de queryFn. Considerar usar useMdzapTemplates dentro do TemplatesManagerDialog para DRY.
fn_mdzap_stats aguardando_humano=0: A RPC retorna 0 mas há conversa com metadata.awaiting_human=true. Isso confirma que a função calcula via tabela leads (JOIN) e não via metadata da conversa. Consistente com o BUG-02 encontrado.
