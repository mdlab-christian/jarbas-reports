Bugs P0 — Críticos (quebram funcionalidade core)

P0 HR2 / VALOR — Valores de restrição exibidos 100x menores
Arquivosrc/components/cliente/tabs/ClienteTabRestricoes.tsx:198 CausaA coluna restricoes.valor armazena valores em reais (float: 607.93) mas o código usa formatBRL(r.restricao_valor) que divide por 100. Resultado: R$ 607,93 exibe como R$ 6,08. EvidênciaQuery direta: valor=607.93 (float real no banco). formatBRL(centavos) divide por 100 — errado para esta tabela. ImpactoTODOS os valores financeiros de restrições são exibidos 100x menores. Decisão jurídica comprometida. SoluçãoUsar formatBRLReais(r.restricao_valor) no lugar de formatBRL(r.restricao_valor) na linha 198. TagDATA UI
P0 HR1/HR2 — Aba Restrições sempre vazia para maioria dos clientes
Arquivosrc/hooks/useClienteRestricoes.ts:49 — RPC get_cliente_restricoes_completo CausaA função PostgreSQL get_cliente_restricoes_completo usa public.get_org_id() internamente para filtrar por organização. Para clientes com restrições na tabela restricoes, a função retorna [] mesmo quando existem registros. Confirmado: cliente com 7 registros diretos na tabela retorna 0 via RPC. O badge da aba lê de cliente.restricoes JSONB (campo diferente) que também é 0 para 94% dos clientes. EvidênciaCliente 6bf1beca: SELECT direto = 7 restrições. RPC = 0. A inconsistência confirma que a função filtra por org mesmo com service_role (SECURITY DEFINER pode estar usando get_org_id() que retorna null quando não há JWT de usuário real). ImpactoAba Restrições mostra vazia para clientes com restrições no banco. Afeta 100% dos clientes com dados na tabela restricoes (9.919 registros). SoluçãoVerificar o body da função get_cliente_restricoes_completo e garantir que aceita qualquer org quando chamada via service_role, ou que o filtro de org está correto para o JWT do usuário logado. Testar via browser com usuário real. TagRPC DATA
P0 HR6 / HR7 — Nova Restrição criada no JSONB, não aparece na aba
Arquivosrc/components/cliente/modals/NovaRestricaoModal.tsx:84 + linha 91 CausaNovaRestricaoModal chama add_cliente_restricao que escreve na coluna JSONB clientes.restricoes. A aba Restrições lê de useClienteRestricoes → get_cliente_restricoes_completo que retorna dados da tabela restricoes. São dois storages diferentes. Após criar, o modal invalida ['cliente', cpfSlug] mas NÃO invalida ['cliente-restricoes', clienteId]. EvidênciaLinha 91: queryClient.invalidateQueries({ queryKey: ['cliente', cpfSlug] }). Falta: queryClient.invalidateQueries({ queryKey: ['cliente-restricoes', clienteId] }). ImpactoUsuário cria restrição, recebe toast "Restrição adicionada", mas a lista não atualiza. Dado salvo em lugar diferente do que é lido. Confusão total. SoluçãoMigrar add_cliente_restricao para inserir na tabela restricoes (não JSONB), e adicionar invalidação de ['cliente-restricoes', clienteId] no onSuccess. TagLOGIC RPC

Notas Técnicas Adicionais

Padrão SECURITY DEFINER + get_org_id()

As RPCs get_cliente_by_cpf, get_cliente_restricoes_completo, get_cliente_documentos, add_cliente_restricao, update_cliente_dados e get_cliente_processos_count usam public.get_org_id() que extrai a organização do JWT do usuário. Isso é correto para segurança multi-tenant. Os testes via service_role retornam vazio/erro porque não há JWT de usuário com org. Os bugs P0 de restrições e documentos devem ser verificados com login real no browser antes de confirmar definitivamente.

Inconsistência JSONB ↔ TABLE nas Restrições

A tabela restricoes (relacional) tem 9.919 registros com schema diferente do JSONB clientes.restricoes (legado). O sistema está em migração entre os dois modelos.
