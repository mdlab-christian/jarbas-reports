FIX 2 — Substituição de jurisprudências POR-BLOCO (APLICADO)

Investigação backend antigo (mdflow / hzqoxkmxvwxejjanstdh)

O CTO afirmou: "No backend antigo já tinha corretamente todas as vinculações de blocos de texto e jurisprudências." Investiguei — o cenário é mais complicado:

MétricaValorSignificado
Total de blocos no antigo 1244 Universo
Blocos com jurisprudencias_vinculadas (ARRAY) populado 7 Schema novo (nunca foi adotado em massa)
Blocos com jurisprudencias (JSONB) populado 1244 Schema legado: {"jurisprudencia_1": "uuid"}
UUIDs distintos referenciados pelo JSONB 550 Universo de juris referenciadas
Quantos desses 550 UUIDs existem em olivia_jurisprudencias do antigo 0 TODAS as 550 referências do JSONB são órfãs no próprio antigo
Quantos existem no novo 0 Idem — IDs não migram
Conclusão: o JSONB jurisprudencias do projeto antigo aponta para 550 UUIDs que não existem nem no antigo nem no novo. Provavelmente ficaram quebrados após uma migração antiga (n8n + GAS) onde as juris foram regeneradas com novos UUIDs mas os blocos não foram reapontados. Migration direta UUID-to-UUID é impossível.

Mas há mapeamento por nome: os blocos têm título tipo "(Serviço de crédito) GRAVACAO TELEFONICA" e existem no novo juris correspondentes "(Serviço de crédito) GRAVACAO TELEFONICA - RS", "- SC", "- SP". O sufixo de tribunal precisaria casar com processos.tribunal.

DECISÃO 1 — Como popular juris_ids dos blocos?

Ver código já pronto em assembleHtml.ts. O dado é o que falta.

A — Migration heurística por título + tribunal médio risco

Para cada bloco, buscar juris no novo onde título do bloco está contido no título da juris, filtrando pelo tribunal do processo associado (TJRS/TJSC/TJSP). Popular olivia_blocos_texto.juris_ids.

  • Resolve em massa (~1244 blocos)
  • Reaproveita o trabalho histórico do escritório
  • Heurística pode errar (false positives)
  • Vinc é "por bloco", mas tribunal varia por processo — vinc fica fixa ao primeiro tribunal
B — UI manual no Step 2 baixo risco

Adicionar dropdown de "vincular juris a este bloco" dentro do card de bloco no Step 2 (Tags + Blocos). Advogado escolhe na hora. Persiste em jurisprudencias_adicionais da sessão.

  • Zero risco de dado errado
  • Já parcialmente implementado (jurisprudencias_adicionais existe)
  • Trabalho manual em cada petição
  • Não resolve "biblioteca de blocos com juris pré-vinc"
C — Híbrido: heurística + revisão UI recomendado

Rodar migration heurística em modo "draft" (popular novo campo juris_ids_sugeridas). Step 2 mostra sugestões com ✓/✗ pra advogado confirmar. Confirmadas migram para juris_ids.

  • Aproveita massa + valida cada vinc
  • Construção de biblioteca curada
  • Mais escopo de implementação (1 nova coluna + UI)
Recomendação: opção B para destravar imediatamente as petições do Midas (já têm a infra de jurisprudencias_adicionais). Opção C como segunda fase (curadoria gradual da biblioteca).

DECISÃO 2 — Endereçamento da vara cível no início

CTO pediu: "endereçamento vinculado ao advogado na tabela advogados, preenchido com o endereçamento da vara cível do processo."

Estado atual:

  • Tabela advogados tem cabecalho_escritorio (texto livre) — não tem coluna enderecamento
  • RPC fn_olivia_smart_blocks_preencher_campos retorna 33 campos — nenhum deles é endereçamento da vara
  • Tabela processos tem vara, comarca, tribunal mas não tem endereço físico
A — Template fixo por tribunal+vara recomendado

Adicionar campo enderecamento_template em advogados (texto com placeholders: "EXMO. SR. DR. JUIZ DE DIREITO DA {{vara}} CÍVEL DA COMARCA DE {{comarca}}/{{uf}}"). RPC compõe na hora usando processos.vara/comarca/uf.

  • Customizável por advogado
  • Padrão jurídico ABNT
  • Precisa migration + popular template default
B — Tabela de endereços de varas grande escopo

Criar tribunais_varas com endereço físico completo de cada vara. Popular ~3000 varas (TJRS+TJSC+TJSP). Endereçamento composto pelo endereço real.

  • Dado mais preciso
  • Manutenção pesada
  • Não é padrão jurídico (peticiona-se com "EXMO. SR. DR. JUIZ" — endereço físico só pra envio postal)
C — Hardcode no template do modelo zero esforço

CTO/equipe edita olivia_modelos_peticao.conteudo_template_html_rico incluindo o endereçamento com {{vara}}, {{comarca}}, {{uf}} hardcoded no header. RPC já retorna esses campos.

  • Zero código
  • Funciona hoje
  • Cada modelo precisa ser editado individualmente
  • Não escala se houver muitos modelos
Recomendação: opção C para resolver agora (1 modelo, 1 edit), opção A como upgrade quando chegar a 5+ modelos.
