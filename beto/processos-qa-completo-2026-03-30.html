🔴 P0 — Cache do Modal não é invalidado após salvar

🔴 P0
useProcessoUpdate invalida ['processo', numeroCnj] mas o ProcessoModalShell usa ['processo-modal', processoId]
src/hooks/useProcessoUpdate.ts · src/components/processo/ProcessoModalShell.tsx

Causa raiz: Ao salvar alterações via rpc_processo_update, o hook invalida a query ['processo', numeroCnj] (usada pela página de detalhe /processos/:cnjSlug). Porém, ProcessoModalShell — o modal aberto a partir da tabela de processos — busca os dados com a chave ['processo-modal', processoId]. Essa key nunca é invalidada após o save, então o modal continua mostrando os dados antigos até o usuário fechar e reabrir.

Impacto: Edição feita no modal não reflete imediatamente. O usuário pode acreditar que o dado não foi salvo.

❌ Código problemático — useProcessoUpdate.ts
// Invalida apenas a página de detalhe (rota /processos/:cnjSlug) queryClient.invalidateQueries({ queryKey: ['processo', numeroCnj] }); queryClient.invalidateQueries({ queryKey: ['processos'] }); // ← FALTA: queryClient.invalidateQueries({ queryKey: ['processo-modal', processoId] })
✅ Correção sugerida — useProcessoUpdate.ts
// Adicionar invalidação da queryKey do modal queryClient.invalidateQueries({ queryKey: ['processo', numeroCnj] }); queryClient.invalidateQueries({ queryKey: ['processos'] }); queryClient.invalidateQueries({ queryKey: ['processo-modal', processoId] }); // ← ADD

🔴 P0 — Realtime de documentos invalida queryKey errada

🔴 P0
useEffect realtime invalida ['processo-documentos', processoId] mas o hook usa ['processo-documentos', processoId, organizacao_id]
src/components/processo/tabs/ProcessoTabDocumentos.tsx:44 · src/hooks/useProcessoDocumentos.ts

Causa raiz: O subscription Realtime do ProcessoTabDocumentos chama queryClient.invalidateQueries({ queryKey: ['processo-documentos', processoId] }). // Opção A: invalidar por prefixo (garante match independente do 3º elemento) queryClient.invalidateQueries({ queryKey: ['processo-documentos', processoId], exact: false }) // Opção B: obter organizacao_id via useAuth() e usar key completa const { organizacao_id } = useAuth(); queryClient.invalidateQueries({ queryKey: ['processo-documentos', processoId, organizacao_id] })

🟠 P1 — QueryKey de useProcessoBySlug não é invalidada pelo useProcessoUpdate

🟠 P1
useProcessoBySlug registra ['processo', numeroCnj, organizacao_id] mas useProcessoUpdate invalida ['processo', numeroCnj]
src/hooks/useProcessoBySlug.ts · src/hooks/useProcessoUpdate.ts

Causa raiz: A página de detalhe /processos/:cnjSlug usa useProcessoBySlug, que define queryKey: ['processo', numeroCnj, organizacao_id]. Ao salvar via useProcessoUpdate, a invalidação usa ['processo', numeroCnj] sem o organizacao_id. Por padrão invalidateQueries faz match por prefixo (exact: false), então normalmente funcionaria — exceto se o usuário tiver múltiplas queries com o mesmo CNJ em diferentes orgs (multi-tenant), onde apenas a primeira seria invalidada.

Impacto: Baixo em produção com org única; alto em ambientes multi-tenant. Vale tornar explícito para evitar regressão futura.

❌ Inconsistência
// useProcessoBySlug.ts queryKey: ['processo', numeroCnj, organizacao_id] // 3 elementos // useProcessoUpdate.ts — invalida queryClient.invalidateQueries({ queryKey: ['processo', numeroCnj] }) // 2 elementos
✅ Correção — uniformizar ou usar prefixo explícito
// useProcessoUpdate.ts queryClient.invalidateQueries({ queryKey: ['processo', numeroCnj], exact: false, // Já é o default, mas tornar explícito remove ambiguidade }); // OU passar organizacao_id para o update hook e usar a key completa

🟠 P1 — Conflito de QueryKey entre useDocumentosProcesso e useProcessoDocumentos

🟠 P1
Dois hooks distintos usam o prefixo 'processo-documentos' com parâmetros diferentes
src/hooks/useDocumentosProcesso.ts · src/hooks/useProcessoDocumentos.ts

Causa raiz: useDocumentosProcesso (usado no shared drawer) registra ['processo-documentos', processoId, restricaoId]. useProcessoDocumentos (usado na tab principal) registra ['processo-documentos', processoId, organizacao_id]. Como ambos usam o mesmo prefixo 'processo-documentos', uma invalidação por prefixo de qualquer um dos dois pode contaminar o cache do outro.

Impacto: Quando o usuário abre o shared drawer e depois o detalhe do processo, pode ver documentos de cache incorreto (ou re-fetch desnecessário). Difícil de reproduzir mas silencioso.

✅ Correção — usar prefixos diferentes
// useDocumentosProcesso.ts (shared drawer) queryKey: ['documentos-processo-drawer', processoId, restricaoId] // useProcessoDocumentos.ts (tab principal) queryKey: ['processo-documentos', processoId, organizacao_id] // mantém

🟡 P2
ProcessoTabOlivia — thread existente buscada do DB mas cachedThreadId sempre vazio (sessionStorage nunca populado no mount)
src/components/processo/tabs/ProcessoTabOlivia.tsx:~L36

Causa raiz: loadExistingThread verifica sessionStorage.getItem(sessionKey), e se nulo, busca do banco e faz sessionStorage.setItem(). Porém na próxima abertura do componente (mesmo sem reload), o sessionStorage já tem o valor — mas a função tenta re-usar o cached path que ignora o DB check. O bug é sutil: se a thread foi encerrada/criada nova externamente, o componente continua enviando mensagens para a thread antiga sem perceber.

Impacto: UX inconsistente — mensagens podem não ser exibidas após navegação entre processos na mesma sessão.

🟡 P2
ProcessoRestricaoPrincipal — staleTime: 30_000 no fallback da restricao é muito baixo (re-fetch a cada 30s)
src/components/processo/sections/ProcessoRestricaoPrincipal.tsx:~L24

Causa raiz: O fallback de restricao usa staleTime: 30_000 (30 segundos). Dados de restrição raramente mudam; esse staleTime causa re-fetches frequentes desnecessários, especialmente com múltiplas aberturas do drawer.

✅ Sugestão

🟢 P3
ProcessoDrawer — SheetContent sem SheetHeader envolvendo o SheetTitle
src/components/shared/processo_drawer/ProcessoDrawer.tsx · ProcessoDrawerHeader.tsx

O SheetTitle do Radix está dentro de ProcessoDrawerHeader, que é um div genérico — não está dentro de um SheetHeader.
