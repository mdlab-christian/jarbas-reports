Auditoria de Schema
P0 SCHEMA-001 restricao_tipo armazenado como INTEGER — spec define TEXT
Tabelas afetadas: processos.restricao_tipo e processos_restricoes.tipo
Tipo atual: integer null
Tipo esperado (spec): TEXT CHECK (tipo IN ('comprador','devedor','fiador','avalista'))
Impacto: Frontend faz mapeamento manual {1:'comprador', 2:'devedor', ...} — acoplamento frágil. Novos valores requerem mudança em dois lugares. Não há constraint de integridade.
P1 SCHEMA-002 processos_movimentacoes sem soft delete e sem updated_at
Tabela com 330k+ linhas sem deleted_at e sem updated_at.
Impacto: Movimentações só podem ser deletadas com hard delete, quebrando histórico. Trigger set_updated_at não existe nesta tabela.
P1 SCHEMA-003 processos_restricoes sem soft delete
Tabela processos_restricoes sem deleted_at e updated_at.
Impacto: Mesmos problemas de SCHEMA-002 — hard delete obrigatório, sem histórico de mudanças.
P2 SCHEMA-004 valor_causa como NUMERIC — padrão MdFlow é BIGINT centavos
processos.valor_causa é numeric.
Padrão MdFlow: BIGINT em centavos.
Impacto: Inconsistência com demais tabelas financeiras. Valores como 1500000.00 em vez de 150000000 centavos. Possível perda de precisão em cálculos acumulados.
P2 SCHEMA-005 RLS usa org_id() em processos mas get_org_id() em processos_restricoes
processos policies: usa org_id()
processos_restricoes policies: usa get_org_id()
Ambas são wrappers em torno da mesma lógica, mas a inconsistência dificulta manutenção e auditoria de segurança.
P2 SCHEMA-006 Tabela processos com 130+ colunas — extremamente larga
A tabela principal tem mais de 130 colunas, incluindo dados desnormalizados de cliente, empresa, advogado.
Impacto: PostgreSQL planner com dificuldade de usar índices, serialização lenta, JSON retornado por RPC tem >100 chaves.
Mitigação a longo prazo: Materialized view + refresh periódico para campos derivados.

Auditoria de RPCs
P0 RPC-001 get_processos_paginados não tem parâmetro p_ia_status — filtro quebrado
Evidência: Frontend (useProcessos.ts:141) passa p_ia_status: filtros.ia_status para a RPC.
A RPC não tem este parâmetro. Resultado: erro PGRST202 quando qualquer usuário usa o filtro "Análise IA".

Resposta observada:
{"code":"PGRST202","message":"Could not find the function public.get_processos_paginados(p_ia_status, p_organizacao_id, p_page, p_page_size) in the schema cache"}
Impacto: 100% dos usuários que tentam filtrar por IA status recebem erro silencioso. Lista não carrega.
P1 RPC-002 fn_processos_busca sem SECURITY DEFINER
Função fn_processos_busca tem security_definer = false.
Risco: Herda permissões do chamador. Se chamada por um usuário não-privilegiado, pode falhar ao acessar tabelas protegidas por RLS ou expor dados indevidos.
P1 RPC-003 fn_processos_cliente_snapshot sem SECURITY DEFINER
Função fn_processos_cliente_snapshot tem security_definer = false.
Risco: Mesmos riscos de RPC-002. Esta função captura snapshot de dados de cliente — potencial vazamento se chamada em contexto errado.
✅ OK RPC-CHECK RPCs críticas com SECURITY DEFINER correto
7 RPCs auditadas e corretas: get_processos_paginados, get_processos_kpis, rpc_processo_get_completo, fn_processo_timeline, fn_restricoes_listar, rpc_processo_update, rpc_processo_soft_delete.
Todas têm SECURITY DEFINER + SET search_path = public.

Parecer Sincero do Dev
Diagnóstico técnico — /processos backend

❌ Críticos P0 — bloqueia em produção

RPC-001: Filtro ia_status quebrado. Frontend manda p_ia_status mas a RPC não tem esse parâmetro. PGRST202 para qualquer usuário que use o filtro IA. Fix: adicionar p_ia_status TEXT DEFAULT NULL e implementar o filtro na RPC.
SCHEMA-001: restricao_tipo é INTEGER. Spec, KB e comentários dizem TEXT. Frontend mantém mapeamento manual frágil. Não é bloqueante no dia a dia mas é tech debt P0 por divergência fundamental de schema.

⚠️ Importantes P1

  • • SCHEMA-002/003: processos_movimentacoes e processos_restricoes sem soft delete — decisão de design que precisa ser intencional, não acidental.
  • • RPC-002/003: fn_processos_busca e fn_processos_cliente_snapshot sem SECURITY DEFINER — risco de segurança se chamadas por contexto não privilegiado.
  • • WH-001: Webhooks arquivar/restaurar não logam em webhook_requests — auditoria zero de quem fez o quê.

✅ O que está bem

  • ✅ RLS impecável: Todos os 5 cenários de isolamento passaram. Zero vazamento de dados entre organizações.
  • ✅ 50+ índices na tabela processos: Coverage excepcional. Sem índice faltando nas queries principais.
  • ✅ Performance sólida: get_processos_paginados ~212ms, get_processos_kpis ~214ms. Acima do benchmark de 150ms mas muito aceitável para 13k registros.
  • ✅ 4 webhooks funcionais: arquivar, restaurar, exportar, arquivar_lote — todos retornam envelope canônico correto.
  • ✅ 7 RPCs críticas com SECURITY DEFINER correto.
  • ✅ SQL injection neutralizado pela parametrização correta.
  • ✅ unaccent funcionando: busca por 'joão' encontra 134 registros com normalização de acentos.
  • ✅ staleTime 30_000ms em todos os hooks React Query.
  • ✅ Export síncrono: webhook retorna action=download_file com URL assinada imediatamente.

📋 Diagnóstico geral

O backend de /processos está fundamentalmente sólido, com RLS correto e performance adequada para o volume atual. O problema mais urgente é o filtro ia_status quebrado — qualquer usuário que clique em "Análise IA" na FilterBar recebe um erro silencioso e a lista não carrega. Isso é um P0 real que afeta UX em produção hoje.

A questão de restricao_tipo como INTEGER é tech debt acumulado — funciona porque o frontend compensa, mas cria acoplamento frágil que vai cobrar seu preço quando alguém precisar adicionar um novo tipo de restrição.

A ausência de log em webhook_requests para arquivar/restaurar é um gap de auditoria que, em um sistema jurídico, pode ser problemático: não dá para saber quem arquivou um processo e quando.
