P0 — CRITICO: Existem 2 overloads de rpc_processo_update com assinaturas diferentes:

8. Todos os Achados

P0-001 — rpc_processo_update duplicada com coluna fantasma

Descricao: Existem 2 overloads de rpc_processo_update. A versao com p_updated_at_check text referencia a coluna responsavel_id que nao existe na tabela processos. Tambem permite override direto de historico e etiquetas sem validacao, e a coluna prioridade e tratada como TEXT quando e BOOLEAN.

Impacto: Se o frontend invocar a versao text (o que depende do tipo do argumento), qualquer update com responsavel_id falhara. Risco de corrupção de dados via historico/etiquetas.

Fix sugerido:

-- Remover a versao legada (text):
DROP FUNCTION IF EXISTS rpc_processo_update(uuid, text, jsonb);
-- Manter apenas a versao (uuid, timestamptz, jsonb)

P1-001 — rpc_processo_soft_delete sem org check

Descricao: SECURITY DEFINER bypassa RLS. A funcao atualiza processos filtrando apenas por id, sem verificar organizacao_id. Qualquer usuario autenticado pode soft-delete um processo de outra org se souber o UUID.

Fix sugerido:

-- Adicionar validacao de org:
v_org_id := (current_setting('request.jwt.claims',true)::jsonb->'app_metadata'->>'organizacao_id')::UUID;
...
WHERE id = p_processo_id AND organizacao_id = v_org_id AND deleted_at IS NULL;

P1-002 — rpc_processo_restaurar sem org check

Descricao: Mesmo problema do soft_delete. Apesar de ter check de acesso_gestor, nao filtra por org. Um gestor de org A pode restaurar processo de org B.

Fix sugerido: Mesmo padrao — adicionar AND organizacao_id = v_org_id.

P1-003 — Indice idempotency_key global viola multi-tenant

Descricao: O indice processos_idempotency_key_unique e UNIQUE sem filtro de organizacao_id. Isso impede que orgs diferentes usem a mesma idempotency_key, violando isolamento multi-tenant.

Fix sugerido:

DROP INDEX processos_idempotency_key_unique;
-- idx_processos_idempotency_key (org-scoped) ja existe e e suficiente

P1-004 — processos_restricoes sem trigger updated_at

Descricao: A tabela processos_restricoes tem coluna updated_at mas nenhum trigger para atualiza-la automaticamente.

Fix sugerido:

CREATE TRIGGER trg_processos_restricoes_updated_at
  BEFORE UPDATE ON processos_restricoes
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

P2-001 — Performance de busca texto: 651ms

Descricao: Busca por texto em get_processos_paginados leva 651ms com 13.754 processos distribuidos. A funcao usa immutable_unaccent(lower()) com ILIKE, o que nao aproveita bem os indices trigram existentes.

Sugestao: Considerar indice GIN trigram composto ou materializar coluna de busca normalizada.

P2-002 — Colunas duplicadas (migracao incompleta)

Descricao: Existem 3 pares de colunas legadas:

  • distribuido_por + distribuido_por_id
  • estado_id (text) + estado_id_new (uuid)
  • restricao_valor + restricao_valor_new

Nenhum impacto funcional imediato, mas aumenta confusao e tamanho da tabela.

P2-003 — 59 indices na tabela processos

Descricao: A tabela processos tem 59 indices (26 MB). Indices de tamanho maior que os dados (21 MB). Cada INSERT/UPDATE precisa manter todos os 59 indices.

Sugestao: Auditoria de uso de indices (pg_stat_user_indexes) para remover indices nao utilizados.

P2-004 — Tabelas possivelmente duplicadas

Descricao: Existem pares suspeitos: processos_fase / processos_fases, processo_documentos / processos_documentos, processo_insights / processos_insights. Podem ser vestigios de migracoes.

P2-005 — processos_movimentacoes sem updated_at

Descricao: A tabela processos_movimentacoes nao tem coluna updated_at e nenhum trigger correspondente. A coluna deleted_at existe mas nao tem trigger associado. Movimentacoes sao append-only, entao o impacto e baixo.

Fix P1-001: Adicionar org check em soft_delete

CREATE OR REPLACE FUNCTION public.rpc_processo_soft_delete(p_processo_id uuid)
RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER SET search_path TO 'public'
AS $function$
DECLARE
  v_user_id UUID;
  v_org_id UUID;
  v_hist_entry JSONB;
BEGIN
  v_user_id := auth.uid();
  v_org_id := (current_setting('request.jwt.claims',true)::jsonb->'app_metadata'->>'organizacao_id')::UUID;

  v_hist_entry := jsonb_build_object(
    'tipo', 'exclusao', 'user_id', v_user_id,
    'timestamp', NOW(), 'descricao', 'Processo excluido'
  );

  UPDATE processos SET
    deleted_at = NOW(),
    historico = COALESCE(historico, '[]'::jsonb) || jsonb_build_array(v_hist_entry),
    updated_at = NOW()
  WHERE id = p_processo_id
    AND organizacao_id = v_org_id  -- FIX: org isolation
    AND deleted_at IS NULL;

  IF NOT FOUND THEN
    RETURN jsonb_build_object('success', false, 'error', 'PROCESSO_NAO_ENCONTRADO_OU_JA_EXCLUIDO');
  END IF;
  RETURN jsonb_build_object('success', true);
END;
$function$;

Fix P1-002: Adicionar org check em restaurar

CREATE OR REPLACE FUNCTION public.rpc_processo_restaurar(p_processo_id uuid)
RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER SET search_path TO 'public'
AS $function$
DECLARE
  v_user_id UUID;
  v_org_id UUID;
  v_hist_entry JSONB;
  v_has_gestor BOOLEAN;
BEGIN
  v_user_id := auth.uid();
  v_org_id := (current_setting('request.jwt.claims',true)::jsonb->'app_metadata'->>'organizacao_id')::UUID;

  SELECT acesso_gestor INTO v_has_gestor FROM users WHERE id = v_user_id;
  IF NOT COALESCE(v_has_gestor, false) THEN
    RETURN jsonb_build_object('success', false, 'error', 'PERMISSION_DENIED');
  END IF;

  v_hist_entry := jsonb_build_object(
    'tipo', 'restauracao', 'user_id', v_user_id,
    'timestamp', NOW(), 'descricao', 'Processo restaurado da lixeira'
  );

  UPDATE processos SET
    deleted_at = NULL,
    historico = COALESCE(historico, '[]'::jsonb) || jsonb_build_array(v_hist_entry),
    updated_at = NOW()
  WHERE id = p_processo_id
    AND organizacao_id = v_org_id  -- FIX: org isolation
    AND deleted_at IS NOT NULL;

  IF NOT FOUND THEN
    RETURN jsonb_build_object('success', false, 'error', 'PROCESSO_NAO_ENCONTRADO_OU_JA_ATIVO');
  END IF;
  RETURN jsonb_build_object('success', true);
END;
$function$;
