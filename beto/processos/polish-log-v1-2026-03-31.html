POLISH-001 FUNC P0
Exportação síncrona via action=download_file
Aplicado

Problema: O webhook mdflow/processos/export é síncrono e retorna action='download_file' com URL diretamente. O frontend ignorava essa resposta e iniciava polling em webhook_requests por até 5 minutos sem resultado.

Arquivo: src/pages/processos/index.tsx — função handleExport

Antes (bug)
// Após res.success:
toast.info('Preparando exportação...', { duration: 10000 });

// Polling for result
const requestId = payload.request_id;
let attempts = 0;
const maxAttempts = 150; // 5min at 2s intervals
// ... polling nunca termina
Depois (fix)
// Verificar se webhook retornou URL diretamente (modo síncrono)
if (res.action === 'download_file' && res.data?.url) {
  window.open(String(res.data.url), '_blank');
  toast.success('Exportação concluída!', { duration: 3000 });
  setIsExporting(false);
  return;
}

// Fallback: polling assíncrono (caso raro)
toast.info('Preparando exportação...', { duration: 10000 });
const requestId = payload.request_id;
POLISH-002 STATE P1
Empty state contextual com filtros avançados ativos
Aplicado

Problema: Filtros avançados (status, advogado, empresa, fase, etc.) retornando 0 resultados com campo de busca vazio exibiam "Nenhum processo cadastrado" — mensagem incorreta que sugeria banco vazio.

Arquivos alterados:

  • src/pages/processos/components/ProcessosTable.tsx — adicionada prop hasActiveFilters?: boolean
  • src/pages/processos/index.tsx — passa hasActiveFilters={"{"}activeAdvancedCount > 0{"}"}
Antes (bug)
if (!isLoading && processos.length === 0) {
  if (searchQuery) {
    return (
      <EmptyState
        icon={SearchX}
        title="Nenhum resultado encontrado"
        description="Tente ajustar os filtros..."
        actionLabel="Limpar filtros"
        onAction={onResetFilters}
      />
    );
  }
  // Sem busca E sem filtros: mesma mensagem!
  return (
    <EmptyState
      icon={Scale}
      title="Nenhum processo cadastrado"
      description="Importe ou distribua..." />
  );
}
Depois (fix)
if (!isLoading && processos.length === 0) {
  if (searchQuery || hasActiveFilters) {
    return (
      <EmptyState
        icon={SearchX}
        title="Nenhum resultado encontrado"
        description={searchQuery
          ? "Tente ajustar os filtros ou o termo de busca"
          : "Nenhum processo corresponde aos filtros ativos"}
        actionLabel="Limpar filtros"
        onAction={onResetFilters}
      />
    );
  }
  return (
    <EmptyState
      icon={Scale}
      title="Nenhum processo cadastrado"
      description="Importe ou distribua..." />
  );
}

POLISH-004 UI P2
Coluna CNJ não sortable (inconsistência visual)

Solicitação: tornar a coluna "Nº CNJ" sortable como as demais (Cliente, Empresa, Valor, Distribuído).

Motivo do skip: A RPC get_processos_paginados (definida em 20260304200000_cnj_normalize_search.sql) não suporta numero_cnj como p_sort_field. O CASE do ORDER BY aceita apenas: created_at, data_distribuicao, updated_at. Qualquer valor fora desse conjunto cai no ELSE silenciosamente (fallback para created_at), gerando inconsistência visual sem erro aparente. Fix requer alteração da migration/RPC — escopo de backend, não polish de frontend.

Próximo passo: Adicionar numero_cnj ao CASE do ORDER BY na RPC e no SORT_OPTIONS do frontend (nova tarefa separada).
