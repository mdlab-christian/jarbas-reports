BUG-1 — ultima_movimentacao_texto nunca populado no banco
P1 — Dado ausente
Evidência: Query direta: ultima_movimentacao_texto=not.is.null → content-range: */0 (0 de 14.455 processos têm o campo preenchido)
Causa raiz: O trigger/pipeline que importa movimentações do EPROC (JARBAS/Cecília) não popula o campo desnormalizado processos.ultima_movimentacao_texto mesmo quando processos.movimentacoes_total > 0 e há registros em processos_movimentacoes.
Impacto: Campo "Última Movimentação" nunca exibido em ProcessoDadosProcesso, ProcessoDrawerTabMovimentacoes (leve). Usuário perde contexto rápido da última ação processual.
Arquivo frontend: src/components/processo/sections/ProcessoDadosProcesso.tsx:190 — código correto, dados ausentes no banco
Fix sugerido: Migration SQL com trigger em processos_movimentacoes para atualizar processos.ultima_movimentacao_texto e ultima_movimentacao_em após INSERT (ORDER BY data_movimentacao DESC LIMIT 1). Ou batch update via RPC para histórico.
BUG-2 — Campos deposito_judicial e alvara não exibidos em nenhum componente
P2 — Missing display
Evidência: Schema tem: com_deposito BOOLEAN, deposito_judicial_valor NUMERIC, deposito_judicial_tipo TEXT, deposito_judicial_em TIMESTAMPTZ, alvara_numero TEXT, alvara_valor NUMERIC, alvara_expedido_em TIMESTAMPTZ. Nenhum aparece em componente React.
Causa raiz: Campos existem no schema (confirmados em supabase.ts e types/processo.types.ts) mas não foram adicionados ao ProcessoCompleto type nem a nenhuma seção de exibição.
Impacto: Informações de depósito judicial e alvará não acessíveis pelo usuário via interface. Baixa criticidade enquanto dados são NULL (só 1 processo com com_deposito=true, sem valores preenchidos).
Fix sugerido: Adicionar campos ao ProcessoCompleto type e criar seção "Depósito / Alvará" em ProcessoDadosProcesso condicionada a com_deposito === true || alvara_numero.

BUG-3 — rpc_processo_get_completo conta movimentações da tabela errada
P2 — Counter incorreto no RPC
Evidência:
-- O RPC usa:
FROM movimentacoes mv WHERE mv.processo_id = p.id  →  330.349 rows = 0 (tabela vazia!)

-- A tabela real é:
processos_movimentacoes  →  330.349 rows para Midas org
Arquivo: supabase/migrations/20260327000010_fix_rpc_processo_get_completo.sql:119-125
Impacto real: BAIXO — total_movimentacoes retornado pelo RPC é sempre 0 para todos os processos. No entanto, o badge de "Movimentações" em ProcessoView NÃO usa esse campo do RPC — usa useProcessoMovimentacoes que query processos_movimentacoes corretamente. Então a UX de badge não é afetada. Apenas o campo ProcessoCompleto.total_movimentacoes retorna 0 errado.
Onde total_movimentacoes é usado: Não usado em ProcessoView. Usado em /controller (ControllerMonitoradoRow.tsx:100) onde exibe item.total_movimentacoes — mas esse vem de outra query, não do RPC desta page.
Fix: Na migration do RPC, substituir FROM movimentacoes mv por FROM processos_movimentacoes mv e remover AND mv.deleted_at IS NULL (processos_movimentacoes não tem deleted_at).
