BUG-1 CRÍTICO Seção "Cliente" mostra "Nenhum cliente vinculado" mesmo com cliente cadastrado Migration Não Aplicada
Arquivos Envolvidos

ProcessoDadosCliente.tsx (toda função) · rpc_processo_get_completo (banco) · 20260327000010_fix_rpc_processo_get_completo.sql

Sintoma

O cabeçalho do Drawer exibe o nome do cliente corretamente (campo processo.cliente_nome), mas a seção "Cliente" na aba Dados mostra "Nenhum cliente vinculado".

Snippet — ProcessoDadosCliente.tsx (linha 13)
const c = processo.clientes;

if (!c) {
  return (
    <GlassCard>
      <h2 className="text-base font-semibold mb-4">Cliente</h2>
      <p className="text-sm text-muted-foreground">Nenhum cliente vinculado</p>
    </GlassCard>
  );
}
Causa Raiz — Confirmada ✅
Migration 20260327000010 não foi aplicada ao banco Supabase.
A rpc_processo_get_completo no banco é uma versão anterior que não realiza JOIN com a tabela clientes. Por isso, o campo processo.clientes retorna null no JSON da RPC.

Prova: A migration nova (aplicada) retornaria ACESSO_NEGADO ao ser chamada com service_role (pois get_org_id() retorna NULL sem sessão, e NULL IS DISTINCT FROM org_id = TRUE). A versão atual retorna PROCESSO_NOT_FOUND para processos ativos → é a versão antiga.
Evidência do Banco
# JOIN direto via PostgREST confirma que os dados EXISTEM no banco:
GET /rest/v1/processos?select=id,numero_cnj,cliente_id,cliente_nome,clientes(id,nome,cpf_cnpj)

→ [
    { "id": "1f0ad0b9...", "cliente_nome": "SOLANGE ARNDT DOS SANTOS",
      "clientes": { "id": "6479f3c6...", "nome": "SOLANGE ARNDT DOS SANTOS", "cpf_cnpj": "269.337.278-07" } },
    { "id": "56eacfba...", "cliente_nome": "DANIEL LEGUICA DOS SANTOS",
      "clientes": { "id": "8ed8d54d...", "nome": "DANIEL LEGUICA DOS SANTOS", "cpf_cnpj": "011.888.970-20" } }
  ]

# Mas RPC retorna PROCESSO_NOT_FOUND (versão antiga sem org check) em vez de dados:
POST /rest/v1/rpc/rpc_processo_get_completo {"p_processo_id": "1f0ad0b9..."}
→ { "error": "PROCESSO_NOT_FOUND" }

# Migration nova teria retornado ACESSO_NEGADO (get_org_id() = NULL ≠ org_id real)
Impacto

Seção "Cliente" do Drawer de Processos sempre vazia, mesmo para processos com cliente_id preenchido. CPF, email, telefone, endereço — nada aparece.

Solução

Aplicar a migration 20260327000010_fix_rpc_processo_get_completo.sql no Supabase (via Studio SQL Editor ou supabase db push). A migration já está no repositório — só falta executar no banco.

Causa Raiz — Confirmada ✅
ProcessoHeader não tem prop isModal (nem similar).
A interface do componente é: interface Props { processo: ProcessoCompleto; isEditing?: boolean; onEdit?: () => void; }
Não há parâmetro para suprimir o breadcrumb ou o CopyableCNJ de título em contexto modal.

Quando ProcessoModalShell renderiza ProcessoView (que renderiza ProcessoHeader), ambos exibem o CNJ: o SheetTitle do modal (ocorrência 1) + breadcrumb (ocorrência 2) + título grande (ocorrência 3).
Impacto

Usuário vê o número CNJ triplicado no Drawer de Processos: na barra do Sheet, no breadcrumb e no título principal — poluição visual grave, especialmente em telas menores.

Solução

Opção A (recomendada): Adicionar prop isModal?: boolean ao ProcessoHeader. Quando isModal=true, suprimir o breadcrumb (ocorrência 2) e o CopyableCNJ de título (ocorrência 3). O SheetTitle do modal mantém a ocorrência 1 como âncora visual do Sheet.

Opção B: Remover o SheetTitle com CNJ do ProcessoModalShell.tsx e manter apenas o header do ProcessoHeader.

BUG-3 CRÍTICO Aba Restrições vazia — fn_restricoes_listar com overloads conflitantes (PGRST203) 500 PostgREST
Arquivos Envolvidos

ProcessoTabRestricoes.tsx · useRestricoes.ts · fn_restricoes_listar (banco — dois overloads)

Cadeia de Falha
ProcessoTabRestricoes → useRestricoes({ organizacaoId, processoId })
  → supabase.rpc('fn_restricoes_listar', { ..., p_processo_id: processoId })
  → PostgREST: PGRST203 — "Could not choose the best candidate function"
  → hook: error !== null, data = []
  → UI: "Nenhuma restrição vinculada a este processo"
Evidência do Banco — Confirmada ✅
Dois overloads conflitantes da mesma função existem simultaneamente no banco:

Overload 1 (migration 20260320020000):
fn_restricoes_listar(p_org, p_cli, p_lead, p_apenas_processaveis, p_apenas_sem_processo, p_processo_id)

Overload 2 (migration 20260330000001):
fn_restricoes_listar(p_org, p_cli, p_lead, p_processo_id, p_apenas_processaveis, p_apenas_sem_processo)

PostgREST envia parâmetros por nome (named params). Com dois overloads que aceitam os mesmos nomes em ordens diferentes, ele não consegue resolver qual chamar → PGRST203. 
# Resposta exata do banco ao chamar fn_restricoes_listar:
{
  "code": "PGRST203",
  "details": null,
  "hint": "Try renaming the parameters or the function itself in the database so function overloading can be resolved",
  "message": "Could not choose the best candidate function between:
    public.fn_restricoes_listar(p_organizacao_id => uuid, p_cliente_id => uuid, p_lead_id => uuid,
      p_apenas_processaveis => boolean, p_apenas_sem_processo => boolean, p_processo_id => uuid),
    public.fn_restricoes_listar(p_organizacao_id => uuid, p_cliente_id => uuid, p_lead_id => uuid,
      p_processo_id => uuid, p_apenas_processaveis => boolean, p_apenas_sem_processo => boolean)"
}
Dados Existem no Banco
# Restrições com processo_id preenchido:
GET /rest/v1/restricoes?select=id,processo_id,empresa_nome,data_restricao&processo_id=not.is.null&limit=5
→ [
    { "processo_id": "0153b84a...", "empresa_nome": "RGE SUL DISTRIBUIDORA DE ENERGIA SA", "data_restricao": "2024-10-31" },
    { "processo_id": "65b46a3c...", "empresa_nome": "EDP SÃO PAULO DISTRIBUIDORA DE ENERGIA SA", "data_restricao": "2024-05-02" },
    { "processo_id": "fdb167e5...", "empresa_nome": "BANCO DIGIO S.A.", "data_restricao": "2022-06-25" },
    ...
  ]
# → Dados existem, mas a função falha antes de retorná-los.
Sintoma 2 — Data não aparece

Mesmo que o erro fosse contornado, a data é renderizada via {r.data_restricao && <span>{formatDate(r.data_restricao)}</span>} — isso funciona quando data_restricao está preenchida (banco confirma que está). O sintoma de "sem data" é consequência da aba vazia (dados nunca chegam ao front).

Impacto

Toda a aba Restrições é inacessível para qualquer processo. Usuário vê sempre "Nenhuma restrição vinculada a este processo" independente dos dados reais.

Solução

DROP do overload conflitante no Supabase Studio:
DROP FUNCTION IF EXISTS public.fn_restricoes_listar(uuid, uuid, uuid, boolean, boolean, uuid);
Manter apenas o overload com p_processo_id como 4º parâmetro (ordem da migration 20260330000001, que é a versão mais recente). Em seguida, recriar com CREATE OR REPLACE FUNCTION para garantir assinatura única.
