🔴 Bugs Críticos (P0/P1)
BUG-001 ✅ Encerrado — Falso Positivo CNPJ começando em 90 — Banco Santander ✓ DADO CORRETO
Confirmado pelo Gustavo: O CNPJ 90.400.888/0001-42 do Banco Santander está correto. CNPJs iniciando em 90 são válidos para bancos com capital estrangeiro operando no Brasil. Pesquisa confirmou que 90.400.888/0001-42 é o CNPJ oficial do Banco Santander (Brasil) S.A. na Receita Federal. Dado legítimo, sem problema.
BUG-002 🔴 P1 ★ Novo TypeError silencioso: err.message em catch block sem tipagem
Em ModalConfirmacaoDelete.tsx e ModalMergeEmpresa.tsx, o catch usa err.message em um bloco tipado como unknown, causando erro TypeScript que pode crashar o build ou swallowing silencioso do erro real.
Arquivo: modals/ModalConfirmacaoDelete.tsx linha ~62
Código problemático:
} catch (err: unknown) { toast.error(err.message || 'Erro ao excluir empresa.'); }

Problema: err.message em unknown não compila sem type narrowing. Provavelmente está funcionando apenas porque TypeScript strict não está habilitado ou há um cast implícito.
Fix: catch (err) { toast.error(err instanceof Error ? err.message : 'Erro ao excluir empresa.'); }
BUG-003 🔴 P1 ★ Novo Aprovação de empresa cria NOVO registro — não atualiza existente
O fluxo de aprovação (mode='approve') sempre executa um INSERT na tabela empresas, mesmo quando a empresa já existe no banco com status_aprovacao='pendente'. Isso pode criar duplicatas de empresas aprovadas.
Arquivo: hooks/use_empresa_modal.ts — função handleSave, branch mode === 'approve'

Comportamento: Quando a IA detecta uma empresa e cria um registro na empresas_aprovacao, a aprovação humana faz INSERT de uma nova empresa em vez de atualizar status_aprovacao da existente.
Cenário problemático: Se empresa já existe aprovada e a IA a re-detecta, a aprovação do item da fila cria uma segunda cópia.
Evidência: empresa_id no FilaPendentesSection é passado para o modal mas o INSERT não verifica existência prévia — apenas faz upsert na fila.
🟡 Bugs Moderados (P2/P3)
BUG-004 🟡 P2 ✦ Core E1 Confirmado Detecção de duplicatas apenas no CREATE — edit ignora similares
O hook handleNomeBlur só busca similares quando mode === 'create'. Ao editar uma empresa, o campo nome_fantasia pode ser alterado para algo já existente sem nenhum aviso de duplicidade.
Código: if (mode !== 'create' || form.nome_fantasia.length < 3) return;
Fix: Remover verificação de mode ou habilitar para mode='edit' com exclusão do próprio ID na RPC.
BUG-005 🟡 P2 ✦ Core R2 Confirmado Empresas sem categoria criam silenciosamente — sem alerta de completude
No modo CREATE e EDIT, categoria_id não é campo obrigatório — mas deveria ser obrigatório apenas para empresas processáveis. Empresas não processáveis podem ser salvas sem categoria (comportamento correto). O problema é que atualmente empresas processáveis também podem ser criadas sem categoria, quebrando o matching da OlivIA.
Validação atual: Categoria obrigatória APENAS em mode='approve'. Em create/edit não há validação condicional por processável.
Regra correta (confirmada pelo Gustavo): categoria_id obrigatório SE processavel = true. Empresas não processáveis podem ficar sem categoria.
Dados: Com 372 empresas processáveis, é provável que algumas não tenham categoria — isso afeta direto o funcionamento da OlivIA.
Fix sugerido: Validar no submit: if (form.processavel && !form.categoria_id) → erro "Categoria obrigatória para empresas processáveis".
BUG-006 🟡 P2 ★ Novo Campo categoria (texto) vs categoria_id (FK) — campo legado não sincronizado
A tabela empresas tem dois campos para categoria: categoria (string legado) e categoria_id (FK para empresas_categorias). O código salva apenas categoria_id, mas pode haver registros antigos com apenas o campo categoria preenchido, causando inconsistência silenciosa.
Schema (types.ts): categoria: string | null e categoria_id: string | null ambos existem.
Código: Todas as queries do QA usam categoria_id e fazem join com empresas_categorias. Empresas antigas com apenas categoria aparecem sem categoria na UI.
Risco: OlivIA pode estar usando o campo categoria legado enquanto a UI usa categoria_id — dados inconsistentes.
BUG-007 🟡 P2 ★ Novo Import CSV: loop sequencial sem batch — risco de timeout em produção
A função handleImportCSV itera linha por linha chamando callWebhook('empresas/capturar', ...) com await e sleep(100ms) entre cada chamada. Um import de 200 empresas leva ~20+ segundos e pode travar a aba.
Impacto estimado: 100 linhas = ~10s, 500 linhas = ~50s. Nenhum progresso visual além do toast inicial.
Fix: Processar em batches de 10-20 por request, ou usar um endpoint de bulk import no backend, ou adicionar progress bar.
BUG-008 🟡 P3 ✦ Core R3 Confirmado Órgão Restritivo não vinculado à empresa — sem campo orgao_id na tabela
A aba "Órgãos Restritivos" gerencia uma tabela separada (orgaos_restritivos), mas a tabela empresas NÃO possui campo orgao_id no schema visto em types.ts. Não há FK ligando empresa ao seu órgão restritivo.
Schema empresas: Nenhum campo orgao_id ou orgao_restritivo.
Impacto: Impossível filtrar empresas por órgão restritivo na listagem. A relação empresa ↔ órgão existe apenas via processos (não via cadastro direto).
R3 do core confirmado: "Validar órgão restritivo incorreto" — impossível validar sem o campo no schema.
BUG-009 🟡 P3 ★ Novo PopupDuplicidade: botão "Ver" não abre a empresa — apenas fecha o popup
Em empresa_tab_cadastro.tsx, o prop onView é passado como () => onDismissSimilares() — apenas fecha o popup, sem abrir a empresa similar para comparação.
Código: <PopupDuplicidade onView={() => onDismissSimilares()} />
Expectativa: O botão "Ver" deveria abrir a empresa similar em modo view/detalhe para o usuário comparar antes de decidir.
Impacto UX: O usuário vê "empresa similar encontrada" mas não consegue ver os dados dela sem fechar o modal e buscar manualmente.
BUG-010 🟡 P3 ★ Novo useQuery com meta.onSuccess deprecated — dados do empresa-detail podem não carregar
Em use_empresa_modal.ts, a query de carregamento usa meta: { onSuccess: undefined } enquanto a segunda query do mesmo key usa enabled: false. Isso é um anti-pattern do TanStack Query v5 que pode causar race condition no carregamento dos dados do formulário.
Código problemático:
const { isLoading: loadingEmpresa } = useQuery({ queryKey: ['empresa-detail', empresa_id], meta: { onSuccess: undefined }, ... });
const { data: empresaData } = useQuery({ queryKey: ['empresa-detail', empresa_id], enabled: false });

Anti-pattern: Duas chamadas useQuery com o mesmo key mas uma com enabled:false só pega cache — ok conceitualmente, mas o useEffect de preenchimento do form depende disso, e se a cache não estiver populada ainda, o form fica vazio.
