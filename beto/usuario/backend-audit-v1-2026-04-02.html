Saude Backend: 82/100
Schema solido, RLS funcional, RPCs bem escritas. 1 achado P0 (privilege escalation via RLS), 1 P1, 3 P2.

Achados por Prioridade

P0   Privilege Escalation — users_update_self permite alterar campos criticos
A policy users_update_self (USING: id = auth.uid(), WITH CHECK: id = auth.uid()) permite que qualquer usuario autenticado faca UPDATE em TODAS as colunas do seu proprio registro. Isso inclui:

perfil, acesso_financeiro, acesso_gestor, organizacao_id, ativo

Um usuario mal-intencionado pode executar diretamente no Supabase JS: 
supabase.from('users').update({ perfil: 'admin', acesso_financeiro: true, acesso_gestor: true }).eq('id', meuId)
Impacto: Escalacao de privilegios completa. Qualquer usuario pode se tornar admin com acesso financeiro e gestor.
Fix recomendado: Criar RPC update_user_profile SECURITY DEFINER que aceita apenas nome, whatsapp, cargo. Remover users_update_self e permitir edits somente via RPC. Ou usar trigger BEFORE UPDATE que impede mudanca em campos protegidos quando auth.uid() = NEW.id e o caller nao e admin.
P1   avatar_url armazena URL publica mas bucket e privado
O bucket avatares tem public: false, porem o registro existente de Christian Paz armazena: 
https://...supabase.co/storage/v1/object/public/avatares/...
Essa URL retorna HTTP 400 (bucket privado). O hook useUserAvatar foi corrigido para salvar o path relativo e gerar signed URLs, mas o registro legado no banco ainda contem a URL publica quebrada.

Impacto: Avatar de Christian pode nao carregar em contextos que usam avatar_url diretamente do banco (ex: fn_hydrate_user retorna avatar_url raw).
Fix: Migration para limpar o registro legado: UPDATE users SET avatar_url = '55a0c7ba.../avatar_df3f3195.webp' WHERE id = '5fe9f43f...' (manter somente o path relativo). Garantir que todo consumer gera signed URL antes de exibir.
P2   Campo telefone duplica whatsapp
A tabela tem ambos telefone (TEXT) e whatsapp (TEXT). Nos 2 registros com telefone preenchido, o valor e identico ao whatsapp. O frontend /usuario so usa whatsapp. O campo telefone nao e referenciado na spec de /usuario.
Recomendacao: Avaliar se telefone e usado em outras paginas. Se nao, depreciar com comentario.
P2   fn_user_convidar aceita perfis divergentes do CHECK constraint
O CHECK chk_users_perfil aceita: admin, juridico, comercial, financeiro, operacional.
A RPC fn_user_convidar valida: juridico, comercial, financeiro, admin, gestor, parceiro.

Divergencia: gestor e parceiro passam na validacao da RPC mas falham no INSERT por violar o CHECK. operacional e valido no CHECK mas nao e aceito pela RPC.
Fix: Alinhar a lista da RPC com o CHECK constraint.
P2   users_select_self nao filtra deleted_at IS NULL
A policy users_select_self usa apenas id = auth.uid(), sem filtrar soft-delete. Se um usuario for soft-deleted, ele ainda consegue ler seu proprio registro (embora nao devesse ter sessao ativa). A policy users_select_own_org filtra corretamente.
Risco: Baixo (usuario deletado nao deveria ter JWT valido), mas e uma inconsistencia.
