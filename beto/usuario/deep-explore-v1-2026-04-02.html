Bugs Encontrados

BUG P1 Dirty state nao bloqueia navegacao SPA (react-router)

Descricao: O handler beforeunload no index.tsx (linhas 38-43) so previne navegacao browser-level (refresh, fechar aba). Navegacao via react-router (cliques na sidebar, links internos) ignora completamente o dirty state, permitindo ao usuario perder alteracoes nao salvas sem nenhum aviso.

Como reproduzir:

1. Na pagina /usuario, clicar Editar em Dados Pessoais
2. Alterar o nome para qualquer valor (dirty banner aparece)
3. Clicar em qualquer link da sidebar (ex: Homepage)
4. A pagina navega sem aviso, alteracoes perdidas

Impacto: Perda silenciosa de dados do usuario. O dirty banner e o beforeunload dao falsa sensacao de seguranca.

Sugestao de fix: Implementar navigation guard com useBlocker do react-router v6.4+ ou useBeforeUnload + unstable_usePrompt. Alternativa: interceptar cliques na sidebar quando isDirty for true e exibir dialog de confirmacao.

Melhorias Sugeridas

MELHORIA P2 Password strength muito permissivo — "Abc12345" e considerado "Forte"

O algoritmo em usePasswordStrength.ts classifica como "Forte" qualquer senha de 8+ chars com 3 tipos de caractere. Senhas triviais como "Abc12345" ou "Password1" passam como fortes.

Sugestao: Adicionar verificacao contra lista de senhas comuns (top 1000) e/ou penalizar padroes sequenciais (abc, 123, qwerty). Bibliotecas como zxcvbn sao uma opcao mais robusta.
MELHORIA P2 Cargo aceita qualquer string — sem validacao de tamanho minimo

O campo Cargo pode ser salvo com 1 caractere (ex: "a") ou com caracteres especiais sem sentido. Tem maxLength=100 mas nenhum minLength. Cargo vazio salva como null (correto).

Sugestao: Considerar minimo de 2 caracteres quando preenchido, e/ou regex basico para evitar lixo.
MELHORIA P2 Logout via confirm dialog nao e acessivel via snapshot (portal rendering)

O ConfirmDialog de logout renderiza em portal do Radix. O agent-browser snapshot nao detecta o dialog via [role="alertdialog"], embora ele exista no DOM. O dialog funciona corretamente quando acessado via eval JS.

Sugestao: Nao e um bug real — e uma limitacao de acessibilidade do snapshot tool. Mas verificar que o dialog tem role="alertdialog" corretamente no HTML final (ja tem, via Radix).
MELHORIA P2 GIF nao rejeitado explicitamente com mensagem amigavel

O bucket Supabase restringe allowed_mime_types a image/jpeg, image/png, image/webp. O input HTML aceita apenas esses tipos. O modal mostra "JPG, PNG ou WebP". Mas se um GIF for arrastado via drag-and-drop (bypassando o accept do input), o validateFile rejeita com "Arquivo invalido. Use uma imagem JPEG, PNG ou WebP." — funciona corretamente.

Sugestao: Nenhuma acao necessaria — validacao frontend + backend esta coberta. Apenas documentando como OK.
MELHORIA P2 Nenhuma inline validation para WhatsApp durante digitacao

A validacao de WhatsApp so ocorre no click de Salvar (toast error). Nao ha feedback inline durante a digitacao como existe para o PIX (chaveError). O usuario so descobre o erro ao tentar salvar.

Sugestao: Adicionar validacao inline similar ao PIX — mensagem abaixo do campo quando digitos < 10 ou > 11 apos debounce.

Analise de Codigo Estatica

OK Validacoes defensivas completas

Nome: trim + min 3 chars. WhatsApp: digits only, 10-11. CPF: check digits (Mod 11). CNPJ: check digits. Email: regex basico. Telefone PIX: 10-11 digits. Aleatoria: 32-36 chars.

OK Hooks seguem padroes MdFlow

useUserProfile com staleTime 30s. useUserPIX, useUserAvatar, useUserSecurity separados por dominio. useAvatarSignedUrl resolve paths relativos vs URLs completas. Todos usam queryClient.invalidateQueries apos mutacao.

OK Design System: tokens CSS, sem hardcode

Todas as cores usam tokens Tailwind (text-foreground, bg-primary/20, text-destructive, etc). STRENGTH_COLORS usa bg-destructive/bg-warning/bg-success. Fonts: display (Manrope) para H1/H2, Inter para corpo.

OK PIX dados salvos sem mascara no banco

useUserPIX.savePIX faz chave.replace(/\D/g, '') para cpf/cnpj/telefone antes de salvar. Banco recebe digits puros. Mascara e reaplicada na exibicao via applyPixMask.

OK Avatar upload: path relativo no DB, signed URL para display

useUserAvatar salva org_id/user_id/avatar_hash.webp no DB via RPC. Gera signed URL (1h) para display imediato. useAvatarSignedUrl resolve paths relativos automaticamente.
