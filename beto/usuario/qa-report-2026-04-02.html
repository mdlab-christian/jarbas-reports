Bugs Encontrados

P0 Overflow horizontal em 375px (mobile) — conteudo extravasa 89px
Teste: A5
Evidencia: scrollWidth=458 vs clientWidth=369 no viewport 375x812
Causa raiz: O container principal (.mx-auto.w-full.max-w-3xl) nao respeita overflow em mobile. Possivelmente o identity card com flex horizontal (avatar 96px + info) nao quebra em coluna no mobile, ou a sidebar/header esta empurrando conteudo.
Impacto: Usuarios em smartphones nao conseguem usar a pagina sem scroll horizontal. Viola principio P-07 (Desktop-first mas mobile desde o inicio).
Solucao proposta:
// UsuarioIdentityCard.tsx — adicionar flex-wrap em mobile
// De: className="flex items-center gap-6"
// Para: className="flex items-center gap-6 max-sm:flex-col max-sm:items-start max-sm:gap-4"

// Tambem considerar: overflow-hidden no container pai
// pages/Usuario/index.tsx:
// className="mx-auto w-full max-w-3xl space-y-6 overflow-hidden px-4 sm:px-0"
P1 Preview 64px do avatar NAO atualiza ao mover/zoom
Teste: H6
Arquivo: ModalAvatarCrop.tsx, linhas 267-278
Causa raiz: O preview canvas usa um ref callback que so executa na montagem inicial do elemento. Quando zoom/pan mudam e o useEffect (linhas 117-121) redesenha o canvas principal, o preview NAO e re-renderizado porque o ref callback nao e re-invocado por React.
Impacto: Usuario ve preview desatualizado durante crop — pode salvar achando que ficou de um jeito mas na verdade ficou de outro.
Solucao proposta:
// Criar um useEffect separado para atualizar o preview canvas:
const previewRef = useRef<HTMLCanvasElement>(null);

useEffect(() => {
  if (!previewRef.current || !canvasRef.current) return;
  const ctx = previewRef.current.getContext('2d');
  if (ctx) {
    ctx.clearRect(0, 0, 64, 64);
    ctx.drawImage(canvasRef.current, 0, 0, 64, 64);
  }
}, [zoom, pan, imageSrc]); // Re-draw preview quando zoom/pan mudam
P2 staleTime 120s no useUserProfile — padrao MdFlow e 30s
Teste: L4
Arquivo: hooks/useUserProfile.ts, linha 23
Causa raiz: staleTime definido como 2 * 60 * 1000 (120s) em vez do padrao de 30_000ms (30s) do projeto.
Impacto: Se usuario alterar dados em outro dispositivo, a UI pode mostrar dados stale por ate 2 minutos.
Solucao: staleTime: 30_000
P2 ModalAvatarCrop sem touch events para mobile
Teste: H5
Arquivo: ModalAvatarCrop.tsx, linhas 139-152
Causa raiz: Handlers de pan usam onMouseDown/onMouseMove/onMouseUp. Nao ha onTouchStart/onTouchMove/onTouchEnd equivalentes.
Impacto: Em dispositivos touch (mobile/tablet), usuario nao consegue arrastar a imagem para reposicionar o crop. Zoom funciona via slider.
Solucao: Adicionar handlers de touch equivalentes com e.touches[0].clientX/clientY.
P2 STRENGTH_COLORS usa Tailwind hardcoded em vez de CSS variables
Arquivo: hooks/shared/usePasswordStrength.ts, linhas 13-15
Causa raiz: bg-amber-500 e bg-emerald-500 sao classes Tailwind com cores fixas, nao CSS variables do Design System.
Impacto: Inconsistencia com DS v5. Cores nao mudam se o tema for alterado.
Solucao:
// De:
weak: 'bg-destructive',
medium: 'bg-amber-500',   // hardcoded
strong: 'bg-emerald-500', // hardcoded

// Para:
weak: 'bg-destructive',
medium: 'bg-warning',     // var(--warning)
strong: 'bg-success',     // var(--success)
P2 UPDATE em users sem organizacao_id — code smell
Teste: L2
Arquivos: useUserPIX.ts (linha 19), UsuarioPersonalForm.tsx (linha 82)
Causa raiz: Queries fazem .eq('id', userId) sem incluir .eq('organizacao_id', orgId). Embora RLS proteja via policy users_update_self, o padrao do projeto recomenda sempre filtrar por org explicitamente.
Impacto: Nenhum impacto de seguranca (RLS protege), mas inconsistencia com padrao do projeto e menos defensivo.
Solucao: Adicionar .eq('organizacao_id', orgId) nas queries de UPDATE.
