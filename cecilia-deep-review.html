5. Issues Identificados
🔴 [CRIT-01] jarbas-state.json ausente — próximo ciclo será baseline
O arquivo logs/jarbas-state.json não existe no momento desta análise. Isso significa que o próximo ciclo do cron-monitorar vai tratar TODAS as OABs como "primeira execução" (isFirstRun=true). Nesse caso: processos são salvos como baseline, mas nenhuma movimentação é enviada ao ingest n8n. Comportamento esperado e correto, mas é bom Christian estar ciente de que o estado de dedup foi resetado.

Causa provável: restart do processo JARBAS ou remoção manual do arquivo de logs.
Impacto: ciclo seguinte terá baseline fresh — a partir do próximo ciclo, dedup funcionará normalmente.
Ação: monitorar os logs do próximo ciclo para confirmar que o baseline foi criado corretamente.
🟡 [WARN-01] skill_rag_mdflow.mjs pode não existir
checkRagReindex() faz import('../skills/skill_rag_mdflow.mjs') dinamicamente aos domingos às 23h. O arquivo não foi encontrado no diretório de skills. Isso causará uma exceção silenciosa (catch local), mas é um ponto de falha que vai gerar noise nos logs semanalmente.

Ação: ou criar o stub skill_rag_mdflow.mjs, ou desativar o checkRagReindex() até que o ChromaDB esteja pronto.
🟡 [WARN-02] Estado de dedup em arquivo local — risco de perda
O estado de dedup (logs/jarbas-state.json) fica em disco local. Se o Mac Mini reiniciar, o arquivo pode ser perdido dependendo de onde está o diretório de trabalho (cwd) do processo JARBAS. Não há persistência no Supabase para esse estado.

Risco: flood de movimentações no próximo ciclo pós-restart (todas as da última semana são tratadas como novas). O isFirstRun=true impede o ingest — mas movimentações muito antigas podem vazar se o arquivo existir e tiver chaves antigas que expiram (TTL 7 dias).
Ação: considerar salvar o state no Supabase (jarbas_logs ou tabela própria) como backup, com sync a cada ciclo.
🟡 [WARN-03] KEYWORDS_URGENTE pode classificar "Publicação de Decisão" como urgente incorretamente
A regex /publicad/i no array KEYWORDS_URGENTE vai casar com "publicado" em qualquer contexto — inclusive "ato publicado no expediente" que deveria ser SKIP. O problema está na ordem de verificação: URGENTE é verificado antes de SKIP, então "publicado" sempre vira urgente, mesmo quando é só uma publicação administrativa.

Ação: refinar a regex para /decisão publicada|sentença publicada|publicação.*decisão/i com lookahead, ou mover para RELEVANTE com menor peso.
🟡 [WARN-04] Triagem ambígua limitada a 50 processos por ciclo
MAX_AMBIGUOUS_CLAUDE = 50 (env TRIAGEM_MAX_AMBIGUOUS). Ambíguos além do limite são classificados como "relevante" por segurança — o que força o acesso desnecessário a processos que provavelmente seriam SKIP. Em dias com muitas movimentações novas, isso pode reduzir a economia do Dual-Flow.

Ação: avaliar se 50 é suficiente para o volume típico. Considerar aumentar para 100 ou implementar um segundo passe com cache de classificações por evento-type.
🔵 [INFO-01] MAX_MOVS_PER_OAB limita envio silenciosamente
MAX_MOVS_PER_OAB = 200 (env). Se uma OAB tiver mais de 200 novidades em um ciclo, as extras são descartadas silenciosamente (só um console.warn). Não há notificação Telegram nem log no Supabase para esse corte.

Ação: adicionar log no Supabase e notificação Telegram quando o limite for atingido.
🔵 [INFO-02] INGEST_ALERT_THRESHOLD compartilhado entre todas as OABs
O contador _ingestConsecutiveFailures é global (module-level), não por OAB. Falhas intercaladas em OABs diferentes podem acumular no contador incorretamente. Ex: RS116571 falha, SC074025 sucesso (reseta?), mas o código só reseta em sucesso — então falhas de OABs diferentes se acumulam no mesmo contador.

Verificar: o código reseta em ingestResult.success = true, mas o contador é module-level e persiste entre chamadas de diferentes OABs. Pode gerar alertas falso-positivos.
✅ [GOOD-01] Headless: false — decisão correta para EPROC
O browser roda com headless: false. Isso é intencional e correto para o EPROC: tribunais frequentemente bloqueiam headless por fingerprinting. O Mac Mini fica com o browser visível — normal e esperado.
✅ [GOOD-02] Dual-Flow economiza acessos de forma inteligente
A Fase 1 (snapshot) nunca entra nos processos. Apenas urgente + relevante passam pela Fase 3. Em um escritório com ~5.800 processos, a maioria das movimentações diárias são conclusões e cargas (SKIP) — o que significa que a Cecília economiza ~60-80% dos acessos ao EPROC em dias típicos.
✅ [GOOD-03] Re-login callback sem reiniciar o ciclo
reloginFn() fecha o browser atual e reabre a sessão sem abortar o ciclo completo. Isso é crucial para OABs que ficam muitas horas processando — a sessão EPROC expira (~2h) e o sistema se recupera automaticamente.
6. Recomendações Priorizadas
P0
Verificar logs do próximo ciclo (jarbas-state.json ausente)
Confirmar que o baseline foi criado corretamente após o próximo ciclo às 13h ou 19h. Se o state não for salvo, investigar o cwd do processo JARBAS.
P1
Criar stub skill_rag_mdflow.mjs ou remover checkRagReindex()
Evitar exception semanal silenciosa. Um stub com export async function skillRagMdflow() { return { skipped: true }; } resolve até a implementação real.
P1
Salvar jarbas-state.json em Supabase como backup
A cada ciclo, fazer upsert do state em uma tabela jarbas_state (ou em jarbas_logs com categoria "state_backup"). Ao inicializar, se o arquivo local não existir, restaurar do Supabase. Evita baseline desnecessário após restart.
P1
Refinar regex /publicad/i para evitar falso-positivo em URGENTE
Trocar por /decisão publicada|sentença publicada|acórdão publicado/i. Movimentações administrativas "publicado no expediente" devem cair em SKIP, não URGENTE.
P2
Tornar _ingestConsecutiveFailures por OAB
Usar um Map: const _ingestFailures = new Map(); indexado por OAB. Alertar apenas quando a mesma OAB falhar 3x consecutivas, não acúmulo cross-OAB.
P2
Log no Supabase quando MAX_MOVS_PER_OAB for atingido
Adicionar logJarbas('alerta', 'Limite MAX_MOVS atingido', {...}) + notificação Telegram para Christian saber quando uma OAB estava com volume muito alto.
P2
Dashboard de métricas da Cecília no MdFlow
Uma view simples em jarbas_execucoes mostrando: ciclos por dia, movimentações por OAB, taxa de sucesso do ingest, economia do Dual-Flow (% SKIP), alertas ativos. Útil para Christian monitorar a saúde sem precisar checar logs.
