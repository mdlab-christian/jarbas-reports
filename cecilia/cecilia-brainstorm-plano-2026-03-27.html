Bugs Críticos a Corrigir Antes do Teste E2E

P1 — Bloqueadores
  • 🔑 Keycloak — Campo senha hidden (skill_login_eproc.mjs)
    O Playwright preenche o campo id="password" (toggle de exibição), mas o campo real é name="password" (oculto). Login retorna erro silencioso de autenticação.
    ✅ Fix: usar page.evaluate() para setar o valor diretamente no campo hidden, ou usar page.fill('[name=password]', senha)
  • 📄 Paginação de processos — skill_listar_processos_eproc.mjs
    Supabase retorna apenas 1000 registros por padrão. O lookup de processo_id só verificava os primeiros 1000 dos 14.642 processos. 29/34 CNJs pareciam "não cadastrados" mas existiam além da página 1.
    ✅ Fix aplicado no cecilia-e2e-v2.mjs: loop com range(offset, offset + 999) até esgotar. Aplicar o mesmo fix nos outros scripts que fazem lookup de CNJ.
  • 🔢 CNJ abreviado em títulos e notificações
    Função gerarTitulo() usava cnj.split('-')[0] gerando "5012667" em vez do CNJ completo. Intimações ficavam com títulos ambíguos.
    ✅ Corrigido no cecilia-e2e-v2.mjs. Verificar nos outros scripts que geram títulos.
  • 🚫 ia_risco_nivel: constraint quebra com valor 'info'
    A constraint intimacoes_ia_risco_nivel_check aceita apenas: baixo, medio, alto, critico. O classificador às vezes retorna 'info', causando erro 400.
    ✅ Fix: mapear 'info' → 'baixo' no classificador. Aplicar em todos os scripts de inserção de intimação.
  • 🔒 Vault Obsidian — permissão negada
    Pastas de ~/mdlab-vault/01_projetos/controller/02-PLAYBOOKS/ têm permissão drwx---r-x. A CECILIA não consegue ler taxonomia, matriz de roteamento nem playbooks.
    ⚠️ Requer Christian executar: chmod -R +rx ~/mdlab-vault/01_projetos/controller/

Fluxo de Monitoramento — Atual vs. Proposto

Refatoração Central

🔴 Fluxo Atual (sequencial, propenso a falha)

Login EPROC
→
Lista processos
→
Acessa processo 1
→
Extrai + Analisa IA
→
Insere no banco
→
Acessa processo 2 ...

❌ Problema: se o processo 3 travar, para tudo. IA é chamada processo a processo (caro). Sem visão do conjunto antes de agir.

🟢 Fluxo Proposto (3 fases paralelas)

Fase 1 — Coleta leve (scraping rápido)

Login EPROC
→
Lista TODAS as páginas
→
Filtra movimentados desde última check
→
Lista de CNJs + última movimentação (sem entrar no processo)

Fase 2 — Triagem IA em lote (1 requisição)

Envia lista de CNJs + eventos para Claude
→
IA decide: quais precisam de leitura de documento?
→
Retorna mapa: CNJ → ação necessária

Fase 3 — Acesso cirúrgico (só os relevantes)

Acessa apenas processos que precisam de doc
→
Download seletivo de PDFs
→
Análise profunda IA
→
Insere intimações + notifica

✅ Vantagens: menos requisições IA, menos tempo de navegação, falhas isoladas por processo, visão do conjunto antes de agir.

Ideia do Christian: "primeiro ele tira a frente de todos os processos que tiveram movimentação sem fazer análise, aí depois em uma única requisição ele pega e processa com IA, e depois ele acessa um a um e já sabe quais documentos ele tem que fazer"

→ Isso é exatamente o que esse fluxo de 3 fases implementa.

📌 Lógica de janela temporal

  • Cada ciclo captura movimentações desde a última execução bem-sucedida
  • Se o ciclo anterior falhou → o próximo captura a janela acumulada (ex: 12h em vez de 6h)
  • Salvar ultima_execucao_sucesso por OAB no Supabase após cada ciclo
  • Processos com erro são re-enfileirados para o próximo ciclo (não bloqueiam os demais)
  • Máximo de páginas: navegar até esgotar — sem limite de página hard-coded

Vision — Resiliência Visual

Confiabilidade

Como a Vision funciona hoje

  • skill_clayton_visual_fallback.mjs — 3 camadas
  • Camada 1: Memória visual salva (zero custo)
  • Camada 2: Playwright DOM selectors (95%)
  • Camada 3: Screenshot → Claude Sonnet → executa
  • Promove seletor para memória após 10 acertos

Melhorias propostas

  • Usar Vision em TODAS as falhas de navegação — hoje é opt-in
  • Fallback automático: se DOM não encontrar em 3s → Vision
  • Incluir Vision na Fase 1 para detectar CAPTCHAs
  • Screenshot comparativo: antes/depois de cada ação crítica
  • Auto-detectar logout via Vision (sessão expirada)
"ver se você tem função de vision também [...] a gente tem que deixar muito bem determinado [...] ela pode operar inclusive pelo nosso computador, pelo usuário, se necessário"

→ A Vision já existe no Clayton. O objetivo é torná-la o fallback padrão em qualquer falha de navegação, não um recurso opcional.

Playwright vs. App Scraping — Análise

Decisão de Arquitetura

✅ Playwright (atual)

  • Controle total do browser real
  • Suporte a JavaScript, SSO, TOTP
  • Vision fallback integrado
  • Pode interagir com qualquer elemento
  • Download de PDFs autenticados
  • Sessão real = cookies válidos

⚠️ App Scraping (alternativa)

  • Mais rápido para dados estruturados
  • Sem overhead de renderização
  • ❌ EPROC não tem API pública
  • ❌ Autenticação Keycloak difícil sem browser
  • ❌ Downloads exigem cookies de sessão
  • ❌ CAPTCHA e 2FA bloqueiam acesso direto
Recomendação: Manter Playwright como estratégia principal. O EPROC foi construído para ser acessado via browser — qualquer abordagem que bypass o browser aumenta o risco de bloqueio e quebra com atualizações do TJRS. O investimento certo é tornar o Playwright mais resiliente (Vision, retry, health check de sessão).

Plano do Teste E2E — 10 Processos × 5 OABs

Próximo Passo

Após correções P1, executar teste E2E completo: 10 últimos processos com movimentação por OAB = 50 processos total.

  • 1

    Corrigir Keycloak (fix campo senha)

    Atualizar skill_login_eproc.mjs. Testar login em todas as 5 OABs. Log: ✅ ou ❌ por OAB.

  • 2

    Listar + filtrar processos com movimentação recente

    Para cada OAB: navegar todas as páginas, ordenar por ultima_movimentacao_em DESC, pegar os 10 primeiros.

  • 3

    Triagem rápida: texto da última movimentação

    Para cada processo: capturar texto do último evento sem entrar no processo. Montar JSON de triagem.

  • 4

    Análise IA em lote (nova Fase 2)

    Uma chamada Claude: todos os 50 CNJs + eventos → retorna classificação, risco, ação recomendada por processo.

  • 5

    Acesso cirúrgico + download

    Para processos com risco >= alto ou tem_documento = true: entrar no processo, baixar doc, análise profunda.

  • 6

    Inserir intimações no banco

    50 processos → banco. Verificar se aparecem no Controller. Registrar CNJs novos (sem processo_id) na tabela processos.

  • 7

    Gerar relatório de diagnóstico E2E

    HTML com: sucesso por OAB, erros encontrados, sugestões de melhoria, tempo de execução, custo estimado de IA.

📝 Gerar relatório ou corrigir inline?

Recomendação: gerar relatório primeiro, corrigir depois.
O E2E vai revelar padrões de erro que só aparecem em escala. Corrigir inline durante o E2E cria uma mistura de versões que dificulta o diagnóstico. Melhor: rodar completo, anotar tudo, fazer um único PR de correções depois.
