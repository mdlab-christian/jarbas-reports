🤝 Análise: Pair Programming com Claude Code

O que foi feito nesta sessão

Esta sessão foi conduzida inteiramente pelo JARBAS usando ferramentas diretas (write/edit/exec) — sem spawn de Claude Code. O motivo: para debug iterativo com ciclo rápido (escreve → testa → analisa erro → corrige), o Claude Code é lento demais. Cada iteração com CC leva 60-90s de boot; com ferramentas diretas, leva 3-5s.

Resultado: 10 bugs corrigidos em ~3h de sessão ativa, com ~25 ciclos de fix/test. Isso seria inviável com CC.

✅ JARBAS Direto — Ideal para:

  • Debug iterativo rápido (3-5s por ciclo)
  • Correções pontuais em arquivos existentes
  • Análise de logs em tempo real
  • Leitura + entendimento de código
  • Testes imediatos após cada fix
  • Bug hunting com stack traces
  • Variáveis de estado entre iterações

🤖 Claude Code — Ideal para:

  • Criação de novos módulos do zero
  • Execução de PRDs grandes (5+ arquivos)
  • Refatoração de codebase inteiro
  • Exploração de estrutura desconhecida
  • Features com muitas dependências
  • Documentação extensa de código
  • Trabalho isolado sem supervisão próxima

Recomendação para próximas sessões

Tipo de TarefaFerramentaMotivo
Debug de skill existente JARBAS direto Ciclo rápido, contexto mantido
Novo skill do zero (>300 linhas) Claude Code Exploração + escrita em paralelo
Análise de resultados / relatórios JARBAS direto Acesso direto ao Supabase + GitHub
Lapidação de página (frontend) Claude Code Precisa de contexto MDFlow completo
Fix de bug com stack trace claro JARBAS direto 2 minutos vs 10 minutos com CC

💡 Conclusão: Pair programming com CC vale a pena em outras conversas? Sim — para features novas. Mas para iteração de debug como esta sessão, JARBAS direto é 5-10x mais eficiente. Use CC quando a tarefa tiver início/meio/fim claros. Use JARBAS direto quando a tarefa for exploratória ou reativa.
