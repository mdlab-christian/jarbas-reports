⚠️ 6. Decisão Arquitetural — Precisa do Christian

🎯 Pergunta central: HTML Supabase ou Google Docs API?

O Make usava Google Docs nativamente ("Create from Template"). O N8n atual usa HTML em documento_templates. Dois caminhos possíveis:

🅰️ Opção A — HTML Supabase (recomendado)

Consistente com a arquitetura atual do N8n

  • ✅ Sem dependência Google Docs API
  • ✅ Consistente com WH-DOC-CONTRATO-MIDAS
  • ✅ Templates editáveis pelo admin no MdFlow
  • ✅ Sem limites de quota Google
  • ❌ Precisamos converter 30+ Google Docs para HTML
  • ❌ Sem geração no Google Drive do cliente
  • ⚠️ Requer definir nova coluna oab_code em documento_templates

🅱️ Opção B — Google Docs API no N8n

Replica exatamente o comportamento do Make

  • ✅ Replica Make 1:1
  • ✅ Mantém docs no Google Drive do cliente
  • ✅ Template já existem e funcionam
  • ❌ Requer Google Drive OAuth configurado no N8n
  • ❌ 2 sistemas de template (HTML + Docs)
  • ⚠️ Depende de credenciais Google no N8n Railway

📝 Perguntas para o Christian

  1. Opção A ou B? HTML Supabase (sem Google Drive) ou Google Docs (com Drive)?
  2. Tabela leads vs clientes: O Make usa leads. O N8n usa clientes. São a mesma? Ou o N8n deve usar leads também?
  3. Armazenamento de URL: O contrato_midas salva em clientes.contrato_midas_url. Para empresa/orgao, onde salvar? Nova coluna? Tabela documentos?
  4. Payload do disparo: Qual será o contrato de chamada? { cliente_id, oab_code, tipo, organizacao_id }?
  5. DDL novo: Adicionar coluna oab_code em documento_templates — pode?
