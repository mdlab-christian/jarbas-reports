2 Validação da Lógica de Filtro de Intimações

A questão central: o sistema capta APENAS intimações com despacho, decisão ou prazo em aberto, para a parte que o advogado representa (autor OU réu)?

✅ O que está correto

✓

Keywords de captura adequadas: INTIMAÇÃO, CITAÇÃO, EXPEDIDA/CERTIFICADA, INTIMADO, CITADO — cobre os principais casos do EPROC RS/SC/SP.

✓

Lista de ignorados (IGNORE_KEYWORDS): exclui movimentações administrativas como "Conclusos para Despacho", "Juntada de Petição", etc.

✓

Deduplicação por SHA-256: hash de CNJ + data_evento + texto evita duplicatas no banco. Erro 23505 é silenciado corretamente.

✓

Urgência automática: keywords EXTINÇÃO, SOB PENA, PRAZO FATAL, 24 HORAS marcam a intimação como urgente = true.

❌ Gaps encontrados

GAP CRÍTICO: Sem filtro por polo (autor/réu) na captura

A skill_captura_intimacoes.mjs captura TODAS as linhas com keywords de intimação, independente de se o escritório representa o autor ou o réu. O polo só é determinado depois pela IA, no analise_bulk. Isso é tecnicamente aceitável, mas gera volume desnecessário no banco e aumenta custo de tokens de IA.

Impacto: Intimações para o réu (ex: "réu intimado a pagar") são capturadas, analisadas pela IA, e só então classificadas como "Ciência com Renúncia". Poderiam ser filtradas antes pela posição processual.

GAP MODERADO: Campos autor/réu extraídos por índice fixo de coluna

O código usa cells[3] para autor e cells[4] para réu — índices fixos que variam conforme versão do EPROC (TJRS vs TJSC vs TJSP têm layouts diferentes). Se o EPROC atualizar o layout, a lógica quebra silenciosamente sem erro óbvio.

GAP MODERADO: extractTexto só para urgentes

O texto completo do documento só é extraído quando extractTexto=true AND intimacao.urgente=true. Intimações não-urgentes ficam apenas com o texto do evento da listagem (~500 chars), que pode ser insuficiente para a IA classificar corretamente (ex: contestação longa).

GAP LEVE: Bug de double-counting de depósitos no analise_bulk

Nas linhas 569-583 do analise_bulk, há lógica de contagem redundante: eventos.depositos++ é chamado antes e depois da chamada handleDeposito(). O contador pode ser incrementado 2x para o mesmo depósito. Não afeta a funcionalidade, mas polui os logs e métricas.

GAP LEVE: Sem filtro de "prazo em aberto" na captura

A skill captura intimações com base em keywords textuais, não pela cor/status do EPROC (Vermelho=aguardando abertura, Amarelo=prazo aberto, Verde=fechado). Intimações com prazo já fechado (verde) poderiam ser descartadas na origem. O atributo nao_lida tenta detectar isso via cor CSS mas é frágil.

8 Nota Final
7.2
/ 10 — Sistema operacional com fundamentos sólidos
Justificativa detalhada:

O sistema de Controller/Distribuição do JARBAS apresenta uma arquitetura bem pensada e modular. O pipeline de 4 camadas (captura → análise IA → feedback → ação) é coerente e cada componente tem responsabilidade clara. O agente controller.md é exemplar — rico em regras processuais específicas, com árvore de decisão clara e roteamento por empresa ré.

O diferencial competitivo do sistema é o loop de aprendizado via skill_controller_feedback: cada decisão humana alimenta o banco de exemplos e melhora automaticamente as próximas análises. Isso é um projeto genuinamente inovador para automação jurídica.

O que segura a nota são os gaps na camada de captura: ausência de filtro por polo (autor/réu) antes da análise IA, extração de campos por índice fixo de coluna (frágil), e a lógica de cor EPROC para prazos abertos/fechados não confiável. Somado a isso, a coexistência de 3 versões do frontend (V1/V2/V3) sem feature parity clara, e a skill de ciência com renúncia que, embora implementada, não tem garantias robustas de sucesso no SP (esaj diferente do EPROC).

O playbook consolidado é excelente e o agente controller está bem calibrado. A maior lacuna técnica é na camada de captura, que é a porta de entrada de todo o sistema.
9 Top 5 Pontos de Melhoria (Priorizado por Impacto/Esforço)
1 Filtro por polo na captura de intimações

Antes de capturar a intimação, verificar na tabela processos se o escritório é autor ou réu (polo_ativo). Intimações para o réu (onde somos autores) → capturar e criar tarefa de ciência com renúncia automaticamente. Intimações para o autor → capturar para análise IA completa. Isso reduz ~30-40% do volume processado pela IA e acelera o pipeline.

Alto impacto Baixo esforço skill_captura_intimacoes.mjs
2 Extração de texto completo para TODAS as intimações (não só urgentes)

Atualmente, o texto completo do documento só é extraído para intimações urgentes (extractTexto=true AND urgente=true). Intimações não urgentes ficam com ~500 chars do evento — insuficiente para a IA classificar contestações longas, saneadores, emendas. Aumentar para todas as intimações melhora drasticamente a acurácia da IA e reduz revisões manuais.

Médio impacto Médio esforço skill_captura_intimacoes.mjs
3 Unificar versões do Controller frontend (V1/V2/V3)

Existe código de 3 versões do Controller no frontend, com componentes duplicados. Isso aumenta a manutenção e pode causar bugs de paridade de features. Definir V3 como única versão suportada, deprecar V1/V2, e garantir que todos os filtros (incluindo polo_intimacao) estejam presentes.

Médio impacto Médio esforço frontend/controller
4 Corrigir bug de double-counting de depósitos + mover filtro pauta para query

Dois bugs menores com impacto real: (1) O contador eventos.depositos pode ser incrementado 2x por depósito — corrigir a lógica de contagem nas linhas 569-583 do analise_bulk. (2) O filtro excluir_pauta_julgamento é feito no cliente após paginação — se paginado, pode mostrar menos itens do que esperado. Mover para uma coluna computada na view.

Médio impacto Baixo esforço analise_bulk + frontend
5 Validação de ciência com renúncia no SP + campo polo_intimacao no frontend

A skill de ciência com renúncia usa seletores genéricos que podem não funcionar no TJSP (esaj.tjsp.jus.br é diferente do EPROC). Implementar fluxo específico para SP. Além disso, adicionar o campo polo_intimacao ao type IntimacaoRow e ao filtro do frontend para que advogados possam filtrar rapidamente.

Médio impacto Médio esforço skill_ciencia_renuncia + frontend types
10 Recomendação de Lapidação Final — Como chegar a 9+/10

Para atingir nota 9+/10, o sistema precisa de 3 frentes simultâneas de melhoria:

🎯

Frente 1 — Filtro inteligente de captura (1-2 dias)

Adicionar lógica de polo na skill_captura_intimacoes.mjs: ao encontrar uma intimação, buscar o processo no banco e verificar se o escritório é parte ativa (autor) ou passiva (réu). Intimações para o réu já teriam tarefa de ciência com renúncia criada automaticamente, sem passar pela análise IA. Reduz custo e acelera pipeline.

🤖

Frente 2 — Automação de ciência com renúncia end-to-end (2-3 dias)

Completar o loop: quando a IA classifica uma intimação como "Ciência com Renúncia", criar automaticamente uma jarbas_tarefa tipo='ciencia_renuncia' sem intervenção humana. Adicionar validação de resultado no EPROC SP (esaj). Adicionar campo polo_intimacao ao frontend para visibilidade. Com isso, a classe mais comum de intimações (~40% do volume) seria 100% automatizada.

📊

Frente 3 — Frontend unificado + filtros avançados (3-4 dias)

Deprecar V1/V2, fixar V3 como único Controller. Adicionar filtros: polo (autor/réu), status da ciência com renúncia, empresa ré. Mover filtro de pauta para a view SQL. Adicionar coluna de confiança da IA (%) no card da intimação para os advogados saberem quando confiar na sugestão da IA sem revisar.

📚

Complementar — Concluir missão de treinamento (1 dia)

O MISSAO.md mostra todas as caixas ainda marcadas como pendentes. Processar os vídeos G1, G3-G10 restantes (alguns já têm relatórios gerados em /treinamentos/) e consolidar no system prompt do agente olivia-controller.md. Meta realista: Com as 3 frentes implementadas, o sistema chegaria a 9.0/10. O 1 ponto restante seria ganho com: seletores EPROC resilientes a atualizações via mapa centralizado JSON, cache do few-shot (Redis/memória), e dashboard de aprendizado IA com taxa de acerto por tipo de intimação.
