O Conceito: Visão Adaptativa (ideia de Christian)
Cleiton vê a tela, pensa e decide — como um humano
Hoje o Cleiton (Controller) usa Playwright engessado: se o EPROC mudou um botão de lugar ou adicionou um passo novo, o script trava. Com Visão Adaptativa, quando o Playwright falha, o Cleiton tira um screenshot da tela, manda para o Claude Vision e pergunta: "O que está acontecendo aqui? Onde devo clicar?" — e segue em frente. E o melhor: salva a decisão para nunca mais precisar perguntar de novo.
👁️
Camada 1 — Playwright normal
Seletores pré-mapeados. Rápido, confiável para fluxos conhecidos. 95% dos casos. Zero custo de API.
🧠
Camada 2 — Fallback Vision
Screenshot → Claude Vision analisa → decide ação → executa. Só ativa quando Playwright falha. Custo: ~R$0,05/tela.
💾
Camada 3 — Memória de Decisões
Decisão que funciona → salvar no banco (cleiton_visual_memory). Na próxima vez que encontrar a mesma tela, usa a decisão salva. Aprende sozinho.
🔄
Camada 4 — Auto-melhoria
JARBAS acompanha decisões de Vision que deram certo → gera novo seletor Playwright baseado nelas → próxima versão não precisa mais de Vision para aquele fluxo.
🎯
Resposta sincera: sim, vai melhorar muito — com ressalvas
O EPROC muda com frequência (atualizações do tribunal). Hoje cada mudança quebra o Cleiton até o João ou Christian mapear o novo fluxo. Com Visão Adaptativa, o Cleiton se recupera sozinho e aprende para as próximas. A ressalva: não usar Vision para tudo — é caro e lento. Só como fallback quando Playwright falha. Estimativa: reduzir de 40% erro atual para menos de 5%.

// Estrutura da tabela de memória visual
CREATE TABLE cleiton_visual_memory (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tribunal    text,        -- 'TJRS' | 'TJSC' | 'TJSP' | 'JF'
  contexto    text,        -- hash da tela (elemento + URL + situação)
  screenshot_hash text,   -- perceptual hash da screenshot
  decisao     jsonb,       -- { acao: 'click', seletor: '#btn-peticionar', coords: [x,y] }
  confianca   int,         -- 1-100 (aumenta conforme acertos)
  acertos     int DEFAULT 0,
  created_at  timestamptz DEFAULT NOW(),
  last_used   timestamptz
);
// skill_cleiton_visual_fallback.mjs
async function adaptiveAction(page, contexto, tribunal) {
  // 1. Tentar decisão salva
  const memória = await buscarMemória(tribunal, contexto);
  if (memória && memória.confianca > 70) {
    await page.locator(memória.decisao.seletor).click();
    await atualizarAcerto(memória.id);
    return;
  }
  
  // 2. Sem memória → Vision
  const screenshot = await page.screenshot({ type: 'png' });
  const decisao = await claudeVision(screenshot, contexto);
  
  // 3. Executar decisão do Vision
  if (decisao.coords) await page.mouse.click(...decisao.coords);
  else await page.locator(decisao.seletor).click();
  
  // 4. Salvar para próxima vez
  await salvarMemória(tribunal, contexto, decisao);
}

Lições dos Vídeos do João — O Que Aprendemos
9 vídeos analisados em 12/03 e 13/03. Relatório completo: relatorio-videos-controller-12-03-2026.html
🔴
TJRS — Distribuição via guia fora da fila
João mostrou que a DR do TJRS requer sair da fila padrão e entrar em uma "guia de distribuição" específica. O Cleiton hoje vai direto pela fila — está errando 100% das vezes no RS. Fix: adicionar passo de navegação para a guia correta antes de distribuir.
Bug crítico TJRSFix: 1 step a mais
🔵
TJSC — Diferença de login e sessão
TJSC tem timeout de sessão mais curto que TJRS. João faz login de novo quando não está dentro de 2h. O Cleiton não detecta sessão expirada e entra em loop. Fix: verificar se está logado antes de cada ação, re-login automático se necessário.
Bug sessão SCFix: session check
🟡
TJSP — 3 passos antes de distribuir
Em SP a distribuição tem 3 confirmações onde RS tem 1. João navega: triagem → confirmação comarca → confirmação competência → distribuição. O Cleiton tenta distribuir direto e falha. Fix: mapear os 3 passos do TJSP como fluxo separado.
3 passos SPFix: fluxo separado
🏛️
Justiça Federal (PJE) — Sistema completamente diferente
O primeiro vídeo de JF gravado pelo João. PJE é um sistema totalmente diferente do EPROC. Login diferente, navegação diferente, campos diferentes. O Cleiton não tem skill para JF ainda. Oportunidade: com Visão Adaptativa, Cleiton pode mapear o PJE sem precisar de seletores pré-programados.
JF PJE novoVision pode mapear
📝
Peticionamento EPROC — Confirmações duplas
João mostrou que protocolar uma petição tem 2 confirmações: "Confirmar tipo" e "Confirmar protocolo". O Cleiton só faz 1. Resultado: petição fica em rascunho, não é efetivamente protocolada. Fix crítico para skill_olivia_eproc_protocolar.mjs.
Bug crítico protocolo2 confirmações
👁️
DRA RS/SC — Intimações de réu adverso exigem atenção
Em recursos, as intimações do adversário também aparecem no EPROC e o João precisa ignorá-las de forma consciente. O Cleiton não distingue intimações do cliente vs do adversário. Fix: filtrar por OAB do advogado responsável antes de processar.

Modal de Intimação — 3 melhorias baseadas nos vídeos
1. Badge "Intimação do Adversário" — quando OAB da intimação não for do escritório, mostrar badge laranja para não processar automaticamente.
2. Link direto para o EPROC — botão "Ver no EPROC" abre direto na intimação correta, sem precisar navegar manualmente.
3. Preview do documento — mostrar as primeiras 3 linhas da intimação no card, sem precisar abrir o EPROC para saber do que se trata.
