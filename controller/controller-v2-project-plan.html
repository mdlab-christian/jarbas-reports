Gaps P0 identificados no deep research
Problemas que travam a integração JARBAS — não estavam no PRD v2 original

🔴 Gap P0-A — controller_responsaveis inexistente

A controller_matriz_roteamento usa texto livre: "Leonardo", "Stephanie", "Heloísa". Quando o JARBAS for criar atividades reais no MdFlow, precisa de user_id — não string. Sem essa tabela, a integração trava na hora de criar a atividade. A tabela precisa existir antes do JARBAS tentar rotear qualquer coisa.

Solução: criar controller_responsaveis com id, nome, codigo, user_id_mdflow, carteiras (jsonb), prazo_padrao_dias, email. Dados dos 9 responsáveis já mapeados no relatório do João.

🔴 Gap P0-B — Estado atual do processo sem fonte definida

A Regra 04 é "SEMPRE cruzar com o status do processo no MdFlow antes de qualquer sugestão". Mas qual tabela o JARBAS consulta? Não existe view consolidada de estado. Sem isso, o JARBAS vai sugerir réplica em processo já arquivado, CS em processo em AJG, etc. Exatamente os erros que já aconteceram nos 3 casos do relatório.

Solução: criar view vw_processo_estado_atual que consolida: status_processo, atividades abertas, atividade pendente, ia_status, ultima_intimacao. JARBAS consulta isso antes de qualquer decisão.

⚠️
Gap P1-A — controller_historico_decisoes sem audit trail
Sem registrar o que o JARBAS decidiu em cada intimação, não tem como auditar erros, medir taxa de acerto real, ou alimentar o feedback loop com evidência. Os 3 casos de erro documentados pelo João seriam impossíveis de rastrear sistemicamente. Adicionado como etapa 1B neste plano.
⚠️
Gap P1-B — Deduplicação sem fingerprint no schema
A Regra 02 define critério (processo + evento + teor + data próxima) mas não há campo fingerprint_hash na tabela intimacoes. O JARBAS não consegue verificar duplicata em O(1). Adicionado como coluna na etapa 1A.

Expandir skill_controller_feedback — endpoint /webhook/feedback
🦞 JARBAS backend · skill complexidade média prioridade 1 depende: etapas 1, 2
Fase 2
O que fazer
Nova função recordFeedback(intimacao_id, aprovado, tipo_correto?):
1. Salva em controller_historico_decisoes.feedback_joao ('positivo'/'negativo')
2. Se aprovado=true → UPDATE jarbas_activity_playbooks SET total_acertos = total_acertos+1
3. Se aprovado=false + tipo_correto fornecido:
  a. Verifica se playbook existe para tipo_correto
  b. Se não existe → cria entrada em jarbas_activity_playbooks com status='aguardando'
  c. Se existe → notifica João via Telegram: "João, o playbook [X] pode precisar de ajuste"
4. Sempre incrementa jarbas_activity_playbooks.total_usos
Critério de aceite
Curl simulando feedback negativo com tipo_correto novo → playbook criado com status='aguardando'.
Curl simulando feedback positivo → total_acertos incrementado no playbook usado.
