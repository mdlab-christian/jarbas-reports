Como o JARBAS vai absorver o conhecimento do controller para substituir o Advbox no MdFlow · 13 mar 2026

🎯 Objetivo

O controller gravou 8 vídeos mostrando o workflow manual que hoje é feito no Advbox (sistema jurídico legado) — que é exatamente a versão antiga da página Controller do MdFlow. Esses vídeos são a fonte de verdade do que precisa ser replicado e automatizado.

O JARBAS vai absorver todo esse conteúdo usando transcrição de áudio + análise visual de frames com Gemini, extrair o workflow passo a passo por tribunal, mapear os gaps entre Advbox e MdFlow, e gerar um relatório com tudo que precisa ser implementado ou melhorado — tanto na página Controller quanto nas skills do JARBAS.

⚙️ Pipeline de Processamento

Cada vídeo passa pelas 4 etapas abaixo. Os vídeos 1–7 (workflow) e o vídeo 8 (melhorias) recebem análises ligeiramente diferentes na etapa 4.

🔊
Etapa 1 — Transcrição de Áudio
Todo o áudio de cada vídeo é transcrito via OpenAI Whisper. O resultado é o texto completo do que o controller falou — instruções, comentários, explicações sobre o que está fazendo na tela, pedidos de melhoria. Esse texto é a base para análise semântica.
OpenAI Whisper ffmpeg (extração de áudio) summarize CLI
🎞️
Etapa 2 — Extração de Frames
Frames são extraídos via ffmpeg em dois ritmos:
• Vídeos de workflow (1–7): 1 frame a cada 20 segundos → captura cada ação na tela sem redundância excessiva
• Vídeo de melhorias (8): 1 frame a cada 15 segundos → mais granular para não perder nenhuma tela mostrada
Frames são salvos em ~/jarbas/playbooks/controller/treinamentos/frames/[video-slug]/
ffmpeg -vf fps=1/20 JPEG 80% quality ~80–100 frames/vídeo
♊
Etapa 3 — Análise Visual com Gemini
Os frames são enviados em batches para Gemini 2.5 Flash com um prompt estruturado que pede:
  • O que está visível na tela (Advbox, EPROC, ou outro sistema)
  • Qual ação o usuário está realizando
  • Campos preenchidos, botões clicados, filtros aplicados
  • Diferenças visuais notáveis em relação ao MdFlow
Contexto injetado: "Advbox é o sistema legado que o MdFlow substitui. Foque em identificar o workflow manual e diferenças de UX/funcionalidade."
gemini-2.5-flash analyze-video-gemini.mjs batch de frames
🧠
Etapa 4 — Síntese e Relatório por Vídeo
Transcrição + análise visual são combinados pelo JARBAS para gerar um relatório estruturado por vídeo:
  • Vídeos 1–7: Workflow passo a passo → mapeamento Advbox vs MdFlow, o que é manual e pode ser automatizado
  • Vídeo 8: Lista estruturada de melhorias/pedidos → prioridade, impacto, esforço estimado
claude-sonnet-4-6 prompt context: PLAYBOOK-CONTROLLER output: markdown report

🔍 O que Será Extraído

🔄 Workflow Manual (Vídeos 1–7)

  • Passo a passo do que o controller faz por tribunal (RS, SC, SP)
  • Diferenças entre OABs (DR Christian vs DRA Heloísa)
  • Como ele identifica intimações relevantes no Advbox
  • Como ele confere no EPROC e decide a ação
  • Quais dados ele preenche manualmente que o JARBAS deveria preencher
  • Tempo estimado por intimação / por sessão

⚡ Automação Possível

  • Ações repetitivas que podem virar skills do JARBAS
  • Regras de decisão implícitas ("se X então Y") usadas pelo controller
  • Filtros e buscas que o JARBAS pode aplicar automaticamente
  • Campos de ciência/resposta que seguem padrão fixo
  • O que hoje leva 30 min que poderia levar 2 min automatizado

🆚 Gaps Advbox → MdFlow

  • Funcionalidades do Advbox que o MdFlow ainda não tem
  • Campos ou visualizações que o controller usa no Advbox mas não existem no Controller do MdFlow
  • Diferenças de UX que causam fricção ou confusão
  • Features que o Advbox tem e o controller usa todo dia

💡 Melhorias Pedidas (Vídeo 8)

  • Lista estruturada de pedidos explícitos do controller
  • Dores e frustrações com o MdFlow atual
  • Priorização por impacto no dia a dia
  • Cada item classificado: Bug / Feature / UX / Automação
  • Entrada direta para o backlog da página-ideas
