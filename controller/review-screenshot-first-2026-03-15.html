Stage 1 — Playwright → screenshot → Storage
9/10 → ✓ Corrigido
crítico ✓ CORRIGIDO Tasks da fila presas em "processando" para sempre
Tasks buscadas com status='pendente' eram marcadas como 'processando' mas nunca atualizadas para 'concluido' ou 'erro' após o batch. A próxima execução do cron nunca as reprocesaria pois não são mais 'pendente'. Acúmulo progressivo de tasks mortas na fila.

skill_analise_visual_eproc.mjs
Stage 2 — Gemini/Claude visual → triagem
8.5/10 → ✓ Corrigido
médio ✓ CORRIGIDO Erros HTTP do Gemini silenciosos
await resp.json() era chamado sem verificar resp.ok. Um erro HTTP 4xx/5xx (chave inválida, quota excedida, rate limit) retornaria um body de erro que falharia silenciosamente no parse JSON, caindo no fallback Claude sem logging adequado. Impossível distinguir "Gemini falhou" de "Gemini respondeu garbage".
Fix: Adicionado if (!resp.ok) throw new Error(\`Gemini HTTP \${resp.status}: \${errText}\`) antes do parse. O fallback para Claude continua funcionando, mas agora com mensagem de erro informativa no log.
info Sem problemas adicionais — Arquitetura correta
Fluxo Gemini Flash → Claude fallback bem implementado. JSON parse com regex seguro. Atualização de movimentações e enfileiramento de Stage 3 corretos. Batch processing com pausa entre análises adequado.
skill_analise_intimacoes_bulk.mjs
Migração Haiku→Sonnet 4.6 — análise em lote
7.5/10 → ✓ Corrigido
crítico lógico ✓ CORRIGIDO handleAlvara() chamado para toda intimação (else branch incorreto)
O bloco else { await handleAlvara(...) } executava para toda intimação que NÃO era depósito/alvará, ou seja, para a maioria esmagadora dos registros. handleAlvara tem guard interno (if (!isAlvara) return) então não causava dano direto, mas adicionava latência desnecessária e mascarava a intenção do código. O counter eventos.alvaras++ dentro do bloco de depósito também estava incorreto.
Fix: Post-processing reescrito: ambos handleDeposito e handleAlvara chamados incondicionalmente (cada um tem seu próprio guard), com contadores baseados nas flags/tipos da análise IA.
crítico lógico ✓ CORRIGIDO Counter eventos.depositos usa padrão dead-code
const beforeDeposito = eventos.depositos; await handleDeposito(...); if (eventos.depositos > beforeDeposito ...) — handleDeposito não tem acesso ao objeto eventos, portanto nunca o modifica. A comparação eventos.depositos > beforeDeposito é sempre false. O contador dependia apenas do "recheck" posterior que também chamava eventos.depositos++ via path diferente, resultando em dupla contagem.
Fix: Contador unificado baseado em raw.flag_deposito_judicial || atividadeTipo === 'deposito_judicial' || titulo.includes('depósito') — uma única source of truth por intimação.
médio ✓ CORRIGIDO Variável MODEL_HAIKU enganosa após migração
const MODEL_HAIKU = 'claude-sonnet-4-6' — nome contradiz o valor. Qualquer dev que leia o código assume que está usando Haiku (barato/rápido) quando na verdade está usando Sonnet (custo 10x maior). Pode levar a surpresas no budget.
Fix: Renomeado para MODEL_ANALISE com comentário de migração preservado.

Filtros novos — Screenshot-First UI
8.5/10 → ✓ Corrigido
médio ✓ CORRIGIDO hasFilters sempre true — estado vazio exibe mensagem errada
Object.values(filters).some(v => v !== null && v !== '' && v !== false...) — arrays vazios como advogado_ids: [] passam todas as condições ([] !== null && [] !== false = true). Resultado: a tela vazia sempre mostrava "Nenhuma intimação encontrada com filtros ativos" mesmo sem nenhum filtro aplicado, nunca exibindo "Tudo em dia por aqui".
Fix: Check refatorado para tratar arrays separadamente: Array.isArray(v) ? v.length > 0 : v !== null && v !== '' && v !== false && v !== undefined.

Recomendações para próximas iterações

🔄
Retry automático para tasks em erro — Adicionar campo retry_count em jarbas_tarefas e reprocessar tasks com status='erro' até 3 tentativas antes de desistir. Hoje tasks com erro ficam paradas.
💰
Monitor de custo Sonnet vs Haiku — A migração Haiku→Sonnet aumentou custo ~10x. Considerar medir input_tokens + output_tokens por batch e logar em tabela de telemetria para controle de budget.
🔐
Hardcoded chat_id nos fallbacks Telegram — O chat_id 1307075495 está hardcoded em ambos os skills como fallback. Deveria ser sempre lido de TELEGRAM_CHAT_ID env var para flexibilidade de ambiente.
📸
Screenshot fullPage=false pode perder movimentações — A captura de screenshot apenas da viewport pode não capturar todas as movimentações se o scroll inicial não atingir a seção correta. Considerar fullPage: true com clip para a área de eventos, ou capturar apenas o elemento #divMovimentacoes.
🧪
Testes E2E do pipeline — Criar teste de integração com screenshot fictício armazenado no Storage para validar o fluxo Stage 1→2→3 sem depender de sessão EPROC ativa.
