Por que o Gustavo não ficou satisfeito:

2 Sistema de Documentos — Análise Profunda

🔴
Este é o ponto central onde o Gustavo não ficou satisfeito. O sistema foi implementado com a arquitetura correta em termos de banco de dados, mas a LÓGICA DE PROPAGAÇÃO (documentos gerais → todas as restrições) está ausente. E a experiência de nomeação de arquivos não segue o padrão esperado.

O que Gustavo pediu nos áudios (Áudios 2, 3, 8, 9, 10, 11)

📁

TIER 1 — Documentos Gerais do Lead (upload central)

⚠️ Lógica incompleta
Áudio 2: "esses documentos é pra ir em todas as páginas, é pra ir depois, em todas as restrições [...] os do 3 até o 5, que o 3 é a identidade ou o CNH, o 4 é o comprovante de residência, o 5 é a carteira de trabalho"
Áudio 10: "documentos gerais do Brasil, eles são ocupados do lead, nível cliente, e depois todos os processos daquele cliente [...] eles são documentos gerais que servem para todos os processos, não para processos específicos. Então, eles podem ir para todas as restrições."

O que foi implementado:

Upload central ✅ categoria: 'documento_lead' ✅ lead_id preenchido ✅ restricao_id=null ✅ invalidateQueries após upload ✅ Propagação automática ❌ Nomenclatura numérica ❌

O gap crítico: Os docs são salvos com restricao_id=null e processo_id=null (correto). Mas quando a distribuição ocorre e os processos são criados, ninguém copia/vincula esses documentos gerais para cada processo criado. A RPC distribuir_lead_completo cria os processos mas não replica os documentos gerais.

🎯

TIER 2 — Documentos Específicos por Restrição

⚠️ Upload faltando
Áudio 2: "tem que aprimorar esse modal de documentos, essas restrições, pra poder enviar documentos por lá também [...] na restrição do Boa Vista é pra ter um upload de documento pra gente mandar o comprovante"
Áudio 10: "documentos específicos (upload por restrição) vão APENAS para aquela restrição/processo"

O que foi implementado:

Seção por restrição no drawer ✅ Botões Gerar Contrato/Procuração ✅ Lista docs gerados por restrição ✅ Upload manual por restrição ❌

O gap: Cada seção de restrição no drawer mostra os botões de gerar, mas não tem campo de upload. O Gustavo pediu poder enviar um comprovante específico para cada restrição (ex: comprovante Boa Vista vai só para o processo do Boa Vista).

⚡

TIER 3 — Documentos Gerados (Contrato + Procuração)

⚠️ Parcialmente funcional
Áudio 2: "as procurações é separada por restrição, porque cada procuração, cada restrição gera uma procuração [...] cada advogado tem procurações e contratos específicos"

O que foi implementado:

Botão Gerar por restrição ✅ Webhook n8n configurado ✅ Cria processo se não existe ✅ 6 workflows ativos ✅ Falha se sem cliente_id ❌ Botão "Abrir" às vezes falha ❌ Auth header incorreto no distribuir ⚠️

O problema do "processo não encontrado": O hook useGerarContrato já foi corrigido — ele cria um processo automaticamente se não existe. MAS só faz isso se o lead já tem cliente_id. Para leads que ainda não foram convertidos (sem cliente_id), o botão aparece mas falha com "Lead ainda não convertido em cliente".

Gustavo pediu que os botões Gerar apareçam a partir do "em atendimento". Hoje os botões aparecem para leads SEM cliente_id mas falham ao clicar.

Lógica de Propagação — O que deveria acontecer

Upload doc geral
restricao_id=null → Distribuição
RPC cria processos → Copia docs gerais
❌ NÃO ACONTECE → processo 1
doc vinculado + processo 2
doc vinculado + processo N
doc vinculado
🔧 Onde corrigir: RPC distribuir_lead_completo
-- No final da RPC, após criar os processos,
-- deve executar algo como:

-- 1. Buscar docs gerais do lead (restricao_id IS NULL)
SELECT id, nome, arquivo_url, tipo_documento, organizacao_id
FROM processos_documentos
WHERE lead_id = p_lead_id
  AND restricao_id IS NULL
  AND categoria = 'documento_lead'
  AND deleted_at IS NULL;

-- 2. Para cada processo criado, copiar os docs gerais
INSERT INTO processos_documentos
  (processo_id, restricao_id, lead_id, organizacao_id,
   nome, arquivo_url, tipo_documento, categoria, status, tipo)
SELECT
  proc.id, proc.restricao_id, p_lead_id, p_organizacao_id,
  doc.nome, doc.arquivo_url, doc.tipo_documento,
  doc.categoria, 'disponivel', 'upload'
FROM novos_processos proc
CROSS JOIN docs_gerais doc;
