🔴 Críticos / Agravados
CRM-09

Deduplicação completamente desativada + zero constraint no banco

✓ Confirmado — Agravado Corrigir M — meio dia
Diagnóstico
🔴
Dedup totalmente comentado. ModalNovoLead.tsx:257: const matches: DedupMatch[] = []; // await checkDedupBeforeInsert(data); — o array é criado vazio, o gate nunca é atingido. O ModalDedup e checkDedupBeforeInsert existem e funcionam, mas são dead code.
🔴
Zero UNIQUE constraint no banco. Query direta em pg_constraint: tabela leads tem apenas leads_pkey (PRIMARY KEY). Nenhum UNIQUE em cpf, whatsapp ou telefone. O banco não impede duplicatas de nenhuma forma.
🟡
checkDedupBeforeInsert busca no banco (não client-side). Faz dois SELECTs: um por cpf = cpfDigits, outro por whatsapp = waDigits — ambos com .is('deleted_at', null). A lógica está correta, só está comentada.
🔴
Sem índice de busca por CPF/WhatsApp. Query confirmou: nenhum index em leads(cpf) ou leads(whatsapp). Quando o dedup for reativado, cada cadastro fará full-scan na tabela.
🟡
Impacto ao reativar. Leads arquivados (deleted_at IS NULL) não são bloqueados — correto. Leads ativos com mesmo CPF geram o ModalDedup, que permite "Continuar assim mesmo" — comportamento controlado pelo atendente.
Solução recomendada (meio dia)
  1. Adicionar índice parcial no banco: CREATE INDEX idx_leads_cpf ON leads(cpf) WHERE deleted_at IS NULL AND cpf IS NOT NULL e mesmo para whatsapp.
  2. Remover o comentário em ModalNovoLead.tsx:257 e restaurar const matches = await checkDedupBeforeInsert(data).
  3. Não adicionar UNIQUE constraint (impede leads arquivados de ter o mesmo CPF reativado). Índice parcial é suficiente para performance.

Follow-up sem indicador de saúde — campo ativo já existe

Diagnóstico
✅
Campo ativo: boolean já existe em crm_followup_config. Confirmado via FollowupConfig interface em useLeadFollowup.ts:16-23 e pela query .eq('ativo', true) na linha 68.
🟡

✅

CRM-10

RLS nas tabelas críticas do CRM

✓ Refutado — RLS está correto Monitorar
Diagnóstico — Verificação direta no banco (pg_tables + pg_policies)
✅
Todas as 6 tabelas têm rowsecurity = true. Confirmado via pg_tables: leads, restricoes, leads_atividades, leads_interacoes, crm_followup_config, processos.
✅
leads — 4 policies CRUD completo, todas usando organizacao_id = org_id(). DELETE exige perfil='admin'. Multi-tenant correto.
✅
leads_atividades + leads_interacoes — 4 policies CRUD cada, todas organizacao_id = org_id(). Sem restrição de perfil para DELETE (qualquer membro pode excluir atividades próprias).
✅
crm_followup_config — 4 policies. DELETE restrito a perfil='admin' — correto para configurações de automação.
✅
restricoes — 1 policy ALL organizacao_id = get_org_id(). Cobre todos os comandos com uma única policy.
🟡
RPCs com SECURITY DEFINER bypassam RLS — intencional e seguro. distribuir_lead_completo, fn_criar_processo_distribuicao e outras RPCs validam p_organizacao_id internamente via parâmetro explícito. O bypass de RLS é necessário para operações atomicas multi-tabela. Risco: código malicioso passando org_id errado — mitigado pelo JWT do Supabase no frontend.
🔵
Inconsistência menor: org_id() vs get_org_id(). Tabelas de leads usam org_id(); restricoes usa get_org_id(). Provavelmente aliases da mesma função — verificar se são idênticas para evitar confusão futura.
Risco de vazamento cross-tenant: BAIXO. RLS está bem configurado em todas as tabelas investigadas. A inconsistência de nomenclatura (org_id vs get_org_id) é cosmética mas vale unificar na próxima migration de manutenção.
CRM-11

Proteção contra double-submit nos modais de distribuição

✓ Refutado — proteção robusta existente Ignorar
Diagnóstico
✅
Três camadas de proteção em ModalDistribuirLead.tsx.
1. if (loading) return (linha 56) — early exit na função.
2. disabled={loading || isLoading || restricoes.length === 0} (linha 214) — botão desabilitado.
3. onOpenChange={loading ? undefined : onOpenChange} (linha 143) — modal não fecha durante processamento.
✅
Loading cobre toda a operação (RPC + loop n8n). O setLoading(false) está no bloco finally — garante reset mesmo em exceção.
✅
Sem indicador de progresso do batch n8n — mas aceitável. O atendente vê "Distribuindo..." durante todo o processo. Para o contexto (max 5-10 restrições por lead), latência total < 5s.
Nenhuma ação necessária. A proteção atual é adequada para o volume de dados do sistema.
CRM-12

Realtime Kanban — funciona, mas sem fallback de polling

◑ Parcial Monitorar
Diagnóstico
✅
Realtime ativo via supabase.channel. useCrmKanban.ts:191-224: watch em leads com filter: organizacao_id=eq.${organizacaoId}, evento * (INSERT+UPDATE+DELETE). Cleanup correto via removeChannel no return do useEffect.
✅
Leads criados por outro atendente aparecem sem refresh. INSERT na tabela leads dispara o channel, que invalida ['crm-leads-kanban', organizacaoId] e o CrmKanban refaz a query na view.
🟡
Conflito de drag simultâneo: last-write-wins. Dois atendentes movendo o mesmo lead ao mesmo tempo fazem UPDATE leads SET coluna_crm=X sem versioning. O segundo sobrescreve o primeiro — sem aviso de conflito. Baixo risco em prática (2 atendentes no mesmo lead simultaneamente é raro).
🟡
Sem fallback de polling. CrmKanban.tsx usa staleTime: 0 mas sem refetchInterval. Se o WebSocket do Supabase cair silenciosamente, o kanban fica desatualizado até reload manual.
🔵
Channel name hardcoded. 'crm-leads-realtime' sem sufixo de org — se o mesmo browser abrir dois orgs (improvável), haveria colisão de channel. Sem impacto prático atual.
Risco real: WebSocket drop silencioso. O Supabase Realtime reconecta automaticamente na maioria dos casos, mas em ambientes de rede instável pode ficar zumbi. Mitigação simples: adicionar refetchInterval: 60_000 na query do CrmKanban como fallback silencioso.
