CRM-01

Falha silenciosa no n8n após distribuição

✓ Confirmado Corrigir M — meio dia
Diagnóstico
Em ModalDistribuirLead.tsx:89–110, após a RPC atômica distribuir_lead_completo ser chamada com sucesso (e retornar processo_ids[]), o código dispara chamadas POST ao n8n em lotes de 5 para gerar petições. O bloco catch (linha 106–109) simplesmente executa um console.warn e continua: 
try {
  await fetch(N8N_DIST_PETICAO_URL, { ... })
} catch {
  // n8n failure não bloqueia — lead já foi arquivado com sucesso
  console.warn('[distribuir] n8n batch falhou, seguindo...')
}
O toast de sucesso (✅ X restrição(ões) distribuída(s)! Lead arquivado.) é exibido incondicionalmente logo após, independente do resultado do n8n. O atendente nunca sabe se as petições foram ou não geradas. Não há retry, não há flag de "petição pendente" no banco, e não há verificação posterior.

Impacto real: processos existem no banco com status ativo mas sem petição inicial. A advogada/equipe jurídica precisaria gerar manualmente ou perceber pela ausência do documento no EPROC.
Solução ideal
  • Contar quantos batches falharam e incluir na mensagem de toast: "3 processos criados, mas 1 petição falhou — gerar manualmente."
  • Salvar flag peticao_pendente=true em processos quando n8n falha, visível no Controller.
  • Alternativamente: não exibir o toast antes do loop n8n terminar — aguardar todos os batches e contar falhas.

CRM-02

Checklist de distribuição não bloqueia prosseguimento

◑ Parcial Monitorar
Diagnóstico
O BETO estava quase certo, mas o diagnóstico é mais sutil. Em ModalDistribuicao.tsx:178–193, o botão "Prosseguir para Triagem" tem: 
disabled={isLoading || loading}
Ele NÃO é desabilitado por itens falhando — a intenção é deliberada. O modal até tem um texto explícito (linha 163–165): 
"Você pode prosseguir mesmo com pendências, mas recomendamos resolver antes."
Também existe um botão secundário "Resolver Pendências" que aparece apenas quando failedCount > 0 && onEditarLead, redirecionando ao modal de edição.

Portanto: o comportamento é by design — gate soft, não hard. A decisão foi explicitamente deixada para o atendente. O risco real existe (distribuir sem advogado ou CPF inválido falhará na RPC), mas a UX avisa e o banco vai rejeitar de qualquer forma se a RPC tiver validação.

Situação de risco identificada: se a RPC distribuir_lead_completo não valida advogado_id NOT NULL no PostgreSQL, um lead sem advogado pode ser distribuído e criar processos sem advogado vinculado.
Recomendação
Verificar se a RPC distribuir_lead_completo valida advogado_id. Se sim, o gate soft é aceitável. Se não, adicionar validação na RPC — não no frontend. A UX atual está correta (avisa sem bloquear).
CRM-03

Drag para "triagem" sem verificar advogado

✓ Confirmado Corrigir P — 1-2h
Diagnóstico
Em useCrmKanban.ts:82–90, a única validação de drag é a verificação de WhatsApp: 
const COLUNAS_REQUER_WHATSAPP = ['em_atendimento', 'aguardando_documentos', 'aguardando_assinatura', 'triagem', 'convertido'];
if (COLUNAS_REQUER_WHATSAPP.includes(novaColuna)) {
  const whatsapp = (lead as any).whatsapp ?? '';
  if (!whatsapp.replace(/\D/g, '')) {
    toast.error('Adicione o WhatsApp antes de avançar o lead', { duration: 4000 });
    // rollback + return
  }
}
Não há verificação de advogado_id. Um lead sem advogado pode ser arrastado para "triagem" sem nenhum aviso. Quando o atendente tentar distribuir via checklist, o ModalDistribuicao marcará "Advogado vinculado" como ❌, mas o lead já está na coluna errada.

A view view_crm_leads_kanban já inclui advogado_id — o dado está disponível no lead em memória, tornando a verificação trivial.
Solução
Adicionar em useCrmKanban.ts, após o check de WhatsApp, uma verificação específica para a coluna "triagem": 
if (novaColuna === 'triagem' && !(lead as any).advogado_id) {
  toast.error('Vincule um advogado antes de mover para Triagem', { duration: 4000 });
  // rollback + return
}
1–2h de trabalho incluindo teste.
CRM-04

JSONB duplicado de restrições (legado vs tabela normalizada)

◑ Parcial Monitorar
Diagnóstico
As duas fontes existem e são usadas em paralelo — o BETO estava certo. Mapeamento real:
  • JSONB (leads.restricoes): usado em ModalDistribuicao.tsx via normalizeJsonbArray() para o checklist de processabilidade.
  • Tabela restricoes: usada em ClienteDrawerTabRestricoes (via useRestricoes hook) e em ModalDistribuirLead (query direta).
A situação é menos grave do que parece: o ModalDistribuicao já faz um check híbrido — linha 69 do checklist: 
passed: restricoesTable.some(r => r.processavel) || hasProcessavel
Ou seja, ele aceita qualquer das duas fontes como válida. A divergência real é que o JSONB pode estar desatualizado (preenchido pela auditoria ApiFull) enquanto a tabela restricoes é a fonte normalizada e editável. Se o usuário editar uma restrição na tabela mas o JSONB não for atualizado, o checklist pode mostrar "Processável" baseado no JSONB stale.

Risco prático: baixo a curto prazo — a tabela restricoes é a fonte canonical para distribuição real. O JSONB é apenas fallback legado no checklist.
Recomendação
Não corrigir agora. Monitorar se surgem discrepâncias em produção. Longo prazo: remover a branch || hasProcessavel do checklist e confiar 100% na tabela restricoes, tornando o JSONB irrelevante para esta lógica.
CRM-05

Timeline de atividades sem paginação (limite 50)

✓ Confirmado Sprint Futuro
Diagnóstico
Em CrmLeadTabAtividades.tsx:105 e :125: 
.limit(50)  // leads_atividades
.limit(50)  // leads_interacoes
A timeline combina as duas (timeline = [...atividades, ...interacoes].sort()), totalizando até 100 eventos. Não há "Carregar mais", infinite scroll ou paginação. Não há aviso ao usuário quando o limite é atingido.

Impacto prático hoje: leads de escritório de advocacia de direito do consumidor com WhatsApp ativo via MDZap podem facilmente ter 50+ mensagens, especialmente em processos de 6+ meses. O atendente perderia histórico de negociação anterior. Para leads novos, é adequado.

Natureza do problema: informacional, não bloqueante. Não afeta operações — apenas auditorias de histórico.
Recomendação
Sprint futuro. Adicionar botão "Carregar mais 50" com offset, ou aumentar o limite para 100 como solução rápida de custo zero. A ausência de aviso é o único ponto a resolver antes: adicionar texto sutil "Mostrando os últimos 50 eventos" quando timeline.length === 100.
CRM-06

Score/AI Report sem geração automática ao cadastrar

✓ Confirmado Monitorar
Diagnóstico
Confirmado: o score só existe se a auditoria for rodada manualmente. O card usa (CrmLeadCard.tsx:50): 
const hasAIReport = lead.score != null && (lead.score as number) > 0;
Quando hasAIReport = false: o badge de score não aparece; o dropdown mostra "Sem relatório IA" (disabled). A opção "Auditar CPF" aparece no dropdown apenas quando lead.cpf && onAuditoriaLead — o caminho para gerar o score está disponível, mas não é proativo.

Não há trigger PostgreSQL, webhook automático ou job n8n criando o score ao inserir um lead. O fluxo é: atendente → card → "Auditar CPF" → ModalNovaAuditoria → n8n roda ApiFull → score persiste no banco.

O impacto operacional depende do processo do escritório. Se auditoria é obrigatória antes de triagem (o checklist não verifica score), o atendente pode esquecer. Se é opcional, o design atual é adequado.
Recomendação
Monitorar. A geração automática de auditoria ao cadastrar tem custo (consumo de créditos ApiFull) e pode ser indesejada para leads descartados. Decisão de negócio do CTO, não técnica. Se quiser sinalizar ausência: adicionar badge sutil "Sem auditoria" no card quando cpf && !hasAIReport.

CRM-07

Follow-up automático depende de job n8n externo

◑ Parcial Sprint Futuro
Diagnóstico
O BETO estava correto que o disparo automático depende de n8n externo. Em useLeadFollowup.ts, o hook apenas:
  • Busca crm_followup_config (regras por coluna: intervalo, max_tentativas, canal)
  • Lê histórico de atividades tipo followup — .limit(20)
  • Permite registro manual de follow-up
O front mostra a config (ex: "Regra: a cada 24h via whatsapp — máx 5 tentativas") mas quem executa o disparo é o n8n. Nenhuma indicação visual de se o job está ativo ou offline.

O campo wa_status (enviado/entregue/lido/respondido/falhou) vem do banco — provavelmente atualizado pelo n8n via webhook MDZap. O front não faz polling em tempo real além do staleTime: 30_000 do React Query.

Impacto: se o n8n estiver offline, os follow-ups automáticos simplesmente não ocorrem. O atendente não tem aviso visual de automação inativa.
Recomendação
Sprint futuro. Adicionar indicador de "automação ativa/inativa" baseado em crm_followup_config.ativo. Se o campo já existe na tabela, é mudança mínima de UI. O status do n8n em si não é acessível diretamente do frontend.
CRM-08

Kanban com limite de 200 leads — aviso enganoso

◑ Parcial — diagnóstico diferente Corrigir (texto) P — 30min
Diagnóstico
A investigação revelou um problema diferente do descrito pelo BETO. LEADS_PER_COLUMN_LIMIT = 200 existe em CrmKanban.tsx:64, mas não é usado para filtrar — é apenas um gatilho de texto: 
// query fetches up to 1000 total:
.limit(1000)

// no render:
{col.leads.length >= LEADS_PER_COLUMN_LIMIT && (
  <p>Mostrando os {LEADS_PER_COLUMN_LIMIT} mais recentes</p>
)}
O problema: a mensagem diz "Mostrando os 200 mais recentes" quando uma coluna atinge 200 leads, mas TODOS os leads ainda são exibidos — a mensagem é tecnicamente falsa, pois nada foi truncado. A query usa .limit(1000) global, então na prática uma coluna pode ter até 1000 leads (improvável, mas possível).

O texto correto seria "Esta coluna tem muitos leads — considere arquivar os inativos" ou similar.
Solução
Corrigir o texto em CrmKanban.tsx:274–276 para algo como: "⚠️ {col.leads.length} leads nesta coluna — desempenho pode ser afetado". P (30min). Ou remover o texto enganoso completamente.
CRM-09

Dedup por email ausente — e dedup INTEIRO está desativado

✓ Confirmado (e agravado) Sprint Futuro
Diagnóstico
O BETO identificou ausência de dedup por email, mas há algo mais grave: o sistema de dedup está completamente desativado. Em ModalNovoLead.tsx:257: 
// TODO: Reativar dedup quando CRM estiver 100% — bloquear CPF duplicado vs base de clientes
const matches: DedupMatch[] = []; // await checkDedupBeforeInsert(data);
A lógica de dedup (CPF + whatsapp) está implementada em checkDedupBeforeInsert() e o ModalDedup existe pronto, mas foram comentados. Duplicatas podem ser cadastradas livremente.

Quanto ao email: mesmo se o dedup for reativado, a função checkDedupBeforeInsert verifica apenas CPF (11 dígitos) e WhatsApp. Email não está na lógica. O email é campo opcional no schema Zod.

Contexto do negócio (negativação indevida): email é canal secundário neste mercado. O dedup por email seria de valor marginal. O dedup por CPF e WhatsApp é o que importa — e está temporariamente desativado.
Recomendação
Sprint futuro: reativar o dedup existente (remover o comentário e restaurar await checkDedupBeforeInsert(data)). Email não precisa ser adicionado — baixo valor para o contexto. Ação: P (30min) quando o CTO decidir ativar.
