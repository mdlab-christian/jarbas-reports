#1
Botão "Baixar" abre arquivo em nova aba em vez de fazer download
DocumentoItem.tsx — handleBaixar  |  ClienteDrawerTabDocumentosLead.tsx — handleBaixarSelecionados
P0 CONFIRMADO
Sintoma
Clicar em "Baixar" abre o arquivo em nova aba igual ao botão "Abrir". Nenhum download acontece.
Causa raiz — código confirmado
🔴 CONFIRMADO — leitura direta do arquivo

handleBaixar (linha 57–61) e handleAbrir (linha 51–55) são idênticos — ambos chamam window.open(url, '_blank'). Não há distinção entre abrir e baixar.

Adicionalmente, handleBaixarSelecionados em ClienteDrawerTabDocumentosLead.tsx (linha 164–176) tem o mesmo problema — também usa window.open(url, '_blank') para os documentos selecionados. Bug #1 tem duas superfícies.

Código problemático
// DocumentoItem.tsx — linhas 51–61
// handleAbrir e handleBaixar são IDÊNTICOS ↓

const handleAbrir = async () => {
  const url = await resolveUrl()
  if (url) window.open(url, '_blank', 'noopener,noreferrer')
  else toast.error('URL não disponível para este documento')
}

const handleBaixar = async () => {
  const url = await resolveUrl()
  if (url) window.open(url, '_blank', 'noopener,noreferrer')   // ← igual ao handleAbrir!
  else toast.error('Documento sem URL disponível')
}

// ClienteDrawerTabDocumentosLead.tsx — linha 173
if (url) window.open(url, '_blank')   // ← mesmo problema no batch
Arquivos afetados
ArquivoLinhaStatusObservação
src/components/shared/cliente_drawer/tabs/DocumentoItem.tsx 57–61 BUG handleBaixar — superfície primária
src/components/shared/cliente_drawer/tabs/ClienteDrawerTabDocumentosLead.tsx 164–176 BUG handleBaixarSelecionados — superfície secundária (nova)
Correção necessária
Fix para DocumentoItem.tsx — handleBaixar
const handleBaixar = async () => {
  const url = await resolveUrl()
  if (!url) { toast.error('Documento sem URL disponível'); return }
  try {
    const res = await fetch(url)
    const blob = await res.blob()
    const a = document.createElement('a')
    a.href = URL.createObjectURL(blob)
    a.download = doc.nome ?? 'documento'
    a.click()
    URL.revokeObjectURL(a.href)
  } catch {
    // fallback: tenta forçar via parâmetro
    window.open(url + '?download', '_blank')
  }
}

Aplicar mesmo padrão em handleBaixarSelecionados no componente principal.

#2
Docs Tier 1 não aparecem nas pastas de restrição após upload
ClienteDrawerTabDocumentosLead.tsx — handleUploadCentral  |  documentos_lead_hooks.ts — useRestricaoDocs
P0 DIAGNÓSTICO REFINADO
Sintoma
Documentos enviados na área central do lead (Tier 1) não aparecem nas pastas de cada restrição imediatamente após o upload. Recarregar a página ou fechar/reabrir o drawer resolve, mas a UX está quebrada.
Diagnóstico v1 vs v2
  • Hipótese 1 do v1 (restricao_id: null não explícito) — DESCARTADA. O insert em handleUploadCentral linha 206 passa restricao_id: null explicitamente.
  • Hipótese 2 do v1 (invalidação incompleta do cache) — CONFIRMADA COMO CAUSA REAL.
  • Finding novo: a lógica de query em useRestricaoDocs está correta — já busca Tier 1 com .is('restricao_id', null). O banco retornaria os dados certos se o cache fosse invalidado.
Causa raiz — código confirmado
🔴 CONFIRMADO — invalidação incompleta de cache React Query

Após upload central (Tier 1), handleUploadCentral invalida apenas o query key ['lead-docs-upload', lead.id, organizacaoId] — que é a lista central.

Os cards de restrição usam o query key ['restricao-docs-upload', restricaoId, leadId, organizacaoId] — nunca invalidado após upload central. O React Query mantém o cache antigo por 30 segundos (staleTime: 30_000), então as pastas não refletem o novo documento.

Código problemático
// ClienteDrawerTabDocumentosLead.tsx — handleUploadCentral (linhas 198–221)
// Após insert bem-sucedido:

toast.success(`"${nomeEscolhido}" enviado`)
await queryClient.invalidateQueries({
  queryKey: ['lead-docs-upload', lead.id, organizacaoId]   // ← apenas lista central
})
// ↑ NÃO invalida ['restricao-docs-upload', restricaoId, ...]
// As pastas de restrição ficam com cache desatualizado por 30s
Correção necessária
Adicionar invalidação das pastas após upload central
// Após o invalidateQueries atual, adicionar:
for (const r of restricoes) {
  queryClient.invalidateQueries({
    queryKey: ['restricao-docs-upload', r.id, lead.id, organizacaoId]
  })
}

restricoes já está disponível no componente (linha 112). Sem custo de rede adicional — apenas invalida o cache, o refetch ocorre quando o usuário abre a pasta.

Erro ao carregar página/modal de processo distribuído
rpc_processo_get_completo — PostgreSQL ERROR 54023 — jsonb_build_object > 100 args
P0 CAUSA RAIZ ENCONTRADA
Sintoma
Ao clicar em qualquer processo na lista (/processos), o modal lateral abre e exibe "Erro ao carregar processo" com botão "Tentar novamente". O erro persiste indefinidamente.
Diagnóstico v1 vs v2
  • v1 dizia "investigação necessária" sem conclusão. v2 encontrou a causa raiz.
  • Causa raiz: rpc_processo_get_completo quebra com erro PostgreSQL 54023 em produção.
  • ProcessoPage (/processos/:cnj) usa useProcessoBySlug — query PostgREST direta, NÃO usa a RPC. Funciona.
  • ProcessoModalShell (modal em /processos, /controller) usa a RPC diretamente — SEMPRE falha.
Causa raiz — erro PostgreSQL confirmado em produção
🔴 PostgreSQL ERROR 54023 — execução real no banco de produção
ERROR: 54023: cannot pass more than 100 arguments to a function QUERY: SELECT jsonb_build_object( -- 40+ campos flat do processo 'id', p.id, 'organizacao_id', p.organizacao_id, 'numero_processo', p.numero_processo, 'numero_cnj', p.numero_cnj, ... (36 campos) -- + 6 relações (clientes, empresas, advogados, ...) -- + 4 counters (total_documentos, total_movimentacoes, ...) -- TOTAL: > 100 argumentos = CRASH ) CONTEXT: PL/pgSQL function rpc_processo_get_completo(uuid) line 8 at SQL statement

O PostgreSQL limita jsonb_build_object() a 100 argumentos (50 pares chave-valor). A RPC tem ~40 campos flat + relações aninhadas = excede o limite em qualquer chamada.

Substituir único jsonb_build_object() por composição com ||

SELECT
  jsonb_build_object(
    'id', p.id, 'organizacao_id', p.organizacao_id,
    'numero_processo', p.numero_processo, 'numero_cnj', p.numero_cnj,

  )
INTO v_result
FROM processos p ...

O operador || de JSONB concatena objetos sem limite de argumentos. Dividir em blocos de ~20 pares cada resolve o limite do PostgreSQL. Requer migration no Supabase.

#4
Upload de imagens (JPG/PNG) não converte para PDF automaticamente
src/hooks/useDocumentUpload.ts — função upload
P1 CONFIRMADO
Sintoma
JPG e PNG são salvos como imagem no Supabase Storage. O sistema jurídico espera PDFs para anexos processuais. Documentos de identidade e comprovantes digitalizados chegam como imagens.
Causa raiz — código confirmado
🟡 CONFIRMADO — ausência de conversão pré-upload

Em useDocumentUpload.ts, o arquivo é enviado diretamente ao storage sem verificação de file.type. Não há função de conversão imagem → PDF. O contentType: file.type é passado literal, mantendo MIME type de imagem.

Nota: este hook é usado pela aba de documentos do processo (pagina_processo). O upload do CRM/lead está em handleUploadCentral diretamente no componente — também sem conversão. Dois pontos de upload sem cobertura.

Arquivos afetados
ArquivoLinhaStatusObservação
src/hooks/useDocumentUpload.ts 34–43 AUSENTE Upload processo — sem conversão
src/components/shared/cliente_drawer/tabs/ClienteDrawerTabDocumentosLead.tsx 198–221 AUSENTE Upload CRM/lead — sem conversão
Correção necessária
Utilitário converterParaPdf() com jsPDF ou pdf-lib
// src/utils/imageUtils.ts — novo arquivo utilitário
import jsPDF from 'jspdf'

export async function converterParaPdf(file: File): Promise<File> {
  const tipos = ['image/jpeg', 'image/png', 'image/webp']
  if (!tipos.includes(file.type)) return file  // já é PDF ou outro

  const url = URL.createObjectURL(file)
  const img = await loadImage(url)
  URL.revokeObjectURL(url)

  const portrait = img.height >= img.width
  const pdf = new jsPDF(portrait ? 'p' : 'l', 'px', [img.width, img.height])
  pdf.addImage(img, 'JPEG', 0, 0, img.width, img.height)

  const blob = pdf.output('blob')
  return new File(
    [blob],
    file.name.replace(/\.(jpg|jpeg|png|webp)$/i, '.pdf'),
    { type: 'application/pdf' }
  )
}

Chamar converterParaPdf(file) antes do upload em ambos os pontos. Requer npm install jspdf. Alternativa mais leve: pdf-lib + canvas.
