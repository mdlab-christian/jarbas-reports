📁 Lógica de Documentos — 3 Tipos
Tipo 1 — Específico por Restrição

Vinculado via restricao_id. Aparece só naquela restrição. Quando a restrição vira processo na distribuição, o documento vai junto para aquele processo específico.

Tipo 2 — Específico por Processo

Vinculado via processo_id. Fica no processo e na restrição vinculada a ele. Não aparece em outros processos do mesmo cliente.

Tipo 3 — Geral do Lead/Cliente

Vinculado via lead_id. Aparece em TODOS os processos e TODAS as restrições do lead. São os documentos pessoais do cliente.

⚠️ Gaps Confirmados — Migrations Necessárias
🔴 restricoes.orgao_id — UUID FK → orgaos_restritivos
Campo orgao_nome (texto livre) existe mas NÃO É suficiente. É preciso vincular via FK para garantir integridade. Gustavo confirmou nos áudios: "tem que ser vinculado ao orgao_id sempre". Etapa E28.
🔴 processos_documentos.restricao_id — UUID FK → restricoes
Para implementar a lógica de documentos por restrição (Tipo 1), é preciso adicionar restricao_id na tabela de documentos. Sem isso, não há como filtrar docs por restrição. Etapa E28.
🔴 processos_documentos.tipo_documento — TEXT
Campo para registrar o nome exato do tipo de documento (ex: "Procuração Empresa", "Contrato Honorários"). Lista definitiva a ser mapeada via Drive (E48). Etapa E28.
⚠️ processos.valor_causa — NUMERIC (verificar se existe)
Campo para armazenar o valor calculado (restrição + R$15.000). Pode já existir — verificar antes de criar migration. Lógica: preenchido automaticamente na RPC de distribuição. Etapa E41.
⚠️ empresas.processavel — BOOLEAN (verificar se existe)
Flag se a empresa é processável ou não. Deve ser puxada no CRM ao cadastrar restrição na auditoria. Pode já existir — verificar. Etapa E42.
⚠️ empresas.categoria_id — UUID FK (verificar se populado)
Categoria da empresa define: qual petição inicial usar, qual OlivIA acionar. Não aparece no front mas é crítico para automação. Verificar se campo existe e está populado. Etapa E42.
