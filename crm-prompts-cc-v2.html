## CONTEXTO Esta é uma tarefa de correção do CRM MdFlow. Uma revisão conjunta (BETO + Claude Code investigação) identificou os seguintes problemas confirmados. Não é exploratória — os arquivos já foram investigados. Execute as correções descritas. Projeto: /Users/beto/beta-mdflow --- ## CORREÇÃO 1 — CRM-01: Feedback de falha n8n após distribuição Arquivo: src/components/crm/modals/ModalDistribuirLead.tsx PROBLEMA CONFIRMADO: Após distribuir o lead (RPC que cria processos e arquiva o lead), o código dispara chamadas ao n8n para gerar petições em batch. Se o n8n falhar (offline, token expirado, timeout), o erro é capturado num catch que faz apenas console.warn e o toast de sucesso é exibido normalmente. O atendente não fica sabendo que as petições não foram geradas. O lead já foi arquivado e os processos criados — isso está correto e não deve ser revertido. O que falta é o feedback adequado. CORREÇÃO DESEJADA: - Após tentar gerar as petições via n8n, verificar se alguma chamada falhou - Se houve falha parcial ou total: exibir um toast de warning (não erro) informando que os processos foram criados com sucesso, mas a geração automática de petições falhou, e que as petições precisam ser geradas manualmente - O toast de warning deve ter duração maior (8-10s) e idealmente um botão de ação se existir uma rota para geração manual - Manter o toast de sucesso da distribuição separado do toast de warning das petições - NÃO reverter a distribuição — apenas informar sobre a falha das petições Exemplo de mensagem para o toast: "Lead distribuído com sucesso! ⚠ Atenção: não foi possível gerar as petições automaticamente. Verifique a conexão com o n8n e gere as petições manualmente." --- ## CORREÇÃO 2 — CRM-03: Verificar advogado ao mover lead para triagem via drag Arquivo: src/hooks/crm/useCrmKanban.ts PROBLEMA CONFIRMADO: A única validação ao mover um lead via drag é checar se o WhatsApp está preenchido (para todas as colunas avançadas). Para a coluna 'triagem' especificamente, o lead deveria ter um advogado vinculado (campo advogado_id na tabela leads). Atualmente é possível arrastar um lead para triagem sem advogado, causando problemas na aprovação jurídica posterior. CORREÇÃO DESEJADA: - No bloco de validação do handleDragEnd, adicionar verificação específica para coluna 'triagem': se o lead não tiver advogado_id preenchido, rejeitar o drag com toast de erro informativo - Mensagem sugerida: "Vincule um advogado antes de enviar o lead para Triagem" - Manter o rollback otimista já existente (mesma lógica do check de WhatsApp) - Verificar se o campo advogado_id existe na ViewLead (view_crm_leads_kanban) — se não existir, consultar como obter essa informação (pode ser que precise adicionar à view ou buscar de outra forma) Se advogado_id não estiver disponível na ViewLead, apenas adicione um comentário TODO indicando que a validação depende de adicionar o campo à view, e documente o que seria necessário. --- ## CORREÇÃO 3 — CRM-08: Texto enganoso sobre limite de 200 leads no kanban Arquivo: src/components/crm/CrmKanban.tsx (ou onde o aviso aparece) PROBLEMA IDENTIFICADO pelo Claude Code: O texto "Mostrando os 200 mais recentes" aparece quando uma coluna tem ≥200 leads, mas a investigação mostrou que a query busca .limit(1000) e todos são renderizados. O texto está errado — nada foi truncado. CORREÇÃO DESEJADA: - Localizar onde esse texto é renderizado - Corrigir para refletir a realidade: ou remover o aviso (se todos são mostrados), ou ajustá-lo para mostrar o número real de leads na coluna sem implicar truncamento - Se existe de fato um limite na renderização (virtualização, etc.), documentar corretamente - Solução mais simples: substituir o texto por "X leads" sem a palavra "mais recentes" se não há truncamento real --- ## ENTREGA Após as correções: 1. Rode o TypeScript check: npx tsc --noEmit 2. Se passar, faça commit com a mensagem: fix(crm): falha silenciosa n8n, validação advogado drag, aviso kanban [CRM-01, CRM-03, CRM-08] 3. Push para o repositório Se encontrar algum impedimento técnico em qualquer correção (ex: advogado_id ausente na view), documente claramente no commit e siga para a próxima correção.

## CONTEXTO Esta é uma tarefa de verificação e hardening do CRM MdFlow. O ModalDistribuicao (frontend) tem um checklist soft que permite prosseguir sem advogado vinculado. O Claude Code investigou e marcou como PARCIAL — a questão é: a RPC distribuir_lead_completo valida advogado_id no PostgreSQL? Projeto: /Users/beto/beta-mdflow --- ## TAREFA — CRM-02: Verificar validação de advogado na RPC de distribuição PASSO 1 — Verifique a RPC no banco: A RPC é chamada como: supabase.rpc('distribuir_lead_completo', { ... }) Localize os parâmetros que ela recebe no frontend (ModalDistribuirLead.tsx) e verifique se advogado_id está entre os parâmetros enviados. PASSO 2 — Acesse a definição da RPC: Procure no projeto por migrations ou arquivos SQL que definem a função distribuir_lead_completo no Supabase. Caminhos prováveis: - supabase/migrations/ - supabase/functions/ PASSO 3 — Determine: A) A RPC recebe advogado_id como parâmetro e valida que não é null? → OK, apenas documentar. B) A RPC recebe advogado_id mas não valida? → Adicionar validação na RPC ou no frontend. C) A RPC não recebe advogado_id? → Identificar se o advogado já está vinculado ao lead antes da distribuição (campo advogado_id na tabela leads) e se a RPC usa esse valor. PASSO 4 — Ação baseada no que encontrar: - Se não há validação em lugar nenhum: adicionar validação no frontend no ModalDistribuirLead (antes de chamar a RPC, verificar se lead.advogado_id está preenchido e bloquear com toast de erro) - Se já existe validação no banco: apenas documentar no código com um comentário - Se a RPC falha naturalmente sem advogado (FK constraint): verificar se o erro é tratado com mensagem amigável para o usuário --- ## ENTREGA Documente o resultado como comentário no próprio código (acima da chamada da RPC no ModalDistribuirLead). Se houver correção de código: commit com mensagem: fix(crm): hardening validação advogado na distribuição [CRM-02] Se não houver correção necessária: commit com apenas o comentário de documentação: docs(crm): documentar validação advogado na RPC distribuição [CRM-02]
