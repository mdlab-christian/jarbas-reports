Máscara maskCNJ formata automaticamente. Após 500ms debounce, chama rpc_validar_cnj_unico. Ícone ✅ verde se único, ✕ vermelho se duplicado — feedback imediato antes do submit.

6
Clica "Confirmar Distribuição"
Chama rpc_efetuar_distribuicao_manual com 11 parâmetros (versão v3, a mais recente). Passa p_estado_id como UF string (ex: "RS") — compatível com coluna estado_id TEXT no banco.
✅ Chama a versão correta (v3 com p_advogado_id)
7
RPC executa (lado banco)
O que a RPC faz:
→ Verifica se processo existe e não está em distribuição paralela
→ Valida CNJ duplicado (dupla checagem server-side)
→ Busca fase "Distribuição" (id: 1ac24ce8 — existe ✅)
→ UPDATE em processos: etapa_distribuicao='distribuido', distribuido=true, fase_id, grau=1, valor_causa calculado
→ UPDATE em processos_agendamentos: marca agendamentos como concluido=true
→ Retorna {'{"success": true}'}
✅ Todas as colunas existem e são compatíveis
8
Frontend recebe sucesso
Toast "Processo distribuído com sucesso!" + confetti 🎉 + invalidateQueries nos 4 caches. Processo some da lista de todos os funcionários instantaneamente via Realtime.
✅ Realtime + cache invalidation corretos
🔍 Checks Técnicos Detalhados
✅
RPC v3 chamada corretamente
Frontend passa p_advogado_id → PostgreSQL resolve overload para a v3 (11 parâmetros). Nenhum conflito com v1 (9 params) ou v2 (10 params).
✅
Tipo de p_estado_id compatível
RPC v3 declara p_estado_id TEXT. Frontend envia UF como string (ex: "RS"). Coluna processos.estado_id é TEXT. Processos já distribuídos têm estado_id = 'RS'. ✓ 100% compatível.
✅
Fase "Distribuição" existe no banco
A RPC busca processos_fases WHERE nome ILIKE '%distribui%'. Encontrou: Distribuição (ordem 2, ativo true). fase_id será preenchida em todos os processos distribuídos.
✅
Juiz — find-or-create funciona
Se funcionário preencher nome do juiz, a RPC busca em julgadores pelo nome (case-insensitive) e cria se não existir. Se comarca selecionada, atualiza comarcas_varas.juiz_titular. Campo opcional — não bloqueia se vazio.
✅
valor_causa calculado automaticamente
RPC define: se valor_causa IS NULL ou 0, preenche com restricao_valor + R$15.000. Nunca sobrescreve valor existente. Automático, sem ação do funcionário.
✅
Agendamentos marcados como concluídos
Após distribuição, a RPC faz UPDATE processos_agendamentos SET concluido=true para o processo. O registro que criamos hoje será marcado corretamente — sem agendamentos fantasmas.
✅
Dupla proteção contra CNJ duplicado
1️⃣ Frontend: rpc_validar_cnj_unico async enquanto digita → feedback visual imediato.
2️⃣ RPC: valida novamente server-side antes do UPDATE → impossível cadastrar duplicado mesmo com dois funcionários simultâneos.
✅
Proteção contra distribuição paralela (JARBAS + funcionário)
Se JARBAS estiver distribuindo automaticamente um processo no mesmo momento, a RPC retorna JARBAS_EM_ANDAMENTO — toast específico, sem crash, sem dados corrompidos.
✅
EXCEPTION handler na RPC
Qualquer erro inesperado retorna {'{"success": false, "error": "mensagem"}'} — nunca lança exceção para o frontend. O modal exibe toast de erro genérico, sem crash.
✅
Realtime atualiza todos os clientes
Supabase Realtime escuta postgres_changes em processos. Após distribuição, invalidate em 4 caches: processos_distribuicao, distribuicao_kpis, distribuicao, processos. Lista de todos os funcionários atualiza instantaneamente.
⚠️
Badge "Vencido" — apenas visual
Os 294 agendamentos criados hoje têm data_agendamento = NOW(). A aba exibirá banner amarelo "X agendamentos vencidos". Não bloqueia nada — é só visual informativo. Distribuição funciona normalmente.
