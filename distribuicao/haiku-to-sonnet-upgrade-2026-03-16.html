🔍 Diagnóstico — Por que o Haiku falha

❌ Limitações do Haiku 4.5

  • Raciocínio contextual fraco — não conecta múltiplas instruções do playbook
  • Extração de dados inconsistente — CPF, valor da causa, endereço da ré às vezes erram com PDFs longos
  • Classificação de documentos falha — confunde COMPROVANTES vs COMPROV RESIDENCIA vs ANEXO
  • Regra comarca ≠ cidade autor — esquece que a comarca é da ré, não do autor
  • Anti-litigância ignorada — não rastreia "1 por empresa por dia" com confiabilidade
  • Sem capacidade de reação visual — quando a tela muda inesperadamente, trava
  • Sem aprendizado — cada erro é idêntico ao anterior, sem memória de solução

✅ O que o Sonnet 4.6 resolve

  • Raciocínio profundo — segue playbook complexo de ponta a ponta
  • Extração robusta — entende contexto jurídico, localiza campos em PDF sem regex frágil
  • Tipificação precisa — entende diferença por advogado (Christian vs Heloísa)
  • Regras de negócio — aplica anti-litigância, anti-CEF, OAB certa por estado
  • Vision nativa — analisa screenshots quando seletor falha
  • Aprendizado persistente — soluções salvas em clayton_visual_memory

⚠️ Problema central identificado: skill_distribuicao_preparar_docs.mjs usa apenas regex pura para extrair CPF, valor da causa, endereço do autor e dados da ré — sem nenhum modelo de linguagem. Isso é a principal fonte de erros de extração.

📋 Plano de Ação — 5 Frentes

Frente 1 — Integrar Sonnet na extração de documentos P0

Substituir regex frágil em skill_distribuicao_preparar_docs.mjs por chamada ao Sonnet 4.6 para extração estruturada da petição inicial.

// NOVO: extractDadosPeticao com Sonnet const resp = await anthropic.messages.create({ model: 'claude-sonnet-4-6', max_tokens: 1024, messages: [{ role: 'user', content: `Extrai do texto da petição inicial (JSON): - cpf (formato: 000.000.000-00) - valor_causa (formato EPROC: 15.758,02) - comarca (cidade da RÉ, não do autor) - endereco_autor (cep, logradouro, numero, complemento, bairro, cidade, uf) - empresa_re (nome, cnpj, cidade, uf) - sexo_autor (M/F) Texto da petição: ${texto.substring(0, 20000)}` }] });

Custo por distribuição: ~R$0,05. Retorno: elimina 80% dos erros de extração.

Frente 2 — Integrar Clayton Visual na distribuição P1

skill_distribuicao.mjs já tira screenshots em cada etapa mas não analisa visualmente quando algo falha. Integrar skill_clayton_visual_fallback.mjs como camada de fallback em cada etapa crítica.

// Em cada etapa: tentar seletor → se falhar → Vision async function executarEtapaComVision(page, etapa, acoes) { try { return await acoes(); // Tenta Playwright normal } catch (e) { // Fallback: Vision analisa screenshot const screenshot = await page.screenshot({ type: 'png' }); return await claytonVisualFallback({ page, tribunal: detectarTribunal(page.url()), contexto: `etapa_${etapa}`, screenshot, instrucao: `Etapa ${etapa} falhou: ${e.message}. O que fazer?` }); } }
Frente 3 — Memória de Erros (Self-Learning) P1

Criar tabela distribuicao_erros_solucoes no Supabase para armazenar erros + soluções. Quando um erro igual ocorrer, aplicar a solução salva automaticamente — sem gastar Vision.

-- Supabase: nova tabela CREATE TABLE distribuicao_erros_solucoes ( id uuid DEFAULT gen_random_uuid(), tribunal text, -- RS | SC | SP etapa text, -- etapa1 | etapa2 | ... erro_pattern text, -- regex do erro solucao_tipo text, -- wait | click | fill | skip | vision solucao_payload jsonb, -- seletor, valor, delay acertos int DEFAULT 0, -- quantas vezes funcionou created_at timestamptz DEFAULT now() );

Clayton já tem isso para Vision geral — aqui é específico para a distribuição com contexto de etapa EPROC.

Frente 4 — Sonnet na validação de regras de negócio P2

Usar Sonnet para validar anti-litigância e identificar casos bloqueados (CEF, 1 por empresa por dia) antes de iniciar a distribuição, com raciocínio explícito.

// Pré-validação com Sonnet const validacao = await sonnet.validar({ cliente: dadosCliente, empresa: dadosRe, restricoes: restricoesHoje, historico: distribuicoesUltimas24h, regras: playbook.regrasBloqueio }); // Retorna: { aprovado: bool, motivo: string, alternativas: [] }
Frente 5 — Relatório visual pós-distribuição P3

Após cada distribuição concluída, gerar mini-relatório com: CNJ gerado, dados extraídos, anomalias detectadas pelo Vision, custo IA utilizado. Publicar no GitHub Pages e notificar.
