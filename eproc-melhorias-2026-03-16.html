Guard "execution context destroyed" — falha silenciosa eliminada

✅ Deployado
❌ Antes

Quando o EPROC fazia redirect inesperado (ex: timeout de sessão), o page.evaluate() jogava exceção não capturada. A tarefa morria com tentativas=0 — sem circuit breaker, sem reagendamento.

✅ Depois

P1

Reagendamento offline para captura_intimacoes sem CNJ

✅ Deployado
❌ Antes

Quando EPROC offline, a captura de intimações falhava e era descartada. O reagendamento existia mas exigia numero_cnj — que a captura não tem. Resultado: ciclo de intimações simplesmente perdido.

✅ Depois

Preflight isEprocAvailable() antes de qualquer login. Se offline → agenda nova tentativa usando oab como chave. Backoff: +2h → +4h → +8h (máx 3x). Nenhuma captura é mais perdida por indisponibilidade.

🔎 Novos pontos identificados nesta rodada

P1 — Novo cerebro_documental sem timeout no orphan recovery

O ORPHAN_TIMEOUT_BY_TYPE no task-consumer tem timeouts para importar_processo, captura_intimacoes, download_documentos etc. — mas não tem para cerebro_documental. Isso significa que uma tarefa cerebro travada cai no timeout padrão de 10 minutos, quando na prática pode levar 30-40min processando 50+ documentos. Resultado observado: tarefa marcada como "Timeout: reenfileirada automaticamente" cedo demais.

💡 Fix: adicionar cerebro_documental: 90 no ORPHAN_TIMEOUT_BY_TYPE — mesmo valor de importar_processo.

P1 — Novo cerebro_documental não tem preflight offline nem reschedule

Após adicionar o reagendamento offline para captura_intimacoes, percebo que o cerebro_documental tem o mesmo problema: se o EPROC estiver offline quando uma tarefa dispara, ela vai direto para erro sem ser reagendada. Além disso, cerebro_documental não está no SKILLS_COM_RESCHEDULE — então erros transitórios (EPROC offline, net, timeout) são descartados sem retry automático.

💡 Fix: (1) adicionar preflight isEprocAvailable() com reagendamento igual à captura. (2) Adicionar cerebro_documental no set SKILLS_COM_RESCHEDULE.

P2 — Novo Sem pool de sessões — browser abre/fecha em cada tarefa

Confirmado: cada tarefa EPROC (captura, cerebro, importar) abre um Chromium novo, faz login (~3-8s) e fecha ao terminar. Com 5 contas e ~250 tarefas/dia, isso gera ~250 ciclos de login. A maioria desses logins é desnecessária porque o perfil persistente já tem cookies válidos — mas o browser é fechado entre tarefas e forçado a fazer o fluxo SSO completo novamente.

Um pool simples de 1 browser por OAB com keepalive e re-login lazy reduziria latência e carga no EPROC (menos hits de login no SSO = menor risco de bloqueio anti-bot).

💡 Fix: criar src/lib/browser-pool.mjs — Map de oab → { browser, page, lastUsed }. Lazy reuse com TTL de 10min. Se expirou → fecha e reabre.
🔍 Causa do erro "OAB RS116571 não encontrada" (11/03)
🟡
Causa provável: circuit breaker temporário, não bug permanente

Em 11/03 às 08:03, cerebro_documental com oab=RS116571 recebeu erro "não encontrada". Hoje a conta está ativa, login_falhas_consecutivas=0 e funcionando (último login às 16:28). A hipótese mais provável: naquele momento a conta tinha falhas consecutivas acima do threshold (≥5) e o loadAccounts() filtrou ela do retorno. O circuit breaker resolveu sozinho quando as falhas foram resetadas. Não é bug — é o sistema funcionando corretamente.
