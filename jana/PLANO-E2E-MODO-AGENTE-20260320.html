Stack e Dependências

ComponenteTecnologiaEndpoint / Path

Fluxo de Dados — Modo Agente

  GeradorStep1 (seleção processo + modelo)
       │
       ▼ processoId, modeloId, versaoId, tipoPeticaoId, advogadoId
  AgenteStep2Analise
       │ POST /olivia/analise-processo → analise_id
       │ poll olivia_analises até status=CONCLUIDO
       ▼ analiseId + analiseResultado
  AgenteStep3Editor
       │ contextoAdicional, qualidadeIa, gerarRelatorio
       │ AgenteChat ↔ POST /olivia/chat (multi-turn)
       ▼ contexto configurado
  AgenteStep4Geracao
       │ POST /olivia/gerar-modo-agente {processo_id, modelo_id, versao_id,
       │   analise_id, qualidade_ia, contexto_adicional, gerar_relatorio}
       │ poll olivia_historico até status=SUCESSO
       ▼ textoGerado, urlPublica, urlPdf, scoreRevisor, tokensUsados
  AgenteStep5Conclusao
       │ exibe texto + links + score + métricas
       ▼ onNovaPeticao → Step 6 OU nova sessão
  AgenteStep6Conclusao
       │ PATCH olivia_historico.feedback_advogado
       ▼ fim da sessão
ℹ️
Sem n8n no Modo Agente. Todo o fluxo é JARBAS (Mac Mini :3012 via Cloudflare tunnel jarbas.jarbas-mdlab.com). Se JARBAS offline → geração falha — não existe fallback automático para Express neste modo.

  • CC corrige: lógica de hooks, bugs de estado, backend calls, novos componentes 21st.dev
  • Prompts CC sempre em accordion com botão copiar — um prompt por bloco lógico
  • Se n8n trigado (não é o caso no Modo Agente) → monitorar via API n8n
  • Sempre verificar Supabase após cada step crítico (Step 2, 4, 6)
  • Se JARBAS offline durante o teste → documentar como issue P1 + tentar retry
  • Decisão crítica de arquitetura → PARAR loop, perguntar Christian, aguardar resposta
  • NUNCA fire-and-forget: se CC demorar >10min sem resposta → alertar Christian
