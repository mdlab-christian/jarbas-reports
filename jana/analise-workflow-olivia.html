Issues Encontrados
P0 — Crítico 21 análises travadas em PROCESSANDO desde 14/03

Todas as análises travadas pararam na mesma fase:

"fase": "Documentos extraídos. Buscando jurisprudência relevante..."
"progresso": 50

Isso indica que o step de RAG/busca de jurisprudência no n8n está falhando silenciosamente — sem disparar a atualização de status para ERRO. O workflow fica suspenso indefinidamente porque:

1. Não há tratamento de erro no step de jurisprudência que force o UPDATE para ERRO no banco

2. O Safety Timeout de 5min no frontend só afeta o estado local (UI) — não altera o banco

3. O constraint do banco (chk_olivia_analises_status) não permite PENDENTE — impossível reprocessar sem criar nova análise

Correção recomendada

No n8n: adicionar Error Handler global no workflow que faça UPDATE olivia_analises SET status='ERRO' em qualquer exceção não tratada. Também considerar adicionar PENDENTE ao constraint para permitir reprocessamento.

P1 — Alto Case sensitivity: frontend compara 'concluida' mas banco salva 'CONCLUIDA'

O frontend verifica o status com letras minúsculas:

// OliviaAnaliseProcesso.tsx + OliviaAnaliseDecisao.tsx
if (data.status === 'concluida' || data.status === 'concluida_parcial') { ... }
if (data.status === 'erro') { ... }

Mas o banco armazena em MAIÚSCULAS (CONCLUIDA, ERRO). O Supabase JS client retorna o valor exatamente como está no banco.

Risco: Se o Realtime ou polling retornar o status em maiúsculas (padrão do banco), o frontend nunca detecta a conclusão — a análise completa no banco mas a UI fica girando infinitamente.

O cache check também usa .eq('status', 'CONCLUIDA') (maiúsculo) — inconsistente com o resto do código.

Correção recomendada

Normalizar no frontend: data.status?.toLowerCase() === 'concluida' — ou padronizar o n8n para sempre salvar em maiúsculas e ajustar todas as comparações.

P2 — Médio Sem status PENDENTE: análises com erro não podem ser reprocessadas

O constraint do banco permite apenas: PROCESSANDO | CONCLUIDA | CONCLUIDA_PARCIAL | ERRO. Não existe PENDENTE para marcar análises para reprocessamento.

Hoje precisei resetar para ERRO (workaround) porque PENDENTE violava o constraint. Isso significa que o advogado/admin precisa criar uma nova análise do zero ao invés de simplesmente reprocessar.

Correção recomendada

Migration: ALTER TABLE olivia_analises DROP CONSTRAINT chk_olivia_analises_status; ALTER TABLE olivia_analises ADD CONSTRAINT chk_olivia_analises_status CHECK (status = ANY (ARRAY['PENDENTE','PROCESSANDO','CONCLUIDA','CONCLUIDA_PARCIAL','ERRO']));

P3 — Baixo tokens_usados e custo_usd sempre zerados

Nas 12 análises concluídas, os campos tokens_usados e custo_usd ficam em 0. Não há rastreamento de consumo de tokens por análise.

Impacto

Impossível calcular custo real por organização. Se o produto escalar, não haverá dados para precificação ou rate limiting.
