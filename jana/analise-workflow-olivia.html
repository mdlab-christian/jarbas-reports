Issues Encontrados — Revisão Nó-a-Nó
P0 — Bloqueante Sem Error Trigger: qualquer falha trava o registro silenciosamente no banco workflow inteiro
O workflow não possui nenhum nó errorTrigger. Todos os 77 nós têm onError: stopWorkflow (padrão). Quando qualquer nó falha, o n8n para a execução, mas o registro em olivia_analises (ou olivia_historico_gerador) fica travado em PROCESSANDO para sempre — sem UPDATE para ERRO.

O OLIVIA-ANALISE-V3 (inativo) tinha um ErrorHandler. Este workflow nunca teve.
Correção
Adicionar nó Error Trigger + nó Code (ErrorHandler) que faça PATCH em olivia_analises e/ou olivia_historico_gerador com status: 'ERRO' + mensagem de erro.
P0 — Bloqueante Unificar OCR acessa formato legado — incompatível com frontend novo Unificar OCR
O nó Extrair URL de petições e provas suporta corretamente dois formatos (A = frontend novo, B = legado). Porém o nó seguinte Unificar OCR acessa diretamente o formato B hardcodado: 
// Hardcodado formato B legado — QUEBRA com formato A
const tipoPeticao = webhookData.documentos.peticao.tipo;
const tiposProvas = webhookData.documentos.provas.map(p => p.tipo);
// Se frontend envia formato A: webhookData.documentos === undefined → TypeError
O frontend OliviaAnaliseProcesso.tsx envia no formato A ({processo_id, arquivos:[...], contexto}). Toda análise disparada pelo frontend novo trava aqui.
Correção
// Detectar formato e normalizar antes de acessar
const body = $('Webhook').first().json.body;
const isFmtA = !body.documentos && (body.arquivos || body.processo_id);
const tipoPeticao = isFmtA ? 'peticao' : body.documentos?.peticao?.tipo;
const tiposProvas = isFmtA ? [] : (body.documentos?.provas || []).map(p => p.tipo);
P0 — Bloqueante :processos filtra por numero_processo mas frontend envia processo_id (UUID) :processos
O nó :processos faz: 
// PROBLEMA: filtra por numero_processo com valor do campo processo.numero
filter: numero_processo = {{ $('Webhook').item.json.body.processo.numero }}

// Mas o frontend envia:
{ processo_id: "48f876f0-d14f-481f-be49-7c096253edac" }  // UUID
// Não existe body.processo.numero → filtro retorna null → :restricoes falha
Correção
// Filtrar por id (UUID) quando formato A, por numero quando formato B
filter: id = {{ $('Webhook').item.json.body.processo_id }}
// ou suportar ambos com lógica condicional no nó de preparação
P0 — Bloqueante :olivia_historico_gerador sem operação configurada :olivia_historico_gerador
O nó Supabase :olivia_historico_gerador está com operação em branco (OP: ""). Sem operação definida, o nó vai retornar erro ou item vazio — todos os nós posteriores que referenciam $(':olivia_historico_gerador').item.json.id vão receber undefined.
Correção
Definir a operação correta: provavelmente create (INSERT) para criar o registro de histórico no início da execução, ou get se o registro já existe.
P1 — Alto Wait sem unit configurada — pode travar ou dar erro de validação Wait
O nó Wait tem amount: 4 mas unit: "" (em branco) e resume: "". Sem unidade de tempo, o n8n pode interpretar como 0 segundos ou lançar erro de validação ao executar. Este nó é usado no loop de chunks quando is_complete é false — ou seja, em toda análise com múltiplos chunks.
Correção
Definir unit: "seconds" e resume: "timeInterval".
P1 — Alto Switch sem fallback: categorias não mapeadas param o fluxo silenciosamente Switch
O Switch tem apenas uma regra: empresa_categoria === "Instituições financeiras". Para outras categorias (Mercado Livre, Telecom, etc.), não há conexão no output main:1 (fallback). O n8n para silenciosamente sem executar nenhum agente nem retornar erro.
Correção
Adicionar output de fallback (default) conectado ao Agente 1 - Petições (ou agente genérico). Ou converter o Switch em If com branch true/false.
P2 — Médio Modelos desatualizados: Agentes 1, 2, 3 usam claude-3-5-sonnet-20241022 Agente 1 · Agente 2 · Agente 3
Os três agentes principais usam claude-3-5-sonnet-20241022 (outubro 2024). O OLIVIA-ANALISE-V3 usava claude-sonnet-4-6 — geração mais recente, melhor performance em análise jurídica.

P2 — Médio OCR Mistral acessa URL de storage sem signed URL — pode retornar 403 OCR - Mistral · C3-FetchArquivo
O nó OCR - Mistral passa a URL do storage diretamente para a API do Mistral: 
document_url: "{{ $json.url }}"
