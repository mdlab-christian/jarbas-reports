Alternativas ao Playwright para Captura EPROC

Avaliação de arquiteturas para monitoramento de movimentações processuais — robustez, custo e prazo

✅ Recomendação

Playwright + HTML Dump + Claude Haiku — não troca o driver, muda a estratégia

O Playwright é a abordagem correta para o EPROC — não existe API pública nem forma de evitar autenticação com sessão real. O problema hoje não é o Playwright em si, é que ele captura DOM instável e usa Vision (screenshot) como fallback para parsing. A solução é inverter: usar scraping de HTML serializado como primário e Vision apenas quando o HTML não tiver o texto estruturado. Isso elimina 90% dos timeouts e é 5× mais rápido que esperar renders visuais.

Abordagens Avaliadas

  • Já implementado
Problemas

  • Overlay/modal quebra fluxo
  • Screenshot para fallback = base64 vazio

Recomendado

Arquitetura Recomendada: Pipeline v5 — HTML Dump First

Fluxo por OAB — ~2 min total

1. Login com TOTP

Igual ao atual — Playwright autentica com 2FA. Sem mudança. skill_login_eproc.mjs

2. Navegar para lista de intimações

Navega para a página de listagem. Sem clicar em nada individual. Rola a página para garantir que todos os itens carregaram.

3. HTML Dump page.content()

Captura o HTML completo da página em memória. Nenhum click, nenhum timeout. const html = await page.content()

4. Claude Haiku parseia HTML

Envia o HTML para Haiku com prompt estruturado: extraia CNJ, tipo evento, data, partes, urgência. Retorna JSON com todas as intimações da listagem em 1 chamada.

~$0.001 por listagem · <3s

5. Filtro inteligente

Aplica regras operacionais do João: descarta antigas, duplicadas, DJEN, pauta. Só processa intimações que passam nos filtros.

6. Busca documento (só se necessário)

Para intimações que passaram no filtro e precisam do texto completo: navega diretamente para o processo pelo CNJ, extrai texto do documento. Só 1 navegação por intimação relevante, não uma por linha da lista.

7. Salva + Classifica + Roteia

Persiste no Supabase, aciona classificador e roteador. Igual ao atual.

Pontos Críticos

💡

Por que HTML e não Screenshot?

O HTML contém todos os dados estruturados (CNJ, datas, links) que o visual não garante. Screenshot é melhor quando o dado está só em imagem (ex: PDF escaneado). Para listagens de tabelas HTML do EPROC, o HTML dump é sempre mais preciso e ~50× mais barato que Vision.

⚠️

Quando usar Screenshot + Vision (fallback)

Manter Vision como fallback apenas quando: (1) page.content() retornar HTML sem tabela de intimações, (2) o Haiku não conseguir extrair dados estruturados do HTML, (3) o documento for um PDF escaneado sem texto selecionável. O bug atual do base64 vazio acontece porque a página ainda está carregando quando o screenshot é tirado — corrigido com waitForLoadState('networkidle') antes do screenshot.

🎯

Treinamentos do João são plenamente compatíveis

As regras operacionais que o João treinou (filtros, tipos de evento, urgência, ciência automática) continuam funcionando igual. A mudança é só na camada de extração de dados — o classificador e o roteador não mudam.

🔄

Sua ideia de Screenshot + Scraping combinado é válida como fallback

O fluxo ideal é: tenta HTML dump → se HTML ok, Haiku parseia. Se HTML falhar (página dinâmica que não renderizou) → screenshot fullpage + Vision. Isso é um fallback em cascata, não o fluxo principal. Assim fica à prova de erros sem pagar Vision em todo caso.
