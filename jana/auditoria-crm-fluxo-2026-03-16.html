Fluxo Completo — Auditoria → CRM

🔍

1. Consulta de Auditoria

Usuário acessa /auditorias → aba Nova Auditoria → informa CPF do cliente. O sistema consulta o bureau (ex: apifull_115) e retorna as restrições encontradas com valores e datas.

📋

2. Histórico e Resultado

Auditoria aparece na aba Histórico com status Concluída e badge de restrição (Processáveis / Limpo / Sem ação). Clicando em ⋮ → Ver resultado abre o modal com resumo: total de restrições, valor e bureau.

🟢

3. Drawer "Cadastrar Restrições"

Botão "Ver Restrições" no modal abre o drawer com todas as restrições encontradas. Cada item exibe:
• Credor (nome normalizado via findBestMatch)
• Bureau, valor, data, contrato
• Badge de certeza: ✓ Certeza / ⚠ Dúvida
• Sugestão de credor correto (linha secundária com botão corrigir)
Usuário seleciona as restrições desejadas e clica em "Cadastrar (N)".

💾

4. Cadastro e Auto-aprendizado

Restrições são salvas na tabela restricoes_clientes vinculadas ao cliente. O sistema aprende os matches credor→nome_normalizado via salvarRestricoes para futuras auditorias do mesmo CPF. Ao concluir, o sistema redireciona automaticamente para o CRM.

👤

5. Lead no CRM

Sistema navega para /crm → Pipeline → Kanban, com o lead recém-criado na coluna "Novos". O lead exibe nome, CPF, estado e indicador de WhatsApp. Abrindo o drawer do lead:

• Aba Dados: nome, CPF, data de cadastro
• Aba Restrições: todas as restrições cadastradas com badge Processável, valor total
• Aba Documentos: upload de documentos + comprovante da auditoria (ver seção de fix)
• Botões: Auditar, Vincular Adv., MDZap, Editar, Triagem

⚖️

6. Triagem

Na aba Restrições do lead → botão "Triagem". Exibe lista de restrições processáveis com opção de incluir cada uma individualmente. Cada restrição incluída gera um processo judicial no sistema. Fluxo: Auditória Concluída → Selecionar restrições → Fazer Triagem → Processos gerados.

Fix Aplicado — Constraint de entidade_tipo

O botão "Criar Lead no CRM" tentava salvar o comprovante da auditoria em processos_documentos com entidade_tipo = 'lead', mas o check constraint da tabela só aceitava 'processo' e 'execucao'. O documento não era salvo e o erro ocorria silenciosamente (sem toast ou log visível ao usuário).

🔧 Solução aplicada — Migration SQL (16/03/2026)

O campo lead_id já existia na tabela — evidência de que a intenção original era suportar leads. O constraint foi atualizado para incluir 'lead':

-- Antes (constraint original)
CHECK (entidade_tipo = ANY (ARRAY['processo'::text, 'execucao'::text]))

-- Depois (fix aplicado em produção)
ALTER TABLE processos_documentos
  DROP CONSTRAINT processos_documentos_entidade_tipo_check;

ALTER TABLE processos_documentos
  ADD CONSTRAINT processos_documentos_entidade_tipo_check
  CHECK (entidade_tipo = ANY (ARRAY['processo'::text, 'execucao'::text, 'lead'::text]));

✅ Migration aplicada diretamente via Supabase Management API em produção. Verificada com pg_get_constraintdef.
