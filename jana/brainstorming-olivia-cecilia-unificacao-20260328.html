Fricção identificada

Christian identificou que há análises de processo que precisam de ambas as perspectivas — monitoramento (Cecília) e estratégia jurídica (OlivIA) — mas não sabe para qual agente encaminhar. A orquestração manual entre dois agentes gera fricção operacional.

O Insight Central

onde está o bug real da arquitetura
Root Cause Analysis

O problema que Christian identificou — "não sei pra qual agente mandar" — não é um problema de arquitetura de agentes. É um problema de interface e roteamento.


Christian não deveria precisar saber que existem dois agentes. Ele deveria mandar qualquer demanda jurídica para um ponto de entrada único — o Jarbas 2.0 — e o Jarbas decide internamente se é: (a) intimação → Cecília, (b) geração de petição → OlivIA, (c) análise cross-domain → Jarbas orquestra os dois.

O Jarbas 2.0 já É esse hub. O problema é que o protocolo de roteamento não está explícito. A solução é definir as regras de handoff, não destruir a especialização.
⚖️ Veredicto do Jarbas 2.0
Análise direta — sem enrolação
Não unifica. Manter OlivIA e Cecília separadas é a decisão correta.

A razão é simples: elas têm naturezas de execução fundamentalmente diferentes. Cecília é um serviço de monitoring — leve, contínuo, Haiku. OlivIA é um motor de geração — pesado, contextual, Sonnet/Opus. Colocar isso no mesmo processo é como rodar um banco de dados transacional no mesmo servidor que um scraper de 24/7. Funciona, mas é um downgrade dos dois.

O que está faltando não é unificação. É um protocolo de handoff claro entre as duas, com o Jarbas 2.0 atuando como roteador central. Christian não deveria nem precisar saber que existem dois agentes — ele manda qualquer demanda jurídica para mim, e eu distribuo.

Exceção válida: se no futuro o fluxo "intimação detectada → análise estratégica → geração de petição" se tornar um pipeline de alta frequência (ex: 50+ petições por dia de forma autônoma), aí pode fazer sentido um agente de pipeline dedicado só para esse fluxo específico — não unificando tudo, mas criando um terceiro agente especializado no flow.

O que implementar

próximos passos concretos
1

Definir regras de roteamento explícitas no Jarbas 2.0

Criar tabela de decisão: qual tipo de demanda vai pra qual agente. Ex: "intimação/prazo/despacho → Cecília", "geração de peça/estratégia/revisão → OlivIA", "análise completa do processo → Jarbas orquestra ambas".

2

Protocolo de handoff Cecília → OlivIA

Quando Cecília identifica intimação que exige petição, ela cria um ticket na tabela jarbas_handoffs (Supabase) com o contexto completo. OlivIA pega o ticket, gera a peça e fecha o loop. Sem intervenção humana para esse fluxo.

3

Interface única para Christian

Christian manda tudo para o Jarbas 2.0 no Telegram. O Jarbas roteia, orquestra, e devolve o resultado consolidado. Christian não precisa saber qual agente fez o quê.

4

Dashboard de status da rede jurídica

Jarbas 2.0 gera relatório diário com: intimações detectadas pela Cecília, petições geradas pela OlivIA, tickets pendentes, KPIs do pipeline. Publicado no GitHub Pages às 23h.

⚠️ Se implementar o handoff automático (Passo 2), começar com notificação obrigatória para João antes de qualquer ação — mesma filosofia da Cecília atual. Validar o pipeline por 30 dias antes de colocar em modo autônomo.

Quando Revisar Esta Decisão

os triggers para reconsiderar
Trigger 1

Volume alto de pipeline

Se o fluxo automático "intimação → petição" ultrapassar 30+ por dia de forma autônoma, criar um agente de pipeline dedicado (terceiro agente, não unificação).

Trigger 2

Latência de handoff crítica

Se o prazo processual for tão curto que a latência do handoff Cecília → OlivIA vira risco jurídico real, aí vale considerar um agente de urgência integrado.

Trigger 3

Multi-tenant MdFlow

Se o MdFlow virar SaaS com múltiplos escritórios, a arquitetura separada se torna ainda mais importante — cada tenant teria sua própria OlivIA, com Cecília compartilhada.
