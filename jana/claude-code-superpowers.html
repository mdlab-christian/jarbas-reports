O gap crítico — Vision para E2E

⚠️ Claude Code não consegue analisar screenshots nativamente

O CC lança o Playwright, captura o screenshot em /tmp/e2e_*.png — mas não consegue ver a imagem para avaliar se a UI está correta, se há sobreposição de elementos, se o modal abriu certo, se o toast apareceu, etc.

Isso é o gap exato que você identificou. E é resolvível com 3 abordagens diferentes.

🟢 Solução A — JANA como Vision reviewer

CC faz o E2E e tira screenshots → envia path via Telegram → JANA analisa as imagens e retorna o veredicto com nota.

Vantagem: Zero custo extra, funciona agora, JANA já tem image tool.
Desvantagem: Loop com Telegram tem latência de ~10s por screenshot.

Disponível hoje Zero config

🔵 Solução B — Script Vision via API Anthropic

CC executa um script Node que chama a API da Anthropic com a imagem base64 e retorna a análise visual diretamente no terminal. CC usa o resultado no fluxo E2E.

Vantagem: Totalmente autônomo, sem Telegram, loop interno do CC.
Desvantagem: Precisa de ~20 linhas de script (simples).

~1h config Recomendada

🟡 Solução C — MCP Vision dedicado

Instalar um MCP server de visão (ex: mcp-vision ou custom) que o CC invoca como tool.

Vantagem: Mais elegante, integrado como tool nativa do CC.
Desvantagem: Mais trabalho de setup, manutenção de servidor MCP.

~3h config Mais robusto
Minha recomendação — Solução B + credenciais no .env

📋 O que implementar

Criar ~/beta-mdflow/.claude/skills/vision-analyzer/ com um script analyze.mjs que o CC invoca assim:

# CC tira screenshot node /tmp/playwright_e2e.js → /tmp/e2e_tela.png # CC analisa com Vision node ~/beta-mdflow/.claude/skills/vision-analyzer/analyze.mjs \ --image /tmp/e2e_tela.png \ --context "Página /processos — modal de criação aberto" \ --expect "Modal aberto com título 'Novo Processo', botão Salvar visível" # Output: JSON com score, issues, veredicto → { score: 9, issues: [], verdict: "APROVADO" }
