🧠 4. Lógica Inteligente — Documentos Duplicados (Proposta Arquitetural)

Gustavo quer que o sistema detecte quando o mesmo tipo de documento já foi solicitado/recebido para o mesmo cliente em outros processos.

✅
O que já existe: ConfirmarDuplicataModal detecta duplicata no mesmo processo. Mostra opções: usar existente, criar mesmo assim, cancelar. Base sólida para expandir.

Regra de Negócio Proposta

A
Duplicata Ativa Cross-Processo
Mesmo cliente + mesmo tipo + status ≠ resolvido em outro processo → aviso: "Já solicitado em processo X". Opções: usar existente / criar novo mesmo assim.
B
Documento Já Recebido (Janela de Validade)
Mesmo cliente + mesmo tipo + status=resolvido + data_entrega ≤ 90 dias atrás → aviso: "Este cliente já possui [Tipo] (recebido em [data], processo [CNJ]). Pode estar válido." Opções: reutilizar / solicitar novo mesmo assim.
C
Resolução em Lote por Similaridade
Ao resolver com upload → verificar se há outras pendências ativas do mesmo tipo para o mesmo cliente. Se sim: "Usar este documento para as outras 2 pendências do mesmo tipo?" com checkbox por processo.
🖥️ Frontend — O que muda
  • Expandir ConfirmarDuplicataModal para aceitar array de duplicatas (multi-processo)
  • Novo estado no NovoDocumentoModal: após submit, verificar cross-processo antes de criar
  • Novo componente ResolveLoteClienteModal — oferece propagar upload
  • Valididade configurável: 90d padrão, vindo de configuração do tenant
🗄️ Backend — O que muda
  • Nova RPC: rpc_check_duplicata_cross_processo(p_org_id, p_cliente_id, p_tipo_id, p_processo_id) → retorna array de pendências similares
  • Opcional: coluna validade_dias INT DEFAULT 90 em documentos_tipos
  • RPC rpc_propagar_resolucao: copia arquivo_url de uma pendência para outras do mesmo cliente+tipo
  • Índice: CREATE INDEX ON documentos_pendentes(organizacao_id, cliente_id, tipo_id) WHERE deleted_at IS NULL

💡 6. Sugestões de Melhoria (Análise Sincera)

P1 — ALTO
Verificar se webhook resolver → n8n faz INSERT em processos_documentos
O frontend está correto: envia arquivo_url no payload. Mas se o workflow n8n de resolver não fizer INSERT em processos_documentos, o documento fica "perdido" — não aparece no /pagina_processo. Abrir o workflow n8n documentos_pendentes/resolver e verificar se esse step existe.
P1 — ALTO
ControllerSheetDocPendente deveria usar webhook, não INSERT direto
Princípio P-03: ações de criação com efeitos colaterais devem passar pelo n8n. Hoje é INSERT direto. OK para agora, mas ao adicionar notificação imediata do responsável, vai exigir refatoração. Considerar migrar para callWebhook('documentos_pendentes/criar').
P2 — MÉDIO
Badge "VENCIDO" visualmente distinto de "CRÍTICO"
Items com prazo no passado e items com prazo daqui 1 dia recebem o mesmo tratamento visual (badge âmbar "X dias" ou "vencido"). Um badge vermelho pulsante para vencidos criaria urgência real e seria instantaneamente legível numa lista de 50+ itens.
P2 — MÉDIO
Contador de pendências por tipo no topo da página (visão analítica rápida)
Além dos 4 KPIs atuais, um mini breakdown por tipo de documento mais frequente ("CNH: 34 | RG: 28 | Holerite: 19") seria valiosa para o gestor entender quais documentos estão bloqueando mais processos.
P2 — MÉDIO
Resolver em lote deveria oferecer campo de observação única
O bulkResolver resolve N pendências de uma vez (Promise.allSettled) mas sem nenhuma observação. Um modal simples "Observação para todas as X pendências?" antes de disparar o lote seria mais auditável.
P2 — MÉDIO
Filtro de prazo na aba "Resolvidos" deveria ser uma barra, não dropdown
O período (30d/90d/ano/todos) para aba resolvidos é um select dentro do componente. Poderia ser chips rápidos mais visíveis — o padrão visual da página são chips, usar select para isso é inconsistente.
P3 — BAIXO
Modo Sequencial deveria mostrar o histórico da pendência atual inline
ModoSequencialSheet tem upload inline, mas para ver o histórico é preciso sair do modo sequencial. Um mini-accordion "histórico" dentro do ModoSequencial removeria esse contexto switch.
P3 — BAIXO
Busca por texto deveria incluir descricao_custom
O normalizeStr client-side busca em: cliente.nome, titulo, documentos_tipos.nome, processos.numero_cnj. Não inclui descricao_custom — que pode conter informações específicas relevantes para busca.
P3 — BAIXO
Animação Framer Motion na BulkBar deveria ter easing mais suave
A barra de ações em lote usa Framer Motion (y animation). Verificar se há bounce residual — bulk bars devem entrar de baixo rapidamente sem quique, pois é UI funcional não decorativa.
