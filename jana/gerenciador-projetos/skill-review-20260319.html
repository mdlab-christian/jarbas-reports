⚙️ Funcionalidades Atuais

🚀 iniciar projeto

Coleta dados, gera preview, cria no banco, cria pasta de screenshots. Oferece gerar PRD geral e etapas iniciais.

  • INSERT em projetos
  • mkdir pasta screenshots
  • Pergunta se quer PRD agora

▶️ executar projeto

Loop de execução: busca próxima etapa pendente, gera prompt CC (5 Pilares), valida, atualiza banco, loop.

  • Respeita depende_de
  • Trava em 2 tentativas falhas
  • Manual ou automático (max 3)
  • Validação vision/e2e/n8n/manual

📋 adicionar etapa

Coleta dados, gera PRD + prompt rascunho com 5 Pilares. Alerta sobre ordenação padrão. Quebra em sub-etapas se complexa.

  • Ordenação sugerida por camada
  • Prompt CC gerado automaticamente
  • granularidade mínima definida

🌐 html projeto

Busca dados + etapas + screenshots. Gera HTML completo com progresso por camada, evidências inline, PRDs em accordion.

  • Publica em GitHub Pages
  • Screenshots inline
  • Progresso por camada

📄 gerar PRD

Gera prompt CC de investigação para documentar o projeto. Resultado salvo em projetos.prd_markdown.

  • Investigação sem alterar código
  • Frontend + Backend + n8n
  • Formato estruturado obrigatório

💡 ideias (projetos_ideias)

Registra, lista e converte ideias em etapas. Diferencia ideias dev vs equipe externa (ideia_equipe).

  • Categorias definidas
  • Ideia → etapa em 2 comandos
  • Filtro por projeto

🔒 anti-alucinação

CC recebe PRD de 1 etapa + resumo geral. Nunca o projeto inteiro. "Se ambíguo → PARA e explica".

  • Max 2 tentativas antes de parar
  • Critério de aceite obrigatório
  • Pontos de falha pré-mapeados

📸 screenshots obrigatórios

CC tira screenshot de cada estado via Playwright. JANA valida com image(). Registrado em projeto_screenshots.

  • Antes/depois/validação/bug
  • JANA valida visualmente
  • evidencia_url na etapa

🔴 Lacunas Identificadas
1
projetos_reports não populado P0 — Corrigido hoje
Os cc_prompts gerados não incluíam instrução de salvar em projetos_reports ao finalizar.

Problema: A tabela projetos_reports foi criada no E11 mas nenhum prompt instrui o CC a salvar nela. Ficava vazia mesmo com etapas concluídas.

Fix aplicado: Adicionado bloco obrigatório "Bloco de encerramento" em todo cc_prompt gerado pela skill — com INSERT em projetos_reports incluindo nota de autoavaliação 0-10.

Status: ✅ Skill atualizada. Próximas etapas já vão incluir automaticamente.

2
Campo revisao em projetos_etapas não usado P1
O campo foi criado pelo E04 (Wave 0) mas a skill não instrui CC a preenchê-lo com QA estruturado.

Problema: O campo revisao JSONB foi planejado como QA por etapa — perguntas, respostas, nota, aprovado. Mas a skill não gera essas perguntas nem instrui o CC a preencher o campo.

Fix sugerido: Após execução de cada etapa, JANA gera checklist de revisão baseado no critério de aceite e tipo de validação, salva em revisao.

3
Progresso % calculado mas não refletido em tempo real P1
O UPDATE de progresso acontece no Passo 7 do loop, mas só depois que JANA valida. Em execuções paralelas (4 terminais), pode ficar defasado.

Problema: Com 4 terminais CC rodando em paralelo, o progresso % pode ser atualizado por um terminal enquanto outro ainda está executando. Pode dar race condition ou % errado.

Fix sugerido: Atualizar progresso apenas com status='concluida' AND testado=true — que só JANA marca. CC não deve atualizar progresso_pct.

4
Sem suporte a multi-terminal explícito P2
A skill foi desenhada para loop sequencial. Multi-terminal paralelo (como OlivIA) não tem instruções específicas.

Problema: Quando Christian roda 4 terminais CC simultâneos, não há instrução na skill sobre como coordenar — quem faz merge de conflito de arquivo, quem tem prioridade, como JANA sabe que etapa foi para qual terminal.

Fix sugerido: Adicionar campo terminal em projetos_etapas (ex: frontend/backend/automacoes/gas) e instrução de como coordenar execução paralela.

5
Sem notificação automática de etapa concluída P2
CC entrega o relatório no terminal, mas não há mecanismo automático de JANA ser notificada.

Problema: Hoje Christian precisa copiar o relatório e enviar manualmente para JANA validar. Em 4 terminais paralelos, isso é trabalhoso.

Fix sugerido: Adicionar ao bloco de encerramento do cc_prompt: curl chamando um webhook ou endpoint que notifique JANA automaticamente com o relatório.

6
Sem histórico de versão de cc_prompt P3
Quando o prompt é refinado (Wave 0 → Wave 1), a versão anterior se perde.

Problema: O campo cc_prompt é sobrescrito. Não há log de versões anteriores do prompt, impossível comparar evolução ou fazer rollback.

Fix sugerido: Adicionar campo cc_prompt_history JSONB[] ou criar tabela projetos_etapas_prompt_history.

💡 Melhorias Recomendadas
A
Suporte nativo a multi-terminal NOVO
Adicionar campo terminal e seção de coordenação paralela na skill

Adicionar à tabela projetos_etapas:

ALTER TABLE projetos_etapas ADD COLUMN terminal TEXT 
  CHECK (terminal IN ('frontend','backend','automacoes','gas','principal','unico'));

E na skill: instrução clara de como JANA coordena quando recebe relatórios de terminais diferentes simultaneamente.

B
Dashboard de progresso em tempo real NOVO
Comando status projeto que mostra o estado ao vivo, sem precisar do HTML completo

Resposta rápida em texto para Telegram — sem precisar gerar HTML. Útil durante execução paralela.
