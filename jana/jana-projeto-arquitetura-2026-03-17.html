💡 O problema que resolve
Por que o modo atual não funciona bem
Perda de contexto: Ideias ficam só na conversa do Telegram. Quando a sessão reseta, some tudo.
Alucinação: Sem base de dados de referência, Jana/JARBAS "inventam" o que foi planejado.
Sem rastreabilidade: Não dá saber o que foi executado, testado, aprovado.
Execução não padronizada: Cada projeto tem um fluxo diferente, sem consistência.
🎯 O que o Modo Projeto resolve
A grande ideia

Tudo que Christian planeja vai para o backend antes de ser executado. Jana e JARBAS lêem do banco — não dependem da memória da sessão. O loop de execução roda tarefa por tarefa, marcando o banco, garantindo que nada é pulado.

📋 Modo Projeto — Como funciona
1️⃣ Fase: Ideação (Christian manda áudio/texto)
Christian entra em modo projeto e despeja tudo — áudios, ideias, problemas, visões. Pode ser longo e caótico. Jana absorve, transcreve e salva cada ideia em jana_ideias (ou jarbas_ideias). Sem filtrar ainda.
2️⃣ Fase: SPEC (Deep Research + GAP + Drift)
Com todas as ideias salvas, Jana roda a skill_gerar_spec. Ela lê do banco, faz deep research, identifica GAPs (o que está faltando), Drifts (o que mudou desde a última decisão), e gera uma SPEC completa — mais detalhada que um PRD. Salva em jana_projetos.
SPEC inclui: contexto, objetivos, critérios de aceitação, stack técnica, dependências, riscos, plano de execução detalhado por camada (frontend, backend, n8n, agente).
3️⃣ Fase: Geração de Tarefas
A SPEC vira tarefas atômicas em jana_tarefas. Cada tarefa tem: camada (frontend/backend/n8n/Jana/JARBAS), componente, o que fazer, critério de aceite, campo concluida e campo testada. Só é marcada concluída quando de fato foi feita. Só é marcada testada quando foi validada (Vision no frontend, teste manual no n8n, etc.).
4️⃣ Fase: Execução em Loop
Jana roda a skill_executar_projeto. Ela lê todas as tarefas do projeto que não estão concluídas, executa uma por uma, atualiza o banco, e segue. Se encontrar bloqueio → notifica Christian. No modo manual: Christian acompanha e Claude Code executa as partes pesadas. No modo automático: roda de noite sem intervenção.
5️⃣ Fase: Teste + Fechamento
Cada tarefa executada passa por teste. Frontend: Vision (screenshot + Claude analisa). n8n: acessa workflow e revisa. Backend: roda query de validação. Só marca testada=true depois de confirmar que funcionou. Projeto fechado = 100% das tarefas concluídas + testadas.
🗺️ HTML de Plano — Formato padrão
Quando Christian gera o plano, o HTML sempre inclui:
• Mapa mental — visualização de todas as ideias e conexões
• Checklist de absorção — confirma que nada foi esquecido do áudio
• SPEC estruturada — o que vai ser feito, por que, como
• Tarefas por camada — frontend, backend, automação, agente
• GAPs identificados — o que ainda falta decidir
• Próximos passos — em ordem de execução

⚙️ Skills a criar
skill_gerar_spec.mjs
Recebe um projeto_id (ou lista de ideias). Lê todas as ideias associadas do banco. Faz deep research se necessário. Identifica GAPs e Drifts. Gera SPEC completa em markdown estruturado. Salva no campo spec do projeto. Gera HTML e publica no GitHub Pages. Retorna link.
anthropic/claude-sonnet-4-6 supabase read/write github pages deep research
skill_criar_projeto.mjs
Cria um projeto em jana_projetos/jarbas_projetos. Vincula as ideias aprovadas. Gera as tarefas atômicas em jana_tarefas/jarbas_tarefas a partir da SPEC. Define ordem, camada, executor e critério de aceite para cada tarefa. Retorna resumo do projeto criado com número de tarefas por camada.
supabase write decompõe SPEC em tarefas calcula dependências
skill_executar_projeto.mjs
O coração do sistema. Lê todas as tarefas não concluídas de um projeto, em ordem. Para cada tarefa: executa (ou prepara prompt pro Claude Code), atualiza banco, valida (Vision/teste), marca concluida + testada. Loop até 100%. Tem dois modos: manual (Christian acompanha, CC executa o pesado) e automático (roda de noite sem intervenção). Notifica Christian no Telegram a cada etapa.
loop de execução modo manual + automático vision para frontend notifica telegram claude code quando necessário
skill_status_projeto.mjs
Dashboard de um projeto em andamento. Lê tarefas do banco, calcula progresso (concluídas/total, testadas/total). Gera HTML de status. Identifica bloqueios (tarefas paradas há muito tempo). Pode ser chamada a qualquer momento por Christian para ver onde está o projeto.
supabase read html de status alerta de bloqueios
🤖 Agente de Projeto
~/.claude/agents/projeto.md
Agente especializado que conhece o Modo Projeto. Sabe ler do banco, gerar SPECs, criar tarefas, executar em loop. É o agente que roda quando Christian usa o Modo Projeto — não o agente genérico. Referencia as skills acima e tem instruções de como orquestrar tudo.
🚀 Dois modos de execução

👨‍💻 Modo Manual

Christian acompanha. Claude Code executa o pesado. Jana/JARBAS orquestram.

  • Christian inicia o projeto
  • Jana gera SPEC + tarefas
  • Para cada tarefa: Jana prepara prompt CC
  • Christian cola no Claude Code (ou Jana spawna)
  • CC executa, Jana valida
  • Banco atualizado em tempo real
  • Christian vê progresso no HTML
  • Melhor para projetos críticos

🌙 Modo Automático

Roda de noite. Sem intervenção. JARBAS orquestra tudo.

  • Christian aprova o projeto antes de dormir
  • JARBAS inicia loop às 00:00
  • Executa tarefa por tarefa
  • CC spawned automaticamente
  • Banco atualizado a cada tarefa
  • Relatório HTML de manhã
  • Notifica Christian no Telegram
  • Melhor para tarefas conhecidas e seguras
