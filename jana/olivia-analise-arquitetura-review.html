🔴 Problemas Reais na Prática
📄
P1 — 70-80% dos documentos são PDFs escaneados
No V3, quando o PDF não tem texto nativo, qualidade = 20 e o Claude analisa sem provas reais. A contestação, a tela sistêmica do Serasa, os contratos físicos — todos escaneados. O V3 simplesmente passa strings vazias para o Claude, que vai inventar ou gerar análise fraca.
FIX: Reintroduzir fallback de Vision (Claude ou Gemini) + Mistral OCR como terceira camada. Sem isso, o workflow falha silenciosamente nos casos mais importantes.
📚
P2 — 100-150 documentos por processo
O loop processa um PDF por vez, em série. 150 PDFs × (fetch URL + extração + possível Vision) = pode levar 10-20 minutos. Além disso, 150 docs × 15k chars = 2.25M chars → muito além do budget de 80k. O V3 silenciosamente ignora tudo acima do budget, sem avisar o usuário.
FIX: Estratégia hierárquica — Claude analisa cada doc individualmente (mini-resumo estruturado) em paralelo, depois análise consolidada. Paralelizar extração com Promise.all. Budget aumentar ou estratégia de seleção inteligente dos docs mais relevantes.
🔗
P3 — Pipeline Gemini → Claude perde contexto
No V1, Gemini extrai dados estruturados → Claude recebe JSON com campos isolados, não o texto original. Claude analisa o extrato de Gemini, não as provas reais. A cada handoff, o raciocínio fica mais raso. Claude Opus no final com input empobrecido é desperdício.
FIX: Claude com Vision lê as provas diretamente (PDF as imagens). Elimina o intermediário Gemini → análise mais profunda com contexto original preservado.
🏷️
P4 — Triagem excessiva quando sem documentos ou docs ruins
Se a IA não lê as provas (escaneadas), confiança em todos os blocos cai para < 0.7. Resultado: 8-12 perguntas de triagem para o usuário responder. O usuário abre o resultado esperando resposta e encontra formulário. Experiência péssima.
FIX: Lógica de confiança revisada — confiança 0.5-0.7 → preenche com melhor bloco + badge "verificar". Só gera pergunta se confiança < 0.3 (realmente sem contexto). Modo "proposta + confirmação" em vez de perguntas abertas.
📊
P5 — Dados ricos do processo não chegam para o Claude
O Supabase tem: restricao_orgao_nome, restricao_valor, restricao_contrato, restricao_data, valor_causa, data_distribuicao, e dados do cliente com restrições históricas. O V3 não injeta esses dados explicitamente no prompt do Claude. O V1 tinha restricoes[] estruturado. Sem esses dados, Claude não consegue cruzar valor da restrição com o que está na contestação.
FIX: Injetar explicitamente no prompt: restricao_orgao, valor_apontamento, data_inclusao, contrato, restrições anteriores do cliente. Claude pode então verificar: "valor da contestação bate com o apontamento?"
💾
P6 — Claude não tem memória entre sessões
Hoje o sistema usa GPT Agent só para armazenar thread_id porque Claude não tem threads. Isso é gambiarr a com ponto de falha adicional (GPT como intermediário).
FIX: olivia_analises já é a memória. Injetar análises anteriores no contexto = Claude lembra via prompt, não via thread. Mais confiável. Eliminar dependência do GPT Agent para memória.
📉
P7 — Tokens/custo sempre zero
Nó Anthropic nativo do n8n não expõe usage. É impossível monitorar custo por análise ou por organização.
FIX simples: substituir nó Anthropic por HTTP Request direto para api.anthropic.com/v1/messages. Response inclui usage.input_tokens e usage.output_tokens.

💡 Resposta à sua pergunta: Claude Opus faz tudo sozinho?
Sim, tecnicamente. E é a melhor opção para o seu caso.

Claude Opus 4+ tem Vision nativo — você envia as páginas do PDF como imagens, ele lê sem OCR intermediário. Para 5-10 docs por processo é viável. Para 150 docs, a estratégia é:

Fase 1 (paralela): Claude Sonnet Vision lê cada doc → gera mini-resumo estruturado (tipo do doc, dados extraídos, relevância jurídica, qualidade). Custo: ~$0.01-0.02/doc.
Fase 2 (única): Claude Opus recebe todos os mini-resumos + contexto completo → análise jurídica final, tags→blocos, triagem, relatório. Custo: ~$0.30-0.80 por análise.

Isso é melhor que Gemini→Claude porque: (1) mesmo ecossistema Anthropic, (2) sem perda de contexto, (3) Opus analisa o que Sonnet extraiu com fidelidade.

RAG — o que está certo e o que pode melhorar
O que funciona: embedding da query com tipo+empresa+tags → fn_rag_search top 8. Boa abordagem.
O que pode melhorar:
💡 Jurisprudência por tipo de réu — filtrar RAG por categoria da empresa, não só score geral. Súmula 385 só é relevante quando há negativação preexistente; não mandar sempre.
💡 Manual de análise de documentos via RAG — você mencionou que a ideia era a IA receber instrução on-demand de como analisar cada tipo de doc. Isso pode ser implementado: ao classificar o documento como "fatura", o RAG busca o chunk de "como analisar faturas" do playbook jurídico. Ideia poderosa — vale implementar.
💡 Petições similares ganhas — se o RAG incluir peças jurídicas de processos similares ganhos, OlivIA pode identificar padrões de argumentação bem-sucedidos.

✅ Recomendações por Prioridade
P0 — Crítico
Reintroduzir Vision + OCR (Gemini Vision ou Claude Vision + Mistral fallback)
Sem isso, 70-80% dos documentos chegam vazios para o Claude. A análise é inútil. Prioridade máxima antes de qualquer outra coisa.
Impacto: sem Vision, o workflow funciona em dev mas falha na prática com clientes reais.
P0 — Crítico
Injetar dados de restrição explicitamente no prompt (orgão, valor, contrato, data)
Claude não consegue cruzar "valor da contestação = valor do apontamento?" sem ter o valor do apontamento. O Supabase tem esses dados em processos. É só uma linha a mais no B1-Process.
Impacto: análise jurídica incompleta sem isso — é o dado mais básico de um caso de negativação indevida.
P1 — Alta
Voltar Claude Opus para análise final (não Sonnet)
Você disse que qualidade é prioridade. O V1 usava Opus — e fazia isso corretamente. V3 mudou para Sonnet, provavelmente para economizar. A diferença em raciocínio jurídico e qualidade de escrita é significativa.
Impacto: análise mais rasa, triagem de menor qualidade, menor confiança nos blocos selecionados.
P1 — Alta
Adicionar campo relatorio_equipe ao schema de saída
O objetivo declarado é que a equipe não precise ler os documentos. Mas o V3 não tem um relatório de provas estruturado — só tem análise. O Gerador Express Etapa 2 precisa disso explicitamente.
Impacto: equipe ainda precisa abrir os PDFs para entender o caso. Derrota o propósito.
P1 — Alta
Classificar tipo de documento antes de extrair
Saber que é uma "fatura" antes de extrair permite: (1) aplicar template de extração específico, (2) buscar playbook correto para aquele tipo, (3) gerar análise especializada. O V1 fazia isso com Gemini (100 tokens, custo mínimo).
Impacto: análise genérica de todos os docs vs análise especializada por tipo.
P2 — Normal
Revisão do threshold de triagem (confiança média → proposta, não pergunta)
Confiança 0.5-0.7 → preenche bloco + flag "verificar". Só gera pergunta aberta se confiança < 0.3. Reduz drasticamente o número de perguntas sem perder cobertura.
P2 — Normal
Corrigir tokens/custo (HTTP Request direto para Anthropic)
Fix de 5 minutos. Substitui o nó Anthropic nativo por um HTTP Request que retorna usage completo. Necessário para monitoramento de custo por organização.
P3 — Evolução
Estratégia hierárquica para 100-150 docs (fase 1: mini-resumos paralelos)
Para volumes grandes: Sonnet Vision em paralelo para cada doc → mini-resumo estruturado → Opus analisa os resumos + docs mais relevantes completos. Escalável, rápido, sem explodir o contexto.
P3 — Evolução
RAG on-demand por tipo de documento (manual de análise)
Ao classificar doc como "fatura", o RAG busca o chunk "como analisar faturas" do playbook jurídico e injeta no prompt. Assim a OlivIA sempre tem o guia correto para cada tipo de prova — on-demand, como você idealizou.
