🐛 Bugs Corrigidos no V3

P0   C3-FetchArquivo: GET → POST para signed URL
A API Supabase Storage exige POST /storage/v1/object/sign/processos/{path} com body {"expiresIn": 3600}. O node usava GET, retornando 404/405. 100% das análises com PDFs falhavam silenciosamente.
Fix: método GET → POST + header Content-Type adicionado.
P0   Step de download binário ausente (C3 → C4)
C4-ExtractPDF espera dados binários ($binary.data), mas C3 retornava JSON com signedURL. Nenhum texto era extraído dos PDFs.
Fix: novo node C3b-DownloadPDF adicionado (HTTP GET da signed URL com responseFormat: "file"). Conexões atualizadas: C3 → C3b → C4.
P1   meta JSONB double-stringify em 4 nodes PATCH
Nodes B4-UPDATE-Classificando, D2-UPDATE-50pct, E1-UPDATE-60pct, F3-UPDATE-100pct enviavam JSON.stringify({meta: JSON.stringify({progresso: 20})}) → meta chegava como string scalar no Supabase. Polling de progresso no frontend (20%/50%/60%/100%) não funcionava — meta->>'progresso' retornava null.
Fix: JSON.stringify() interno removido em todos os 4 nodes.

⚠️ Issues em Aberto no V3

P1   ErrorHandler: extração de analise_id frágil
Caminho error.execution?.data?.resultData?.runData?.['A4-INSERT-olivia_analises']?.[0]?.data?.main?.[0]?.[0]?.json?.id quebra com atualizações do n8n.
Recomendação: usar $execution.customData.set('analise_id', ...) logo após A4-INSERT.
P1   Sem limite de docs em E2-PrepararInput-Claude
Cada doc limita a 15k chars mas sem limite de quantidade. Com 10+ PDFs pesados pode estourar a context window do Claude.
Recomendação: limitar total a ~100k chars ou top-N docs por qualidade.
P2   F4-LogWebhook: só console.log
Node não atualiza webhook_requests com status final como o nome sugere.
P2   D3-GerarEmbedding: rag_query vazia → 400 OpenAI
Edge case: processo sem empresa e sem tags → input: "" enviado para OpenAI → 400 Bad Request. Fallback via continueOnFail em D4, mas RAG fica vazio.
Recomendação: fallback de query genérica: "análise jurídica processo consumerista".

🏗️ Arquitetura: V2 vs V3

V2 — Pontos negativos

  • 4 provedores de IA (Gemini + Claude + Mistral) — manutenção complexa
  • 50 nodes — mais verboso
  • P0s bloqueantes que fazem 100% das análises com PDF falhar
  • Progress bar broken por double-stringify
  • Sub-workflows (executeWorkflow) adicionam latência
  • Fluxo bypass sem PDFs não claro
  • Context passthrough inconsistente

V3 — Pontos positivos

  • Stack única: Claude Sonnet 4-6 para tudo
  • 43 nodes — mais enxuto e legível
  • P0s corrigidos — extração de PDF funcional
  • Progress bar 20/50/60/100% funcionando
  • Fases A-F claramente nomeadas
  • Context passthrough consistente via PassCtx
  • Bypass limpo para processos sem documentos
  • Output estruturado completo para frontend

🚀 Recomendações — Próximas Versões

1. ErrorHandler com customData (V3.1)
Substituir extração frágil de analise_id por $execution.customData.set('analise_id', id) logo após A4-INSERT. Torna o error workflow robusto entre versões do n8n.
2. Download direto do Storage (V3.1 — performance)
Substituir signed URL (C3 POST + C3b GET) por download direto: GET /storage/v1/object/processos/{path} com service_role_key. Elimina C3b e reduz 1 request por PDF.
3. Limite de docs em E2 (robustez)
Limitar total de chars a ~100k ou top-N docs ordenados por qualidade. Previne estouro da context window com processos com muitos PDFs.
4. Fallback RAG query (edge case)
Quando rag_query vazia (processo sem empresa/tags), usar fallback genérico: "análise jurídica processo consumerista" em vez de enviar string vazia para OpenAI.
5. F4-LogWebhook real (qualidade de auditoria)
Atualizar webhook_requests com status final (CONCLUIDA/ERRO) e analise_id para rastreabilidade completa.

V3 supera V2 em todos os critérios críticos
