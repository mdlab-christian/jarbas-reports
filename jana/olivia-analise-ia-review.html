🖥️ 4. Análise do Frontend — Steps 1 e 2
1

Step 1 — Identificação + Upload de Provas

8.5/10

O que funciona bem ✅

  • Upload inline via UploadProvasInline — Realtime + poll a cada 5s
  • Disparo automático do webhook olivia/analise-decisao 1s após upload
  • Análise em background — não bloqueia o avanço do wizard
  • Feedback visual durante análise ("Analisando em background...")
  • Timeout de segurança 3 minutos com toast de erro
  • Seleção de tipo de prova por arquivo (extrato, contrato, etc.)
  • Validação analise_obrigatoria via olivia_modelo_tags
  • Aviso visual quando processo sem documentos

Gaps encontrados ⚠️

  • ⚠️ Upload chama olivia/analise-decisao mas o webhook real é mdflow/olivia/analise — verificar mapeamento
  • ⚠️ Sem progress bar do processamento (0%→100%) — usuário não sabe quanto falta
  • ⚠️ Máximo 5 arquivos / 10MB por arquivo — pode ser limitante para casos com muita documentação
  • ⚠️ Sem preview de PDFs carregados antes do upload
2

Step 2 — Triagem: Relatório IA + Perguntas

7.5/10

O que funciona bem ✅

  • Componente TriagemAnaliseCard exibe resultado antes das perguntas
  • Mostra: score, pontos favoráveis, riscos, lacunas, tese sugerida
  • Aguarda análise concluir antes de exibir (poll 2.5s, max 30s)
  • Campo instruções adicionais sempre disponível
  • Autosave a cada 30s via useAutosave
  • Animação fluida entre perguntas (framer-motion)
  • Progress bar entre perguntas
  • Tecla Enter navega entre perguntas
  • Botão "Pular triagem" após 15s timeout

Gaps críticos ⚠️

  • ⚠️ TriagemAnaliseCard não exibe perguntas_triagem do workflow — as perguntas do card são da tabela de templates, não as geradas pela IA
  • ⚠️ Score do card usa score/100 mas workflow retorna chance_sucesso 1-3 — mapeamento quebrado
  • ⚠️ Campo visao_geral não exibido no card (apenas score + listas)
  • ⚠️ Sem distinção visual entre "análise sem PDFs" vs "análise com PDFs"
  • ⚠️ Sem exibição de documentos_faltantes no Step 2

🚨 Disconnect crítico identificado

As perguntas do StepTriagem vêm do webhook mdflow/olivia/triagem (que usa fn_olivia_get_triagem_by_grupo), não das perguntas_triagem geradas pelo OLIVIA-ANALISE-V3. Isso significa que mesmo com 9 perguntas contextuais geradas pelo Claude, o Step 2 provavelmente exibe as perguntas genéricas do template — perdendo todo o contexto probatório gerado.

📋 5. Plano de Melhorias
🚨

[P0] Usar perguntas_triagem da análise IA no Step 2

O StepTriagem deve verificar se existe analise_resultado.resultado.perguntas_triagem e, se sim, usá-las como perguntas ao invés de chamar mdflow/olivia/triagem. As perguntas geradas pelo Claude são contextualizadas ao caso específico — muito superiores às genéricas do template.

P0 — Crítico StepTriagem.tsx ~2h dev
🚨

[P0] Corrigir mapeamento score no TriagemAnaliseCard

O workflow retorna chance_sucesso: 1|2|3 (fraco/médio/forte) mas o card espera score: 0-100. Mapear: 1→25, 2→60, 3→90 (ou exibir o label textual) para que a barra de progresso faça sentido visualmente.

P0 — UX quebrado TriagemAnaliseCard.tsx ~30min dev
⚠️

[P1] Corrigir captura de tokens/custo no E4-ParseResultado

O nó Anthropic do n8n entrega usage dentro de content[].usage (não output.usage). Corrigir E4 para iterar content array e extrair input_tokens e output_tokens. Essencial para rastrear custo por análise.

P1 — Custo E4-ParseResultado (n8n) ~20min dev
⚠️

[P1] Adicionar visao_geral e documentos_faltantes no TriagemAnaliseCard

O campo visao_geral contém o resumo narrativo completo do caso — deve ser o primeiro item exibido no card. Os documentos_faltantes (8 no E2E) são altamente relevantes para o advogado saber o que pedir antes de gerar a petição.

P1 — UX TriagemAnaliseCard.tsx ~1h dev
⚠️

[P1] Verificar endpoint: olivia/analise-decisao vs mdflow/olivia/analise

O UploadProvasInline chama olivia/analise-decisao mas o workflow real está em mdflow/olivia/analise. Verificar se há roteamento/alias configurado no n8n ou se o Step 1 está disparando para o workflow correto.

P1 — Conectividade UploadProvasInline.tsx ~1h investigação
💡

[P2] Progress bar real no Step 1 durante análise

O workflow já atualiza progresso_pct... mas verificamos que o campo não existe na tabela. Adicionar coluna progresso_pct int default 0 em olivia_analises e usar os UPDATEs que já existem no workflow (50%, 60%, 100%). Frontend assina via Realtime e exibe barra real.

P2 — UX Migration + UploadProvasInline ~2h dev
💡

[P2] fase_processual: detectar quando sem PDFs

O campo retorna vazio quando não há documentos. O workflow deveria inferir a fase processual a partir dos metadados do processo (grau, fase_id, ultima_decisao_tipo) mesmo sem PDFs, para que o output sempre tenha esse dado.

P2 — Qualidade E2-PrepararInput-Claude (n8n) ~1h dev
✨

[P3] Indicar "análise sem PDFs" visualmente no Step 2

Quando a análise não teve documentos, o card deveria mostrar um banner "⚠️ Análise realizada sem documentos — as perguntas abaixo ajudam a complementar o contexto". Isso educa o advogado sobre a limitação da análise.

P3 — Nice to have TriagemAnaliseCard.tsx ~30min dev
✨

[P3] Aumentar limite de arquivos no Step 1 (5→10)

Casos complexos podem ter contestação + vários anexos. O budget de 80k chars no E2 já trata o volume no backend. O limite de 5 arquivos no frontend é desnecessariamente restritivo.

P3 — Capacidade UploadProvasInline.tsx (maxFiles=10) ~5min dev
