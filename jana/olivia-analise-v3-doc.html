O que esse workflow faz

O OLIVIA-ANALISE-V3 é o coração da análise jurídica automatizada do MdFlow. Ele recebe uma requisição do frontend com o ID do processo e (opcionalmente) arquivos, então:

  1. Valida e registra a requisição no Supabase
  2. Carrega todo o contexto do processo (dados, empresa, categoria, cliente)
  3. Busca as tags da categoria e os blocos de texto disponíveis
  4. Carrega o playbook específico da categoria/empresa
  5. Extrai texto dos PDFs via loop (se houver arquivos)
  6. Busca jurisprudência relevante via RAG (embeddings OpenAI + fn_rag_search)
  7. Envia tudo para o Claude Sonnet 4.6 como OlivIA com prompt especializado
  8. Persiste o resultado estruturado no Supabase
  9. Atualiza o processo com resumo e chance de sucesso
Endpoint de entrada
POST /webhook/mdflow/olivia/analise
{
  "request_id": "uuid",
  "organizacao_id": "uuid",
  "user_id": "uuid",
  "data": {
    "processo_id": "uuid",
    "tipo": "PROVAS",              // opcional
    "arquivos": [...],             // opcional
    "contexto": "texto adicional"  // opcional
  }
}
Resposta imediata (202 Async)
HTTP 202 — Processamento assíncrono
{
  "success": true,
  "analise_id": "uuid",
  "status": "PROCESSANDO",
  "message": "Análise iniciada. Use analise_id para polling."
}
O frontend usa analise_id para polling no Supabase. A análise continua rodando em background.
🔄 Fluxo Completo
A
Estágio A — Recepção e Validação
Entry Point
Recebe o webhook, registra no log e valida o payload obrigatório antes de qualquer processamento.
WH-Analise
Webhook POST /webhook/mdflow/olivia/analise — ponto de entrada
A1-LogWebhook-INSERT
INSERT em webhook_requests com status processing
Rastreia todas as chamadas para auditoria
A2-ValidatePayload
Code: valida campos obrigatórios — request_id, organizacao_id, user_id, processo_id
A3-PayloadValido?
IF: bifurca para caminho feliz (→ A4) ou erro 400 (→ A-Respond-Error)
A4-INSERT-olivia_analises
INSERT em olivia_analises com status PROCESSANDO e progresso 5%
Gera o analise_id que o frontend irá usar para polling
A-Respond-202
Responde HTTP 202 com analise_id — libera o frontend imediatamente
Processamento continua em background após esta resposta
B
Estágio B — Carregamento de Contexto
Context Loading
Carrega todo o contexto necessário para a análise: processo, empresa, categoria, tags, blocos de texto e playbook.
B1-Prep
Extrai analise_id, processo_id e monta objeto base
B1-HTTP-Processo
GET Supabase: busca o processo com dados completos (empresa, categoria, cliente, advogado, comarca)
B1-HTTP-AnaliseAnt
GET Supabase: busca análise anterior do mesmo processo (para contexto histórico)
B1-Process
Code: consolida processo + empresa + categoria + análise anterior em um objeto unificado
B2-HTTP-Tags
RPC Supabase: busca tags e blocos da categoria via fn_get_tags_blocos
Tags = perguntas que a IA deve responder | Blocos = opções de resposta disponíveis
B2-Process
Code: processa tags/blocos, limita a 50 blocos, gera query RAG dinâmica
B3-HTTP-Playbook
RPC Supabase: busca playbook específico por categoria/empresa via fn_get_playbook
B3-Process
Code: consolida playbook (fallback para "Genérico" se não encontrado)
B4-UPDATE-Classificando
PATCH olivia_analises: atualiza progresso para 20% — "Contexto carregado. Classificando documentos..."
C
Estágio C — Extração de Documentos (Loop)
PDF Processing
Processa cada arquivo PDF individualmente em loop. Se não há arquivos, bypassa direto para o estágio D.
C1-SplitDocs
Code: split dos arquivos em items individuais. Se arquivos.length === 0, retorna item com _has_docs: false
C1b-HasDocs?
IF: verifica _has_docs. Se false → bypass para D1 sem processar PDFs
✓ Bypass correto — processos sem documentos funcionam normalmente
C2-Loop-Docs
SplitInBatches (batch=1): itera um PDF por vez
C3-FetchArquivo
POST Supabase Storage: gera URL assinada (1h) para o arquivo no bucket processos
C4-ExtractPDF
n8n Extract From File (PDF): extrai texto nativo do PDF via URL assinada
C5-FormatPDF
Code: formata texto extraído, trunca em 20k chars/doc, calcula qualidade (0-100)
Qualidade ≥ 80 se texto > 100 chars, senão 20 (PDF scaneado/vazio)
⚠️ C2-Loop retorna ao início para cada doc. Ao terminar todos, segue para D1-ConsolidarDocs.
D
Estágio D — Consolidação + RAG
RAG + Embeddings
Consolida todos os documentos extraídos e busca jurisprudência relevante via embeddings vetoriais.
D1-ConsolidarDocs
Code: agrega todos os docs extraídos, calcula qualidade média
D2-UPDATE-50pct
PATCH: atualiza progresso para 50% — "Documentos extraídos. Buscando jurisprudência..."
D3-GerarEmbedding
POST OpenAI: gera embedding da query RAG via text-embedding-ada-002
Query = tags da categoria + empresa + tipo de processo (dinâmico)
D4-BuscarRAG
RPC Supabase: fn_rag_search — busca top 8 chunks de jurisprudência/petições com score ≥ 0.3
Fonte: source_types: ['jurisprudencia', 'peticao'] da organização
D5-MontarPacote
Code: monta pacote final com TUDO — processo, docs, tags, blocos, playbook, jurisprudência, análise anterior
Trunca blocos em 200 chars/preview para não explodir o contexto
E
Estágio E — Análise com Claude (OlivIA)
IA Core
O coração do workflow. Claude Sonnet 4.6 recebe todo o contexto e retorna a análise estruturada em JSON.
E1-PassCtx
Code: recupera contexto de D5 (passthrough)
E2-PrepararInput-Claude
Code: monta system prompt + user message completos para Claude
Budget total de docs: 80k chars | 15k chars por documento individual
E3-Claude-Analise-Sonnet
Nó Anthropic Claude: claude-sonnet-4-6 | max_tokens: 8192 | temperature: 0.2
Temperature baixa para outputs JSON consistentes e determinísticos
E4-ParseResultado
Code: parse JSON do output, normaliza campos, calcula tokens/custo, extrai tags resolvidas vs. pendentes
⚠️ tokens_usados e custo_usd chegam zerados — o nó Anthropic do n8n não expõe usage
F
Estágio F — Persistência e Notificação
Finalization
Salva resultado, atualiza processo, marca análise como concluída e fecha o log.
F1-PATCH-olivia_analises
PATCH: salva resultado JSON completo, atualiza progresso para 95%, registra modelo usado
F2-UPDATE-processos
PATCH processos: atualiza ia_resumo (visão geral + analise_id) e chance_sucesso_ia (1-3)
F3-UPDATE-100pct
PATCH: status final CONCLUIDA, progresso 100%, tags_resolvidas, tags_pendentes, perguntas de triagem
F4-LogWebhook-UPDATE
PATCH webhook_requests: fecha o log com status: completed
ErrorHandler
Catch-all: captura qualquer erro, atualiza olivia_analises com status: ERRO e a mensagem

User Prompt — Estrutura dinâmica
<processo>
Número CNJ, tipo, objeto, empresa ré, categoria, cliente

Texto dos PDFs (budget 80k chars total, 15k por doc) ou "(Nenhum documento enviado)"
<tags_da_categoria>
Lista de tags com ID, código e nome — são as "perguntas" que a IA deve responder com blocos
<blocos_disponiveis>
Blocos de texto pré-escritos (opções de resposta) com ID, título e preview
<jurisprudencia_rag>
Top 8 chunks de jurisprudência/petições relevantes via RAG (embedding search)
<contexto_historico>
Análise anterior do processo (se existir) para comparação e evolução
<instrucoes_resposta>
Schema JSON exato que deve ser retornado — com todos os campos e tipos
✅ Threshold de confiança: ≥ 0.7 → usa o bloco diretamente. < 0.7 → bloco_id = null + gera pergunta_usuario para triagem.
📋 Schema de Saída
Tabela olivia_analises — campos salvos
resultado JSONB — objeto completo retornado pelo Claude
status PROCESSANDO → CONCLUIDA ou ERRO
meta JSONB — progresso (0-100), fase atual, modelo, tokens, parse_ok
tokens_usados int — ⚠️ sempre 0 (limitação do nó Anthropic n8n)
custo_usd float — ⚠️ sempre 0 (mesmo motivo)
modelo_usado text — "claude-sonnet-4-6"
concluida_em timestamp — quando a análise foi concluída
Schema JSON do campo resultado
visao_geral string Resumo executivo do caso — avaliação geral das chances
pontos_fortes string[] Lista de argumentos/provas favoráveis ao cliente
pontos_fracos string[] Lista de vulnerabilidades ou ausência de provas
analise_probatoria string Análise detalhada de cada documento e sua relevância jurídica
riscos object[] Riscos com descricao, severidade (ALTO/MEDIO/BAIXO), mitigacao
recomendacoes string[] Ações recomendadas para fortalecer o caso
chance_sucesso 1 | 2 | 3 1 = baixa, 2 = média, 3 = alta — usado no card do processo
documentos_faltantes string[] Documentos que deveriam ser solicitados ao cliente
analise_tags object[] Para cada tag: tag_id, bloco_id, confianca (0-1), justificativa, pergunta_usuario
perguntas_triagem object[] Perguntas geradas para tags com confiança < 0.7 — aparecem como formulário de triagem no frontend
chance_sucesso 1|2|3 Salvo também em processos.chance_sucesso_ia
tags_resolvidas number Tags com confiança ≥ 0.7 e bloco atribuído
tags_pendentes number Tags que precisam de triagem manual

🔍 Avaliação Técnica
✅ Fluxo assíncrono correto
202 imediato + polling no Supabase. O frontend nunca fica bloqueado esperando resposta do n8n.
✅ Bypass de docs funcionando
C1b-HasDocs? corretamente pula extração de PDF quando arquivos.length === 0. Testado com processo real sem PDFs.
✅ RAG com embeddings
OpenAI ada-002 gera embedding → fn_rag_search busca top 8 chunks de jurisprudência com similaridade vetorial.
✅ Playbook dinâmico
Instruções específicas por categoria/empresa injetadas no system prompt. Fallback para "Análise padrão" se não houver playbook.
✅ Parse robusto do JSON
E4 trata markdown fences, whitespace, e encontra o início do JSON mesmo com texto extra. Normaliza todos os campos obrigatórios.
✅ Error handling global
ErrorHandler captura qualquer falha no workflow, atualiza olivia_analises com status ERRO e fase descritiva.
✅ Contexto histórico
B1-HTTP-AnaliseAnt busca análise anterior e injeta no prompt — OlivIA sabe o que foi analisado antes e pode evoluir o diagnóstico.
✅ Context window controlada
Budget total 80k chars de docs, 15k por arquivo, 200 chars por preview de bloco, 500 chars por chunk RAG. Não estoura o contexto do Claude.
⚠️ tokens_usados / custo_usd = 0
O nó Anthropic do n8n não expõe usage no output. E4 tenta capturar mas sempre recebe zeros.
Fix possível: usar nó HTTP Request direto para a API Anthropic em vez do nó nativo, que retorna o usage completo.
⚠️ PDF scan/imagem = qualidade 20
PDFs escaneados sem texto nativo extraem texto vazio → qualidade 20. Não há OCR implementado ainda.
Possível evolução: fallback para Anthropic vision quando qualidade < 30.
