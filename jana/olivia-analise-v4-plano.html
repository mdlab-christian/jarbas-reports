🔌 Mudanças de Integração Express (5 alterações)
MUDANÇA 1 Novo campo de entrada — modelo_grupo_id B1-Prep + B1-Process

Adicionar suporte ao modelo de petição do advogado. O workflow busca o template e passa para a análise.

  • B1-Prep: extrair modelo_grupo_id: $json.body?.data?.modelo_grupo_id || null
  • B1-Process: se modelo_grupo_id presente → GET olivia_modelos_grupo?id=eq.[id]&select=active_version_id
  • Com active_version_id → GET olivia_modelos_peticao?id=eq.[id]&select=id,nome,conteudo_template
  • Salvar como: modelo_template = { id, nome, conteudo_template } ou null se não encontrar
  • Adicionar ao context_block: modelo_template: ctx.modelo_template || null
MUDANÇA 2 E2-PrepararInput-Claude — System prompt + schema de saída E2-PrepararInput-Claude

Três adições ao nó E2: regra de jurisprudência, injeção do modelo template e schema JSON expandido.

  • System prompt — após instrução de análise: REGRA ABSOLUTA: citar SOMENTE jurisprudências presentes no banco fornecido. NÃO inventar STJ/STF/TJ. Se não há juris relevantes, omitir a seção inteiramente.
  • User prompt — antes de [DOCUMENTOS]: Se modelo_template presente → injetar conteudo_template como esqueleto obrigatório de seções/tags com instrução de seguir a estrutura exatamente
  • Schema JSON — adicionar peticao_base e relatorio_processo (ver abaixo)
  • Confiança calibrada: instrução explícita "confiança > 0.7 exige documento primário com dados identificáveis"

⚠️ Riscos e Mitigações
CRÍTICO
PDF → imagem: blocker não mencionado no plano original
Claude Vision precisa de imagens. n8n não converte PDFs. Sem resolver, Vision não existe na prática.
✓ FIX: Gemini File API aceita PDF diretamente — upload uma chamada, processa todas as páginas. Implementar como rota principal de fallback.
ALTO
OOM no Railway: 150 docs × imagens base64 = 750MB+
Split in Batches paralelo sem controle vai derrubar o container (512MB RAM starter).
✓ FIX: lotes de 10-15 docs, cada doc grava resultado no banco e descarta da memória. Consolidação lê do banco.
MÉDIO
Context window na consolidação (Opus)
500 tokens × 150 docs + context_block + playbook pode chegar a 100-120K tokens. Opus tem 200K, não quebra, mas custo sobe.
✓ FIX: monitorar na primeira semana. Ativar prompt caching no system prompt do playbook (economiza ~90% tokens de input recorrentes).
MÉDIO
Rate limits do Claude Opus
Opus tem limites mais restritivos que Sonnet. 150 docs em lotes pode bater o rate limit.
✓ FIX: retry com exponential backoff no sub-workflow. Verificar tier atual da conta Anthropic antes de ir pra produção.
MÉDIO
Confiança auto-reportada não-determinística
Mesmo doc pode receber 0.65 numa chamada e 0.72 em outra. Problemático para triagem jurídica.
✓ FIX: temperatura 0.1-0.2 na consolidação. Instrução explícita de calibração no prompt ("confiança > 0.7 exige documento primário").
BAIXO
Duplicatas de docs
Advogados incluem o mesmo contrato em múltiplos formatos. Infla contexto desnecessariamente.
✓ FIX: hash MD5 do conteúdo extraído antes do mini-resumo. Descartar duplicatas silenciosamente.
BAIXO
Docs com 50+ páginas (laudos, extratos bancários)
Gemini File API suporta até 2000 páginas mas processa lentamente. Um único doc pode dominar o budget.
✓ FIX: limite de 30 primeiras + 5 últimas páginas por doc. Flag de aviso no relatorio_processo.
