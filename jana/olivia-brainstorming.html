🤖 6. Modo Agente — Arquitetura Nova

💡 Visão do Christian: O Modo Agente é o modo mais avançado. Liberdade criativa total. Roda via JARBAS/JANA. Usa RAG jurídico. Antes de gerar a petição, percorre internamente o caminho Tags → Blocos → gera uma petição Smart Blocks para si mesmo como referência, depois redige a versão final enriquecida.

Arquitetura proposta — Agente Único

FRONTEND (4 steps) │ └─ Step 1: Identificação (processo + modelo + advogado + qualidade) │ └─ Step 2: Análise & Triagem │ └─ JANA: POST /olivia/gerar-triagem (síncrono, ~30-60s) │ Payload: processo_id, modelo_id, org_id, arquivos[] │ Retorna: perguntas[], analise_provas, sugestões de respostas │ └─ Step 3: Revisão das perguntas (usuário valida/edita + inputs livres) │ └─ Step 4: Gerar └─ JARBAS: POST /olivia/gerar-modo-agente (fire-and-forget, 30s timeout) Payload: historico_id, processo_context, triagem_respostas, instrucoes │ Internamente (JARBAS): ├─ 1. Buscar modelo de petição (esqueleto) ├─ 2. Percorrer tags do modelo ├─ 3. Para cada tag → buscar blocos → selecionar melhor bloco ├─ 4. Montar "petição Smart Blocks" internamente (referência) ├─ 5. Agente redator (Claude Sonnet 4.6) — com petição base + RAG jurídico ├─ 6. Agente revisor (Claude Haiku) — score 0-10 ├─ 7. Criar Google Doc (GAS ou JARBAS /olivia/criar-doc) └─ 8. UPDATE olivia_historico SET status='SUCESSO' WHERE id=historico_id FRONTEND: polling olivia_historico a cada 3s → detecta SUCESSO

Steps do Modo Agente — Detalhado

StepO que aconteceNovo vs Mantém
1 — Identificação Mesmo step 1 unificado. Processo + tipo + modelo + versão + advogado + qualidade IA + upload de provas. Reconstruir (corrigir bug versão)
2 — Análise & Triagem JANA analisa documentos + gera perguntas de triagem. Mostra: análise das provas, sugestão de respostas com confiança, perguntas para o usuário validar. Input livre de texto por pergunta. Melhorar (adicionar inputs livres)
3 — Revisão Usuário revisa as perguntas/respostas geradas pela IA. Pode editar qualquer resposta. Campo de "instruções adicionais" global. Novo (step separado e limpo)
4 — Gerar Botão "Gerar com JARBAS". Progress bar. Polling. Exibe resultado com score, Google Doc URL, métricas. Reconstruir (corrigir P0 endpoint)

Bugs P0 a corrigir no Modo Agente

  • P0-01: Criar endpoint POST /olivia/gerar-modo-agente no JARBAS (não existe)
  • P0-02: olivia_historico criado com status=PENDENTE nunca é atualizado para SUCESSO (JARBAS não faz UPDATE)
  • P0-03: Skill do JARBAS faz INSERT em vez de UPDATE no olivia_historico — polling nunca encontra o resultado

Perguntas de triagem — Filosofia

💡 Christian: Perguntas devem ser apenas as que o usuário precisa validar. O agente deve pré-responder o que consegue extrair dos documentos e do banco. O usuário só vê o que realmente precisa de input humano.

Proposta:
• confidence >= 0.85: resposta automática, usuário pode expandir para ver/editar
• confidence 0.5-0.84: mostrar com sugestão, aguardar confirmação
• confidence < 0.5: campo aberto para o usuário preencher
• Input livre adicional por pergunta: textarea "Observação adicional"

🔬 7. Análise IA — Reconstrução

Sai do workflow n8n OLIVIA-ANALISE-V4. Passa a rodar 100% no JARBAS/JANA.

Aba: Analisar Processo

Seleciona processo via searchbox. JARBAS analisa: documentos, movimentações, fatos relevantes, pontos jurídicos. Retorna relatório estruturado com seções colapsáveis.

Aba: Analisar Decisão

Upload de documento + seleção de processo (opcional). JARBAS analisa a decisão judicial: favorável/desfavorável, fundamentos, prazo para recurso, sugestão de próximo passo.

Novo endpoint JARBAS para análise

POST /olivia/analisar Body: { tipo: 'processo' | 'decisao', processo_id?: string, org_id: string, arquivos?: [{ path, url, mime }], instrucoes?: string } Response stream SSE ou polling via olivia_historico (modo='ANALISE')

Agente único de análise

💡 Christian: Unificar o agente de triagem (que analisava provas) e o agente que gerava perguntas em um único agente. Resposta sempre em JSON.

Agente único recebe: documentos + dados do processo + modelo de petição selecionado
Retorna JSON com: { analise_provas, pontos_relevantes, perguntas_triagem[], respostas_sugeridas{} }

Sobre a Spec definitiva — Formato XML ou Markdown?

💡 Recomendação JANA: XML/tags para a spec definitiva.

Para projetos longos com múltiplas camadas (frontend / backend / banco / automação), XML permite:
• Delimitar seções com precisão (<section name="step-1">, <layer name="backend">)
• Ler partes específicas sem contexto inteiro
• IA encontra o que precisa sem ambiguidade

Markdown fica mais legível para você enquanto escreve. Proposta: escrever em Markdown + converter para XML com tags de seção antes de usar com CC.
