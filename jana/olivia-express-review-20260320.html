§2 Bugs P1 Encontrados (5 novos)
🔴
BUG-01 · P1 · salvarRevisao tem lógica de meta CORROMPIDA
Em useExpressGeneration.ts linha 185:
meta: supabase.rpc ? undefined : { ... }
O problema: supabase.rpc é sempre uma função (truthy), então meta será sempre undefined. Os dados de escritorio_contraparte_id, chance_sucesso e instrucoes_finais NUNCA são persistidos no banco. A condição foi mal construída — parece um placeholder que ficou.
🔴
BUG-02 · P1 · QualidadeIa type desalinhado com banco e constants
types.ts linha 71: QualidadeIa = 'padrao' | 'pensamento_estendido'
constants.ts linha 25: value 'pensamento_estendido' (label "Avançado")
O banco CHECK espera: conforme master plan DIV-04 resolvida — deve ser 'padrao' | 'avancado'
A migration do DB usa avancado, mas o frontend envia pensamento_estendido. INSERT falhará com CHECK constraint no Modo Agente.
🟡
BUG-03 · P1 · GeradorStep1 usa modo v2 (ProcessoSelector legado) para Express
OliviaGeradorExpress.tsx: passa modo="express" para GeradorStep1, que cai no branch v2 (sem session prop). O branch v2 usa o ProcessoSelector legado em vez da busca v3 (searchbox com busca real por CNJ/nome). Segundo spec D-FE-01, o searchbox moderno deve estar em todos os modos.
🟡
BUG-04 · P1 · dispararGeracao ignora instrucoes_finais da revisão
O payload enviado ao n8n em dispararGeracao() inclui instrucoes_adicionais da triagem (Step 2), mas quando o usuário faz revisão no Step 4 e adiciona instrucoes_finais, esse campo nunca chega ao n8n porque a geração já foi disparada. Há dois caminhos possíveis:
(a) Salvar revisão e re-disparar no n8n OU (b) enviar instrucoes_finais via webhook separado após aprovação do rascunho — o que não existe hoje.
🟡
BUG-05 · P1 · ExpressStep4Rascunho reinicia polling se re-renderizar
O useEffect em ExpressStep4Rascunho.tsx tem como deps apenas [historicoId, organizacaoId, stopPolling]. Se o componente desmonta e remonta (ex: tab blur/focus, hot reload), o polling recomeça do zero, gerando múltiplos intervalos paralelos. O stopPolling é chamado no cleanup, mas pollStartRef é resetado via closure sem referência estável. Isso pode causar timeout prematuro ou duplo polling.
