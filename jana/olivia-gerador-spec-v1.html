⚡ Anti-padrão de variação por advogado
💡
Cada advogado tem seu modelo de petição vinculado. O sistema usa esse modelo como esqueleto. A IA não precisa de style guides separados — ela simplesmente preenche/adapta o esqueleto existente do advogado. Isso garante que petições de advogados diferentes pareçam ter sido escritas por pessoas diferentes (anti-padrão de processo em massa).

⚡ 04 — Modo Express (IA Assistida)
⚡
Modo Express
N8N • Sonnet 4.6 (agente único) • Google Doc • Rate limit: 10/hora
Step 1
Identificação + Upload opcional
Campos padrão + Modal de Upload de Provas (opcional — ver Seção 06).
Deve permitir avançar sem upload — petições simples não precisam de documentos.
Input de texto + microfone para considerações iniciais do usuário.
Step 2
Análise IA + Triagem Inteligente
O agente lê processo + documentos → gera relatório completo em JSON (não resumo).
Frontend renderiza o JSON dinamicamente em sections/acordions — não texto corrido.
Em seguida: perguntas de triagem inteligentes (geradas por IA, baseadas no contexto).
Relatório completo: o usuário pode responder as perguntas sem abrir os PDFs do processo.
Step 3
Revisão + Instruções adicionais
Editor Markdown com autosave. Usuário pode ajustar outline/estrutura antes de gerar.
Step 4
Geração → Google Doc
Agente único (Sonnet 4.6) recebe: modelo esqueleto + blocos do advogado + triagem + análise → gera petição.
Responde em JSON por tag (uma entrada por tag do modelo). Frontend monta e envia pro Apps Script → Google Doc.
🏗️ Mudança Arquitetural Chave: Multi-agente → Agente Único
✅
Antes: múltiplos agentes N8N em sequência (GERAR → REDATOR → REVISOR) — contexto fragmentado.
Agora: um único Claude Sonnet 4.6 que faz análise E geração. Mais contexto, menos handoffs, petição mais coerente.
Processo com 100 páginas? Sonnet 4.6 tem 200k tokens — processa tudo em uma passagem.

🤖 05 — Modo Agente (JARBAS Autônomo)
⭐
Este é o diferencial competitivo. Não existe nada igual no mercado jurídico brasileiro. Liberdade criativa total — construímos do zero.
🤖
Modo Agente
JARBAS interno • Sem N8N • Sem Google externo • Opus/Sonnet • Máxima qualidade
Step 1
Identificação + Intenção do Usuário
Campos padrão (processo, tipo petição, modelo, advogado).
Input expandido: campo de texto longo + microfone para o usuário explicar o que quer fazer.
Opção: "Usar modelo" (default) ou "Petição livre" (sem modelo, só tipo de petição).
Modal de upload de provas (pré-carregado com docs do JARBAS já baixados).
No modo agente, o JARBAS já baixou todos os documentos relevantes — o usuário raramente precisa fazer upload manual.
Step 2
Análise Autônoma + Triagem
JARBAS lê processo + documentos on-demand (ele sabe quais documentos são relevantes para a petição solicitada).
Gera relatório HTML completo (não JSON) — análise profunda, layout rico, para o usuário ler confortavelmente.
Perguntas de triagem inteligentes e personalizadas: baseadas no processo real, não em templates genéricos.
Inputs de resposta: texto + microfone.
Jurimetria integrada: exibe chances do processo, valor estimado, precedentes do mesmo tipo de ação e tribunal.
Step 3
Revisão do Rascunho
JARBAS gera uma proposta inicial da petição antes do usuário aprovar.
Editor com preview ao vivo. Usuário pode ajustar, comentar, pedir reformulação de trechos.
Chat integrado para refinamento: "reformula esse parágrafo", "adiciona jurisprudência sobre X".
Step 4
Geração Final + Extração de Dados
Petição final gerada internamente no JARBAS.
Extração automática de dados de contato do escritório adverso → proposta de acordo preparada.
Output: HTML interno (formatado) + opção de exportar como Google Doc / PDF.
Salva em olivia_historico com metadados completos.

No relatório (Step 2)

  • Chance de sucesso (chance_sucesso_pct)
  • Valor estimado recuperável
  • Prazo estimado resolução
  • Precedentes similares (tribunal + tipo de ação)

Após geração (Step 4)

  • Proposta de acordo com valor calculado
  • Dados do escritório contrário extraídos automaticamente
  • Histórico de acordos similares
⭐
Diferencial vs JurídicoAI e concorrentes: Todos os concorrentes usam IA genérica sem base jurídica própria. Nossa vantagem é o RAG proprietário (blocos, jurisprudências selecionadas, playbooks do escritório) + contexto completo do processo no Supabase + documentos já baixados pelo JARBAS. A IA nunca inventa — só usa conhecimento interno.
📁 06 — Modal de Upload & Documentos do JARBAS
🗂️ Modal "Documentos & Provas"

Abre ao clicar em "Adicionar provas". Disponível no Express e no Agente. Pré-carregado com documentos já baixados pelo JARBAS.

📥 Documentos do JARBAS (já disponíveis)

  • Organizados por evento processual (cada evento = 1 accordion card)
  • Card do evento mostra: data, tipo (Contestação, Recurso...), parte responsável
  • Expandir accordion → lista de documentos do evento com preview
  • Selecionar: evento inteiro (checkbox) ou documentos específicos
  • Motivo: não sobrecarregar contexto da IA com documentos irrelevantes

⬆️ Upload manual (sempre disponível)

  • Drag & drop de PDF, imagem, Word
  • Após upload → classificação obrigatória (tipo de prova)
  • Otimizado para volume alto (múltiplos arquivos de uma vez)
  • Tipos de prova via olivia_tipos_prova
✅
Resultado: usuário não precisa fazer upload manual na maioria dos casos. JARBAS já baixou os documentos relevantes. O modal é a interface de seleção/curadoria, não de upload forçado.

🏷️ 07 — Formatação: Tags, Blocos e Hierarquia
⚠️
Problema crítico: A formatação é um diferencial do escritório. Os modelos antigos têm formatação específica em texto/HTML. A migração para o novo sistema deve preservar 100% da formatação — se quebrar, a petição fica inutilizável.
🏷️ Hierarquia Atual (Smart Blocks)
Modelo de Petição (olivia_modelos_grupo) └── Versão (olivia_modelos_peticao) ├── Tags: {{FATOS}}, {{FUNDAMENTOS}}, {{PEDIDOS}}, {{JURISPRUDENCIA}}... │ └── (olivia_tags + olivia_modelo_tags) ├── Blocos de Texto por Tag │ └── (olivia_blocos_texto) └── Jurisprudências vinculadas └── (olivia_modelo_juris)
⚡ Express & Agente — IA preenche as tags

A diferença do Smart Blocks é que a IA não é limitada aos blocos existentes:

Smart Blocks

  • Tags → seleciona bloco existente
  • Sem blocos = tag vazia
  • Ordem fixa do modelo

Express / Agente

  • Tags → IA gera conteúdo por tag
  • IA pode adicionar tags novas intermediárias
  • IA pode remover tags irrelevantes para o caso
  • IA pode reposicionar tags (se fizer sentido)
  • Modelo = esqueleto, não camisa de força
// Resposta JSON do agente (Express/Agente): { "tags_mantidas": ["{{FATOS}}", "{{FUNDAMENTOS}}", "{{PEDIDOS}}"], "tags_removidas": ["{{CONTESTACAO_DOCUMENTAL}}"], // irrelevante p/ este caso "tags_novas": ["{{EXCECAO_PRESCRICAO}}"], // adicionada pela IA "conteudo": { "{{FATOS}}": "Conteúdo gerado...", "{{FUNDAMENTOS}}": "Art. 51 CDC...", ... } }
💾 Importação de Modelos Antigos → HTML
💡
Proposta: ao importar modelos antigos (blocos de texto em formato legado), gerar um conteudo_html para cada bloco que preserva a formatação original (negrito, recuo, espaçamento de petição).

HTML é mais robusto que texto puro para preservar formatação. O Apps Script já processa HTML. Isso resolve o problema de formatação tanto no Smart Blocks quanto no Express/Agente.

Campo novo sugerido: olivia_blocos_texto.conteudo_html + campo existente conteudo como fallback.
🛡️ 08 — Regras de IA & Anti-Alucinação
🚨
Regra P0 absoluta: A IA NUNCA inventa fundamentações, artigos, jurisprudências ou fatos. Tudo que ela gera deve estar ancorado em: (1) RAG interno, (2) dados do processo no Supabase, (3) documentos fornecidos, (4) input do usuário.
📋 Regras Fixas do Gerador (extraídas do sistema antigo)
#RegraImplementação
R1 Jurisprudência = estado do processo. Usar TJRS para processo RS, TJSC para SC, TJSP para SP. Filtrar olivia_jurisprudencias por estado_id do processo. Fallback: versão mais recente do assunto.
R2 Zero invenção. Sem fundamentações não mapeadas no RAG. System prompt com instrução explícita. Validar output contra lista de fundamentos conhecidos.
R3 Variação por advogado. Mesmo fato → petições diferentes por advogado. Modelo/versão vinculados ao advogado. Cada modelo tem estrutura e linguagem distintas.
R4 Processo tipo empresa vs pessoa física. Lógicas diferentes. Verificar tipo_processo e empresa_id vs cliente_id. Prompt diferente para cada tipo.
R5 Dados da restrição do processo. Usar restricao_valor, restricao_contrato, etc. Incluir no contexto do agente automaticamente.
R6 Formatação obrigatória. Petições seguem formato jurídico padrão do escritório. Esqueleto do modelo + HTML dos blocos. Apps Script aplica template de página.

🤔 D1 — Modo Express: Pensamento normal ou avançado (thinking)?
Express hoje usa chain de agentes N8N. Migrar para agente único Sonnet 4.6. Mas vale colocar uma opção de "modo pensamento avançado" (Claude extended thinking)?

Recomendação JANA: Opção A para V1. Lançar com Sonnet 4.6 unificado. Se precisar de pensamento avançado no Express, colocar no Modo Agente (que tem liberdade total). Express = velocidade. Agente = qualidade máxima.
🤔 D2 — Smart Blocks e Express via JARBAS (sem N8N)?
Você mencionou avaliar se vale a pena ter Smart Blocks e Express rodando via JARBAS (sem N8N), já que o fluxo já está feito e o JARBAS seria mais rápido e estável.
A
Manter N8N para Smart Blocks e Express. Funciona, já está feito, é resiliente (sem JARBAS).
B
Migrar para JARBAS. Mais rápido, mais controle, sem latência N8N. Mas perde resiliência offline.
C
Híbrido: JARBAS por padrão, fallback automático para N8N se JARBAS offline. Melhor dos dois mundos, mas mais complexo.
Recomendação JANA: Opção A para V1, Opção C no roadmap. O princípio de resiliência (escritório não para sem JARBAS) é fundamental. Manter N8N como primary por enquanto, avaliar migração após validação do Modo Agente.
