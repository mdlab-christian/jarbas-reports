💥 Por Que o Claude Code Falhou — Diagnóstico Honesto

🔴 Causa Raiz #1 — Escopo sem fronteiras

Quando você entrega o projeto inteiro de uma vez pro CC, ele tenta satisfazer o PRD inteiro num único contexto de 200k tokens. Inevitavelmente: alucina nas partes finais, ignora regras do início, cria variações não documentadas. O CC não sabe o que é P0 vs P3.

🔴 Causa Raiz #2 — n8n tem regras únicas que IA ignora

O CC não sabe que fetch() não existe em Code nodes do n8n. Ele escreveu o que parece correto, mas quebra em produção. A descoberta de require('https') só veio com teste E2E real. IA não testa, ela escreve o que parece certo.

🔴 Causa Raiz #3 — Contexto multi-domínio demais

React + TypeScript + Supabase + n8n + GAS + Skills JARBAS + banco com 22 tabelas + regras de negócio jurídico. O CC prioriza o que é mais comum (React/TS) e simplifica o que é específico (n8n, GAS, hierarquia de blocos).

🟡 Causa Raiz #4 — Sem feedback loop em tempo real

CC gera tudo, você vai dormir, acorda com resultado ruim. Sem revisão por camada, sem "fez isso certo, agora próximo passo". A iteração é o antídoto para alucinação.

🟡 Causa Raiz #5 — Workflows n8n em desuso conectados

500+ workflows no n8n é um campo minado. O CC (e a IA em geral) conecta pelo nome — quando tem workflows antigos com nomes parecidos, ela conecta o errado. Precisamos de nomes canônicos e IDs fixos no prompt.

🔵 Por que JANA fez melhor na noite anterior?

Porque eu executei por partes menores, com revisão a cada passo, usando os arquivos reais do projeto. Isso confirma a abordagem: blocos pequenos + revisão por camada = menos alucinação.

🧠 Gestão de Contexto — Como Não Alucinar

✅ O que funciona para reduzir alucinação

  • Bloco ≤ 200 linhas de código: Nunca pedir mais de 1 componente por vez
  • Critério de aceite antes do código: "Isso vai estar pronto quando..." definido antes de escrever
  • IDs explícitos no prompt: Sempre passar IDs reais (workflow ID, table name, column names)
  • Arquivo de referência junto: Passar o bloco relevante do PRD no prompt, não o PRD inteiro
  • Revisar antes de avançar: Testar bloco no browser antes de ir pro próximo
  • OLIVIA-STATUS.md no início: Ler estado antes de qualquer ação

❌ O que causa alucinação — evitar sempre

  • "Faz o projeto inteiro": Nunca. Sempre por bloco.
  • PRD completo no prompt: Passa só a seção relevante
  • Sem especificar IDs: "chama o n8n" sem passar o workflow ID → CC inventa
  • Misturar domínios: "Faz o frontend E o n8n" no mesmo prompt → CC simplifica o n8n
  • Verificar depois de muito tempo: Revisar a cada bloco, não no final
  • Contexto >80%: Reiniciar sessão antes de degradar

📊 Sobre o RAG de projetos — vale usar?

Diagnóstico honesto: O RAG de projetos que você criou é valioso para consultas, mas não resolve o problema de execução sem alucinação. A alucinação não vem de falta de memória — vem de escopo grande e feedback lento.

O que o RAG faz bem: Permite perguntar "o que decidimos sobre o modo Agente?" e obter resposta rápida. Útil para consultas durante execução.

O que o RAG NÃO resolve: O CC gerando código errado com contexto correto. A alucinação é do modelo, não da falta de dados.

Recomendação: Usar o RAG como consulta rápida durante blocos, não como substituto do OLIVIA-STATUS.md. O arquivo STATUS é mais confiável porque é atualizado manualmente.

🤖 Sobre criar skills específicas por categoria de empresa

Ideia excelente, timing errado. A lógica de ter uma skill por categoria (Cessão, Energia Elétrica, Telefonia, etc.) é muito válida — cada categoria tem padrões de argumentação diferentes, teses que funcionam, jurisprudências específicas.

Pré-requisito: OlivIA v3 precisa estar funcionando primeiro. As skills precisam do banco populado (olivia_modelo_juris com vinculações reais, texto_integral das jurisprudências — E27). Sem isso, as skills trabalham no vazio.

Plano futuro: Após OlivIA entregue → criar skill por categoria → integrar com categoria_empresa_id que já está no banco (E11 concluído) → testar com casos reais do Midas.

🔄 Sobre grupos paralelos — eles consomem contexto entre si?

Não — grupos Telegram são sessões completamente independentes no OpenClaw. Contexto de um grupo não afeta outro.

Mas atenção: Se você tiver 3 grupos rodando simultâneamente com Claude Code paralelo, os custos se somam. Não é problema de contexto, é questão de orçamento.

Estratégia recomendada: 1 grupo ativo por vez. Quando um bloco está sendo executado, os outros esperam. Isso também evita conflitos de Git (dois grupos modificando o mesmo arquivo).
