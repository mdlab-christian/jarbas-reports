1. Por que o CC falhou — diagnóstico real

❌ O que não funcionou (CC direto)

  • Leu 11k+ linhas de spec antes de começar → contexto saturado
  • Perdeu o fio condutor entre steps → alucinação nas conexões
  • Não sabia o que já foi testado vs o que falta
  • Sem rastreamento de estado → recomeça do zero a cada sessão
  • Correções sem validação visual → bugs que pareciam corrigidos
  • Prompt único com 27 etapas → responsabilidade demais de uma vez

✅ O que vai funcionar (RAG + JANA Loop)

  • RAG entrega só o trecho relevante (~2k tokens) → contexto limpo
  • Estado persistido em projetos_etapas → nunca perde onde parou
  • JANA tira screenshot antes + depois de cada fix → validação real
  • Commit após cada correção → histórico granular, rollback fácil
  • JANA conhece banco, browser, git → sem intermediário
  • 1 componente = 1 iteração do loop → foco total
Insight central: A tabela projetos_etapas foi projetada exatamente para isso — cada linha tem cc_prompt, criterio_aceite, arquivos_alvo. A falha foi chamar tudo de uma vez. Com RAG + loop JANA, usamos essa estrutura como foi concebida: 1 etapa → 1 contexto → 1 validação → avança.

7. Arquitetura do Loop JANA

loop_olivia():
  etapa = SELECT * FROM projetos_etapas
          WHERE projeto_id = '48d9bfaa-...'
          AND status = 'rascunho'
          ORDER BY prioridade ASC, ordem ASC
          LIMIT 1

  if etapa == null:
    → todas concluídas → validação E2E → relatório → notificar Christian

  UPDATE projetos_etapas SET status = 'em_execucao' WHERE id = etapa.id

  # 1. Consultar RAG
  spec_trecho = exec("node rag_projeto.mjs search '{etapa.componente}' --projeto olivia-v3")

  # 2. Ler código atual
  codigo_atual = read(etapa.arquivos_alvo[0])

  # 3. Screenshot (componente visual)
  if etapa.camada == 'frontend':
    navigate("http://localhost:5173/olivia")
    screenshot_antes = browser.snapshot()

  # 4. Comparar spec vs real → identificar divergências
  divergencias = comparar(spec_trecho, codigo_atual, screenshot_antes)

  # 5. Corrigir
  for d in divergencias:
    edit(etapa.arquivos_alvo, d.fix)

  # 6. Commit
  exec("cd ~/beta-mdflow && git add {etapa.arquivos_alvo} && git commit -m 'fix(olivia): {etapa.componente} — {divergencias}'")

  # 7. Validar
  screenshot_depois = browser.snapshot()
  passou = valida_criterio(screenshot_depois, etapa.criterio_aceite)

  if passou:
    UPDATE projetos_etapas SET status = 'concluida',
                               evidencia_url = screenshot_depois,
                               resultado = 'OK'
    continue loop

  else:
    tentativas += 1
    if tentativas >= 2:
      UPDATE projetos_etapas SET status = 'falhou', pontos_falha = [motivo]
      NOTIFY Christian: "⚠️ {etapa.titulo} falhou 2x — intervenção necessária"
      continue loop  # avança para próxima (não trava tudo)
    retry

8. Probabilidades Reais de Sucesso

Frontend conforme spec (campos, layout, fluxo de steps) 95%

JANA já corrigiu bugs similares com browser + edit + commit. Com RAG confirmando spec antes de cada fix, chance de alucinação cai drasticamente.

Loop de revisão completo (~15 componentes) enquanto você dorme 75%

O risco não é o RAG nem o loop — é a complexidade dos flows n8n (tipo petição, modelos, versões). Com RAG entregando spec correta, JANA tem tudo que precisa. Risco: fix que exige teste de backend real (n8n offline, seed incompleto).

Integração completa (frontend → n8n → GAS → backend) 55%

Depende de serviços externos (n8n Railway, GAS). Se n8n offline ou webhook incorreto → loop para nessa etapa e registra bloqueio. Não trava o restante.

Expectativa realista: Você acorda com o frontend 85–90% conforme spec, cada divergência registrada como etapa no banco, e os bloqueios de integração claramente identificados com causa raiz. Não "pronto pra produção" — mas pronto para lapidação final rápida.

10. Melhorias Sugeridas por JANA

💡 cc_prompt com header padrão

Cada prompt deveria começar com: credenciais, caminho dos arquivos, como commitar, onde notificar ao terminar. Hoje alguns têm, outros não. Padronizar evita que o executor "adivinhe" onde estão as credenciais.

💡 Ambiente de teste isolado n8n

Antes de modificar workflows de produção, copiar com _test suffix, testar com pin data, só então substituir. Já fazemos no frontend (branch feature), deveria ser padrão no n8n também.

💡 Playwright E2E como gate obrigatório

Para etapas de camada "frontend", critério de aceite deveria incluir Playwright snapshot obrigatório. Suite E2E OlivIA (37 testes, 100% pass) já existe — integrá-la como gate evita regressions silenciosas.

💡 RAG Phase 2 em background imediato

Após E27 ter ≥80% do texto_integral, iniciar embeddings imediatamente em background. Quando você acordar, o RAG Controller já teria base para funcionar — não precisa esperar mais uma sessão.
