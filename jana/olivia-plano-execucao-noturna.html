1. Diagnóstico — Por que a IA está se perdendo

Problema raiz: Prompts muito longos com muita responsabilidade numa única chamada. A IA recebe 27 etapas de uma vez e tenta executar tudo sem contexto incremental — resultado: alucinação, pular etapas, inventar comportamentos.

❌ O que não funciona

  • Prompt único com 27 etapas + spec + PRD
  • Claude Code lendo a tabela inteira do banco
  • Contexto enorme → janela saturada → alucinação
  • Sem verificação de sucesso entre etapas
  • IA ignora critério de aceite (não foi forçado a checar)

✅ O que vai funcionar

  • 1 etapa = 1 CC spawn (contexto pequeno e focado)
  • Cada CC lê só o cc_prompt da etapa + arquivos alvo
  • JANA valida critério de aceite antes de avançar
  • Status no banco: rascunho → em_execucao → concluido/falhou
  • Loop controlado por JANA — não pelo CC

🔬 Análise técnica do problema

A tabela projetos_etapas tem a estrutura certa — cada etapa tem cc_prompt bem definido, criterio_aceite, arquivos_alvo, e depende_de. O problema é que o executor (CC) está sendo chamado com contexto demais de uma vez. A solução é usar essa estrutura como foi projetada: JANA como orquestradora, CC como executor atômico de 1 etapa por vez.

2. Solução — Execução Granular em Etapas Atômicas

Arquitetura proposta: JANA lê a próxima etapa pendente do banco → spawna CC com prompt focado → monitora saída → valida critério de aceite → marca concluída → avança para próxima. Cada etapa: contexto mínimo, objetivo único, verificação obrigatória. 
JANA (Orquestradora)
  │
  ├── 1. Lê projetos_etapas WHERE status = 'rascunho' ORDER BY ordem LIMIT 1
  │
  ├── 2. Atualiza status → 'em_execucao'
  │
  ├── 3. Spawna CC com cc_prompt da etapa (NÃO com spec inteira)
  │       └── CC faz apenas aquela etapa, faz build check, commita
  │
  ├── 4. JANA lê output do CC e valida:
  │       └── Critério de aceite passou? → status → 'concluido'
  │       └── Falhou? → retry 1x → se falhou 2x → status → 'falhou', alerta
  │
  └── 5. Avança para próxima etapa → loop

🎯 Vantagens

  • CC tem contexto de 1 etapa, não 27
  • Alucinação praticamente impossível
  • Rastreabilidade total no banco
  • Retry automático se falhar
  • JANA acorda se alguma etapa travar

5. Estratégia de Execução Noturna

Minha proposta: JANA executa como orquestradora inteligente. Cada etapa = 1 CC spawn com contexto mínimo e focado. Validação obrigatória antes de avançar. Retry automático (max 2x). Nenhuma etapa fica no limbo.
  1. E05 — Fix n8n fetch() [P0, ~30min] CC conecta na API n8n, exporta 2 workflows, substitui fetch() por require("https") em 15 nodes, reimporta e ativa. JANA valida: curl nos webhooks Express e Smart Blocks → sem ReferenceError.
  2. E08 — Fix ORG_ID hardcoded [P2, ~10min] CC edita skill_olivia_agente_v2.mjs: 1 linha. Commit. JANA valida: grep confirma que ORG_ID usa params.organizacao_id.
  3. E24 — Skill relatório HTML [P2, ~45min] CC cria skill_olivia_relatorio.mjs no JARBAS (via ssh mac-mini). Testa com historico_id real. JANA valida: URL pública acessível no browser + campo olivia_historico.url_publica preenchido.
  4. E27 — Reimport RAG texto_integral [P1, ~60min] CC cria script reimport no JARBAS, executa em background. Bate de 50 por vez, 2s de pausa. JANA monitora e valida quando script terminar: ≥80% das jurisprudências com texto_integral.
  5. Validação E2E Final [~20min] JANA abre browser, faz login como mdlab.equipe@gmail.com, testa modo Express (processo de negativação) e modo Smart Blocks. Tira screenshot. Se OK → relatório final.
  6. Relatório final + notificação JANA gera HTML de resultado, publica em jarbas-reports, notifica Christian no Telegram com link e resumo do que foi feito.

Arquitetura do loop JANA

loop:
  etapa = SELECT * FROM projetos_etapas
          WHERE projeto_id = '48d9bfaa...'
          AND status = 'rascunho'
          ORDER BY prioridade ASC, ordem ASC
          LIMIT 1

  if etapa == null:
    → todas concluídas → gerar relatório final → notificar

  UPDATE projetos_etapas SET status = 'em_execucao' WHERE id = etapa.id

  resultado = executa_cc(etapa.cc_prompt)

  if valida_criterio(resultado, etapa.criterio_aceite):
    UPDATE projetos_etapas SET status = 'concluido' WHERE id = etapa.id
    tentativas = 0
    continue loop
  else:
    tentativas += 1
    if tentativas >= 2:
      UPDATE projetos_etapas SET status = 'falhou' WHERE id = etapa.id
      NOTIFY Christian → "⚠️ E{ordem} falhou 2x — precisa de intervenção"
      break
    retry

8. Minhas Ideias de Melhoria para o Projeto

Perspectiva JANA: Se este projeto fosse meu, faria algumas coisas diferentes. Não para mudar o que está definido na spec — mas para garantir que o que está na spec funciona de forma robusta e escalável.

💡 1. Skill de Orquestração Dedicada

Criar uma skill projeto-executar que encapsule o loop descrito acima. JANA só precisaria receber projeto_id e a skill faz tudo: ler etapas, spawnar CC, validar, atualizar banco, reportar. Reusável para qualquer projeto na tabela projetos.

💡 2. cc_prompt com Template Padrão

Cada cc_prompt deveria começar com um header padrão: credenciais, caminho dos arquivos, como commitar, onde notificar ao terminar. Hoje alguns prompts têm isso, outros não. Padronizar evita que o CC precise "adivinhar" onde estão as credenciais.

💡 3. Ambiente de Teste Isolado para n8n

Antes de modificar workflows n8n de produção, copiar com _test suffix, testar com pin data, e só então substituir o original. Evita quebrar produção durante o fix. Já fazemos isso no frontend (branch feature), deveria ser padrão no n8n também.

💡 4. Playwright E2E como Gate de Conclusão

Para etapas de camada "frontend", o critério de aceite deveria incluir um Playwright snapshot obrigatório. Hoje temos a suite E2E da OlivIA (37 testes, 100% pass). Integrá-la como gate automático evitaria regressions silenciosas.

💡 5. RAG em 2 Fases (já está planejado, mas confirmando)

E27 preenche texto_integral (Phase 1). Phase 2 seria gerar embeddings e vetorizar. Sugiro que após E27 ter ≥80%, eu imediatamente inicie os embeddings em background (mesmo que demore 2h). Quando você acordar, o RAG Controller já teria base para funcionar.
