🧱 Smart Blocks — Análise Fallback Legado — ideia de Christian, avaliação JANA
🦊 Minha análise da ideia (como CTO)

Você sugeriu usar o mecanismo do sistema antigo como fallback para o Smart Blocks enquanto o novo não funciona. A ideia tem mérito — os problemas de vinculação de tabelas que emperram o Smart Blocks novo podem ser desacoplados com uma view que "traduz" o schema antigo para o novo. Veja minha análise:

✅ Por que pode funcionar

  • O sistema antigo já gerou petições reais — mecanismo comprovado
  • Uma VIEW no Postgres pode fazer a ponte sem alterar o schema novo
  • Entregas valor imediato enquanto o novo é construído corretamente
  • Reduz pressão: Smart Blocks novo pode ser construído sem urgência
  • Fallback transparente: o usuário nem percebe

❌ Riscos reais

  • Sistema antigo pode ter dívida técnica que vai para o novo via GAS
  • Dois mecanismos em paralelo = manutenção dupla
  • Views complexas podem ter performance ruim com volume
  • GAS do sistema antigo pode ter credenciais/configs diferentes
  • Risco de "fallback que vira permanente" — nunca terminar o novo
D8

🔧 Skill Gerenciamento-Projetos — mudanças + framework reutilizável
🦊 Transformar isso em framework, não só em skill do OlivIA

Esta é uma oportunidade de ouro. O que estamos construindo — projetos com conversas paralelas, estado compartilhado no Supabase, blocos de execução revisados pelo CC — é um framework de desenvolvimento agêntico que vai funcionar para qualquer projeto futuro. A skill deve ser genérica o suficiente para reutilizar no MdFlow, em futuros módulos, em qualquer coisa que precise de execução distribuída.

AGORA (modelo JANA-first)

Estrutura padrão para JANA gerar o prompt de revisão. Inclui: bloco executado, spec do modo, arquivos alterados, regras específicas (n8n, GAS, Supabase), o que NÃO pode mudar.

3. Verificar spec do modo: o bloco segue exatamente o que foi especificado?
4. Listar no máximo 3 arquivos por bloco (evitar contexto longo)
5. Após executar: validar que nenhuma regra de negócio jurídica foi ignorada
6. Salvar report + atualizar status antes de prosseguir

🎯 JANA como CTO — O que faria diferente — considerações honestas
🦊 Se este projeto fosse meu, faria diferente em 5 pontos
  • 1. RAG antes de qualquer código: Antes de escrever uma linha de código para qualquer modo, eu consultaria o RAG especificamente sobre aquele componente. "Como funciona o Step 2 do Smart Blocks segundo a documentação?" → RAG me dá o contexto exato. Isso elimina 80% das alucinações.
  • 2. n8n com IDs canônicos no prompt: Todo prompt que menciona n8n vai ter o ID do workflow, não só o nome. workflow_id: "abc123" não "workflow que gera petição express". CC não pode conectar o workflow errado se der o ID.
  • 3. E2E antes de concluir qualquer modo: Nenhum modo é marcado como "concluído" sem um E2E visual passando. Não importa se o código parece certo — se o teste não passa, não está pronto.
  • 4. Spec viva por conversa: Cada sala de execução tem uma spec-sala.md que é atualizada a cada bloco concluído. Não uma spec estática. A spec reflete o estado real do código, não o estado planejado.
  • 5. Limite de 200 linhas por bloco: Qualquer bloco que precise alterar mais de 200 linhas em um arquivo vira dois blocos. CC alucina em diff grandes. Blocos menores = revisão mais precisa.

🧠 RAG — como usar aqui

Antes de cada bloco complexo: buscar no RAG da JANA os documentos relevantes ao componente. O RAG já tem toda a documentação do OlivIA. Garantir que a query seja específica: não "OlivIA" mas "Smart Blocks Step 2 hierarquia de blocos".

📋 Google Apps Script

GAS é o motor do Smart Blocks (geração de documento). Verificar se a documentação do GAS está no RAG. Definir na spec: qual script, qual deployment, como é chamado, credenciais, timeout esperado.

🤖 Agentes Especializados

Modo Agente: skills por categoria (trabalhista, cível, etc.) + usuários isolados no macOS no MacBook Air. Cada agente tem acesso só às skills da sua categoria. Resultado esperado: melhor que agente genérico.

🔴 O maior risco do projeto — ainda
  • Vinculação de tabelas: O Smart Blocks novo trava aqui. Antes de construir qualquer frontend, precisamos entender exatamente quais tabelas estão envolvidas e por que a vinculação falha. Se não entendermos isso agora, o Smart Blocks novo vai ter o mesmo problema.
  • Contexto entre conversas: A tabela projetos_status mitiga isso, mas só se todas as conversas atualizarem religiosamente. Qualquer conversa que "esquece" de atualizar quebra a coordenação.
  • n8n legacy: 500+ workflows com nomes parecidos é uma bomba. Antes de qualquer trabalho n8n, catalogar os workflows canônicos que o OlivIA usa — e apenas eles.
