🔎 Por que o CC falhou e o que muda

❌ CC — O que não funcionou

  • Leu 11k linhas de spec antes de começar → contexto saturado
  • Perdeu o fio condutor entre steps — alucinação nas conexões
  • Não sabia o que já tinha sido testado vs o que faltava
  • Sem rastreamento de estado — recomeçava do zero em cada sessão
  • Correções sem validação visual → bugs que pareciam corrigidos

✅ RAG + JANA Loop — O que muda

  • RAG entrega só o trecho relevante (~2k tokens) → contexto limpo
  • Estado persistido em projetos_etapas → nunca perde o que fez
  • JANA tira screenshot antes + depois de cada fix → validação real
  • Commit após cada correção → histórico granular, rollback fácil
  • JANA conhece o banco, o browser, o git — não precisa de intermediário

75%

O principal risco não é o RAG nem o loop — é a complexidade dos flows n8n (B-01 tipo petição, modelos, versões). Com o RAG entregando a spec correta, JANA tem tudo que precisa. O que pode falhar: fix que exige teste de backend real (n8n offline, seed incompleto).

95%
De entregar o frontend conforme spec (campos, layout, fluxo de steps)

JANA já corrigiu bugs similares com browser + edit + commit. Com RAG para confirmar spec antes de cada fix, chance de alucinação cai drasticamente.

55%
De testar integração completa (frontend → n8n → GAS → backend) em loop único

Integração real depende de serviços externos (n8n Railway, GAS). Se n8n estiver offline ou webhook incorreto, o loop para nessa etapa e registra o bloqueio. Não trava o restante.

Expectativa realista: você acorda com o frontend 85-90% conforme spec, cada divergência registrada como etapa no banco, e os bloqueios de integração claramente identificados com causa raiz. Não "pronto pra produção" — mas pronto para lapidação final sua.
