Issues Críticos — Ordenado por Prioridade
P0 CRÍTICO Step 1 não unificado — 3 implementações paralelas divergindo
Violação de D-ARQ-11. Existem 3 implementações do Step 1: GeradorStep1.tsx (Express/v3), AgenteStep1.tsx (Agente) e SmartBlocksStep1.tsx (Smart Blocks). Cada uma tem lógica diferente, inconsistências nos campos exibidos e potencial para drift. AgenteStep1 usa ProcessoSelector legado que não enriquece o processo com restricoes_ativas, restricao_valor, cidade_distribuicao etc. As correções no SearchPanel v3 nunca chegam ao Agente.
P0 CRÍTICO AgenteStep1 sem seletor Padrão/Avançado (D-FE-14)
O modo Agente deve ter seletor de qualidade IA: Padrão (Sonnet 4.6) vs Avançado (Opus 4.6). Esse campo alimenta olivia_historico.qualidade_ia e controla qual modelo Claude usa o JARBAS. Atualmente ausente — o sistema usa Sonnet por padrão mas o usuário não pode escolher Opus para casos complexos.
P1 ALTO SmartBlocksStep1: JARBAS_ENABLED hardcoded = true
const JARBAS_ENABLED = true está hardcoded em SmartBlocksStep1.tsx. Isso significa que o toggle de relatório HTML nunca é desabilitado mesmo quando JARBAS estiver offline. Deve usar o mesmo health check de useGeradorStep1 que já faz fetch(JARBAS_BASE_URL/health).
P1 ALTO Validação ao avançar não verifica documentos (D-FE-15)
Modos Express e Agente devem exigir pelo menos 1 documento uploaded OU confirmação explícita "Gerar sem documentos". Atualmente canAdvance verifica apenas processo + tipo + modelo + versão. Usuários podem gerar petições sem nenhuma prova/contestação, reduzindo qualidade.
P1 ALTO Abas Chrome multi-petição ausentes (D-FE-07)
index.tsx usa selectedMode simples — só 1 petição por vez. O plano prevê WizardTabBar (abas tipo Chrome, max 5) com usePeticaoStore (Zustand). Ausente completamente. Usuários que iniciam uma petição e querem iniciar outra perdem o contexto da primeira.
P1 ALTO Express usa GeradorStep1 v2 (ProcessoSelector legado) em vez de v3
O fluxo Express herda do GeradorStep1 mas cai no v2 path que usa ProcessoSelector legado — sem o search panel enriquecido, sem enriquecimento de restrições, sem cidade_distribuicao. Deveria usar o v3 path com GeradorStep1SearchPanel.
P2 MÉDIO UploadDrawer sem categorias de documento e sem drag-over visual
O plano prevê categorias: Comprovante restrição, SMS, Email, Correspondência, Contestação, Laudo, Outros. Sem categorias o JARBAS não consegue priorizar documentos corretamente na análise. Também falta feedback visual de drag-over na área de upload.
P2 MÉDIO ModeCard: status JARBAS/n8n não dinâmico por modo
Os cards de modo sempre mostram status "online" (valor fixo). Deveriam refletir: Agente = JARBAS health check, Express = n8n ping, Smart Blocks = GAS disponível. Usuários não sabem quando um modo está offline antes de iniciar.
