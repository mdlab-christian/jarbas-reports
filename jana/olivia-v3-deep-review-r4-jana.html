🔴 P0 — Bugs críticos (quebram em produção)
P0 BUG-R4-01 SmartBlocks — INSERT em olivia_historico sem qualidade_ia
O hook useSmartBlocksGeneration.ts faz o INSERT de olivia_historico sem incluir o campo qualidade_ia. O campo pode ter NOT NULL com DEFAULT no banco — mas o SmartBlocksState não tem qualidadeIa no state e o SmartBlocksStep1 não expõe seletor de qualidade.

Resultado: todas as gerações Smart Blocks ficam sem qualidade_ia registrada → analytics/relatórios quebrados; se a coluna for NOT NULL sem DEFAULT, o INSERT vai falhar silenciosamente (capturado no try/catch mas usuário só vê "Falha ao criar histórico").

Comparação: Express e Agente passam qualidade_ia corretamente. Smart Blocks não.
📄 hooks/useSmartBlocksGeneration.ts:insert (linha ~150) · hooks/SmartBlocksState (sem qualidadeIa)
P0 BUG-R4-02 useExpressGeneration — startPolling declarado DEPOIS de dispararGeracao no closure
dispararGeracao (linha 89) chama startPolling(historicoId) (linha 157) e declara startPolling como dependência do useCallback (linha 168). Mas startPolling só é declarado na linha 216 — depois no arquivo.

Em JavaScript, useCallback com const tem temporal dead zone — ao montar, startPolling pode ser undefined dentro do closure de dispararGeracao na primeira renderização. Isso causa: geração dispara, historicoId criado, mas polling nunca inicia → usuário fica preso em "EM_PROGRESSO" eternamente.

O R3 "corrigiu" adicionando startPolling nas deps, mas a ordem de declaração ainda é o problema real.
📄 hooks/useExpressGeneration.ts:89 (dispararGeracao) vs :216 (startPolling) — inversão de ordem
P0 BUG-R4-03 GeradorStep1 ainda tem dual API v2/v3 — drift do cleanup
O split do GeradorStep1 (R3) manteve a dual API PropsV2 | PropsV3 com isV3() guard (11 ocorrências confirmadas). O PR de limpeza (R2-prompt 5) prometia remover WizardContainer e simplificar, mas a complexidade da dual API foi apenas movida para o hook useGeradorStep1.

O problema real: Express usa GeradorStep1 via v2 (sem session) enquanto Smart Blocks usa v3 (com session). Mas o Express não tem WizardSession — usa ExpressState. Isso cria dois contextos diferentes usando o mesmo componente de formas incompatíveis, o que vai causar bugs quando alguém alterar um path sem saber do outro.
📄 components/shared/GeradorStep1.tsx:15-45 (PropsV2/V3) · hooks/useGeradorStep1.ts
🟡 P1 — Issues importantes
P1 ExpressState.qualidadeIa tipado como string (deveria ser QualidadeIa)
ExpressState declara qualidadeIa: string (linha 16 de useExpressGeneration.ts), não QualidadeIa ('padrao' | 'avancado'). O tipo QualidadeIa foi criado e unificado na R3 — mas o Express não foi atualizado. Qualquer valor pode ser passado para qualidade_ia no INSERT, incluindo valores inválidos que violam o CHECK constraint do banco.
📄 hooks/useExpressGeneration.ts:16 — qualidadeIa: string → deve ser QualidadeIa
P1 AgenteSession tem campos de feedback redundantes (feedbackNota, feedbackComentario) que não são usados
AgenteSession (types.ts) declara feedbackNota?: number | null e feedbackComentario?: string | null. Esses campos existiam antes da migração para JSONB (feedback_advogado). Agora o AgenteStep6Conclusao usa estado local (useState) para nota e comentário, e salva direto no banco como JSONB — os campos na interface ficaram orphaned e criam confusão sobre qual é a fonte de verdade.
📄 types.ts:AgenteSession.feedbackNota/feedbackComentario → remover ou usar
P1 ExpressStep4Rascunho recebe historicoId mas nunca usa para nada funcional
A prop historicoId: string é declarada na interface do ExpressStep4Rascunho, mas não aparece em nenhuma chamada Supabase ou lógica dentro do componente (verificado via grep). O componente trabalha com textoGerado e generationStatus vindos do hook — mas se precisar salvar estado intermediário ou permitir "continuar de onde parou", o historicoId seria necessário. Por ora é prop morta que confunde quem mantém.
📄 components/express/ExpressStep4Rascunho.tsx:34 — historicoId declarado, não usado
P1 Agente tem 6 steps mas WizardStepper provavelmente mostra 5 — inconsistência UX
OliviaGeradorAgente renderiza steps 0–5, mas o array AGENTE_STEPS precisa ser verificado — se tiver 5 entradas e o WizardStepper mostrar step 5 como "além do último", o usuário vê o stepper quebrado (índice fora do range). O Step 4 (AgenteStep5Conclusao) e o Step 5 (AgenteStep6Conclusao = feedback) são dois componentes de conclusão que poderiam ser unificados.
📄 components/gerador/OliviaGeradorAgente.tsx:16 (AGENTE_STEPS) — verificar count vs session.step range
P1 ExpressStep5Conclusao busca dados via Supabase no mount — sem loading skeleton adequado
O componente faz um fetchDados() no useEffect de mount e mostra Loader2 enquanto carrega. Mas o historicoId já deveria ter os dados finais disponíveis via polling do hook (que já selecionou google_doc_url, tokens_usados, etc.). Há duplicação de query: o hook já buscou esses dados e os tem em state.googleDocUrl, state.tokensUsados — o Step5 faz uma segunda query desnecessária para os mesmos campos.
📄 components/express/ExpressStep5Conclusao.tsx:32-55 — re-fetch desnecessário, dados disponíveis no hook state
🔵 P2 — Melhorias
P2 Smart Blocks sem seletor de qualidade IA (sempre usa default do banco)
Enquanto Express e Agente têm seletor de qualidade IA (padrao/avancado), o Smart Blocks não tem. O SmartBlocksState não tem campo qualidadeIa e o INSERT não passa o campo. O usuário Smart Blocks não tem controle sobre custo vs qualidade — sempre usa o default. Pode ser intencional (Smart Blocks é o modo "manual", não IA pesada), mas se for, deveria ser documentado.
📄 hooks/useSmartBlocksGeneration.ts · components/smart-blocks/SmartBlocksStep1.tsx
P2 Acentos removidos em textos visíveis ao usuário (AgenteStep3, Step6)
Vários textos no AgenteStep3Editor e AgenteStep6Conclusao perderam acentuação: "Instrucoes adicionais", "Geracao rapida", "Analise profunda", "Conclusao", "Avaliacao da peticao", "Comentario". Isso aparece diretamente na UI do usuário — profissional que usa o sistema vê português sem acento.
📄 components/agente/AgenteStep3Editor.tsx + AgenteStep6Conclusao.tsx — textos sem acento
P2 validatePlaceholders — função toCamelCase não exportada, mapa de placeholders hardcoded
A lib validatePlaceholders.ts tem PLACEHOLDER_LABEL hardcoded com 9 placeholders. Se o sistema tiver modelos com outros placeholders (ex: NUMERO_PROTOCOLO, DATA_AUDIENCIA), a validação retorna o nome técnico bruto sem label amigável. Ideal seria buscar os labels do banco ou tornar o mapa extensível. Também: toCamelCase é função interna não exportada, não testável.
📄 lib/validatePlaceholders.ts:PLACEHOLDER_LABEL hardcoded
