🎯 Problema Raiz

O Google Docs exporta HTML sem estilos inline — apenas <p><span>texto</span></p>. Negrito, itálico, tamanho de fonte e recuos não sobrevivem ao export HTML. Os blocos de texto no banco são texto puro com HTML entities — zero formatação.

A jurisprudência especificamente tem dois problemas: (1) fonte igual ao corpo (deveria ser 10–11pt) e (2) a origem do texto vem dos blocos do banco sem marcação de "isso é jurisprudência". O Claude gera o trecho mas a skill de geração de DOCX não sabe diferenciar jurisprudência de texto normal.

2. Decisão Estratégica — Google Apps Script vs Solução Interna

🔧 Google Apps Script

Como funciona: Script rodando no GDocs/GDrive, recebe o texto gerado e aplica formatação diretamente no documento via Google Docs API nativa.


Vantagens:

  • API nativa do GDocs — formatação 100% fiel
  • Sem depender de bibliotecas DOCX
  • Pode ler o template original e copiar estilos pixel a pixel
  • Já tem infraestrutura de Drive instalada no Midas

Desvantagens:

  • Deploy/auth complexo (OAuth service account)
  • Depende de serviço externo (Google) sempre online
  • Mais um endpoint para manter
  • Latência extra (chamada HTTP → GAS → GDocs)
  • Debugging difícil (logs no GAS, não no servidor OlivIA)

⚡ Solução Interna (python-docx)

Como funciona: Skill gera-peticao.mjs gera o DOCX diretamente usando python-docx (ou docx.js) com estilos programáticos — sem depender de exportação Google.


Vantagens:

  • Controle total — cada parágrafo, cada estilo
  • Sem dependências externas — roda no Mac Mini
  • Mais rápido (sem round-trip HTTP)
  • Debuggable com logs normais
  • Jurisprudências com fonte diferente: 3 linhas de código
  • Já tem o pipeline de geração no lugar

Desvantagens:

  • Precisa de Python + python-docx instalado no Mac Mini
  • Template base precisa ser criado em código (não visual)
  • Primeiro setup leva ~2h

🏆 Decisão: Solução Interna com python-docx

O GAS seria necessário se a OlivIA precisasse editar documentos já existentes no Drive ou se a formatação fosse tão complexa que impossibilitasse reprodução programática. Não é o caso.

O problema é bem definido: jurisprudências com fonte 10pt, recuo e itálico + recuo de parágrafo + espaçamento. São estilos estáticos que python-docx resolve com precisão cirúrgica. GAS seria over-engineering aqui.

Dado que já temos o pipeline rodando no Mac Mini com Node.js, a skill gera-peticao.mjs pode chamar um script Python que gera o DOCX — ou usar a lib docx (npm) diretamente no Node.

3. Arquitetura da Solução

Como o DOCX será gerado com formatação correta

O Claude gera a petição em Markdown estruturado com marcações semânticas. A skill de geração converte esse Markdown para DOCX usando a lib docx (npm) com estilos explícitos.

3.1 Marcações Semânticas no Output do Claude

O prompt S6 (gera-peticao) deve pedir ao Claude para usar marcações especiais que a skill saiba interpretar:

## DO DIREITO
### Negativa de Prestação de Serviço

Texto normal do corpo da petição com recuo de parágrafo.

>>> JURIS
BRASIL. Superior Tribunal de Justiça. Recurso Especial nº 1.649.087...
"A responsabilidade civil objetiva do fornecedor de serviços..."
<<< JURIS

Outro parágrafo normal.

>>> CITE
Neste sentido, o Enunciado 37 do FONAJE...
<<< CITE

A skill detecta os marcadores >>> JURIS / <<< JURIS e aplica estilo específico no DOCX.

3.3 Sanitização — Remover Instruções Internas

Antes de passar o texto para o DOCX, a skill sanitiza:

// Padrões de instrução interna a remover
const INSTRUCOES_PATTERNS = [
  /JUNTAR\s+O?\s*PRINT\b.{0,100}/gi,
  /COLOCAR\s+O?\s*PRINT\b.{0,100}/gi,
  /INSERIR\s+(?:AQUI|O)\b.{0,100}/gi,
  /SE\s+FOR\s+O\s+CASO\b.{0,150}/gi,
  /ALTERAR\s+(?:PARA|CONFORME)\b.{0,100}/gi,
  /\[[\w\s\u00C0-\u024F]{2,60}\]/g,   // [DATA DA RESTRIÇÃO] etc

6. Resumo Executivo

Estratégia: solução interna com lib docx (npm)

GAS é desnecessário. O problema é bem delimitado e solucionável em código Node.js puro. Sem novos serviços, sem autenticação OAuth, sem round-trips.

Jurisprudências: 10pt + recuo 3cm + itálico

Marcador semântico >>> JURIS no output do Claude → estilo aplicado na geração do DOCX.

Sanitização: obrigatória antes de gerar

JUNTAR/COLOCAR/INSERIR/ALTERAR + texto todo em CAPS + [VARIÁVEIS] — todos removidos por regex antes de montar o DOCX.

Subtítulos duplicados: instrução no prompt S6

"Os subtítulos de seção já estão nos blocos. Não adicione títulos de seção — adapte apenas o corpo."
