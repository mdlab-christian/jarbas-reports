2. Problemas residuais — análise técnica completa

Problema A — Jurisprudências: fonte menor + formato diferente do original

No GDocs, as jurisprudências são coladas no corpo do documento com formatação especial: fonte 10pt, itálico, recuo bilateral (esquerda + direita). O puppeteer atual usa blockquote com recuo somente à esquerda (4cm) e 11pt — visualmente parecido, mas não idêntico.

O texto das jurisprudências no banco (texto_integral) chega como texto plano sem formatação HTML. Quando o GDocs original insere, ele copia a formatação do GDoc da jurisprudência.

Atual: <blockquote><p><em>"texto da decisão..."</em></p> <p><strong>TJSC — 50135...</strong></p></blockquote> → recuo só esquerda, font-size 11pt, sem recuo direito Original: Recuo de 3cm BILATERAL (esquerda + direita), fonte 10pt, espaçamento 1.0 Número do acórdão em linha separada, tribunal + câmara + relator + data

Problema B — Blocos sem html_rico (814/1244 blocos)

Os arquivos .txt dos blocos no Google Drive não são acessíveis via OAuth externo — retornam 404. O importer só conseguiu importar ~430 blocos cujos GDocs estavam compartilhados publicamente ou com a conta de serviço.

Resultado: 814 blocos usam o campo conteudo (texto com HTML entities &ccedil;) sem bold/italic. O S5 faz decode das entities mas não recupera a formatação (negrito em títulos de seção).

Problema C — DOCX sem conteúdo (1.5KB)

O S8 usa docxtemplater com o template-base.docx (que tem um único placeholder {conteudo}). O template recebe o HTML mas o HtmlModule não está instalado no mac-mini, então só o template vazio é gerado.

Biblioteca necessária: docxtemplater-plugin-html (ou @xl-soft/docxtemplater-html-module) Status: NÃO instalada em /Users/Shared/olivia-skills/node_modules/

Problema D — Acesso Drive: precisamos da Service Account ou compartilhamento

O OAuth usa as credenciais do cliente GOOGLE_CLIENT_ID_MDFLOW — que tem acesso apenas aos arquivos explicitamente compartilhados com o email autorizado. Os 814 blocos .txt estão em uma pasta do Drive que não foi compartilhada.

3. ✅ Não precisamos do GAS — temos tudo no Node.js

O GAS seria necessário se precisássemos de acesso ao Drive via conta do usuário. Mas já temos OAuth funcionando. A única diferença é o compartilhamento dos arquivos. Solução:

SOLUÇÃO DEFINITIVA
Compartilhar a pasta "Blocos de Texto" do Drive com o email do OAuth — ou usar a Service Account do Google Cloud (que já existe no projeto). Depois rodar o importer novamente. Isso resolve B e D de uma vez.
Email OAuth atual: perinottipazadvogados.com (titular do GOOGLE_CLIENT_ID_MDFLOW) Pasta no Drive: OlivIA/MP Advocacia/Blocos de Texto/ Ação necessária: Compartilhar essa pasta (ou o Drive inteiro) com o email do OAuth

Alternativamente, usar text_integral + decode HTML — o conteúdo textual já está no banco, só falta o bold/italic. Isso pode ser resolvido com um parser HTML simples que aplica negrito nos títulos de seção detectados por padrão (CAPS ou primeira linha do bloco).

5. GAS vs Node.js puro — decisão definitiva

GAS (Google Apps Script)

✅ Acesso nativo ao Drive (sem OAuth externo)

✅ DriveApp.getFileById() funciona para qualquer arquivo

✅ Pode copiar formatação Word diretamente

❌ Rodaria fora da OlivIA (script separado no Google)

❌ Não tem acesso ao Supabase nativamente

❌ Latência alta (chamada HTTP para GAS)

❌ Mantém dependência de infraestrutura Google

Node.js + Drive API (atual)

✅ Tudo na mesma stack — sem dependência externa

✅ OAuth já funciona para arquivos compartilhados

✅ mammoth converte DOCX → HTML com fidelidade

⚠️ Precisa que os arquivos estejam compartilhados

✅ Fases 1-3 resolvem 90% dos problemas de formatação

✅ Fase 4 resolve os 10% restantes (com compartilhamento)

Recomendação: Node.js puro. Não precisa de GAS. A única ação necessária fora do código é compartilhar a pasta no Drive (5 minutos).
