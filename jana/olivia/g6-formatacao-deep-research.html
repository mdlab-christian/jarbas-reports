1. Por que a formatação está sendo perdida

Origem dos dados

Todo o conteúdo da OlivIA foi criado no Google Docs e importado para o banco. A cadeia é:

Google Docs → Export HTML → olivia_modelos_peticao.conteudo_template (HTML) Google Docs → Export HTML → olivia_blocos_texto.conteudo (HTML)

Problema: Export HTML do Google Docs perde formatação

O Google Docs exporta em HTML sem classes CSS nem estilos inline para formatação visual. Os elementos gerados são apenas <p><span>texto</span></p>. Negrito, itálico e outros estilos não estão no HTML exportado.

Verificação real no banco:

Tags HTML presentes nos blocos: NENHUMA — apenas entidades HTML (&ccedil; etc.) Template: apenas <p><span> sem class, style, font-weight, <b> ou <strong> Blocos: texto puro com HTML entities — zero tags de formatação

O que precisamos

O arquivo DOCX original do Google Docs contém toda a formatação em XML Word (runs com <w:b/>, <w:i/>, estilos de parágrafo, etc.). Temos duas opções:

OpçãoComo funcionaViabilidade
A — Export DOCX via Google Drive API GET /drive/v3/files/{id}/export?mimeType=docx → DOCX com formatação original ⚠️ Precisa OAuth correto
B — DOCX stored no Supabase Storage Se o DOCX já foi exportado e salvo no bucket olivia-modelos → usar direto ❌ Bucket vazio
C — Re-importar com formatting preservation GAS exporta cada Doc como DOCX → salva no bucket → skill usa mammoth.js para HTML com formatação ✅ Mais robusto
D — HTML fiel ao GDocs (mammoth reverse) mammoth.js converte DOCX → HTML preservando negrito, itálico, listas ✅ Junto com C

2. Solução técnica recomendada — Opção C+D

A solução completa tem dois componentes independentes:

2A. One-time import: Google Drive → DOCX → Supabase + HTML rico

Script GAS ou n8n que roda uma vez (e repete quando houver novos docs):

Para cada olivia_modelos_peticao e olivia_blocos_texto com google_doc_id: 1. GET /drive/v3/files/{google_doc_id}/export?mimeType=docx 2. POST /storage/v1/object/olivia-modelos/{versao_id}/template.docx ← bucket 3. Converter DOCX → HTML com mammoth.js (preserva b, i, listas) 4. UPDATE olivia_modelos_peticao SET conteudo_html_rico = <html com formatação> UPDATE olivia_blocos_texto SET conteudo_html_rico = <html com formatação>

Credenciais disponíveis: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GOOGLE_REFRESH_TOKEN já estão no .env. O access_token atual retornou 401 — provável que o token foi gerado para outro escopo. Precisa ser re-autorizado com escopo https://www.googleapis.com/auth/drive.readonly.

2B. Pipeline S5: usar conteudo_html_rico se disponível, fallback para conteudo

// Em olivia-smartblocks.mjs const bloco = await buscarMelhorBloco(tagId); const html = bloco.conteudo_html_rico // com <b>, <i>, <u> preservados || bloco.conteudo; // fallback: HTML entities (atual) substituir(placeholder, html);

2C. S8 (cria-doc): DOCX via docxtemplater com template rico

// Em olivia-cria-doc.mjs // 1. Baixar template.docx do bucket (tem formatação Word original) const docxTemplate = await downloadFromBucket('olivia-modelos', versao_id + '/template.docx'); // 2. Docxtemplater com HTML injection via htmltodocx const doc = new Docxtemplater(docxTemplate, { modules: [new HtmlModule()], // converte {{html_placeholder}} → rich text }); doc.render({ conteudo_bloco1: htmlRico, ... }); // 3. Resultado: DOCX com formatação idêntica ao modelo original

Módulo necessário: docxtemplater-module-html — disponível no npm, permite injetar HTML formatado direto no DOCX.

SELECT p.numero_processo, p.reu_nome as empresa_re, p.valor_causa, p.valor_condenacao, c.nome as parte_autora, c.estado_civil, cv.nome as comarca_vara, a.nome as advogado_nome, a.oab, a.oab_uf, a.cabecalho_escritorio, a.meta FROM processos p LEFT JOIN clientes c ON c.id = p.cliente_id LEFT JOIN advogados a ON a.id = p.advogado_id LEFT JOIN comarcas_varas cv ON cv.id = p.comarca_vara_id WHERE p.id = $processo_id
