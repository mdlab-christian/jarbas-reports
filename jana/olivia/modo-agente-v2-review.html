Bugs, Gaps e Issues

🚨 [P0] Tunnel jana.jarbas-mdlab.com não existe / server não rodando

BLOQUEADOR

O frontend chama https://jana.jarbas-mdlab.com/olivia/gerar-agente mas o servidor não está rodando na porta 18788 e o tunnel Cloudflare não está configurado. Qualquer tentativa de usar o Modo Agente cai imediatamente no fallback JARBAS (que usa a skill legada, não a v2).

✅ Fix: (1) iniciar node ~/Desktop/modo-agente/server.mjs, (2) configurar tunnel Cloudflare porta 18788 → jana.jarbas-mdlab.com

⚠️ [P1] modelo_grupo_id vs modelo_id — mismatch frontend→backend

P1

O wizard passa modelo_id: ws.modelo_id ?? ws.modelo_grupo_id no gerarPeticao. Mas ws.modelo_id é o ID do modelo_peticao (versão específica) enquanto modelo_grupo_id é o grupo. A skill v2 usa modelo_id para buscar instrucoes_ia na tabela olivia_modelos_peticao. Se vier o grupo ID, a query falha silenciosamente e o prompt fica sem instruções do modelo.

✅ Fix: garantir que Step 1 sempre popula ws.modelo_id com o ID da versão publicada, não do grupo. Ou adicionar resolução automática no fetchContexto.

⚠️ [P1] GAS_URL_NOVO pode ser endpoint SmartBlocks (espera payload diferente)

P1

O .env tem GAS_URL_NOVO apontando para o GAS Script. Mas dependendo de qual versão do GAS está publicada (Express mode ou SmartBlocks), o payload esperado é diferente. A skill v2 envia peticao_html diretamente, enquanto o SmartBlocks espera blocos + titulo_documento. Se o GAS estiver em SmartBlocks mode, o Google Doc não vai ser criado.

✅ Fix: confirmar qual modo do GAS está publicado. Se necessário, criar endpoint dedicado para HTML puro (modo Express) separado do SmartBlocks.

⚠️ [P1] advogado_id não enviado no payload de geração

P1

O frontend tem advogado_id no wizardState (Step 1 seleciona o advogado), mas o objeto params em handleGerar não inclui advogado_id. A skill tenta recuperar via contexto.advogado?.id (busca pelo processo), mas se o processo não tiver advogado associado, o campo fica null no histórico e os dados do advogado podem ficar errados na petição.

✅ Fix: adicionar advogado_id: ws.advogado_id nos params do handleGerar e usar no salvarHistorico.

🔸 [P2] Server sem autostart — cai se MacBook Air reiniciar

P2

O server.mjs precisa ser iniciado manualmente. Não há LaunchAgent nem pm2 configurado. Após qualquer reinicialização, o Modo Agente vai para fallback JARBAS sem aviso.

✅ Fix: criar LaunchAgent plist ou adicionar ao processo de inicialização do OpenClaw.

🔸 [P2] Fallback JARBAS usa skill legada, não skill v2

P2

Quando JANA offline, o hook usa JARBAS_BASE_URL. Mas o JARBAS não tem a skill v2 — tem a skill legada (skill_olivia_gerar_peticao.mjs). Isso significa que ao cair no fallback, a triagem inteligente, Vision e revisor v2 não estão disponíveis. O usuário pode não perceber que está usando o motor antigo.

✅ Fix (curto prazo): mostrar badge "Motor: JARBAS (legado)" quando em fallback. Longo prazo: implantar skill v2 no JARBAS também.

🔸 [P2] StepAnaliseTriagem: modeloGrupoId prop não usado internamente

P2

O componente recebe modeloGrupoId como prop mas não a passa para gerarTriagem (que usa apenas modeloId). Se modeloId for null mas modeloGrupoId tiver valor, a triagem falha.

✅ Fix: const effectiveModeloId = modeloId ?? modeloGrupoId e passar para gerarTriagem.

🔸 [P2] STATUS.md desatualizado — ainda mostra tudo como ⏳

P2

O STATUS.md em ~/Desktop/modo-agente/ ainda lista todos os itens como "A criar". Isso é confuso se alguém (ou outro CC) tentar retomar o contexto — vai achar que nada foi feito.

✅ Fix: atualizar STATUS.md para refletir o estado real.
