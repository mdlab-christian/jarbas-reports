Dave Swift — "My OpenClaw Is Forgetting Everything" (19min)
Problema de memória → Solução Obsidian · configuração openclaw.json · indexação
▼
🎯
Problema central: a memória do OpenClaw "mente".

A promessa de memória persistente do OpenClaw não funciona out-of-the-box. O Obsidian resolve isso criando um vault como second brain do agente — os arquivos .md são a memória real.

Principais insights:

  • Vault = pasta de markdown files. Não inventou nada novo — o agente já opera com markdown.
  • Configurar o vault no openclaw.json (bloco memory) — agente indexa automaticamente ao reiniciar.
  • Dica crítica: sem espaços no nome do vault — "Dave's Vault" quebrou a indexação, "Daves-Vault" funcionou.
  • Pedir ao agente para construir a estrutura de pastas (ele sabe sobre o artigo do Heinrich → constrói corretamente).
  • Obsidian Sync $4-5/mês — sincroniza entre Mac Mini e MacBook sem configuração complexa.
  • Segurança: vault compartilhado com LLM cloud (Anthropic) → tudo que está no vault pode ser visto. Considerar o que colocar.
  • Tailscale resolve acesso remoto ao vault quando fora de casa.
V2
Alex McFarland — "Perfect Co-Working System" (24min)
user.md como fonte da verdade · pasta organizada · 2 máquinas · plugins Obsidian
▼
🎯
Foco em co-working — agente como funcionário, não substituto.

A sacada principal: tratar o agente como novo funcionário. Começa pelas tarefas que você nunca teve tempo de fazer. Vai dando mais responsabilidade conforme ganha confiança.

Principais insights:

  • user.md é o arquivo mais importante. Tudo sobre você/empresa em um lugar. O agente SEMPRE carrega. Substituiu múltiplos "context profiles" espalhados.
  • Obsidian como janela para o workspace do agente — visualiza e edita markdown files do workspace direto no Obsidian.
  • Abrir o vault do OpenClaw workspace no Obsidian = ver tudo: SOUL.md, MEMORY.md, skills, dashboards.
  • Plugin 1: Data Files Editor — visualizar/editar JSON, TXT e outros non-markdown no Obsidian.
  • Plugin 2: HTML Reader — renderizar HTMLs no Obsidian. Dashboards do agente aparecem no Obsidian.
  • Settings → Sync → "Sync all other types" — necessário para sincronizar HTML, JSON, TXT entre máquinas.
  • Dar ao agente conta própria: email, número, cartão, Obsidian account separado. Shared vault via Obsidian Sync.
  • Dois papéis claros: sua máquina = trabalho criativo + cloud code; Mac Mini = overflow + tarefas que não tinha tempo.
  • todo.md compartilhado no workspace = task list colaborativa entre você e o agente.
V3
Rafael Quintanilha (QuantBrasil) — "Cérebro para Agentes IA" (27min)
CLI Obsidian · Headless Sync · Zettelkasten · recuperação de notas via agente
▼
🎯
Foco: Obsidian CLI + Headless Sync = memória do agente rodando sem interface.

Rafael usa VPS (equivalente ao nosso Mac Mini). A CLI do Obsidian permite operações no vault sem abrir o aplicativo — ideal para o agente rodar sozinho.

Principais insights:

  • Obsidian CLI lançada recentemente — comandos: abrir daily note, adicionar item, buscar notas, encontrar notas órfãs. Menos tokens que operações manuais.
  • Headless Sync — vault sincronizado na VPS/Mac Mini sem o cliente Obsidian aberto. O agente opera nas notas sem interface visual.
  • Comando obsync — sincroniza vault antes de operar. Garante que o agente tem versão mais recente.
  • Para notas acionáveis (checklists, tasks de projeto) → dentro do repositório. Para conhecimento macro, direção, filosofia → no Obsidian vault.
  • Anti-Zettelkasten: delegar a escrita dos seus pensamentos para a IA remove o aprendizado. IA escreve o factual (quem veio, o que aconteceu). Pensamentos e insights — sempre seus.
  • Fricção zero para capturar: WhatsApp → agente → Obsidian com links criados automaticamente.
  • Notas como claims (afirmações), não tópicos. Ex: "smart-blocks-devem-vir-antes-da-geracao" não "smart-blocks".
  • Podcast Greg Isenberg + Vim: comandos para contradizer tópico, desafiar tópico, responder como outra pessoa — baseado nas notas do vault.
🗂️ Decisão: Estrutura de Vaults
✅
Vaults separados — decisão correta.

Christian confirmou: mdlab-vault (não jarbas-vault) para JANA + JARBAS, e olivia-vault separado para jurídico. Os 3 vídeos reforçam: cada vault tem uma filosofia diferente e não deve ser misturado.

🧠 mdlab-vault

Para JANA + JARBAS — conhecimento técnico, projetos, decisões, arquitetura MdLab.

  • Sync: Mac Mini + MacBook Air (ambos via Obsidian Sync)
  • Agente: JANA lê e escreve; JARBAS também acessa
  • Conteúdo: projetos, decisões, lições aprendidas, stack, specs
  • Filosofia: conhecimento técnico operacional
  • Path Mac Mini: /Users/jarbas/mdlab-vault/
  • Path MacBook Air: ~/mdlab-vault/

⚖️ olivia-vault

Para OlivIA — conhecimento jurídico, RAG, processos (já existe).

  • Sync: Mac Mini only (Headless — OlivIA opera sem interface)
  • Agente: OlivIA lê; JANA pode escrever via SSH
  • Conteúdo: modelos, blocos, juris, tags, playbooks
  • Filosofia: conhecimento jurídico RAG-first
  • Path: /Users/Shared/olivia-vault/ (já existe)
  • Isolado — clientes nunca misturam com técnico
⚡ Melhorias recomendadas — O que implementar
1
Criar mdlab-vault com estrutura correta
Pasta única compartilhada entre JANA e JARBAS · filosofia técnica · CLAUDE.md
▼

Criar a estrutura no Mac Mini e sincronizar para o MacBook Air via Obsidian Sync:

mdlab-vault/ ├── 00_inbox/ # captura rápida — zero fricção ├── 01_projetos/ # MdFlow, OlivIA, Controller, etc. │ ├── mdflow/ │ ├── olivia/ │ └── controller/ ├── 02_decisoes/ # decisões arquiteturais permanentes ├── 03_licoes/ # lições aprendidas por projeto ├── 04_stack/ # documentação de tecnologias usadas ├── 05_pessoas/ # equipe, perfis, contexto ├── 06_specs/ # specs de features em desenvolvimento ├── 07_ideias/ # ideias não prontas para projeto ├── 08_archive/ # conteúdo inativo ├── 09_system/ # templates e configs ├── CLAUDE.md # filosofia do vault para o agente └── INDEX.md # índice com 1 linha por nota (orientação rápida)
💡
CLAUDE.md é o equivalente ao nosso SOUL.md mas para o vault — ensina o agente como navegar, o que priorizar, como nomear notas, qual a filosofia.

Regra do Heinrich: notas com nomes de afirmações, não tópicos. Ex: smart-blocks-devem-vir-antes-da-geracao.md em vez de smart-blocks.md.

2
Configurar Obsidian Sync entre Mac Mini e MacBook Air
Vault compartilhado · plugins Data Files Editor + HTML Reader · sync all types
▼

Passos:

  1. Criar vault mdlab-vault no Mac Mini (usuário jarbas) e abrir no Obsidian
  2. Habilitar Obsidian Sync (já contratado) — compartilhar vault para o email da JANA/MacBook Air
  3. Instalar plugins nos 2 dispositivos: Data Files Editor + HTML Reader
  4. Settings → Sync → Toggle: "Sync all other types" em AMBAS as máquinas (sincroniza HTML, JSON, TXT)
  5. Configurar openclaw.json do JARBAS para indexar o vault (bloco memory)
  6. Configurar openclaw.json da JANA para indexar o mesmo vault
  7. Reiniciar ambos os agentes para indexação inicial
⚠️
Sem espaços no nome do vault — Dave Swift descobriu que espaços quebram a indexação. Usar mdlab-vault (com hífen).

Resultado: abrir o MacBook Air com Obsidian e ver em tempo real tudo que o JARBAS criou no Mac Mini. HTMLs de dashboards renderizam direto no Obsidian. JSON configs editáveis visualmente.

3
Configurar openclaw.json para indexar o vault
Bloco memory no config · SessionStart hook · INDEX.md
▼

No /Users/jarbas/.openclaw/openclaw.json (JARBAS) e no equivalente da JANA, adicionar/atualizar o bloco de memória:

"memory": { "enabled": true, "path": "/Users/jarbas/mdlab-vault", "indexOnStart": true, "watchForChanges": true }

Hook SessionStart — o agente roda automaticamente ao iniciar sessão para se orientar:

"hooks": { "SessionStart": [{ "type": "command", "command": "tree -L 3 -a -I '.git|.obsidian' --noreport /Users/jarbas/mdlab-vault" }] }

INDEX.md — arquivo especial que o agente lê para orientação rápida sem abrir cada nota:

# INDEX — mdlab-vault ## Projetos ativos - [[01_projetos/mdflow/readme]] — MdFlow LegalTech · stack React/TS/Supabase - [[01_projetos/olivia/readme]] — Gerador de petições com IA · V1 em desenvolvimento - [[01_projetos/controller/readme]] — Agente Controller EPROC · planejamento ## Decisões permanentes - [[02_decisoes/olivia-sem-jarbas]] — OlivIA não usa JARBAS para geração - [[02_decisoes/gas-eliminado]] — Puppeteer substituiu GAS - [[02_decisoes/dual-format-obrigatorio]] — sempre HTML + MD ## Lições críticas - [[03_licoes/cc-nao-faz-a-mais]] — CC deve fazer exatamente o pedido - [[03_licoes/n8n-fetch-quebrado]] — usar require('https'), não fetch() [...]
4
Migrar decisões + lições para o vault
memory/decisions.md · memory/lessons.md → notas separadas no vault com links
▼
💡
O vault substitui o sistema de memory/*.md fragmentado.

Atualmente temos memory/decisions.md, lessons.md, projects.md — arquivos monolíticos. No vault, cada decisão/lição vira uma nota separada com links contextuais. O agente encontra por busca semântica, não lendo o arquivo inteiro.

Migração:

  • Cada decisão em memory/decisions.md → nota individual em 02_decisoes/
  • Cada lição em memory/lessons.md → nota individual em 03_licoes/
  • Projetos ativos → 01_projetos/{projeto}/readme.md
  • Manter os memory/*.md como backup/índice por enquanto

Formato de cada nota (padrão afirmação):

--- tags: [decisao, olivia, arquitetura] data: 2026-03-21 projeto: olivia --- # OlivIA não depende do JARBAS para geração de petições O motor de geração é 100% interno via skills da OlivIA. [[jarbas]] não participa do fluxo de geração — apenas roteia chamadas. **Por quê:** risco de quebra de contexto + limite de 4096 chars do Telegram **Decidido em:** 2026-03-21 · [[01_projetos/olivia/readme]] **Irrevogável:** sim

🧠 Filosofia de cada vault (Heinrich adaptado)
📖
Heinrich: "todo vault precisa de sua própria filosofia."

O mesmo padrão (markdown + links + CLAUDE.md), regras diferentes. Exatamente como codebases — você não usa a mesma arquitetura para um CLI e um web app.

🧠 mdlab-vault — filosofia

  • Profundidade > breadth. Uma decisão bem documentada vale mais que 10 rascunhos.
  • Notas como afirmações. Título é uma claim, não um tópico.
  • Links contextuais. Dentro da frase, não como referência no rodapé.
  • Cada nota autocontida. Pode ser lida sem precisar das outras para fazer sentido.
  • Agente enriquece, humano decide. IA registra fatos. Christian escreve insights.
  • Todo projeto tem README. Uma nota de orientação que o agente lê antes de agir.

⚖️ olivia-vault — filosofia

  • RAG-first. Toda nota é fonte de contexto jurídico para a OlivIA.
  • Formato YAML frontmatter obrigatório. Tags, tribunal, empresa — para filtragem.
  • Notas composáveis. Cada bloco de argumentação funciona sozinho + em combinação.
  • Curação > volume. 1 jurisprudência bem tageada vale mais que 10 genéricas.
  • Crescimento contínuo. OlivIA adiciona novas notas ao vault após cada petição gerada.
  • Nunca misturar com técnico. Isolamento absoluto de dados de clientes.
