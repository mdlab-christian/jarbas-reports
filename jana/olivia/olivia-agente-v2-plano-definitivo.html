Conceito Central

A lógica que une SmartBlocks + IA + dados reais

SmartBlocks como Esqueleto — Agente como Redator MasterInsight chave
SMARTBLOCKS (atual): Modelo de petição → Tags → Advogado escolhe blocos de texto → GAS monta o Google Doc Resultado: petição mecânica, determinística, sem análise de provas, sem jurimetria AGENTE v2.0 (novo): Modelo de petição → Tags → Agente RESOLVE as tags com blocos aprovados do banco + analisa provas com Vision + cruza com dados de restrição do processo + consulta jurimetria real (50k+ decisões) + RAG jurídico → Monta peticao_base estruturada (o esqueleto que SmartBlocks geraria) → Claude Opus APRIMORA cada seção: mais argumentação, mais específico, mais forte → Revisor score loop ≥ 9.0 → GAS cria Google Doc com formatação ABNT preservada Resultado: petição de elite, com provas, jurimetria e argumentação cirúrgica KEY INSIGHT: O Agente não inventa do zero — ele tem um ESQUELETO VALIDADO (os blocos aprovados pelo escritório) e só enriquece o que já funciona.
🏷️
Tags = Perguntas
Cada tag no modelo de petição é uma "seção a preencher". A tag tem código, tipo, categoria de empresa e bloco padrão vinculado.
📦
Blocos = Respostas
1244 blocos de texto aprovados pelo escritório. Cada tag tem múltiplos blocos — o agente escolhe o melhor baseado no contexto do caso.
✍️
Opus = Aprimorador
Com a petição base montada, Claude Opus lê as provas, cruza com a restrição, injeta jurimetria e eleva cada argumento ao máximo.

Aprendizado Contínuo — O Agente que Evolui Sozinho

Como o sistema fica mais inteligente a cada petição gerada

Loop de Feedback AutomáticoAuto-evolutivo
  • Advogado aprova petição → feedback_advogado='aprovei' → entra no RAG de exemplos aprovados
  • Processo ganho (decisoes.outcome='GANHO') → petição usada vira exemplo de elite no RAG
  • Score revisor ≥ 9.0 → bloco de alta qualidade disponível para próximas gerações similares
  • Processo perdido → decisoes_analises extrai quais argumentos NÃO funcionaram → feed negativo ao revisor
  • Acordo aceito → jurimetria_acordos atualizada com valor real → calibra pedidos futuros
  • Decisão favorável de juiz específico → jurimetria_juizes atualizada → argumentos desse juiz priorizados
Como o Agente Aprende na Prática
// Ao gerar nova petição, fetchContextoRico() faz: // 1. RAG de exemplos aprovados (petições que ganharam) const exemplosAprovados = await supabase .from('olivia_historico') .select('texto_gerado, score_revisor') .eq('organizacao_id', orgId) .in('feedback_advogado', ['aprovei']) .eq('tipo_reu', tipoReu) .gte('score_revisor', 9.0) .order('score_revisor', { ascending: false }) .limit(2); // 2. Jurimetria da empresa (dados reais de vitórias/derrotas) const jurimetria = await supabase .from('jurimetria_empresas') .select('taxa_vitoria, indenizacao_media, argumentos_eficazes[]') .eq('empresa_id', empresaId) .eq('uf', processoUf) .single(); // 3. Argumentos que NÃO funcionaram (evitar) const feedNegativo = await supabase .from('decisoes_analises') .select('argumento, motivo_rejeicao') .eq('empresa_id', empresaId) .eq('outcome', 'PERDIDO') .limit(5); // → Tudo isso vai no prompt → Agente aprende

  • Mês 1: petições baseadas em blocos genéricos + jurisprudência padrão
  • Mês 3: petições com argumentos calibrados pelo histórico real de decisões do TJRS
  • Mês 6: o agente sabe exatamente quais argumentos funcionam para cada juiz e cada empresa
  • Mês 12: taxa de vitória mensurável — cada geração aumenta a probabilidade de ganho

Resposta: JANA/JARBAS pode substituir o n8n inteiramente?

Análise técnica honesta — Modo Express + SmartBlocks dentro da rede

Comparativo: n8n Railway vs Rede JANA/JARBAS
Aspecton8n Railway (atual)JANA/JARBAS Network
Disponibilidade☁️ 99.9% (Railway cloud)🏠 MacBook Air sempre ativo + Mini 24/7 = alta disponibilidade
Latência⚡ Railway próximo do Supabase🏡 Máquinas locais — pode ser mais lento em chamadas externas
Flexibilidade🔧 Limitado ao que o n8n suporta♾️ Node.js puro — qualquer lógica, qualquer lib, sem limites
Debugging😤 UI n8n é frágil, erros obscuros✅ console.log real, stack trace limpo, dev experience excelente
SSE / Streaming❌ n8n não suporta SSE nativamente✅ Express.js + EventEmitter = SSE nativo
Custo💸 Railway ~$20-30/mês✅ Zero — hardware já pago
Manutenção😰 Update n8n = risco de quebrar workflows✅ Código versionado em Git, testes unitários possíveis
Velocidade de desenvolvimento🐢 Visual + JSON = lento para lógica complexa⚡ TypeScript/JS direto = 3-5x mais rápido
Dependência externa❌ Railway fora do ar = sistema para✅ Rede interna independente de cloud externo

💡 Minha recomendação

  • Modo Agente v2 (este plano): JANA/JARBAS — 100% sem n8n. Mais estável, SSE nativo, melhor debugging.
  • Express + SmartBlocks: manter no n8n por enquanto — já funcionam, migrar é risco sem ganho imediato.
  • Futuro (V3): se o Agente v2 provar que a rede é confiável, migrar Express e SmartBlocks também. Eliminar n8n completamente. Custo zero, flexibilidade total.
  • Posso criar Express + SmartBlocks dentro da rede? Sim. Tenho o código do n8n, o GAS, todos os endpoints. É um sprint de 1-2 dias. O risco seria o período de migração — mas com testes E2E paralelos, é seguro.
