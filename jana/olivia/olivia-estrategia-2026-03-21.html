🏗️ Arquitetura: OpenClaw vs N8N — Visão de Longo Prazo

Você está certo sobre a direção

OpenClaw Skills são de fato a evolução natural do n8n para o seu caso. A diferença fundamental é:

• n8n = Workflows visuais, bom para integração simples, frágil quando tem lógica complexa
• OpenClaw Skills = Código Node.js direto, executa dentro do agente, tem acesso a contexto, memória, outras tools

Quem está migrando de n8n para Skills faz isso exatamente pelo que você está sofrendo: fetch() quebrado, dependência de runtime específico, debugging difícil.

🔄 Conceito Workflow de Skills

Você levantou um ponto importante: alguns Workflows do n8n têm dezenas de nodes em sequência. Como replicar isso?


Resposta: Uma Skill "orquestradora" que chama outras Skills em sequência:

// skill_olivia_express_v3.mjs
async function gerarExpressV3(params) {
  // 1. Buscar contexto do processo
  const contexto = await buscarContextoProcesso(params);
  // 2. Análise IA + triagem
  const triagem = await analisarDocumentos(contexto, params.docs);
  // 3. Montar outline
  const outline = await montarOutline(triagem, params.modelo);
  // 4. Gerar rascunho Claude
  const rascunho = await gerarComClaude(outline);
  // 5. Criar Google Doc
  const doc = await criarGoogleDoc(rascunho, params);
  return doc;
}

Isso é um "Workflow de Skills" — uma função que orquestra sub-funções, cada uma fazendo uma coisa.

🤖 Ideia: Skill de Conversão N8N → OpenClaw

💡 Viável e inteligente

Você mencionou criar uma Skill que lê um workflow n8n e converte para Skill. Aqui está como isso funcionaria:

// skill_n8n_to_skill.mjs
// Input: JSON exportado do n8n (workflow)
// Output: Arquivo .mjs de Skill OpenClaw equivalente

1. Parsear nodes do workflow JSON
2. Mapear cada node para equivalente em código:
   - HTTP Request → fetch() ou https.request()
   - Set node → atribuição JS
   - IF/Switch → if/switch JS
   - Code node → extrair código (já está em JS)
   - Merge node → Promise.all() ou desestruturação
   - Webhook trigger → parâmetro de entrada da função
3. Gerar função principal com o fluxo em sequência
4. Adicionar comentários explicando cada etapa
5. Retornar o .mjs gerado como texto

⚠️ Limitação importante

Workflows n8n complexos com 50-100 nodes não convertem 1:1. O conversor seria melhor como "ponto de partida" — gera 70% do código, você ajusta o resto. Para Express (16 nodes) e Smart Blocks (21 nodes), a conversão ficaria muito boa.

Mas hoje isso não é prioridade. Hoje é colocar o Modo Agente no ar.

🧩 Smart Blocks como Base para o Modo Agente

A Arquitetura de "Petição de Referência" é Muito Boa

A ideia de usar Smart Blocks para gerar uma petição-esqueleto antes do Agente gerar a petição final é arquiteturalmente sólida. É anti-alucinação por design:

1

Smart Blocks gera petição-esqueleto

Executa a lógica mecânica: modelo → tags → blocos → template → Google Doc. Sem IA, sem risco de alucinação. Resultado: petição estruturalmente correta mas genérica.

2

Claude recebe o esqueleto + contexto do caso

O Agente recebe: (a) petição-esqueleto como referência, (b) dados reais do processo, (c) documentos uploadados. Claude sabe que tem que seguir a estrutura mas personalizar com os fatos reais.

3

Claude gera petição final personalizada

Resultado: petição que segue os modelos do escritório (garantia de estrutura) mas com argumentação personalizada para o caso específico (qualidade jurídica real).

Na prática: a skill_olivia_agente_v2.mjs já tem referência ao GAS v4.0 no checkpoint. A lógica de chamar Smart Blocks internamente pode ser adicionada como função auxiliar dentro da skill — sem precisar do n8n.
