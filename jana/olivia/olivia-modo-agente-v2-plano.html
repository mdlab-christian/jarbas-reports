🧠 Conceito Central: Tags → Blocos
  • Tags são perguntas — campo variável do modelo de petição que precisa ser preenchido
  • Blocos de texto são respostas — conteúdo HTML aprovado que responde a cada tag
  • SmartBlock: mecânico — pega os blocos do Drive e monta
  • Agente v2: inteligente — gera a resposta para cada tag usando contexto do processo
  • Depois de resolver todas as tags → tem uma "petição base" completa
  • Essa petição base é entregue ao Claude Opus como referência
  • Claude aprimora a petição com provas, jurisprudência, análise de restrições
📡 SSE vs PollingNovo
  • Polling atual: frontend faz GET a cada 3s no jarbas_tarefas — ineficiente
  • SSE v2: JANA envia eventos em tempo real via Server-Sent Events
  • Frontend usa EventSource — conexão persistente, zero overhead
  • Eventos: progress, log, result, error
  • Fallback automático para polling se SSE falhar (resiliência)
  • Token streaming opcional — usuário vê a petição sendo escrita em tempo real

Pipeline de Geração

Node a node — inspirado no Express, executado sem n8n

🔍

N1 — fetchContexto() Manter da skill atual

Busca tudo: processo + cliente + empresa + advogado + modelo + documentos + movimentações + playbook + style guide + agentConfig. Adicionar: restricao_orgao_nome, restricao_valor, restricao_codigo_contrato do processo. Buscar também olivia_jurisprudencias filtradas por UF + tipo_reu.

🏷️

N2 — resolverTags() NOVO

Busca as tags do modelo de petição filtradas pela categoria da empresa. Para cada tag: verifica se existe bloco_texto aprovado no histórico → usa como resposta. Se não tem bloco: Claude Haiku gera resposta rápida. Monta peticao_base_html como o SmartBlock faria — esta é a referência estrutural para o Opus.

🔬

N3 — analisarProvas() Reescrever

Triple fallback: (1) Claude Vision nativo — envia PDF como imagem diretamente no prompt, analisa página a página. (2) Gemini 2.0 Flash Vision — fallback para PDFs muito grandes (>20MB). (3) Mistral OCR Latest — fallback para PDFs escaneados ruins. Foco jurídico: analisa cada prova contra os dados de restrição (restricao_orgao_nome, restricao_valor, restricao_codigo_contrato) verificando se a empresa comprovou a origem do débito conforme inversão do ônus probatório. Retorna analise_provas[] estruturada com: tipo_documento, comprova_debito (boolean), observacoes, pontos_fracos_reu[].

🔗

N4 — gerarTriagem() Integrar na skill

Elimina dependência do n8n WH-OLIVIA-TRIAGEM. A skill gera as perguntas diretamente usando Claude Haiku com contexto do processo + tags do modelo. Retorna perguntas[] com confiança ≥ 0.7 pré-preenchidas automaticamente. Frontend só mostra as com baixa confiança para o advogado confirmar. Latência: 3-5s (era 8-12s via n8n).

🧩

N5 — buildPromptMestre() Evolução

Monta o prompt completo para o Claude Opus com: system prompt do banco (ai_agents_config), peticao_base_html do N2 como referência estrutural, analise_provas do N3, triagem respondida, dados de restrição (valor/orgao/contrato), jurisprudências por UF+tipo_reu, style guide, playbook, RAG chunks. Prompt caching: system prompt marcado com cache_control: ephemeral — reduz custo ~80% em re-gerações da mesma sessão.

✍️

N6 — gerarPeticao() Claude Opus

Motor: Claude Opus 4-6 — sem intermediário, SDK direto. Streaming via SSE para o frontend. Instrução central: "você tem uma petição base gerada mecanicamente — melhore-a com base nas provas analisadas, jurisprudência e contexto do processo. Mantenha a estrutura e formatação HTML. Adicione análise estratégica das provas da ré." Anti-fingerprint: seed CNJ, 4 camadas de variação aplicadas.

⚖️

N7 — revisarPeticao() Manter + melhorar

Loop revisor com Claude Sonnet: score < 9.0 → regenera N6 com feedback estruturado (max 2x). Critérios: corretude_juridica (3pts), qualidade_escrita (3pts), completude_pedido (2pts), adequacao_caso (2pts). Novo: revisor também verifica se as provas da ré foram adequadamente rebatidas com base na analise_provas do N3. Retorna criticas[] + partes_reescrever[] específicas.

📊

N8 — parseRelatorio() NOVO

Claude Haiku analisa a petição gerada + contexto do processo e extrai: pontos_fortes[], pontos_fracos[], riscos_processuais[], chance_sucesso (0-100%), estrategia_recomendada, valor_sugerido_acordo. Salvo em meta.relatorio_processo do olivia_historico (mesmo formato que o Express salva no B8). Exibido no RelatorioCasoAccordion do Step 4.

📄

N9 — criarGoogleDoc() GAS v4.0

POST para GAS v4.0 (mesmo endpoint atual). Payload: blocos_html + tags_substituicao + configuracoes. Formatação preservada: o HTML é enviado para o GAS que cria o Google Doc mantendo: negrito, itálico, parágrafos alinhados, recuos, listas numeradas. Sem DOCX intermediário — direto HTML → Google Doc → link imediato para o advogado.

⚡ Sobre o Motor: Claude Opus vs Claude Code Agent

  • Skill atual usa execSync('claude --agent olivia-juridico') — frágil, timeout 300s, não funciona sempre
  • v2.0 usa SDK Anthropic direto com Claude Opus — estável, streaming nativo, sem processo filho
  • Claude Opus = melhor modelo disponível — qualidade máxima, vale o custo
  • Prompt caching + re-uso de contexto → custo estimado: ~$0.15-0.25 por petição

Análise de Provas v2 — Vision Nativo

Como o agente vai analisar documentos jurídicos com foco em restrições de crédito

Triple Fallback Vision
1

Claude Vision Nativo

PDF → converter para imagens → enviar inline no prompt. Melhor para documentos textuais nativos. Limite: 20MB/imagem — paginar PDFs grandes.

2

Gemini 2.0 Flash Vision

Fallback para PDFs >20MB ou quando Claude falhar. Gemini suporta PDFs diretamente sem converter para imagem. Mais rápido para docs grandes.

3

Mistral OCR Latest

Fallback final para PDFs escaneados (imagens). Melhor OCR do mercado para documentos em português. Extrai texto que outros não conseguem.

Foco Jurídico: Inversão do ÔnusCrítico
  • Em relações de consumo: empresa deve provar a origem do débito
  • O agente analisa se os documentos da empresa comprovam a restrição
  • Dados injetados na análise: restricao_orgao_nome, restricao_valor, restricao_codigo_contrato, restricao_data
  • Pergunta central: "esse documento prova que o débito de R$ X junto à empresa Y com contrato Z é legítimo?"
  • Retorna: comprova_debito (boolean) + pontos_fracos_reu[] + observacoes
  • Resultado alimenta diretamente o buildPromptMestre() — argumentos específicos das provas

Riscos & Mitigações

O que pode dar errado e como mitigar

🔴

PDFs muito grandes (>20MB) com Claude Vision

Risco: Claude Vision tem limite de 20MB por imagem — PDFs jurídicos grandes falharão.
Mitigação: Paginação automática (max 10 páginas por chamada) + fallback automático para Gemini se >20MB.

🟡

Latência do Opus (modelo maior = mais lento)

Risco: Claude Opus é ~3-4x mais lento que Sonnet — pode levar 2-3min por petição.
Mitigação: SSE streaming mascara a latência (usuário vê o texto sendo gerado). Prompt caching reduz tokens re-processados. Score alto na primeira tentativa evita re-geração.

🟡

SSE vs CORS em produção

Risco: EventSource não suporta headers custom (sem Authorization). Produção pode bloquear.
Mitigação: Auth por query param (token na URL) ou cookie. Fallback automático para polling se SSE falhar na conexão.

🟡

Compatibilidade da skill com versão nova

Risco: Skill legada tem 1825 linhas com dependências específicas — portar pode introduzir bugs.
Mitigação: Criar v2 em arquivo novo, manter legado funcionando. Migrar gradualmente após validação E2E completa.

🟢

O que mais pode ser implementado para tornar o Modo Agente o mais avançado do mercado

🧪 Modo ComparaçãoFuturo
  • Gerar 2 versões com seeds diferentes lado a lado
  • Advogado escolhe a melhor (ou faz merge manual)
  • Versão escolhida vai para RAG como exemplo aprovado
  • Melhora o banco de exemplos automaticamente
📈 Score de Viabilidade Pré-GeraçãoCurto prazo
  • Antes de gerar: analisar processo e estimar chance de sucesso
  • Baseado em: tipo de réu, dados da restrição, histórico similar
  • Alerta se processo tem pontos fracos significativos
  • Advogado decide se continua com plena consciência
🗂️ Batch GenerationCurto prazo
  • Selecionar múltiplos processos similares
  • Gerar petições em batch (fila JANA/JARBAS)
  • Útil para campanha de réplicas no mesmo devedor
  • Notificação quando todas concluídas

🧠 Ideia Criativa: "Modo Contraditório"

  • Antes de gerar a petição final: o agente simula ser o advogado da ré e identifica os pontos mais fracos da petição
  • Essa "análise adversarial" é injetada no prompt do revisor — que passa a fortalecer exatamente esses pontos
  • Resultado: petição blindada contra os argumentos mais comuns da contestação antes mesmo de ela existir
  • Inspirado na técnica de red-teaming em segurança — aplicada ao direito
