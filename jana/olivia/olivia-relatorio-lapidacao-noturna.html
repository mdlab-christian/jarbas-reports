🔍 Análise Detalhada — Bug B3 (Download 404)

Diagnóstico confirmado via API: bucket olivia-peticoes tem "public": false. As URLs geradas pelo servidor seguem o padrão /object/public/olivia-peticoes/... que só funciona em buckets públicos.

Solução Recomendada: Signed URL via Edge Function

✅ Opção A — Tornar bucket público

Simples, rápido. Petições ficam acessíveis por URL direta. Risco: qualquer um com a URL acessa — mas URLs contêm UUID de historico_id que é difícil de adivinhar.

-- Via Supabase Dashboard:
UPDATE storage.buckets
SET public = true
WHERE name = 'olivia-peticoes';

⚠️ Opção B — Signed URLs (seguro)

Frontend chama supabase.storage.from('olivia-peticoes').createSignedUrl(path, 3600) antes de redirecionar para download. Mais seguro mas requer lógica adicional no Step5.

const { data } = await supabase.storage
  .from('olivia-peticoes')
  .createSignedUrl(path, 3600); // 1h
window.open(data.signedUrl);
Recomendação: Opção A (tornar público) é suficiente para o contexto — UUIDs de historico_id são unguessable e petições são documentos do próprio escritório. Resolve em 1 linha.

## Bug B3 — Download DOCX/PDF retorna 404
Arquivo: src/pages/olivia/components/express/ExpressStep5Conclusao.tsx (ou onde os botões Download ficam)
Sintoma: clicar em "Download .docx" ou "Download .pdf" retorna 404. 
Causa: bucket olivia-peticoes é privado, URLs /object/public/ não funcionam.
Solução: nos botões de download, antes de redirecionar, gerar uma signed URL temporária:

  window.open(data.signedUrl, '_blank');
O storagePath é a parte da URL após '/olivia-peticoes/' (ex: 'abc-123/peticao.docx').
Implemente a função de download com signed URL em todos os botões .docx e .pdf do Step5.
