🎯 O problema raiz do Smart Blocks
🚨

Causa raiz confirmada: bloco_grupo_id NULL em 100% dos blocos

O Smart Blocks funciona em cima de uma hierarquia: Grupo de Bloco → Versões do Bloco → Tags → Conteúdo. No banco novo, todos os 1.244 blocos têm bloco_grupo_id = NULL. Isso significa que o sistema de versões do Smart Blocks está completamente cego — ele não consegue nem achar os blocos, quanto mais selecionar a versão correta para uma petição.

Os 329 modelos de petição têm modelo_grupo_id preenchido (169 grupos existem), então modelos funcionam. Mas os blocos, que são a matéria-prima, estão "órfãos".

⚠️

Google Docs IDs: todos preenchidos, mas precisam validação cruzada

Tanto no banco antigo quanto no novo, google_doc_id está preenchido (0 NULLs). Os arquivos existem no Drive. Mas não sabemos se os IDs do Drive ainda são válidos — o sistema antigo usava Apps Script que referenciava esses IDs direto. Se as pastas do Drive foram reorganizadas, os links estão quebrados.

⚖️ Reimportar tudo vs. Corrigir cirurgicamente

❌ Opção A — Reimportar tudo do zero

  • Risco: apagar 1.244 blocos e 329 modelos que já estão corretos
  • Trabalho: precisa reescrever todo o script de migração
  • Problema: vai encontrar os mesmos conflitos de IDs de antes
  • Tempo: dias de trabalho + risco de regressão
  • Quando faz sentido: se a estrutura do banco antigo for o único source-of-truth confiável

✅ Opção B — Corrigir cirurgicamente (recomendado)

  • Os dados já estão no banco novo — não precisa reimportar
  • Problema 1: criar os grupos de bloco e vincular os 1.244 blocos
  • Problema 2: validar se tag_ids e juris_ids referenciam IDs válidos
  • Problema 3: verificar Google Doc IDs ainda válidos no Drive
  • Resultado: Smart Blocks funciona sem reimportar nada
✅

Recomendação: Opção B — corrigir, não reimportar

Com 70% de confiança digo que reimportar é desnecessário e mais arriscado. Os dados estão lá. O que precisa ser feito é um script de correção que:

1. Cria olivia_modelos_grupo (equivalente) para blocos de texto, agrupando por título base

2. Popula bloco_grupo_id nos 1.244 blocos

3. Valida referências cruzadas de tag_ids e juris_ids

Os 30% restantes de incerteza vêm de não saber se os Google Doc IDs ainda são válidos. Isso precisa ser verificado antes de qualquer decisão final.

⚠️ Quando reimportar SERIA a decisão certa

Se a validação dos Google Doc IDs mostrar que mais de 30% dos documentos estão quebrados no Drive, aí sim reimportar com uma estrutura nova de pastas faz sentido — porque não adianta ter os IDs corretos no banco se os arquivos não existem mais no Drive.

Nesse caso, o fluxo seria: criar estrutura de pastas organizada no Drive → reimportar documentos do sistema antigo → migrar com script que já conhece o mapeamento correto.

📋 Plano recomendado para a conversa de Backend
1

Validar Google Doc IDs (30min)

Pegar amostra de 20 blocos e 20 modelos do banco novo. Verificar se os google_doc_id ainda abrem no Drive. Se >70% válidos → confirma Opção B. Se <30% válidos → aciona Opção A com reimportação.

2

Criar tabela olivia_blocos_grupo + script de agrupamento

O banco novo tem olivia_modelos_grupo para modelos — mas não tem equivalente para blocos. Precisa criar a tabela (ou verificar se bloco_grupo_id deveria referenciar olivia_modelos_grupo) e rodar script que agrupa os 1.244 blocos por título base.

3

Validar referências cruzadas tag_ids e juris_ids

Verificar se os UUIDs nos arrays tag_ids e juris_ids dos blocos existem como IDs válidos nas tabelas olivia_tags e olivia_jurisprudencias. Se os IDs foram preservados na migração, as referências já estão corretas.

4

Mapear olivia_jurisprudencias_assuntos antigo → novo

O banco antigo tem 570 assuntos de jurisprudência com hierarquia (assunto → categoria). Verificar se o banco novo importou essa estrutura completa ou apenas os IDs sem a hierarquia.

5

Alimentar RAG com os blocos de texto

Com bloco_grupo_id preenchido e referências corretas, rodar o pipeline de RAG (needs_rag_sync = true nos blocos) para indexar os 1.244 blocos no vector store. Sem isso, o Smart Blocks não consegue fazer busca semântica.

🦊

Veredito final

Com segurança — baseado nos dados reais dos dois bancos

✅

Não reimporta. Corrige.

Os dados estão nos dois bancos com exatamente os mesmos números. O problema não é que a importação foi errada — é que ela foi incompleta: copiou as linhas mas não criou as relações entre elas no novo schema.

A correção cirúrgica (passo 2 do plano) resolve o Smart Blocks sem tocar nos modelos de petição que já funcionam, sem risco de regressão e em horas, não dias.

⚠️

Exceção: se os Google Doc IDs estiverem quebrados

Aí o plano muda para reimportação planejada com nova estrutura de pastas no Drive. Mas isso só se confirma com a validação do passo 1. Não descarta essa hipótese sem checar primeiro.
