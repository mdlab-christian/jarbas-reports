1️⃣ Step 1 — Identificação do Processo

O que o usuário faz

  1. Busca e seleciona o processo (SearchBoxProcesso — busca por CNJ ou nome do cliente)
  2. Vê o painel de informações: partes, vara, objeto, documentos já no sistema
  3. Seleciona o Tipo de Petição (Réplica, Apelação, Contestação…)
  4. Seleciona o Modelo filtrado por categoria da empresa do processo
  5. Seleciona a Versão do modelo (ex: V3 · Pré-configurado)
  6. Clica em "Avançar" → dispara a Fase 1 em background

O que o sistema faz ao avançar

  • Cria registro em olivia_historico com status PROCESSANDO
  • Chama POST /olivia/gerar com Fase 1 (sem perguntas_respondidas)
  • Pipeline roda em background: S1 → S2 → S3
  • Muda status para AGUARDA_RESPOSTAS ao final da S3
  • Seta meta.relatorio como AnaliseSection[] com resultado da análise

Componentes principais

  • GeradorStep1.tsx — container do step
  • SearchBoxProcesso.tsx — busca de processos
  • SeletorTipoPeticao.tsx — filtrado por categoria da empresa
  • SeletorModelo.tsx — filtra por status='published' e categoria
  • SeletorVersao.tsx — versões do modelo (V1, V2, V3…)
  • useOliviaWizard.ts → função iniciarAnalise(dados)

Payload enviado ao servidor (Fase 1)

{
  historico_id,
  organizacao_id,
  processo_id,
  tipo_peticao_id,
  modelo_id,          // = olivia_modelos_grupo.id (o grupo, não a versão)
  modelo_grupo_id,    // mesmo que modelo_id
  versao_id,          // olivia_modelos_peticao.id (active_version) — DEVE ser enviado
  qualidade_ia,       // 'padrao' | 'avancado'
  analisar_eproc,     // boolean (default true — futuro)
}

⚠️ Bug ativo: versao_id não é enviado pelo frontend

O SmartBlocks precisa do versao_id para buscar o template correto. O server.mjs tem um auto-resolve (busca active_version_id pelo modelo_grupo_id), mas o ideal é o frontend enviar explicitamente.

2️⃣ Step 2 — Triagem / Perguntas

O que o usuário vê

  1. Skeleton loading enquanto a Fase 1 processa (status PROCESSANDO)
  2. Quando status vira AGUARDA_RESPOSTAS → perguntas aparecem via Realtime
  3. Card "Análise do caso" com accordions colapsáveis (AnaliseSection[]):
    • Documentos Analisados
    • Pontos Fortes
    • Alertas e Fragilidades
    • Pilares de Prova
    • Tom Recomendado
  4. Badge de chance de ganho (Alta/Média/Baixa) no topo
  5. Perguntas personalizadas baseadas nas tags do modelo
  6. Textarea de instruções adicionais
  7. Botão "Gerar petição" → avança para Step 3 e dispara Fase 2

Como as perguntas devem ser geradas (olivia-gera-perguntas.mjs)

  • Busca as tags do modelo_grupo_id via active_version_id
  • Filtra tags já respondidas pela analise-provas (tags_respostas)
  • Para cada tag pendente: gera UMA pergunta específica (não genérica)
  • Usa nome do réu, tipo de contrato e fatos dos documentos nas perguntas
  • Tags obrigatórias (obrigatoria=true) sempre geram pergunta
  • Máximo 6 perguntas — não 14
  • Não perguntar sobre dados já no processo (nome, número CNJ, objeto)

Como o relatório deve aparecer

meta.relatorio é um array de AnaliseSection: { titulo, conteudo, tipo }. Deve ter ≥3 seções após a geração. O frontend renderiza como Collapsibles dentro do card "Análise do caso". Atualmente não aparece por bug no mapeamento do campo tom (objeto, não string).

Realtime

Dois canais: triagem-rt-{historicoId} (ExpressStep2Triagem) e olivia-wizard-{id} (useOliviaWizard). O wizard ouve SUCESSO/CONCLUIDO/ERRO. A triagem ouve AGUARDA_RESPOSTAS para exibir as perguntas.

❌ Bugs ativos no Step 2

  • Relatório não aparece: campo tom é objeto, server trata como string → [object Object]
  • Perguntas genéricas: SYSTEM_PROMPT não instrui usar tags como guia
  • 14 perguntas quando deveria ser máximo 6

3️⃣ Step 3 — Estrutura (Outline)

O que o usuário vê

  1. Lista ordenável (drag-and-drop via @dnd-kit) das seções da petição
  2. Agrupadas em 3 seções: Introdução → Fundamentação → Conclusão
  3. Items obrigatórios mostram badge "Obrigatório" e não podem ser removidos
  4. Items opcionais têm toggle on/off
  5. Contador de seções por grupo
  6. Usuário pode reordenar dentro do grupo e ativar/desativar itens opcionais
  7. Clica em "Gerar petição" → avança para Step 4 e a Fase 2 continua em background

Como o outline deve ser montado

  • Fonte de dados: perguntas_geradas.estrutura retornada pelo S3 (gera-perguntas)
  • Ordem canônica por secao_id: f9105bb6 (intro) → 06f1e1cc (fundamentação) → f1426f75 (conclusão)
  • Dentro de cada grupo: ordem definida por olivia_modelo_tags.ordem ASC
  • Função sortByGroup() deve aplicar essa ordenação ao receber initialOutline

IDs fixos das seções

intro:        f9105bb6-7c59-4ca6-9a85-56dfac261605
fundamentacao: 06f1e1cc-e13c-4253-a852-7c555b40d9ba
conclusao:    f1426f75-5aaa-46cf-93c0-17058d40f6aa

❌ Bug ativo: tags fora de ordem

As tags chegam sem respeitar olivia_modelo_tags.ordem. A função sortByGroup() agrupa por seção mas não ordena por ordem dentro de cada grupo.

4️⃣ Step 4 — Geração da Petição

O que o usuário vê

  1. Barra de progresso com 5 substeps animados (Framer Motion)
  2. Mensagem de status dinâmica (progresso_msg do banco via polling a cada 8s)
  3. Formulário de revisão final (aparece quando generationStatus === 'AGUARDA_REVISAO'):
    • Campo "Escritório da contraparte" (autocomplete)
    • Seletor "Chances de ganho" (Alta/Média/Baixa)
    • Switch "Pular revisor" (agilidade)
    • Textarea de instruções finais
  4. Botão "Aprovar e gerar" → endpoint olivia/criar-doc

Substeps e ranges de progresso corretos

SubstepRange pctDuração estimada

Polling fallback (implementado hoje)

A cada 8s, o frontend faz query direta no Supabase para progresso_pct e progresso_msg. Garante que a tela não trava mesmo se o Realtime dropar durante a chamada Claude (~90s).

❌ Bug ativo: Step 4 trava em 65%

O server.mjs não muda o status de AGUARDA_RESPOSTAS para PROCESSANDO ao iniciar a Fase 2. O wizard fica aguardando CONCLUIDO mas o status não sinaliza que a geração está rodando. Fix: setar status=PROCESSANDO + progresso_pct=50 no início da Fase 2.

5️⃣ Step 5 — Conclusão / Download

O que o usuário vê

  1. Confetti animado de sucesso
  2. Badge com score do revisor (ex: "9.0 / 10 ✅")
  3. Botão de download do DOCX (signed URL do Supabase)
  4. Botão de download do PDF
  5. Seção de feedback (nota 1-10 + comentário)
  6. Botão "Nova petição" (reset do wizard)

O que o sistema deve garantir

  • DOCX ≥ 10KB (petição real, não template vazio)
  • PDF ≥ 50KB
  • URLs de download via docx_url e pdf_url do histórico
  • Score do revisor visível mas sem detalhes de tokens/custo

🔧 Skills do Pipeline — Como Cada Uma Deve Funcionar

S1 — olivia-analisa-provas.mjs (v3)

Quando: Fase 1, primeira skill chamada. Roda sempre.

Input: { historico_id, organizacao_id, processo_id, documentos_reu_urls, qualidade_ia, dados_processo }

O que faz:

  • Busca dados do processo no banco (partes, vara, objeto, categoria do réu)
  • Busca documentos com texto_extraido IS NOT NULL da tabela processos_documentos
  • Roda 22 perguntas do playbook jurídico sobre os documentos
  • Classifica cada documento por pilar (relação jurídica / contratação / origem do débito)
  • Define tom: incisivo (réu fraco) / neutro / defensivo (réu forte)
  • Calcula chance de ganho (0–100%)
  • Detecta alertas: litigância má-fé, multa/ofício OAB, 3 pilares comprovados

Output: { analise_por_doc, pontos_fortes, alertas, chance_ganho, tags_respostas, tom: {tom, descricao}, estrutura_sugerida, municao_por_pilar }

Atenção: tom é um OBJETO, não string. O server.mjs deve tratar: typeof tom === 'string' ? tom : tom?.tom

Limite atual: texto_extraido.slice(0, 2000) por documento (fix aplicado hoje — era 300)

S2 — olivia-busca-juris.mjs

Quando: Fase 1, após analisa-provas.

Input: { historico_id, organizacao_id, modelo_grupo_id, analise_provas, processo_id }

O que faz:

  • Busca jurisprudências vinculadas ao modelo via olivia_modelo_juris
  • Filtra por status='published' (não 'ativo')
  • Fallback: vault Obsidian para jurisprudências adicionais
  • Retorna até 10 jurisprudências ranqueadas por relevância

Output: { jurisprudencias: [...] }

S3 — olivia-gera-perguntas.mjs

Quando: Fase 1, última skill. Define as perguntas que aparecem no Step 2.

Input: { historico_id, organizacao_id, modelo_grupo_id, analise_provas, dados_processo, qualidade_ia }

O que deve fazer (comportamento correto):

  • Buscar as tags do modelo via active_version_id do grupo
  • Filtrar tags já respondidas pela analise-provas (tags_respostas)
  • Para cada tag pendente: gerar UMA pergunta específica usando nome do réu e dados concretos
  • Tags obrigatórias (obrigatoria=true) sempre geram pergunta
  • Máximo 6 perguntas
  • Não perguntar sobre dados já no processo
  • Salvar em olivia_historico.perguntas_geradas
  • Setar status AGUARDA_RESPOSTAS

Output salvo no banco: { perguntas_usuario: [...], respostas_automaticas: [...] }

S4 — olivia-smartblocks.mjs (v4)

Quando: Fase 2, primeira skill. Gera o template base da petição.

Input: { historico_id, organizacao_id, modelo_id, versao_id, modelo_grupo_id, processo_id, perguntas_respondidas, jurisprudencias }

O que faz:

  • Busca as tags do modelo via active_version_id
  • Auto-resolve versao_id pelo modelo_grupo_id se não vier
  • Busca blocos de texto para cada tag
  • Substitui placeholders dinâmicos com dados reais do processo
  • Monta HTML rico com blocos e jurisprudências
  • Complementa com vault Obsidian (playbooks, teses)

Output: { peticao_base_texto, peticao_html_base, blocos_usados, juris_usadas, instrucoes_ia }

Regra crítica: Se tags.length === 0, retorna mock mínimo com _fallback: true. Verificar sempre que o log mostra "N tags para [grupo]" e não "sem tags".

S5 — olivia-gera-peticao.mjs (v2)

Quando: Fase 2, após SmartBlocks. Chama Claude API (~60–120s).

Input: { peticao_base_texto, peticao_html_base, analise_provas, instrucoes_ia, dados_processo, respostas_advogado, qualidade_ia }

Comportamento correto (MODO REFINAMENTO):

  • Recebe template SmartBlocks de ~44.000 chars
  • Claude REFINA — não reescreve do zero
  • Substitui variáveis {{NOME}} pelos dados reais
  • Adapta introdução com fatos concretos dos documentos
  • Output deve ter ≥ 80% do tamanho do input
  • Nunca mencionar CONTR7, CONTR8, CONTR9 no texto
  • Data sempre preenchida automaticamente (nunca placeholder)
  • Seção === FATOS EXTRAÍDOS DOS DOCUMENTOS === deve aparecer no prompt

Output: { peticao_texto, peticao_html_final }

S9 — olivia-formata-peticao.mjs

Quando: Fase 2, após gera-peticao. Determinística (sem LLM).

O que faz: Aplica formatação ABNT ao HTML: marcadores semânticos JURIS → 10pt + recuo 3cm + itálico, remove markdown literal (**, ##), normaliza headings, sanitiza.

Output: { peticao_html_formatada, score_formatacao }

S6 — olivia-revisor.mjs

Quando: Fase 2, após formatadora. Avalia qualidade.

O que faz: Compara petição gerada vs template SmartBlocks. Score 0–10. Se qualidade_ia is not defined, usa heurística local (bug ativo — verificar desestruturação do input).

Output: { score, feedback }

S7 — olivia-cria-doc.mjs

Quando: Fase 2, última skill. Gera os arquivos finais.

O que faz:

  • Gera DOCX via docx-generator.mjs (lib docx npm)
  • Gera PDF via Chromium headless
  • Faz upload para bucket olivia-peticoes no Supabase
  • Atualiza olivia_historico com docx_url, pdf_url, status CONCLUIDO

DOCX esperado: ≥ 10KB | PDF esperado: ≥ 50KB
