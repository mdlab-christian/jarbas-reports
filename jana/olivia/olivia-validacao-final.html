Pontos de Risco — Mapear antes de executar
▼

🔴 RISCO 1 — Qualidade da petição Express vs Agente

O Modo Agente (skill_olivia_agente_v3.mjs no JARBAS) gerou SUCESSO em 2 runs com motor='jarbas'. O Modo Express via N8n gerou 4 sucessos mas nunca foi validado em produção real. A chain skills é nova — primeira execução pode revelar problemas de prompt ou contexto incompleto. Mitigação: Fase 1 = validar com 5 petições reais antes de ir para produção. Se qualidade < 7/10 no revisor → ajustar prompts.

🟡 RISCO 2 — Supabase Realtime + olivia_historico updates frequentes

Realtime em JSONB pode ter latência. Se a skill fizer muitos UPDATES em sequência, o frontend pode receber eventos fora de ordem. Mitigação: usar campo progresso (inteiro 0-100) + progresso_msg (text) separados. Mais simples e confiável que JSONB nested.

🟡 RISCO 3 — GAS quota e latência

GAS tem quota de execução (6min max). Para petições complexas (muitos blocos), pode atingir o limite. Mitigação: olivia-cria-doc deve ter timeout de 90s + retry automático + fallback: salvar texto_gerado em olivia_historico.meta.texto_fallback se GAS falhar. Botão "Copiar texto" no Step 4.

🔵 RISCO 4 — Contexto N8n (para scripts de referência)

Os workflows N8n usam fetch() que não existe no N8n. Fix: require('https') + wrapper httpRequest. Todos os Code nodes dos workflows N8n de referência usam o padrão validado (Test 2 E2E). Skills OpenClaw usam Node.js nativo — sem esse problema.
