📍 O que é OlivIA

Gerador de petições jurídicas do MdFlow, sistema da MdLab LegalTech. Focado em ações de massa do mesmo tipo (Cumprimento de Sentença, Ação Revisional, etc.), o que permite alto nível de automação e qualidade. O escritório Midas é o cliente principal.


O sistema usa modelos de petição como esqueleto (sempre, mesmo nos modos avançados). Tags = perguntas. Blocos de texto = respostas. Múltiplas versões de modelo por tipo de petição.

✅ DECISÕES FINALIZADAS
✅ D1 — Chaves API hard-coded no n8n
Credenciais ficam no n8n por enquanto. Revisar na versão de produção escalável. Sem externalização.
✅ D2 — Fila de Geração dentro do Gerador
Aba "Fila de Geração" separada excluída. Recriada no lugar de "Últimas Petições" dentro da aba Gerador. Mantém: status em tempo real, botão Abrir, filtro por modo.
✅ D3 — Smart Blocks: wizard por tag
Cada tag = 1 step do wizard. 1 bloco de texto por vez (versão selecionada). Sem criação de bloco personalizado. Sem abertura de todos os blocos.
✅ D4 — QualidadeIA: Sonnet 4.6 padrão / Opus 4.6 pensamento estendido
Funcional em todos os modos. Frontend passa qualidade_ia=padrao|pensamento_estendido. n8n roteia para modelo correto.
✅ D5 — Análise IA: recriar 100% no JANA/JARBAS
Frontend e mecanismo novos. Sem dependência de serviço externo. Workflow: 8ntAvTGMhCfrlfRt.
✅ D6 — Chat Olivia: modal de início
"Chat Livre" vs "Informar número de processo". Se informar processo → chat contextualizado com dados do processo.
✅ D7 — Agente: upload docs no Step 1
Estava no Step 2. Mover para Step 1 junto com identificação. JANA precisa dos docs para gerar triagem.
✅ D8 — Step 1 unificado: painel do Smart Blocks como referência
Smart Blocks tem o melhor painel de processo. Usar como referência para todos os modos. Diferente para Empresa vs Órgão. Debounce 300ms, exibir CNJ + nome cliente + nome ré.
✅ D9 — Seletor de Playbook: removido do Step 1
IA seleciona automaticamente o playbook por categoria de empresa/tipo de processo. Usuário não escolhe manualmente.
✅ D10 — HTML Reports: salvar no backend
Não publicar no GitHub. Salvar no Supabase Storage ou tabela dedicada. Link disponível no histórico do processo.
✅ D11 — Vibe.lau: fora desse projeto
Deixar de fora por enquanto. Não incluir na SPEC.
✅ D12 — Sistema de abas tipo Chrome
Múltiplas petições simultâneas, modos e steps diferentes. Usuário navega entre abas enquanto uma gera. Já existe WizardTabBar no Express — padronizar para todos os modos.
✅ D13 — Modelo de petição: sempre usado como esqueleto
Mesmo nos modos mais avançados, o modelo de petição é o esqueleto. Agente usa Smart Blocks como base adicional (petição mecânica gerada primeiro, depois melhorada).
✅ D14 — Tags = perguntas / Blocos = respostas (regra máxima)
Todos os agentes IA devem respeitar esta hierarquia. Agente pode criar/excluir/reposicionar tags. Tags com múltiplas escolhas = blocos disponíveis. Editar tópico = editar tag diretamente.
✅ D15 — Modo Agente usa Smart Blocks como base
Fluxo: gera Smart Blocks primeiro → Agente usa como comparativo/base → melhora com análise de provas e impugnações específicas. Nunca inventa fundamentações.
✅ D16 — Smart Blocks Step 2: botão "Gerar Relatório HTML" (opcional)
Só aparece se usuário ativou toggle no Step 1. Não trava o Smart Blocks se JARBAS offline. JARBAS gera relatório completo do caso.
✅ D17 — Modo Agente Step 1: toggle "Gerar Relatório HTML"
Opcional para economizar custo. Usuário escolhe se quer relatório HTML junto com a petição.
✅ D18 — RAG Controller: plug and play
2 RAGs separados: RAG Olivia (jurídico, já em desenvolvimento) + RAG Controller (jurimetria: taxa de sucesso por juiz, argumentos que funcionam, teses vencedoras). Deixar arquitetura pronta para plugar.
✅ D19 — Modo Petição Livre: modo próprio
Inspirado no Modo Agente. Seletor opcional: usar modelo de petição como base ou não. Frontend próprio. Funcionamento igual ao Agente por baixo.
✅ D20 — Chat Olivia: editor de petição integrado
Chat serve para conduzir usuário E para alterar petição. Usuário pede alterações via chat → chat absorve e propõe mudanças na petição. Drawer/sidebar com editor.
✅ D21 — BI Jurídica no sistema
Quando processo transita em julgado → análise automática do que levou à decisão (argumentos, provas, juiz). Alimenta RAG Controller. Melhora petições futuras com base em taxa de sucesso real.
✅ D22 — Versão V1/V2 implica em modelo + blocos de texto
Seletor de versão afeta tanto o modelo de petição quanto os blocos de texto e jurisprudências disponíveis. Igual ao BackEngine antigo.
⚠️ DECISÕES PENDENTES (recomendações JANA)
⚠️ P-D1 — Memória para agentes Claude
Problema: Claude não tem memória entre calls. Para petições longas ou contexto rico, pode ser limitante.
Recomendação JANA: Criar tabela olivia_agente_contexto com histórico de interações por historico_id. Passa como "memory" no prompt de sistema. Custo baixo, ganho alto.
⚠️ P-D2 — Relatórios HTML: onde salvar exatamente
Problema: Decidido que vai no backend, mas não definido se Supabase Storage (arquivo HTML) ou coluna JSONB/text na tabela.
Recomendação JANA: Storage (Supabase Storage, bucket olivia-reports) com path salvo em olivia_historico.relatorio_html_path. Fácil de servir como link de download.
⚠️ P-D3 — Importação de modelos HTML antigos
Problema: Existem petições antigas em HTML que podem ser importadas como modelos.
Recomendação JANA: Usar CC (terminal dedicado) para investigar estrutura dos HTMLs antigos e criar script de importação. Confirmar com Christian se vale a pena antes de implementar.
⚠️ P-D4 — Fallback JANA offline no Modo Agente
Problema: Se JANA cair, triagem via JARBAS ou bloquear?
Recomendação JANA: Fallback para JARBAS com aviso visual "Motor alternativo ativo". Implementar depois dos P0.
⚠️ P-D5 — Smart Blocks: edição de bloco persiste no Google Doc?
Problema: Usuário edita bloco no Step 2 mas webhook envia só IDs — n8n busca conteúdo original do banco.
Recomendação JANA: Enviar blocos_conteudo_customizado: [{id, conteudo}] no payload para que n8n use versão editada.

🏗 Arquitetura JARBAS — Motor Principal
Como funciona o JARBAS na OlivIA

JARBAS é o motor de geração de petições mais avançado. Para o Modo Agente, ele recebe contexto completo (processo + triagem JANA + rascunho Smart Blocks) e gera a petição final via Claude. A habilidade de trabalhar com ações de massa do mesmo tipo de processo é o que torna o sistema excepcionalmente preciso.
