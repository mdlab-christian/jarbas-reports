1. Contexto Geral

O Modo Express é o fluxo principal de geração de petições da OlivIA no MdFlow. É um wizard de 5 etapas que guia o advogado desde a identificação do processo até a geração da petição final com IA, integrando:

  • Análise automática de provas via ANALISE-V4 (modelos com tags)
  • Triagem inteligente com pré-fill de alta confiança e filtragem de perguntas
  • Revisão da minuta com editor rico + MicrophoneButton
  • Entrega via Google Docs + DOCX (docx@8)
  • Relatório do caso gerado pelo Opus na conclusão

Esta sprint cobriu a integração com ANALISE-V4, smart blocks, lapidações pós-deepreview e commits de produção.

3. Arquitetura e Lógica

3.1 Fluxo Ponta a Ponta

[Step 1: Identificação] → Seleciona processo + tipo petição + modelo_grupo + advogado → modelo_tem_tags=true + sem docs? → bloqueia (análise obrigatória) → modelo_tem_tags=false → avança direto (modelo simples) → cria olivia_historico + dispara ANALISE-V4 → polling olivia_analises + overlay animado com Progress bar [Step 2: Triagem] → pré-preenche respostas com confianca >= 0.7 → perguntasFiltradas: omite perguntas já cobertas → todasAltaConfianca: badge Sparkles + skip direto → layout 2 colunas: preview petição base ← → perguntas restantes → MicrophoneButton em perguntas texto + instrucoes_adicionais [Step 3: Revisão] → Editor rico com seções + MicrophoneButton instrucoes_finais → Autosave 30s [Step 4: Geração] → useGerarPeticao → callWebhook('mdflow/olivia/express', payload) → polling olivia_historico (PENDENTE → EM_PROGRESSO → SUCESSO/ERRO) → ProgressSubsteps visual [Step 5: Conclusão] → Métricas: score, modelo, tokens, custo, tempo → RelatorioCasoAccordion: visão geral + fortes/fracos + riscos + chance + recomendações → Download DOCX (docx@8) + Google Docs

3.2 Análise ANALISE-V4

Disparada ao avançar Step 1 quando modelo_tem_tags=true. Endpoint: mdflow/olivia/analise

Retorna:

  • analise_tags[] — { id, bloco_id, confianca } — mapeamento tag→bloco
  • peticao_base.blocos_resolvidos[] — blocos resolvidos com conteudo_html + confianca

Polling via Supabase olivia_analises (interval 2s, timeout 5min). Frontend exibe overlay com <Progress> bar.

3.3 Triagem Inteligente

Pré-fill: tags com confianca >= 0.7 → resposta automática via useEffect

Filtro: perguntasFiltradas = perguntas.filter(q => tag.confianca < 0.7)

Skip total: se perguntasFiltradas.length === 0 → badge Sparkles "IA analisou tudo" + botão Concluir direto

Layout: quando blocosResolvidos disponível → grid md:2cols com ScrollArea preview

processo_context: 30+ campos enviados ao webhook (id, número, tipo, objeto, polo, fase, restrição, cliente, empresa, financeiro, tribunal, vara, ia_resumo, analise_id, analise_resultado)

3.4 Fix: Race Condition no Login

Problema: React 18 batching fazia ProtectedRoute ver user=null após login e redirecionar de volta ao /login.

Fix: flushSync() em auth_context.tsx — força commit síncrono de setUser + setOrgSuspended antes do navigate().

6. Decisões Arquiteturais

DecisãoMotivo
modelo_tem_tags controla se análise é obrigatóriaModelos simples sem tags não precisam de análise prévia — reduz fricção para o caso mais comum
Pré-fill com confiança ≥ 0.7Threshold conservador — abaixo disso o usuário confirma. Evita erros silenciosos em casos ambíguos
Layout 2 colunas no Step 2 quando há previewUX: advogado vê a petição base sendo construída enquanto responde as perguntas restantes
flushSync no auth_contextReact 18 batching causava race condition — flushSync força commit síncrono antes do navigate()
processo_context rico (30+ campos)IA precisa de máximo contexto para gerar petições precisas sem depender de Q&A adicional
instrucoes_adicionais + instrucoes_finais unificados no payloadAdvogado complementa instrução em dois momentos (Step 2 e Step 3) — backend recebe tudo concatenado
RelatorioCasoAccordion colapsávelNão sobrecarrega tela de conclusão — relatório disponível sem intrometer no fluxo principal
Bulk upload 50 arquivos / 100MBProcessos INSS/bancário têm muitos documentos — limite anterior de 5/10MB era impeditivo

8. Pontos de Atenção / Riscos

🔴
relatorio_processo não populado no n8n

A UI está preparada para renderizar o RelatorioCasoAccordion, mas depende do prod-olivia-express popular meta.relatorio_processo no JSONB do olivia_historico. Se o campo não existir → accordion não aparece (defensivo), mas o valor do relatório é perdido. Verificar no workflow.

🟡
Timeout 5min na análise

Usuários com conexão lenta ou ANALISE-V4 sobrecarregado ficam presos no overlay. Considerar botão "pular análise / tentar depois".

🟡
loadingAnaliseMsg.includes() para calcular Progress

Frágil — se as mensagens mudarem no hook, a porcentagem fica errada. Substituir por enum de estados ou valor numérico retornado diretamente pelo webhook.
