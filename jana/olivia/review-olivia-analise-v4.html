Pontos Fortes
🏗️
Arquitetura bem segmentada em fases A → F
Cada fase tem responsabilidade clara. Facil de debugar, modificar e escalar. 48 nodes bem distribuidos.
🔄
Fallback de OCR em camadas (Gemini → Mistral)
Documentos sem texto extraivel passam por Gemini Vision e depois Mistral OCR. Robusto para PDFs escaneados.
📊
Progress tracking granular (20% → 50% → 60% → 95% → 100%)
Frontend consegue mostrar progresso real da analise. Boa UX para o advogado acompanhar.
🛡️
Error Trigger central + ErrorHandler
Qualquer erro no pipeline cai no ErrorHandler que loga e atualiza o status da analise. Nao deixa registros em limbo.
🔍
RAG de jurisprudencia com embedding
D3 gera embedding dos docs e D4 busca jurisprudencias relevantes no banco. Prompt instrui Claude a nao citar fora do RAG — correto.
✅
202 imediato + processamento async
Responde 202 ao webhook logo apos inserir a analise. Processamento continua em background. Correto para analises longas.
Pontos de Melhoria
⚠️
P2 Tokens sempre chegando 0 no monitoramento
E3 usa fullResponse: false — o envelope com usage e descartado antes de chegar no E4. Fix: ativar fullResponse: true no E3 e ajustar o E4 para ler $input.item.json.body.usage. Impacto: custo_usd sempre 0 no banco, sem rastreabilidade de gasto por analise.
⚠️
P2 F1 e F3 salvam campos sobrepostos
F1 ja salva resultado, status e meta com progresso 95%. F3 faz PATCH novamente com os mesmos campos (progresso 100%). Duas escritas no banco para o mesmo registro em sequencia imediata. Poderia ser consolidado em um unico PATCH final. Nao e bug, mas e custo desnecessario.
⚠️
P2 Nodes criticos sem continueOnFail
Nodes como E3-Claude-Opus-HTTP, F1-PATCH-olivia_analises, F2-UPDATE-processos nao tem continueOnFail. Se E3 der timeout (raro mas possivel), o ErrorHandler captura — mas o registro fica sem resultado salvo. Melhor ter try/catch explicito no E4 cobrindo tambem falha do E3.
💡
P3 executionOrder ainda em "v1"
Settings usa executionOrder: "v1" (legado). Migrar para "v1" e o padrao atual — mas vale confirmar se o n8n desta versao suporta "v2" para melhor paralelismo em branches.
💡
P3 Dead ends sem saida explicita
F4-LogWebhook-UPDATE e ErrorHandler sao terminadores do fluxo mas sem node de encerramento formal. Nao e problema tecnico — o n8n aceita — mas dificulta leitura do grafo.
💡
P3 Chave Supabase hardcoded no F4
O node F4-LogWebhook-UPDATE usa a SUPA_KEY hardcoded no jsCode (ao inves de credencial do n8n). Ja existe na maioria dos nodes HTTP via header parametrizado — padronizar o F4 para usar a mesma credencial injetada.
