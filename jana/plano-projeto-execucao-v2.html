Divisão de Papéis: JANA vs Claude Code — Definição Oficial

Decisão Arquitetural: CC é o executor principal. JANA é a planejadora, orquestradora e validadora. Esta divisão deve ser gravada permanentemente na skill projeto-executar.

🦊 JANA — Planejamento & Orquestração

  • Lê spec e cria etapas no banco
  • Gera prompts CC completos e estruturados
  • Valida resultados (vision, e2e, n8n-review)
  • Tira screenshots via browser tool
  • Atualiza banco com status + evidências
  • Identifica próxima tarefa no loop
  • Cria tarefas de fix quando CC erra
  • Notifica Christian com relatório final
  • Mantém PRD e spec atualizados
  • NÃO executa código diretamente (evita travar/interromper)

⚡ Claude Code — Execução

  • Executa tarefas seguindo prompt JANA
  • Não improvisa além do prompt
  • Gera relatório obrigatório ao final
  • Tira screenshots de componentes criados
  • Salva relatório em ~/jarbas-reports/jana/
  • Testa funcionalidade após implementar
  • Documenta o que não pôde fazer e por quê
  • Vantagem: não quebra no meio, mais confiável para execução longa

PRD Geral do Projeto — Por que é crítico

Insight: Sem um PRD geral do projeto, o CC não tem contexto suficiente para decisões arquiteturais. Hoje só há PRD por etapa — o CC não "vê" o projeto inteiro.

📄 O que o PRD Geral deve conter

  • Objetivo do projeto (o que estamos construindo e por quê)
  • Stack técnica completa (React, Supabase, n8n, etc.)
  • Arquitetura de dados (principais tabelas e relações)
  • Principais componentes e suas responsabilidades
  • Fluxos críticos de usuário
  • Restrições e regras de negócio
  • O que NÃO deve ser alterado
  • Dependências entre camadas

⚡ Como alimentar o PRD Geral

  • CC investiga e escreve o PRD inicial (prompt abaixo)
  • JANA valida e complementa
  • Salvo em projetos.prd_markdown
  • Cada prompt de etapa inclui um resumo do PRD geral
  • CC atualiza o PRD quando descobre algo novo
  • PRD vira a "bíblia" que evita retrabalho entre etapas

Browser Testing + Screenshots Automáticos

Ideia brilhante: O CC tira print de tudo que constrói. JANA valida visualmente. Histórico permanente no banco. Daqui a 6 meses você sabe exatamente o que foi feito e quando.

🔄 Fluxo por etapa frontend

  • CC implementa componente/modal/página
  • CC abre browser na rota
  • CC tira screenshot de cada estado (tab 1, tab 2, modal aberto, modal fechado, estado de erro, estado de sucesso)
  • Screenshots salvos em ~/jarbas-reports/jana/screenshots/olivia/[etapa-id]/
  • JANA recebe relatório + paths dos screenshots
  • JANA analisa com image() tool
  • JANA registra na tabela projeto_screenshots
  • JANA atualiza etapa.evidencia_url

📁 Estrutura de Pastas

  • ~/jarbas-reports/jana/screenshots/ — raiz
  • └── olivia/ — por projeto (slug)
  •     └── etapa-[id]/ — por etapa

  •         ├── depois.png
  •         ├── tab-1.png
  •         ├── tab-2.png
  •         └── validacao.png
⚠️ Limitação: CC (Claude Code no terminal) não tem browser tool nativo — ele pode usar Playwright (CLI) para tirar screenshots. JANA usa browser tool para validação visual. O ideal é: CC tira screenshot via Playwright, salva localmente, JANA lê o arquivo e analisa com image().

Sincronização Bilateral JANA ↔ Claude Code

Viabilidade: Sim, é viável — e é um diferencial enorme. Skills são arquivos Markdown. Se JANA pode ler/escrever arquivos, pode manter CC sincronizado.

✅ O que sincronizar

  • Skills (SKILL.md) — criou na JANA → replica ao CC e vice-versa
  • AGENTS.md, SOUL.md, TOOLS.md — contexto de identidade
  • NÃO sincronizar: MEMORY.md (contexto pessoal), credenciais

⚙️ Como implementar

  • CC lê skills de ~/.claude/commands/ ou CLAUDE.md
  • JANA escreve atualização de skill
  • Script sync-skills.sh copia para pasta do CC
  • CC lê na próxima sessão automaticamente
  • Bidirecional: CC também pode gerar skill nova → JANA incorpora

Fluxo de Sync

JANA cria/atualiza skill X

Arquivo salvo em ~/.openclaw/workspace-jana/skills/X/SKILL.md

Script de sync detecta mudança

Próximo prompt CC inclui referência

CC executa ciente da skill

Não precisa ser "treinado" separadamente — o prompt carrega o contexto

Pontos de Quebra e Riscos

🔴
CC executa algo fora do escopo da spec
CC é poderoso demais — pode alterar banco, arquivos não relacionados, introduzir bugs em áreas não tocadas.
✅ Solução: prompt inclui lista explícita de arquivos permitidos + "NÃO altere nada fora deste escopo". JANA valida diffs após execução.
🟡
Loop automático vai longe demais sem Christian saber
Modo automático pode executar 5+ etapas sem supervisão e causar regressões em cadeia.
✅ Solução: limite de 3 etapas por loop automático. JANA manda update após cada etapa. Christian pode pausar a qualquer momento com "para o loop".
🟡
Screenshot não reflete estado real (cache, loading)
CC tira screenshot mas componente ainda está carregando, ou há dados de produção misturados com teste.
✅ Solução: Playwright com wait-for-network-idle antes do screenshot. Sempre usar conta de teste (mdlab.equipe@gmail.com) nos E2E.
🟢
Overhead de planejamento maior que ganho de execução
Para bugs pequenos (1 arquivo, 30min de trabalho), o overhead de criar etapa no banco + gerar prompt + validar pode ser maior que simplesmente pedir pra JANA resolver.
✅ Solução: manter dois fluxos. Bugs pequenos → JANA direta. Projetos >3 etapas ou >1h → Sistema de Projetos com CC.
🟢
PRD desatualizado conforme projeto evolui
CC implementa algo que muda a arquitetura, mas o PRD não é atualizado → próximas etapas usam PRD errado.
✅ Solução: cada prompt CC inclui instrução: "se você descobriu algo que muda a arquitetura, documente no relatório em seção ATUALIZAÇÃO_PRD". JANA incorpora automaticamente.
