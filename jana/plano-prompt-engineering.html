📦 Nova Estrutura de Projeto

🗂️ Anatomia de um projeto com prompt engineering

Cada projeto passa a ter a seguinte estrutura:

# Projeto: [Nome] PRD: objetivo: "..." contexto: "stack, dependências, arquitetura" requisitos: [...] restrições: [...] etapas: - ordem: 1 nome: "Banco de dados / Supabase" descricao: "..." prompt: | ## Contexto PRD: {prj.prd} ## Tarefa Implementar migrations para... ## Regras - Sempre usar RLS... ## Output esperado - Migrations testadas...

📋 O que o prompt de cada etapa deve conter

  • Contexto PRD: referência ao objetivo geral e requisitos
  • Contexto técnico: stack, arquivos relevantes, dependências
  • Tarefa específica: o que fazer nessa etapa, sem ambiguidade
  • Regras e constraints: o que não fazer, padrões a seguir
  • Exemplos: few-shot quando aplicável
  • Output esperado: o que define "etapa concluída"
  • Critério de revisão: como verificar antes de avançar
🔢 Ordenação de Etapas — Lógica Proposta

⚙️ Sequência recomendada para projetos MdFlow

A ordem deve garantir que cada etapa tem o que precisa pronto:

# Etapa Depende de Executor
1 📐 Spec / PRD — spec skill
2 🗄️ Banco de dados (Supabase migrations) PRD CC — Supabase
3 🔧 Backend / Edge Functions / RLS Migrations CC — Backend
4 ⚡ Automações n8n Backend pronto JARBAS — n8n
5 🎨 Frontend React/TypeScript Backend + tipos CC — Frontend
6 🔍 Revisão de integridade Tudo acima CC — Review
7 ✅ Correção e ajuste final Review CC — Fix
⚡ Modo Fila — Claude Code
1

Geração do projeto

skill projeto-executar lê o spec → gera PRD + prompts para todas as etapas em sequência

2

Enfileiramento

Prompts são passados um a um para o Claude Code (--print --permission-mode bypassPermissions), em ordem de dependência

3

Execução sequencial

CC executa cada etapa com o prompt estruturado como instrução. Ao terminar, aguarda confirmação ou avança automaticamente.

4

Revisão automática

Etapa de review inclui prompt de verificação — se detectar problema, CC corrige antes de encerrar
