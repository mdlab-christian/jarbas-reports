## Bug B01 — deleted_at ausente na query de versões

### Problema
Em usePetitionModels.ts existe uma query que busca versões de modelos de petição.
Essa query NÃO filtra por deleted_at IS NULL.
Resultado: modelos soft-deleted com ativo=true ainda aparecem na listagem para o usuário.

### Fix
Localizar a query de versões (tabela: modelos_peticao_versoes ou similar).
Adicionar .is('deleted_at', null) como filtro adicional.
Exemplo do padrão correto de query Supabase no projeto:
  .from('tabela')
  .select('...')
  .eq('organizacao_id', orgId)
  .is('deleted_at', null)
  .order('created_at', { ascending: false })

### Critério de aceite
- Query de versões NUNCA retorna registros onde deleted_at IS NOT NULL
- TypeScript sem erros após o fix

## Bug B03 — useProcessoDetalhe sem filtro organizacao_id

### Problema
A função/hook useProcessoDetalhe em useProcessSearch.ts faz query no banco
mas NÃO inclui filtro explícito de organizacao_id.
Atualmente depende 100% do RLS do Supabase para isolamento multi-tenant.
Isso é frágil: se RLS falhar ou for bypass'd, dados de outros escritórios vazam.

### Fix

// Antes (frágil):

// Depois (seguro):

### Critério de aceite
- Query inclui filtro organizacao_id além do RLS
- TypeScript sem erros

## Bug B04 — useEffect enrichment com deps incompletas

### Critério de aceite

- TypeScript sem erros, comportamento preservado

## Bug B05 — ProcessoSelector onChange sem memoização — risco de loop infinito

### Problema
Em ProcessoSelector.tsx existe um useEffect que chama onChange dentro do seu body.
Se o parent que passa onChange não memoizar a função com useCallback,
cada render do parent recria onChange → dispara o effect → pode causar loop infinito.

### Fix — opção 1 (preferida, local)
Adicionar onChange ao array de deps do useEffect MAS proteger com um ref para
evitar execuções desnecessárias:
  const onChangRef = useRef(onChange)
  useEffect(() => { onChangeRef.current = onChange }, [onChange])
  // e dentro do effect usar: onChangeRef.current(...)

### Fix — opção 2 (se opção 1 não couber no padrão do projeto)
Mover a chamada de onChange para fora do useEffect — para um handler de evento
ou para um callback direto, sem effect intermediário.

### Documentar no relatório
- Qual opção foi escolhida e por quê
- Se necessário adicionar useCallback no parent, indicar QUAL arquivo/componente
  precisa dessa mudança (não alterar, apenas documentar)

### Critério de aceite
- Sem risco de loop infinito por onChange recreado no parent
- TypeScript sem erros
