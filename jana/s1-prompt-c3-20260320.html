## Item 1 — B05: Parents do ProcessoSelector sem useCallback

### Contexto do fix anterior (C2/B05)
Na correção B05 foi adicionado um onChangeRef estável dentro do ProcessoSelector.tsx
para evitar loops causados por onChange recriado no parent a cada render.
A abordagem com ref é defensiva e funciona, mas não resolve a causa raiz:
os parents que passam onChange ao ProcessoSelector podem estar recriando a função
sem useCallback, gerando renders desnecessários mesmo sem loop.

### Tarefa
1. Buscar todos os componentes que importam e usam ProcessoSelector no projeto:
   grep -r 'ProcessoSelector' src/ --include='*.tsx' -l
2. Para cada caller encontrado, verificar se o onChange passado usa useCallback
3. Se NÃO usa useCallback → adicionar useCallback com deps corretas
   Exemplo do fix:
   // Antes (recria a função a cada render):
    setState(prev => ({ ...prev, processo: p }))} />

   // Depois (estável entre renders):
   const handleProcessoChange = useCallback((p: ProcessoOption | null) => {
     setState(prev => ({ ...prev, processo: p }))
   }, []) // deps: apenas variáveis do escopo que mudam

4. Não alterar a lógica interna — apenas memoizar as funções

### Critério de aceite
- Todos os callers do ProcessoSelector passam onChange via useCallback
- Sem re-renders desnecessários ao selecionar processo
- TypeScript sem erros

## Item 2 — D-FE-11: versaoNumero assíncrono pode chegar undefined no Step 2

### Contexto do fix anterior (D-FE-11)

2. Identificar onde versaoNumero é passado para o Step 2 (via state, props, ou contexto)

### Tarefa — Parte B: fix
Implementar uma das abordagens abaixo (escolher a mais adequada ao padrão do projeto):

    && versaoNumero !== undefined && versaoNumero !== null
