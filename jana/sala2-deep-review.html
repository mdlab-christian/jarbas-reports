🚨 Bugs P0 — Runtime Crash (devem ser corrigidos antes de qualquer teste)

BUG P0-1 · gerarPeticao() ignora blocos e respostas — envio sempre vazio
P0 CRASH

Arquivo: OliviaGeradorSmartBlocks.tsx + useSmartBlocksGeneration.ts

O container chama gerarPeticao(blocosSelecionados, respostasWizard) com 2 parâmetros, mas o hook define a função como async () => sem parâmetros. Os blocos selecionados pelo usuário são silenciosamente ignorados. O webhook recebe blocos_selecionados: [] sempre.

// OliviaGeradorSmartBlocks.tsx (linha 42) — INCORRETO
await gerarPeticao(blocosSelecionados, respostasWizard); // parâmetros ignorados!

// useSmartBlocksGeneration.ts (linha 83) — não aceita parâmetros
const gerarPeticao = useCallback(async () => { // ← zero params
  const { blocosIds, ... } = state; // lê blocosIds do state, que nunca foi atualizado
}

Fix: Alterar hook para aceitar (blocosIds: string[], respostasWizard: Record<string, unknown>) e usar no payload, em vez de ler do state.

BUG P0-2 · SmartBlocksStep3Geracao usa state.status que não existe
P0 LOGIC BREAK

Arquivo: SmartBlocksStep3Geracao.tsx

O Step3 lê state.status para detectar loading/sucesso/erro. Mas SmartBlocksState não tem campo status — tem step: SmartBlocksStep. Os valores que o Step3 checa ('EM_PROGRESSO', 'PENDENTE') nunca existem; os valores reais são 'GERANDO', 'POLLING', 'SUCESSO', 'ERRO'.

// SmartBlocksStep3Geracao.tsx (linha 16) — BUG
const isProcessing = state.status === 'EM_PROGRESSO' || state.status === 'PENDENTE'; // NUNCA true
const isError = state.status === 'ERRO'; // NUNCA true
const isSucesso = state.status === 'SUCESSO'; // NUNCA true

// Fix: usar state.step
const isProcessing = state.step === 'GERANDO' || state.step === 'POLLING' || state.step === 'CRIANDO_HISTORICO';
const isError = state.step === 'ERRO';
const isSucesso = state.step === 'SUCESSO';

Também: effectiveCompleted no container checa state.status === 'SUCESSO' — mesmo bug.

BUG P0-3 · state.blocosSelecionados.length — campo inexistente causa TypeError
P0 CRASH

Arquivo: SmartBlocksStep3Geracao.tsx linha 101

SmartBlocksState tem campo blocosIds: string[], mas Step3 acessa state.blocosSelecionados.length — campo que não existe. Isso causa TypeError: Cannot read properties of undefined (reading 'length') na tela de sucesso.

// BUG — campo não existe
{state.blocosSelecionados.length} bloco{state.blocosSelecionados.length > 1 ? 's' : ''}...

// Fix
{state.blocosIds.length} bloco{state.blocosIds.length > 1 ? 's' : ''}...

DRIFT-2 · olivia_modelo_tags usa modelo_id mas SmartBlocks passa modeloGrupoId
DRIFT

Problema: O hook useSmartBlocks.ts busca tags via .eq('modelo_id', modeloGrupoId!). O schema de olivia_modelo_tags tem coluna modelo_id (UUID), mas em olivia_modelos_peticao há tanto id quanto modelo_grupo_id. Se SmartBlocksModo passar session.modeloId (o id da versão), pode não bater com o modelo_id em olivia_modelo_tags.

O container ativo OliviaGeradorSmartBlocks passa diretamente state.modeloId para SmartBlocksStep2TagWizard — que usa esse valor para query em olivia_blocos_texto com campo errado. Mas a correlação modeloId → tags → blocos precisa ser validada em produção.
