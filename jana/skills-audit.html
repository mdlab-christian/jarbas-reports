O que a Anthropic diz sobre skills
📐 Estrutura obrigatória
Pasta com SKILL.md (exato, case-sensitive). Subpastas opcionais: references/ (docs extras), scripts/ (código), assets/ (templates). Nome da pasta: só kebab-case. Sem README.md dentro da skill — vai para o nível do repo.
🎯 Frontmatter YAML — é o que faz a skill disparar automaticamente
Sem --- frontmatter, Claude não consegue auto-detectar quando usar a skill. O campo description (até 1024 chars) deve ter: o que faz + quando usar + frases-gatilho exatas do usuário. Campos obrigatórios: name (kebab-case) + description.
📦 Progressive Disclosure — 3 níveis
Nível 1: frontmatter YAML — sempre carregado no system prompt (mínimo de tokens).
Nível 2: corpo do SKILL.md — carregado quando skill é relevante.
Nível 3: arquivos em references/ — carregados sob demanda. Isso reduz uso de tokens drasticamente.
✍️ Description eficaz = [o que faz] + [quando usar] + [frases exatas]
Exemplo ruim: "Helps with projects."
Exemplo bom: "Manages Linear project workflows including sprint planning. Use when user mentions 'sprint', 'Linear tasks', or asks to 'create tickets'."
🔬 Critérios de sucesso
Skill deve disparar em ≥90% das queries relevantes. Workflow completa sem o usuário precisar redirecionar. Resultados consistentes entre sessões. Novo usuário consegue usar na primeira tentativa.

Problemas prioritários — o que corrigir primeiro

🔴 P1 — 8 skills sem frontmatter YAML (o problema mais crítico)

Sem o bloco --- no topo do SKILL.md, Claude não consegue detectar automaticamente quando usar a skill. Elas só funcionam quando você cita explicitamente o nome. Isso mata o principal valor das skills.

  • cc-prompt-gen — usada toda hora, não dispara automático
  • cc-paralelo — idem, crítico para tarefas cross-domínio
  • spec — vital para planejamento, nunca dispara sozinha
  • 21st-dev — não detecta quando gerar componentes UI
  • ideias — precisa de trigger explícito o tempo todo
  • distribuicao-master, n8n-monitor, projeto-executar

🔴 P1 — reconstrucao-pagina com description de 15 chars

A description diz apenas "Reconstrução completa" — sem triggers, sem contexto de quando usar. Claude não sabe que deve usar isso quando você diz "lapida turbo /processos" ou "reconstrói /pagina do zero".

🟡 P2 — Nenhuma skill usa references/ para progressive disclosure

O guia da Anthropic recomenda mover documentação longa para references/. Skills como frontend, reconstrucao-pagina e spec têm SKILL.md com centenas de linhas — isso ocupa tokens no system prompt desnecessariamente. O ideal é manter o SKILL.md enxuto e mover o conteúdo técnico para references/design-system.md, references/exemplos.md, etc.

🟡 P2 — Descriptions genéricas sem frases-gatilho

Muitas descriptions descrevem O QUE a skill faz mas não QUANDO usar nem quais frases exatas do usuário a disparam. Isso faz Claude hesitar ou usar a skill errada.

❌ Como está (supabase-ops)
Manages Supabase migrations, types generation, RLS policies, and edge functions
✅ Como deveria ser
Manages Supabase for MdFlow: migrations, RLS policies, RPCs, types generation. Use when user asks to "create migration", "add RLS", "generate types", "run query no Supabase", "criar tabela", "política de acesso" or mentions edge functions.

🔵 P3 — Agents CC sem metadata estruturada

Os agents em ~/.claude/agents/ estão bem estruturados com frontmatter YAML (name, description, tools, model). Mas poderiam se beneficiar de: campo compatibility (quais MCPs precisam), metadata.version para rastrear versões, e descriptions mais ricas com exemplos de trigger.

O que o CC deve implementar — plano de lapidação

Tarefa CC — Fase 1: Adicionar frontmatter nas 8 skills críticas

Para cada skill sem frontmatter, o CC deve:

  • Ler o corpo existente do SKILL.md para extrair nome, triggers e funcionalidade
  • Gerar o bloco YAML com name, description rica (200-400 chars com triggers), e metadata.version
  • Inserir no topo do arquivo sem modificar o conteúdo existente
  • Validar que não tem tags XML e que o name é kebab-case

Tarefa CC — Fase 2: Enriquecer descriptions das skills com frontmatter fraco

Para skills com description <200 chars (reconstrucao-pagina, supabase-ops, pdf-extract, etc.), expandir a description para incluir: contexto de uso, frases-gatilho exatas, casos típicos. Limite: 1024 chars.

Tarefa CC — Fase 3: Criar references/ nas skills longas

Skills com SKILL.md >200 linhas (frontend, spec, reconstrucao-pagina, cc-prompt-gen) devem ter o conteúdo técnico movido para references/, deixando o SKILL.md como índice enxuto. Isso reduz tokens no system prompt.

Tarefa CC — Fase 4: Versionar e padronizar metadata

Adicionar em todas as skills: metadata.author: JANA, metadata.version: X.Y.Z, metadata.updated: 2026-03-18. Isso facilita rastrear qual versão está ativa.
