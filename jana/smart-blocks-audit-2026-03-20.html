🐛 Issues Identificados

P0-01
N8N chama JARBAS em vez do GAS — placeholders não substituídos
P0 Bloqueador n8n · C2-ChamarJARBAS

O nó C2-ChamarJARBAS aponta para https://jarbas.jarbas-mdlab.com/olivia/criar-doc. Esta era a rota JARBAS que foi descartada em favor do GAS direto (decisão arquitetural da spec). O n8n nunca foi atualizado para apontar para o GAS.

Resultado: o documento é gerado com {{bloco_1}}, {{bloco_2}}, {{advogado_nome}} etc. sem substituição — exatamente o que você viu no Google Doc.

Root Cause
Nó C2 nunca foi migrado de JARBAS → GAS após a decisão arquitetural. C1-MontarPayload monta um payload compatível com GAS v4.0 (campo gasPayload) mas C2 o manda para o endpoint JARBAS que não reconhece esse formato.
Fix
Atualizar C2 para chamar a URL do GAS v4.0 (Web App deployado no Google Apps Script). Payload já está correto em C1 — apenas trocar a URL e ajustar body para ={{ JSON.stringify($json.gasPayload) }}.
P0-02
Seletor de Versão exibe nome do modelo repetido ao invés de "Versão N"
P0 Bloqueador Frontend · SmartBlocksStep1.tsx

O Select de versões prioriza v.nome para exibir o label. O campo nome em olivia_modelos_peticao contém o nome completo do modelo (ex: "Instituição Financeira - Cartão de Crédito"), não a versão. Resultado: todas as 3 versões aparecem com o mesmo texto.

Root Cause
Código atual: v.nome ? v.nome : v.versao != null ? `Versão ${v.versao}` : 'Versão sem nome'
v.nome nunca é null/empty — sempre tem o nome do modelo. Deveria sempre mostrar "Versão N" usando o campo versao (número inteiro), e o nome apenas como tooltip ou subtítulo.
Fix
Trocar label para: v.versao != null ? `Versão ${v.versao}` : v.nome || 'Versão sem nome'. Opcionalmente adicionar v.nome como subtítulo menor embaixo do número.
P1-01
Tags no Step 2 ignoram campos obrigatório/múltipla-seleção de olivia_modelo_tags
P1 Importante Frontend · useSmartBlocks.ts

A junction table olivia_modelo_tags tem colunas próprias: obrigatoria e ordem. Essas colunas são por modelo — uma mesma tag pode ser obrigatória num modelo e opcional em outro.

O hook useSmartBlocks busca essas colunas na query (obrigatoria, ordem), mas ao montar o objeto TagComSecao usa os campos globais da tag (olivia_tags.selecao_obrigatoria e olivia_tags.multipla_selecao), ignorando os valores específicos do modelo.

Root Cause
Em useSmartBlocks.ts, o mapeamento deveria priorizar row.obrigatoria ?? tag.selecao_obrigatoria mas usa apenas tag.selecao_obrigatoria. E a ordem deveria usar row.ordem ?? tag.ordem.
Fix
No mapeamento de tagsData → TagComSecao[]: usar row.obrigatoria ?? row.olivia_tags.selecao_obrigatoria para selecao_obrigatoria e row.ordem ?? row.olivia_tags.ordem para ordem.
P1-02
Tags sem ordem definida (86/183) → exibição desordenada no Step 2
P1 Importante Banco · olivia_tags olivia_modelo_tags

86 das 183 tags ativas têm ordem = 0 ou NULL. A query ordena por ordem ASC, então tags sem ordem ficam todas no início agrupadas — a ordem exibida no wizard não reflete a sequência lógica da petição.

Fix
1. Garantir que olivia_modelo_tags.ordem esteja preenchido para todos os modelos ativos (admin ou migration).
2. Fallback no frontend: se ordem for null/0, ordenar por nome como critério secundário.
3. Curto prazo: .order('ordem', { ascending: true, nullsFirst: false }) já evita nulos no topo.
P1-03
Descrição dos blocos exibida em formato estranho no Step 2
P1 Importante Frontend · SmartBlocksStep2TagWizard.tsx

O campo bloco.conteudo está sendo renderizado como texto plano com slice(0,120). Para blocos jurídicos o conteúdo pode conter HTML/markdown ou ser muito técnico, ficando ilegível como preview.

Fix
Duas opções: (a) Remover descrição dos blocos no card — mostrar apenas título + badge Drive + indicador de seleção; (b) Exibir conteúdo sanitizado como HTML com dangerouslySetInnerHTML + DOMPurify limitado a 2 linhas. Recomendo (a) — mais limpo.
P2-01
Métricas do Step 3 sempre vazias (TEMPO, CUSTO, TOKENS)
P2 Qualidade Frontend · SmartBlocksStep3Geracao.tsx

O Step 3 exibe métricas de tempo, custo e tokens que sempre aparecem como "—". O modo Smart Blocks usa GAS (não IA), então custo e tokens são sempre zero — mas o tempo de processamento pode e deve ser calculado.

Fix
Calcular tempo de processamento no frontend (delta entre início do polling e SUCESSO). Ocultar campos "Custo" e "Tokens" para modo Smart Blocks (não faz sentido sem IA). Mostrar apenas: Tempo total, Blocos montados, Status GAS.
