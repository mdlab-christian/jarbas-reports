Step 2 — Bug: Lista flat sem wizard de tags
P0 Frontend Step 2 exibe TODOS os blocos ao mesmo tempo, sem tags, sem wizard

O componente SmartBlocksStep2Blocos carrega todos os blocos do modelo e os exibe em uma lista flat com checkboxes. O comportamento esperado é completamente diferente:


Esperado: cada olivia_tag do modelo deve ser um step do wizard. A tag é a "pergunta" (ex: "Qual documento o réu apresentou?"). Os blocos daquela tag são as "respostas" (ex: "Documento e selfie normal", "Documento ilegível", etc). Se multipla_selecao=false → radio (escolha 1). Se true → checkboxes. Uma tag por vez — usuário avança pelo wizard respondendo cada pergunta.


Impacto atual: Advogado vê 50+ blocos misturados sem contexto. Não sabe quais são mutuamente exclusivos. Seleciona blocos incompatíveis. Gera petição com contradições.

🔧 ARQUITETURA DO FIX
NOVO FLUXO Step 2:
1. Buscar tags do modelo via olivia_modelo_tags (já existente no código)
2. Para cada tag: buscar blocos com tag_ids @> [tag.id] (blocos desta tag)
3. Renderizar wizard: tagIndex = 0..N
   - Header: tag.nome (a "pergunta")
   - tag.descricao (se houver, como subtítulo)
   - Blocos = as "respostas"
     - multipla_selecao=false → RadioGroup (1 opção obrigatória)
     - multipla_selecao=true → CheckboxGroup (múltiplas)
     - selecao_obrigatoria=true → não permite avançar sem selecionar
4. Nav: [← Tag anterior] [Tag N de N] [Próxima tag →]
5. Ao completar todas as tags → habilitar botão "Ir para Gerar"
6. REMOVER botão "Adicionar bloco personalizado" (conforme spec)
TypeScript — Nova estrutura de estado
// Adicionar ao SmartBlocksWizardState interface TagSelection { tag_id: string; tag_nome: string; multipla_selecao: boolean; selecao_obrigatoria: boolean; blocos: SmartBlocItem[]; selected_ids: string[]; // IDs selecionados para esta tag } // No state, substituir blocos_ids simples por estrutura por tag: interface SmartBlocksWizardState { // ... campos existentes ... tag_selections: TagSelection[]; // wizard por tag tag_wizard_index: number; // qual tag está sendo respondida // blocos_ids continua existindo (derivado de tag_selections) }

Step 3 — Root Cause: Petição completamente errada
P0 Backend JARBAS espera secoes[].conteudo — n8n envia secoes[].google_doc_id

Evidence do banco (olivia_historico, top erros):


• "Cada seção deve ter campo 'conteudo' (índice 0)" — 2 ocorrências
• "titulo_documento é obrigatório e deve ser string não-vazia" — 4 ocorrências
• "Drive upload falhou: Quota exceeded" — 3 ocorrências (quando consegue gerar, sobrecarrega a API do Drive)


Root cause: O nó C1-MontarPayloadGAS no n8n monta secoes com esta estrutura:

n8n C1 — O que envia (ERRADO)
secoes = blocos.map(b => ({ ordem: b.ordem, tipo: 'bloco', google_doc_id: b.google_doc_id, // ← envia google_doc_id placeholder: b.placeholder })); // JARBAS valida: secoes[0].conteudo → undefined → ERRO
JARBAS skill_olivia_criar_doc — O que espera
// JARBAS espera um de dois formatos: // Formato A (texto direto): secoes = [{ conteudo: "texto do bloco aqui...", tipo: 'bloco' }] // Formato B (referência Google Doc — precisa copiar doc): secoes = [{ google_doc_id: "1xxx", tipo: 'bloco' }] // → JARBAS precisa estar preparado para lidar com google_doc_id

Existem 2 abordagens para o fix:

🔧 OPÇÃO A (RECOMENDADA) — n8n busca conteúdo dos blocos e envia como texto
No nó B2-BuscarBlocos do n8n, adicionar 'conteudo' no select:
  ?select=id,titulo,google_doc_id,conteudo

No C1-MontarPayloadGAS, popular secoes com conteudo:
  secoes = blocos.map(b => ({
    ordem: b.ordem,
    tipo: 'bloco',
    conteudo: b.conteudo || '',   // ← adicionar conteúdo
    google_doc_id: b.google_doc_id,
    placeholder: b.placeholder
  }));

PRÓ: Simples, funciona com a API atual do JARBAS.
CON: Payload maior. Mas conteúdo dos blocos é texto curto, OK.
🔧 OPÇÃO B — JARBAS API aceita google_doc_id e copia o conteúdo
Atualizar skill_olivia_gerar_peticao.mjs no JARBAS para:
- Se secoes[i].google_doc_id → buscar conteúdo do Google Doc via API
- Substituir o placeholder no template com o conteúdo obtido

PRÓ: Usa os Google Docs dos blocos (fonte de verdade formatada).
CON: Mais lento (N chamadas à API Google Docs), rate limit por bloco.
     Quota exceeded já acontece — piorar com mais chamadas é arriscado.
🚨
Quota do Google Drive está sendo excedida
3 erros de "Drive upload falhou: Quota exceeded" no banco. Cada geração faz N+1 chamadas ao Drive (1 por bloco + 1 para criar o doc final). Com 1244 blocos e múltiplas gerações simultâneas, isso vai estourar o plano gratuito. A Opção A (usar o campo conteudo do Supabase) elimina completamente esse problema para leitura, mantendo apenas 1 chamada de write para criar o doc final.
P1 Backend titulo_documento vazio quando processo sem numero_cnj

4 erros de "titulo_documento é obrigatório e deve ser string não-vazia". O nó C1 constrói o nome do arquivo assim:

n8n C1 — Linha problemática
const nomeArquivo = `Petição - ${processo.numero_cnj || ''} - ${cliente.nome || ''}`.trim(); // Se numero_cnj=null e cliente.nome=null → "Petição - - ".trim() = "Petição - -" // JARBAS valida: se string vazia ou só espaços → lança erro // BUG: "Petição - -" não é vazia mas pode ser inválida
🔧 FIX
const parts = ['Petição'];
if (processo.numero_cnj) parts.push(processo.numero_cnj);
if (cliente.nome) parts.push(cliente.nome);
const nomeArquivo = parts.join(' - ');
// Garantia: sempre começa com "Petição" → nunca vazio

P2 Backend Timeout sem PATCH de status no banco

Se o frontend estoura o timeout de 3 minutos mas o n8n ainda está rodando, o registro fica EM_PROGRESSO indefinidamente. O frontend atual faz apenas update local do state sem PATCH no banco após timeout. Com 37 registros ABANDONADO acumulados, isso já é um problema recorrente.

🔧 FIX — Frontend SmartBlocksStep3Gerar.iniciarGeracao
// Na lógica de timeout (após os 3min):
setState(p => ({ ...p, status_geracao: 'ERRO' }));
setErrorMsg('Tempo limite excedido...');
// ← ADICIONAR:
await supabase
  .from('olivia_historico')
  .update({ 
    status: 'ABANDONADO',
    error_message: 'Timeout no frontend (3min) — n8n pode ainda estar processando',
    finished_at: new Date().toISOString()
  })
  .eq('id', historicoId)
  .eq('status', 'PENDENTE'); // só atualiza se ainda PENDENTE, não sobrescreve SUCESSO

Melhorias Sugeridas — JANA
💡 Melhoria 1 — Wizard com progresso visual por tag
Ao refatorar o Step 2 para wizard por tags, adicionar um stepper interno ao Step 2 mostrando "Pergunta N de Total". Ex: "3 de 8 perguntas respondidas". Cada tag que tem selecao_obrigatoria=true deve ter um indicador vermelho se não respondida ao tentar avançar. Tags opcionais (selecao_obrigatoria=false) podem ser puladas com botão "Pular".

UX esperada: Similar a um quiz jurídico — cada pergunta é uma tela limpa com título da tag, subtítulo explicativo, e as opções de bloco como cards clicáveis com preview do conteúdo em hover/tooltip.
💡 Melhoria 2 — Preview de conteúdo do bloco inline
Ao exibir os blocos como opções do wizard, mostrar as primeiras 2-3 linhas do conteudo do bloco diretamente na card (truncado). Botão "Ver completo" expande em Sheet lateral. Isso permite o advogado tomar decisão informada sem precisar clicar em cada opção.

Dado disponível: olivia_blocos_texto.conteudo está no banco, só precisa ser carregado na query.
💡 Melhoria 3 — Limpar lixo do banco automaticamente
37 registros ABANDONADO + 25 ERRO acumulados no banco. Sugestão: criar um cron no n8n que roda semanalmente e move registros ERRO/ABANDONADO com mais de 30 dias para uma tabela de arquivo (olivia_historico_arquivo), ou simplesmente faz soft-delete. Mantém o banco limpo e as queries de polling mais rápidas.
💡 Melhoria 4 — Resumo da petição antes de gerar (Step 3)
No Step 3 (antes de clicar em "Gerar"), mostrar um resumo estruturado das escolhas: uma linha por tag respondida com a opção selecionada. Ex:

Dano Moral → "Não entendeu que houve dano"
Preliminares → "Incompetência do juízo" + "Ilegitimidade passiva"
Documento → "Documento e selfie normal"

Isso permite o advogado revisar antes de gerar e voltar se errou alguma resposta.
💡 Melhoria 5 — Indicador de "tags sem blocos" no Step 1 ao selecionar modelo
Ao selecionar um modelo no Step 1, fazer uma query rápida para verificar se o modelo tem tags com blocos vinculados. Se alguma tag não tem blocos, mostrar um badge de aviso antes do usuário avançar. Evita o usuário chegar no Step 2 e ver "Nenhum bloco encontrado".
