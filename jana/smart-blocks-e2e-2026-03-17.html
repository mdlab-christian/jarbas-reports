🦞 JARBAS — /olivia/criar-doc (olivia-doc-creator v2.0)
📥 Payload recebido do n8n
POST https://jarbas.jarbas-mdlab.com/olivia/criar-doc Headers: Content-Type: application/json mdflow: MdL1501@ (auth header) { titulo_documento: string, google_folder_id: "1JTbKJqbmPzR...", peticao_html: string (conteudo_template), modelo: { google_doc_id? }, blocos: [ { ordem, placeholder, google_doc_id } ], secoes: [ { ordem, tipo, google_doc_id, placeholder } ], jurisprudencias: [ { placeholder, google_doc_id, titulo } ], tags: { {{campo}}: valor }, campos_substituicao: { ... }, configuracoes: { ... }, config: { fonte, tamanho, ... } }
📤 O que o JARBAS faz
1.Valida blocos_html / peticao_html (mín 50 chars)
2.Se sem google_folder_id → pandoc DOCX + Supabase Storage
3.Com google_folder_id → olivia-doc-creator.mjs
4.Drive API multipart upload (HTML → Google Doc nativo)
5.Aplica tags {{campo}} via regex
6.ABNT wrapper: margem 3.17cm/2.54cm, fonte configurável
7.Retorna { success, google_doc_url, google_doc_id }
⚠️ Recebe peticao_html mas precisa montar os blocos — lógica interna a confirmar

🚨 Gaps e Problemas Identificados
🔴
G1 — Registros PENDENTE stuck no banco (9 registros)
Geração foi disparada mas nunca recebeu callback de SUCESSO/ERRO. Causa provável: JARBAS /olivia/criar-doc falhou silenciosamente ou n8n não chegou ao node D1/D2. Isso polui o histórico e faz o frontend de recovery mostrar dados inválidos.
P0
🔴
G2 — C2-ChamarJARBAS: JARBAS não concatena blocos em série
O payload envia peticao_html (conteudo_template do modelo) + array de blocos com google_doc_id. O olivia-doc-creator.mjs atual cria um Doc com peticao_html mas NÃO busca e concatena os Google Docs dos blocos. O JARBAS precisa de lógica específica para: 1) buscar conteúdo de cada bloco via Drive API, 2) substituir placeholders {{bloco_N}} no HTML base, 3) então criar o Doc final.
P0
🟡
G3 — versao_id enviado mas ignorado no workflow n8n
O frontend envia versao_id (olivia_modelos_peticao.id) no payload. O n8n recebe mas C1-MontarPayloadGAS usa B1e-BuscarModelo baseado em modelo_id (grupo). Se versao_id não é usado para buscar o conteudo_template correto, pode buscar a versão errada do modelo.
P1
🟡
G4 — B1 (dados do processo) não verificado nos nodes analisados
O C1-MontarPayloadGAS referencia B1-BuscarProcesso, B1b-BuscarAdvogado, B1c-BuscarCliente, B1d-BuscarEmpresa, B1e-BuscarModelo — mas esses nodes não apareceram no dump da API do n8n (provavelmente fora do range de exibição). É necessário verificar se esses nodes estão corretamente conectados e retornando dados.
P1
🟡
G5 — instrucoes_adicionais enviado mas não integrado ao JARBAS
O frontend Smart Blocks tem campo instrucoes_finais que é enviado como instrucoes_adicionais no payload. O n8n recebe mas o prompt enviado ao JARBAS não inclui essas instruções no contexto da geração. Para o Modo Agente está melhor resolvido; para Smart Blocks, falta.
P1
🔵
G6 — Petição Inicial bloqueada no Gerador (comportamento esperado)
Confirmado via análise do código: Petição Inicial não deve aparecer no gerador Smart Blocks nem Express. Só é gerada no fluxo de Distribuição. Isso é correto — mas vale documentar explicitamente nos tipos de petição (olivia_tipos_peticao.ativo=false para Inicial no contexto do gerador).
P2 — doc
🔵
G7 — Recovery banner (D6) pode mostrar gerações antigas de blocos diferentes
O banner de recovery busca qualquer SUCESSO do processo + modo=SMART_BLOCKS sem verificar se os blocos_ids são os mesmos. Se o usuário tentou uma réplica antes e agora quer uma AR Negativo para o mesmo processo, o banner vai sugerir recuperar a geração antiga irrelevante.
P2
