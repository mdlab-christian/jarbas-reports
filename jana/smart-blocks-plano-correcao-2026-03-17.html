🔴 P0 — Crítico (bloqueiam funcionamento)

P0
GAS v4.0 não está integrado ao workflow Smart Blocks
n8n / N8N-OLIVIA-GERAR-MANUAL / node "Inserir Historico"

O workflow atual NÃO chama o GAS. Ele busca o conteudo (texto plano) dos blocos, concatena e faz PATCH do historico com google_doc_url: null. O GAS deployado em AKfycbzoyKb... nunca é chamado.

Resultado: o advogado vê tela de "SUCESSO" mas sem link para o Google Doc.

  • O n8n busca conteudo dos blocos mas o GAS precisa de google_doc_id de cada bloco
  • O workflow também não busca o google_doc_id do modelo de petição
  • Nenhum nó do workflow faz HTTP Request para o GAS
✅ Fix necessário

Adicionar ao n8n (após buscar blocos):

1. Buscar google_doc_id do modelo: SELECT google_doc_id FROM olivia_modelos_peticao WHERE id = versao_id

2. Buscar google_doc_id, placeholder, ordem de cada bloco em olivia_blocos_texto

3. Montar payload GAS v4.0 (modelo + blocos[] + juris[] + tags_substituicao)

4. HTTP Request POST para https://script.google.com/macros/s/.../exec

5. UPDATE historico com google_doc_url da resposta do GAS

P0
Campos google_doc_id ausentes nos blocos — dados faltantes no banco
Supabase / olivia_blocos_texto / olivia_modelos_peticao

O GAS v4.0 precisa de google_doc_id para cada bloco e para o modelo de petição. Sem esse ID, o GAS não tem de onde copiar o conteúdo formatado.

Situação atual: o n8n usa conteudo (texto plano), ignorando o google_doc_id. Isso indica que esses campos podem estar:

  • Nulos no banco para a maioria dos blocos (nunca foram preenchidos)
  • Só preenchidos nos blocos da Olivia Express antiga

Diagnóstico necessário antes de integrar o GAS: checar quantos blocos têm google_doc_id não-nulo.

✅ Fix necessário

Query de diagnóstico:

SELECT COUNT(*), COUNT(google_doc_id) FROM olivia_blocos_texto WHERE organizacao_id = '...'

Se a maioria estiver nula: definir estratégia — ou popular os IDs ou criar docs no Drive para cada bloco automaticamente.

🟠 P1 — Alto (bugs funcionais)

P1
Step 2 blocos não carregam — race condition na cadeia de queries
SmartBlocksStep2Blocos.tsx / query blocosDb

A query de blocos tem enabled: !!organizacao_id && (modeloTagsIds !== undefined). O problema: modeloTagsIds passa por uma cadeia de 3 queries:

modeloGrupo → junctionTagIds → modeloTagsIds

Se junctionTagIds ainda está em loading, modeloTagsIds é undefined, então enabled=false e a query de blocos nunca dispara. Em ambientes lentos (Supabase cold start), isso trava.

O E2E confirmou: 0 checkboxes de blocos encontrados mesmo depois de selecionar modelo.

✅ Fix

Mudar condição para: enabled: !!organizacao_id && !!state.modelo_id && modeloTagsIds !== undefined

E adicionar fallback: se junctionTagIds.isLoading, mostrar skeleton. Se junctionTagIds.isSuccess && !junctionTagIds.data.length → usar modeloGrupo.tags_ids ou carregar todos os blocos.

P1
E2E: busca de processo retorna 0 resultados
SmartBlocksStep1Identificacao.tsx / query sb-step1-processos

No E2E automatizado, a busca de processo retornou 0 resultados. A query só executa se debouncedSearch.length >= 2 ou sem filtro (carrega 20 processos).

Causa provável: sem debouncedSearch, a query executa sem filtro e deveria retornar os 20 mais recentes. O E2E abriu o popover mas não esperou o resultado aparecer antes de tentar interagir — timing issue no Playwright.

No browser real, a query funciona. O bug é no script de E2E (não no componente).

✅ Fix no E2E

No script E2E: após abrir o popover, aguardar networkidle ou esperar que ao menos 1 item apareça antes de tentar selecionar. Adicionar waitForSelector('.w-full.px-3') com timeout de 5s.

P1
versao_id enviado ao n8n mas completamente ignorado
N8N-OLIVIA-GERAR-MANUAL / node "Inserir Historico"

O frontend envia versao_id (ID do olivia_modelos_peticao) no payload. O n8n recebe e valida, mas não usa. O google_doc_id do modelo — que deveria ser buscado usando esse versao_id — nunca é consultado.

Isso é exatamente a informação que o GAS precisa para saber qual template copiar (modelo.google_doc_id).

✅ Fix

No n8n, adicionar consulta: SELECT google_doc_id, nome FROM olivia_modelos_peticao WHERE id = data.versao_id e usar google_doc_id no payload do GAS.

🟡 P2 — Médio (drifts e inconsistências)

P2
n8n gera "texto_gerado" concatenando conteúdo — abordagem errada para Smart Blocks
N8N-OLIVIA-GERAR-MANUAL / node "Inserir Historico"

O workflow monta um textoGerado concatenando conteudo dos blocos com --- entre eles e salva em olivia_historico.texto_gerado. Isso faz sentido para Olivia Express (geração por IA), mas para Smart Blocks a saída correta é um Google Doc com formatação preservada.

O campo texto_gerado sendo preenchido com texto plano dá falsa impressão de que o documento foi gerado quando na verdade é só um dump do banco.

✅ Fix

Para Smart Blocks, o workflow deveria salvar google_doc_url + google_doc_id (vindos do GAS) e deixar texto_gerado = null. A concatenação de conteúdo pode ser removida ou mantida como fallback/log.

P2
Modelo n8n aceita dois nomes de campo conflitantes
N8N / Validar Payload + Inserir Historico

O campo de instruções extras tem dois nomes: o frontend envia instrucoes_finais, o n8n aceita instrucoes_adicionais || instrucoes_finais. O banco salva como instrucoes_adicionais. Funciona, mas é drift que vai causar confusão.

✅ Fix

Padronizar para instrucoes_adicionais em todos os lugares. Atualizar o frontend para enviar instrucoes_adicionais e o n8n para só aceitar esse nome.

P2
Duplo registro potencial em olivia_historico
Frontend (OliviaGeradorSmartBlocks.tsx) + N8N

O frontend cria um registro com status=PENDENTE e envia o historico_id ao n8n. O n8n faz PATCH pra SUCESSO. Até aí OK.

Problema: se o frontend falhar após o INSERT mas antes de enviar o webhook (ex: timeout de rede), o registro fica em PENDENTE para sempre. Se o usuário tentar de novo, cria outro registro — dois PENDENTEs para o mesmo processo.

O n8n tem um fallback de INSERT sem historico_id mas isso criaria um terceiro registro.

✅ Fix

Adicionar limpeza de PENDENTEs antigos antes de criar novo: DELETE FROM olivia_historico WHERE processo_id = X AND modo = 'SMART_BLOCKS' AND status = 'PENDENTE' AND created_at < now() - interval '10 minutes'

Ou usar o RecoveryBanner já existente para detectar e reutilizar o registro.

P2
Skill JARBAS (skill_olivia_gerar_peticao) usa abordagem errada para Smart Blocks
~/jarbas/src/skills/skill_olivia_gerar_peticao.mjs (Mac Mini)

O JARBAS tem um endpoint /olivia/criar-doc que recebe peticao_html e cria o doc. Isso é correto para Olivia Express (IA gera o HTML).

Para Smart Blocks, o correto é chamar o GAS v4.0 (copia template + substitui blocos com formatação preservada). A skill atual passa peticao_html e tags vazias — não passa modelo.google_doc_id nem blocos[].google_doc_id.

Isso significa que mesmo se o JARBAS for chamado pelo Smart Blocks, o documento seria gerado do zero em HTML (perdendo a formatação dos templates).

✅ Fix

Criar rota separada no JARBAS para Smart Blocks: POST /olivia/smart-blocks que recebe o payload v4.0 e chama o GAS diretamente. Ou: o n8n chama o GAS diretamente (sem passar pelo JARBAS).

🔵 P3 — Baixo (UX e débitos técnicos)

P3
Step 3: download DOCX ainda presente mas Smart Blocks não gera DOCX
OliviaGeradorSmartBlocks.tsx / função downloadDocx

Existe uma função downloadDocx no Step 3 mas o Smart Blocks não passa por geração de texto via IA — apenas cria um Google Doc. O botão de download de DOCX pode aparecer sem conteúdo ou não aparecer, dependendo do estado.

✅ Fix

Remover a função downloadDocx do Smart Blocks ou, se desejado, adicionar botão "Exportar DOCX" que converte o Google Doc via API.

P3
Polling timeout de 3min pode deixar usuário preso
OliviaGeradorSmartBlocks.tsx / iniciarGeracao

Se o GAS demorar mais de 3 minutos (petições longas, cold start), o frontend exibe "Tempo limite excedido" mas o histórico pode ter sido gerado com sucesso depois. O usuário não tem como retomar — teria que abrir /olivia novamente e usar o RecoveryBanner.

✅ Fix

Ao exibir timeout, checar o status do historico uma última vez antes de mostrar erro. Se status=SUCESSO, exibir sucesso. Também deixar mensagem de erro com link "Verificar histórico" para o usuário acessar o resultado.

P3
Script E2E não testa com dados reais — gaps de cobertura
~/beta-mdflow/scripts/smart-blocks-e2e-review.mjs

O script E2E usa a conta mdlab.equipe@gmail.com mas depende de processos existentes nessa conta no Supabase. Se não houver processos cadastrados (ou se o dev server apontar para banco diferente), toda a cadeia após Step 1 falha.

Também não testa o fluxo de geração real (n8n + GAS) — só testa a UI.

✅ Fix

Adicionar seed de dados de teste antes do E2E: criar processo + bloco + modelo com google_doc_id real via script. Ou usar um processo já existente no banco com CNJ conhecido e hard-codar no E2E para buscar.
