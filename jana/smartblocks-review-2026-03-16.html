🚨 Issues Priorizados

🔴 P1 — Bloqueia geração correta
Polling do frontend não encontra o SUCESSO

O frontend cria um registro em olivia_historico com ID X, chama o webhook mas não passa historico_id no payload. O workflow cria um novo registro (ID Y) e salva SUCESSO no ID Y. O frontend faz polling no ID X — que nunca muda. Resultado: o usuário vê "gerando…" para sempre até timeout de 2 minutos.

Fix: Em OliviaGeradorSmartBlocks.tsx, adicionar historico_id: hist.id no payload antes de chamar callWebhook. O workflow já tem lógica para usar o historico_id quando recebido (A1c-MergeHistorico).
// Linha ~after do INSERT hist
const webhookPayload = {
  request_id: requestId,
  organizacao_id,
  actor_user_id: userId,
  historico_id: hist.id, // <-- adicionar isso
  data: { ... }
};
🟡 P2 — UX degradada
Step 2 mostra todos os 1.244 blocos sem filtro

Como olivia_modelo_tags está vazia e a maioria dos olivia_modelos_grupo.tags_ids está [], o Step 2 (seleção de blocos) exibe todos os blocos sem filtro. O advogado precisa navegar por centenas de blocos para encontrar os relevantes.

Fix: Popular olivia_modelos_grupo.tags_ids com os IDs de tags relevantes para cada modelo, ou popular a junction table olivia_modelo_tags. Isso filtrará automaticamente os blocos por contexto.
🟡 P2 — Dados incompletos
104 modelos sem conteudo_template (31,6%)

329 modelos publicados, mas apenas 225 têm conteudo_template preenchido. Para os 104 restantes, o JARBAS recebe <p>Petição gerada via Smart Blocks</p> como template base — o Google Doc gerado pode ter conteúdo minimal se o modelo não tiver Google Doc ID válido.

Mitigação: O JARBAS também usa o google_doc_id_origem do modelo como template. Se este estiver preenchido (todos 329 têm), o documento é gerado a partir do Google Doc diretamente.
ℹ️ P3 — Não urgente
olivia_modelo_juris vazia — jurisprudências não vinculadas

O workflow já lida com isso retornando jurisprudencias: [] — não quebra. Mas o recurso de jurisprudências no Smart Blocks está inativo enquanto a tabela estiver vazia.
