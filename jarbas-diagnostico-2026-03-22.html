Watchdog + KeepAlive = Loop infinito P0 · CRÍTICO
Root cause do flood de restarts — 407x em 7h

O script jarbas-watchdog.sh mata o processo do gateway quando detecta muitos sendMessage failed. Mas o plist do gateway tem KeepAlive: true — então o launchd sobe o gateway automaticamente de novo, em ~62 segundos. O watchdog detecta o "flood" de novo, mata, o launchd sobe... loop infinito.

Resultado: 407 ciclos de boot/shutdown entre 00:00 e 07:00 BRT. O gateway nunca ficou estável por mais de 1 minuto.

2026-03-22T00:02:53 → Listening (62s depois)
2026-03-22T00:03:55 → Listening (62s depois)
# ThrottleInterval no plist = 5s, mas o processo demora ~60s pra iniciar e ser detectado
# Total: 407 restarts em 7h = $0 de AI (gateway morreu antes de processar) MAS...
✅ Correção
  1. Remover ou refatorar o watchdog — ele não deve matar o gateway diretamente
  2. Se o flood de sendMessage for real → o próprio OpenClaw tem flood protection. Deixe ele gerenciar.
  3. O watchdog só deve bootar o gateway se ele estiver offline (o que ele já faz corretamente via pgrep)
  4. Remova o kill do watchdog ou adicione um guard: "só mata se o processo estiver travado > 5min sem resposta"

Sessão principal de 808KB corrompida P1 · ALTO
Sessão gigante causando "session file repair skipped: invalid session header"

O MEMORY.md também alerta: "Workspace context > 20KB = flood de sendMessage no gateway". Com 808KB de sessão, esse limite foi ultrapassado em 40x.

  2. Ou apagar e deixar o gateway criar uma nova sessão fresca
  3. Configurar contextTokens: 60000 (atual: 100000) para evitar sessões gigantes

4
Heartbeat a cada 1h dispara Claude mesmo sem uso P1 · ALTO
HEARTBEAT.md tem 6 verificações ativas — todas consomem tokens

O HEARTBEAT.md tem 6 verificações configuradas para rodar a cada heartbeat (1h):

  • Monitor de contexto dos grupos (verifica sessions via API)
  • Tarefas presas no Supabase
  • Saúde do gateway
  • Health check dos tribunais EPROC (a cada 2h)
  • Saúde da JANA via SSH
  • GROUP_CONTEXT_CHECKPOINT (cron 1h) — salva arquivos de estado

Cada heartbeat = 1 chamada completa ao Claude Sonnet (com todo o system prompt + MEMORY.md + SOUL.md + workspace). Isso custa em torno de $0.15-0.30 por heartbeat, dependendo do contexto carregado.

Com 407 restarts, muitos heartbeats foram disparados em sequência. Mesmo que a maioria tenha retornado HEARTBEAT_OK sem chamar a API, os que chamaram a API causaram os 4 erros de billing.

✅ Correção
  1. Mover verificações automáticas (EPROC, JANA, grupos) para scripts n8n — não para o Claude
  2. O heartbeat do Claude deve ser simples: só checar HEARTBEAT.md e retornar HEARTBEAT_OK se vazio
  3. Reduzir frequência para a cada 2h em vez de 1h

Dois LaunchAgents controlando o mesmo processo P2 · MÉDIO
guardian.plist + ai.openclaw.gateway.plist = conflito de controle

Dois LaunchAgents têm autoridade sobre o gateway:

  • ai.openclaw.gateway.plist — gerenciado pelo OpenClaw, com KeepAlive: true
  • jarbas.openclaw.guardian.plist — seu script customizado, roda a cada 30s

O conflito ficou evidente quando o gateway subiu com nome "JARBAS (OpenClaw) (2)" — sinal de duas instâncias via Bonjour. O guardian ativa o plist do openclaw via launchctl bootstrap, mas o openclaw já está gerenciado — resultado: duas instâncias em portas diferentes ou conflito de PID.

✅ Correção
  1. Desativar o guardian — o KeepAlive nativo do OpenClaw já faz o trabalho
  2. Se quiser um guardian customizado, ele só deve verificar, nunca reiniciar
