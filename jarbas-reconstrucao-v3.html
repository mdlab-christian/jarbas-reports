Problemas Identificados no JARBAS Atual

Diagnóstico

🔴 P0 — Consumo de tokens inflado

SOUL.md, MEMORY.md, HEARTBEAT.md, skills de OlivIA, Controller e grupos todos carregados em TODA conversa. O sistema injeta >20KB de contexto por requisição — mesmo quando não é necessário.

🔴 P0 — Contexto monolítico

Uma única instância JARBAS responde Telegram DM, grupos OlivIA, Controller, CRM, Network. Contexto de OlivIA vaza para DM do Christian e vice-versa. Isolamento é workaround manual.

🔴 P0 — Heartbeat + crons competindo com fila

Heartbeats a cada 5min + cron GROUP_CONTEXT_CHECKPOINT (1h) + SYNC_CC_AGENTS (30min) interrompem tasks críticas. Semáforo Claude mitiga mas não resolve o root cause.

🟡 P1 — MEMORY.md > 20KB causa flood

Problema documentado: workspace context >20KB = flood de sendMessage no gateway. Obriga manter MEMORY.md artificialmente enxuto, perdendo histórico importante.

🟡 P1 — Skills de domínio cruzado

Skills de OlivIA registradas no mesmo start.mjs do JARBAS principal. Quando OlivIA processa petição, trava Claude semaphore para o DM do Christian.

🟡 P1 — Confusão de workspaces

~/jarbas/ (Node.js broker) vs ~/.openclaw/workspace/ (OpenClaw) têm arquivos duplicados/conflitantes. SOUL.md existe nos dois. Gera alucinação sobre onde alterar arquivos.

🔵 P2 — ChromaDB RAG não operacional

skill_rag_mdflow.mjs e skill_memoria_contextual.js dependem do ChromaDB mas ele nunca foi populado. 2 skills registradas mas nunca usadas.

🔵 P2 — Instâncias de agentes sem limite

OlivIA gera petições em fila mas pode criar múltiplos processos Claude simultâneos se a fila acumular. Sem circuit-breaker por domínio.

Contexto por Instância — Regra dos 8KB

Token Budget
🚨
Regra absoluta: Nenhum workspace de instância deve exceder 8KB total. Workspace context > 20KB = flood de sendMessage no gateway (problema confirmado). Cada instância carrega APENAS o contexto do seu domínio.

🦞 JARBAS Core (~6KB)

  • 📄 SOUL.md — versão enxuta (v4.1 atual OK)
  • 📄 USER.md — mantém
  • 📄 MEMORY.md — manter < 4KB rigoroso
  • 📄 HEARTBEAT.md — apenas verificações do Core
  • ❌ SEM skills OlivIA no contexto
  • ❌ SEM HEARTBEAT de Controller

⚖️ OlivIA (~5KB)

  • 📄 SOUL_OLIVIA.md — persona especializada
  • 📄 MEMORY_OLIVIA.md — teses + lições jurídicas
  • 📄 Vault Obsidian (carregado via tool, não no contexto fixo)
  • ❌ SEM MEMORY.md do JARBAS Core
  • ❌ SEM skills de distribuição EPROC
  • ❌ SEM heartbeat de monitoramento

🎛️ Controller (~5KB)

  • 📄 SOUL_CONTROLLER.md — persona operacional
  • 📄 MEMORY_CONTROLLER.md — estado tribunais + crons
  • 📄 HEARTBEAT_CONTROLLER.md — verificações EPROC
  • ❌ SEM contexto OlivIA
  • ❌ SEM MEMORY.md pessoal do Christian

📊 Economia estimada

  • Atual: ~20KB por req × todas instâncias
  • Novo: ~6KB Core | ~5KB OlivIA | ~5KB Ctrl
  • 💰 Redução ~60-70% tokens input
  • Heartbeats do Controller não tocam o Core
  • OlivIA processa sem bloquear DM do Christian

Problemas Específicos a Resolver

Checklist
  • P0
    Conflito de pastas: ~/jarbas/ vs ~/.openclaw/workspace/
    Claude Code deve auditar ambas as pastas e identificar arquivos duplicados. SOUL.md existe em ~/jarbas/SOUL.md e ~/.openclaw/workspace/SOUL.md. Definir ~/jarbas/ como source-of-truth para Node.js broker; ~/.openclaw/workspace/ apenas para OpenClaw context injection. Remover duplicatas.
  • P0
    Heartbeat interferindo na fila de tasks
    Heartbeat a cada 5min acorda OpenClaw e consome context window. No novo modelo: Core heartbeat só verifica DM + custo. Controller heartbeat verifica EPROC/intimações. OlivIA sem heartbeat (event-driven). Cada heartbeat usa < 1KB de contexto.
  • P0
    MEMORY.md pessoal do Christian carregado em grupos
    Regra D-012 existe mas tem falhas. No novo modelo: MEMORY.md só existe no workspace do Core (jarbas). OlivIA e Controller têm MEMORY próprio sem dados pessoais. Impossível vazar por arquitetura.
  • P1
    SYNC_CC_AGENTS e GROUP_CONTEXT_CHECKPOINT no mesmo processo
    Atualmente ambos crons disparam no Core. No novo modelo: GROUP_CONTEXT_CHECKPOINT vira cron por instância (cada instância salva seu próprio checkpoint). SYNC_CC_AGENTS permanece no Core mas isolado em worker separado, sem bloquear OpenClaw.
  • P1
    Claude semaphore monolítico (MAX_CONCURRENT=1 global)
    Quando OlivIA gera petição, JARBAS Core fica bloqueado. No novo modelo: cada instância tem seu próprio semáforo. OlivIA pode usar MAX_CONCURRENT=2, Controller MAX_CONCURRENT=3 (tarefas mais curtas). Core fica responsivo 100% do tempo.
  • P1
    ChromaDB nunca populado — 2 skills mortas
    Decidir: ativar ChromaDB (popular com processos Supabase) ou substituir por Supabase pgvector (já disponível). Recomendação: usar pgvector no Supabase — zero infra adicional, já temos extension disponível, mesma API.
  • P2
    skill_ciencia_renuncia.mjs STUB registrada
    Skill registrada mas retorna stub. Aguarda vídeo do João. Remover do start.mjs até estar pronta — evita confusion quando o skill_immune_system monitora tarefas dessa skill.
  • P2
    skill_archon.mjs não implementada
    MODO ARCHON (Opus 4.6) está documentado no SOUL.md mas a skill não existe. Claude Code deve criar a implementação: detecta trigger "archon:", chama claude-opus via API com contexto completo, retorna resposta premium.

Riscos e Mitigações

Risco Probabilidade Impacto Mitigação
OlivIA separada perde acesso ao Supabase compartilhado Média Alto Shared Skills Bus expõe API interna HTTP em localhost. Cada instância usa endpoint interno, não conexão direta. Alternativa: mesmas credenciais Supabase em cada instância via .env separado.
3 instâncias OpenClaw × heartbeats = 3× custo base Média Médio Controller e OlivIA sem heartbeat de 5min. Controller: heartbeat a cada 30min apenas. OlivIA: event-driven puro (só acorda quando grupo manda mensagem).
Browser pool duplicado: 3 instâncias abrindo Chromium Alta Alto Browser pool permanece centralizado no JARBAS Core (jarbas user). OlivIA e Controller pedem páginas via API interna. Mac Mini 16GB suporta 1 pool de até 3 tabs simultâneas.
Conflito de tokens Telegram (mesmo bot, 2 instâncias) Alta Alto Telegram não permite 2 pollers para o mesmo bot token. Solução: usar o mesmo poller centralizado no Core e fazer dispatch interno por chat_id para cada instância via IPC ou HTTP.
Migração de skills quebra tasks pendentes na fila Média Médio Antes de migrar: zerar fila jarbas_tarefas (processar ou marcar como erro). Fazer migração em horário de baixo volume (madrugada). Testar nova instância com 1 task antes de ativar cron.
