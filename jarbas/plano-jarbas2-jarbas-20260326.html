5 Erros e Armadilhas — O Que Não Deve se Repetir
❌ MEMORY.md desatualizado = agente desorientado
Após restart, a única âncora de contexto longo é o MEMORY.md. Se estiver desatualizado, o agente age como se fosse o primeiro dia. Regra: atualizar ao final de cada sessão importante. Tamanho máximo: ~6000 chars (acima disso custa muito por call).
❌ Workspace context > 20KB = flood de sendMessage
Descobri isso da pior forma. Arquivos grandes no workspace context aumentam exponencialmente o custo e podem travar o gateway. Manter SOUL.md, MEMORY.md, AGENTS.md enxutos. Referências detalhadas ficam em arquivos externos (~/jarbas/docs/).
❌ Credentials em arquivos de identidade públicos
Token Telegram exposto no GitHub Pages da JANA. Senha sudo no SOUL.md. Credenciais só no .env — nunca em arquivos que podem ir pro repo ou ser expostos.
⚠️ "Mental notes" não sobrevivem a restarts
Qualquer informação importante que o agente "lembrará" mas não escreveu → perdida no próximo restart. Regra: se é importante, escreve no MEMORY.md ou num arquivo memory/YYYY-MM-DD.md agora.
⚠️ Dizer "não posso" sem verificar
Primeira instância: verificar se existe skill ou se Claude Code resolve. "Não posso" é último recurso — antes: skill, cc:, exec direto, sessão isolada.
⚠️ Processar arquivos grandes inline
PDFs, ZIPs, vídeos >1MB ou >50k tokens: usar sessions_spawn modo run (sessão isolada que descarta contexto). Inline contamina o context window e encarece todas as calls seguintes.
💡 SUPABASE_KEY ≠ SUPABASE_SERVICE_KEY
Nome correto da variável é SUPABASE_KEY (service_role). Parece bobagem mas causou quebra real. Documentar o nome exato no .env e no TOOLS.md.
💡 Isolamento de contexto por grupo
Sem isolamento, o agente mistura contexto de grupos diferentes. Memory files por grupo (group-[chat_id].md) resolvem isso. No Jarbas 2.0 (só DM) não é problema — mas documentar se grupos forem adicionados.
