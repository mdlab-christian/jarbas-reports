Análise de Viabilidade

✅ VIAVEL — Justificativas técnicas
🧠
Context window folgado Claude suporta até 64K tokens de saída. 15 páginas equivalem a ~8K tokens — margem de 8x sobre o necessário.
📐
Estrutura jurídica previsível Intro → Dos Fatos → Do Direito → Dos Pedidos. Claude navega essa estrutura sem ancoragem por seção.
📦
Todos os dados disponíveis no Step 2 OCR docs, tópicos aprovados, Smart Blocks, jurisprudências, instruções do modelo — tudo pronto antes da geração.
🎯
Input confortável (~30-40K tokens) Mega-prompt com tudo montado cabe bem no Sonnet 4.6 e Opus 4.7 sem truncagem.
✍️
Coesão narrativa superior Claude escreve sabendo o que vem antes e depois — sem repetições entre seções, sem inconsistências de linha argumentativa.
⚠️ Riscos e Mitigações
⏳
Sem progresso granular Mitigação: streaming SSE no Mac Mini, ou progress bar determinístico baseado em tempo médio histórico. UX: "Redigindo petição completa..."
💥
Falha total em caso de erro Mitigação: retry automático silencioso 1x antes de fallback para modo tópicos. O usuário não percebe na maioria dos casos.
🔭
Qualidade não comprovada ainda Mitigação: shadow mode lado-a-lado com modo atual por 3-5 dias. Critério objetivo: nota média ≥ 8.5/10 pelo score_revisor.
🔌
Timeout em conexões longas Mitigação: timeout estendido no Mac Mini + Edge Function keepalive. Janela de 60-90s é previsível e gerenciável.

Plano de Implementação — 6 Fases

0
Fase 0 — Backup e Feature Flag  P0  2h
Zero risco. Nenhuma funcionalidade atual alterada. Revertível com um único comando.
  • Renomear gerador atual no Mac Mini para gerador-topicos.mjs (preservado como fallback permanente)
  • Feature flag MODO_GERACAO TEXT CHECK ('topicos','unico') na tabela olivia_historico
  • Toggle UI na Step 2 — "Modo Avançado (Beta)" visível apenas para perfil admin
1
Fase 1 — Backend Mac Mini: gerador-unico.mjs  P0  1 dia
Novo arquivo paralelo ao gerador-topicos. Mesma estrutura de módulo, megaprompt diferente.
  • Novo gerador-unico.mjs no Mac Mini (user olivia), sem alterar arquivos existentes
  • Seção CONTEXTO: processo, réu, tipo de ação, modelo selecionado
  • Seção DOCUMENTOS: textoOcr + resumos de todos os documentos
  • Seção TÓPICOS: lista aprovada pelo usuário + considerações preenchidas
  • Seção BLOCOS: conteúdo HTML completo dos Smart Blocks selecionados
  • Seção JURISPRUDÊNCIAS: ementas + tribunais das jurisprudências vinculadas
  • Seção INSTRUÇÕES: instrucoes_modelo do template + instruções customizadas
  • Instrução final: "Gere petição HTML completa, coesa, sem repetições, estrutura Fatos → Direito → Pedidos"
  • Temperature respeitada (alta=0.2, equilibrada=0.5, livre=0.7)
2
Fase 2 — Edge Function: olivia-gerar-v2  P1  4h
Cópia de olivia-gerar com ajuste mínimo de sinalização. Reutiliza todo código de montagem de peticao_referencia.
  • Nova EF olivia-gerar-v2 no Supabase — cópia da atual com sinalização ajustada
  • UPDATE historicoId: modo_geracao='unico', etapa_atual='geracao_unica'
  • Mesma estrutura de peticao_referencia reutilizada integralmente
  • Mac Mini detecta etapa_atual='geracao_unica' e roteia para gerador-unico.mjs automaticamente
3
Fase 3 — Frontend: Toggle de Modo  P1  2h
Feature-flagged para admin. Usuário comum não vê diferença. Loading state diferenciado no Step 3.
  • useOliviaWizardV3.ts: adicionar estado modoGeracao: 'topicos' | 'unico'
  • Componente ModoGeracaoToggle no Step 2 — visível apenas para perfil admin
  • olivia.service.ts: chamar olivia-gerar-v2 quando modo='unico'
  • UX Step 3 modo único: "Redigindo petição completa..." com progress bar baseado em tempo médio histórico
4
Fase 4 — Shadow Mode  P2  3-5 dias
Validação empírica com petições reais. Critério de aprovação objetivo antes de qualquer rollout.
  • Gerar 10-20 petições reais em modo 'unico' lado-a-lado com modo atual
  • Comparar: tempo total de geração, qualidade (score_revisor), coerência narrativa
  • Critério de aprovação: nota média ≥ 8.5/10 E tempo ≤ 60% do modo atual
  • Documentar resultados em tabela de comparação no vault antes de avançar para Fase 5
5
Fase 5 — Rollout Gradual  P2  1 semana
Progressivo, com fallback automático a cada etapa. Zero downtime.
  • 5a: Admin + Christian — validação final em produção real
  • 5b: Advogados internos (Heloísa e equipe)
  • 5c: Modo 'unico' passa a ser o padrão para todos os usuários
  • Modo 'topicos' mantido como fallback automático: retry silencioso 1x se falha

// gerador-unico.mjs — estrutura do mega-prompt montado pela Edge Function const megaPrompt = ` == CONTEXTO DO PROCESSO == Processo: [processo.numero_cnj] Tipo: [processo.tipo] | Subtipo: [processo.subtipo] Reu: [processo.reu_nome] Modelo de peticao: [modelo.titulo] == DOCUMENTOS ANALISADOS == [documentos.map(d => nome + textoOcr).join('\n\n')] == TOPICOS APROVADOS PELO USUARIO == [topicos.map((t, i) => n + titulo + consideracoes).join('\n')] == SMART BLOCKS SELECIONADOS == [blocos.map(b => titulo + conteudo_html).join('\n\n')] == JURISPRUDENCIAS == [juris.map(j => tribunal + numero + ementa).join('\n\n')] == INSTRUCOES DO MODELO == [modelo.instrucoes] == INSTRUCAO FINAL == Gere a peticao juridica completa em HTML. Seja coeso, sem repeticoes entre secoes. Estruture: Dos Fatos → Do Direito → Dos Pedidos. Fidelidade: [fidelidade] (temperature: [temperature]) `; // 1 chamada — HTML completo retornado de uma vez const response = await claude.messages.create({ model: modelo_claude, // sonnet-4-6 ou opus-4-7 max_tokens: 16000, // folga 2x — 15 pags = ~8K tokens temperature: temperature, // alta=0.2 | equilibrada=0.5 | livre=0.7 messages: [{ role: 'user', content: megaPrompt }] }); // Mac Mini detecta etapa_atual='geracao_unica' e roteia aqui return { peticao_html: response.content[0].text, modo: 'unico' };

Recomendação do Jarbas

🐙
RECOMENDO: Implementar modo único como padrão futuro, com rollout gradual.

Implementar o modo único não é uma aposta — é a arquitetura correta para o problema. Petições jurídicas são documentos com coerência narrativa obrigatória. Fragmentar a geração em N chamadas independentes é o que cria inconsistências que já observamos. A solução é dar ao Claude o quadro completo de uma vez e deixar ele escrever o argumento inteiro. O único trade-off real (sem progresso granular) tem mitigação técnica clara com streaming SSE.

Jarbas 2.0 · MacBook Pro · 20/04/2026
1
Qualidade: Claude com visão total do documento gera texto mais coeso, sem repetições e sem inconsistências entre seções. A argumentação jurídica flui como um documento único — porque é.
2
Velocidade: Redução de ~40-60% no tempo de geração. Para o Opus 4.7, que é mais lento por chamada, o ganho percentual é ainda maior — N chamadas lentas vs 1 chamada única.
3
Simplicidade: Backend mais simples, menos pontos de falha, mais fácil de debugar. O gerador atual tem complexidade de estado que não agrega valor ao produto.
4
15 páginas é trivial: 8K tokens de saída contra 64K de limite — menos de 13% da capacidade do Claude. Volume de output não é um desafio aqui.
5
Backup intacto: O gerador atual fica preservado como gerador-topicos.mjs e funciona como fallback automático. Zero risco operacional.
Trade-off real (unico)
Sem feedback granular de progresso durante geração. Mitigação recomendada: streaming SSE no Mac Mini (nativo do Claude SDK) ou progress bar determinístico baseado em tempo médio histórico das últimas petições. Ambos implementáveis na Fase 3 sem bloquear o rollout.
