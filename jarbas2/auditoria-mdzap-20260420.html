A auditoria identificou 9 falhas P0, sendo 5 delas relacionadas a ausência de organizacao_id em queries/updates — permitindo, em teoria, leitura ou escrita de dados de outras organizações via service role. Os dois casos mais críticos são o MdzapConversaBulkBar (L89-99) e o endpoint retry-media no webhook, ambos sem qualquer validação de ownership.

Os 14 bugs P1 concentram-se em race conditions nos hooks useMdzapMensagens e useMdzapSendMessage, além de falhas de feedback ao usuário quando operações assíncronas falham silenciosamente.

A Fase 1 (Segurança) deve ser prioridade absoluta — estimativa de 2 a 3 dias — antes de qualquer deploy em produção. 6 issues foram resolvidos nesta sessão, incluindo os P0 de TemplatesManagerDialog e o sdrMode hardcoded no desktop.

Camada 7 · Backend Webhook

2 P0 5 P1 2 P2
mdzap-webhook/index.ts 585 linhas · CRÍTICO
P0 Bug L275-282: Retry mídia sem validação auth
POST ?action=retry-media + mensagem_id. SELECT sem organizacao_id. Atacante com UUID consegue fazer trigger de download de mídia de qualquer organização.
P0 Bug L287-291: SELECT mensagem sem organizacao_id
supabase.from('whatsapp_mensagens').select(...).eq('id', mensagem_id).maybeSingle() — service role lê mensagem de qualquer organização.
P1 UX L426: contato_nome normalization backward
.replace(/[.\s]/g, '') remove pontos e espaços, verifica .length > 0. "." → "" → false → usa número. Lógica invertida, nomes com apenas pontos são ignorados.
P1 Bug L449-472: Media download failure silencioso
Se downloadAndStoreMeta falha, mensagem entra com mídia vazia sem indicação. Frontend renderiza bolha vazia, sem botão de retry.
P1 Logic L400-405: numero not found retorna 200
Deveria retornar erro (4xx). Status 200 faz a Meta acreditar que o webhook foi processado com sucesso. Pode gerar loops de retry desnecessários.
P1 Perf Sem circuit breaker para n8n
Se n8n está down, retryFetch tenta 3x com backoff. Webhook demora 6 segundos. Frontend fica pendurado aguardando.
P2 Arch L551-577: DLQ sem processamento
DLQ insert sem background job de processamento. Mensagens perdidas após 24h sem nenhuma tentativa de reprocessamento.
P2 Logic L536-537: Mode check com null/undefined
Se modo está null/undefined, assume Julia ativa. Conversa nova sem modo → Julia dispatcher ativa. Comportamento pode ser intencional, mas não está documentado.

Recomendações por Fase

1
Segurança
2–3 dias
  • Adicionar organizacao_id em TODOS os queries/updates sem filtro de org
  • Validar ownership de conversa/mensagem em todas as operações
  • Remover uso de service_role desnecessário em retry mídia
  • Corrigir endpoint retry-media (L275-282 e L287-291)
2
Bugs Críticos
3–5 dias
  • Corrigir QUERY_KEY em useMdzapConversas
  • Implementar retry logic no webhook
  • Adicionar UI feedback para falha de mídia
  • Cancelar promises em effect cleanup ao trocar conversa
  • Corrigir race condition no envio (limpar texto após confirmação)
3
UX
2–3 dias
  • Adicionar Error Boundaries nos componentes principais
  • Toast de sucesso somente após webhook confirmar
  • Botão retry na bolha de mensagem com falha
  • AlertDialog de confirmação para bulk archive
  • Tooltip nos badges CRM e placeholder de conversa vazia

Performance
2–3 dias
  • Debounce na busca de ConversaList
  • Memoizar conversaByCliente com useMemo
  • Batch realtime updates: 1 refetch/segundo máximo
  • Configurar lifecycle TTL no bucket S3
  • Implementar circuit breaker para chamadas n8n
