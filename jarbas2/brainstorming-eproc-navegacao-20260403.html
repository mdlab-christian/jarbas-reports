Issues Encontrados

CRITICO
I-01: page.goto() em skill_ler_documento.mjs

Linha ~39: await page.goto(docHref, { waitUntil: 'domcontentloaded', timeout: 30000 })

INCONSISTENTE com o padrao window.location.href usado em TODOS os outros skills. O page.goto() reinicializa o contexto da pagina, matando cookies de sessao do EPROC. O fallback (newPage) funciona, mas o path principal esta errado.

CRITICO
I-02: Browser novo por OAB

Cada OAB abre um browser Chromium completamente novo. RS116571 e RS131163 sao ambos TJRS — poderiam reusar o mesmo browser com re-login.

Overhead: ~15-30s por launch de browser. Com 5 OABs, sao ~1.5-2.5min gastos so com browser launch.

WARNING
I-03: Processamento sequencial de OABs

Loop sequencial: RS116571 → RS131163 → SC074025 → SC075466 → SP535101. Delay de 4.5-5.5 min entre cada. Total: ~35-45 min.

TJRS, TJSC e TJSP sao tribunais independentes — poderiam rodar em paralelo.

WARNING
I-04: Documentos quase nao lidos

documentos_lidos = 0 na maioria dos ciclos. Apenas 1 doc lido no ciclo SP535101 manual. Os eventos sao classificados pelo TEXTO do evento, nao pelo conteudo do documento.

Para classes E/F, o texto do documento e critico para analise precisa (sentencas, decisoes, depositos). Sem ler o doc, a IA classifica apenas pelo titulo do evento.

WARNING
I-05: Vision L4 restrito a docs criticos

Se isDocCritico = false, Vision L4 e pulado. Docs nao-criticos onde L1/L2/L3 falharam retornam success: false — dados perdidos.

WARNING
I-06: Prompt Sonnet 15-25K tokens

Sistema carrega inline: 14 arquivos do vault + 60 playbooks + 70 regras + equipe. A maior parte e redundante para classificacoes simples.

Propostas de Otimizacao

O7: Corrigir page.goto() em skill_ler_documento.mjs
P0 — CRITICO

Atual: await page.goto(docHref, { waitUntil: 'domcontentloaded', timeout: 30000 })

Proposta: Usar page.context().newPage() para abrir doc em nova aba, preservando sessao na pagina principal.

// ANTES (mata sessao): await page.goto(docHref, { waitUntil: 'domcontentloaded', timeout: 30000 }); // DEPOIS (preserva sessao): const docPage = await page.context().newPage(); await docPage.goto(docHref, { waitUntil: 'domcontentloaded', timeout: 30000 }); const result = await extractText(docPage, cnj, isDocCritico); await docPage.close();
Pro
Elimina morte de sessao no path principal. O fallback ja usa newPage() — so falta o path principal.
Risco
Minimo. O fallback ja funciona com newPage(). Basta mover a logica.
O3: Reduzir Prompt Sonnet
P0 — ALTO IMPACTO

Atual: ~15-25K tokens (vault 14 arquivos + 60 playbooks + 70 regras + equipe)

Proposta: Prompt tiered — carregar contexto relevante por classe, nao vault inteiro.

TierQuandoTokensEconomia
CompactClasses B, C, G (rotina)~5K-80%
StandardClasses D (prazo simples)~10K-50%
FullClasses E, F (estrategica)~20Katual

Estimativa: -40-60% tokens input Sonnet → economia de ~$0.30/dia ($9/mes).

O4: Structured Outputs com tool_use
P1 — CONFIABILIDADE

Atual: Resposta JSON livre, parsing com parseJsonArrayRobusto() (3-stage fallback: regex → partial → line-by-line).

Proposta: Usar tool_use da API Anthropic com schema definido. JSON garantido sem parsing.

// tool_use schema para classificacao tools: [{ name: "classificar_intimacao", input_schema: { type: "object", properties: { classe: { type: "string", enum: ["A","B","C","D","E","F","G"] }, acao: { type: "string", minLength: 20 }, urgente: { type: "boolean" }, responsavel: { type: "string", nullable: true }, confianca: { type: "number", minimum: 0, maximum: 1 }, }, required: ["classe","acao","urgente","confianca"] } }]
Pro
Elimina erros de parse JSON. Schema enforced pela API. Validacao automatica. Sem fallback 3-stage.
Contra
tool_use nao suporta batch nativo (1 tool call por intimacao). Pode precisar adaptar para multiple tool calls ou manter batch com JSON.
O1: Paralelismo por Tribunal
P1 — ALTO IMPACTO

Atual: Sequencial: RS→RS→SC→SC→SP com 5min delay entre cada. Total: ~35-45 min.

Proposta: 3 threads paralelas, uma por tribunal:

// Paralelo por tribunal (Promise.all) await Promise.all([ executarTribunal('TJRS', [RS116571, RS131163]), // sequencial entre si executarTribunal('TJSC', [SC074025, SC075466]), // sequencial entre si executarTribunal('TJSP', [SP535101]), // sozinha ]);

Tempo estimado: 35-45min → 12-15min. Bottleneck passa a ser o tribunal mais lento (TJRS com RS131163: 504s + oabDelay).

Pro
Reducao de 60-65% no tempo total. Cada tribunal e independente — sem conflito de sessao.
Risco
3 browsers Chromium simultaneos: ~1.8GB RAM. Mac Mini M4 tem 16GB — viavel mas monitorar.
O5: Haiku-Only para Classes A/B/G
P1 — CUSTO

Atual: Haiku faz triagem, Sonnet analisa TODOS D/E/F. Mas A/B/G ja sao simples — nao precisam de Sonnet.

Proposta: Se Haiku classifica como A, B ou G com confianca >0.85 → aceitar sem Sonnet.

Dados reais: B tem confianca 0.98, G tem 0.90. Sao classes claras que Haiku resolve bem. Economia: ~30% menos chamadas Sonnet.

O6: Selecao Inteligente de Documentos
P2 — QUALIDADE

Atual: documentos_lidos = 0 na maioria dos ciclos. IA classifica apenas pelo titulo do evento.

Proposta: Ler documento apenas quando a classificacao depende do conteudo:

ClasseLer Doc?Razao
A, B, GNaoTitulo do evento e suficiente
COpcionalVerificar quais docs faltam
DSimPrecisar prazo e acao especifica
ESim — PrioritarioSentenca/decisao: precisa texto completo
FSim — PrioritarioDeposito/alvara: validar valor
O2: Browser Reuse Entre OABs do Mesmo Tribunal
P2 — TEMPO

Atual: Browser novo por OAB. RS116571 fecha browser, RS131163 abre novo browser no mesmo TJRS.

Proposta: Manter browser aberto, fazer logout/login rapido com nova OAB no mesmo contexto.

Economia: ~15-30s por switch (browser launch). Risco: sessao contaminada entre OABs (cookies compartilhados).

O8: Expandir Vision Fallback
P3 — DADOS

Atual: Vision L4 so para docs criticos (isDocCritico = true). Docs nao-criticos com L1/L2/L3 falhando → dados perdidos.

Proposta: Habilitar Vision para TODOS os docs onde os 3 primeiros niveis falharam. Custo: Haiku Vision ~$0.01/imagem.
