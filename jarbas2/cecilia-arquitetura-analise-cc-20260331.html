Impacto no Enrichment — Playwright Continua Necessario

A promessa vs a realidade
Promessa da Split: "Node.js vira coletor puro, muito mais simples"

Realidade: O Node.js CONTINUA precisando do Playwright para:

  • Login no EPROC (Keycloak)
  • Gerar planilha XLS (navegar no EPROC)
  • Acessar pagina do processo (enrichment: juiz, vara, advogado_reu)
  • Ler documentos PDF (conteudo para analise)
  • AI Vision fallback (screenshots de telas travadas)

O PRD v5 adiciona extractProcessMetadata() que extrai juiz, vara, advogado_reu, AJG da pagina EPROC. Isso EXIGE browser. Se o Node.js "vira coletor puro", ele ainda faz 5 das 6 stages com Playwright. A unica coisa que sai e a chamada fetch() para a API do Claude — que e a parte mais simples de todo o pipeline.

🤔
A complexidade real do sistema esta no Playwright (login, anti-bot, DOM scraping, race conditions), nao na chamada de API do Claude. Mover a IA para n8n remove a parte simples e mantem a parte complexa.

Onde Divirjo do Jarbas

6 Divergencias Tecnicas
1. "Ciclo EPROC cai de 90min para 15-20min" (Jarbas)

Minha analise: Errado. O delay anti-bot entre processos (800-2500ms) domina o tempo. Stages 1-5 levam ~65min mesmo sem IA. O ganho real e ~25min, nao 70min. E o ciclo de captura pura AINDA precisa do Playwright aberto esse tempo todo.

2. "4 responsabilidades eliminadas do Node.js" (Jarbas)

Minha analise: So 1 responsabilidade sai: a chamada Claude API. As outras 3 que Jarbas lista (busca Supabase, retry, circuit breaker) continuam necessarias na captura. E com enrichment, MAIS responsabilidades entram no Node.js.

3. "n8n tem retry automatico, exponential backoff nativo" (Jarbas)

Minha analise: Verdade, mas parseJsonArrayRobusto() (ja no PRD) resolve o mesmo problema com ~20 linhas. E o retry do n8n so ajuda se o erro e transiente — para input sujo (10 dos 12 bugs), retry nao ajuda.

4. "Estimativa: 2-3 semanas de CC para migracao limpa" (Jarbas)

Minha analise: Subestimado. Precisa: novo schema (intimacoes_raw ou status_analise), novo workflow n8n completo, redesenhar notificacoes Telegram, testar E2E do zero, documentar. Realista: 3-5 semanas, com risco de regredir bugs ja corrigidos.

5. Jarbas nao explorou Opcao C (Hibrido)

Minha analise: Jarbas apresentou binario: manter ou splittar. A opcao hibrida — manter monolitico mas com pontos de extensao (status_analise + edge function para re-analise) — captura 80% do beneficio com 10% do custo.

6. "Plano B valido — e o Plano Definitivo para escala" (Jarbas)

Minha analise: Concordo que e o plano para escala, mas nao para o volume atual. Com 5 OABs e ~400 intimacoes/dia, o monolitico com bugs corrigidos e suficiente. A split vira necessidade com 15+ OABs — e esse dia pode nunca chegar, ou chegar em 6 meses. A Opcao C preserva a opcionalidade sem pagar o custo agora.

3 Concordancias com Jarbas
  • Arquitetura split e superior no longo prazo — sim, se o volume crescer 3x+
  • Training loop via frontend e o argumento killer — sim, implementavel em qualquer opcao
  • Nao fazer agora — sim, CC deve terminar os 12 fixes primeiro

Decisao de Prep — O Que o CC Pode Fazer AGORA

Migration SQL (adicionar ao PRD v5)
-- Prep para split futura: status_analise permite pipeline assincrono ALTER TABLE intimacoes ADD COLUMN IF NOT EXISTS status_analise text DEFAULT 'concluida' CHECK (status_analise IN ('pendente', 'processando', 'concluida', 'erro')); -- Indice para query rapida de pendentes (n8n ou edge fn) CREATE INDEX IF NOT EXISTS idx_intimacoes_status_analise ON intimacoes(status_analise) WHERE status_analise = 'pendente'; -- Tabela de feedback para training loop CREATE TABLE IF NOT EXISTS feedback_ia ( id uuid DEFAULT gen_random_uuid() PRIMARY KEY, intimacao_id uuid REFERENCES intimacoes(id) ON DELETE CASCADE, classe_original text, classe_corrigida text, motivo text, usuario_id uuid REFERENCES auth.users(id), organizacao_id uuid, created_at timestamptz DEFAULT now() ); CREATE INDEX IF NOT EXISTS idx_feedback_ia_org ON feedback_ia(organizacao_id, created_at DESC);

Custo: 1 migration. Risco: zero (default='concluida' = comportamento atual inalterado). Beneficio: pipeline pronta para split quando necessario.

Mudanca minima no orquestrador
// Em persistirIntimacao() — ANTES da analise IA: status_analise: 'pendente' // novo campo // APOS analise IA bem-sucedida: await supabase.from('intimacoes') .update({ classe_cecilia: r.classe, confianca_ia: r.confianca, analise_ia: r.analise, status_analise: 'concluida' // novo }) .eq('id', intimacaoId); // Se analise IA falhar: await supabase.from('intimacoes') .update({ status_analise: 'erro' }) .eq('id', intimacaoId);
