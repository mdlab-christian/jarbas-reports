✅ Prós da Arquitetura Split

1. Node.js muito mais simples e estável

Hoje o Node.js faz: login Playwright, download XLS, parse, busca Supabase, análise IA, retry, circuit breaker. São 6 responsabilidades em um processo.

Na split, o Node.js faz apenas: login → download → parse → INSERT raw. 4 responsabilidades eliminadas. Ponto de falha principal (IA) sai do Playwright.

Benefício: ciclo EPROC de 90min → 15-20min. Sem Haiku → sem timeout de 10.3s/processo. Playwright não fica aberto enquanto a IA pensa.

2. n8n = retry nativo, filas, observabilidade

n8n tem retry automático, exponential backoff, queue management, webhook triggers e dashboard de execuções nativo. Tudo que a gente está tentando reimplementar no Node.js manualmente.

Uma análise IA falha? n8n retenta automaticamente. Sem contaminar batch inteiro. Sem `parseJsonArray` frágil.

3. Treinamento via Playbooks vira realidade

Hoje o vault Obsidian da Cecília precisa ser sincronizado via SSH. Com n8n + Supabase, o João edita no frontend → Supabase atualiza → n8n lê o playbook na próxima análise.

Loop de treinamento: João classifica errado → feedback vai pro Supabase → n8n usa o histórico de correções como few-shot examples. Sem tocar código.

4. Escala independente por camada

Hoje se SC074025 tem 392 processos e trava o ciclo de 69min, todas as outras OABs esperam. Com split:

  • Node.js coleta todas as OABs em paralelo (sem IA travando)
  • n8n analisa o batch inteiro em paralelo (múltiplos workers)
  • OABs grandes não bloqueiam as menores

5. Observabilidade real

n8n tem dashboard de execuções, logs por nó, histórico de runs, alertas. Hoje para saber o que aconteceu num ciclo precisa ir no Supabase e decodificar logs manuais.

Com n8n: Christian abre o painel → vê cada execução → click numa falha → vê o erro específico → retenta manual se quiser.

6. Modelos de IA facilmente intercambiáveis

No Node.js atual, trocar de Haiku para GPT-4o-mini exige alterar código. No n8n, é trocar um nó por outro. Dá pra testar modelos A/B em produção sem deploy.

❌ Contras e Riscos

1. Latência aumenta (pode ser problema)

Hoje: XLS capturado → análise IA → João vê no controller, tudo num ciclo de 90min.

Com split: XLS capturado → INSERT raw → trigger n8n → análise → UPDATE. São 2 passos assíncronos. O João abre o controller e vê intimações sem classificação por alguns minutos.

Mitigação: mostrar badge "Analisando..." no controller enquanto análise está pendente. O João já vê a intimação crua, IA classifica em background.

Impacto real: baixo. O ciclo atual já demora 90min. Adicionar 5-10min de processamento n8n não muda a experiência do João.

2. Complexidade de infraestrutura aumenta

Hoje: 1 processo Node.js + Supabase. Com split: Node.js + n8n + Supabase + possível queue (Redis ou Supabase Realtime como trigger).

Mais componentes = mais coisas para monitorar, mais pontos de falha de infra.

Mitigação: n8n já está rodando no Railway. O trigger pode ser simples: cron no n8n que faz SELECT em `intimacoes_raw` WHERE analisada = false.

3. Migração exige reescrita significativa

O skill_analisar_ia.mjs e skill_orquestrador.mjs precisam ser reescritos. O schema do Supabase precisaria de uma nova tabela intermediária (`intimacoes_raw`) ou um campo `status_analise` em `intimacoes`.

Estimativa: 2-3 semanas de CC para migração limpa. Hoje o CC está resolvendo os 12 bugs da v2 — não é o momento de mudar arquitetura.

4. n8n Railway tem custo e limites

A instância n8n no Railway é compartilhada. 300-600 análises IA/ciclo × 2 ciclos/dia = 600-1200 execuções/dia. Verificar se o plano Railway aguenta a carga.

Alternativa: n8n no Mac Mini (já instalado) rodando 24/7 — zero custo adicional de infra.

5. Problema de contexto: Node.js perde acesso ao resultado IA

Hoje o Node.js analisa e persiste num loop sincronizado. Com split, o Node.js não sabe se a análise IA foi bem-sucedida. O fluxo de notificação (Telegram → João) precisa ser redesenhado para disparar depois do n8n terminar.

Fluxo completo da proposta Split

LAYER 1 — CAPTURA (Node.js Cecília, 24/7)
══════════════════════════════════════════════════════════
Cron 00h, 06h, 12h, 18h BRT
  → Playwright: login EPROC (RS116571, RS131163, SC074025...)
  → Gerar planilha XLS por OAB
  → Parse XLS → lista de CNJs + evento bruto
  → Lookup MdFlow: processo_id por CNJ
  → INSERT intimacoes (status = 'pendente_analise', sem classe_cecilia)
  → Trigger: UPDATE cecilia_ciclos (status='captura_concluida')

LAYER 2 — ANÁLISE IA (n8n, assíncrono)
══════════════════════════════════════════════════════════
Trigger: SELECT * FROM intimacoes WHERE status='pendente_analise'
  (cron n8n a cada 5 minutos OU Supabase Realtime webhook)
  → Para cada intimação:
    ① Filtro zero-custo: polo=reu → Classe B automático
    ② Filtro ruído: RUIDO_PATTERNS → Classe A automático
    ③ Claude Haiku: classificação D/E/F/C/G com contexto MdFlow
    ④ Validação JSON robusta (n8n tem parse nativo)
    ⑤ UPDATE intimacoes SET classe_cecilia=X, confianca_ia=Y, status='nova'
  → Se falhar: retry automático n8n (3x com backoff)
  → Se falhar 3x: status='erro_analise', notificar Jarbas

LAYER 3 — CONTROLLER (Frontend MdFlow)
══════════════════════════════════════════════════════════
João abre /controller:
  → Ver intimações com status='pendente_analise' → badge "Analisando..."
  → Ver intimações com status='nova' → classificadas, prontas para ação
  → Playbooks: editar taxonomia no frontend → Supabase → n8n usa no próximo ciclo
  → Feedback: João discorda da classificação → INSERT feedback_ia → few-shot para n8n

🎯 Veredicto — O Que Fazer

Resposta direta: A ideia é boa, mas é Fase 2 — não agora

A arquitetura split é claramente superior no longo prazo. Node.js mais simples, n8n mais robusto, treinamento via frontend funcionando de verdade, ciclos mais rápidos, debug visual.

Mas tem um problema: o CC está consertando 12 bugs agora.

Fazer essa migração agora seria como trocar os pneus de um carro com o motor ainda em chamas. A Cecília v2.3 precisa ficar estável primeiro — 2 semanas de E2E limpos com fallback D abaixo de 40 — antes de mexer na arquitetura.

✅ O que fazer agora (Cecília v2.3)

  • CC terminar as 12 correções
  • E2E limpo: fallback D < 40, zero Classe A no banco
  • Estruturar intimacoes com campo status_analise (preparação para split)
  • Documentar schema atual antes de migrar

📅 O que fazer em 2-4 semanas (Fase 2)

  • Migrar análise IA para n8n (workflow dedicado)
  • Node.js vira coletor puro (muito mais simples)
  • Implementar Playbooks com loop de feedback
  • Dashbboard n8n como monitoramento principal

🔑 Decisão de prep que o CC pode tomar AGORA

Adicionar coluna status_analise em intimacoes:

ALTER TABLE intimacoes
ADD COLUMN IF NOT EXISTS status_analise text DEFAULT 'concluida'
  CHECK (status_analise IN ('pendente', 'processando', 'concluida', 'erro'));

-- Índice para o n8n fazer SELECT rápido depois
CREATE INDEX IF NOT EXISTS idx_intimacoes_status_analise
  ON intimacoes(status_analise) WHERE status_analise = 'pendente';

Isso não quebra nada hoje (default='concluida' = comportamento atual). Mas prepara o banco para a split quando chegar a hora.
