🔴 Problemas Identificados — Pontos de Quebra

P0 Circuit breaker sem reset automático

Situação atual: Circuit breaker desativa OABs após 6 falhas consecutivas. Reset é manual (SQL). Se o EPROC ficar 40 minutos fora durante um ciclo → todas as 5 OABs desativadas → ciclos seguintes rodando com 0 contas → silêncio total.

Impacto real: Ciclo das 12h roda, não processa nada, não alerta. Christian descobre horas depois que perdeu intimações urgentes.

Fix: Após desativar, agendar reset automático com backoff: 10min → 30min → 1h → alerta Telegram.

P0 Silêncio em caso de falha total do ciclo

Situação atual: Se o ciclo falha (0 contas ativas, erro de rede, EPROC fora), o sistema não notifica João nem Christian.

Impacto real: "Fire and forget" silencioso — exatamente o antipadrão do nosso sistema.

Fix: Ao finalizar ciclo com 0 processamentos ou erro crítico → notificar imediatamente com causa.

P1 Schema mismatch _analista_raw

Fix parcial aplicado hoje — skill_analisar_ia.mjs v2.2 agora gera _analista_raw wrapper.

Ainda precisa verificar: o normalizeAnaliseIA() no controller MdFlow está consumindo todos os campos? Especialmente responsavel_sugerido_nome e resumo_controller. Testar um ciclo real e checar o JSON gravado em cecilia_intimacoes.

P1 Cache de sessão 14min pode expirar no meio do ciclo

Situação: 5 OABs processando sequencialmente. RS116571 pode ter 100+ processos. Se cada OAB leva 4-5 min → ao chegar na 4ª OAB, o cache da 1ª já expirou.

Fix: Aumentar TTL do cache de sessão para 25-30 min. Ou implementar renovação silenciosa quando TTL < 3 min.

P1 Validar formato XLS por tribunal

Risco: TJRS, TJSC e TJSP podem gerar planilhas com cabeçalhos diferentes. O parser atual foi validado com RS116571 (TJRS). SP535101 e SC074025 não testados ainda.

Fix: Rodar skill_gerar_planilha para cada tribunal no modo debug + logar os cabeçalhos reais antes de ativar produção completa.

P1 Filtro de polo pode descartar processos legítimos

Situação: Processos onde OAB é polo passivo técnico mas somos advogados do autor. Polo passivo → Classe B automático → nunca analisa IA.

Fix: Validar com João: o filtro de polo da planilha XLS é confiável? Se sim, documentar explicitamente. Se não, substituir por validação mais granular no texto.

P2 Sem monitoramento de custo IA por ciclo

Em produção plena: 5 OABs × 3 ciclos/dia × N intimações = potencialmente centenas de análises/dia. Sem monitoramento, custo pode explodir silenciosamente.

Fix: Registrar tokens usados em cecilia_custo_ia por ciclo. Alertar se custo/dia > $5.

P2 Responsável sugerido pode ser nome inválido

Situação: A IA pode inventar nomes para responsavel_sugerido_nome se não souber a lista válida. O arquivo equipe-responsaveis.md agora está no vault (fix v2.2), mas o prompt deve reforçar os nomes exatos.

Fix: Adicionar ao system prompt: "Responsáveis válidos: Eloisa, Leonardo, Dr. Christian, João. NUNCA inventar nome fora desta lista."

🔄 M2 — Circuit Breaker com Reset Automático + Backoff

Substituir o circuit breaker atual por versão com backoff inteligente:

// accounts-loader.mjs — circuit breaker v2
// Ao desativar OAB:
// 1. Gravar jarbas_accounts.circuit_reset_at = now + backoff_ms
// 2. Backoff: 1ª falha=10min, 2ª=30min, 3ª=1h, 4ª=2h → alerta humano
// 3. Ao iniciar ciclo: checar se circuit_reset_at < now → resetar automaticamente
// 4. Se OAB desativada → notificar Telegram imediatamente

async function checkCircuitReset(accounts) {
  const now = new Date();
  for (const acc of accounts) {
    if (!acc.ativo && acc.circuit_reset_at && new Date(acc.circuit_reset_at) < now) {
      await supabase.from('jarbas_accounts')
        .update({ ativo: true, login_falhas_consecutivas: 0, circuit_reset_at: null })
        .eq('login_eproc', acc.login_eproc);
      notify(`🔄 OAB ${acc.login_eproc} — circuit breaker resetado automaticamente`);
    }
  }
}
