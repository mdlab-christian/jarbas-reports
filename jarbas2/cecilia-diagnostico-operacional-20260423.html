MECANISMO FUNCIONA — MAS COM 3 FALHAS CRÍTICAS QUE ANULAM O VALOR ENTREGUE

O pipeline técnico (login → scrape → persist → análise IA) está operacional. Mas 3 problemas críticos fazem com que a Cecilia capture lixo em vez de prazos reais e analise quase nada de substancial:

01
S3 não filtra ruído — 80%+ dos eventos capturados são eventos de cartório/expediente que o próprio João Victor disse "jogar fora sem analisar". O playbook eventos-ruido.md existe e está bem escrito — mas o S3 não está aplicando as regras.
02
S5 não lê documentos — apenas 1 de 619 intimações tem conteúdo real do documento (0,16%). Sem documento, a Cecilia classifica baseada só no título — quase tudo vira Classe A (ciência simples, sem ação).
03
Confiabilidade técnica baixa — RS116571 em 9% de sucesso; ciclos falham por erros Playwright (page.evaluate: Execution context was destroyed). Fix parcial aplicado em 22/04.
Resposta direta sobre treinamento: SIM, treinamento novo resolve os problemas de qualidade (S3 + classificação). Mas NÃO resolve o S5 (leitura de documentos) que é problema técnico. Prioridade: corrigir S3 e S5 no código ANTES de escrever playbooks, caso contrário o treinamento vai ser aplicado sobre dados errados.

S5 falha na leitura de documentos — bug de código (navegação Playwright, iFrame EPROC, timeout PDF). Treinamento não muda como o código acessa o DOM.
Confiabilidade técnica 41% — estabilidade Playwright + EPROC session é problema de infraestrutura, não de treinamento
RS116571 com 9% — problema de sessão/credencial específica deste OAB; precisa investigação técnica independente
Classificação sem substância — se S5 continua quebrado, você vai treinar a Cecilia para classificar lixo melhor. Prazos reais continuam invisíveis.
Conclusão: Treinamento novo é necessário e vai ter impacto real — mas só APÓS corrigir S3 e S5 no código. Caso contrário, o treinamento vai ser aplicado sobre dados errados e os prazos reais vão continuar passando invisíveis.

Conclusão Final

O mecanismo da Cecilia é sólido. O problema não é a IA — é o que ela está recebendo para analisar.
→
Resolvendo S3 (filtro de ruído): a Cecilia passa a processar ~120 eventos relevantes em vez de 619 lixos. Redução de 80% no custo de IA e aumento massivo de signal-to-noise.

→
Com treinamento novo (Playbooks 3+4+5): a Cecilia vai saber exatamente o que fazer com cada tipo de evento — qual atividade criar, qual prazo, quem notificar.
