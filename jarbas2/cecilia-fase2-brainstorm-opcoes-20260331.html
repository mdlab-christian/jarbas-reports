Por Que n8n Nao E a Resposta

n8n e uma ferramenta de orquestracao visual, nao de inteligencia
  • Para analise IA, tu ta encapsulando um fetch("api.anthropic.com") de 3 linhas em 12 nodes visuais. A inteligencia vem do Claude, nao do n8n.
  • Cada node e rigido — nao "pensa", nao se adapta. Para trocar a logica, tu edita JSON num editor visual em vez de editar codigo.
  • 100 workflows ja rodando — o n8n do MdFlow ja ta lotado de orquestracao operacional (distribuicao, CRM, parceiros). Misturar analise IA ali polui o sistema.
  • Sem acesso a Obsidian — n8n nao le cecilia-vault. Precisaria migrar playbooks para Supabase primeiro.
  • Sem raciocinio — se o prompt precisa mudar, alguem precisa ir la editar manualmente. Nao aprende sozinho.

n8n e bom para: webhooks, CRUD, filas, automacoes operacionais (o que ja faz). n8n e ruim para: analise IA, raciocinio, adaptacao, Obsidian.

As 3 Opcoes Reais

⚡ Opcao 1: Supabase Edge Function
Funcao serverless no Supabase. Triggered por cron ou INSERT. Zero infra.
  • ✅ Zero infra — roda no Supabase, sem servidor
  • ✅ Event-driven — triggered por INSERT (status=pendente)
  • ✅ Mesmo datacenter — acesso Supabase nativo, ultra rapido
  • ✅ Auto-scale — 10 ou 1000 items, escala sozinho
  • ✅ TypeScript — deploy via supabase functions deploy
  • ❌ Timeout 150s — precisa processar item a item, nao batch grande
  • ❌ Sem Obsidian — nao le vault, playbooks precisam ir pra tabela
  • ❌ Sem raciocinio — executa template, nao pensa
  • ❌ Debug limitado — logs do Supabase, sem breakpoints
7.0
🧠 Opcao 2: Claude Code Agent
Agent CC com acesso MCP a Supabase + 4 vaults Obsidian. Raciocina, adapta, aprende.
  • ✅ Obsidian nativo — le cecilia-vault, jarbas-vault, mdlab-vault via MCP
  • ✅ Supabase nativo — query, update, migrate via MCP
  • ✅ RACIOCINA — entende feedback, adapta prompt, pensa sobre edge cases
  • ✅ Training loop real — le feedback do Joao, raciocina, gera few-shot
  • ✅ Pode chamar Haiku/Sonnet — como sub-tarefa via API
  • ✅ 68 agents como referencia — patterns ja validados
  • ⚠ Custo Opus — cada invocacao do agent usa Opus tokens (~$0.50-2 por run)
  • ⚠ Nao ideal para batch 400 — processar 1 a 1 com Opus e caro
  • ❌ Nao roda sozinho 24/7 — precisa trigger (cron, manual, ou skill)
8.5
📦 Opcao 3: Node.js Skill (Evoluido)
Evoluir o skill_analisar_ia.mjs existente. Adicionar Obsidian + feedback. Sem nova infra.
  • ✅ Ja existe — skill_analisar_ia.mjs (600 linhas) funcional
  • ✅ Zero custo infra — roda no Mac Mini como hoje
  • ✅ Batch eficiente — 10 items por chamada Claude, otimizado
  • ✅ Obsidian possivel — readFileSync("~/cecilia-vault/...")
  • ✅ CC supervisiona — CC escreve/corrige o script, Node.js executa
  • ❌ Nao raciocina — executa template, nao pensa
  • ❌ Atualizacao manual — mudar prompt = editar codigo = redeploy
  • ❌ "Ultrapassado" — script cron, sem observabilidade moderna
  • ❌ Acoplado a Cecilia — roda dentro do pipeline de captura

O CC Agent pode:

  • Ler playbooks do Obsidian: Read ~/cecilia-vault/playbooks/playbook-analise-processo.md
  • Entender a taxonomia A-G e as nuances que o Joao documentou
  • Comparar com feedback_ia no Supabase: "Joao reclassificou 5 items de C para E esta semana"
  • Raciocinar: "O prompt atual nao distingue tutela de despacho. Vou adicionar um few-shot example."
  • Atualizar cecilia_config.prompt_system no Supabase com o prompt melhorado
  • Edge Function le o prompt atualizado na proxima execucao — sem deploy, sem codigo
✅
Loop completo: Joao edita playbook no Obsidian → CC Agent le via MCP → CC Agent atualiza prompt no Supabase → Edge Function usa prompt novo → classificacao melhora → Joao ve resultado melhor. Tudo automatico, 1x/dia.
Alternativa: Frontend edita playbooks direto no Supabase

Para o futuro, o Joao poderia editar playbooks no frontend MdFlow (aba "Treinamento") em vez do Obsidian. Os dados iriam para cecilia_config direto. Mas isso e Fase 3 — por agora, Obsidian + CC Agent funciona perfeitamente.

Resumo da Decisao

PerguntaResposta
n8n?NAO. Orquestracao visual pra API call e overengineering. Sem Obsidian, sem raciocinio.
So Edge Function?Quase. Excelente pra execucao, mas nao raciocina e nao le Obsidian.
So CC Agent?Melhor em inteligencia, mas caro demais pra batch 400/dia.
So Node.js?Funciona, mas "ultrapassado" e acoplado a Cecilia.
Edge Fn + CC Agent?SIM. Edge Fn executa barato 24/7. CC Agent pensa 1x/dia com Obsidian. ~$113/mes.
Conecta com Obsidian?SIM. CC Agent le 4 vaults via MCP e sincroniza para Supabase.
Training loop?SIM. Joao da feedback → CC Agent raciocina → atualiza prompt → Edge Fn usa.
Fase 1 precisa mudar?Minimo: INSERT com status_analise='pendente', remover chamada analise_ia.
Pode comecar agora?SIM. Migration + Edge Fn + teste isolado. Conectar depois.

Arquitetura final: Edge Function (execucao 24/7) + CC Agent (cerebro 1x/dia) + Obsidian (playbooks) + Supabase (tudo conectado). Zero n8n pra isso. Zero Node.js novo. Moderno, barato, inteligente.
