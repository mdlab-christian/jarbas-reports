Especificacao node-a-node
1. Schedule Trigger — Cron 5min

Tipo: n8n-nodes-base.scheduleTrigger

Config: A cada 5 minutos, 24/7

Porque 5min: Fase 1 roda ciclos de ~90min. Com 5min, a Fase 2 pega os pendentes dentro de no maximo 5min apos captura. Para o Joao, e quase real-time.

Alternativa futura: Supabase Realtime webhook (zero latencia) — mas cron e mais simples para v1.

2. Fetch Pendentes — Supabase SELECT

Tipo: n8n-nodes-base.httpRequest (Supabase REST API)

GET {SUPABASE_URL}/rest/v1/intimacoes ?status_analise=eq.pendente &select=id,processo_id,tipo_evento,conteudo,data_intimacao, fonte,cnj,tribunal,signature_hash,evento_eproc, processos(id,numero_cnj,cliente_nome,empresa_nome, fase_processual,polo_representado,valor_causa, vara_distribuicao) &order=created_at.asc &limit=50 Headers: apikey: {{$credentials.supabaseApi.apiKey}} Authorization: Bearer {{$credentials.supabaseApi.apiKey}} Prefer: return=representation

Limit 50: Processar em batches de 50 para nao sobrecarregar o n8n. Se houver mais, o proximo ciclo de 5min pega o resto.

JOIN processos: O PostgREST do Supabase faz JOIN automatico via FK processo_id. Traz contexto MdFlow (empresa, fase, polo) sem query extra.

3. Tem items? — IF

Tipo: n8n-nodes-base.if

Condicao: {{ $json.length > 0 }}

Se nao tem pendentes, o workflow para (no-op). Evita processar Claude com 0 items.

4. Pre-Filter — Code (zero-cost classification)

Tipo: n8n-nodes-base.code

Objetivo: Classificar 40-60% das intimacoes SEM chamar Claude. Economia direta de tokens.

// Pre-filtros zero-custo (mesmo logic do Node.js atual) const RUIDO_PATTERNS = [ /^concluso[sa]?\s+(ao|para|de\s+)/i, /^redistribu[ií]/i, /^ato\s+ordinat[oó]rio/i, /^juntada?\s+de?\s+(peti[çc]|doc|certid|procura)/i, /^certid[aã]o\s+de\s+publica/i, /^termo\s+de\s+juntada/i, /^baixa\s+(definitiva|provis)/i, /ju[ií]zo\s+100%\s+digital/i, /^vista\s+ao\s+(minist|mp)/i, /^dados\s+complement/i, ]; const items = $input.all(); const needsIA = []; const autoClassified = []; for (const item of items) { const evento = (item.json.tipo_evento || '').trim(); const polo = item.json.processos?.polo_representado; // Auto-classe A: ruido if (RUIDO_PATTERNS.some(r => r.test(evento))) { autoClassified.push({ ...item.json, _classe: 'A', _confianca: 0.95, _motivo: 'Pre-filtro ruido (regex match)' }); continue; } // Auto-classe B: polo=reu → ciencia simples if (polo === 'reu') { autoClassified.push({ ...item.json, _classe: 'B', _confianca: 0.90, _motivo: 'Pre-filtro polo reu (ciencia automatica)' }); continue; } // Precisa IA needsIA.push(item.json); } return [ { json: { autoClassified, needsIA, stats: { total: items.length, auto: autoClassified.length, ia: needsIA.length }}} ];

Output: Dois arrays — autoClassified (vai direto pro UPDATE) e needsIA (vai pro Claude).

5. Build Prompt — Code (montar prompt com contexto)

Tipo: n8n-nodes-base.code

Objetivo: Montar prompt para Claude com: evento, contexto MdFlow, playbook, feedback historico.

// Buscar feedback recente para few-shot (se houver) const items = $input.first().json.needsIA; // Template do prompt (pode ser editado na tabela cecilia_config) const systemPrompt = `Voce e Cecilia, assistente juridica... Taxonomia: A=irrelevante, B=ciencia, C=operacional, D=juridico simples, E=estrategico, F=financeiro, G=encerramento. ...`; // Agrupar em batches de 10 para eficiencia const batches = []; for (let i = 0; i < items.length; i += 10) { const batch = items.slice(i, i + 10); const userPrompt = batch.map((item, idx) => { const ctx = item.processos || {}; return `[${idx+1}] CNJ: ${item.cnj || ctx.numero_cnj} Empresa: ${ctx.empresa_nome || 'N/A'} Fase: ${ctx.fase_processual || 'N/A'} Polo: ${ctx.polo_representado || 'autor'} Evento: ${item.tipo_evento} Conteudo: ${(item.conteudo || '').slice(0, 2000)}`; }).join('\n---\n'); batches.push({ systemPrompt, userPrompt, items: batch }); } return batches.map(b => ({ json: b }));
6. Claude Haiku — HTTP Request (Triage)

Tipo: n8n-nodes-base.httpRequest

POST https://api.anthropic.com/v1/messages Headers: x-api-key: {{$credentials.anthropicApi.apiKey}} anthropic-version: 2023-06-01 content-type: application/json Body: { "model": "claude-haiku-4-5-20251001", "max_tokens": 2000, "system": "{{$json.systemPrompt}}", "messages": [{ "role": "user", "content": "{{$json.userPrompt}}\n\nResponda em JSON array..." }] }

Custo: Haiku 4.5 — $0.80/MTok input, $4/MTok output. Para 400 intimacoes/dia com pre-filter 50%: ~200 items × ~1K tokens = ~200K tokens/dia = ~$1/dia.

Retry: n8n configurado para retry 3x com backoff exponencial (1s, 4s, 16s).

7. Parse Response — Code (validacao robusta)

Tipo: n8n-nodes-base.code

// Parse JSON robusto — resolve BUG-03 por design const response = $input.first().json; const text = response.content?.[0]?.text || ''; const batchItems = $input.first().json.items; const results = []; // Tentar parse do array completo try { const arr = JSON.parse(text.match(/\[[\s\S]*\]/)?.[0] || '[]'); for (let i = 0; i < arr.length; i++) { results.push({ intimacao_id: batchItems[i]?.id, classe: arr[i]?.classe || 'D', confianca: arr[i]?.confianca || 0.3, motivo: arr[i]?.motivo || 'Analise IA', prazo_dias: arr[i]?.prazo_dias || null, risco: arr[i]?.risco || 'medio', acao_sugerida: arr[i]?.acao_sugerida || '', }); } } catch { // Fallback: parse individual (nunca contamina batch) const objects = text.match(/\{[^{}]*\}/g) || []; for (let i = 0; i < objects.length; i++) { try { const obj = JSON.parse(objects[i]); results.push({ intimacao_id: batchItems[i]?.id, classe: obj.classe || 'D', confianca: obj.confianca || 0.3, motivo: obj.motivo || 'Parse individual', prazo_dias: obj.prazo_dias || null, risco: obj.risco || 'medio', }); } catch { /* skip broken item, nao contamina */ } } } return results.map(r => ({ json: r }));

Key insight: Se o parse do array falha, tenta parse individual. Cada item e independente. Isso resolve BUG-03 por design no n8n.

8. Switch D/E/F — Classe que precisa Sonnet

Tipo: n8n-nodes-base.switch

Regra: Se classe D, E ou F com confianca < 0.8 → enviar para Claude Sonnet para analise profunda. Senao → direto para UPDATE.

Racional: Haiku faz triagem rapida. Sonnet refina os casos complexos (decisoes judiciais, prazos fatais, valores). Economiza ~70% dos tokens Sonnet.

9. Claude Sonnet — HTTP Request (Analise Profunda)

Tipo: n8n-nodes-base.httpRequest

Modelo: claude-sonnet-4-6-20250514

Diferenca do Haiku:

  • Prompt mais detalhado com playbook completo do Joao
  • Inclui few-shot examples do feedback_ia
  • Pede: acao_sugerida detalhada, prazo_dias, risco_nivel, fundamentacao
  • 1 item por vez (nao batch) para maximizar qualidade

Custo: Sonnet — $3/MTok input, $15/MTok output. Para ~60 items D/E/F/dia × ~2K tokens: ~$2-3/dia.

Tipo: n8n-nodes-base.errorTrigger + n8n-nodes-base.code

Logica:

  • Se falhar 3x: Error Trigger captura
  • UPDATE intimacao com status_analise = 'erro'
  • INSERT em cecilia_erros com detalhes
  • Telegram para Jarbas: "Falha na analise de X intimacoes"

Migrations SQL — Pre-requisito

Migration 1: status_analise + feedback_ia
-- 1. Status para pipeline assincrono ALTER TABLE intimacoes ADD COLUMN IF NOT EXISTS status_analise text DEFAULT 'concluida' CHECK (status_analise IN ('pendente','processando','concluida','erro','reanalisar')); CREATE INDEX IF NOT EXISTS idx_intimacoes_status_analise ON intimacoes(status_analise) WHERE status_analise IN ('pendente','reanalisar'); -- 2. Tabela de feedback para training loop CREATE TABLE IF NOT EXISTS feedback_ia ( id uuid DEFAULT gen_random_uuid() PRIMARY KEY, intimacao_id uuid REFERENCES intimacoes(id) ON DELETE CASCADE, classe_original text, classe_corrigida text, motivo text, usuario_id uuid REFERENCES auth.users(id), organizacao_id uuid, created_at timestamptz DEFAULT now() ); CREATE INDEX IF NOT EXISTS idx_feedback_ia_org ON feedback_ia(organizacao_id, created_at DESC); -- 3. Config para few-shot examples e prompts CREATE TABLE IF NOT EXISTS cecilia_config ( key text PRIMARY KEY, value jsonb NOT NULL, updated_at timestamptz DEFAULT now(), updated_by text DEFAULT 'system' ); -- 4. Seed: prompt e few-shot iniciais INSERT INTO cecilia_config (key, value) VALUES ('prompt_system', '"Voce e Cecilia..."'::jsonb), ('prompt_triage', '"Classifique..."'::jsonb), ('few_shot_examples', '[]'::jsonb), ('ruido_patterns', '["concluso","redistribu","ato ordinatorio"]'::jsonb) ON CONFLICT (key) DO NOTHING; -- 5. Marcar intimacoes existentes como 'concluida' -- (default ja faz isso, mas garantir) UPDATE intimacoes SET status_analise = 'concluida' WHERE status_analise IS NULL AND classe_cecilia IS NOT NULL;
✅
Zero breaking change: Default 'concluida' = comportamento atual inalterado. Fase 1 continua funcionando mesmo se Fase 2 ainda nao existir. A migration pode ser aplicada AGORA.
