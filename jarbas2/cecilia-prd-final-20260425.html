🧭 PB6 — Navegação EPROC: Deep Dive DEFINITIVO
9.5/10
Documento DEFINITIVO. Resolve o problema mais crítico da Cecilia.

O PB6 ensina a Cecilia onde olhar no EPROC. É a peça que faltava para o pipeline ter sentido end-to-end.

O que o PB6 entrega
  • Fluxo-mãe: 15 etapas (Relação de processos → análise → saída MdFlow → próximo)
  • 10 Regras estruturadas: REGRA/QUANDO/AÇÃO/POR QUÊ/EXCEÇÃO
  • Checklist: 14 perguntas obrigatórias antes de classificar
  • Prompt operacional: pronto para injeção no SOUL.md
  • 134 prints: 7 grupos (login → relação → processo → eventos → ruídos → docs → UI)
  • 3 camadas de leitura: navegação / leitura visual / decisão operacional
  • Tabela de situações: 11 cenários EPROC → playbook aplicável
  • Compliance: o que Cecilia NUNCA faz (não protocola, não assina, não modifica)
As 10 Regras — resumo
  1. Começar SEMPRE pela Relação de processos (nunca busca aleatória)
  2. Ordenar por Data/hora do último evento (coluna-chave)
  3. Abrir processos de cima para baixo (fila auditável)
  4. Confirmar cabeçalho antes de analisar (número, classe, OAB)
  5. Último movimento é ponto de partida, não conclusão
  6. Abrir evento de referência obrigatório ("Refer. ao Evento X")
  7. Documento vinculado deve ser lido quando conteúdo útil
  8. Ruído NÃO vai para o controller
  9. Relevante vira item controller com análise IA
  10. Após concluir, voltar à fila e continuar (ciclo contínuo)
Por que o PB6 resolve o problema

Antes do PB6, a Cecilia capturava via XLS estático — não sabia navegar a interface web do EPROC. Com PB6, ela sabe: onde clicar, o que confirmar antes de analisar, como abrir documentos, quando parar e voltar. Isso mapeia diretamente para S2 (scrape) e S4 (navigate) do pipeline.

Separação de responsabilidades elegante: PB6 é conhecimento conceitual (o que fazer), os seletores CSS/XPath são implementação (como fazer no código Playwright). Esta separação é correta e deve ser mantida.

O que ainda falta (PB6 v2.0 — futura)
  • Seletores CSS/XPath exatos por tribunal (validação pendente com Christian)
  • Fluxo de autenticação por tribunal (TJRS vs TJSC vs TJSP — são diferentes)
  • Screenshots dos prints reais integrados ao markdown
  • Tratamento de erro por tela (502, 503, Cloudflare bloqueio)
  • Fluxo para OABs diferentes na mesma sessão
  • Documentação de edge cases (processo em sigilo, prazo expirado na tela)
