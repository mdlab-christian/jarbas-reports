1. TL;DR Executivo — Por que Cecília não acessa documentos nem faz triagem

Diagnóstico em uma frase:

Existem 6 causas raízes simultâneas (todas P0) — 3 estruturais identificadas na revisão inicial + 3 adicionais descobertas pela revisão adversarial Codex. Mesmo se uma fosse corrigida, as outras ainda bloqueariam o pipeline. Por isso o sintoma é "captura mas não triagem".

Causa 1 — Código
Bug arquitetural S4↔S5
Pipeline navega N processos em loop, depois tenta ler N documentos em UMA chamada. A página fica no último processo aberto, então as leituras dos N-1 anteriores falham silenciosamente.
Causa 2 — Filtro
Pre-classify "djen+ruido" descarta tudo
Regex DJEN_FILTER_KEYWORDS tem termos genéricos demais (EXPEDIDA CITAÇÃO, DISPONIBILIZAÇÃO). Resultado: 100% das intimações descartadas em S3, S4-S8 nem rodam.
Causa 3 — Infra
Cron oficial não está carregado
LaunchAgent com.mdflow.controller-agent.cecilia existe no disco mas não está em launchd. Sistema antigo ai.cecilia.controller está enabled mas sem PID há 3 dias.
Codex 1 — Contrato dados
Eventos sem evento_texto
S2 cria objetos com APENAS titulo (linha 568-571). S5 espera evento_texto/conteudo. Mesmo com fix S4↔S5, toRead zera antes de ler.
Codex 2 — Matcher campo errado
navigateToDocumento e tryL3
s5-read-docs.mjs:277,352 usa evento_texto || conteudo SEM fallback para titulo. Resultado: 0 docs encontrados em #tblEventos.
Codex 3 — Degradação silenciosa
S6 sem documento
Quando S5 falha, S6 ANALYZE roda IA com conteúdo vazio e classifica como "G — ignorável" sem alertar. Sintoma: 1.684 NULL + class A/B inflado.
Métrica brutal (últimas 24h): 1.730 intimações capturadas → 21 com documento (1.2%) → 1.684 pendentes (97% sem IA). Praticamente nenhum valor está sendo entregue ao usuário final.

3. Causas Raízes Detalhadas

P0 — Bug arquitetural: S4 e S5 desacoplados

Arquivo: /Users/cecilia/cecilia-vault/controller-agent/controller-agent.mjs:605-660

Código atual (defeituoso)
// S4 NAVIGATE: navega processo a processo, page fica NO ULTIMO for (const item of docCandidates) { await navigateToProcessByCnj(page, cnj); // navega processo X // ... } // Apos o loop: page esta SOMENTE no ultimo processo // S5 READ_DOCS: chamado UMA vez com array de N intimacoes const paraLeitura = docCandidates.map(p => { ... }); await s5ReadDocs(page, paraLeitura, ...); // tenta ler N docs DA PAGINA DO ULTIMO

Dentro de s5ReadDocs → extractDocumento → navigateToDocumento, o seletor busca #tblEventos da página atual. Se a intimação é do processo X mas page está no Y, a busca falha silenciosamente — sem erro fatal, sem retry, sem log de alerta.

Sintoma observado em produção: SP535101 = 1.200 processos verificados → 0 analisados → 0 docs. SC074025 mesma coisa. A pipeline literalmente "passa por cima" e marca como concluída.
P0 — S3 pre-classify descarta 100% das intimações

Arquivo: /Users/cecilia/cecilia-vault/controller-agent/lib/pre-classify.mjs:169-200

O array DJEN_FILTER_KEYWORDS inclui termos AMPLO demais — descartando intimações relevantes:

  • EXPEDIDA CITAÇÃO — qualquer citação eletrônica (nossa OU adversário)
  • DISPONIBILIZAÇÃO — termo genérico, atinge tudo
  • CONCLUSOS AOS AUTOS — pode ser despacho recente
  • CITAÇÃO ELETRÔNICA EXPEDIDA + variantes
Log evidência — test-sc075466.log
[S3] Triagem: total=11 | descartados=11 (marcoZero=0 parteReu=0 djen+ruido=11 dedup=0) | ciencia=0 | preClass=0 | paraIA=0 | triageIA=1 [S3] Nenhuma intimacao para processar - pulando S4-S8
Resultado catastrófico: 100% das 11 intimações descartadas como "djen+ruido". Os estágios S4 a S8 (navegação, leitura de documento, IA, persistência, alertas) são integralmente pulados. Isso explica os 1.684 pendentes sem classificação.
