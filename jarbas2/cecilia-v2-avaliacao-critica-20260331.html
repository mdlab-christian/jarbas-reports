🔍 Por que a nota geral é 7.2 e não mais?

A arquitetura está certa. O custo e a velocidade estão ótimos. O problema real é que o sistema classifica bem tecnicamente mas roteia mal operacionalmente. Sem contexto do MdFlow no prompt (réu, fase, AJG), a IA classifica "D — revisão manual" para 60%+ dos processos. O João vai ver uma fila enorme de itens sem responsável definido. Isso vai exigir trabalho manual, que é exatamente o que o sistema deveria eliminar. As Fases 1-3 do plano resolvem isso — por isso o potencial sobe para 8.8.

✅ O que vai funcionar bem desde o primeiro ciclo real

1. O pipeline não vai travar. A arquitetura Node.js standalone é muito mais robusta que a v1. Circuit breaker por OAB, retry automático, logs centralizados. Mesmo que o EPROC esteja lento, o sistema vai continuar.

2. O controller vai ter dados. As intimações vão aparecer com schema correto (_analista_raw), compatíveis com o normalizer do frontend. O filtro de DJE e pauta vai funcionar (~40% descartados automaticamente).

3. Classes E e F vão ser sinalizadas. A notificação imediata via Telegram para E/F já está implementada (O12). Alvará expedido e sentença improcedente vão chegar em segundos no grupo de emergência.

4. O custo vai ser irrisório. $19/mês é praticamente nada. Podem rodar sem preocupação com escala de volume.

⚠️ O que vai exigir atenção no início

1. Muita classe D genérica. Enquanto a Fase 1 (contexto MdFlow) não estiver implementada, a IA vai classificar ~50-60% como "D — providência jurídica" sem responsável definido. O João vai ter trabalho manual para rotear essas.

2. Intimações da parte ré. Sem o filtro M2 (polo_representado), algumas intimações direcionadas à parte ré podem aparecer no controller. Não é catastrófico, mas é ruído que o João vai notar.

3. Ações genéricas. "Verificar prazo processual" ainda pode aparecer em alguns resultados antes de M6 (instrução de ação específica no prompt) estar ativa.

4. RS131163 (Heloísa) sem TOTP. Essa OAB pode continuar falhando se o EPROC exigir segundo fator. Monitorar nos primeiros ciclos.

❌ O que não funciona ainda (requer Fases 1-3)

1. Roteamento automático. O sistema não sabe que Boa Vista vai para Leonardo, banco vai para Stephanie, improcedente vai para Heloísa. Tudo isso exige que a Fase 1-2 do plano seja implementada.

2. Criação de tarefas no MdFlow. As intimações ficam na tabela intimacoes mas não viram atividades na fila do responsável no MdFlow. O João precisa ir no controller manualmente — que é melhor que antes, mas não é o fluxo ideal.

3. Verificação de tarefa aberta recente. Pode criar intimação duplicada para processo que já tem tarefa ativa. Ruído na fila.

⚡ 8 Pontos de Atenção Críticos

1
TOTP da RS131163 (Heloísa) — A OAB sem segundo fator configurado vai continuar travando ciclos. Se o EPROC-RS exigir OTP, o login vai falhar silenciosamente. Verificar com Heloísa se o EPROC pede segundo fator para a conta dela e configurar no sistema antes de reativar os crons.
2
maxEventos ainda pode ser insuficiente em processos muito ativos — 50 eventos é muito melhor que 3, mas processos com >50 movimentações desde a última análise (possível após longos períodos parados) ainda vão perder eventos. O filtro real é desdeTimestamp — mas vale monitorar se algum processo está sendo truncado.
3
Tempo de ciclo pode estourar com pico de movimentações — A estimativa de ~40min é para dia normal. Em dias após feriado prolongado (ex: depois de carnaval), podem haver 3x mais movimentações. Um ciclo pode levar 2-3h. O sistema aguenta, mas o cron de 12h em 12h pode se sobrepor. Considerar timeout máximo por OAB.
4
Sessão EPROC expira em 25 minutos — O SESSION_TTL_MS foi aumentado de 14min para 25min. Se um processo for muito lento (PDF grande via Vision), a sessão pode expirar no meio do ciclo. O sistema faz re-login, mas isso adiciona ~30s de latência por ocorrência. Monitorar nos primeiros ciclos.
5
Calibração do prompt vai levar 2-4 semanas — Nenhum prompt de IA funciona perfeitamente de primeira em produção real. Vai haver casos edge (perícia com contexto ambíguo, cancelamento que parece desistência, AJG pendente vs concedida). O João vai precisar auditar os primeiros 200-300 resultados e reportar os erros para calibração contínua. Isso é normal e esperado.
6
Dados de polo_representado podem estar inconsistentes — Se o campo polo_representado não estiver preenchido para todos os processos (ou estiver com valores diferentes do esperado), o filtro M2 vai falhar silenciosamente. Antes de ativar, checar: SELECT polo_representado, COUNT(*) FROM processos GROUP BY 1 e confirmar os valores reais.
7
Processos sem numero_cnj no MdFlow — O batch lookup M1 usa numero_cnj como chave. Se houver processos cadastrados sem CNJ (possível em casos de migração parcial), o enriquecimento vai retornar null para esses. A IA vai classificar sem contexto — comportamento aceitável, mas vale monitorar a taxa de _mdflow: null nos logs.
8
page.goto() no skill_ler_documento — O CC identificou que page.goto() é usado para navegar a documentos. Isso pode resetar a sessão EPROC em alguns contextos. O fallback window.location.href é mais seguro. Não é crítico (funciona), mas se houver erros de sessão intermitentes nos logs, esse é o primeiro lugar a olhar.

🔮 É a Solução Definitiva?

É a fundação certa. Não é o produto final.

O Node.js standalone é a arquitetura correta para esse problema. Escalável, barato, independente, observável. O sistema atual (v2.2) resolve os problemas de infraestrutura que quebravam a v1. Mas o produto final — onde o João abre o controller e vê 50 intimações classificadas corretamente, com responsável definido, ação específica e prazo — ainda depende das Fases 1-3 sendo implementadas e calibradas em produção real.

✅ Recomendação Final do Jarbas 2.0

O que fazer agora (essa semana)

1. Rodar o teste E2E com 10 processos por OAB após o CC aplicar Fase 0. Verificar se o controller popula corretamente.

2. Mostrar pro João os primeiros resultados. Ele vai apontar o que está errado — esse feedback vale ouro para calibrar o prompt.

3. Não reativar os crons antes de validar com o João. Melhor um ciclo manual validado do que 3 ciclos automáticos gerando lixo no controller.

4. CC implementar Fases 1-2 em paralelo enquanto o João valida o E2E. Não precisa esperar — são mudanças independentes.

O sistema vai atender as necessidades de vocês?

Sim — mas em fases. Para o volume de 5.800 processos e meta de 80-100 distribuições/dia, o sistema tem capacidade técnica suficiente. A restrição não é computacional — é de qualidade de classificação. Com as Fases 1-3 implementadas e 2-3 semanas de calibração com o João, o controller vai funcionar como ele imaginou: filtrando o ruído, classificando corretamente, apontando para o responsável certo. A Cecília não vai substituir o João — vai liberar o tempo dele para focar no que importa.
