⚠️ Análise de Riscos — O que Pode Dar Errado

🔴 Risco Alto — Playwright instabilidade (BUG-07)

139 erros eram race conditions profundos. O fix de isClosed() + retry 3x ajuda, mas não resolve 100%. O EPROC é um sistema dos anos 2000 com comportamento imprevisível. Se o E2E rodar em horário de pico no tribunal, pode ter 30-50 erros ainda. Isso não quebra o sistema — só reduz cobertura naquele ciclo.

Mitigação: browser recicla a cada 50 processos + AI Vision como fallback. Não é solução definitiva, mas segura.

🟡 Risco Médio — SSH sem PTY pode timeout em Fases longas

O CC está editando arquivos via SSH no Mac Mini. Se o E2E demorar mais que ~60min, a conexão SSH pode cair. O plano usa nohup pro E2E, mas as edições de arquivos precisam de conexão ativa. O CC sabe disso — o plano está bem estruturado para minimizar.

🟡 Risco Médio — AI Vision pode aumentar custo

O visionFallback() (Claude Haiku Vision) é chamado quando selector falha. Com 139 erros/ciclo, pode custar ~$0.28/ciclo só em Vision. Em 3 ciclos/dia = ~$0.84/dia. Não é caro, mas é novo custo que não estava mapeado. Vale monitorar na primeira semana.

🟢 Risco Baixo — Processos novos não cadastrados

Os ~21 órfãos reais (processos que não existem no MdFlow) vão continuar gerando intimações sem processo_id. Isso é um gap de processo operacional, não de código. A Cecília agora notifica João para esses casos — mas alguém precisa cadastrar o processo no MdFlow.

🟢 Risco Baixo — Seletores CSS mudarem no EPROC

O EPROC atualiza periodicamente. Os seletores confirmados hoje (span#txtMagistrado, span#txtOrgaoJulgador, etc.) podem mudar numa atualização futura. Baixo risco no curto prazo — o EPROC TJRS é estável. Médio prazo (6-12 meses): pode precisar re-confirmar.

Monitoramento manual. João precisava entrar no EPROC de cada OAB, verificar cada processo individualmente, e decidir o que fazer. Com 5.800 processos ativos, era impossível — alguma coisa sempre escapava.

Mesmo com 50% de fallback D e ruídos, a Cecília já alimentava o /controller automaticamente. João tinha uma fila priorizada. Intimações E (estratégicas) e F (SISBAJUD — urgente) eram notificadas imediatamente. Isso por si só já era uma virada.

O que a Cecília v2.3 entrega depois de hoje

O mesmo fluxo, mas limpo. A fila que João vê vai ter 90%+ de intimações reais. O banco vai acumular dados que hoje estão em 0% (advogado_reu, AJG, juiz). Cada ciclo de monitoramento vai enriquecer silenciosamente 30+ processos com dados que antes não existiam.

Em 30 dias rodando em produção, o banco vai ter uma camada de dados que vai permitir análises que hoje são impossíveis: qual advogado da Serasa perde mais? Qual vara decide mais rápido? Quais processos têm AJG e podem priorizar execução?
