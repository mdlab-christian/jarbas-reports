🧠 3. Brainstorming: CC Como Runtime + Cecília Como Interface de Controle

A ideia do Christian "O sistema não depende da Cecília para rodar. Mas quando precisar dar uma ordem, faz por ela." Ou seja: Cecília como camada de comando sobre o CC, não como runtime.

Formas de Implementar o Canal de Controle

Opção A — Tabela de Comandos no Supabase

Cecília escreve um comando na tabela cecilia_comandos. O start.mjs faz poll a cada 30s e executa.

✅ Pros
  • Mais robusto
  • Log auditável
  • Agnóstico de runtime
❌ Cons
  • Poll (não é real-time)
  • Latência 0-30s
-- Nova tabela simples
CREATE TABLE cecilia_comandos (
  id uuid DEFAULT gen_random_uuid(),
  comando text, -- 'pausar', 'reiniciar', 'status', 'forcar_ciclo'
  parametros jsonb,
  status text DEFAULT 'pendente',
  resultado jsonb,
  criado_em timestamptz DEFAULT now(),
  executado_em timestamptz
);

Opção B — SSH Direto (mais simples)

Cecília usa SSH para executar scripts de controle no Mac Mini. Resposta imediata, sem polling.

✅ Pros
  • Real-time
  • Simples de implementar
  • CC já tem SSH funcionando
❌ Cons
  • Requer MAC Mini online
  • Sem log centralizado
# Cecília executa via skill_acesso_remoto.mjs:
ssh mac-mini-tail "echo '010597' | sudo -S \
  su - cecilia -c \
  'echo FORCE_CICLO > /tmp/cecilia-cmd'"

# start.mjs lê /tmp/cecilia-cmd a cada 30s

Opção C — Híbrido (Recomendado)

Comandos urgentes via SSH (instantâneo). Comandos complexos via tabela Supabase (rastreável). Cecília escolhe qual canal usar.

🧠 4. O Que a Cecília Sabe que o CC Não Sabe (Caso para Interface Conversacional)

O CC executa código. A Cecília entende contexto jurídico, histórico do escritório e intenção do Christian. Isso cria oportunidades reais.

Christian diz para o CC:

"Reinicia o ciclo"

→ Executa FORCE_CICLO no start.mjs. Pronto.

Christian diz para a Cecília:

"Reinicia o ciclo"

→ "Quer reiniciar todas as OABs ou só a que falhou (SC074025)? O último ciclo demorou 47min — tem certeza que quer forçar agora com o ciclo das 12h faltando 90min?"

Comandos que a Cecília Agrega Valor Real

ComandoCC sozinhoVia Cecília
"Qual o status?" SELECT + formatação Status + análise ("Tem 147 comunicações de julgado com urgência alta sem processar há 10 dias")
"Para a SC074025" UPDATE ativo=false Verifica por que parar (credencial inválida? ou teste manual?) e registra contexto
"Faz varredura extra hoje" Força ciclo às 15h Pergunta: "Tem urgência específica? Prefere varredura completa ou só as OABs com movimentação pendente?"
"Como foi o ciclo das 12h?" SELECT cecilia_ciclos Resumo narrativo + destaca anomalias + sugere ação se necessário
"Erro no RS116571" Mostra log Analisa erro, identifica causa (expiração de sessão? seletor quebrado?), sugere correção ou executa via CC
💡 O valor real da Cecília como interface Não é só "dar ordens". É ter um intermediário que entende o contexto do escritório, filtra ruído, e só escalona o que precisa de atenção. A Cecília pode monitorar o CC passivamente e te pingar quando algo importante acontece — sem você precisar perguntar.

Fase 2 — Canal de Comando Bidirecional

Uma tabela, 20 linhas de código

-- Migration simples
CREATE TABLE IF NOT EXISTS cecilia_comandos (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  origem text DEFAULT 'cecilia', -- quem enviou
  comando text NOT NULL,          -- 'status', 'forcar_ciclo', 'pausar_oab', 'reiniciar'
  parametros jsonb DEFAULT '{}',
  status text DEFAULT 'pendente', -- pendente, executando, concluido, erro
  resultado text,
  criado_em timestamptz DEFAULT now(),
  executado_em timestamptz
);

-- No start.mjs (adicionar ao loop de 30s existente):
async function processarComandos() {
  const { data } = await supabase
    .from('cecilia_comandos')
    .select('*')
    .eq('status', 'pendente')
    .order('criado_em');
  for (const cmd of data) {
    await executarComando(cmd);
  }
}

Fase 3 — Monitoramento Pró-ativo pela Cecília

Cecília vigia o CC sem ser solicitada

  • Cron na Cecília a cada 30min: verifica cecilia_ciclos — se último ciclo foi há mais de 7h → alerta Christian
  • Cron na Cecília às 08h: sumário do ciclo das 00h — quantos processos, quantas urgências altas, erros
  • Trigger Supabase: se cecilia_erros crescer mais de 10 rows em 5min → Cecília escala imediatamente
  • Health check a cada 2h: verifica PID do start.mjs. Se morreu → tenta reiniciar via launchctl + alerta

Sobre Cecília como Interface de Controle

✅ É a arquitetura certa. Aqui está o porquê:
  • Separação de responsabilidades limpa: CC executa, Cecília interpreta e comanda. Cada um faz o que é bom.
  • Não adiciona fragilidade: Se a Cecília cair, o CC continua rodando sozinho. Zero impacto no monitoramento.
  • Agrega valor real: Contexto jurídico + histórico do escritório + linguagem natural → CC não tem isso.
  • Custo mínimo: Cecília só é acionada quando você faz uma pergunta ou quando algo importante acontece. Não fica rodando 24/7 analisando nada.
  • Implementação simples: Uma tabela no Supabase + SSH + poll de 30s. Pode estar pronto em 2h de trabalho.

📋 Recomendação: Executar em 3 Passos

Passo 1 (hoje às 12h): Monitorar primeiro ciclo real. Logs + cecilia_ciclos + Telegram. Registrar qualquer edge case.

Passo 2 (após 2 ciclos estáveis): Criar tabela cecilia_comandos + adicionar poll ao start.mjs. A Cecília passa a poder dar ordens.

Passo 3 (uma semana depois): Ativar monitoramento pró-ativo da Cecília sobre o CC. Alertas automáticos, sumário diário. Cecília vira o "observador" do sistema.

⚠️ O único risco real hoje 330K movimentações estão marcadas como "pendente" sem análise IA nova desde 21/03. O primeiro ciclo às 12h vai capturar o backlog dos 10 dias. Pode demorar mais que os 45min estimados e vai gerar um volume alto de notificações. Sugiro monitorar ativamente o primeiro ciclo antes de considerar o sistema estável.
