🐛 4 Bugs Críticos Identificados (análise do código + banco)

1
CRÍTICO — tipo_evento = nome da classe do processo, não o evento real

O campo tipo_evento está sendo preenchido com o nome da classe processual do processo ("Indenização por dano moral, Responsabilidade civil, DIREITO CIVIL") em vez do nome do evento (ex: "Disponibilizado no DJEN"). Isso acontece porque o XLS do EPROC tem uma coluna "Classe" (tipo do processo) e o campo mapeado para ultimoEvento está capturando a coluna errada para alguns processos.

Evidência no banco: 23x "Indenização por dano moral" como tipo_evento — isso não é um evento processual, é o assunto do processo.

Impacto: A IA recebe o assunto do processo como "evento" e não consegue classificar corretamente. Acaba classificando como A ou D sem fundamento real.

✅ Fix — skill_processar_planilha.mjs
// O problema está no mapeamento de colunas para alguns processos.
// Verificar qual índice corresponde ao último evento no XLS de cada tribunal.
// O campo correto é o "Último Evento" ou "Evento" — não "Classe" ou "Assunto".

// Adicionar validação: se tipo_evento parece ser assunto (>30 chars sem verbos de ação):

  if (!texto) return false;

CRÍTICO — Eventos "Conclusos" e "DJEN" chegando ao controller

Na amostra do banco aparecem: "Conclusos para decisão/despacho" (7x A), "Conclusos para julgamento" (3x A), "Publicado no DJEN" (2x A). Esses deveriam ser filtrados no XLS pelo isRuido() e nunca chegar ao banco.

O problema: O RUIDO_PATTERNS atual tem /conclus[oã]o?\s+(ao|para)/i, mas "Conclusos para decisão/despacho" usa a forma "Conclusos" (plural com 's') + "para" sem a preposição "ao". O pattern não está matchando essa variação.

Além disso: Processos com tipo_evento = assunto do processo (Bug 1) passam pelo isRuido() porque nenhum pattern de ruído pega texto genérico de assunto como "DIREITO CIVIL".

CRÍTICO — Classe A inserida no banco (deveria ser descartada)

Classe A = irrelevante, não deveria aparecer no controller. Mas 85 de 158 intimações (54%) são classe A. O problema tem duas causas:

Causa A: Processos pré-classificados como A pelo preClassificarEvento() estão sendo persistidos no banco. Eles deveriam ser descartados, não inseridos.

Causa B: A IA também está classificando como A processos que chegaram com evento genérico (Bug 1). Com texto como "DIREITO CIVIL", a IA não sabe o que fazer e marca A por segurança.

Evidência: fonte_preclass = 'IA' em todos os A do banco — esses não vieram do pré-classificador, vieram da IA com input errado.

ALTO — Classe D sem responsável (confiança 0.3 = Haiku não leu documento)

53 intimações Classe D, maioria com confianca_ia = 0.3. Quando a confiança é 0.3, o Haiku classificou sem texto de documento — só com o nome do evento. Isso significa que o documento do evento não foi lido (não chegou no L1/L2/L3) ou que o evento não tinha documento vinculado.

Exemplos reais: "Decisão de saneamento e de organização do processo" com D e confiança 0.3 — o Haiku recebeu só o título do evento, sem o texto da decisão. Com o texto real, provavelmente classificaria corretamente (D ou E dependendo do conteúdo).

Regra que está faltando: Se confiança < 0.5, o sistema deve tentar Sonnet antes de persistir — não insere com confiança baixa sem escalação.

✅ Fix — skill_analisar_ia.mjs — escalação por confiança baixa

function mereceSonnet(mov, haikuClasse, haikuConfianca) {
  if (!['D', 'E', 'F'].includes(haikuClasse)) return false;
  if (mov.isAssinatura) return false;

}

MÉDIO — JavaScript injetado aparecendo como evento

Na amostra do banco apareceu: "26 if (window.carregarTooltipSuspensaoEFeriado === undefined) {" como tipo_evento. O skillAcessarProcesso está capturando texto de scripts JavaScript da página do EPROC como se fosse texto de evento. Acontece quando o DOM não foi carregado completamente antes da extração ou quando o seletor pega elementos errados.

✅ Fix — skill_acessar_processo.mjs
// Ao extrair texto de eventos do DOM, filtrar texto que parece JavaScript:
function pareceJavaScript(texto) {
  if (!texto) return false;
  return (
    texto.includes('function ') ||
    texto.includes('window.') ||
    texto.includes('if (') ||
    texto.includes('var ') ||
    texto.includes('const ') ||
    /^\d+\s+(if|var|function|window|const|let)\b/.test(texto)
  );
}

// No loop de extração de eventos:
if (pareceJavaScript(textoEvento)) {
  console.error(`   ⚠️ [BUG5] JS injetado como evento — ignorando: ${textoEvento.slice(0,50)}`);
  continue;
}

🎯 Problema Raiz Central

O XLS do EPROC não está sendo lido corretamente para todos os tribunais

O skill_processar_planilha.mjs tem detecção dinâmica de colunas, mas para alguns processos o campo ultimoEvento está capturando a coluna de assunto/classe processual em vez do último evento. Isso contamina todo o pipeline:

  1. XLS lê coluna errada → ultimoEvento = "Indenização por dano moral..."
  2. isRuido() não pega (não é ruído, é assunto) → passa para análise
  3. Pré-classificador não pega (nenhum pattern) → vai para EPROC
  4. EPROC extrai eventos e documentos mas o contexto já está errado
  5. IA classifica com confiança baixa (0.3) sem saber o que está analisando
  6. Resultado incorreto vai para o banco e aparece no controller

Fix raiz: Inspecionar o cabeçalho real do XLS para cada tribunal (TJRS, TJSC, TJSP) e confirmar qual coluna é o último evento. Adicionar log explícito: console.log('Cabeçalho detectado:', JSON.stringify(rows[0])).
