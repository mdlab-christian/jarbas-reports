2. Resposta: Qualidade NAO Sera Perdida

Por que as otimizacoes MELHORAM a qualidade

O sistema atual tem um problema fundamental: envia lixo para a IA. 84% dos documentos lidos sao assinaturas e distribuicoes. A IA nao consegue classificar texto tipo "RS055359 GIANMARCO COSTA BEBER PROCURADOR" — entao classifica como D (manual).

Com as otimizacoes:

  1. Pre-classificacao por nome — "Baixa Definitiva" = classe G com 95% confianca. Nao precisa de IA, nao precisa abrir EPROC. Resultado identico ao que um humano daria.
  2. Ler so o evento recente — Se o processo teve "Decisao Interlocutoria" hoje, abrir e ler SÓ esse documento. Nao os 40 historicos.
  3. Filtrar assinaturas — Documentos com pattern "OAB+NOME+ADVOGADO" sao IGNORADOS (nao enviados para IA). Isso MELHORA a qualidade porque a IA recebe so conteudo real.
  4. Sonnet para decisoes, Haiku para triagem — Decisoes judiciais (o que importa) analisadas com Sonnet 4.6. Triagem simples (A/B/G) com Haiku ou regras.
Resultado: Em vez de classificar 136 assinaturas como "D" (inutil), o sistema classifica 6 decisoes reais com contexto completo. Qualidade SOBE, custo DESCE.

3. Sobre a Planilha XLS

O que o EPROC retorna: Uma planilha com 1 linha por processo, contendo: CNJ, Classe, Autores, Reu, Localidade, Assunto, Ultimo Evento, Data Ultimo Evento, Data Autuacao, Valor da Causa.

NAO e uma lista de todas as movimentacoes — e uma lista de todos os processos do advogado, cada um com seu evento mais recente. O XLS do teste tinha 5 linhas (5 processos) e pesava 7.7KB.

Nao ha filtro de data server-side — o EPROC retorna TODOS os processos. A filtragem por data acontece no nosso codigo (etapa 3: comparar dataEvento com desdeTimestamp).

Para ~200 processos com movimentacao por OAB, o XLS teria ~200 linhas (~50KB). Download trivial (~2s).

So processos marcados "Precisa Doc" chegam aqui. Para cada um:

  1. Abrir processo no EPROC
  2. Extrair eventos do DOM mas ler documentos APENAS dos 1-3 eventos mais recentes
  3. Filtrar assinaturas: pular qualquer documento cujo texto contenha pattern /[A-Z]{2}\d{5,6}.*(?:ADVOGAD|PROCURAD)/i

  5. Resultado: 1-2 documentos relevantes por processo (vs. 20+ antes)

6. Otimizacoes Adicionais Descobertas

Alem das O1-O7 do plano anterior, a investigacao profunda revelou mais oportunidades:

P0 O8 — Dedup de Documentos por Assinatura

No teste, o documento "RS055359 GIANMARCO COSTA BEBER PROCURADOR" foi lido 27 vezes (cada evento do mesmo processo tinha essa assinatura anexa). Mesmo que nao filtremos assinaturas, precisamos deduplica-las.

Fix: Antes de ler, verificar se o doc.href ja foi lido nesta sessao. Manter um Set de hrefs ja processados.

P1 O9 — Pular Processos Arquivados/Transitados

A tabela processos tem 170 arquivados + 312 transitados = 482 processos que nao precisam de monitoramento ativo. Se aparecerem no XLS, pular.

Fix: Na etapa 3, fazer lookup de status do processo. Se status IN ('arquivado', 'transitado_julgado'), pular (ou classificar como G automaticamente).

P1 O10 — Batch Lookup de cecilia_ultima_analise_em

Na etapa 3, em vez de 1 query por processo (N queries), buscar todos de uma vez:

// Em vez de:
for (proc of procs) { await getUltimaAnalise(proc.cnj); }

// Fazer:
const cnjs = procs.map(p => p.cnj);
const { data } = await supabase.from('processos')
  .select('numero_cnj, cecilia_ultima_analise_em, status')
  .in('numero_cnj', cnjs)
  .eq('organizacao_id', ORG_ID);
const mapaTimestamps = Object.fromEntries(data.map(d => [d.numero_cnj, d]));

Impacto: De N queries Supabase para 1. Economia de ~2-5 segundos por OAB.

P2 O11 — Polling Adaptativo (ciclos_sem_movimento)

O campo cecilia_ciclos_sem_movimento pode ser usado para reduzir a frequencia de checagem de processos "dormentes":

  • 0-2 ciclos sem mov: checar normalmente
  • 3-5 ciclos sem mov: checar apenas no ciclo das 00h
  • 6+ ciclos sem mov: checar 1x por semana

Isso e util quando o sistema estiver maduro. Nao implementar agora.

P1 O12 — Notificacao Imediata para Classes E/F

Atualmente, notificacoes sao enviadas no final do ciclo. Para classes E (Estrategica) e F (Valores), notificar imediatamente apos classificacao, sem esperar o ciclo terminar.

Impacto: Advogado recebe alerta de penhora/bloqueio em ~1 min em vez de ~40 min.

P1 O13 — Console.warn → Console.log (Visibilidade)

Descobrimos que console.warn vai para cecilia.err.log (stderr), nao para o log principal. Os 138 erros de persistencia ficaram invisiveis!

Fix: Trocar console.warn para console.log em mensagens de erro criticas, ou unificar stdout/stderr no plist.

SET cecilia_ultima_analise_em = COALESCE(
  ultima_movimentacao_em - interval '1 day',
  date_trunc('day', NOW() AT TIME ZONE 'America/Sao_Paulo') AT TIME ZONE 'America/Sao_Paulo'
)
WHERE organizacao_id = '55a0c7ba-1a23-4ae1-b69b-a13811324735'
  AND ativo = true
  AND deleted_at IS NULL
  AND cecilia_ultima_analise_em IS NULL;

-- SELECT count(*) FROM processos
-- WHERE organizacao_id = '55a0c7ba-...' AND cecilia_ultima_analise_em IS NOT NULL;
Por que ultima_movimentacao_em - 1 dia? Se a ultima movimentacao registrada (via ADVBOX) foi dia 20/03, o sistema vai analisar movimentacoes desde 19/03. Isso da margem para nao perder nada no gap entre ADVBOX e Cecilia. Se nao existir essa data, usa hoje 00:00 (ignora historico).
