Problema Raiz: Le TODOS os eventos e documentos de cada processo

A planilha do EPROC lista 5 processos com movimentacao recente. Mas ao acessar cada processo, o sistema le TODOS os eventos historicos (41-44 eventos) e tenta ler TODOS os documentos de cada evento.

Resultado: 3 processos com ~46 documentos cada = 138 movimentacoes para a IA classificar, quando bastaria 1 evento recente por processo = 3 movimentacoes.

Fator de desperdicio: 46x

Problema 2: Documentos de assinatura lidos via Vision ($$$)

Eventos como RS078546CELSO DE FARIA MONTEIROADVOGADO sao apenas assinaturas digitais de advogados — nao tem conteudo juridico util. O sistema tenta ler esses "documentos" e cai no L4 Vision (screenshot + Claude Haiku), gastando ~$0.002 cada.

No teste: ~80 das 100 chamadas Vision foram para assinaturas inuteis.

Problema 3: Sonnet 4.6 para classificacao A-G e overkill

A taxonomia A-G tem 7 classes com regras claras. claude-sonnet-4-6 custa $3/MTok input + $15/MTok output. Para classificacao com regras explicitas, claude-haiku-4-5 ($0.80/MTok in + $4/MTok out) da conta com precisao semelhante — e 4x mais barato.

Problema 4: desdeTimestamp null no primeiro ciclo

Como cecilia_ciclos estava vazio, getUltimoCiclo() retorna null → desdeTimestamp = null → sem filtro temporal. Todos os eventos historicos passam.

Quando null, deveria defaultar para hoje 00:00 BRT.

🔧 Pipeline Otimizado — 7 Mudancas

Fluxo Atual (problemático)
Login → Gerar XLS → Filtrar processos → Acessar processo → Ler TODOS documentos → Enviar 138 movs pra Sonnet
Fluxo Otimizado
Login → Gerar XLS → Filtrar processos → Pre-classificar por evento → So acessar se C-F → Ler so 1 doc recente → Classificar com Haiku
  • P0 O1 — Ler so o evento mais recente, nao o historico inteiro
    Arquivo: skill_acessar_processo.mjs
    A planilha ja diz qual e o ultimo evento de cada processo. Nao precisamos acessar o processo no EPROC e extrair 41+ eventos. A abordagem correta:
    • A planilha ja traz: CNJ, classe, autores, reu, ultimo evento, data
    • Usar o texto do ultimo evento da planilha como input para a IA
    • So acessar o processo no EPROC se a pre-classificacao indicar que precisamos ler o documento real (classes C, D, E, F)
    • Quando acessar: ler APENAS o documento do evento mais recente, nao todos

    Impacto: 46 docs/proc → 0-1 doc/proc (reduz ~98% das leituras)

  • P0 O2 — Pre-classificacao zero-custo antes de acessar processo
    Arquivo: skill_orquestrador.mjs
    Muitos eventos podem ser classificados APENAS pelo nome do evento na planilha, sem abrir o processo:
    • Classe A (DJE, publicacao, remessa, concluso, pauta) → ja filtrado como ruido
    • Classe B (despacho generico, saneador, ato ordinatorio) → classificar pelo nome
    • Classe G (extincao, baixa) → classificar pelo nome
    • So acessar o EPROC para: sentenca, decisao, pericia, deposito, alvara (classes C-F)

    Impacto: ~60-70% dos processos nao precisam ser acessados no EPROC

  • P0 O3 — Trocar Sonnet 4.6 por Haiku 4.5 na classificacao
    Arquivo: skill_analisar_ia.mjs
    A taxonomia A-G tem regras claras e explicitas. Haiku 4.5 classifica com precisao semelhante a 1/4 do custo:
    ModeloInputOutputCusto p/ 100 proc
    Sonnet 4.6$3/MTok$15/MTok$4.50
    Haiku 4.5$0.80/MTok$4/MTok$0.35

    Fallback: se Haiku retornar confianca <0.6, re-classificar com Sonnet (so ~10% dos casos).

    Impacto: 4x reducao no custo de IA

  • P1 O4 — Eliminar leitura de assinaturas digitais
    Arquivo: skill_acessar_processo.mjs
    Documentos cujo texto comeca com padrao RS\d{6}|SC\d{6}|SP\d{6} seguido de nome e "ADVOGADO" sao assinaturas digitais. Pular completamente.

    Impacto: elimina ~60% das leituras de documento quando precisar acessar processo

  • P1 O5 — Default desdeTimestamp = hoje 00:00 quando null
    Arquivo: skill_orquestrador.mjs
    Quando getUltimoCiclo() retorna null (primeiro ciclo ou ciclo anterior falhou), usar meia-noite BRT do dia atual em vez de null. Isso garante que so processos do dia sao analisados.

    Impacto: evita reprocessar historico inteiro no primeiro ciclo

  • P1 O6 — Reduzir system prompt da IA
    Arquivo: skill_analisar_ia.mjs
    O system prompt atual tem ~15K tokens (14 arquivos vault + 52 playbooks + 67 regras). Otimizar:
    • Condensar vault: dos 14 arquivos, so 4 sao essenciais (taxonomia, regras, roteamento, treinamento Joao)
    • Carregar playbooks e regras como resumo, nao texto completo
    • Meta: system prompt <5K tokens (reducao 3x)

    Impacto: 3x menos tokens de input por batch

  • P2 O7 — Vision apenas como ultimo recurso para docs criticos
    Arquivo: skill_ler_documento.mjs
    L4 Vision so deve ser usado para documentos de classes E/F (sentencas, depositos) quando L1-L3 falharem. Para classes C/D, se L1-L3 falharem, classificar com texto do evento sem leitura completa.

    Impacto: elimina ~90% das chamadas Vision
