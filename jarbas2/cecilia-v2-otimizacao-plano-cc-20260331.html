🚨 Problema Especial — Primeiro Setup do Sistema

Por que o primeiro ciclo é catastroficamente lento

A Cecília usa cecilia_ultima_analise_em na tabela processos para saber a partir de quando filtrar movimentações. Problema: esse campo tem 0 registros preenchidos — nunca rodou antes.

Resultado: desdeTimestamp = null → sem filtro temporal → sistema pega TODOS os eventos históricos de todos os processos desde sempre.

Com 200 processos e histórico médio de 40+ eventos cada = potencialmente 8.000 eventos históricos sendo processados.

Solução: Preencher cecilia_ultima_analise_em com hoje 00:00 BRT para todos os processos ativos

Por que é válido: A Cecília vai capturar a partir da última data registrada. Se marcarmos "hoje 00:00 BRT" como base, o sistema vai buscar apenas movimentações de hoje em diante — que é exatamente o comportamento correto para o primeiro setup.

Alternativa ainda mais precisa: Usar ultima_movimentacao_em da tabela processos como base onde disponível (8.103 processos já têm), e cair para hoje 00:00 apenas para os 1.850 que não têm.

SQL necessário:

-- Preencher cecilia_ultima_analise_em para todos os ativos
-- Usa ultima_movimentacao_em se disponível, senão hoje 00:00 BRT
UPDATE processos
SET cecilia_ultima_analise_em = COALESCE(
  ultima_movimentacao_em,
  date_trunc('day', NOW() AT TIME ZONE 'America/Sao_Paulo') AT TIME ZONE 'America/Sao_Paulo'
)
WHERE organizacao_id = '55a0c7ba-1a23-4ae1-b69b-a13811324735'
  AND ativo = true
  AND deleted_at IS NULL
  AND cecilia_ultima_analise_em IS NULL;

### Fase 3 — Implementar após aprovação

**Implementar na seguinte ordem:**

**O1 — Fix desdeTimestamp default (1 linha, zero risco)**
Em `skill_orquestrador.mjs`, na função `getUltimoCiclo()` ou onde `desdeTimestamp` é definido:
```javascript
// Antes:
const desdeTimestamp = ultimoCiclo ? new Date(ultimoCiclo) : null;

// Depois:
const hoje00h = new Date();
hoje00h.setHours(0, 0, 0, 0); // 00:00 local
const desdeTimestamp = ultimoCiclo 
  ? new Date(ultimoCiclo) 
  : hoje00h; // nunca null
```

Também considerar usar `cecilia_ultima_analise_em` da tabela `processos` como fallback por processo (mais preciso do que o último ciclo global).

**O2 — Pré-classificação por nome do evento (antes de acessar EPROC)**
Criar função `preClassificarPorNome(nomeEvento)` em `eproc-utils.mjs`:
- Retorna `{ classe, confianca, precisaDocumento }` ou `null` se inconclusivo
- Lista de mapeamentos baseada nos playbooks do vault da Cecília
- Se `precisaDocumento = false` → não abrir EPROC, classificar direto
- Se `precisaDocumento = true` → abrir EPROC e ler APENAS o documento mais recente (não todos)

**O3 — Usar texto da planilha como input IA (não abrir processo)**
Quando o orquestrador tem o nome do evento da planilha:
1. Tentar pré-classificar por nome → se confiante, registrar e seguir
2. Se inconclusivo: enviar apenas o nome/texto do evento para IA classificar (sem abrir EPROC)
3. Se IA precisar de documento (classes C-F + confiança baixa): aí sim abrir EPROC, mas ler APENAS o documento mais recente

**O4 — Trocar Sonnet 4.6 por Haiku 4.5**
Em `skill_analisar_ia.mjs`:
- Modelo padrão: `claude-haiku-4-5` (ou `claude-haiku-3-5` se 4.5 não disponível via API)
- Fallback para Sonnet: se confiança retornada < 0.60 OU classe E ou F (alto impacto)
- Registrar qual modelo foi usado por classificação em `_analista_raw`

**O5 — Filtrar assinaturas digitais**
Em `skill_acessar_processo.mjs`, antes de tentar ler documento:
- Se nome do evento contém padrões como OAB number + nome + "ADVOGADO": ignorar, retornar null
- Regex: `/\b(OAB|ADV|ADVOGADO|PROCURADOR)\b/i` ou padrão específico que você identificar no código

**O6 — Update `cecilia_ultima_analise_em` ao final de cada ciclo**
Ao finalizar o ciclo de uma OAB, atualizar `cecilia_ultima_analise_em` para `NOW()` para todos os processos que foram processados nesse ciclo:
