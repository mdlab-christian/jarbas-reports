O Problema Central

O campo cecilia_ultima_analise_em da tabela processos esta NULL em 100% dos registros. O orquestrador usa cecilia_ciclos.concluido_em (per-OAB) como fallback. Na primeira execucao, sem ciclo anterior, desdeTimestamp = null — o que faz o sistema processar TODOS os eventos historicos de cada processo.

Teste real (31/03): 3 processos de RS116571 geraram 138 movimentacoes, 63 documentos lidos (muitos via Vision L4), 14 batches IA com Sonnet 4.6. Tempo: ~25 minutos. Custo estimado: ~$2.50 para apenas 3 processos.
Projecao para escala real: Se cada OAB tiver ~20-50 processos com movimentacao por dia (estimativa conservadora), e cada um gerar ~46 eventos, seriam ~5.000 movimentacoes por OAB = 25.000 total por ciclo. Inviavel em tempo e custo.

Fluxo Atual (6 etapas por OAB)

[1/6] Login EPROC — SSO + 2FA + TOTP (~15s)
[2/6] Gerar XLS — relatorio processos do procurador (~10s)
[3/6] Processar XLS — filtrar por data + ruido. Sem desdeTimestamp = 0 filtrados!
[4/6] Acessar processo — navega ao EPROC, extrai TODOS os eventos da pagina
[5/6] Ler documentos — 5 niveis (L1-L5). Le TODOS os docs de TODOS os eventos
[6/6] Analise IA — Sonnet 4.6 + system prompt 15K tokens, batches de 10

3. Otimizacoes Propostas

P0 O1 — Inicializar cecilia_ultima_analise_em

Problema

0% dos processos tem o campo preenchido. Sem ele, o sistema nao sabe "desde quando" buscar movimentacoes e processa TUDO.

Solucao

Executar SQL que preenche cecilia_ultima_analise_em com hoje 00:00 BRT para todos processos ativos:

UPDATE processos
SET cecilia_ultima_analise_em = date_trunc('day', NOW() AT TIME ZONE 'America/Sao_Paulo') AT TIME ZONE 'America/Sao_Paulo'
WHERE organizacao_id = '55a0c7ba-1a23-4ae1-b69b-a13811324735'
  AND ativo = true
  AND deleted_at IS NULL
  AND cecilia_ultima_analise_em IS NULL;

Efeito: No proximo ciclo, so processos com movimentacao a partir de hoje 00:00 BRT serao analisados. Movimentacoes historicas sao ignoradas (ja estao no sistema via ADVBOX).

Risco

Movimentacoes que ocorreram entre a ultima sync ADVBOX e agora serao "perdidas". Mitigacao: se necessario, ajustar para ontem 00:00 ou para COALESCE(ultima_movimentacao_em, hoje_00h).

Impacto

De ~138 movimentacoes por 3 processos para ~1-3 movimentacoes por processo. Reducao de 95-98% do volume.

P0 O2 — Usar cecilia_ultima_analise_em per-processo (nao per-OAB)

Problema

O orquestrador usa getUltimoCiclo(oab) que busca cecilia_ciclos.concluido_em — um timestamp global por OAB. Se o ciclo processa 3 de 200 processos e falha, os outros 197 ficam sem timestamp.

Solucao

Alterar skill_orquestrador.mjs:

  1. Na etapa 3 (processar XLS), para cada processo no XLS, buscar cecilia_ultima_analise_em da tabela processos pelo CNJ
  2. Comparar dataEvento do XLS com cecilia_ultima_analise_em do processo
  3. So processar se dataEvento > cecilia_ultima_analise_em
  4. Apos processar cada processo com sucesso, atualizar cecilia_ultima_analise_em = NOW()

Arquivo

skill_orquestrador.mjs — funcoes getUltimoCiclo() (linhas 56-67) e executarCiclo() (etapa 3)

Pseudo-codigo

// Nova funcao: buscar timestamp per-processo
async function getUltimaAnalise(cnj) {
  const { data } = await supabase.from('processos')
    .select('cecilia_ultima_analise_em')
    .eq('numero_cnj', cnj)
    .eq('organizacao_id', ORG_ID)
    .limit(1).single();
  return data?.cecilia_ultima_analise_em
    ? new Date(data.cecilia_ultima_analise_em)
    : new Date(); // fallback: agora (nao processa se nunca analisou)
}

// Na etapa 3: filtrar por processo individual
for (const proc of processosXLS) {
  const ultimaAnalise = await getUltimaAnalise(proc.cnj);
  if (proc.dataEvento && new Date(proc.dataEvento) > ultimaAnalise) {
    processosParaAnalisar.push(proc);
  }
}

// Apos persistir cada resultado com sucesso:
await supabase.from('processos')
  .update({ cecilia_ultima_analise_em: new Date().toISOString() })
  .eq('numero_cnj', cnj)
  .eq('organizacao_id', ORG_ID);

Impacto

Garante que processos sao re-analisados apenas quando tem movimentacao nova. Elimina reprocessamento.

P0 O3 — Pre-classificar por nome do evento (zero-custo)

Problema

Hoje, todo processo que passa o filtro de data e ruido vai para o EPROC (navegacao web, ~10-30s por processo) para extrair eventos e ler documentos. Mas ~60-70% dos eventos podem ser classificados apenas pelo nome.

Solucao

Criar funcao preClassificarPorNome(eventoTexto) em eproc-utils.mjs:

// Mapa de pre-classificacao por pattern de evento
const PRE_CLASSIFICACAO = [
  // Classe A — Irrelevante (zero-custo, nao precisa doc)
  { pattern: /^DJE\b|^DJM\b|^Diário/i, classe: 'A', confianca: 0.95 },
  { pattern: /certidão\s+(de\s+)?(publicação|intimação)/i, classe: 'A', confianca: 0.90 },
  { pattern: /juntada\s+de\s+petição/i, classe: 'A', confianca: 0.85 },
  { pattern: /prazo\s+decorrido/i, classe: 'A', confianca: 0.90 },
  { pattern: /PROCURAÇÃO|procuracao/i, classe: 'A', confianca: 0.95 },
  { pattern: /cancelamento?\s+de?\s+distribuição/i, classe: 'A', confianca: 0.95 },
  { pattern: /ciência,?\s+com\s+renúncia/i, classe: 'A', confianca: 0.90 },

  // Classe B — Informativo (zero-custo, nao precisa doc)
  { pattern: /^PETIÇÃO$/i, classe: 'B', confianca: 0.80 },
  { pattern: /despacho\s+de\s+mero\s+expediente/i, classe: 'B', confianca: 0.85 },
  { pattern: /conclusos?\s+para\s+despacho$/i, classe: 'B', confianca: 0.80 },
  { pattern: /remetidos?\s+(os\s+)?autos/i, classe: 'B', confianca: 0.85 },
  { pattern: /recebidos?\s+(os\s+)?autos/i, classe: 'B', confianca: 0.85 },

  // Classe G — Encerramento (zero-custo, nao precisa doc normalmente)
  { pattern: /baixa\s+definitiva/i, classe: 'G', confianca: 0.95 },
  { pattern: /transitado\s+em\s+julgado/i, classe: 'G', confianca: 0.95 },
  { pattern: /arquivamento\s+definitivo/i, classe: 'G', confianca: 0.90 },

  // Classe C — Decisao Pendente (PRECISA abrir EPROC e ler doc)
  { pattern: /conclusos?\s+para\s+(decisão|julgamento)/i, classe: 'C', confianca: 0.70, precisaDoc: true },

  // Classe E/F — Potencialmente critico (PRECISA doc)
  { pattern: /sentença|acórdão|decisão\s+interlocutória/i, classe: null, confianca: 0, precisaDoc: true },
  { pattern: /tutela|liminar|antecipação/i, classe: null, confianca: 0, precisaDoc: true },
  { pattern: /penhora|bloqueio|arresto/i, classe: null, confianca: 0, precisaDoc: true },
];

export function preClassificarPorNome(eventoTexto) {
  if (!eventoTexto) return { classe: null, confianca: 0, precisaDoc: true };
  for (const rule of PRE_CLASSIFICACAO) {
    if (rule.pattern.test(eventoTexto.trim())) {
      return {
        classe: rule.classe,
        confianca: rule.confianca,
        precisaDoc: rule.precisaDoc || false,
        regra: rule.pattern.source,
      };
    }
  }
  // Desconhecido: precisa doc + IA
  return { classe: null, confianca: 0, precisaDoc: true };
}

Uso no Orquestrador

// Etapa 3.5 (nova): pre-classificar antes de acessar EPROC
const processosParaEPROC = [];
const jaClassificados = [];

for (const proc of processosRelevantes) {
  const pre = preClassificarPorNome(proc.ultimoEvento);
  if (pre.classe && pre.confianca >= 0.80 && !pre.precisaDoc) {
    // Classificado com confianca — nao precisa abrir EPROC
    jaClassificados.push({
      ...proc,
      classe: pre.classe,
      confianca: pre.confianca,
      fonte: 'pre_classificacao',
      texto: proc.ultimoEvento,
    });
  } else {
    // Precisa abrir EPROC para ler documento
    processosParaEPROC.push(proc);
  }
}

console.log(`   Pre-classificados: ${jaClassificados.length} | Para EPROC: ${processosParaEPROC.length}`);

P0 O4 — Ler SOMENTE o evento recente (nao todos os historicos)

Problema

skill_acessar_processo.mjs extrai TODOS os eventos da pagina do processo e depois filtra por data client-side. Com processos que tem 20-40+ eventos, isso gera dezenas de documentos para ler.

Solucao

Quando o orquestrador chama skillAcessarProcesso, ja sabe qual e o evento-gatilho (do XLS). Alterar para:

  1. Extrair TODOS os eventos do DOM (necessario para navegacao)
  2. Mas so ler documentos do evento mais recente (ou eventos desde cecilia_ultima_analise_em)
  3. Ignorar documentos de eventos antigos

Alteracao em skill_acessar_processo.mjs (linhas 90-149)

// ANTES: retorna TODOS os eventos com data >= desdeTs
// DEPOIS: retorna apenas os N mais recentes (ex: top 3) desde desdeTs

// Apos extrair eventos, ordenar por data DESC e pegar apenas os primeiros
const eventosRecentes = eventosExtraidos
  .filter(ev => !desdeTs || ev.dataDate >= new Date(desdeTs))
  .sort((a, b) => b.dataDate - a.dataDate)
  .slice(0, 3); // MAX 3 eventos mais recentes

return { eventos: eventosRecentes };

Complemento: Filtro de Assinaturas

Dentro dos eventos, filtrar documentos que sao assinaturas de advogado:

// Filtrar docs de assinatura antes de ler
const docsRelevantes = evento.documentos.filter(doc => {
  const texto = (doc.texto || '').toUpperCase();
  // Pular: OAB + ADVOGADO/PROCURADOR (assinaturas)
  if (/[A-Z]{2}\d{5,6}.*(?:ADVOGAD|PROCURAD)/i.test(texto)) return false;
  // Pular: certificados digitais
  if (/CERTIFICADO\s+DIGITAL|ASSINATURA\s+DIGITAL/i.test(texto)) return false;
  return true;
});

P1 O6 — Reduzir system prompt (15K → 5K tokens)

Problema

O system prompt carrega 14 arquivos do vault + 52 playbooks + 67 regras = ~15K tokens. Com cache Anthropic, a primeira chamada paga tudo; as demais pagam 10%. Mas o custo base ainda e alto.

Solucao

  1. Compilar os 14 arquivos vault em 1 prompt condensado com as regras essenciais
  2. Mover playbooks/regras para um formato compacto (tabela em vez de texto livre)
  3. Remover duplicatas entre vault + playbooks + regras
  4. Manter prompt cacheavel (mesmo conteudo entre batches)

Impacto

System prompt de 15K → ~5K tokens. Economia de ~$0.03 por batch (10 movs). Com Haiku, a economia e menor mas o prompt menor tambem acelera a resposta.

P2 O7 — Vision apenas para classes criticas (E/F)

Problema

Quando L1-L3 falham na leitura de documento, L4 (Vision + Haiku) e acionado. Para documentos de classe A/B, isso e desperdicio — nao precisavamos ler o documento.

Solucao

Apos O3 (pre-classificacao), so documentos de processos que precisam de leitura chegam na etapa de ler documentos. Para esses, se L1-L3 falham:

  • Classes C/D: Registrar como "documento ilegivel" com confianca baixa → revisao manual
  • Classes E/F (ou incerto): Usar Vision L4 — justificavel pelo impacto critico

Impacto

Vision L4 de ~15 calls/processo para 0-1 calls/processo. Economia de ~$0.01-0.03 por processo.
