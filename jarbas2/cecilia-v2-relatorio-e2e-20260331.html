Issue 2 — Extracao Corrompida

P0 Critico
🔍 Descricao

O campo tipo_evento extraido do XLS do EPROC contem dados corrompidos em multiplas categorias. Isso causa falha em cascata: se o input e lixo, nenhum modelo de IA vai classificar corretamente.

🧪 Categorias de Corrupcao
Tipo Qtd Exemplo Causa Provavel

Classe Juridica ~8 Indenizacao por dano moral, DIREITO CIVIL Coluna errada no XLS ou merge de colunas
Nome de Juiz ~1 c.rohenkohlCaio Eduardo Rohenkohl Concatenacao de colunas adjacentes
DJEN Publicacao ~10 Publicacao no DJEN Nao e corrupcao, e noise nao filtrado
🧬 Analise de Causa Raiz

O skill skill_processar_planilha.mjs processa o arquivo Excel exportado pelo EPROC. O problema esta em como os dados sao parseados:

// Problema 1: XLS do EPROC tem colunas com conteudo misto // A coluna "Evento" pode conter scripts JS embutidos no HTML // Problema 2: Merge de colunas adjacentes // "classe juridica" e "magistrado" aparecem colados no tipo_evento // Solucao: Sanitizar ANTES de enviar para IA function sanitizeTipoEvento(raw) { // Remove JS fragments if (raw.includes('window.') || raw.includes('function(')) return null; // Remove classes juridicas if (raw.includes('DIREITO') && raw.includes(',')) return null; // Remove nomes de magistrado (pattern: username+nome+MAGISTRADO) if (raw.includes('MAGISTRADO')) return null; return raw.trim(); }
🛠️ Solucao Proposta
  1. Adicionar sanitizacao no skill processar_planilha — regex para detectar e descartar JS, classes juridicas, nomes de magistrado
  2. Validar header do XLS — logar quais colunas o EPROC exportou, conferir que estao no indice esperado
  3. Expandir RUIDO_PATTERNS em eproc-utils.mjs para filtrar assuntos como tipo_evento
  4. Log de descarte — registrar items descartados para auditoria

Impacto: Elimina ~14 das 51 falhas Haiku. Previne classificacoes impossiveis. Zero JS/assunto como tipo_evento.

Issue 9 — parseJsonArray() Contamina Batch

P0 Critico — NOVO
🔍 Descricao

Esta e a causa raiz real dos "32.3% de falha Haiku". Nao e o modelo falhando — e o parser de JSON.

Mecanismo de contaminacao:

  1. Haiku recebe batch de N processos
  2. Haiku retorna JSON com formatacao ligeiramente invalida (trailing comma, markdown wrapper, etc.)
  3. parseJsonArray() tenta JSON.parse() — falha
  4. Retorna [] (array vazio)
  5. Orquestrador interpreta: "nenhum resultado" = TODOS os N processos recebem fallback
  6. Cada processo: classe='D', confianca=0.3, acao='Classificacao Haiku falhou'

Um unico erro de JSON contamina o batch INTEIRO. Se o batch tem 10 processos, 10 recebem Classe D falsa.

🛠️ Solucao: parseJsonArrayRobusto()
function parseJsonArrayRobusto(text) { // Estrategia 1: JSON.parse direto try { return JSON.parse(text); } catch {} // Estrategia 2: Extrair array de dentro de markdown ```json ... ``` const match = text.match(/```json?\s*([\s\S]*?)```/); if (match) try { return JSON.parse(match[1]); } catch {} // Estrategia 3: Regex para extrair objetos individuais const objects = [...text.matchAll(/\{[^{}]*\}/g)]; if (objects.length) { return objects.map(m => { try { return JSON.parse(m[0]); } catch { return null; } }).filter(Boolean); } // Estrategia 4: Retry individual quando batch falha return null; // null (nao []) — sinaliza para retry individual }

Mudanca critica: Retornar null em vez de [] quando todas as estrategias falham. null sinaliza para o orquestrador reprocessar o batch individualmente em vez de aplicar fallback cego para todos.

Issue 4 — Eventos "Concluso" no Controller

P1 Importante
🔍 Descricao

13 eventos do tipo "concluso" chegaram ao controller durante o E2E. Segundo o treinamento do Joao, todos os conclusos sao Classe A (irrelevante) e devem ser auto-arquivados.

Distribuicao:

  • 10x "Conclusos para decisao/despacho"
  • 3x "Conclusos para julgamento"

Classificacao recebida:

  • 10x Classe A (correto)
  • 3x Classe D (Haiku falhou — deveria ser A)
🛠️ Solucao Proposta

Adicionar regra deterministica no noise filter antes da IA:

// Regra: Concluso = sempre Classe A const CONCLUSO_PATTERNS = [ /conclus[oa]s?\s+(para|ao)/i, /conclus[oa]s?\s+p\//i, ]; function isConcluso(tipoEvento) { return CONCLUSO_PATTERNS.some(p => p.test(tipoEvento)); } // No pipeline, ANTES de chamar Haiku: if (isConcluso(intimacao.tipo_evento)) { intimacao.classe = 'A'; intimacao.classe_source = 'regra_deterministica'; return; // Skip IA — zero tokens gastos }

Impacto: 13 chamadas IA eliminadas por ciclo. Zero falsos positivos para conclusos. Zero "Conclusos para decisao" no controller.

📝 Mensagens de Erro
// 104 ocorrencias (75.9%) "Target page, context or browser has been closed" // 33 ocorrencias (24.1%) "Execution context was destroyed, most likely because of navigation"

Ambos indicam que o browser ou a page foi destruida durante uma operacao assincrona — tipicamente quando a pagina navega enquanto o script tenta ler elementos.

🧬 Analise de Causa Raiz
  • Navegacao do EPROC: O EPROC usa redirects e reloads frequentes que destroem o contexto da page
  • Memory leak: Playwright com 1.122 processos pode acumular memoria e crashar
  • Timeout insuficiente: EPROC pode demorar e o browser fecha antes da resposta
  • Sessao expirada: A sessao do EPROC expira e forca redirect para login
🛠️ Solucao Proposta
  1. safeEvaluate() com retry — wrapper para page.evaluate() que verifica isClosed() antes de executar e retenta em caso de context destroyed
  2. Reciclar browser context — criar novo context a cada 50 processos
  3. Retry com backoff — retry 2x com delay exponencial antes de marcar como erro
  4. Verificar sessao antes de navegar — checar se a sessao EPROC ainda esta ativa

Nota: Os 137 erros nao impactaram a classificacao (sao do skill acessar_processo, nao do analisar_ia). Mas impedem a leitura de detalhes do processo quando necessario.
