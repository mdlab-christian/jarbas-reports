Issue 1 — Falhas Haiku (32.3%)

Issue 2 — Extracao Corrompida

P0 Critico
🔍 Descricao

O campo tipo_evento extraido do XLS do EPROC contem dados corrompidos em multiplas categorias. Isso causa falha em cascata: se o input e lixo, nenhum modelo de IA vai classificar corretamente.

🧪 Categorias de Corrupcao
Tipo Qtd Exemplo Causa Provavel
JS Fragment ~5 if (window.carregarTooltip... XLS parser captura conteudo de <script> tag
Classe Juridica ~8 Indenizacao por dano moral, DIREITO CIVIL Coluna errada no XLS ou merge de colunas
Nome de Juiz ~1 c.rohenkohlCaio Eduardo Rohenkohl Concatenacao de colunas adjacentes
DJEN Publicacao ~10 Publicacao no DJEN Nao e corrupcao, e noise nao filtrado
🧬 Analise de Causa Raiz

O skill extrair_xls processa o arquivo Excel exportado pelo EPROC. O problema esta em como os dados sao parseados:

// Problema 1: XLS do EPROC tem colunas com conteudo misto // A coluna "Evento" pode conter scripts JS embutidos no HTML // Problema 2: Merge de colunas adjacentes // "classe juridica" e "magistrado" aparecem colados no tipo_evento // Solucao: Sanitizar ANTES de enviar para IA function sanitizeTipoEvento(raw) { // Remove JS fragments if (raw.includes('window.') || raw.includes('function(')) return null; // Remove classes juridicas if (raw.includes('DIREITO') && raw.includes(',')) return null; // Remove nomes de magistrado (pattern: username+nome+MAGISTRADO) if (raw.includes('MAGISTRADO')) return null; return raw.trim(); }
🛠️ Solucao Proposta
  1. Adicionar sanitizacao no skill extrair_xls — regex para detectar e descartar JS, classes juridicas, nomes de magistrado
  2. Validar estrutura do XLS — conferir que as colunas estao no indice esperado antes de processar
  3. Log de descarte — registrar items descartados para auditoria (tabela cecilia_descartados)

Impacto: Elimina ~14 das 51 falhas Haiku. Previne classificacoes impossíveis.

Issue 3 — 44 Intimacoes Orfas

P0 Critico
🔍 Descricao

44 intimacoes (27.8%) nao possuem processo_id, o que significa que nao podem ser vinculadas a um processo no MdFlow. Isso inclui intimacoes urgentes como Decisoes Interlocutorias (Classe E).

📝 Exemplos de Orfas com Conteudo Legitimo
tipo_evento Classe Urgencia Consequencia
Decisao Interlocutoria E (Estrategico) ALTA Perda de prazo potencial
Decisao de saneamento e de organizacao do processo D (Juridico) Media Tarefa nao roteada
Ato ordinatorio praticado D (Haiku falhou) Baixa Fila poluida
if (window.carregar... D (lixo) Nenhuma Deveria ser descartado
🧬 Analise de Causa Raiz

O XLS do EPROC nem sempre contem numero de processo na mesma linha da intimacao. O skill processar_xls tenta vincular via regex de numero CNJ, mas falha quando:

  • O numero do processo esta em uma linha anterior (agrupamento no XLS)
  • O formato do numero varia entre tribunais (TJRS vs TJSC)
  • A intimacao e de publicacao DJEN (nunca tem numero inline)
🛠️ Solucao Proposta
  1. Lookup reverso — buscar processo_id na tabela processos do MdFlow usando numero CNJ parcial ou nome das partes
  2. Heranca de contexto no XLS — quando uma linha nao tem processo_id, herdar da linha anterior (padrao de agrupamento do EPROC)
  3. Fila de pendentes — orfas que nao resolvem automaticamente vao para fila manual do Joao (em vez de serem perdidas)

Impacto: Recupera ate ~30 das 44 orfas (estimativa conservadora). Decisoes Classe E nunca mais seriam perdidas.

Issue 4 — Eventos "Concluso" no Controller

P1 Importante
🔍 Descricao

13 eventos do tipo "concluso" chegaram ao controller durante o E2E. Segundo o treinamento do Joao, todos os conclusos sao Classe A (irrelevante) e devem ser auto-arquivados.

Distribuicao:

  • 10x "Conclusos para decisao/despacho"
  • 3x "Conclusos para julgamento"

Classificacao recebida:

  • 10x Classe A (correto)
  • 3x Classe D (Haiku falhou — deveria ser A)
🛠️ Solucao Proposta

Adicionar regra deterministica no noise filter antes da IA:

// Regra: Concluso = sempre Classe A const CONCLUSO_PATTERNS = [ /conclus[oa]s?\s+(para|ao)/i, /conclus[oa]s?\s+p\//i, ]; function isConcluso(tipoEvento) { return CONCLUSO_PATTERNS.some(p => p.test(tipoEvento)); } // No pipeline, ANTES de chamar Haiku: if (isConcluso(intimacao.tipo_evento)) { intimacao.classe = 'A'; intimacao.classe_source = 'regra_deterministica'; return; // Skip IA — zero tokens gastos }

Impacto: 13 chamadas IA eliminadas por ciclo. Zero falsos positivos para conclusos.

Issue 5 — Noise Filter Incompleto

P1 Importante
🔍 Descricao

O noise filter atual filtra apenas um subconjunto dos eventos irrelevantes. Publicacoes DJEN, conclusos e varios eventos automaticos passam direto para a IA, consumindo tokens desnecessariamente.

📋 Eventos que Devem ser Filtrados (Pre-IA)
Pattern Classe Qtd no E2E Status Atual
Conclusos para * A 13 Nao filtrado
Publicacao no DJEN A ~10 Nao filtrado
Ato ordinatorio praticado A ~6 Nao filtrado
Despacho de mero expediente A ~4 Nao filtrado
Juntada de peticao A/B ~3 Parcial

Economia estimada: ~36 chamadas IA por ciclo eliminadas (~23% do total)

🛠️ Solucao: Tabela de Regras Deterministicas

Em vez de hardcodar patterns no codigo, criar uma tabela SQL editavel pelo Joao:

CREATE TABLE cecilia_noise_rules ( id serial PRIMARY KEY, pattern text NOT NULL, -- regex ou texto exato match_type text DEFAULT 'contains', -- 'contains', 'regex', 'exact' classe text DEFAULT 'A', -- classe a atribuir ativo boolean DEFAULT true, criado_por text DEFAULT 'sistema', created_at timestamptz DEFAULT now() );

Beneficio: Joao pode adicionar/remover regras sem deploy. Auditavel. Reduz custo de tokens em ~23%.

Issue 6 — 118 Erros Playwright

P1 Importante
🔍 Descricao

118 erros de Playwright registrados durante o E2E, 100% concentrados no skill acessar_processo. O browser crashou repetidamente durante navegacao no EPROC.

📊 Distribuicao dos Erros

88.1%

11.9%

📝 Mensagens de Erro
// 104 ocorrencias (88.1%) "Target page, context or browser has been closed" // 14 ocorrencias (11.9%) "Execution context was destroyed, most likely because of navigation"

Ambos indicam que o browser ou a page foi destruida durante uma operacao assincrona — tipicamente quando a pagina navega enquanto o script tenta ler elementos.

🧬 Analise de Causa Raiz
  • Navegacao do EPROC: O EPROC usa redirects e reloads frequentes que destroem o contexto da page
  • Memory leak: Playwright com 1.122 processos pode acumular memoria e crashar
  • Timeout insuficiente: EPROC pode demorar e o browser fecha antes da resposta
  • Sessao expirada: A sessao do EPROC expira e forca redirect para login
🛠️ Solucao Proposta
  1. Reciclar browser context — criar novo context a cada N processos (ex: a cada 100)
  2. Retry com backoff — retry 2x com delay exponencial antes de marcar como erro
  3. Verificar sessao antes de navegar — checar se a sessao EPROC ainda esta ativa
  4. Monitorar memoria — log de uso de memoria a cada 50 processos

Nota: Os 118 erros nao impactaram a classificacao (sao do skill acessar_processo, nao do analisar_ia). Mas impedem a leitura de detalhes do processo quando necessario.

Issue 7 — Gap no Filtro do Frontend

P2 Menor
🔍 Descricao

O frontend do MdFlow nao filtra eventos "concluso" explicitamente na visualizacao de intimacoes. Se o backend nao resolver, o usuario ve conclusos na fila de trabalho.

Mitigacao: Se a Issue 4 (concluso no backend) for resolvida, este gap se torna irrelevante. Prioridade baixa.

🛠️ Solucao (se necessario)

Adicionar filtro no componente de listagem de intimacoes:

// Frontend: filtrar conclusos na query const { data } = await supabase .from('cecilia_intimacoes') .select('*') .neq('classe', 'A') .not('tipo_evento', 'ilike', '%conclus%');
