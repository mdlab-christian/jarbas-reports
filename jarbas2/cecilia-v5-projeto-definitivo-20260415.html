Resultado da Pesquisa — Por que mudamos de plano

SECAO 1
1.1 O que investigamos

Claude Managed Agents DESCARTADO

  • Cloud-only — $0.08/session-hour + tokens
  • NAO roda no Mac Mini (requisito hard)
  • Rate limit: max 5 sessoes simultaneas
  • Custo proibitivo: $15-25/ciclo

Skyvern DESCARTADO

  • TOTP nativo (ponto positivo)
  • NAO exporta cookies (Issue #3980, #1649) — DEAL BREAKER
  • 10-20x mais lento que Playwright (60-90s vs 5s por login)
  • Custo LLM por step ($0.04-0.16 por task)
  • Browser profiles so funcionam com workflows, nao com runTask

Browser Use DESCARTADO

  • Python (stack e Node.js)
  • Sem TOTP nativo
  • 3-8s por step
  • Inferior ao Skyvern em features

Firecrawl DESCARTADO

  • NAO suporta autenticacao Keycloak+TOTP
  • Apenas sites publicos
  • Inviavel para EPROC autenticado

Datajud API INCORPORADO

  • API publica do CNJ, endpoint REST
  • Retorna metadados + movimentacoes de processos
  • NAO retorna conteudo de documentos
  • Util como fonte COMPLEMENTAR (cross-reference)
  • Endpoint: POST https://api-publica.datajud.cnj.jus.br/api_publica_{tribunal}/_search
1.2 A descoberta critica sobre o EPROC

TODAS as URLs internas do EPROC contem hash CSRF. E impossivel construir URLs manualmente. Links devem ser extraidos do DOM renderizado. Isso torna QUALQUER abordagem sem Playwright inviavel para operacoes autenticadas.

Implicacao: Mesmo com cookies validos exportados, voce NAO consegue navegar o EPROC via HTTP puro. Cada link tem um hash de sessao unico que muda a cada renderizacao. Skyvern, Browser Use, Firecrawl e HTTP puro estao todos descartados para operacoes que exigem navegacao autenticada.

1.3 Conclusao da pesquisa

Playwright e a UNICA opcao viavel para interagir com o EPROC. A solucao nao e substituir o Playwright — e FORTIFICAR o pipeline ao redor dele com auto-correcao, retry inteligente, e multiplas fontes de dados.

Camada Self-Healing — Detalhamento

SECAO 4

4.1 Session Health Monitor

Roda ENTRE cada stage (S1→S2, S2→S3, etc.) para garantir que a sessao do EPROC ainda e valida.

  • Verifica: page.url() contem "usuario_logar" ou "login" ou "sso."?
  • Se sim: executa S1 re-login automatico
  • Log: "[SELF-HEAL] Sessao expirada detectada entre S4→S5, re-login executado em 12s"
  • Metricas: quantos re-logins por ciclo (target: 0-1)

4.3 Schema Validator

Contract entre stages: verifica que cada objeto tem os campos obrigatorios antes de avancar.

  • S5→S6: conteudo_completo presente para classes C-F?
  • S6→S7: analise_ia tem classe, confianca, resumo?
  • Se falhar: log P0 + Telegram alert + pular item (nao crashar pipeline)
  • Arquivo: lib/schema-validator.mjs

4.4 Budget Guardian

Controla custo acumulado de chamadas Claude API durante o ciclo.

  • Acumula custo_estimado_usd de cada chamada S6
  • Se custo >= $2.00: ABORT OAB restante
  • Telegram alert: "Budget cap atingido: $2.15 / $2.00. X intimacoes nao analisadas."
  • Budget configuravel via cecilia_config.budget_cap_usd

4.5 Circuit Breaker

Protege contra falhas repetitivas que indicam problema estrutural.

  • 3 ciclos consecutivos com status='falhou' → bloquear OAB
  • v5 novo: Auto-reset apos ciclo com status='concluido' (T02)
  • v5 novo: Reset por tempo (24h sem ciclo OK → reset)
  • Comando manual: /cecilia forcar RS116571

4.2 Error Classifier

Cada erro e classificado em 3 categorias com acoes automaticas distintas:

RETRY

Erro transiente: timeout, network, 429, 500, 529

→ Retry com backoff exponencial

SKIP

Erro local: selector nao encontrado, documento vazio

→ Skip item, continuar pipeline

ABORT

Erro fatal: login falhou 3x, budget estourou, circuit breaker

→ Parar OAB, alerta Telegram

Camada Comando — MdOs + Telegram + OpenClaw

SECAO 5
5.1 Modos de Operacao

CRON (automatico)

Execucao periodica sem intervencao humana

  • Horarios configuraveis (default: 6h, 13h, 19h BRT)
  • Configuracao via tabela cecilia_config no Supabase
  • Mudanca de horario via MdOs ou Telegram
  • start-cron.mjs le config do banco a cada poll (60s)

ON-DEMAND (manual)

Execucao sob demanda por operador

  • Via MdOs: /cecilia analisar RS116571 --max 10
  • Via Telegram: @cecilia analisa processos RS116571
  • Via OpenClaw: mensagem natural no Telegram/WhatsApp
  • Via CLI SSH: node controller-agent.mjs --oab RS116571

BULK (lote)

Processamento em massa configuravel

  • Via MdOs: /cecilia bulk --oab RS116571 --max 200
  • Via Telegram: @cecilia analisa 200 processos RS116571
  • Usa mesmo pipeline, so muda MAX_PROCESSOS_POR_OAB
  • Relatorio consolidado ao final

Datajud API — Fonte Complementar

SECAO 6

6.1 O que e

API publica do CNJ que retorna metadados e movimentacoes de TODOS os tribunais brasileiros. Nao requer autenticacao EPROC — acesso via API key publica. Retorna dados estruturados em formato Elasticsearch.

6.2 Como usar no pipeline

  • S0 Antes do login EPROC, consultar Datajud para lista de processos com movimentacoes recentes
  • S2 Apos scrape EPROC, cross-reference com Datajud para detectar movimentacoes perdidas
  • Val Comparar dados EPROC vs Datajud para detectar inconsistencias

6.4 Limitacoes

  • ● NAO retorna conteudo de documentos (so titulos de movimentacoes)
  • ● Dados podem ter delay (nao real-time)
  • ● Processos sigilosos excluidos

Por que NAO usar Skyvern / Browser Use / etc

SECAO 10

Apos pesquisa extensiva de 5 alternativas ao Playwright, concluimos que NENHUMA e viavel para o EPROC por 2 razoes fundamentais:

1. O EPROC usa hash CSRF em TODAS as URLs internas. Qualquer abordagem que nao renderize o DOM completo (como HTTP puro com cookies) falha porque nao consegue construir URLs de navegacao. Os links sao dinamicos, gerados no servidor, e mudam a cada renderizacao da pagina.

2. O Skyvern, a alternativa mais promissora, NAO exporta cookies do browser (Issue #3980, #1649 no GitHub). Isso impede o modelo hibrido onde Skyvern faria login e HTTP faria a coleta. Sem cookies exportaveis, o Skyvern nao pode ser usado como "ponte" para outros coletores.

A solucao correta nao e substituir o Playwright — e FORTIFICAR o pipeline ao redor dele. O Playwright e rapido ($0, <5s por operacao), tem acesso total a cookies e DOM, e ja esta integrado na Cecilia desde a v1. O que faltava era resiliencia, auto-correcao e monitoramento — que e exatamente o que a v5.0 Self-Healing Pipeline adiciona.

O resultado: um pipeline que detecta seus proprios erros, se recupera automaticamente, controla custos, e pode ser operado remotamente via Telegram, MdOs ou OpenClaw — sem nunca depender de ferramentas externas que nao entendem a complexidade do EPROC.
