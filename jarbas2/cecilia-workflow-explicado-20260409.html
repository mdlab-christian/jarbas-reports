1. Visao Geral do Sistema

A Cecilia e um sistema de scraping automatizado que monitora processos judiciais no EPROC (sistema dos tribunais do sul do Brasil). Ela roda no Mac Mini do escritorio, faz login como advogado, baixa planilhas de movimentacoes, le documentos e classifica tudo com IA. O resultado alimenta o frontend /controller no MdFlow.

2. O Cron — Como Tudo Comeca

O arquivo start.mjs e o entry point. Ele faz 3 coisas: configura os crons a partir do banco, inicia os task consumers e sobe o health endpoint. A cada 5 minutos ele relele a config do banco para pegar mudancas de horario sem reiniciar o processo. 

start.mjs (entry point — PID 56457)
    │
    ├── Le cecilia_capture_config no Supabase (poll a cada 5 min)
    │   └── Quais OABs estao ativas? Quais horarios?
    │   └── Resposta: [{ oab: "RS116571", ativo: true, horarios: ["00:00","12:00","18:00"] }, ...]
    │
    ├── Cria cron jobs via node-cron
    │   ├── 00:00 BRT ─── dispara ciclo para TODAS as 5 OABs simultaneamente
    │   ├── 12:00 BRT ─── dispara ciclo para TODAS as 5 OABs simultaneamente
    │   └── 18:00 BRT ─── dispara ciclo para TODAS as 5 OABs simultaneamente
    │
    ├── Task Consumer (4 workers paralelos)
    │   ├── Poll jarbas_tarefas a cada 1-5s
    │   ├── Tipos: captura_manual, reprocessar_ia, importacao_diaria
    │   └── Permite disparo manual pelo frontend (botao "Capturar agora")
    │
    └── Health endpoint (porta 19200)
        └── GET /health → { status: "running", uptime: "12h", ultimo_ciclo: "2026-04-08T18:00" }
Problema critico: Execucao simultanea
Todas as 5 OABs disparam ao MESMO TEMPO via Promise.all(). Isso significa 10+ instancias de Chromium headless rodando simultaneamente, competindo pelo SSO do Keycloak. Resultado: 55% success rate. A correcao e trivial: trocar para for...of sequencial com stagger de 5-10 minutos entre OABs.
🔄

3. Pipeline de 6 Stages — Passo a Passo

Para CADA OAB, o skill_orquestrador.mjs executa estes 6 stages em sequencia. Se qualquer stage falha, o ciclo aborta para aquela OAB e registra o erro em cecilia_ciclos. Os stages sao independentes por OAB — uma OAB falhando nao afeta outra. 

Para cada OAB (ex: RS116571):

  [ STAGE 1 ]    [ STAGE 2 ]    [ STAGE 3 ]    [ STAGE 4 ]    [ STAGE 5 ]    [ STAGE 6 ]
    LOGIN    ───>  PLANILHA   ───> PROCESSAR  ───>  ACESSAR   ───>  LER DOC   ───>  ANALISE IA
  Keycloak SSO    Export XLS    Parse+Filter   Enriquecer    5 fallbacks    Sonnet 4.6
  ~15-30s         ~10-20s       ~2-5s          ~3-5 min      ~1-3 min       ~30-60s

  Total por OAB: ~5-10 minutos (depende da quantidade de movimentacoes novas)
STAGE 1
LOGIN — Autenticacao no EPROC
skill_login_eproc.mjs v2.7
Primeiro, tenta restaurar uma sessao salva no banco. Se nao houver ou estiver expirada, abre um navegador headless e faz o fluxo completo de login.
  • Verifica cecilia_session_cache no Supabase
  • Se expirou: abre Chromium headless via Playwright
  • Navega para o EPROC do tribunal (TJRS/TJSC/TJSP)
  • Redirect automatico para SSO Keycloak
  • Preenche usuario e senha via keyboard.insertText() (nunca page.fill por causa do @)
  • Gera TOTP via authenticator.generate(totp_secret) com otplib
  • Seleciona perfil ADVOGADO no dropdown
  • Extrai sessionHash da URL do painel
  • Salva sessao no banco para reuso
Status: Funciona
5/5 OABs fazem login com sucesso quando executadas individualmente.
Bug: Session Cache TTL
O campo expires_at esta gravando 1970-01-01 (Unix epoch zero). Consequencia: a sessao SEMPRE e considerada expirada, forcando login completo em TODO ciclo. Sao 5 logins SSO desnecessarios por cron, 15 por dia.
▼
STAGE 2
GERAR PLANILHA — Export XLS do EPROC
skill_gerar_planilha.mjs
Com a sessao ativa, navega ao painel do advogado e exporta a planilha de movimentacoes.
  • Navega para o painel do advogado no EPROC
  • Configura filtro de data: desde o ultimo ciclo bem-sucedido OU MARCO_ZERO (01/04/2026)
  • Clica no botao "Exportar XLS"
  • Aguarda download do arquivo .xls
  • Valida que o arquivo nao esta vazio
Status: Funciona
OK quando o login (Stage 1) foi bem-sucedido. Arquivo chega em 5-15 segundos.
▼
STAGE 3
PROCESSAR PLANILHA — Parse e Filtragem
skill_processar_planilha.mjs
Faz parse do XLS, detecta colunas dinamicamente e aplica 5 camadas de filtro para eliminar ruido.
Arquivo XLS (~200-500 linhas por OAB)
    │
    ├── Deteccao de cabecalho
    │   Coluna A = CNJ (25 chars, formato NNNNNNN-DD.AAAA.J.TR.OOOO)
    │   Coluna C = Autor
    │   Coluna D = Reu
    │   Coluna H = Data evento
    │   Coluna I = Distribuicao
    │   Coluna J = Valor da causa
    │
    ├── Filtro 1: MARCO_ZERO
    │   └── Descarta qualquer evento anterior a 01/04/2026
    │
    ├── Filtro 2: Timestamp per-OAB
    │   └── Pula processos ja analisados NESTE ciclo
    │
    ├── Filtro 3: Timestamp per-Processo
    │   └── Pula se cecilia_ultima_analise_em > data_evento
    │
    ├── Filtro 4: Status
    │   └── Pula processos com status "arquivado"
    │
    ├── Filtro 5: Pre-classificacao de ruido
    │   └── 15+ regex eliminam eventos irrelevantes
    │   └── Ex: "Juntada automatica", "Certidao de decurso de prazo"
    │
    └── Output: lista de processos com movimentacao nova
        Tipicamente 20-40 processos por ciclo (de ~500 linhas iniciais)
Status: Funciona
Filtragem eficiente. De ~500 linhas, restam ~20-40 com movimentacao relevante.
▼
STAGE 4
ACESSAR PROCESSOS — Scraping + Enriquecimento
skill_acessar_processo.mjs
Para cada processo da lista, navega a pagina dele no EPROC e extrai metadados. Este e o stage mais lento.
  • Itera sequencialmente (NUNCA paralelo — anti-bot)
  • Delay aleatorio de 800-2500ms entre cada processo
  • Navega para a pagina do processo no EPROC
  • Extrai: vara, juiz, AJG (justica gratuita), parte contraria, valor da causa
  • Lista eventos com links para documentos PDF
  • Faz UPDATE na tabela processos no Supabase (enriquecimento)
Status: Funciona
OK mas lento. ~3-5 minutos para processar 20-40 processos por causa dos delays anti-bot.
▼
STAGE 5
LER DOCUMENTOS — Extracao de Texto
skill_ler_documento.mjs
Le o conteudo dos documentos judiciais. Tem 5 niveis de fallback para garantir que algo e extraido.
Documento no EPROC
    │
    ├── L1: page.innerText() — funciona para HTML simples
    │   └── Se falhar ↓
    │
    ├── L2: page.evaluate() — extracao manual do DOM
    │   └── Se falhar ↓
    │
    ├── L3: pdf-parse — para documentos PDF
    │   └── Se falhar ↓
    │
    ├── L4: Apache Tika — se instalado no Mac Mini
    │   └── Se falhar ↓
    │
    └── L5: GPT-4.1 Vision — screenshot → OCR (caro, so para criticos)

Dedup: nunca le o mesmo documento (por href) 2x no mesmo ciclo
Status: Funciona
L1-L3 cobrem 95%+ dos documentos. L5 (Vision) raramente necessario.
▼
STAGE 6
ANALISE IA — Classificacao com Sonnet 4.6
skill_analisar_ia.mjs
Envia as movimentacoes em batch para o Claude Sonnet 4.6 com contexto completo do processo e 12 playbooks do vault.
  • Batch de 10 movimentacoes por chamada
  • Carrega contexto do processo no MdFlow (empresa, fase, AJG, polo, historico)
  • Carrega 12 playbooks do cecilia-vault (taxonomia A-G, regras operacionais, treinamento Joao)
  • Chama Sonnet 4.6 com prompt estruturado
  • IA retorna: classe, acao, resumo, urgencia, confianca, prazo_dias, tipo_providencia

Problema critico: Dados da IA nao chegam ao frontend
A IA retorna tudo corretamente dentro do JSONB analise_ia. Porem, o skill NAO copia esses dados para as colunas da tabela intimacoes que o frontend usa para filtrar e exibir: 
analise_ia.resumo          →  resumo_ia = NULL          (0% preenchido)
analise_ia.urgente         →  urgencia = NULL           (0% preenchido)
analise_ia.prazo_dias      →  data_prazo = NULL         (0% preenchido)
analise_ia.tipo_providencia →  tipo_acao = NULL          (0% preenchido)
Sem essa "promocao de campos", o frontend nao consegue filtrar por urgencia, nao tem prazos para monitorar, e nao exibe resumos. Os dados EXISTEM no JSONB, mas estao inacessiveis para queries e views.

4. Pos-Pipeline — Da Cecilia ao Frontend

Depois que o pipeline completa para uma OAB, os dados fluem automaticamente ate o frontend via Supabase Realtime. Nao ha intervencao humana — o advogado abre o /controller e ve as novas intimacoes aparecendo. 

Cecilia Stage 6 completa
    │
    ├── INSERT INTO intimacoes
    │   ├── status = 'nao_lida'
    │   ├── provider = 'cecilia_v3'
    │   ├── classe_cecilia = 'B'
    │   ├── analise_ia = { ...JSONB completo... }
    │   ├── data_prazo = NULL ← PROBLEMA (deveria ter valor)
    │   ├── urgencia = NULL  ← PROBLEMA
    │   └── resumo_ia = NULL ← PROBLEMA
    │
    ├── UPDATE cecilia_ciclos
    │   └── status, duracao_s, processos_verificados, intimacoes_criadas
    │
    ├── UPDATE processos (enriquecimento)
    │   └── ultima_movimentacao, cecilia_ultima_analise_em
    │
    ├── Telegram (classes E/F apenas)
    │   └── Notificacao imediata para grupo do escritorio
    │
    ▼
v_controller_intimacoes_completa (VIEW)
    │   JOIN intimacoes + processos + clientes + empresas + advogados + decisoes
    │   + Campos calculados: dias_ate_prazo, status_prazo
    │
    ▼
fn_controller_v5_kpis() (RPC)
    │   Retorna: { nao_lidas: 14, urgentes: 1, execucoes: 0, com_deposito: 0 }
    │
    ▼
Supabase Realtime Channel
    │   postgres_changes → INSERT/UPDATE em intimacoes
    │   → Evento propagado para todos os clientes conectados
    │
    ▼
Frontend /controller (React)
    ├── React Query invalidate → re-fetch automatico
    ├── KPI Cards: nao lidas, urgentes, execucoes, com deposito
    ├── Lista Virtualizada: 50 items/page, infinite scroll
    ├── Drawer Detalhado: 6 tabs (resumo, processo, docs, chat, historico, analise)
    └── Bulk Actions: marcar lida, finalizar, criar atividade
Ciclo de vida de uma intimacao
nao_lida  ───>  lida  ───>  em_andamento  ───>  finalizada
                                     │
                                     └───>  arquivada (sem acao necessaria)

Cada transicao e feita pelo advogado no frontend. O Realtime propaga a mudanca para todos os usuarios conectados.
