01 · Contexto

Por que este sistema existe

O problema que nenhum sistema no mercado resolve de forma confiável.

⏰

O prazo não pode falhar

Um advogado que perde um prazo processual pode perder o processo, responder disciplinarmente perante a OAB e sofrer ação de reparação civil. É o maior risco da advocacia.

📋

EPROC é a fonte da verdade

O sistema e-Proc do CNJ é o único repositório oficial das movimentações processuais nos tribunais federais e estaduais participantes. Qualquer delay na leitura desse sistema é delay de prazo.

🐌

Mercado tem 18 dias de delay

APIs como DataJud têm delay de até 18 dias na captura de movimentações. Para controle de prazos isso é inutilizável. O escritório precisa de movimentações com no máximo 24h de delay.

💡 A oportunidade
Após 18 meses construindo o sistema internamente, com 5.800 processos monitorados em 3 tribunais, identificamos que a mesma infraestrutura pode ser transformada em uma API comercial que atende qualquer escritório de advocacia. O diferencial: captura real de movimentações diretamente do EPROC, com delay máximo de 24h e análise IA das decisões.

S0 · PLAN — Inicialização

Carrega o vault de playbooks jurídicos (~50.801 chars de contexto), lista OABs ativas do banco, cria registro na tabela cecilia_ciclos e inicializa o orçamento do ciclo.

// Vault loader — carrega playbooks do filesystem
const playbooks = await loadVault('/cecilia-vault/02-PLAYBOOKS/');
// ~50.801 chars de contexto jurídico injetado no S6

S1 · LOGIN — Autenticação EPROC

Usa Playwright para navegar ao EPROC do tribunal, fazer login via Keycloak SSO (usuário = número OAB, senha), e resolver 2FA via TOTP com otplib quando solicitado. Detecta seletor de perfil (Advogado / Advogado Titular) e escolhe o correto.

// TOTP com otplib
import { authenticator } from 'otplib';
const token = authenticator.generate(totp_secret);
await page.fill('#otp', token);

S2 · SCRAPE — Captura de movimentações

Navega para Relatório de Processos no EPROC, baixa a planilha XLS de "Últimas Movimentações" com filtro de 3 dias, processa cada linha e insere novos eventos na tabela intimacoes. Integra também com DJEN (Diário de Justiça Eletrônico Nacional) para captura paralela.

// Download XLS via Playwright
await page.click('#btnGerarPlanilha');
const download = await page.waitForEvent('download');
// Parse XLS → array de eventos
// Coluna A=CNJ, C=autor, H=data evento, J=valor causa

S3 · FILTER — Dedup e Triagem

Elimina duplicatas via SHA256 do hash (CNJ + data + título). Aplica IRRELEVANT_KEYWORDS (~80 padrões) para classificar automaticamente como classe A sem custo de IA. Ratio atual: ~74% triados sem IA.

// Pre-classify — exemplos de keywords ignoradas:
const IRRELEVANT = [
  'DECORRIDO PRAZO', 'CONFIRMADA A INTIMAÇÃO ELETRÔNICA',
  'JUNTADA DE CARTA PELO CORREIO', 'MERO EXPEDIENTE',
  'ALTERADO O ASSUNTO PROCESSUAL', 'BOLETO CANCELADO'
  // ~80 padrões no total
];

S4 · NAVIGATE — Acesso ao processo

Para cada intimação que requer leitura de documento, navega ao processo pelo número CNJ usando o campo de busca do EPROC. Detecta o polo do cliente (autor/réu) via DOM para contextualizar a análise IA.

// Navegação por CNJ — CUIDADO: form.submit() é anti-pattern
// Correto:
await Promise.all([
  page.waitForNavigation({ waitUntil: 'domcontentloaded' }),
  page.locator('form[name="consultarProcesso"]').evaluate(f => f.submit())
]);

S5 · READ_DOCS — Leitura de documentos

Sistema de 5 níveis de fallback para extrair o texto dos documentos processuais. Prioriza DOM direto, depois iframes, depois PDF download, depois OCR com Claude Vision.

// 5-Level Fallback:
// L1: .infraAreaTexto (DOM direto — rápido)
// L2: 15+ seletores EPROC + iframes
// L3: texto da linha do evento (fallback contextual)
// L4: Download PDF + pdf-parse
// L5: Screenshot + Claude Vision OCR (último recurso)

// ALWAYS_READ (lê independente do pre-classify):
// sentença, decisão, despacho, acórdão, tutela, liminar

S6 · ANALYZE — Análise Claude Sonnet 4.6

Envia o documento + contexto do processo + playbooks jurídicos para o Claude Sonnet 4.6. Recebe classificação A-G, urgência, prazo detectado, dispositivo da decisão e atividade sugerida. Custo médio: ~$0.059 por item.

// Budget cap — verifica antes de cada chamada:
if (custoAcumulado >= BUDGET_CAP) {
  console.log('Budget cap atingido — abortando S6');
  break;
}
// Auto-loop: Claude revisa a própria análise antes de retornar

S7 · PERSIST — Gravação no banco

Upsert na tabela intimacoes com todos os campos preenchidos. Guard especial para classe A: só grava se confiança >= 0.95. Dedup por signature_hash impede duplicatas.

// Schema relevante da tabela intimacoes:
// ia_status: 'pendente'|'processando'|'concluido'|'erro'
// classe_cecilia: 'A'..'G'
// urgencia: 'imediata'|'alta'|'normal'|'informativa'
// analise_ia: JSONB (prazo, dispositivo, atividade_sugerida)
// conteudo_completo: TEXT (documento lido pelo S5)
// signature_hash: SHA256 para dedup

S8 · ACT — Ações automáticas

Classes E e F disparam Telegram imediato. Classe F (decisão favorável) popula tabela merito_julgamentos. Classe A com confiança alta é auto-finalizada. Classes B-D aguardam validação humana no frontend.

S9 · REPORT — Relatório e orçamento

Gera sumário do ciclo (itens processados, custo total, ratio PRE/IA), atualiza cecilia_ciclos com status final, libera PID lock e encerra o processo.

04 · Fluxo EPROC

Navegação passo a passo no EPROC

Capturas reais do sistema. Cada step representa o que o agente (e a API) deve executar.

1
Login — Autenticação SSO Keycloak
A página de login do EPROC usa o protocolo Keycloak SSO com dois campos: usuário (número da OAB, ex: RS116571) e senha. Após o submit, o sistema pode ou não solicitar autenticação de dois fatores (TOTP). O 2FA aparece de forma intermitente — o sistema não solicita em todo login, apenas periodicamente ou após novos dispositivos.

O que a API deve fazer nesta etapa

  • Navegar para a URL do tribunal (diferente por tribunal: TJRS, TJSC, TJSP)
  • Preencher campo username com o número da OAB
  • Preencher campo password com a senha
  • Submeter o formulário e aguardar redirect
  • Detectar se a página seguinte é o 2FA ou o dashboard
  • Se 2FA: gerar código TOTP com otplib.authenticator.generate(secret) e submeter
  • Após login bem-sucedido: verificar se aparece seletor de perfil
  • Possível seletor: "Advogado" vs "Advogado Titular" (depende do histórico do advogado no sistema)
⚠️ Variabilidade por advogado
Cada advogado pode ter configurações diferentes no EPROC. Alguns têm apenas 1 perfil, outros têm "Advogado" e "Advogado Titular" como dois acessos separados. Processos podem estar distribuídos entre os dois. A API deve verificar todos os perfis disponíveis e consolidar.
2
Painel do Advogado — Tela principal pós-login
Após o login bem-sucedido, o sistema cai no Painel do Advogado. Esta tela mostra resumos de citações, intimações, audiências e pendências. Para o monitoramento de prazos, NÃO é aqui que o sistema busca os dados — é necessário navegar para o Relatório de Processos.

O que a API deve fazer nesta etapa

  • Confirmar que o login foi bem-sucedido verificando a presença do título "Painel do Advogado"
  • Registrar a sessão ativa (cookies/tokens)
  • Navegar imediatamente para Relatórios → Relatório de Processos no menu lateral
  • Alternativa rápida: usar menu lateral > "Últimas movimentações" — que é mais direto
  • IMPORTANTE: nosso sistema usa "Últimas movimentações", NÃO o Painel do Advogado
🚫 Armadilha conhecida
O Painel do Advogado tem contadores de processos com prazo, mas esses dados são agregados e não servem para varredura. É preciso ir para o Relatório de Processos para ter a lista completa ordenada por data de movimentação.
3
Relatório de Processos — Lista de movimentações
Esta é a tela central do sistema. O Relatório de Processos exibe uma tabela com todos os processos que tiveram movimentação, ordenável por data. Cada linha contém: número CNJ, classe, autores, réus, comarca, assunto, último evento e data. O sistema exporta essa tabela como XLS via botão "Gerar Planilha".

O que a API deve fazer nesta etapa

  • Navegar para Relatório de Processos via menu lateral
  • Aplicar filtro de data: últimos 1-3 dias (nunca mais que 7 — evita timeout)
  • Ordenar por "Data/Hora do Último Evento" descendente
  • Clicar em "Gerar Planilha" para download do XLS
  • Alternativamente: paginar a tabela HTML (50 resultados/página) e scraping direto
  • Colunas relevantes: CNJ (col A), Último Evento (col H), Data da Atualização (col I)

Estrutura do XLS (colunas mapeadas)

  • Col A (25 chars): Número CNJ completo
  • Col C: Nome do autor principal
  • Col D: Nome do réu principal
  • Col H: Data do último evento
  • Col I: Data de distribuição
  • Col J: Valor da causa
💡 DJEN como alternativa / complemento
O Diário de Justiça Eletrônico Nacional (DJEN) publica diariamente as intimações por número de OAB. Fazer scraping do DJEN às 00h e usar essa lista para navegar diretamente aos processos no EPROC é uma abordagem complementar que pode reduzir o volume de processos a verificar. O DJEN tem delay de 1 dia mas é uma fonte estruturada.
4
Página do Processo — Árvore de eventos
Ao clicar em um processo, abre a Consulta Processual com dados completos: partes, representantes, juiz, vara, situação, e a árvore de eventos cronológica. Esta é a tela mais crítica do sistema — cada linha da árvore é um evento processual, com data, descrição e link para documentos. O agente deve identificar quais eventos são destinados ao polo que representa.

O que a API deve fazer nesta etapa

  • Navegar ao processo pelo número CNJ (campo de busca no topo ou via URL)
  • Extrair dados das partes: autor, réu, advogados, CPF/CNPJ
  • Identificar qual polo o advogado representa (autor ou réu)
  • Ler a árvore de eventos: cada evento tem ID, data, descrição, documentos
  • Filtrar eventos relevantes: intimações, decisões, sentenças, despachos
  • Identificar se o evento tem documento vinculado (ícone/link de PDF)
  • Para eventos relevantes: acessar o documento (próximo step)

Estrutura da árvore de eventos (DOM)

  • Seletor principal: tabela com class infraTable ou similar
  • Cada linha: ID evento, data/hora, descrição, usuário, coluna de documentos
  • Eventos com PDF: coluna de documentos tem link clicável ou ícone
  • Eventos de intimação: têm flag visual (envelope) e prazo calculado
  • Prazos: exibidos inline na descrição do evento ("Prazo: 15 dias")
🚫 Desafio de URL — Hash de autenticação
A URL do EPROC para um processo específico contém hashes de sessão que expiram. Não é possível acessar diretamente via URL sem sessão autenticada. O sistema precisa navegar a partir da sessão ativa, não construir URLs do zero.

https://eproc1g.tjrs.jus.br/eproc/controlador.php?acao=processo_selecionar&num_processo=5036746-53.2024.8.21.0008&hash=XXXXX

O hash muda a cada sessão e não pode ser previsto externamente.
5
Documento do Processo — Leitura da decisão
Ao acessar um documento (sentença, decisão, despacho), o EPROC pode exibir o conteúdo de três formas: texto direto no DOM via .infraAreaTexto, dentro de um iframe, ou como PDF para download. A API precisa implementar os 5 níveis de fallback para garantir extração em todos os casos.

O que a API deve fazer nesta etapa

  • L1 — DOM direto: tentar document.querySelector('.infraAreaTexto')?.innerText
  • L2 — Múltiplos seletores: testar 15+ seletores alternativos do EPROC
  • L3 — Iframe: se há iframe, trocar contexto para o frame e repetir
  • L4 — PDF: se botão de download PDF existe, baixar e usar pdf-parse
  • L5 — OCR: screenshot da área + Claude Vision (Haiku) como último recurso
  • Truncar em maxChars por tipo: sentença 10k, decisão 8k, despacho 3k

Tipos de documento com prioridade máxima (ALWAYS_READ)

  • Sentença — decisão final do processo
  • Acórdão — julgamento em 2ª instância
  • Decisão — despacho do juiz com efeitos
  • Tutela de urgência / liminar
  • Decisão interlocutória
  • Despacho com prazo
05 · Desafios Técnicos

Problemas que já encontramos (e como resolvemos)

18 meses de desenvolvimento produziram uma lista detalhada de armadilhas. Economiza meses de debugging.

CRÍTICO

Playwright crasha sem contexto GUI

No macOS, rodar sudo -u cecilia node script.mjs no terminal causa SIGTRAP no Chromium. Não funciona. Ponto.

Solução validada

  • Usar LaunchAgent registrado no gui/UID do usuário
  • No Linux (servidor), usar display virtual com Xvfb
  • Em Docker: imagem playwright com Xvfb incluído
  • Comando: launchctl bootstrap gui/506 /path/to/agent.plist
CRÍTICO

form.submit() destrói o contexto

Usar page.evaluate(() => form.submit()) causa erro "Target page closed" porque o form.submit() navega a página antes do evaluate retornar.

// ❌ Anti-pattern — NÃO fazer:
await page.evaluate(() => form.submit());

// ✅ Correto:
await Promise.all([
  page.waitForNavigation({waitUntil:'domcontentloaded'}),
  page.locator('form').evaluate(f => f.submit())
]);
ALTO

Sessão EPROC expira mid-pipeline

O EPROC tem timeout de sessão agressivo. No meio do processamento de uma lista de processos, a sessão pode expirar causando erro em toda a fila.

Solução

  • Session Monitor: verifica page.url() antes de cada operação
  • Se URL mudou para login: executa re-login automático
  • Mantém referência ao browser/page e reconecta transparentemente
  • Fallback: fechar browser inteiro e abrir nova sessão
ALTO

Janela temporal muito larga = timeout

Capturar 30 dias de histórico num único ciclo gera centenas de eventos. O processamento de cada evento consome tempo e o ciclo estoura o timeout de 30-60 minutos.

Solução

  • Janela padrão: 3 dias máximo por ciclo
  • Modo "harvest": ciclo especial para processar backlog histórico
  • Dedup via SHA256: eventos já processados são ignorados
  • Budget cap: limite de $2 por ciclo (abandona quando atingido)
ALTO

URL com hash de sessão impossibilita acesso direto

A URL de um processo específico no EPROC contém tokens de sessão que expiram. Não é possível montar uma URL e acessar diretamente sem navegação autenticada prévia.

Alternativas investigadas

  • Usar número CNJ no campo de busca (funciona, mas lento)
  • API interna do EPROC via engenharia reversa (parcialmente viável)
  • DJEN + acesso direto combinado (mais promissor)
  • Manter sessão ativa e reutilizar o contexto do Painel
ALTO

Seletor de perfil varia por advogado

Alguns advogados têm "Advogado" e "Advogado Titular" como dois cadastros distintos. Processos podem estar em ambos. Se o sistema seleciona só um, perde processos do outro.

Solução

  • Detectar na tela pós-login se há seletor de perfil
  • Processar cada perfil em sequência
  • Unificar os resultados antes de persistir
  • Configurar por advogado qual perfil usar (ou todos)
MÉDIO

2FA intermitente

O EPROC não solicita 2FA em todos os logins — apenas periodicamente ou após mudanças de IP/dispositivo. O sistema deve detectar a tela de 2FA e agir apenas quando ela aparece.

Solução

  • Após submit do login, verificar qual tela aparece
  • Se URL contém "totp" ou "otp": gerar código e submeter
  • Se URL é o dashboard: login completo sem 2FA
  • Secret TOTP armazenado de forma segura (variáveis de ambiente)
MÉDIO

Documentos em iframes aninhados

Muitos documentos do EPROC são renderizados dentro de iframes. O seletor que funciona para uma versão do tribunal pode não funcionar para outra.

Solução

  • 5-level fallback com 15+ seletores CSS testados
  • Tentar todos os frames disponíveis na página
  • Fallback para PDF download quando disponível
  • OCR com Claude Vision como último recurso

Modelo de dados multi-tenant

-- Tabela de escritórios (tenants)
CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  nome TEXT NOT NULL,
  api_key TEXT UNIQUE NOT NULL,
  plano TEXT CHECK (plano IN ('basic', 'pro', 'enterprise')),
  max_oabs INTEGER DEFAULT 2,
  max_processos INTEGER DEFAULT 500,
  webhook_url TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Tabela de OABs cadastradas por tenant
CREATE TABLE oabs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id),
  numero TEXT NOT NULL,           -- ex: RS116571
  tribunal TEXT NOT NULL,         -- TJRS | TJSC | TJSP
  login TEXT NOT NULL,
  senha_encrypted TEXT NOT NULL,  -- AES-256 encrypted
  totp_secret_encrypted TEXT,     -- null se sem 2FA
  ativo BOOLEAN DEFAULT true,
  ultima_sincronizacao TIMESTAMPTZ,
  falhas_consecutivas INTEGER DEFAULT 0
);

-- Tabela de movimentações capturadas
CREATE TABLE movimentacoes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL,
  oab_id UUID REFERENCES oabs(id),
  cnj TEXT NOT NULL,
  tribunal TEXT NOT NULL,
  evento_id TEXT,                 -- ID interno EPROC
  data_evento TIMESTAMPTZ NOT NULL,
  titulo TEXT NOT NULL,           -- descrição do evento
  polo TEXT CHECK (polo IN ('autor', 'reu', 'desconhecido')),
  conteudo_documento TEXT,        -- texto extraído do doc
  -- Análise IA (opcional, tier pro+)
  classe TEXT CHECK (classe IN ('A','B','C','D','E','F','G')),
  urgencia TEXT,
  prazo_detectado TIMESTAMPTZ,
  dispositivo TEXT,               -- dispositivo da decisão
  atividade_sugerida TEXT,
  -- Controle
  status TEXT DEFAULT 'nova',     -- nova | lida | arquivada
  signature_hash TEXT UNIQUE,     -- SHA256 dedup
  created_at TIMESTAMPTZ DEFAULT now()
);

08 · Agente Orquestrador

O orquestrador — Cecília como referência

O "Super Agente" que coordena todo o pipeline. Eduardo mencionou a ideia de orquestrador — é exatamente o que temos.

✅ Já construído e validado
O orquestrador já existe como o arquivo controller-agent.mjs da Cecília. O que precisamos fazer para a API é extrair essa lógica de orquestração e torná-la multi-tenant — capaz de gerenciar múltiplos escritórios e OABs simultaneamente.

Responsabilidades do orquestrador

Agendamento

Roda automaticamente 3x/dia (6h, 13h, 19h). Distribui as OABs em sequência com intervalo de 5-10 minutos entre cada uma para evitar ban. Suporta ciclos manuais (trigger via API).

Gestão de sessões

Mantém um pool de sessões Playwright ativas por tribunal. Detecta sessões expiradas e re-autentica automaticamente sem interromper o ciclo. PID lock evita execuções paralelas.

Circuit breaker

Se uma OAB falhar N vezes consecutivas (senha errada, bloqueio do tribunal), o circuit breaker isola essa OAB e notifica o tenant. O resto do ciclo continua normalmente.

Budget cap

Cada tenant tem um orçamento de IA por ciclo. Quando atingido, os itens restantes são enfileirados para o próximo ciclo. Itens triados sem IA continuam sendo processados gratuitamente.

Vault de conhecimento

Carrega playbooks jurídicos (por especialidade do escritório) antes de iniciar a análise IA. Isso garante que o Claude entende o contexto legal específico de cada tipo de processo.

Relatório e alertas

Ao final de cada ciclo, gera sumário (processos verificados, custo, prazo iminentes) e dispara alertas via webhook/Telegram para movimentações urgentes (classes E e F).

Arquitetura do orquestrador para a API

// Stack sugerida para produção:
// - Node.js com BullMQ (Redis-backed job queue)
// - Playwright cluster com N workers paralelos
// - PostgreSQL multi-tenant com Row Level Security
// - Redis para sessões, cache e fila
// - Docker com Xvfb para Playwright headless

// Fluxo do Job Queue:
// 1. Scheduler cria jobs "sync_oab" para cada OAB ativa
// 2. Worker pega job da fila → executa S1-S9
// 3. Após conclusão: dispara webhooks configurados
// 4. Falha: retry com backoff exponencial (3x)

class OrchestratorAPI {
  async scheduleAllTenants() {
    const oabs = await db.query('SELECT * FROM oabs WHERE ativo = true');
    for (const oab of oabs) {
      await queue.add('sync_oab', { oab_id: oab.id }, {
        delay: oab.index * 5 * 60 * 1000,  // stagger 5min entre OABs
        attempts: 3,
        backoff: { type: 'exponential', delay: 30000 }
      });
    }
  }
  
  async processOAB(oab) {
    const session = await this.sessionPool.acquire(oab.tribunal);
    try {
      await s1_login(session, oab);
      const events = await s2_scrape(session, oab, { dias: 3 });
      const filtered = await s3_filter(events, oab.tenant_id);
      for (const event of filtered) {
        if (event.requiresDoc) {
          await s4_navigate(session, event.cnj);
          event.documento = await s5_readDoc(session);
        }
        if (event.requiresIA) {
          event.analise = await s6_analyze(event, oab.tenant_config);
        }
        await s7_persist(event, oab.tenant_id);
        await s8_act(event, oab.tenant_config);
      }
    } finally {
      this.sessionPool.release(session);
    }
  }
}

Fila de intimações por urgência
