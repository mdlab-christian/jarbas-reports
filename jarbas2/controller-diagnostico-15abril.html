Bugs Identificados — Análise de Causa Raiz

P0-A S5 (read_docs) NUNCA chamado — Classificação cega
A IA classifica eventos com apenas o título, sem conteudo_completo. Resulta em 100% dos eventos classificados como A ou G, sem raciocínio registrado (analise_ia={}).
Prova: 1.198 registros classe A com conteudo_completo=NULL e analise_ia={}. "Comunicação eletrônica recebida - julgado" (22 casos) podem ser sentenças. "Confirmada a citação eletrônica" (29 casos) = prazos para contestar. Perdidos silenciosamente.
Root cause: S3 (pre-classify.mjs) classifica como A antes de S5. S5 só é chamado quando read_doc=true, que só ocorre para C/D/E/F. Como tudo vira A sem documento, S5 nunca roda. Lógica invertida: "dúvida = descarta como A" em vez de "dúvida = lê o doc".
P0-B TJSC — Zero análises em 2 ciclos completos (734 processos)
SC074025 (687 processos) e SC075466 (47 processos) detectam movimentações mas processos_analisados=0 nos dois ciclos. 57min + 65min + 71min de execução desperdiçados.
Prova: Ciclo 09h — SC074025: 597 com movimentação, 0 analisados. Ciclo 16h — SC074025: 596 com movimentação, 0 analisados. Padrão idêntico nas duas rodadas.
Root cause provável: XLS scraping TJSC retorna movimentações com formato diferente do TJRS. Títulos dos eventos TJSC podem ter acentuação, capitalização ou encoding diferente, fazendo pré-classify filtrar tudo para a lista IGNORE. Impacto: 734 processos sem nenhum monitoramento efetivo.
P0-C TJSP permanentemente travado — Playwright sem timeout efetivo
SP535101 iniciou às 09:00 e ainda aparecia como "executando" às 16:00 (7+ horas depois). O ciclo das 16h iniciou novo registro SP535101 como "executando" sem verificar status do ciclo anterior.
Prova: 2 registros em cecilia_ciclos para SP535101 com status="executando" simultaneamente. 0 processos verificados em ambos os registros.
Root cause provável: Playwright travado numa página TJSP. Timeout de 30min por OAB em start-cron.mjs não está funcionando ou PID lock impede o kill. O segundo ciclo iniciou sem verificar se o primeiro terminou. Impacto: 100% dos processos TJSP (~450) sem monitoramento no dia inteiro.
P1-A Budget cap violado sem enforcement incremental
RS131163 às 09h gastou $2.42 vs. budget cap de $2.00 — violação de 21%.
Prova: custo_total=$2.42 no registro cecilia_ciclos para RS131163 09h.
Root cause: Budget cap verificado em S9 após o ciclo terminar, não durante. Sem verificação incremental em S6 antes de cada chamada Claude, o custo pode extrapolar livremente em ciclos com muitos processos.
P1-B 882 itens do provider "jarbas" nunca processados — 42% das intimações do dia
Sistema paralelo Jarbas/TJSC insere eventos na tabela intimacoes com provider="jarbas". O pipeline Cecília não processa esses itens.
Prova: 882 registros com ia_status="pendente", provider="jarbas", todos criados hoje, deleted_at IS NULL.
Root cause: Cecília filtra por provider="cecilia" no início de cada ciclo. Nunca foi implementado modo de reprocessamento para itens inseridos por outros agentes.
P1-C polo_detectado = "desconhecido" em 100% dos casos
Sem polo correto, é impossível saber se "Juntada de Petição" é nossa petição (irrelevante → A) ou do adversário (requer resposta → D). A classificação contextual fica comprometida.
Root cause: S4 (navigate) não implementa detecção de polo. O campo polo_detectado é gravado como string literal "desconhecido" sem tentativa de verificar CPF/CNPJ nos polos do processo.
P2-A DJEN descartado por filtro — intimações formais com prazo perdidas
Filtro DJEN_FILTER_KEYWORDS em pre-classify.mjs foi criado para reduzir ruído, mas 378 "Publicado no DJEN" + 45 "Disponibilizado no DJEN" passaram como A sem leitura de documento. O teor das intimações DJEN nunca é verificado.
Impacto: Se houver prazo numa dessas intimações, ele é perdido silenciosamente sem nenhum alerta. DJEN = intimações formais com prazo D+3 úteis — as mais importantes do sistema.

Log verboso em S3 para TJSC: quantos eventos filtrados, por qual motivo, qual keyword casou. Hipótese principal: títulos TJSC com acentuação/capitalização diferente caem na lista IGNORE. Validar com 5 processos TJSC manualmente antes de corrigir.

S6 verifica custo acumulado do ciclo antes de cada chamada Claude. Se > $1.80, pausar e marcar ciclo como "budget_cap_atingido" para review manual.

Separar responsabilidades: S3 output = read_doc: bool + candidato: 'A' | 'provável_relevante' | 'indefinido'. S5 roda para todo candidato diferente de 'A'. S6 recebe conteudo_completo e classifica definitivamente A–G.

DJEN — Análise de Viabilidade Técnica

Veredicto
Vale a pena — Resolve 3 problemas simultaneamente
1. Completude — 100% das intimações formais, inclusive as que EPROC pode perder
2. Prazos automáticos — dataPublicacao oficial → D+3 úteis calculado sem scraping
3. Conteúdo sem Playwright — Texto completo da intimação via API estruturada
Limitações
Cadastro por OAB — CPF de cada advogado registrado no DJEN separadamente
Rate limiting — API pode ter limites de requisições por hora
Escopo limitado — Apenas intimações formais, não movimentações processuais gerais
Complementar — Não substitui EPROC scraping, apenas enriquece
Posicionamento na Arquitetura — 3 Fontes Complementares
EPROC (Playwright)
Movimentações processuais completas. Histórico integral. Scraping via Playwright — frágil mas insubstituível para dados históricos completos.
DJEN (API REST)
Intimações formais com prazo D+3 oficial. Dados estruturados, sem scraping. Cobre todos os tribunais. Texto completo via API. Nova fonte prioritária.
Datajud (API CNJ)
Metadados do processo: juiz, vara, valor da causa, movimentações públicas. Complementa dados de polo e enriquece contexto para análise IA.
Os três juntos formam cobertura quase completa sem depender exclusivamente de scraping frágil. Cobertura estimada: 95%+ das intimações relevantes.
