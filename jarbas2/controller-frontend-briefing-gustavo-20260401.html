1. TL;DR — Resumo Executivo

O que ja existe (backend completo)

O mecanismo do Controller v3 foi 100% implementado. Isso inclui:

  • 8 RPCs com SECURITY DEFINER, soft-delete compliance, multi-tenant
  • View completa v_controller_intimacoes_completa com 65+ colunas, JOINs, campos calculados
  • 11 migrations aplicadas (status enum, IA columns, playbooks, ciencia com renuncia)
  • 8 workflows n8n ativos (ingest, reprocessar, transfer, export, heartbeat, retry)
  • Edge Function analyze-intimacao (classificacao IA com Haiku 4.5 + Sonnet 4.6)
  • Cecilia v3 monitorando 5 OABs em 3 tribunais (TJRS, TJSC, TJSP)
O que o frontend precisa fazer
  • Consumir a view v_controller_intimacoes_completa via Supabase SDK
  • Chamar RPCs via supabase.rpc('nome_da_rpc', { params })
  • Disparar webhooks n8n via POST quando necessario
  • Tratar o JSONB analise_ia para exibir acao recomendada, resumo, flags
  • NUNCA escrever logica de negocio no frontend — tudo via RPC ou n8n
⚠ REGRA CRITICA: O frontend e passivo. Ele exibe dados da view, chama RPCs para acoes, e executa response.action de webhooks. Nunca decide fluxo por conta propria (Action-Driven UI).

2. 9 Premissas Inviolaveis

Estas premissas sao regras de negocio absolutas. Violar qualquer uma = bug bloqueante.
#PremissaImpacto no Frontend
P1 Fluxo de Status: nao_lida → lida → finalizada Nunca exibir "arquivar". Nunca status "processada" ou "ignorada".
P2 Lida = apenas visual. Card permanece na lista. Mudar font-weight de semibold→normal. NAO remover da lista.
P3 Finalizada = visual. line-through + opacity 0.4. Card fica visivel (strikethrough + fade) ate refresh. Nao some.
P4 ZERO prazo nos cards da lista. Prazo aparece SOMENTE no drawer, KPIs, e aba Equipe. NUNCA no card da intimacao.
P5 Tags coexistem: urgente + importante simultaneamente. Dot vermelho + ring amarelo quando ambos ativos.
P6 IA/Classes OCULTAS do usuario. NUNCA exibir classe A-G, confianca IA %, score. Sao dados internos.
P7 Sugestao de fase via dialog, nunca auto. Ao criar atividade que sugere mudanca de fase → dialog "Mover processo para [Financeiro]?"
P8 Ciencia com Renuncia SEMPRE requer confirmacao humana. Dialog de confirmacao obrigatorio. Nunca auto-executar.
P9 5 OABs parametrizadas Filtro advogado deve incluir: RS116571, SC074025, SP535101, RS131163, SC075466.

5. View Principal — v_controller_intimacoes_completa

Esta e a UNICA fonte de dados para a aba Intimacoes. O frontend faz supabase.from('v_controller_intimacoes_completa').select('*') e recebe tudo pronto.
Colunas disponiveis (65+)

Conteudo titulo, tipo, tipo_evento, conteudo, conteudo_completo TEXT
Status status (nao_lida|lida|finalizada|erro|transferida), ia_status (pendente|processando|concluido|erro) TEXT CHECK
Flags urgente, importante, is_duplicada BOOLEAN
IA Analysis classe_cecilia (A-G), confianca_ia (0-1), analise_ia (JSONB), resumo_ia, ia_acao_recomendada, ia_risco_nivel (baixo|medio|alto|critico), motivo_ia, ia_prazo_sugerido Diversos

Prazo data_prazo, dias_ate_prazo (calculado), status_prazo (sem_prazo|vencido|hoje|critico|atencao|normal) DATE / INT / TEXT

6. RPCs — Contratos de API Backend

Todas as RPCs sao SECURITY DEFINER com deleted_at IS NULL compliance. Chamar via supabase.rpc('nome', { params }).
1. rpc_controller_batch_update_status

Atualiza status de multiplas intimacoes (bulk). Usado pelo BulkBar.

// Assinatura rpc_controller_batch_update_status( p_organizacao_id: UUID, p_intimacao_ids: UUID[], // max 100 p_novo_status: TEXT, // 'nao_lida' | 'lida' | 'finalizada' | 'erro' | 'transferida' p_actor_user_id: UUID ) // Retorno { "success": true, "updated": 3, "status": "lida" } // Exemplo frontend const { data } = await supabase.rpc('rpc_controller_batch_update_status', { p_organizacao_id: orgId, p_intimacao_ids: selectedIds, p_novo_status: 'lida', p_actor_user_id: userId });

⚠ Se p_novo_status = 'finalizada', a RPC automaticamente seta concluida_em e concluida_por_id.

2. rpc_controller_finalizar

Finaliza uma unica intimacao. Usado pelo botao "Finalizar" no drawer.

rpc_controller_finalizar( p_intimacao_id: UUID, p_user_id: UUID ) // Retorno { "success": true } // ou { "success": false, "error": "not_found" }
3. rpc_controller_criar_atividades_lote

Cria atividades em lote a partir de intimacoes selecionadas. Extrai sugestao da IA automaticamente.

rpc_controller_criar_atividades_lote( p_organizacao_id: UUID, p_intimacao_ids: UUID[], p_actor_user_id: UUID ) // Retorno { "success": true, "total_criadas": 5, "total_ignoradas": 2, // status ja finalizada/erro "ids_criados": ["uuid1", "uuid2", ...] } // Comportamento: // - Pula intimacoes com status 'finalizada' ou 'erro' // - Extrai titulo de analise_ia.atividade_sugerida // - Cria link em intimacoes_atividades (bridge table) // - Seta status da intimacao para 'lida'
4. rpc_controller_criar_atividade

Cria UMA atividade a partir de UMA intimacao. Usado pelo SheetAtividade.

rpc_controller_criar_atividade( p_organizacao_id: UUID, p_intimacao_id: UUID, p_titulo: TEXT, p_descricao: TEXT, p_responsavel_id: UUID, // organizacoes_usuarios.id p_processo_id: UUID, p_prazo: DATE, p_criar_participante: BOOLEAN DEFAULT true, p_tipo: UUID DEFAULT NULL, // tipo da atividade p_actor_user_id: UUID DEFAULT NULL, p_data_audiencia: DATE DEFAULT NULL ) // Retorno { "success": true, "atividade_id": "uuid", "participante_criado": true }
5. rpc_controller_transferir_lote

Transfere intimacoes para outro membro da equipe. Cria atividade automatica.

rpc_controller_transferir_lote( p_organizacao_id: UUID, p_intimacao_ids: UUID[], p_destino_user_id: UUID, // organizacoes_usuarios.id p_actor_user_id: UUID DEFAULT NULL, p_prazo_atividade: DATE DEFAULT NULL ) // Retorno { "success": true, "transferidas": 3, "atividades_criadas": 3 }
6. rpc_controller_criar_doc_pendente

Cria documento pendente vinculado a intimacao. Usado pelo SheetDocPendente.

rpc_controller_criar_doc_pendente( p_organizacao_id: UUID, p_intimacao_id: UUID, p_processo_id: UUID, p_cliente_id: UUID DEFAULT NULL, p_solicitante_id: UUID DEFAULT NULL, p_tipo: TEXT DEFAULT NULL, p_prazo: DATE DEFAULT NULL, p_actor_user_id: UUID DEFAULT NULL, p_urgente: BOOLEAN DEFAULT false, p_responsavel_id: UUID DEFAULT NULL ) // Retorno { "success": true, "doc_pendente_id": "uuid" }
7. rpc_controller_registrar_feedback_ia

Registra feedback do usuario sobre a classificacao IA. Usado para treino.

rpc_controller_registrar_feedback_ia( p_organizacao_id: UUID, p_intimacao_id: UUID, p_aprovado: BOOLEAN, // true = IA acertou, false = errou p_motivo: TEXT, // texto livre do usuario p_user_id: UUID ) // Retorno { "success": true }
8. rpc_controller_retry_falhas

Re-enfileira intimacoes com erro para reprocessamento IA. Usado no Dashboard.

rpc_controller_retry_falhas( p_organizacao_id: UUID, p_ids: UUID[] DEFAULT NULL, // null = todas com erro p_max_retries: INTEGER DEFAULT 3 ) // Retorno { "success": true, "reenfileiradas": 5 }
BONUS: rpc_controller_v3_kpis (para KPI cards)

Retorna todos os contadores para os KPI cards de uma vez.

rpc_controller_v3_kpis(p_org_id: UUID) // Retorno { "total": 1247, "nao_lidas": 247, "lidas": 980, "finalizadas": 20, "erro": 3, "urgentes": 12, "com_prazo": 890, "prazo_vencido": 5, "prazo_critico": 8, "classes": { "B": 400, "C": 300, "D": 200, "E": 50, "F": 20, "G": 10, "null": 267 }, "ia_status": { "pendente": 5, "processando": 2, "concluido": 1200, "erro": 3 } } // Frontend mapeia para 4 KPI cards: // Nao lidas = result.nao_lidas // Urgentes = result.urgentes // Vence hoje = COUNT(*) WHERE data_prazo = CURRENT_DATE (ou fazer query separada) // Estrategicas = result.classes.E + result.classes.F (label "Estrategicas", SEM mostrar a classe)
