Resumo Executivo

O Controller v3.0 apresenta uma arquitetura bem projetada com 5 layers (Capture, Storage, Analysis, Intelligence, Interface), classificacao IA 2-tier (Haiku + Sonnet), sistema de classes A-G, realtime funcional e keyboard shortcuts. A UI V4 e moderna, com virtualizer para performance e design system correto.


No entanto, existe um desalinhamento critico entre frontend e backend: a RPC de batch update ainda valida os status antigos (nova, processada, ignorada) enquanto o frontend V4 envia os novos (nao_lida, finalizada). Isso causa falhas em runtime em todas as acoes de status do drawer e em parte das bulk actions. Alem disso, 3 acoes no drawer sao placeholders (Reprocessar IA, Ciencia com Renuncia, Chat IA).


Veredito: Aprovacao condicional — requer correcao dos 3 P0 antes de ir para producao. O backend (migrations, Edge Functions, jurimetria) esta solido. O problema principal e a lacuna entre as migrations aplicadas e o codigo frontend V4.

P0 — Blockers Criticos
P0-01 Status Enum Mismatch: RPC vs Frontend Frontend + Backend

A RPC rpc_controller_batch_update_status valida status contra os valores antigos (v2), mas o frontend V4 envia os valores novos (v3). Resultado: EXCEPTION em runtime para toda acao de status.

-- RPC valida (migration 20260305070001, linha 31):
IF p_novo_status NOT IN ('nova','lida','processada','ignorada','erro','transferida')
  RAISE EXCEPTION 'Status invalido: %'

-- Frontend V4 envia:
'nao_lida'   -- REJECTS (nao esta na lista)
'finalizada' -- REJECTS (nao esta na lista)
'lida'       -- OK (existe em ambas versoes)

Arquivo: supabase/migrations/20260305070001_c11_controller_rpcs.sql:31


Impacto no Drawer (ControllerV4Drawer.tsx:440-447):

// TODAS as 3 acoes enviam 'finalizada' que a RPC rejeita:
<DropdownMenuItem onClick={() => updateStatus('finalizada')}>Finalizar</DropdownMenuItem>
<DropdownMenuItem onClick={() => updateStatus('finalizada')}>Ignorar</DropdownMenuItem>
<DropdownMenuItem onClick={() => updateStatus('finalizada')}>Arquivar</DropdownMenuItem>

Arquivo: src/pages/controller/ControllerV4Drawer.tsx:440-447


Impacto nas Row Actions (ControllerV4Intimacoes.tsx:219-224):

// Mapeamento de acoes para status v3:
const novoStatus = action === 'marcar_nao_lida' ? 'nao_lida'    // REJECTS
  : action === 'marcar_lida' ? 'lida'                            // OK
  : action === 'arquivar' ? 'finalizada'                         // REJECTS
  : action === 'processada' ? 'finalizada'                       // REJECTS
  : action === 'ignorada' ? 'finalizada'                         // REJECTS
  : action;

Arquivo: src/pages/controller/ControllerV4Intimacoes.tsx:219-224


Fix necessario: Recriar a RPC com os status v3 (nao_lida, lida, finalizada, erro, transferida) e ajustar a logica de concluida_em/concluida_por_id para o status finalizada.

P0-02 Bulk Actions Enviam Status Invalidos Frontend

As bulk actions no index.tsx enviam status que nao correspondem nem a v2 nem a v3:

// index.tsx:345-347 — Bulk bar passa status ANTIGOS:
onProcessar={() => handleBulkStatus('processada')}   // RPC aceita, mas DB CHECK pode rejeitar
onIgnorar={() => handleBulkStatus('ignorada')}       // RPC aceita, mas DB CHECK pode rejeitar
onArquivar={() => handleBulkStatus('arquivada')}     // NAO EXISTE em nenhuma validacao - SEMPRE falha!

Arquivo: src/pages/controller/index.tsx:345-347


O valor 'arquivada' nao esta na lista de validacao da RPC (nova, lida, processada, ignorada, erro, transferida) e tambem nao esta nos valores v3 (nao_lida, lida, finalizada). Toda tentativa de arquivar em bulk resulta em exception.


Fix necessario: Alinhar bulk actions para usar 'finalizada' (v3) e atualizar a RPC simultaneamente.

P0-03 rpc_finalizar Inexistente Backend

O commit 24c0f28d diz "adicionar rpc_finalizar" mas a funcao nao existe em nenhum arquivo de migration, nao esta nos types do Supabase, e nao e chamada pelo frontend.

$ grep -rn "rpc_finalizar" --include="*.tsx" --include="*.ts" --include="*.sql"
(nenhum resultado)

Fix necessario: Se a funcao foi planejada no PRD, criar a migration. Se foi absorvida pela rpc_controller_batch_update_status, remover a referencia do commit e atualizar a RPC conforme P0-01.

P1-07 KPI RPC Dual Naming Backend

O arquivo de types define duas RPCs: rpc_controller_kpis (com p_organizacao_id, p_user_id) e rpc_controller_v3_kpis (com p_org_id). Apenas a versao antiga e utilizada pelo frontend. A v3 pode nao existir no banco.

// supabase/types.ts:38360 — Usado pelo frontend:
rpc_controller_kpis: { Args: { p_organizacao_id, p_user_id }; Returns: Json }

// supabase/types.ts:38397 — Nunca usado, pode nao existir:
rpc_controller_v3_kpis: { Args: { p_org_id?: string }; Returns: Json }

Se a v3 nao retorna as KPIs corretas para o novo enum de status, os contadores podem estar errados (ex: contando nova ao inves de nao_lida).

P1-09 Migration CHECK Constraint Faltando Backend

Nenhuma migration no repositorio atualiza o CHECK constraint de intimacoes.status para os valores v3. O PRD diz que o C6 migration fez isso, mas o arquivo nao existe no diretorio supabase/migrations/.

Se o constraint no banco ainda aceita os valores antigos (nova, processada, ignorada), o frontend funciona parcialmente. Se ja foi atualizado para v3 (nao_lida, lida, finalizada), as bulk actions em index.tsx falham tambem no nivel do banco.

P1-10 47 Arquivos / 11k Linhas de Codigo Legacy Frontend

O diretorio src/pages/controller/ contem componentes de 4 versoes diferentes (V1, V2, V3, V4), todos coexistindo. Apenas os V4 sao usados. Os demais criam:

  • ~4.000 linhas de dead code (estimativa)
  • Confusao no build/bundle size
  • Risk de import acidental de componentes legados
  • Dificuldade de manutencao

Componentes legados: ControllerIntimacoes, ControllerModalIntimacao, ControllerFilterBar, ControllerKpis, ControllerBulkActionsBar, todo o diretorio painel-joao/, e mais 15+ arquivos.

P2 — Prioridade Media
P2-11 Frontend Types Incompletos

O tipo IntimacaoRow em types.ts nao inclui colunas adicionadas pelo PRD v3: modelo_ia, custo_ia, tokens_ia, analise_versao. Esses campos existem no banco mas nao sao tipados nem exibidos na UI.

P2-12 KPI Grid Nao Responsivo

O componente ControllerV4KPIs usa grid-cols-4 fixo sem breakpoints mobile. Em telas menores, os 4 cards ficam comprimidos. Deveria usar grid-cols-2 md:grid-cols-4.

Arquivo: src/pages/controller/ControllerV4KPIs.tsx:89

P2-13 staleTime Inconsistente

CLAUDE.md define staleTime: 30_000 para todos os hooks. KPIs usam 60_000 (ControllerV4KPIs.tsx:39). Deveria ser 30_000 para consistencia.

P2-14 Busca Sem Unaccent

A busca em ControllerV4Intimacoes.tsx:116 usa .ilike direto sem unaccent(). Buscar "Jose" nao encontra "Jose" e vice-versa. CLAUDE.md exige unaccent(lower()) para buscas textuais.

P2-15 Realtime Channel Name

O hook usa 'controller-realtime' como nome do channel. O PRD especifica 'controller-intimacoes-{org_id}'. Nao e bug funcional (o filter por org_id funciona), mas diverge do contrato documentado.

P2-16 confianca_ia Sem Guard Para Zero

No drawer, confianca_ia e exibido como (val * 100).toFixed(0)%. Se o valor for 0, exibe "0%" o que e correto, mas se for null o guard funciona. Porem se for 0.0 (number, nao null), o != null passa e exibe "0%" — que pode ser confuso ao usuario (IA sem confianca?). Considerar threshold minimo ou label contextual.

O Que Funciona Bem
Realtime Subscriptions

O hook useControllerRealtime e bem implementado: 6 eventos distintos (RT-001 a RT-006), cleanup correto com removeChannel, invalidacao granular de queries, toasts informativos. Cobre INSERT/UPDATE em intimacoes, atividades e documentos_pendentes.

Virtualizer + Infinite Scroll

O uso de @tanstack/react-virtual com pages de 50 items garante performance mesmo com milhares de intimacoes. O acumulador de items e a deteccao de sentinel para load more sao implementados corretamente.

Keyboard Navigation

Sistema completo de atalhos: J/K para navegar, U/N para urgente, Enter para abrir drawer, Escape para fechar, Ctrl+K para command palette, ? para help. Guards corretos para inputs/modals.

Filter Architecture V4

Filtros multi-select com popovers, debounced search (300ms), chips removiveis, sync com URL params, reset completo. Tipos bem definidos (ControllerV4Filters). Dedup de advogados por user_id.

Error Boundaries

Cada tab envolta em ErrorBoundary com TabErrorFallback. O drawer tambem usa DrawerTabErrorBoundary por tab. Falha em uma tab nao derruba a pagina.

Backend Solid: Migrations, Edge Functions, Jurimetria

As migrations sao bem estruturadas. Edge Function analyze-intimacao com 2-tier funcional. Jurimetria com 760 julgadores, 478 varas, 640 escritorios. Training loop com 52 playbooks. Dedup via evento_eproc UNIQUE index.

BUG-11 Fix: Conclusos Noise Filter

Hard filter com regex /concluso[sa]?\s+(ao|para|de\s+)/i para excluir conclusos que escapam sem classificacao. Solucao pragmatica e eficaz.
