Conexao com Cecilia v3 (Backend)

Pipeline de 5 Camadas

O mecanismo Cecilia v3 alimenta a tabela intimacoes que este frontend consome:

  1. Camada 1 — Captura: Node.js no Mac Mini navega EPROC via Playwright, extrai movimentacoes
  2. Camada 2 — Storage: INSERT em processos_movimentacoes + intimacoes. Trigger pg_net dispara Edge Function
  3. Camada 3 — Analise: Edge Function classifica com 2-tier IA (Haiku 80% + Sonnet 20%)
  4. Camada 4 — Inteligencia: CC Agent treina prompts 1x/dia via cecilia-vault
  5. Camada 5 — Interface: ESTE FRONTEND — Controller UI onde Joao valida/corrige

Taxonomia A-G: A=ruido (filtrado no SQL, nunca aparece), B=ciencia, C=operacional, D=decisao, E=estrategica, F=SISBAJUD, G=encerramento.

Regra critica: Classe A NUNCA aparece no frontend (WHERE clause na view). Classes B-G sao usadas internamente para roteamento e priorizacao — NAO sao visiveis ao usuario (Premissa 6).

Objetivo

Reconstruir o frontend do Controller como pagina de referencia do MdFlow — design system model. Criar uma experiencia otimizada para volume (300-600+ items), com drawer rico para tomada de decisao, e total alinhamento com as premissas operacionais do Joao Victor.

NAO inclui: Matriz de roteamento (removida do projeto). Apenas front-end — o backend (Cecilia v3) e construido separadamente.

2. Premissas Operacionais — 9 Regras Inviolaveis

ATENCAO: Estas regras foram definidas por Joao Victor (controller operacional) e aprovadas por Christian (CTO). Violacao de qualquer premissa = implementacao PAUSADA ate corrigir.

1
Intimacao FINALIZA, nao arquiva. Status flow: nao_lida → lida → finalizada. NAO existe conceito de "arquivar intimacao". Arquivamento e fase do PROCESSO, nao da intimacao.
2
Lida = alteracao APENAS visual. A intimacao permanece na lista, na mesma posicao. Nao desaparece. Apenas muda a aparencia: font-weight normal (vs bold quando nao_lida).
3
Finalizada = riscada + visivel. Card fica com line-through e opacity: 0.4. NAO desaparece imediatamente — permanece visivel ate atualizacao da pagina. Toggle "Mostrar finalizadas" controla visibilidade.
4
Prazo NAO aparece nos cards da lista. Intimacoes normalmente entram antes da abertura do prazo no EPROC. Prazo e responsabilidade do advogado apos receber a atividade. Prazo aparece APENAS: (a) dentro do drawer, (b) nos KPI cards ("Vencem hoje"), (c) na aba Equipe.
5
Etiquetas coexistem. urgente + importante podem estar ativas ao mesmo tempo. NAO sao excludentes. Dot vermelho = urgente, dot amarelo = importante, dot vermelho+borda = ambos.
6
IA e Classes ocultas na interface. Classes internas (A-G), scores de IA, confianca — NAO aparecem para o usuario. Nunca badge de "E — Estrategica" ou "0.97" visivel. Backend usa internamente para priorizacao e roteamento — UI nao exibe.
7
Criar atividade pode sugerir mover processo de fase. Ao criar atividade tipo "Expedição de Alvará", o sistema SUGERE mover para fase "financeiro". Mas NUNCA move automaticamente — sempre dialog de confirmacao.
8
Ciencia NUNCA automatica. O sistema pode sugerir ciencia com renuncia, mas SEMPRE depende do clique humano. Nunca executar ciencia automaticamente num processo. Fluxo: sugestao → usuario confirma → executa via webhook.
9
5 OABs parametrizadas. Christian: RS116571, SC074025, SP535101. Heloisa: RS131163, SC075466. O sistema deve estar preparado para inclusao futura de novas OABs (ex: TJRJ). Filtro por OAB/advogado na filter bar.

  total: number;

// Query implementation:
const { data } = useQuery({
  queryKey: ['controller-v5-intimacoes', organizacao_id, filters, page],
  queryFn: async () => {
    let query = supabase
      .from('v_controller_intimacoes_completa')
      .select('*', { count: 'exact' })
      .eq('organizacao_id', organizacao_id)
      .is('deleted_at', null);

    // Status filter
    if (!filters.mostrar_finalizadas) {
      query = query.neq('status', 'finalizada');
    }
    if (filters.status.length > 0) {
      query = query.in('status', filters.status);
    }

    // Search
    if (filters.search && filters.search.length >= 2) {
      query = query.or(
        `processo_cnj.ilike.%${filters.search}%,` +
        `cliente_nome.ilike.%${filters.search}%,` +
        `empresa_nome.ilike.%${filters.search}%`
      );
    }

    // Tribunal
    if (filters.tribunal !== 'todos') {
      query = query.eq('tribunal', filters.tribunal);
    }

    // Advogado
    if (filters.advogado_ids.length > 0) {
      query = query.in('advogado_id', filters.advogado_ids);
    }

    // Sort
    switch (filters.sort) {
      case 'data_recente': query = query.order('created_at', { ascending: false }); break;
      case 'data_antigo': query = query.order('created_at', { ascending: true }); break;
      case 'parte_az': query = query.order('cliente_nome', { ascending: true }); break;
      default: query = query.order('created_at', { ascending: false });
    }

    // Special KPI filters
    if (filters.urgente) query = query.eq('urgente', true);
    if (filters.prazo_hoje) query = query.eq('data_prazo', new Date().toISOString().split('T')[0]);
    if (filters.estrategicas) query = query.in('classe_cecilia', ['E', 'F']);

    // Pagination
    const from = page * pageSize;
    query = query.range(from, from + pageSize - 1);

    const { data, error, count } = await query;
    if (error) throw error;
    return { items: data ?? [], total: count ?? 0 };
  },
  staleTime: 30_000,
});

CREATE OR REPLACE VIEW v_controller_intimacoes_completa AS
SELECT
  i.*,
  p.numero_cnj AS processo_cnj,
  p.fase_processual AS processo_fase,
  p.status AS processo_status,
  p.origem AS processo_origem,
  c.nome AS cliente_nome,
  c.id AS cliente_id,
  e.nome AS empresa_nome,
  e.id AS empresa_id_joined,
  u.nome AS advogado_nome,
  u.id AS advogado_id,
  -- Campos Cecilia v3 (ja existem na tabela intimacoes)
  i.classe_cecilia,
  i.confianca_ia,
  i.tribunal,
  i.responsavel,
  i.motivo_ia,
  i.cnj,
  -- Prazo calculado
  i.data_prazo,
  CASE
    WHEN i.data_prazo IS NULL THEN NULL
    ELSE (i.data_prazo::date - CURRENT_DATE)
  END AS dias_ate_prazo,
  CASE
    WHEN i.data_prazo IS NULL THEN 'sem_prazo'
    WHEN (i.data_prazo::date - CURRENT_DATE) < 0 THEN 'vencido'
    WHEN (i.data_prazo::date - CURRENT_DATE) = 0 THEN 'hoje'
    WHEN (i.data_prazo::date - CURRENT_DATE) <= 3 THEN 'critico'
    WHEN (i.data_prazo::date - CURRENT_DATE) <= 7 THEN 'atencao'
    ELSE 'normal'
  END AS status_prazo
FROM intimacoes i
LEFT JOIN processos p ON i.processo_id = p.id
LEFT JOIN clientes c ON p.cliente_id = c.id
LEFT JOIN empresas e ON p.empresa_id = e.id
LEFT JOIN organizacoes_usuarios ou ON i.advogado_id = ou.id
LEFT JOIN users u ON ou.user_id = u.id
WHERE i.deleted_at IS NULL

CREATE OR REPLACE FUNCTION rpc_controller_v5_kpis(
  p_organizacao_id UUID,
  p_user_id UUID DEFAULT NULL
)
RETURNS JSONB
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  result JSONB;
BEGIN
  SELECT jsonb_build_object(
    'nao_lidas', COUNT(*) FILTER (WHERE status = 'nao_lida'),
    'urgentes', COUNT(*) FILTER (WHERE urgente = true AND status != 'finalizada'),
    'vence_hoje', COUNT(*) FILTER (WHERE data_prazo::date = CURRENT_DATE AND status != 'finalizada'),
    'estrategicas', COUNT(*) FILTER (WHERE classe_cecilia IN ('E','F') AND status != 'finalizada'),
    'total', COUNT(*) FILTER (WHERE status != 'finalizada')
  ) INTO result
  FROM intimacoes
  WHERE organizacao_id = p_organizacao_id
    AND deleted_at IS NULL
    AND (classe_cecilia IS NULL OR classe_cecilia != 'A');

  RETURN result;
END;
$$;

RETURNS JSONB

BEGIN
  UPDATE intimacoes
  SET status = 'finalizada',

      concluida_por_id = p_user_id,
      updated_at = now()
  WHERE id = p_intimacao_id

  RETURN jsonb_build_object('success', true);
