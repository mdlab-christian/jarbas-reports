Por que isso importa

CRM e MdZap são um sistema único. Lead nasce no MdZap (WhatsApp → Julia), vive no CRM, volta ao MdZap. Gaps na integração causam:

  • Leads fantasma na lista de conversas (leads deletados visíveis)
  • Métricas de dashboard zeradas — tempo_médio e taxa_handoff retornam 0 sempre
  • Atendentes sem notificação quando auditoria CRM conclui
  • Risco de segurança — whatsapp_mensagens sem RLS (mensagens visíveis entre organizações)

Gaps Identificados

P0 Assumir/Devolver n8n — status desconhecido

Frontend implementado (botões assumir/devolver presentes na UI) mas n8n não verificado. Bloqueia handoff humano↔Julia. Se o webhook não existe no n8n, os botões não funcionam silenciosamente.

P1 fn_mdzap_conversas sem deleted_at

A função server-side que alimenta a lista de conversas do MdZap não filtra leads com deleted_at. Leads deletados no CRM aparecem como conversas fantasma no MdZap.

P1 fn_mdzap_stats — hardcoded zeros

A função de estatísticas do dashboard MdZap retorna tempo_médio=0 e taxa_handoff=0 em todos os casos. Nenhum KPI real é exibido para o time.

P1 whatsapp_mensagens sem RLS

A tabela whatsapp_mensagens não possui Row Level Security ativa. Mensagens de diferentes organizações ficam expostas entre si — risco de segurança direto em produção.

P2 pending_lead_data = 0 registros

O Doc Reader deveria popular dados pré-cadastro na tabela pending_lead_data, mas não há nenhum registro. O fluxo de enriquecimento antes do cadastro não está funcionando.

P2 Sem notificação auditoria → MdZap

Quando uma auditoria CRM conclui, o atendente no MdZap não recebe nenhuma notificação. O atendente precisa verificar manualmente se a auditoria foi concluída.
