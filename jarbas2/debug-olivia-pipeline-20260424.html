Contexto da Sessão

O erro reportado superficialmente como "Step 2 → Step 3 travado" encobria dois problemas completamente distintos acumulados no ambiente do Mac Mini: um conflito de porta entre LaunchAgent e LaunchDaemon, e falhas silenciosas de JSON parse no gerenciador de seções jurídicas.

Esta sessão tratou o conflito de porta e implementou a correção definitiva do parser. A correção do TAG is not defined havia sido deployada na sessão anterior (commit b7cdeb4d).

Causa Raiz Real do Erro "Step 2 → Step 3"

SEÇÃO 02
Sessão anterior Problema 1 — TAG is not defined ✅ já corrigido

Commit: b7cdeb4d fix(olivia): TAG is not defined no gerador + import quebrado em /feedback

Causa: As funções loadMemoriaFiles, buildPrompt e buildPromptSecao chamavam log(TAG, ...) sem declarar const TAG = 'geracao' no escopo local do módulo. O erro lançava ReferenceError: TAG is not defined durante a fase de prompt assembly, impedindo qualquer geração de seção.

Correção: Declaração de TAG adicionada no topo de geracao.mjs + import corrigido em /feedback. Deployado e confirmado antes desta sessão.

Esta sessão Problema 2 — EADDRINUSE: porta 19100 ocupada ✅ corrigido

Causa raiz: O LaunchAgent do usuário olivia — arquivo ai.olivia.chat-server.plist — iniciava o olivia-chat-server.mjs sem variável de porta configurada, resultando em bind na porta padrão 19100.

O LaunchDaemon com.olivia.server (responsável pelo server.mjs, pipeline de petições) tentava subir na mesma 19100 e falhava com EADDRINUSE. O processo entrava em loop: terminava com SIGTERM (-15) e reiniciava a cada 10 segundos indefinidamente.

Evidência nos logs do Mac Mini:

/var/log/olivia/server.log · erro repetido a cada ~10s
[OlivIA] listen EADDRINUSE: address already in use 0.0.0.0:19100 [server] AVISO: OLIVIA_GATEWAY_TOKEN nao configurada. Endpoints de gateway sem autenticacao. [OlivIA] listen EADDRINUSE: address already in use 0.0.0.0:19100 [server] AVISO: OLIVIA_GATEWAY_TOKEN nao configurada. Endpoints de gateway sem autenticacao.

Correção aplicada:

ai.olivia.chat-server.plist — variável de porta adicionada
<key>EnvironmentVariables</key> <dict> + <key>OLIVIA_HTTP_PORT</key> + <string>19102</string> </dict>

Sequência de recovery:

Passo 1
Chat server parado via launchctl
launchctl unload ~/Library/LaunchAgents/ai.olivia.chat-server.plist
Passo 2
plist editado com OLIVIA_HTTP_PORT=19102
Porta 19100 liberada para o pipeline de petições
Passo 3
Chat server reloadado em :19102
launchctl load ~/Library/LaunchAgents/ai.olivia.chat-server.plist
Passo 4
server.mjs kickstart — v4.1 confirmado em :19100
Pipeline de petições respondendo normalmente

Problema Adicional — parseSecao JSON Parse Failures

SEÇÃO 03
Falha silenciosa JSON parse falha após 4 tentativas — score_topico zerado ✅ corrigido — tentativa 5

Sintoma nos logs de produção:

/var/log/olivia/geracao.log · múltiplas ocorrências por geração
[geracao] parseSecao: ERRO parsing "DA RESPONSABILIDADE OBJETIVA..." — JSON parse falhou após 4 tentativas. Primeiro: Expected ',' or '}' after property value in JSON at position 2091

Causa identificada: Campos conteudo_html com aspas não escapadas no texto jurídico — citações diretas do autor, nomes de documentos entre aspas (ex: "Contrato de Serviço"), nomes de partes. O Claude retorna JSON válido semanticamente, mas com aspas literais dentro de strings JSON, violando a spec.

Impacto: Seções ainda geravam HTML via fallback regex, mas os campos score_topico, juris_usados e tags_usados ficavam zerados/vazios — degradando métricas de qualidade RAG sem alertar explicitamente.

Por que as 4 tentativas anteriores falhavam:

Tentativas 1–4 (insuficientes)

T1: JSON.parse direto do raw

T2: Extrair bloco ```json ... ```

T3: Remoção de BOM + trailing chars

T4: escapeCtrlCharsEmStrings (controles ASCII)

Nenhuma resolvia aspas literais em valores de string
Tentativa 5 — Surgical Repair

Usa T4 como base, itera até 15x localiza a posição exata do erro de parse, busca a aspa não escapada mais próxima antes dessa posição, e a escapa cirurgicamente — sem tocar no restante da string.

Resolve aspas literais múltiplas iterativamente

Implementação da Tentativa 5:

// geracao.mjs — parseJsonTolerante — Tentativa 5: Surgical Repair Iterativo // Aplica escapeCtrlCharsEmStrings como base, depois itera até 15x // localizando e escapando aspas não escapadas na posição do erro let repaired = escapeCtrlCharsEmStrings(raw); for (let _i = 0; _i < 15; _i++) { try { return JSON.parse(repaired); } catch (eN) { // Extrair posição do erro da mensagem do engine V8 const m = eN.message.match(/position (\d+)/); if (!m) break; const errPos = parseInt(m[1]); // Buscar " não escapado ANTES da posição do erro let fixAt = -1; for (let p = errPos; p >= 0; p--) { if (repaired[p] === '"' && (p === 0 || repaired[p - 1] !== '\\')) { fixAt = p; break; } } if (fixAt < 0) break; // nada para reparar // Substituição cirúrgica: " → \" apenas na posição exata repaired = repaired.slice(0, fixAt) + '\\"' + repaired.slice(fixAt + 1); } } return JSON.parse(repaired); // lança se ainda inválido após 15 iterações
