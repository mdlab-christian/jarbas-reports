Analise Tecnica Profunda

BrowserManager — Core (870 linhas)

O daemon/src/browser-manager.ts e o coracao do sistema:

  • ensureBrowser() — lanca Chromium ou conecta via CDP a Chrome existente
  • autoConnect() — detecta Chrome rodando e conecta automaticamente
  • getPage(name) — retorna pagina persistente por nome (cria se nao existe)
  • listPages() — lista todas as paginas ativas com URL e titulo
  • launchPersistentContext() — contexto com userDataDir (cookies persistem)
  • closePage(name) / closeAllPages() — lifecycle management
HostBridge — Sandbox ↔ Playwright

O daemon/src/sandbox/host-bridge.ts faz a ponte:

// Scripts rodam no QuickJS WASM (sandbox isolado) // HostBridge serializa chamadas Playwright via JSON dispatcherConnection.onmessage = (message) => { sendToSandbox(JSON.stringify(message)); }; // Sandbox envia comandos, HostBridge executa no Playwright real async receiveFromSandbox(json: string) { await dispatcherConnection.dispatch(JSON.parse(json)); }

Cada chamada Playwright passa por serializacao JSON → QuickJS → HostBridge → Playwright. Overhead minimo para operacoes normais, mas latencia extra em operacoes de alta frequencia.

Script API — Como se usa
// dev-browser run script.js --page eproc-rs116571 const page = await getPage(); await page.goto('https://eproc1g.tjrs.jus.br'); await page.fill('#txtLogin', 'RS116571'); await page.fill('#txtSenha', '***'); // TOTP const totp = authenticator.generate(secret); await page.fill('#txtToken', totp); await page.click('#btnEntrar'); await page.waitForURL('**/painel_usuario*'); // Snapshot para IA analisar a pagina const snapshot = await snapshotForAI();

Pros e Contras para o MdFlow/EPROC

Vantagens

  • Paginas persistentes — sessao EPROC sobrevive entre scripts, sem re-login
  • Named pages — --page eproc-rs116571 isola cada OAB
  • Auto-connect — pode conectar a Chrome com sessao ja logada
  • snapshotForAI() — util para debugging e analise de paginas complexas
  • Sandbox seguro — scripts nao acessam filesystem, bom para multi-tenant
  • Daemon sempre ativo — browser nao morre entre execucoes
  • Persistent profiles — cookies EPROC persistem entre sessoes

Desvantagens

  • Sandbox limita — nao pode importar otplib (TOTP), fs, supabase-js
  • Overhead de serializacao — cada chamada PW passa por JSON bridge
  • Dependencia Rust CLI — precisa compilar CLI ou usar binario pre-built
  • Playwright 1.58.2 fixo — nao acompanha versao mais recente
  • File I/O restrito — so grava em ~/.dev-browser/tmp/
  • Sem acesso direto a APIs — nao pode chamar Supabase, Anthropic do sandbox
  • Projeto jovem — v0.2.6, comunidade pequena, pode ter breaking changes

Blocker Critico: TOTP no Sandbox

BLOCKER — Geracao de TOTP impossivel no QuickJS

O EPROC exige autenticacao de dois fatores (TOTP) para todas as 5 OABs. O codigo TOTP e gerado com otplib:

import { authenticator } from 'otplib'; const code = authenticator.generate('HEZDAMZQ...'); // Retorna 6 digitos validos por 30 segundos

No sandbox QuickJS do dev-browser:

  • Nao pode importar modulos npm (otplib, speakeasy)
  • Nao tem crypto nativo suficiente para HMAC-SHA1 (base do TOTP)
  • Workaround: Seria necessario pre-gerar o TOTP fora do sandbox e passar como parametro — adiciona complexidade sem beneficio

Resultado: Para login EPROC, o dev-browser sandbox nao funciona standalone. Precisaria de um script Node.js externo gerando o TOTP e injetando — anulando a proposta de valor do sandbox.

Ideias Aproveitaveis (sem instalar dev-browser)

Padroes para incorporar nos scripts Node.js atuais

Named Pages / Session Manager

Criar um SessionManager que gerencia contextos Playwright por OAB com storageState persistente. Mesmo conceito das named pages do dev-browser, mas nativo.

Persistent Login com Cookie Reuse

Salvar cookies EPROC apos login e reutilizar em proximas execucoes. Evita 5 logins + TOTP a cada monitoramento. Dev-browser faz isso com launchPersistentContext().

AI Snapshot para Debug

Implementar funcao snapshotForAI(page) que extrai texto semantico da pagina para quando um script quebra. Util para Claude Code diagnosticar falhas.

Daemon Pattern (long-running)

Em vez de abrir/fechar browser a cada script, manter um processo daemon com browser ativo. Reduce startup de ~3s para ~0s por execucao.

Recomendacao Final

Veredicto: NAO instalar dev-browser para EPROC

Dev-browser e um projeto excelente em qualidade de codigo e arquitetura. O sandbox QuickJS e inovador e a API de persistent pages e elegante. Porem, para o caso de uso do MdFlow (EPROC com TOTP 2FA + Supabase + pipeline IA), o sandbox e um blocker, nao um beneficio.

Os scripts Node.js diretos que rodamos hoje via Claude Code tem acesso total: otplib para TOTP, supabase-js para banco, Anthropic SDK para IA, fs para arquivos. O dev-browser removeria essas capacidades em troca de isolamento que nao precisamos (somos nos mesmos rodando os scripts, nao terceiros).

Quando dev-browser FARIA sentido: se tivessemos usuarios externos rodando scripts de browser no nosso sistema (multi-tenant browser automation) — ai o sandbox seria essencial. Para automacao interna (EPROC, monitoramento), os scripts diretos sao superiores.

Acao recomendada: Nao instalar dev-browser. Em vez disso, incorporar as 4 ideias acima (session manager, persistent login, AI snapshot, daemon pattern) nos scripts Node.js atuais. Mesmo resultado, sem overhead de sandbox.
