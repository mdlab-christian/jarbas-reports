Resumo Executivo

O sistema de matching usa tokenSimilarity + nome_variacoes com auto-aprendizado via drawer de auditoria. Os dados revelam 1.401 restrições sem empresa_id para credores cujas empresas já existem no banco — o maior gap sendo BOA VISTA SERVICOS S.A. (389 casos) sem vínculo mesmo a empresa existindo com esse nome quase idêntico. O principal problema estrutural é que restrições criadas fora do fluxo de auditoria (migração/CRM antigo) nunca acionam o auto-aprendizado e ficam permanentemente sem vínculo. Adicionalmente, foram detectados 3 drifts críticos em nome_variacoes: dados de teste em produção, cross-contaminação entre fundos FIDC e mapeamento NPL I → NPL II incorreto.

Como Funciona o Matching

Algoritmo — 3 passos em sequência
1
Match exato em nome_variacoes score 1.0 — EXATO
Normaliza o credor recebido (remove acentos, lowercase, caracteres especiais) e compara contra todas as strings do array nome_variacoes de cada empresa cadastrada. Match case-insensitive após normalização.
// Exemplo: "SCPC - Boa Vista Serviços" → normaliza → "scpc boa vista servicos" // Compara com nome_variacoes de BOA VISTA SERVICOS S.A.: // ["BOA VISTA", "BOA VISTA SERVICOS", "SCPC - BOA VISTA SERVIÇOS"] // → normalizado "scpc boa vista servicos" encontrado → score 1.0 → match EXATO
2
tokenSimilarity fuzzy — nome, razao_social, nome_fantasia, nome_variacoes score ≥ 0.45 → aceito
Tokeniza ambos os strings por espaço, calcula interseção de tokens sobre o máximo dos dois sets. Compara o credor contra todos os campos textuais de todas as empresas e retorna o maior score.
// Exemplo A: "BANCO DO BRASIL S/A" vs "BANCO DO BRASIL S.A." // tokensA = ["banco","do","brasil","s"] tokensB = ["banco","do","brasil","s","a"] // common = 4 | max(4,5) = 5 → score = 4/5 = 0.80 → match ALTO ✅ // // Exemplo B: "MERCADO PAGO INSTITUICAO DE PAGAMENTO LTDA" vs "MERCADO PAGO INST. DE PAGAMENTO LTDA" // tokensA = ["mercado","pago","instituicao","de","pagamento","ltda"] (6) // tokensB = ["mercado","pago","inst","de","pagamento","ltda"] (6) // common = ["mercado","pago","de","pagamento","ltda"] = 5 → score = 5/6 = 0.83 → match ALTO ✅ // // Exemplo C: "LEBES FINANCEIRA S.A." vs qualquer empresa // → nenhuma empresa com tokens similares → score < 0.45 → SEM MATCH
3
Classificação do match_type
Determina a confiança do match baseado no score final obtido nos passos 1 e 2.
EXATO
variação ≥ 0.6
Match por nome_variacoes com score alto
ALTO
score ≥ 0.75
Alta confiança — aceito automaticamente
MEDIO
0.45 — 0.74
Confirmação humana recomendada
SEM MATCH — score < 0.45 empresa_id = NULL, orgao_nome = nome do credor raw
Limite de 0.45 — impacto real: "BANCO DO BRASIL S/A" (tokens: banco,do,brasil,s) vs "BANCO DO BRASIL S.A." (tokens: banco,do,brasil,s,a) → score 0.80. O algoritmo deveria matchar. Se esses 144 casos aparecem sem vínculo, o problema é de fluxo, não do algoritmo — foram inseridos diretamente no banco sem passar pelo pipeline de matching.

O matching nunca foi executado nesses registros — portanto adicionar variações ao banco não resolve retroativamente. Será necessário um script de re-matching em batch.

Drifts em nome_variacoes

O auto-aprendizado não valida unicidade global — apenas verifica se a variação já existe para a mesma empresa. Isso permite que variações de empresas diferentes sejam adicionadas incorretamente, causando falsos positivos no matching.
VERO S.A.
CRITICO
O array nome_variacoes contém "Teste Smoke 16:38:34" — dado de teste que foi persistido em produção via auto-aprendizado. Toda restrição com credor "Teste Smoke 16:38:34" vai ser vinculada à VERO S.A. com score 1.0 (exato), gerando vínculo completamente falso.
Acao: executar fn_empresas_remove_variacao (ou UPDATE direto) para remover "Teste Smoke 16:38:34" do array. Auditoria: verificar se existem restrições já vinculadas com esse credor raw.
FIDC IPANEMA IV
CRITICO
Tem como variação "FIDC MULTISEGMENTOS NPL IPANEMA VI RESPONSABILIDADE LIMITADA", porém FIDC IPANEMA VI é uma empresa diferente cadastrada separadamente no banco. Qualquer restrição com credor IPANEMA VI vai ser vinculada incorretamente ao FIDC IV, causando mistura de passivos entre fundos distintos.
Acao: remover variação IPANEMA VI do FIDC IV. Verificar se restrições IPANEMA VI estão vinculadas ao ID errado e corrigir empresa_id. Checar como essa variação foi inserida (provavelmente confirmacao humana errada no drawer).
NPL I (FUNDO CREDITORIOS NAO-PADRONIZADOS)
CRITICO
Tem como variação "NPL II", mas NPL I e NPL II são fundos juridicamente distintos com carteiras independentes. Restrições do NPL II serão vinculadas ao NPL I — erro grave em relatórios de passivo por credor.
Acao: remover "NPL II" das variacoes do NPL I. Auditar restrições vinculadas ao NPL I com credor_raw contendo "NPL II". Corrigir empresa_id para o UUID do NPL II.
VUON CARD / CIB CONSULTORIA
ATENCAO
VUON CARD tem "CIB CONSULTORIA ADMINISTRACAO E PARTICIPACOES S/A." como variação, mas CIB CONSULTORIA existe como empresa separada cadastrada com razão social similar. Potencial duplicata — restrições da CIB podem estar sendo vinculadas ao VUON CARD.
Acao: verificar se CIB CONSULTORIA e VUON CARD são de fato a mesma entidade (CNPJ). Se forem diferentes: remover variação do VUON CARD e garantir que CIB tem seu próprio cadastro. Se forem a mesma: consolidar em uma empresa e marcar a outra com deleted_at.
CELESC — Possível Duplicata
ATENCAO
Existem duas entradas: CELESC DISTRIBUICAO S.A. e CELESC - CENTRAIS ELETRICAS DE SANTA CATARINA SA. Podem ser a mesma empresa (holding vs subsidiária de distribuição) ou entidades distintas. Matching pode estar dividindo restrições entre os dois cadastros.
Acao: verificar CNPJs. Se subsidiária separada: manter ambas com nomes claros. Se mesma entidade: consolidar mantendo CELESC DISTRIBUICAO como principal e adicionando variação "CELESC CENTRAIS ELETRICAS".
BANRISUL — 17 variações por agência
INFO
BANRISUL tem 17 variações no padrão "BANRISUL - TAQUARI", "BANRISUL - PELOTAS" etc. Funcionalmente correto para matching (credores chegam com nome da agência), mas aumenta o risco de falso positivo se outra empresa tiver "BANRISUL" no nome no futuro.
Acao: nenhuma urgente. Monitorar se variações por agência estão causando falsos positivos. Considerar regex matching para padrão "BANRISUL - *" em vez de variações individuais (melhoria futura no algoritmo).

Auto-Aprendizado — Onde Funciona e Onde Não Funciona

RPC fn_empresas_append_variacao
-- Lógica da RPC (comportamento confirmado): -- 1. Recebe: empresa_id UUID, credor_raw TEXT, organizacao_id UUID -- 2. Verifica se credor_raw já existe em nome_variacoes (case-insensitive) -- 3. Se NÃO existe: faz array_append(nome_variacoes, credor_raw) -- 4. Se JÁ existe: noop silencioso -- 5. NÃO valida se credor_raw já existe em OUTRA empresa (gap crítico que gerou drifts FIDC/NPL) -- Acionada: ao salvar restrição confirmada pelo drawer de auditoria -- quando credor_raw ≠ nome canônico da empresa confirmada
✅ Funciona — auto-aprende
🔍
Drawer de auditoria (fluxo normal)
Auditor abre restrição, confirma empresa correta. Se credor_raw difere do nome, variação é adicionada automaticamente.
📎
Match de média confiança confirmado
Score entre 0.45–0.74 validado pelo usuário aprende a variação para futuras ocorrências idênticas.
🚀
Efeito acumulativo correto
Cada confirmação melhora matching futuro. Empresas com muitas restrições (BOA VISTA 1202, MERCADO PAGO 597) acumulam variações ricas.
❌ Não funciona — gap estrutural
📦
Restrições importadas / migração
orgao_nome = NULL. Foram inseridas diretamente no banco sem passar pelo pipeline de matching. O auto-aprendizado nunca é chamado. São os 1.401 casos do gap acima.
📋
Aba Restrições do CRM (sem auditoria)
Restrições salvas diretamente na aba do CRM, fora do fluxo de auditoria, não acionam a RPC de aprendizado.
✅
Match exato de primeira (correto omitir)
Quando credor_raw = nome canônico da empresa, não há variação nova a aprender. Comportamento correto por design.
⚠️
Cross-empresa não validado (gerou drifts)
RPC não verifica se variação já pertence a outra empresa. Confirmação humana errada no drawer gerou os drifts FIDC IV/VI e NPL I/II.

Investigar por que ITAU UNIBANCO S.A. (nome idêntico) tem 78 casos sem vínculo
O nome é exatamente igual ao cadastrado. Isso confirma que esses registros foram inseridos por fluxo que bypassa o matching. Identificar a origem (migração batch, import CSV, endpoint legado) e garantir que o fluxo atual de inserção sempre acione o matching.
P1
Adicionar validação cross-empresa na RPC fn_empresas_append_variacao
Antes de fazer array_append, verificar se a variação já existe em OUTRA empresa do mesmo organizacao_id. Se sim, retornar erro com o nome da empresa conflitante para o auditor decidir. Isso previne novos drifts como os do FIDC e NPL.
P1
Cadastrar empresas ausentes: LEBES FINANCEIRA, BANCO BRADESCO, BANCO CSF
LEBES FINANCEIRA (64 restrições), BANCO BRADESCO (44), BANCO CSF S/A (33) — empresas com volume relevante mas não cadastradas. Criar registros mínimos para que o matching possa vincular retroativamente após o batch de re-matching (P1 acima).
P2
Investigar e consolidar VUON CARD / CIB CONSULTORIA duplicata
Verificar CNPJs. Se mesma entidade: consolidar com soft delete na duplicata. Se entidades distintas: remover variação CIB do VUON CARD. Impacto estimado baixo — volume de restrições não mencionado.
P2
Avaliar CELESC DISTRIBUICAO vs CELESC CENTRAIS ELETRICAS — possível duplicata
Comparar CNPJs. Holding (CELESC SA) e subsidiária (CELESC DISTRIBUICAO) são CNPJs distintos mas frequentemente tratados como mesmo credor em cobranças. Decisão de negócio: tratar como uma empresa ou manter separadas com variações cruzadas.
