Secao 1 — O que o Epic CLI faz por dentro (mecanismos reais do codigo-fonte)

Epic e um workflow manager que chama Claude Code internamente. Usa Bun como runtime + React/Ink para UI terminal.

1.1 Arquitetura interna do Epic
ND()
Funcao que chama Claude com flags completas
claude -p <prompt> --output-format stream-json --verbose --dangerously-skip-permissions
ZZ()
Execucao via tmux, isolada por sessao
claude "implement issue ${id}" --dangerously-skip-permissions
gh CLI
GitHub CLI para sync de issues bidirecional (create, list, close)
worktrees
Git worktrees em .worktrees/ — isolamento de branch por issue, pre-trust Claude no .claude.json do worktree
React/Ink
UI terminal renderizada com React (precisa de Bun). Componentes: lista de issues, status, progresso.
1.2 Template de spec generate (prompt interno OX)
# Spec
Instructions: $ARGUMENTS

- Create a spec for this project using the following format.
- Create a spec.md file in the root of the project.
- Try to build an MVP, only the most essential commands and operations. We will iterate later.
- Think about what is the core job to be done of this project and focus on it.
- An Operation is an action the user can perform within a Command.
- A job story: "When <situation>, I want to <motivation>, so I can <expected outcome>."
- A Flow is a user workflow that connects operations across commands in order.

## Specification Format
# [Project Name]
## Overview
## [command-name]
### Arguments & Flags
### Operations
## Flows
### [Flow Name] — 1. [verb] [subject] -- command operation
1.3 Template de spec break (prompt interno zX)
- Break spec into issues: one prototype per page + one per behavior.
- Create issue "Implement [page name] prototype" per page (components only, no behaviors)
- Create one issue per behavior listed in spec
- Use Flows section to determine ordering and dependencies
- Each behavior depends on its page prototype issue
- Within a flow, each behavior depends on behaviors before it in that flow
- Only list DIRECT dependencies (not transitive)
- Order: components/prototypes before behaviors
1.4 Template de issue plan (prompt interno fZ)
- Read issue file + spec.md for context
- Navigate to page folder to understand what's already implemented (READ ONLY - no changes)
- Look at current schema.ts if necessary
- Update the issue file IN-PLACE with the plan following the Issue Template:

  # Functional Specification
    ## Operation: [name]
    ### Preconditions
    ### [Primary Use Case]
      #### Input / Workflow / Output
    ### [Edge Case or Error Flow]
  # Technical Specification
    ## Function: [operationName]
    ### Implementation steps
    ## Utils, Lib, External Dependencies
  # Tasks
    - [ ] Create operation file
    - [ ] Add to command router
    - [ ] Implement utils
    - [ ] Write tests
    - [ ] Update docs
  # Notes

- Se o issue e implementar componentes de pagina: apenas componentes e estrutura, sem behaviors
1.5 Prompt de issue execute
"Read the issue file at ${path} and execute the implementation plan."

# Executa via:
claude <issue_file> --dangerously-skip-permissions

Notavel pela simplicidade: o prompt e minimalista porque todo o contexto ja esta no arquivo da tarefa.

1.6 Prompt de issue verify (Playwright)
You are verifying that an issue's implementation is working correctly by using the browser.
Use npx @playwright/cli@latest to interact with the browser.
The application is running at http://localhost:$PORT.

Workflow:
1. Read issue file to understand what was built
2. Open browser and navigate
3. For each acceptance criterion: snapshot -> interact -> snapshot -> verify
4. Report PASS/FAIL per criterion

Secao 3 — Plano de Absorcao: 8 Melhorias para /projeto v3

M1 Flows no Spec Alto impacto
O que e
Secao ## Flows na spec que captura workflows do usuario conectando operacoes em sequencia. Determina dependencias e ordem de implementacao automaticamente.
De onde vem
Epic templates OX (spec generate) + zX (spec break). Flows sao a fonte de verdade para depends_on.
Como absorver
Adicionar secao Flows ao template de spec.md. No break, usar Flows para auto-determinar depends_on das tarefas geradas. Elimina dependencias manuais e garante ordem de execucao correta.
M2 Issue Auto-Contido com Functional + Technical Spec Alto impacto
O que e
O arquivo da tarefa contem o plano embutido: Functional Spec + Technical Spec + Tasks + Acceptance Criteria. Nao ha arquivo separado em plans/.
De onde vem
Epic template fZ (issue plan). Execute prompt minimalista funciona porque todo contexto ja esta no arquivo da tarefa.
Como absorver
Quando /projeto plan executar, ATUALIZAR o arquivo tasks/<task-id>.md in-place adicionando as secoes. Manter plans/ apenas como output legado. Tarefa torna-se auto-contida e pode ser passada diretamente ao Claude via claude <task-file>.
M3 Acceptance Criteria por Tarefa Alto impacto
O que e
Cada tarefa define criterios de aceitacao explicitos e verificaveis. Usados tanto pelo developer quanto pelo verify automatico via Playwright.
De onde vem
Epic issue template (Tasks section). O verify prompt itera sobre cada acceptance criterion com snapshot > interagir > verificar.
Como absorver
Adicionar secao # Acceptance Criteria ao template de tarefa durante /projeto plan. Usar como input para /projeto validate. Validate passa a ser deterministico em vez de subjetivo.
M4 Exploracao Read-Only Antes de Planejar Medio impacto
O que e
Instrucao explicita: navegar para a pasta da pagina, entender o que ja esta implementado ANTES de escrever o plano. Fase de leitura pura, sem modificar nada.
De onde vem
Epic template fZ step 1: "Navigate to page folder... READ ONLY - no changes". Reduz alucinacoes sobre estado atual do codigo.
Como absorver
Adicionar Passo 0 explicito no /projeto plan: "Explore o codigo, leia os arquivos relevantes, entenda o estado atual. PROIBIDO modificar qualquer arquivo nesta fase." Incluir verificacao de schema Supabase via MCP antes de propor modelo de dados.
M5 Verify com Playwright Medio impacto
O que e
Pos-execucao, usar npx @playwright/cli@latest para verificar acceptance criteria no browser real. Fluxo: snapshot > interagir > snapshot > PASS/FAIL por criterio.
De onde vem
Epic verify prompt. Valida UI real, nao apenas build/lint. Detecta regressoes visuais e fluxos quebrados.
Como absorver
Adicionar /projeto verify <task-id> que usa playwright-cli se disponivel. Integrar ao validate existente como fase final. Porta padrao do Vite: 5173. Requer app rodando localmente.
M6 GitHub Issues Sync Medio impacto
O que e
Apos break, sincronizar tarefas como GitHub Issues via gh issue create. Issues ficam visiveis no repositorio para rastreamento e colaboracao.
De onde vem
Epic spec break + issue sync. Salva github_id no frontmatter do arquivo da tarefa para rastreabilidade bidirecional.
Como absorver
Adicionar flag --sync ao /projeto break. Se --sync: rodar gh issue create para cada tarefa criada. Salvar github_id no frontmatter do tasks/<task-id>.md. Nao obrigatorio por padrao.
M7 style apply (tweakcn -> globals.css) Baixo impacto
O que e
Comando para aplicar design tokens do tweakcn.com no globals.css seguindo ordem exata de secoes: imports > dark > @theme > :root > .dark > @layer base.
De onde vem
Epic style apply. Especifico para projetos React com shadcn/ui. Util ao mudar temas visuais (novo cliente, novo branding).
Como absorver
Adicionar /projeto style como novo subcomando. Pede URL ou texto do tweakcn, faz merge no src/index.css respeitando a ordem de secoes. Preserva Warm Charcoal + primary #20c997.
M8 Git Worktrees por Tarefa Baixo impacto
O que e
Criar .worktrees/<task-id>/ para cada tarefa com branch propria e Claude pre-trusted. Permite execucao paralela sem conflitos de estado.
De onde vem
Epic issue start. Util para isolamento, mas adiciona complexidade no fluxo Pro + Air + Mac Mini onde o repositorio e compartilhado via git.
Como absorver
Adicionar flag --worktree ao /projeto execute. Se --worktree: criar worktree em .worktrees/<task-id>/, pre-trust Claude no .claude.json, executar la, merge automatico ao finalizar. Nao ativar por padrao.

Secao 5 — Conclusao e Veredicto

Veredicto: NAO vale manter a assinatura do Epic.
x Execute prompt e minimalista: "Read the issue file at X and execute the implementation plan." — nenhum contexto MdFlow.
x Plan template e generico para CLIs — nao conhece SaaS multi-tenant, camadas MdFlow, CLAUDE.md, organizacao_id, RLS.
x Zero integracao com Supabase MCP, n8n MCP, Action-Driven UI, cegueira financeira.
x Requer Bun + React/Ink + autenticacao paga — dependencia externa desnecessaria para o stack MdFlow.
x Break por pagina/behavior nao mapeia para camadas backend/front/n8n/polish do MdFlow.
O epic tem valor educacional — seus mecanismos internos sao elegantes (especialmente Flows para dependencias e o issue auto-contido). A /projeto skill ja e superior no que importa pro stack MdFlow. Com as 8 melhorias absorvidas, /projeto v3 tera tudo que o epic tem + o que o epic nao tem: Supabase MCP, n8n MCP, contexto CLAUDE.md, camadas MdFlow, multi-projeto Pro+Air+Mini, git pull automatico e zero dependencia de runtime externo.

Custo estimado: ~3-4 sessoes de trabalho. ROI: eliminar dependencia de runtime externo + assinatura + ganhar skill especificamente treinada para o stack MdFlow.
