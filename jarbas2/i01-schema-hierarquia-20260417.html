P0 A-P0-03 Regra universal dividida entre modo_alcance e categoria_empresa_id IS NULL
Tipo
Semântica SSOT
Local
olivia_modelos_grupo
Evidência
177/177 linhas com modo_alcance=1 · 2 registros com categoria_empresa_id IS NULL (tipo 0d241e7d)

A decisão D-C2-01 exige uma semântica canônica ÚNICA para "modelo universal". Hoje existem dois sinais coexistentes: modo_alcance (nunca usado de fato — sempre 1) e categoria_empresa_id IS NULL (utilizado). Ambiguidade em produção = bugs silenciosos.

Impacto: Decisões de roteamento de modelo dependem de qual coluna o consumidor ler. Recomendação: Opção A — deprecar modo_alcance.

P0 A-P0-04 fn_olivia_get_tags_com_blocos não filtra organizacao_id em olivia_modelo_tags
Tipo
Segurança RPC (SECURITY DEFINER sem guard)
Local
public.fn_olivia_get_tags_com_blocos(...)
Evidência
JOIN olivia_modelo_tags omt sem AND omt.organizacao_id = p_org_id

A RPC é SECURITY DEFINER, portanto bypassa RLS. Filtra org na tabela-raiz (modelos/tags/blocos), mas a junção com olivia_modelo_tags não tem guard. Vazamento latente: uma tag de Org A associada a modelo de Org B via row intrusiva em modelo_tags poderia aparecer para usuários da Org A.

Impacto: Risco de cross-tenant data leak em cenários de dados sujos. Baixa probabilidade, alta severidade.

-- FIX CIRÚRGICO (patch in-place na RPC): -- ... dentro da query ... LEFT JOIN olivia_modelo_tags omt ON omt.modelo_peticao_id = m.id AND omt.organizacao_id = p_org_id -- <<< adicionar esta linha LEFT JOIN olivia_tags t ON t.id = omt.tag_id AND t.organizacao_id = p_org_id

P1 A-P1-04 Banco antigo hzqoxkmxvwxejjanstdh ainda ativo

Projeto legado V1 continua recebendo tráfego de pipelines auxiliares. Decisão DEC-A05 sunset 30d recomendada. Sem sunset, dupla verdade operacional.

P1 A-P1-05 Arrays residuais empresas_categorias/especificas em olivia_modelos_peticao

Com o grupo_id canônico em olivia_modelos_grupo, os arrays denormalizados em olivia_modelos_peticao.empresas_categorias e empresas_especificas viraram sombra. Marcar como DEPRECATED e planejar drop.

P1 A-P1-06 olivia_modelo_juris sem ordem explícita

Associação modelo↔jurisprudência não tem coluna ordem INT nem relevancia NUMERIC. Consumidores dependem de ordenação incidental (created_at), o que quebra quando a UI permite reordenar.

P1 A-P1-07 fn_olivia_get_model_skeleton sem parâmetro advogado_id (GAP-01 lateral)

A RPC de skeleton resolve escopo categoria/universal mas ignora advogado_id. Modelo específico de Heloísa não é retornado quando ela está logada. Adicionar parâmetro p_advogado_id UUID DEFAULT NULL.

P1 A-P1-08 Dualidade olivia_playbooks vs olivia_playbook_juridico

Duas tabelas com finalidade sobreposta sem convenção documentada. Subproj 06 exige estender olivia_playbooks. Dualidade atual impede consenso de qual é a canônica.
