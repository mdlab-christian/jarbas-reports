4. Drifts · Gaps · Bugs (9 achados)

A-01 P0 DRIFT · Observabilidade
olivia_boris_outputs tabela inexistente — DDL aprovado, migration nunca aplicada. Observabilidade ZERO: não há onde escrever outputs de planner/juris/revisor. Bloqueante para shadow 14d.
A-02 P0 ESTADO · Feature fantasma
ENABLE_BORIS=false, shadow 14d nunca iniciado, Boris 100% dormente. Decisão CTO executada no código mas não no ambiente — feature aparece em specs e não gera valor.
A-03 P1 DUPLICAÇÃO · Código morto
2 runBoris divergentes (super_agente/boris/index.mjs vs pipeline/boris.mjs) — versão pipeline é código morto, grava ndjson no vault (risco LGPD) e não segue contratos novos.
A-04 P1 INCONSISTÊNCIA · Módulo assimétrico
juris_validator.mjs é shell de 16 linhas (re-export), enquanto planner (350L) e revisor (334L) são módulos completos. Implementação real está em pipeline/boris-juris-validator.mjs (443L), violando a organização esperada de super_agente/boris/.
A-05 P1 OBSERVABILIDADE · DEC-16
modelo_usado=NULL em todas as 5 execuções inspecionadas em olivia_historico — viola DEC-16 (congelamento Sonnet 4.6). Impossível auditar drift de modelo ou validar custo por execução.
A-06 P2 LÓGICA · Gate mal calibrado
Gate Revisor score < 8 raramente dispara — scores observados 8.2–8.6 em testes internos, estima-se acionamento em < 5% dos casos. Revisor praticamente inerte. Recalibrar com dados reais pós-shadow.
A-07 P2 AUDIT · LGPD art. 20
olivia_audit_log sem eventos boris_* (zero rows). Fragiliza direito à explicação (LGPD art. 20): não há como reconstruir decisões automatizadas do Boris.
A-08 P2 CIRCUIT · Nunca exercitado
Row subsistema='boris' em circuit breaker pronta, lógica implementada, mas nunca exercitada em produção nem staging. Risco de falhar em primeira ativação real.
A-09 P2 OPENCLAW · Dormência correta
@boris_mdbot corretamente ausente (CTO-03 aprovou blueprint dormente). Não é bug, apenas confirmação de que plano OpenClaw foi respeitado. Ativação futura condicional.

6. Avaliação crítica

Pontos fortes
  • Arquitetura coerente — 3 módulos focados, handshake claro
  • Silent degradation robusto (R3) — nunca quebra geração principal
  • Sonnet 4.6 congelado conforme DEC-16
  • Cache ephemeral em revisor reduz custo em revisões iterativas
  • Feature flag lazy + cache evita overhead quando OFF
  • Blueprint OpenClaw conservador e corretamente dormente (CTO-03)
Pontos fracos
  • Observabilidade inexistente — sem tabela, sem audit
  • Duplicação de runBoris (pipeline vs super_agente)
  • juris_validator é shell de 16L, quebra simetria
  • Shadow 14d planejado mas nunca executado
  • Gate Revisor potencialmente mal calibrado (< 5% casos)
  • modelo_usado=NULL viola DEC-16 na prática
Risco estratégico
  • CTO aprovou 3 módulos + shadow — ninguém exercitou o plano
  • Boris vira feature fantasma: aparece em decisões, não gera valor
  • Primeira ativação real em prod pode revelar falhas nunca testadas
  • Risco LGPD por ndjson em vault (pipeline/boris.mjs)
  • Custo oportunidade: Heloísa poderia estar validando juris há 14 dias
