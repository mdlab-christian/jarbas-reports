Sumário executivo

A Julia v2 reconstrói o agente como sistema robusto rodando no Mac Mini com Anthropic SDK nativo, julia-vault Obsidian (espelho do olivia-vault) e mem0 pra memória persistente multi-signal. Substitui Wati por mdzap (centralização total no MdFlow). Adota Karpathy LLM Wiki pattern (token discipline <5k tok/query mesmo com 300+ páginas). Em paralelo, o dashboard Vinicius /julia/operacao dá UI rica pra operação humana enquanto Julia opera o início do funil (recepção, auditoria, transferência). Multi-vendedor desde dia 1. 500 leads/dia ≥50% otimização.

04 🏛️

Decisões arquiteturais aprovadas pelo CTO

4.1 Substituir Wati pelo mdzap (centralizado)

Por quê: mdzap já existe, multi-número, Storage configurado, edge function pronta. Wati é intermediário desnecessário.

Como: Edge function mdzap-webhook desvia eventos agente_slug=sdr_leads pro Mac Mini. Julia responde direto via Meta API e salva em whatsapp_mensagens. UI /mdzap funciona normalmente.

4.2 User dedicado julia no Mac Mini (separação)

Decisão recomendada: criar user julia (não usar mia/mdos). Padrão já estabelecido (cecilia tem próprio user, mia também).

Estrutura:
/Users/julia/julia-vault/ ← Obsidian /Users/julia/julia-v2/ ← Express server + skills

LaunchAgent dedicado: com.midas.julia-v2.plist

4.3 Identidade dinâmica como vendedor

Julia se passa pelo Vinicius (vendedor único hoje). Multi-vendedor desde dia 1: campo whatsapp_conversas.metadata.vendedor_atribuido define qual persona Julia usa. Quando lead pergunta "você é robô?" — responde com transparência.

4.4 Híbrido humano + IA (não substitui)

Julia NÃO substitui humano. Julia faz INÍCIO do funil:

  • Recepção
  • Auditoria (consulta APIFULL, parser 6 tipos, classificação)
  • Mensagem com lista de restrições
  • Pedido de docs + transferência pro Vinicius

Vinicius pega daí. Julia ainda faz follow-up nos estados em_atendimento, aguardando_documentos, aguardando_procuracoes.

4.5 Margem segura (zero erro)

Sempre que Julia tem dúvida → escala pro humano. Não responde dúvidas técnicas/jurídicas.

Auditoria PDF + criação de lead no CRM

10.1 Pipeline auditoria
  1. Lead chega via WhatsApp com CPF
  2. Julia consulta APIFULL POST /api/scpc-boavista
  3. Parser extrai 6 tipos: PEND_FINANCEIRAS, PROTESTOS, CCF_BACEN, CCF_VAREJO, FALENCIAS, ACOES_CIVEIS
  4. Categoriza empresas via regex (cartão, banco, telefonia, energia, varejo, financeira, etc)
  5. Cria lead na coluna novos se restrições>0 (CPF limpo NÃO entra no CRM)
  6. Preenche aba "Restrições" do drawer com cada item classificado
  7. Classifica processável vs não-processável (cruzamento com tabela empresas)
  8. Gera PDF de auditoria com UI bonita (templates em 03_templates/pdf-auditoria/)
  9. Manda PDF via Meta API
  10. Move lead novos → em_atendimento
  11. Mensagem texto explicativa + lista de docs
