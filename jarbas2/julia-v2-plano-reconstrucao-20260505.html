Sumário executivo

A Julia v1 funciona em produção mas o n8n vira gargalo conforme escalamos. Code nodes JS sem typing/tests, estado distribuído n8n + Supabase + metadata, e Anthropic via HTTP raw sem prompt caching custam complexidade e dinheiro. A proposta é reconstruir Julia como super agente IA no Mac Mini com Anthropic SDK nativo, skills declarativas em Markdown (igual MdOs/Olivia), TypeScript + tests, e Express server recebendo webhooks Wati. Mantém n8n só pra integrações simples e crons auxiliares. Em paralelo, dashboard /julia/operacao no MdFlow dá ao Vinicius gestão total dos leads via UI rica enquanto a v2 está sendo construída.

Pros & contras detalhados

n8n (atual)

✅ PROS

  • Já implementado e validado em produção
  • UI visual permite Heloísa/Vinicius debugar
  • Cron, webhook, retry embutidos
  • Logs centralizados na UI
  • Toggle simples on/off por workflow
  • Comunidade grande, plugins disponíveis

❌ CONTRAS

  • Code nodes JS = sem typing, sem lint, sem tests
  • Workflow JSON não é git-friendly (diff impossível visualmente)
  • Anthropic via HTTP raw perde 50% de cost via prompt cache não usado
  • Estado espalhado (n8n vars + Supabase + lead.metadata) = debug caótico
  • Custos escalam com executions
  • Lock-in difícil (migrar workflow é trabalhoso)
  • Debugging timeout/failure complexo
  • Sem typing forte = bugs encontrados só em runtime

Mac Mini super agente (proposto)

✅ PROS

  • TypeScript + zod = bugs pegos em build
  • Anthropic SDK oficial = prompt caching (−50% custo), structured outputs, streaming, retry sofisticado
  • Skills declarativas em Markdown (igual MdOs/Olivia) — hot-reload sem restart
  • Reuso da arquitetura MdOs existente
  • Performance superior (sem overhead n8n)
  • Replay engine: reprocessar leads antigos com prompt v2
  • Tests automatizados (vitest)
  • git PRs com diff visual
  • Memória persistente sofisticada (Postgres ou Redis)
  • Express server já existe (/Users/mia/diagnostico-server/)
  • Backup Mac Mini local
  • Custo marginal zero (Mac Mini é seu)

❌ CONTRAS

  • Mac Mini = SPOF local (sem HA)
  • Internet escritório cair = Julia para
  • Webhook Wati precisa expor publicamente (Cloudflare Tunnel ou ngrok)
  • Vinicius/Heloísa não-dev tem mais dificuldade pra ver/editar
  • Concorrência (vários inbounds simultâneos) precisa locks/queue
  • Manutenção: updates, certs, monitoramento (mas já existe pra OlivIA/Cecilia)
  • Sem queue nativo (precisa BullMQ ou similar)

Anthropic SDK nativo
  • @anthropic-ai/sdk oficial
  • Prompt caching (BLOCO 1+2 cacheado, ~70% hit) → −50% custo
  • Structured outputs com tool_use → { messages, next_state, transferred } JSON
  • Streaming opcional pra responder mais rápido
  • Retry com exponential backoff embutido
Circuit breaker

Se Anthropic retornar rate limit ou erro 5xx > N vezes em janela X: pausa Julia automaticamente (ai_enabled = false), notifica Vinicius via Telegram/dashboard, tenta voltar após cooldown, logs estruturados pra debug.

Replay engine
node scripts/replay.sh \ --lead-id=acea528b... \ --prompt-version=v2

Reprocessa lead histórico com prompt novo, gera diff. Útil pra A/B test e validação.

Concorrência e locks

Lock por lead_id via Postgres advisory lock — evita race condition se 2 inbounds chegarem do mesmo lead simultaneamente.

Persistência

Continua Supabase como single source of truth. Tabelas existentes: leads, whatsapp_mensagens, auditorias, julia_ia_log, julia_followup_log, julia_config. Nada novo necessário.

Riscos e mitigações

Risco 1 · Mac Mini cai (luz, internet, hardware)

Mitigação: Tailscale + VPN backup · Health check externo (Better Stack) → SMS Christian · Fila persistente Postgres (processa backlog quando voltar) · Failback temporário: webhook aponta pro n8n se Mac Mini >5min offline (último recurso).

Risco 2 · Hot reload introduz regressão

Mitigação: Skills versionadas em git (cada save = commit automático) · Validação Zod do frontmatter antes de aceitar · Modo "dev" vs "prod" — dev hot-reload, prod só após git push tagged.

Risco 3 · Anthropic rate limit

Mitigação: Circuit breaker · Throttling (max N chamadas/min) · Tier 3 da conta (já tem) · Prompt caching reduz tokens consumidos.

Risco 4 · Vinicius/Heloísa não conseguem operar sem UI n8n visual

Mitigação: Dashboard /julia/operacao rico (substitui n8n UI) · Logs estruturados acessíveis via dashboard MdFlow · Skills em Markdown editáveis via UI ("Configurações Julia") · Documentação treinamento.

Risco 5 · Concurrency bug

Mitigação: Postgres advisory lock por lead_id · Tests de carga simulando 100 inbounds simultâneos · Idempotência via wa_message_id UNIQUE.

Ideias bônus pra evoluir Julia v2

🎓 Modo aprendizado supervisionado

Vinicius aprova/edita/rejeita cada mensagem Julia nas primeiras semanas. Cada decisão vira training data → fine-tuning prompt v3.

🧪 A/B test de copies

2 versões de recepção (alta classificação) → escolhe a que converte mais (taxa msg → docs).

🎯 Lead scoring dinâmico

Score = classificação_apifull × peso + UTM × peso + horário × peso + DDD × peso. Score alto vai pra fila prioritária Vinicius.

📥 Inbox unificado Vinicius

Combina /mdzap + /julia/operacao + auditoria do lead numa view só. Vinicius vê tudo do lead em 1 tela.

🧠 Memória persistente Julia

Lead retornou após 30 dias? Julia lembra do contexto anterior (não pergunta de novo). Tabela julia_memoria_leads.

🆘 Backup Julia → Vinicius

Se Julia falhar X vezes consecutivas pra um lead, escala automaticamente pro Vinicius com flag "atendimento manual urgente".

📡 Modo offline Julia

Se Mac Mini cair, fila persistente em Postgres. Quando voltar, processa backlog em ordem (cap 100/min pra não sobrecarregar Wati).

🔁 Replay de execuções

Reprocessar lead histórico com prompt v2 e comparar saída v1 vs v2. Útil pra validar mudanças de prompt antes de deploy.

🛍️ Skills marketplace interno

Heloísa cria skills novas (ex: "recepcao-natal", "promocao-junho") e ativa/desativa via dashboard. Hot-reload aplica imediato.

📲 Multi-canal (futuro)

Mesma Julia v2 atende WhatsApp + Instagram DM + Telegram + Web Chat. Skills agnósticas de canal.
