P1 — Funcional mas frágil

❌
continueOnFail ausente no Meta API Send
Se o token expirar, o HTTP node lança exceção não capturada. Frontend fica sem resposta e o workflow fica com execution travada. Fix: adicionar continueOnFail: true + tratar erro downstream.
❌
UNIQUE constraint em wa_message_id pode estar ausente
Se a Meta reenviar o webhook (timeout ou falha), a mesma mensagem pode ser inserida duas vezes. Fix: verificar e garantir UNIQUE(wa_message_id) em whatsapp_mensagens.
❌
Webhook responde 200 DEPOIS do processamento pesado
A Meta considera falha e reenvia se não receber 200 em ~20s. Download de vídeo de 50MB pode demorar mais. Fix: responder 200 imediatamente e processar de forma assíncrona (ver Seção 5).
❌
Sem logs de erro estruturados
Quando Save Message falha (Storage indisponível, etc.), o erro desaparece. Sem registro em webhook_requests. Impossível diagnosticar falhas em produção.

P2 — Melhorias

⚠️
Tipos de mensagem não suportados no recebimento
sticker, location, interactive (botões/listas) e contacts recebidos da Meta não são mapeados. Parse Message ignora ou falha silenciosamente.
⚠️
Retry de media download ausente
Se a Graph API retorna 401 (token expirado) ou 500 ao baixar a mídia, a mensagem é salva sem media_url. Mídia perdida permanentemente. Fix: 3 tentativas com backoff exponencial.
⚠️
mdzap_raw_logs.organizacao_id nullable
Viola isolamento multi-tenant. Logs podem conter dados de uma org sem referência. Corrigir com NOT NULL + default via trigger.
⚠️
fn_mdzap_conversas LEFT JOIN leads sem filtro deleted_at
A função pode retornar conversas vinculadas a leads deletados (soft delete). Adicionar AND leads.deleted_at IS NULL no JOIN.

5.1 — Resposta 200 Imediata + Processamento Assíncrono

O maior problema de webhooks Meta é o timeout de 20s. Se o workflow demora (download de vídeo 50MB, múltiplos inserts), a Meta considera falha e reenvia — causando duplicatas e instabilidade.

Meta POST → Webhook → Validar assinatura (x-hub-signature-256) → INSERT webhook_requests (status=pending) → Responder 200 OK IMEDIATAMENTE ← aqui, não depois → Continuar processamento em background (sub-workflow assíncrono via Execute Workflow trigger)
🔐

5.2 — Validação de Assinatura (Segurança)

A Meta assina cada requisição com HMAC-SHA256 usando o App Secret. Atualmente não validamos — qualquer pessoa pode postar no endpoint e injetar mensagens falsas.

const crypto = require('crypto'); const signature = $input.headers['x-hub-signature-256']; const rawBody = JSON.stringify($input.body); const expected = 'sha256=' + crypto .createHmac('sha256', process.env.META_APP_SECRET) .update(rawBody) .digest('hex'); if (signature !== expected) { return [{ json: { error: 'Unauthorized', statusCode: 403 }}]; }

Como Adicionar um Novo Número

O workflow de recebimento é 100% dinâmico. O node Lookup Numero usa o wa_instance_id (phone_number_id da Meta) para encontrar o registro em whatsapp_numeros e derivar organizacao_id, agente_slug e ai_enabled. Não é necessário alterar nenhum workflow n8n.

Passo 1 — Cadastrar no Supabase

-- INSERT em whatsapp_numeros (produção: qdivfairxhdihaqqypgb) INSERT INTO whatsapp_numeros ( organizacao_id, nome, numero, wa_instance_id, -- phone_number_id da Meta (campo "From" no webhook) agente_slug, -- slug único para este número ai_enabled, -- false = manual, true = Julia responde config ) VALUES ( 'uuid-da-organizacao', 'Nome do Operador', '5551988887777', '107856789012345', -- obtido no Meta Business Manager 'sdr_novo', -- deve ser único no sistema false, '{"access_token": "EAAxxxx...token-permanente"}' );

Passo 2 — Atualizar constante no Frontend

// Adicionar o novo slug à constante de roteamento const VALID_SDR_MODES = [ 'sdr_leads', 'sdr_docs', 'sdr_acordos', 'sdr_acordos_hm', 'sdr_novo' // ← adicionar aqui ];

Passo 3 — Configurar Webhook na Meta

  • Acessar Meta Business Manager → WhatsApp → Configuração → Webhooks
  • URL: https://primary-production-e209.up.railway.app/webhook/mdzap-facebook
  • Verify Token: configurar com o mesmo valor esperado pelo workflow GET
  • Campos a assinar: messages, message_deliveries, message_reads
  • Zero mudança no n8n — Lookup Numero roteia automaticamente pelo wa_instance_id

Passo 4 — Verificar roteamento

-- Verificar que o número está sendo encontrado corretamente SELECT id, nome, numero, wa_instance_id, agente_slug, ai_enabled, config->>'access_token' IS NOT NULL AS tem_token FROM whatsapp_numeros WHERE deleted_at IS NULL ORDER BY created_at DESC;
⚠️
Access Token permanente obrigatório
O campo config.access_token deve conter um token de longa duração (System User token), NÃO o token temporário de 24h do painel Meta. Tokens expirados causam falha silenciosa no envio e no download de mídia.
