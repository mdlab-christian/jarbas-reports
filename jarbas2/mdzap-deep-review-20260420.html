Fluxo Completo de Recebimento — Step by Step

9 passos · supabase/functions/mdzap-webhook/index.ts
STEP 1 Entry Point — GET / POST Handler
mdzap-webhook/index.ts:284-427
Ponto de entrada da Edge Function. GET retorna hub.challenge para verificação Meta. POST contém a mensagem real. Retorna HTTP 200 imediato via EdgeRuntime.waitUntil() — Meta recebe confirmação antes do processamento completar.
✅ GET/POST separados ✅ 200 imediato para Meta
STEP 2 normalizePayload() — Parsing de 2 formatos
mdzap-webhook/index.ts:69-158
Abstrai diferenças entre Z-API e Meta Cloud API num payload canônico.

Z-API Format (linhas 71-97): extrai phone, chatName, text.message — sem wa_message_id (retorna null).
Meta Cloud API (linhas 101-154): extrai de entry[0].changes[0].value.messages[0] — apenas o índice [0], descarta o restante do array.
⚠️ Meta batch ignorado — só messages[0] ⚠️ Z-API sem wa_message_id
STEP 3 Lookup do número em whatsapp_numeros
mdzap-webhook/index.ts:435-449
OR composto: numero.eq.${num} OR wa_instance_id.eq.${id} OR mdzap_instance_id.eq.${id}. Extrai organizacao_id, ai_enabled, agente_slug para roteamento. Retorna HTTP 200 mesmo quando número não encontrado — evita reenvio Meta.
✅ OR triplo robusto ✅ 200 mesmo sem match
STEP 4 Upsert de conversa com proteção de race condition
mdzap-webhook/index.ts:453-513
Busca conversa ATIVA por (contato_numero, numero_id, organizacao_id, status='ativa'). Se não encontrar: upsert com ignoreDuplicates: true (ON CONFLICT DO NOTHING). Se upsert retornar null (conflito UNIQUE): busca novamente — garante atomicidade. UNIQUE constraint: idx_unique_conversa_ativa em (contato_numero, numero_id) quando status='ativa'.
✅ Race condition tratada corretamente ✅ UNIQUE INDEX + ignoreDuplicates
STEP 5 Download e armazenamento de mídia
mdzap-webhook/index.ts:160-269
Meta: obtém URL de download via Graph API → baixa bytes → valida MIME → upload Storage → retorna publicUrl.
Z-API: URL pública direta → baixa bytes → upload Storage.
retryFetch: 3 tentativas com backoff LINEAR (1s, 2s) — não exponencial. Todas as falhas são silenciosas — retornam null, mensagem salva sem mídia. Mitigação parcial: media_id_failed setado em metadata para retry manual.
⚠️ Falhas de mídia silenciosas ⚠️ Retry linear — sem jitter 🔵 media_id_failed em metadata
STEP 6 INSERT mensagem com deduplicação condicional
mdzap-webhook/index.ts:544-580
Se tem wa_message_id (Meta): upsert + onConflict: 'wa_message_id' + ignoreDuplicates: true — deduplicação robusta via UNIQUE INDEX.
Se não tem (Z-API): insert puro sem deduplicação — vulnerável a duplicatas.
Duplicata detectada: retorna 200 com { status: "duplicate_ignored" }.
✅ Meta: deduplicação robusta ❌ Z-API: sem deduplicação
STEP 7 Atualização da conversa — contador nao_lidas
mdzap-webhook/index.ts:606-617
UPDATE ultimo_msg_at, nao_lidas = (conversa.nao_lidas || 0) + 1, last_msg_preview.
❌ Race condition: lê do cache em vez de UPDATE atômico
-- BUG atual: UPDATE SET nao_lidas = (conversa.nao_lidas || 0) + 1 -- CORRETO: UPDATE whatsapp_conversas SET nao_lidas = nao_lidas + 1 WHERE id = $1
STEP 8 Dispatch n8n — fire-and-forget com DLQ
mdzap-webhook/index.ts:619-662
Só executa se ai_enabled=true AND mode !== 'human'. EdgeRuntime.waitUntil() → Edge Function retorna 200 ANTES do n8n completar. Retry: 3 tentativas com backoff linear. Se falhar → INSERT em mdzap_message_queue (DLQ, TTL 24h).
✅ fire-and-forget correto ❌ DLQ sem consumidor — msgs somem em 24h
STEP 9 Dispatch transcrição de áudio — fire-and-forget
mdzap-webhook/index.ts:664-680
Se tipo=audio/voice → POST mdzap/audio-transcribe (workflow n8n separado). Assíncrono, falha silenciosa com log de erro. Não bloqueia o retorno.
✅ Assíncrono correto ⚪ Falha logada mas não alertada

STEP 1 Validação pré-envio
useMdzapSendMessage.ts:84
Verifica isNumeroOnline(conversa.numero_id) → se offline, aborta com toast.warning. Para mídia: valida tamanho (MEDIA_LIMITS) e tipo MIME antes de qualquer operação.
✅ Validação de online/offline ✅ Validação de MIME e tamanho
STEP 2 INSERT otimista — status 'pending'
useMdzapSendMessage.ts:90-110
Insere mensagem com status='pending', direcao='saida', remetente_tipo='human' antes de chamar n8n. opts.onInserted?.() dispara refetch imediato. Realtime notifica outros usuários instantaneamente.
✅ Mensagem aparece na UI imediatamente ✅ Realtime para múltiplos operadores
STEP 3 Para mídia — upload direto para Supabase Storage
useMdzapSendMessage.ts:152-243
supabase.storage.from(BUCKET).upload(path, file) → retorna publicUrl. setUploadProgress(0→80→100) para feedback visual progressivo. Diferença vs. recebimento: frontend faz o upload diretamente, n8n apenas consome a URL já existente.
✅ Upload progress feedback ✅ URL resolvida antes do n8n
STEP 4 callWebhook('mdzap/enviar-mensagem') → n8n
src/lib/webhook.ts:33-100
POST para VITE_N8N_WEBHOOK_URL/mdzap/enviar-mensagem. Headers: mdflow: MdL1501@, x-org-id, x-user-id, x-request-id. Timeout: 60s. Retry: apenas em HTTP 5xx, 3 tentativas, delays [1000, 2000, 4000]ms (linear, sem jitter). Payload inclui mensagem_id → n8n faz UPDATE, não INSERT.
✅ mensagem_id no payload ⚠️ Retry linear sem jitter
STEP 5 n8n processa e envia para Meta Cloud API
n8n recebe: { numero_id, conversa_id, contato, mensagem, tipo, mensagem_id, ... }. Faz UPDATE whatsapp_mensagens SET status='sent', wa_message_id=<wamid_real>. POST para Meta Cloud API → retorna wamid.
✅ n8n faz UPDATE (não INSERT) ✅ wa_message_id real registrado
STEP 6 markAsSent ou removePending — finalização
useMdzapSendMessage.ts:64-82
Sucesso: UPDATE status='sent' + queryClient.invalidateQueries.
Falha: UPDATE status='failed' + queryClient.invalidateQueries.
UPDATE filtrado apenas por id — confia no RLS. Sem retry automático: falha = mensagem fica como 'failed', usuário deve reenviar manualmente.
⚠️ Sem retry automático em falha ⚠️ Sem organizacao_id no UPDATE (depende RLS)
Estados possíveis de uma mensagem de saída:
pending → sent → delivered → read ↘ failed (retry manual)

Pontos de Falha Identificados

2 P0 · 4 P1 · 5 P2
P0 — CRÍTICO Perda de dados em produção hoje
P0-01 DLQ sem consumidor — mensagens somem após 24h
Onde mdzap-webhook/index.ts:645-658 O que mdzap_message_queue recebe mensagens que falharam no n8n, mas nenhum job as reprocessa. Impacto Toda mensagem que chegou durante um deploy ou falha do n8n some silenciosamente após 24h. Usuário nunca recebe resposta, equipe não sabe que falhou. Solução Criar workflow n8n MDZAP-DLQ-PROCESSOR com Schedule */5 * * * * — ver Seção 6, item 5.
P0-02 Z-API sem deduplicação — mensagens duplicadas
Onde mdzap-webhook/index.ts:568-574 O que Mensagens Z-API não têm wa_message_id → INSERT puro sem ON CONFLICT. Impacto Se Z-API reenviar webhook (retry interno), mesma mensagem aparece 2x na conversa. Julia responde duplicado, histórico corrompido. Solução Dedup temporal: UNIQUE INDEX em (conversa_id, conteudo_hash, DATE_TRUNC('minute', criado_em)) com janela de 5min, ou campo content_hash computado na Edge Function.
P1 — ALTO Degradação de experiência — corrigir em 2 semanas
P1-01 Falhas de mídia silenciosas — usuário não sabe que imagem não chegou
Onde mdzap-webhook/index.ts:161-269 O que Download/upload de imagem, áudio ou documento falha → retorna null → mensagem salva sem mídia. Nenhum aviso. Impacto Operador vê mensagem vazia. Usuário enviou imagem de documento, sistema registra "" e Julia responde sem contexto. Solução Campo media_download_status + retry via DLQ + banner no frontend quando media_id_failed está setado.
P1-02 Sem trace ID — debug de mensagem específica leva horas
Onde mdzap-webhook/ + workflows n8n O que Não há ID único que atravesse webhook → banco → n8n → resposta enviada. Impacto Para debugar um problema específico, correlacionar logs manualmente por conversa_id + timestamp — processo de 10 a 30 minutos. Solução traceId = crypto.randomUUID() no início do handler, propagado para todos os logs e payload n8n.
P1-03 Sem debounce de mensagens — Julia responde 3x para "oi preciso de ajuda"
Onde n8n orchestrator (não implementado) O que Usuário que envia 3 mensagens em 3 segundos gera 3 execuções Claude independentes. Impacto Julia responde 3 vezes fragmentado (confuso), custo triplicado, contexto incompleto em cada resposta. Solução Debounce no n8n: aguardar 3s, verificar se chegou msg mais nova. Ou Redis INCR + RPUSH + LRANGE (pattern Evolution API — ver Seção 6, item 1).
P1-04 Race condition em nao_lidas — contador pode ficar errado
Onde mdzap-webhook/index.ts:609 O que nao_lidas: (conversa.nao_lidas || 0) + 1 — lê valor do cache em JavaScript e escreve de volta. Impacto Duas mensagens simultâneas: ambas leem nao_lidas=5, ambas escrevem 6. Resultado: 6 em vez de 7. Badge de não lidas fica defasado. Solução UPDATE whatsapp_conversas SET nao_lidas = nao_lidas + 1 WHERE id = $1 — operação atômica no PostgreSQL.
P2 — MÉDIO Melhoria de qualidade — próximo mês
P2-01 Retry linear sem jitter — thundering herd
Onde index.ts:36-52 + webhook.ts Solução delay = Math.pow(2, attempt) * 1000 + Math.random() * 1000
P2-02 Realtime fallback sem backoff — 15 RPCs em 2min
Onde useMdzapConversas.ts:75-84 Solução Polling com backoff: 8s → 16s → 32s → máx 60s
P2-03 Meta batch ignorado — messages[0] apenas
Onde index.ts:105 Solução for (const msg of value.messages ?? []) {...}
P2-04 Status de entrega em JSONB — sem tick duplo/azul
Onde metadata JSONB não estruturado Solução Campo delivery_status TEXT CHECK (...) dedicado
P2-05 Workflows n8n provavelmente monolíticos — falha no Claude afeta o loop inteiro
O que Provavelmente um único workflow grande por agente_slug — difícil de testar/atualizar partes isoladas. Impacto Falha no nó Claude derruba envio de mensagem. Atualizar prompt requer deploy do workflow inteiro. Solução Separar em 4 workflows: RECEIVER → ROUTER → AI-PROCESSOR → SENDER. RECEIVER leve, retorna 200 imediato. SENDER isolado com retry independente.

Comparação com Soluções de Mercado

Chatwoot v3 · Evolution API v2

O que a nossa arquitetura faz MELHOR que o mercado

  • Persistência ANTES de chamar n8n — se n8n cair, msg já está salva no banco
  • conversa_id como session key em vez de phone number — multi-tenant correto
  • Roteamento semântico por agente_slug — mais expressivo que phone_number_id raw
  • Hub dinâmico via mdzap_ai_modules — sem deploy para adicionar número
  • waitUntil garante retorno 200 imediato sem bloquear Meta
  • fire-and-forget para n8n — sem timeout risk (Evolution API aguarda resposta síncrona)
  • UNIQUE INDEX por conversa ativa — race condition de abertura tratada corretamente
  • Realtime em 3 canais simultâneos — colaboração multi-operador nativa
Chatwoot v3 — Ruby on Rails + Sidekiq (22k stars)
Aspecto Chatwoot MdZap
Webhook response Controller retorna 200 imediato, Sidekiq processa async Edge Function retorna 200 via waitUntil ✅
Deduplicação Redis SET NX EX (atômico) + fallback DB PostgreSQL UNIQUE INDEX + ignoreDuplicates (Meta) ✅ · Z-API sem dedup ❌
Routing por número phone_number + phone_number_id dupla validação wa_instance_id simples (sem dupla validação) ⚠️
DLQ Sidekiq retry com backoff exponencial + dead queue permanente mdzap_message_queue sem consumidor — TTL 24h ❌
Status de entrega source_id + delivery_status campo dedicado no banco metadata JSONB — não estruturado ⚠️
Bot por inbox AgentBotInbox (FK, histórico, trocável por inbox) ai_enabled direto em whatsapp_numeros ⚠️
Logs Estruturados com trace_id em todos os eventos console.log texto livre — sem correlação ❌
Human handoff Assignee system + bot_paused flag nativo Não implementado ❌

n8n integration Aguarda resposta n8n síncrona (timeout risk!) fire-and-forget — Meta não é bloqueada ✅

Message Debounce com Redis
community.n8n.io/t/whatsapp-debounce-flow/225494
Redução de 45% em chamadas LLM. INCR counter com TTL=25s → só primeira msg aguarda 15s → RPUSH concatena → LRANGE envia ao AI.
INCR counter_{{sessionId}} TTL = 25s IF count == 1: WAIT 15s LRANGE chat_{{sessionId}} 0 -1 → concat → Claude/GPT → resposta ELSE: RPUSH chat_{{sessionId}} message
Adaptar: trocar remoteJid por conversa_id como session key. Resolve P1-03.

-- Anti race condition SELECT * FROM followup_queue WHERE scheduled_at <= now() AND status = 'pending' LIMIT 10 FOR UPDATE SKIP LOCKED; -- Supressão: NÃO dispara se: -- human_mode=true, opt_out=true, -- agendamento confirmado, D+3 já enviado

REF-04
Elena AI — Sub-workflows por Departamento

Hub central → Switch por intenção → Execute Workflow isolado por especialidade. Análogo ao agente_slug mas com sub-workflows independentes deployáveis.
MDZAP-RECEIVER (leve, 200 imediato) └── MDZAP-ROUTER (Switch agente_slug) ├── Execute: MDZAP-AI-JULIA ├── Execute: MDZAP-AI-ACORDOS └── Execute: MDZAP-NOTIFY-HM └── MDZAP-SENDER (isolado)
Adaptar: resolver P2-05 — separar workflows monolíticos em 4 responsabilidades claras.
REF-05
DLQ Processor — Padrão Geral para mdzap_message_queue
Pattern interno — resolve P0-01 diretamente
Workflow dedicado que consome a fila de mensagens falhadas a cada 5 minutos. Resolve completamente o P0-01 sem alteração na Edge Function.
-- Query para o nó Supabase no workflow SELECT id, payload, retry_count, conversa_id, criado_em FROM mdzap_message_queue WHERE status = 'error' AND expires_at > now() AND retry_count < 5 AND deleted_at IS NULL ORDER BY criado_em ASC LIMIT 10; -- Loop: para cada item: -- POST para orchestrator com payload original -- Sucesso → UPDATE status='processed' -- Falha → UPDATE retry_count = retry_count + 1 -- retry_count >= 5 → UPDATE status='dead_letter' (alertar equipe)
Criar: workflow MDZAP-DLQ-PROCESSOR · Schedule Trigger */5 * * * * · Notificação Slack/email quando status='dead_letter'.
