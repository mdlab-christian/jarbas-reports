Resumo Executivo — 5 Verdades que Doem

Antes de qualquer detalhe, saiba isto:

01
Nenhuma mensagem enviada pelo sistema sabe se foi entregue. O campo wa_message_id nunca e gravado apos o envio. Webhooks de status da Meta procuram um ID que nao existe no banco. O sistema inteiro simula confirmacao de entrega.
02
O sistema falha silenciosamente para contatos frios sem avisar o usuario de forma clara. A janela de 24h da Meta e ignorada. O frontend mostra erro generico enquanto a razao real e uma regra conhecida e documentada da API.
03
Row Level Security esta desabilitado na tabela de mensagens. whatsapp_mensagens e a unica tabela critica do sistema sem RLS. Dados de todos os tenants dependem exclusivamente da disciplina do codigo frontend para nao vazar.
04
O n8n e um Single Point of Failure sem circuit breaker, sem DLQ real e sem redundancia. Se o Railway cair, nenhuma mensagem e enviada, Julia para de responder e o frontend nao sabe imediatamente.
05
O MdZap foi construido task-a-task (~39 tasks) sem revisao sistemica. Cada task resolvia um problema pontual. Bugs que nao eram visiveis na task viravam "funciona, proxima feature". O resultado e um sistema com alta complexidade acidental acumulada.

Secao 1 — O que e o MdZap e por que e complexo

O MdZap e o modulo de mensageria WhatsApp do MdFlow — SaaS Legal Ops que gerencia ~5.800 processos do escritorio Midas. Integra Meta Business Cloud API + n8n no Railway + Claude Sonnet (Julia) numa arquitetura de 5 camadas com 3 numeros de WhatsApp com comportamentos radicalmente diferentes.

Secao 2 — A Causa Raiz Real: Complexidade Acidental

O MdZap tem 5 camadas de falha independentes. Um erro em qualquer ponto nao e necessariamente visivel para as outras camadas. O sistema foi construido iterativamente (~39 tasks documentadas), e cada task adicionou complexidade sem revisao sistemica.

Camada 1: React Frontend
  → INSERT otimista | Action-Driven UI | guards de slug | optimistic updates
     FALHA: data consistency, guards esquecidos, race conditions JSONB

Camada 2: Edge Functions Supabase (n8n-proxy + mdzap-webhook)
  → n8n-proxy: valida header, roteia para n8n Railway
  → mdzap-webhook: ~1000 linhas, 2 parsers, normaliza, persiste, dispara orchestrator
     FALHA: SSRF [corrigido], OOM em midia grande [corrigido], RLS ausente [ABERTO]

Camada 3: n8n Railway
  → MDZAP-ENVIAR-MENSAGEM | Orchestrator Julia | Templates | Campanhas
     FALHA: single point of failure, sem DLQ real, wa_message_id nunca gravado [P0 ABERTO]

Camada 4: Meta Graph API
  → POST /messages | GET /media | webhooks delivered/read/failed
     FALHA: janela 24h [P0 ABERTO], wamid lifecycle, rate limiting, MIME types inconsistentes

Camada 5: PostgreSQL / Supabase
  → whatsapp_conversas, whatsapp_mensagens, whatsapp_numeros, mdzap_ai_modules
     FALHA: RLS desabilitado em mensagens [P1 ABERTO], JSONB race conditions [corrigido]
A Meta Graph API para WhatsApp Business e conhecida como uma das APIs mais complexas do ecossistema de mensageria. A janela de 24h, o ciclo de vida dos wamids, os rate limits por numero e as restricoes de MIME type sao armadilhas que a maioria dos times descobre em producao, nao na documentacao.

Secao 3 — Bugs Criticos (P0)

P0 — BLOQUEANTE Bug #1: wa_message_id nunca gravado apos envio ABERTO

Sintoma: Mensagens enviadas pelo sistema nunca mostram entregue ou lido. Status fica preso em "enviado" para sempre, independentemente do que aconteceu com a mensagem.

Fluxo de falha passo a passo:

[0ms] Frontend faz INSERT em whatsapp_mensagens status: 'pending', wa_message_id: null, id: 'abc-123-uuid'
[50ms] Frontend chama n8n MDZAP-ENVIAR-MENSAGEM passando mensagem_id: 'abc-123-uuid'
[500ms] n8n chama Meta Graph API POST /messages Meta retorna: {"messages": [{"id": "wamid.HBgN5...XXXX"}]}
[501ms] n8n recebe o wamid da Meta n8n NAO faz PATCH em whatsapp_mensagens com wa_message_id = wamid. O wamid some.
[minutos] Meta dispara webhook de status: delivered Webhook busca WHERE wa_message_id = 'wamid.HBgN5...' — 0 linhas encontradas — UPDATE ignorado
[para sempre] Mensagem fica com status 'sent', wa_message_id NULL Confirmacoes de entrega e leitura sao descartadas silenciosamente para 100% das mensagens enviadas

Codigo do webhook que falha silenciosamente:

// supabase/functions/mdzap-webhook/index.ts linha ~638 // Tenta atualizar status da mensagem ao receber webhook da Meta const { error } = await supabase .from("whatsapp_mensagens") .update({ status, updated_at: new Date().toISOString() }) .eq("wa_message_id", msgId) // ← msgId EXISTE na Meta, NAO existe no banco .select("id, conversa_id, status") .single(); // Ha ate um retry de 2s porque o codigo ja previu que o n8n poderia ser lento // O problema: o n8n simplesmente nunca grava o wamid. EdgeRuntime.waitUntil( sleep(2000).then(() => updateStatusWithRetry(msgId, status)) // retry inutil );

Fix necessario no workflow n8n MDZAP-ENVIAR-MENSAGEM:

// Apos o node "Chamar Meta API", adicionar node HTTP Request: // PATCH https://[supabase]/rest/v1/whatsapp_mensagens?id=eq.{{$node["Receber Input"].json["mensagem_id"]}} // Authorization: Bearer [service_role_key] // Content-Type: application/json { "wa_message_id": "{{$node["Meta API"].json["messages"][0]["id"]}}", "status": "sent", "updated_at": "{{new Date().toISOString()}}" }
Este bug afeta 100% das mensagens enviadas pelo sistema desde o primeiro dia de producao. Nao e um bug raro — e o comportamento padrao. A confirmacao de entrega e uma ilusao no estado atual.
P0 — BLOQUEANTE Bug #2: Janela 24h da Meta nao tratada ABERTO

Sintoma: Mensagens para contatos que nao respondem ha mais de 24h falham com erro generico. Audio enviado nao chega. O usuario tenta repetir e recebe o mesmo erro sem entender o motivo real.

Regra da Meta (documentada, ignorada pela implementacao):

A Meta rejeita qualquer mensagem de conteudo livre (texto, audio, imagem, documento) para contatos que nao enviaram uma mensagem nas ultimas 24 horas. Aceita APENAS HSM templates pre-aprovados. Codigo de erro: 131047 — "Re-engagement message".

O que acontece hoje:

// Usuario tenta enviar audio para contato frio (>24h sem resposta) // Meta retorna: { "error": { "message": "Message failed to send because there were one or more errors", "type": "OAuthException", "code": 131047, // ← janela de 24h expirada -- causa real "error_subcode": 2494044 } } // n8n retorna para o frontend: { "success": false, "message": "Erro ao enviar mensagem" } // ← generico, sem contexto // Frontend mostra: "Erro ao enviar mensagem" com icone de X vermelho // Usuario entende: "o sistema esta quebrado" // Realidade: e uma restricao conhecida e documentada da Meta

Fix necessario no Orchestrator n8n:

// Antes de chamar Meta API, verificar ultima mensagem de entrada const { data } = await supabase.rpc('fn_mdzap_check_window', { p_conversa_id: conversaId }); // fn retorna { dentro_janela: boolean, ultima_entrada: timestamptz } if (!data.dentro_janela && tipo !== 'template') { return { "success": false, "action": "need_template", "message": "Janela de 24h expirada. Use um template aprovado para reengajar este contato.", "error_code": "WINDOW_EXPIRED" }; }

Secao 4 — Bugs P1 (Impacto Direto no Usuario)

P1 — IMPACTO DIRETO Bug #3: RLS desabilitado em whatsapp_mensagens ABERTO

Sintoma: Nao ha sintoma visivel em condicoes normais. Em condicoes de bug no frontend (filtro esquecido, query sem .eq('organizacao_id', orgId)), mensagens de outros tenants podem ser retornadas e exibidas.

Estado atual no banco:

-- whatsapp_mensagens hoje SELECT relname, relrowsecurity FROM pg_class WHERE relname = 'whatsapp_mensagens'; -- Resultado: -- relrowsecurity = FALSE ← RLS DESABILITADO

O que deveria existir:

ALTER TABLE whatsapp_mensagens ENABLE ROW LEVEL SECURITY; CREATE POLICY "tenant_isolation" ON whatsapp_mensagens USING ( conversa_id IN ( SELECT id FROM whatsapp_conversas WHERE organizacao_id = get_org_id() ) );
Todo o resto do sistema (leads, processos, acordos, clientes, financeiro) tem RLS habilitado. whatsapp_mensagens e a excecao critica. Numa auditoria de conformidade LGPD, este e o primeiro item a reprovar.
P1 — IMPACTO DIRETO Bug #4: fn_mdzap_conversas sem filtro deleted_at em leads ABERTO

Sintoma: Conversas aparecem vinculadas a leads que foram deletados (soft delete). O nome do lead aparece em branco ou como valor nulo dependendo de como o frontend trata a ausencia.

-- fn_mdzap_conversas hoje (simplificado) LEFT JOIN leads l ON wc.lead_id = l.id -- Falta: AND l.deleted_at IS NULL -- Resultado: leads com deleted_at IS NOT NULL aparecem vinculados -- Fix: uma linha de SQL LEFT JOIN leads l ON wc.lead_id = l.id AND l.deleted_at IS NULL -- adicionar esta linha

Secao 5 — Bugs Corrigidos em 2026-04-22

CORRIGIDO Race Condition JSONB — metadata da conversa corrompido sob concorrencia CORRIGIDO 2026-04-22

O que era: Toda atualizacao de metadata da conversa usava read-then-write no frontend, criando uma janela de concorrencia que corrompria dados com multiplos atores simultaneos (Julia + humano).

Codigo antigo (race condition garantido):

// CODIGO ANTIGO -- race condition const { data } = await supabase .from('whatsapp_conversas') .select('metadata') .eq('id', conversaId) .single(); // snapshot stale! await supabase .from('whatsapp_conversas') .update({ metadata: { ...data.metadata, // spread do snapshot mode: 'human' // sobrescreve tudo! } });

Cenario de corrupcao real:

// Em 200ms, dois eventos: // Julia responde: // metadata = { mode: 'julia', last_msg_preview: 'Ola!' } // Humano assume (leu snapshot antigo): // metadata = { mode: 'human' } // ^ apagou o preview da Julia! // Resultado: // - conversa some da lista (preview null) // - modo incorreto exibido // - dados corrompidos sem erro visivel

Solucao aplicada — 2 RPCs PostgreSQL com merge atomico:

-- fn_mdzap_update_conversa_saida: jsonb_set em cadeia, sem SELECT previo UPDATE whatsapp_conversas SET metadata = jsonb_set( jsonb_set(metadata, '{mode}', '"human"'), '{last_msg_preview}', to_jsonb(p_preview) ) WHERE id = p_conversa_id; -- fn_mdzap_update_conversa_metadata: operador || para merge generico UPDATE whatsapp_conversas SET metadata = metadata || p_patch_jsonb -- merge atomico, sem read previo WHERE id = p_conversa_id;
CORRIGIDO OOM Risk na Edge Function — downloads de midia carregavam arquivo inteiro na RAM CORRIGIDO 2026-04-22

O que era: Downloads de midia usavam response.arrayBuffer() que carrega o arquivo inteiro na RAM. Documentos Meta podem ter ate 100MB. Edge Functions tem ~150MB de RAM. Uma unica mensagem com documento pesado poderia derrubar a Edge Function e interromper o processamento de todos os webhooks em fila.

// CODIGO ANTIGO -- OOM em arquivos grandes const buffer = await response.arrayBuffer(); // 100MB na RAM de uma vez // CODIGO NOVO -- safeReadBody() com streaming e limite 50MB async function safeReadBody(response, maxBytes = 50 * 1024 * 1024) { const reader = response.body.getReader(); const chunks = []; let totalBytes = 0; while (true) { const { done, value } = await reader.read(); if (done) break; totalBytes += value.length; if (totalBytes > maxBytes) { reader.cancel(); // aborta e libera memoria throw new Error(`Media excede limite de ${maxBytes / 1024 / 1024}MB`); } chunks.push(value); } return mergeChunks(chunks); }
CORRIGIDO SSRF no n8n-proxy — destinos internos do Railway alcancaveis CORRIGIDO 2026-04-22

O que era: O n8n-proxy validava o path com regex ^[a-z0-9/_-]+$ mas nao restringia os prefixos de destino. Qualquer path valido na regex podia ser chamado — incluindo paths internos do Railway, metadata endpoints do container ou outros servicos na rede privada.

// CODIGO ANTIGO -- sem restricao de destino if (!/^[a-z0-9\/_-]+$/.test(path)) { return error(400); } // Qualquer path passava -- /internal/admin, /../etc tambem // CODIGO NOVO -- whitelist de prefixos const ALLOWED_PREFIXES = ['mdflow/', 'mdzap/', 'olivia/', 'jarbas/', 'ctrl/']; if (!ALLOWED_PREFIXES.some(p => path.startsWith(p))) { return error(403, 'Destino nao permitido'); }

Secao 7 — n8n Railway: Single Point of Failure

P1 — IMPACTO DIRETO n8n sem redundancia, circuit breaker ou DLQ real ABERTO

O que NAO existe hoje:

  • Circuit breaker (se n8n cair, frontend nao sabe imediatamente)
  • Fila de retry persistente real
  • Health check ativo com alertas automaticos
  • Dead letter queue com reprocessamento automatico
  • Redundancia de container (single instance Railway)
  • Memory leak detection / auto-restart em OOM

Cenarios de falha e impacto:

  • Railway cai: zero envios, Julia para, sem aviso imediato
  • Leak de memoria: timeout crescente nos webhooks
  • Workflow trava: mensagem fica pending para sempre
  • Cold start apos inatividade: 2-3s de latencia extra
  • Deploy do n8n: janela de downtime sem rollback instantaneo
O useMdzapStatus faz ping a cada ~30s. Em caso de queda, o usuario so percebe apos 30 segundos mais o tempo de tentativa. Mensagens enviadas nesse intervalo ficam pending sem feedback claro de que o sistema caiu.

Secao 8 — Dois Provedores no Mesmo Endpoint: Z-API + Meta

P2 — DIVIDA TECNICA mdzap-webhook com ~1000 linhas e dois parsers distintos

O mdzap-webhook suporta dois formatos de payload distintos no mesmo endpoint. Um bug em qualquer parser afeta o processamento de todos os webhooks de todos os tenants.

// mdzap-webhook/index.ts -- dois branches de normalizacao no mesmo endpoint // Formato Meta Cloud API (oficial): // entry[0].changes[0].value.messages[0].type // entry[0].changes[0].value.contacts[0].profile.name // Suporta: text, image, audio, video, document, sticker, reaction, interactive // Formato Z-API (legado): // phone, chatName, type, body, caption, fileName... // Suporta: text, image, audio, document function normalizePayload(body) { if (isMetaFormat(body)) { return parseMetaPayload(body); // ~200 linhas de logica } else if (isZApiFormat(body)) { return parseZApiPayload(body); // ~150 linhas de logica } else { throw new Error('Formato desconhecido'); } // Total da funcao: ~1000 linhas com todos os tipos de midia // Complexidade ciclomatica: muito alta -- teste unitario seria extenso }

Secao 13 — Por que tudo isso aconteceu: a verdade brutal

Razao 1: Construcao iterativa sem revisao sistemica

O MdZap foi construido task-a-task (~39 tasks documentadas). Cada task resolvia um problema pontual sem auditar o sistema inteiro. Bugs que nao eram visiveis na task viravam "funciona, proxima feature". O bug do wa_message_id e o exemplo perfeito: o workflow n8n funcionava (Meta recebia a mensagem), mas o feedback de volta para o banco nunca foi implementado. Passou por 39 tasks sem ser detectado porque nenhuma task verificou o banco apos o envio.

Razao 2: Complexidade da Meta API subestimada desde o inicio

A API do WhatsApp Business e conhecida por ser complexa, com restricoes nao obvias: janela de 24h, wamid lifecycle, MIME types inconsistentes entre provedores, webhooks agressivos que podem chegar fora de ordem, rate limiting por numero de telefone e namespace de templates. Cada uma dessas restricoes foi descoberta quando quebrou em producao, nao na documentacao durante o design.

Razao 3: n8n como caixa preta com contrato implicito

Os workflows n8n sao desenvolvidos separadamente (Mateus), testados separadamente, e o contrato de dados (frontend → n8n → banco) raramente e auditado end-to-end. O bug do wa_message_id so foi descoberto numa auditoria de integracao. Sem testes automatizados do contrato de dados, qualquer mudanca num workflow pode quebrar o pipeline silenciosamente e ninguem percebe ate verificar o banco manualmente.

Razao 4: JSONB usado como document store em campos que mudam frequentemente

Usar metadata JSONB para campos que mudam frequentemente (mode, agente_slug, last_msg_preview) cria problemas de concorrencia que so aparecem sob carga real ou com multiplos atores simultaneos. O PostgreSQL tem operadores atomicos para JSONB, mas o padrao mais natural para desenvolvedores frontend (read-then-write) cria race conditions garantidas que corrompem dados silenciosamente.

Razao 5: Tres modulos com uma implementacao — crescimento organico sem refatoracao

Em vez de 3 rotas distintas com componentes separados desde o inicio, o sistema cresceu com condicionais. O primeiro numero (sdr_leads) foi implementado completo com Julia. Os outros dois foram adicionados como "variantes" do mesmo componente. O resultado e um sistema onde adicionar um comportamento especifico para um slug exige lembrar de todos os pontos onde existe um conditional — que hoje sao 6+, e provavelmente existem mais nao detectados ainda.
