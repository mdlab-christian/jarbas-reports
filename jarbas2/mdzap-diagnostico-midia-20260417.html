Seção 2 — Diagnóstico Detalhado

P0 BUG-01 Áudio recebido — não toca, 0 segundos
Causa raiz confirmada
A Edge Function mdzap-webhook salva a URL CDN da Meta diretamente em whatsapp_mensagens.media_url. Essa URL expira em ~5 minutos e requer autenticação Bearer token da Meta. Não é pública e é bloqueada por CORS no browser.
Evidência no código
// MdzapMsgAudio.tsx <audio crossOrigin="anonymous" src={m.media_url}> // Quando media_url = "https://lookaside.fbsbx.com/..." (Meta CDN) // 1. CORS preflight → bloqueado // 2. audio.duration = NaN → fmt(NaN) retorna "0:00" // 3. .play().catch(() => null) → erro silenciado
⛔
Workflow inbound não trata mídia
Message Grouper DmVY7i9I3UIzP08U não faz download nem reupload de mídia recebida.
⚠️
Impacto
100% dos áudios recebidos são inacessíveis após ~5 minutos. Player mostra "0:00" permanentemente.
P0 BUG-02 Áudio enviado — aparece no front, não chega ao cliente
Causa raiz A — Formato de áudio incompatível
MdzapAudioRecorder.tsx usa getSupportedMimeType() que retorna audio/ogg;codecs=opus como primeira opção no Chrome. O MediaRecorder do Chrome produz OGG sem cabeçalho DURATION — bug conhecido da MediaRecorder API. A Meta processa o envio e retorna wa_message_id, mas o receptor não consegue reproduzir o áudio no WhatsApp.
// PROBLEMA: getSupportedMimeType() — ordem atual no código const types = [ 'audio/ogg;codecs=opus', // Chrome seleciona este → OGG corrompido ❌ 'audio/mp4', 'audio/mpeg' ]; // CORREÇÃO: inverter — mp4 (AAC) primeiro const types = [ 'audio/mp4', // AAC/MP4 = formato nativo WhatsApp ✅ 'audio/mpeg', 'audio/ogg;codecs=opus' // fallback apenas se mp4/mpeg não suportado ];
Causa B — descartada
Bucket mdzap-media: public: true confirmado correto. A Meta consegue baixar o arquivo. Não é a causa do bug.
P0 BUG-03 PDF recebido — "anonymous", não baixa
Causa A — filename não extraído do payload
O workflow inbound não extrai message.document.filename do payload Meta e não salva em metadata.filename. O componente exibe {filename || 'Documento'} e cai no fallback. Quando a URL Meta CDN expirada abre em _blank, o browser exibe "anonymous" como nome.
Causa B — URL Meta CDN expirada (mesma raiz do BUG-01)
O <a href="[meta-cdn-url]" target="_blank"> abre página de erro 403. Nenhum arquivo é baixado pelo usuário.
Causa C — Workflow crítico INATIVO
Confirmado: "/MDZAP — Document Received" está com "active": false. Este workflow é responsável por processar documentos recebidos (download → Supabase Storage → URL permanente). Como está inativo, nenhum documento recebido tem URL permanente desde o deploy.
P1 BUG-04 Preview inline de mídia — feature gap
Situação atual
MdzapMsgThread usa MdzapMediaPreview como modal full-screen (fixed inset-0). Ao selecionar arquivo, abre popup cobrindo toda a tela — UX bloqueante e fora do padrão esperado.
Comportamento esperado
Preview compacto acima do campo de input: thumbnail, nome do arquivo, legenda editável, botão enviar — sem modal, sem overlay. Igual ao WhatsApp Web e GPT.
Arquivos a modificar
MdzapMsgThread.tsx MdzapMsgInput.tsx MdzapMediaPreview.tsx
Layout do preview inline
┌─────────────────────────────────────────────────────┐ │ [thumbnail] nome-arquivo.pdf [X] │ ← preview strip │ [_____ legenda opcional _______________________] │ ← caption input ├─────────────────────────────────────────────────────┤ │ [📎] [✨] [ textarea mensagem... ] [🎙️] [▶] │ ← input normal └─────────────────────────────────────────────────────┘
P1 BUG-05 Alta latência (3–8 segundos)
1 — n8n cold start Railway
Container pode estar suspenso no Railway. Primeira execução após inatividade adiciona +2–3s antes de qualquer processamento real.
2 — Cadeia sequencial no n8n
Workflow 5nZxUubzqoeYInSO: Log → Auth → Rate Limit → Config Fetch → Meta API → Save = 6 HTTP requests sequenciais = 2–5s acumulados
3 — Realtime debounce duplo
Debounce 300ms no Supabase Realtime + re-fetch 300ms em useMdzapMensagens = +600ms por mensagem recebida.
4 — Race condition optimistic vs webhook
INSERT otimista aparece primeiro no chat. UPDATE de status chega após webhook retornar, criando "saltos" visuais na thread. Afeta especialmente envios em SDR/Acordos.

Seção 6 — Verificação de Sucesso

  • BUG-01 resolvido — Áudio recebido de cliente: player mostra duração correta (ex: "0:23"), reproduz sem erros de CORS. URL permanente no Supabase Storage em vez de CDN Meta.
  • BUG-02 resolvido — Áudio enviado pelo escritório: status muda de ⏱ (pending) para ✓ (sent) em menos de 5s. Receptor ouve o áudio normalmente no Android e iOS.
  • BUG-03 resolvido — PDF recebido: exibe nome correto extraído do payload Meta (ex: "comprovante_pagamento.pdf"), botão download funciona sem erro 403.
  • BUG-03 complementar — Imagem recebida: carrega dentro de 2s, thumbnail visível na thread do chat, sem URL expirada.
  • BUG-04 resolvido — Preview inline ao selecionar arquivo: strip compacto aparece acima do textarea, não como modal full-screen. Campo de legenda editável. Botão X cancela a seleção.
  • BUG-05 — latência — Envio de mensagem de texto: menos de 2s do clique em Enviar até aparição confirmada para o cliente. Sem cold start perceptível no Railway.
  • BUG-05 — race condition — Mensagens enviadas e recebidas quase simultaneamente aparecem em ordem correta de timestamp, sem "saltos" visuais na thread.
  • Regressão — Envio de texto e imagem via link externo continuam funcionando. Message Grouper não alterou comportamento para mensagens de texto simples.
