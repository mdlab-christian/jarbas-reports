Seção 2 — Pontos de Atenção do CC

P0
criarLeadEAuditarInline — vínculo da conversa
Verificar se após criar o lead ele é corretamente vinculado à conversa (whatsapp_conversas.lead_id = novo_lead_id) antes de disparar a auditoria Boa Vista. Se o UPDATE falhar silenciosamente, o lead existe mas a conversa continua desvinculada.
P0
Redirecionamento para aba Restrições
Após "Auditoria + Cadastro", confirmar que setActiveTab('restricoes') é chamado e a aba realmente muda. Verificar se o state de tab está elevado corretamente (prop drilling vs. estado local do componente).
P1
onCadastrarClick no CrmAuditoriaPanel
Confirmar que onCadastrarClick está sendo passado corretamente para o painel e que o callback abre o MdzapLeadCadastroDrawer. Verificar se o prop está definido na interface do componente e passado no JSX pai.
P1
Doc Reader badges — docReaderFields (Map)
Confirmar que docReaderFields (Map) está populado quando pending_lead_data existe em conversa.metadata. Um Map vazio ou undefined não exibirá badges e não dará erro visível.
P1
auditoriaId — prop drilling até o drawer
Confirmar que auditoriaId está sendo passado na cadeia completa: CrmAuditoriaPanel → LeadRestricoesMdzapTab → MdzapLeadPanel → MdzapLeadCadastroDrawer. Qualquer quebra na chain resulta em drawer sem dados de restrição.
P2
Empty state — sem lead e sem pending_lead_data
Confirmar que o formulário exibe campos vazios (strings vazias) e não undefined. Acessar undefined.campo causaria crash silencioso ou erro de runtime não tratado.

Seção 3 — Prompt para Lovable

🤖
Prompt pronto para enviar ao Lovable
Cole diretamente no chat do Lovable
Preciso que você faça uma auditoria completa do fluxo de "Informações do Lead" no MdZap. **O que deve funcionar:** **Aba Dados (painel direito do MdZap):** Quando a conversa NÃO tem lead vinculado: - O formulário aparece vazio (ou preenchido com dados extraídos pelo Doc Reader) - Existe um banner azul se há dados do Doc Reader disponíveis - Existe um botão verde "Auditoria + Cadastro" no final do formulário - O botão fica desabilitado se o CPF não estiver preenchido ou for inválido - Ao clicar no botão: cria o lead, vincula à conversa, dispara a consulta Boa Vista, e muda automaticamente para a aba "Restrições" Quando a conversa JÁ TEM lead vinculado: - O formulário aparece preenchido com os dados do lead - Campos que o Doc Reader extraiu mostram um badge colorido (verde/amarelo) - Se o Doc Reader encontrou valores diferentes dos cadastrados: aparece um banner laranja - O botão "Salvar dados" aparece APENAS quando o operador alterou algum campo **Aba Restrições:** Quando NÃO tem lead: aparece mensagem "Cadastre o lead" + botão para ir à aba Dados Quando TEM lead: - Aparece o painel de auditoria Boa Vista com status em tempo real (aguardando → concluído → erro) - Quando a auditoria conclui: aparece botão "Cadastrar Restrições" - Ao clicar em "Cadastrar Restrições": abre um drawer lateral com 2 etapas - Etapa 1: confirmar os dados do lead (editável) - Etapa 2: ver as restrições encontradas e selecionar quais cadastrar - Abaixo do painel de auditoria: lista de restrições já cadastradas para este lead **O que eu preciso que você verifique:** 1. O botão "Auditoria + Cadastro" existe e funciona na aba Dados? 2. Após clicar, a aba muda para "Restrições" automaticamente? 3. O CrmAuditoriaPanel aparece corretamente na aba Restrições? 4. O botão "Cadastrar Restrições" abre o drawer de 2 etapas? 5. Os badges do Doc Reader aparecem nos campos quando há dados extraídos? 6. O lead criado fica vinculado à conversa (o nome aparece no cabeçalho)? Para cada item: diga se está implementado, se está parcialmente implementado, ou se está faltando. Se estiver com problema, explique o que está errado no código. Gere um HTML com o diagnóstico completo, fluxograma de como está funcionando atualmente, e lista de pendências com prioridade.
