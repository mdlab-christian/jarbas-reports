MdZap Atual — Diagnóstico

Arquitetura vigente, números ativos, pontos fortes e gaps identificados

Fluxo completo — inbound a realtime
Meta API Payload inbound: phone_number_id, wa_instance_id, mensagem, mídia
mdzap-webhook — Supabase Deno Edge Function Lookup whatsapp_numeros por phone_number_id / wa_instance_id → resolve agente_slug
whatsapp_conversas UPSERT: cria ou encontra conversa. metadata.agente_slug = numero.agente_slug (3 níveis de cascata)
whatsapp_mensagens INSERT mensagem com metadata completo (tipo, mídia, status)
Dispatch condicional — n8n Orchestrator SE ai_enabled == true E mode != "human" → POST n8n orchestrator → Julia processa e responde
Frontend — Realtime Supabase canal mdzap-conversas-rt-{orgId} → setQueryData otimista | canal mdzap-thread:{conversaId} → refetch debounce 300ms

Pontos Fortes — Arquitetura Atual

01
Isolamento de IA determinístico

ai_enabled por dado no banco, não por código. Impossível vazar IA entre números.

02
Webhook universal

Suporte nativo Meta Cloud API + Z-API com normalização de payload no edge.

03
Cascata de agente_slug

3 níveis: metadata → número → default. Garante retrocompatibilidade total.

04
Hub dinâmico

Novo número via mdzap_ai_modules sem deploy de frontend.

05
Realtime com fallback

Se Supabase Realtime cair, polling agressivo a cada 8s mantém UX.

06
INSERT otimista no send

Mensagem aparece imediatamente antes do webhook n8n retornar.

07
Media retry assíncrono

Campo media_id_failed no metadata para retry de mídia sem bloquear fluxo.

08
Dead Letter Queue

Tabela mdzap_message_queue com TTL 24h para mensagens que falharam no dispatch.

Gaps Identificados

B01 — Bug P0 fn_mdzap_conversas sem deleted_at

LEFT JOIN leads sem filtro deleted_at IS NULL. Leads deletados aparecem nas conversas.

B02 — Segurança P0 RLS desabilitado em whatsapp_mensagens

Risco real de vazamento cross-tenant. Policy modelo A (organizacao_id = get_org_id()) ainda não aplicada.

Realtime — P1 Canal por orgId, não por numero_id

Invalidações cruzadas entre agentes diferentes da mesma org. Todos os agentes reagem a eventos de qualquer número.

Sem debounce — P1 Rajadas geram múltiplas execuções Julia

Usuário enviando "oi" + "preciso de ajuda" em 3s dispara 2 execuções n8n e 2 respostas da IA.

Sem handoff — P1 Human handoff não implementado

Não há como pausar a IA por conversa. Operador não pode assumir sem desabilitar o módulo inteiro.

Contexto IA — P2 Sessão de IA em JSONB da conversa

Contexto acumulado em metadata JSONB. Sem janela de memória estruturada nem TTL por sessão.

Soluções de Referência

Open-source e comerciais analisados como benchmark para arquitetura multi-número e multi-agente

Chatwoot
github.com/chatwoot/chatwoot  ·  Ruby on Rails + Vue.js
⭐ 22k
Arquitetura Multi-Número

Cada número é um Inbox com Channel::Whatsapp. Relação polimórfica (channel_type, channel_id) — Inbox agnóstico ao tipo de canal. Suporta WhatsApp, Email, Slack, API simultaneamente.

Roteamento

WhatsApp::WebhookSetupService registra webhook por phone_number_id. Inbound extrai phone_number_id → lookup Channel::Whatsapp → pipeline agnóstica (Inbox → Conversation → Message).

Isolamento de IA

Tabela AgentBotInbox (junção bot↔inbox): um bot por inbox via FK. Conversa criada com status pending → bot recebe eventos → ao decidir handoff, muda status para open → para de receber. Troca de bot = atualizar FK.

tabela mdzap_numero_bots(numero_id, modulo_slug, ativo, ativado_em) como substituição futura de campos diretos em whatsapp_numeros
Multi-Inbox Handoff nativo Open-source
Evolution API
github.com/EvolutionAPI/evolution-api  ·  Node.js / TypeScript
⭐ 14k
Arquitetura Multi-Número

Cada número é uma Instance gerenciada pelo WAMonitoringService. Cada instância tem conexão própria, FK instanceId no banco, cache prefixado com nome da instância, webhook independente.

Roteamento — invariante chave

Payload SEMPRE inclui instanceName e instanceId no nível RAIZ — roteamento acontece ANTES de abrir o conteúdo da mensagem. Exemplo:

{ "event": "messages.upsert", "instance": "meu-numero-1", "data": { "key": { "remoteJid": "5551999...@s.whatsapp.net" } } }
garantir que instancia_id seja a PRIMEIRA operação no webhook, documentado como invariante — antes de qualquer parsing de payload
Multi-Instância Cache prefixado TypeScript
OpenClaw
github.com/openclaw/openclaw  ·  Multi-Agent Framework
Binding declarativo de agente

Session key: agent:<agentId>:<channel>:<accountId>:<peerId>. Cada agente tem workspace isolado, personalidade própria (AGENTS.md, SOUL.md), memória separada. Zero cross-talk entre agentes salvo configuração explícita.

Rastreabilidade por mensagem

agente_slug registrado POR MENSAGEM DE SAÍDA, não apenas por conversa. Preserva histórico de qual bot respondeu cada mensagem individualmente.

registrar agente_slug por mensagem de saída em whatsapp_mensagens (metadata.respondido_por) para rastreabilidade completa
Multi-Agente Isolamento total
Whatomate
github.com/shridarpatil/whatomate  ·  Multi-tenant SaaS
Modelo de dados idêntico ao nosso

Organization → Numbers → Conversations → Messages. Multi-tenant com organizações isoladas, múltiplos provedores WhatsApp (Meta Cloud, Twilio), chatbot IA (OpenAI, Anthropic, Google).

Confirmação de arquitetura

Valida que nossa estrutura de tabelas (organizacao_id em tudo, RLS, provider-agnóstico) está alinhada com o estado da arte do segmento.

confirma que a hierarquia Organization → Numbers → Conversations é o padrão correto para multi-tenant WhatsApp
Multi-Provider Multi-Tenant IA integrada
✅
WASenderApi — Confirmação de padrão (wasenderapi.com/blog)

"Gateway centralizado recebe todo tráfego, inspeciona instance_id/phone_number_id, faz DB lookup, roteia para inbox correto. A separação por número deve acontecer no gateway ANTES de qualquer lógica de negócio." — Exatamente o que nosso mdzap-webhook já faz. Nossa arquitetura está correta.

Cenários n8n Encontrados

Templates e padrões de automação WhatsApp + IA encontrados na comunidade n8n

⏱️
1. Message Debounce com Redis
community.n8n.io · WhatsApp + Redis + LLM
🔗 Ver discussão na comunidade

Agrega mensagens rápidas em uma só chamada à IA. Janela: 10–15 segundos. Redução de 45% em chamadas à API de LLM. Evita múltiplas respostas para rajadas de mensagens do mesmo usuário.

INCR counter_{{sessionId}} TTL=25s # Se counter == 1 → primeira mensagem → aguarda 15s RPUSH chat_{{sessionId}} → armazena texto LRANGE chat_{{sessionId}} → recupera fila concatenada → envia ao AI Agent como mensagem única
⚠ GAP no MdZap Adaptar: usar conversa_id como session key (não remoteJid)
🤝
2. Human Handoff com Flag Supabase
n8n.io/workflows/11648 · Gemini + Twilio + Supabase RAG
🔗 Ver workflow

Campo human_active na tabela de conversas. Webhook verifica flag ANTES de chamar IA. Timeout automático: 2h sem resposta humana → flag limpa → IA retoma automaticamente.

Adaptação direta: adicionar ai_paused_until TIMESTAMPTZ em whatsapp_conversas. Verificar no início do orchestrator n8n: se now < ai_paused_until → não chamar Claude.
⚠ GAP no MdZap ✓ Adaptação simples
🎯
3. ZapPro — Triage Dupla + Handoff + Follow-up
community.n8n.io · Template de produção completo
🔗 Ver template

Template de produção com 4 módulos: Triage dupla (regex urgência + Claude 3.5 Haiku tagging com marcador {handoff}), Handoff (notifica time → "unlock" reativa bot), Follow-up (cron D+1 e D+3 com supressão inteligente), Concorrência (FOR UPDATE SKIP LOCKED no PostgreSQL).

Útil para: fluxo sdr_acordos — follow-up automático de propostas D+1 e D+3 para escritórios de advocacia.
✓ Oportunidade sdr_acordos P3
🕸️
4. Elena AI — Orquestrador com Sub-workflows
n8n.io/workflows/5415 · GPT-4o + WhatsApp + Redis
🔗 Ver workflow

Hub central → Switch → sub-workflows especializados (agenda, despesas, contatos, email). Session ID: remoteJid da Evolution API. Memória: Redis.

Análogo ao nosso agente_slug roteando para orchestrators diferentes. Cada agente é um workflow n8n separado chamado via Execute Workflow.
✓ Padrão confirmado Já fazemos
🏥
5. Healthcare Multi-Agent Orchestrator
n8n.io/workflows/7794 · GPT-4 + PostgreSQL + Google Sheets
🔗 Ver workflow

Orchestrator como "traffic controller" → despacha para agentes especializados. PostgreSQL para conversation memory. Buffer window: últimas 10 mensagens por user ID.

Padrão de memória: tabela n8n_chat_histories(session_id TEXT, message JSONB, created_at). Usar conversa_id como session_id.
⚠ Melhoria futura P2
🧭
6. Intent Routing + Buffer Window Memory
n8n.io/workflows/10240 · AI intent classification
🔗 Ver workflow

AI classifica mensagem → Switch → handler correto (FAQ, escalonamento, reserva, etc.). Alternativa ao roteamento por número: roteamento por intenção dentro do mesmo número.

Padrão alternativo: triage por intenção antes de despachar para Julia, podendo filtrar urgências e reduzir carga no modelo principal.
✓ Referência triage P3

Recomendações Priorizadas

10 ações ordenadas por impacto e urgência — P0 a P3

P0
Bugs Críticos — Implementar imediatamente
Funcionamento incorreto ou risco de segurança em produção
01
fn_mdzap_conversas — filtro deleted_at

LEFT JOIN leads sem deleted_at IS NULL. Leads deletados aparecem como contatos ativos nas conversas. Adicionar filtro na query da RPC.

02
Habilitar RLS em whatsapp_mensagens

Risco cross-tenant confirmado. Aplicar policy modelo A: organizacao_id = get_org_id(). Checar índice em organizacao_id antes.

P1
Alto Impacto — Próximas 2 semanas
Gaps com impacto direto na qualidade das conversas IA
03
Message Debounce Redis

Evita múltiplas respostas Julia para rajadas. Adaptar: usar conversa_id como session key (não remoteJid). Janela recomendada: 12s. Integrar como primeiro nó do orchestrator n8n.

04
Human Handoff Flag

Adicionar ai_paused_until TIMESTAMPTZ em whatsapp_conversas. No orchestrator n8n: verificar now < ai_paused_until → noop. Timeout automático: 2h. UI: botão "assumir conversa" no header da thread.

P2
Melhorias de Escala
Refinamentos arquiteturais para crescimento sustentável
05
Realtime filter por numero_id

Canal atual por orgId gera invalidações cruzadas. Filtrar por numero_id elimina ruído entre agentes.

06
Tabela mdzap_numero_bots

Inspirado no AgentBotInbox do Chatwoot. Histórico de ativação, múltiplos bots por número, auditoria de troca.

07
agente_slug por mensagem de saída

Registrar metadata.respondido_por em cada mensagem enviada pela IA. Rastreabilidade completa de qual bot respondeu o quê.

P3
Oportunidades Futuras
Features de diferenciação para próximo ciclo
08
Follow-up automático D+1/D+3

Pattern ZapPro para sdr_acordos. Cron n8n com supressão inteligente: não dispara se já houve resposta do destinatário.

09
Triage dupla pré-Julia

Regex urgência + Claude Haiku tagging antes de despachar para Julia. Reduz custo de tokens e filtra ruído.

10
RAG contextual com pgvector

Infra já disponível no Supabase. Base de conhecimento da Julia com busca semântica por documentos e histórico de casos.
