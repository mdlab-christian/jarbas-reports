P0 — Bloqueantes para Producao Manual

3 bugs que impedem o atendimento basico de funcionar
P0-01
Erro ao salvar dados do lead no sdr_leads
Atendente relata "sempre da algum erro" ao tentar salvar dados. Sem feedback visual, sem toast, sem indicacao do motivo.
Causa raiz
Conversa nao tem lead_id. Hook useLeadMutations.saveLead() retorna cedo com if (!conversa?.lead_id || !organizacao_id) return — retorno silencioso sem toast.
Solucao necessaria
1. Toast de erro quando lead_id ausente: "Lead nao vinculado. Crie o lead primeiro."
2. Botao "Criar lead" visivel e proeminente quando conversa.lead_id === null
3. Apos criar lead: reload do painel + toast de sucesso
hooks/useMdzapLeadMutations.ts (ou equivalente)
P0-02
Conversa nao vinculada ao lead_id automaticamente
Quando lead entra pela primeira vez, a conversa em whatsapp_conversas nao tem lead_id preenchido. Todo o painel lateral fica vazio.
Responsavel: workflow A0DEvhxj (inbound-upsert-lead)
Deve detectar CPF na primeira mensagem → upsert em leads → atualizar whatsapp_conversas.lead_id + contato_nome. Verificar se esta ativo e funcionando.
Fallback garantido
Botao "Criar lead" no painel lateral da conversa que cria lead com dados basicos (telefone, nome do contato) + vincula lead_id via RPC.
n8n A0DEvhxj + SELECT em whatsapp_conversas.lead_id
P0-03
Julia ainda ativa — indicadores aparecem indevidamente
Apesar de falhas no orchestrator, ai_enabled = TRUE no banco. UI exibe "Julia respondendo, assuma para escrever" — confunde o atendente.
Acoes necessarias
1. Banco: UPDATE whatsapp_numeros SET ai_enabled = false WHERE agente_slug = 'sdr_leads'
2. Frontend: suprimir texto "Julia respondendo..." quando ai_enabled = false
3. Frontend: suprimir toggle "Assumir conversa" quando ai_enabled = false
4. Manter nome "Julia" nos registros — apenas ocultar indicadores de atividade
SQL imediato + condicional no header da thread MdZap

P2 — Melhorias e Dividas Tecnicas

4 itens que nao bloqueiam mas precisam de atencao
P2-01
contato_nome vs lead_nome — duplicacao confusa
Dois campos com propositos sobrepostos: whatsapp_conversas.contato_nome (bruto WhatsApp) e leads.nome (oficial cadastrado).
Definicao clara de precedencia
UI deve sempre exibir: leads.nome ?? conversa.contato_nome. Documentar no codigo para evitar ambiguidade futura.
P2-02
Indicador "Julia processando" aparecendo indevidamente
Loading spinner de Julia pode aparecer mesmo quando ai_enabled = false ou mode = 'human'.
Condicao correta
ai_enabled === true AND conversa.metadata?.mode !== 'human' — ambas obrigatorias para exibir qualquer indicador de Julia.
P2-03
DLQ processor — tentativas sem filtro adequado
mdzap_message_queue pode reprocessar mensagens infinitamente se o cron nao filtrar por numero de tentativas.
Fix
Adicionar &tentativas=lt.3 na query do cron DLQ processor.
P2-04
Template Disparo CP — continueOnFail ausente
Workflow T22: se Meta API retornar 400/500, no meta-tpl-08 nao tem continueOnFail: true. Frontend fica sem resposta (timeout).
Fix
Adicionar continueOnFail: true no no meta-tpl-08 + resposta canonica de erro: success: false, action: "toast_error".
n8n workflow T22, no meta-tpl-08

Controle de IA — Shadow Mode e Manual

Flags que controlam comportamento da Julia por nivel
Nivel 1 — Numero (global)
whatsapp_numeros.ai_enabled TRUE → Julia responde automaticamente FALSE → modo manual para o numero inteiro
Afeta TODAS as conversas do numero
Alteravel via SQL direto no Supabase
Acao P0-03: setar FALSE para sdr_leads
Nivel 2 — Conversa (individual)
conversa.metadata.mode "auto" → Julia responde (padrao) "human" → Julia silenciada nesta conversa Shadow Mode: processa mas nao envia
Atendente assume conversa especifica
Julia gera resposta internamente (supervisao)
Util para treinamento e auditoria
Logica correta de exibicao na UI
// Condicao para mostrar qualquer indicador de Julia const juliaAtiva = numero.ai_enabled === true && conversa.metadata?.mode !== 'human'; // Badge "Julia respondendo" {juliaAtiva && <JuliaBadge />} // Spinner Julia processando {juliaAtiva && isProcessing && <JuliaTyping />} // Toggle "Assumir conversa" {juliaAtiva && <AssumeButton />}

Questoes em Aberto

03
Como Julia acessa dados de restricoes para informar o lead?

04
Renda Receita — como Julia sabe que foi concluida?
Analise de declaracao IR feita por humano. Julia precisa saber se ja foi feita.

05
Follow-up automatico para assinatura — como programar?

A*Trigger no Supabase ao atualizar coluna_crm = 'aguardando_assinatura' → chama n8n webhook que cria job com timer
