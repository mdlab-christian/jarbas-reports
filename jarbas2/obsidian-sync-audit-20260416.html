Diagnóstico rápido — veredito

O que funciona bem: Pro↔Air (jarbas-vault) e vaults específicos de cada worker user (MdOs-vault em cecilia/olivia/jarbas, beto-vault em beto). O que está quebrado: o jarbas-vault no Mac Mini está 6 dias atrasado — não tem sync rodando para ele; users do Mini não compartilham vaults por filesystem symlink (os symlinks apontam para diretórios de outro user, o que quebra por falta de permissão).

Evidências críticas

P0 · jarbas-vault NO MAC MINI está 6 dias fora de sync

00_cerebro/CONTEXTO.md:

Pro:     16 abr 02:34  6348 bytes
Air:     16 abr 02:34  6348 bytes   ← idêntico
Mini:    10 abr 02:40  2583 bytes   ← DEFASADO

Causa: /Users/jarbas/jarbas-vault/.obsidian/ está VAZIO (nenhuma config), e não existe processo obsidian-headless sync --path /Users/jarbas/jarbas-vault. O CLAUDE.md do Jarbas assume que esse vault sincroniza, mas no Mini ele é uma pasta morta — qualquer código que leia ~/self-improving ou ~/jarbas-vault/10_sync-macs/ no Mini está olhando estado antigo.

P0 · olivia-vault NO MAC MINI está fora do sync

User olivia tem só ai.olivia.chat-server, server-v3, watchdog, sync-skills. Nenhum LaunchAgent obsidian-headless para /Users/olivia/olivia-vault. Só o MdOs-vault sincroniza.

Resultado: qualquer coisa que a OlivIA v3 gera/lê no próprio vault dela fica trapped no Mini — não chega ao Pro. Inclui: playbooks, treinos, skills movidas para o vault dela.

P1 · cecilia-vault no Mini tem sync ambíguo

LaunchAgent aponta para /Users/cecilia/cecilia-vault, mas /tmp/obsidian-cecilia.err repete em loop:

No sync configuration found for /Users/cecilia/Documents/cecilia-vault
Run 'ob sync-setup' first.

Tem dois caminhos competindo (home vs Documents). Rodar ob sync-setup no path certo e matar o processo órfão.

P1 · Divergência de naming: MdOs-vault vs MdOs-Vault

Pro / Mini-jarbas / Mini-cecilia / Mini-olivia / Mini-beto usam MdOs-vault (v minúsculo). Air usa MdOs-Vault (V maiúsculo em ~/Documents/). macOS é case-insensitive no HFS+/APFS padrão, mas alguns caminhos hardcoded em scripts podem quebrar. O Obsidian Sync trata o vaultName pelo ID interno, então provavelmente ainda conversam — mas vale padronizar.

P1 · Symlinks residuais apontando p/ users alheios

No Air (~/cecilia-vault, ~/mdlab-vault, ~/olivia-vault) e no Mini-beto (~/jarbas-vault, ~/cecilia-vault, ~/mdlab-vault, ~/olivia-vault) existem symlinks para /Users/<outro_user>/.... No Air não existe esse outro user — symlink morto. No Mini-beto os paths existem mas beto não tem permissão (drwxr-xr-x do user dono). Código que faça cat ~/mdlab-vault/... no beto vai falhar silenciosamente.

OK · backbone principal funciona

Pro ↔ Air (jarbas-vault) está 100% em sync — conferido via bytes idênticos em CONTEXTO.md. A arquitetura "Obsidian Sync como bus de mensagens entre Pro e workers" está certa para esse vault. LaunchAgents do Air rodam com KeepAlive=true e RunAtLoad=true (reiniciam automático).

Plano de correção (ordem sugerida)

Correção 1 — restaurar jarbas-vault no Mini (P0)

ssh mac-mini-tail bash -lc '
  # 1. Backup do estado atual
  mv ~/jarbas-vault ~/jarbas-vault.broken-20260416

  # 2. Criar vault novo via obsidian-headless
  obsidian-headless create --name jarbas-vault --path ~/jarbas-vault
  obsidian-headless sync-setup --path ~/jarbas-vault
  # (login interativo — preencher credenciais Obsidian Sync)

  # 3. Criar LaunchAgent
  cat > ~/Library/LaunchAgents/md.obsidian.sync.jarbas-vault.plist <<EOF
<plist version="1.0"><dict>
 <key>Label</key><string>md.obsidian.sync.jarbas-vault</string>
 <key>ProgramArguments</key><array>
  <string>/opt/homebrew/bin/node</string>
  <string>/opt/homebrew/lib/node_modules/obsidian-headless/cli.js</string>
  <string>sync</string><string>--path</string><string>/Users/jarbas/jarbas-vault</string><string>--continuous</string>
 </array>
 <key>RunAtLoad</key><true/><key>KeepAlive</key><true/>
</dict></plist>
EOF
  launchctl load ~/Library/LaunchAgents/md.obsidian.sync.jarbas-vault.plist
'

Correção 2 — olivia-vault no Mini user olivia (P0)

Mesmo procedimento que acima, mas rodando via sudo -u olivia ou SSH direto no user olivia (se tiver). Criar LaunchAgent em /Users/olivia/Library/LaunchAgents/ai.olivia.obsidian-sync.plist.

Correção 3 — resolver conflito cecilia-vault (P1)

# Descobrir qual path é o canônico
sudo -u cecilia ls -la /Users/cecilia/cecilia-vault /Users/cecilia/Documents/cecilia-vault 2>/dev/null

# Se o sync real quer Documents, criar symlink e matar orphan:

No Air, renomear ~/Documents/MdOs-Vault → ~/Documents/MdOs-vault e atualizar o LaunchAgent. Procurar hardcodes:
