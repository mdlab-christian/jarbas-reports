Diagnóstico: O Problema Real

🎯

A arquitetura está certa. Os dados não chegam.

O problema não é a forma como o gerador foi desenhado — é que 4 bugs de fluxo de dados quebram a cadeia de contexto antes mesmo do Claude ver qualquer coisa. Você tem um motor Ferrari com o cano de combustível furado. Corrigir os gaps é muito mais rápido e eficaz do que reconstruir tudo no N8N.

P0 CRÍTICO
OCR dos documentos sumiu
A EF olivia-gerar busca o historico_id com analise_provas do banco, mas não inclui esse texto no payload enviado ao Mac Mini. O pipeline gera a petição sem nunca ver os documentos do cliente.
→ Fix: EF olivia-gerar passa texto_ocr + pipeline S1 injeta em buildUserPromptV3[ANÁLISE DE PROVAS]
P0 CRÍTICO
Instruções adicionais descartadas
O advogado digita instrucoes_adicionais no Step 1 (ex: "enfatizar que o banco nunca enviou notificação"). Esse campo é salvo no banco mas nunca passa para o pipeline de geração. Claude nunca lê essa instrução.
→ Fix: Incluir instrucoes_adicionais no payload de olivia-gerar + injetar no system prompt
P1 ALTO
Respostas das perguntas ignoradas (V3 vs V4)
O advogado responde perguntas no Step 2 (campo perguntas_topico.valor). O pipeline S1.5 ainda lê o campo antigo perguntas (formato V3). As respostas do advogado são completamente descartadas na geração.
→ Fix: S1.5 SELECT perguntas_topico em vez de perguntas, lê .valor corretamente
P1 ALTO
Revisor de qualidade desativado
O Stage S6 (revisor o4-mini) existe no código mas nunca é chamado no run(). O score de qualidade é sempre null. Não há loop de melhoria automática — petições ruins saem sem revisão.
→ Fix: Reativar S6 no run(), loop máx 3 tentativas, threshold 8.0
P2 MÉDIO
Mac Mini: ponto único de falha
Todo o pipeline pesado (9 stages, geração Claude) roda exclusivamente no Mac Mini na porta 19100. Se o servidor cair, o gerador para completamente. Não há fallback. O advogado fica preso no Step 3 com loading infinito.
→ Fix: EF com Sonnet direto como fallback se timeout 10s no Mac Mini
P2 MÉDIO
Embeddings V2 não populados
A tabela olivia_blocos_texto tem 1.244 blocos, mas embedding_v2 (768-dim, mais preciso) está 0% preenchido. O RAG está rodando com o embedding antigo (1536-dim OpenAI). Busca semântica potencialmente menos precisa.
→ Fix: Job de repopulação async dos embeddings V2

O Que Você Tem de Bom (e é muito)

📚 1.244 blocos de argumentação jurídica
Organizados por tag (tag_1 a tag_6) e tema — preliminares, provas unilaterais, invalidade de notificações, dano moral, requisitos, dinâmica probatória. 26.612 linhas de conteúdo estruturado. RAG com pgvector (1536-dim OpenAI). Cobertura por segmento de réu: bureau, bancário, FIDC, telecom, energia, varejo, cessão.
📄 169 modelos de petição
Distribuídos em 10 tipos (réplica, apelação, contrarrazões, cumprimento de sentença, agravo, alegações finais, manifestação, emenda, petição simples, alvará). 9 categorias de réu. Todos com conteudo_template + instrucoes_ia específicas. Versionados por grupo.
⚖️ 1.392 jurisprudências catalogadas
TJRS, TJSC, TJSP. Sincronizadas via trigger automático de decisões aprovadas. Busca semântica por tribunal + tipo. Injetadas no user_prompt como "inteligência jurídica" combinada com taxa de sucesso do julgador específico e perfil da empresa ré.
🎯 Inteligência do julgador
Por processo: taxa_favoravel do julgador específico, taxa_sucesso_contra_empresa (ESSA empresa nesse juízo), perfil_decisorio (favorece consumidor, aplica CDC, reconhece dano moral). Também: valor_medio_condenacao e valor_medio_acordo por empresa ré. Tudo vai no prompt.
🔄 Self-improving (S9)
Quando feedback_advogado = 'aprovei' e score >= 8.0: extrai patterns da petição aprovada, atualiza olivia_playbooks com probabilidades ajustadas, salva exemplos em olivia_blocos_texto. O sistema literalmente aprende com cada petição aprovada pela equipe.
✏️ VibeLaw (5 comandos de edição)
Editor TipTap no Step 3 com slash menu: /reescrever, /expandir, /jurisprudencia, /simplificar, /formalizar. Cada comando envia trecho para EF com RAG injetado. Autosave a cada 30s. Chat conversacional lateral sobre a análise.

Valeria Voltar para o N8N Mecânico?

❌ Não recomendado
O modo Smart Blocks N8N funcionava porque era mecânico e simples — não analisava nada, só preenchia template com blocos selecionados. Era previsível mas superficial. Não lia documentos, não entendia contexto, não aplicava inteligência jurídica.

O problema atual NÃO é que o gerador de IA é complexo demais — é que os dados não chegam. Regredir para N8N mecânico seria trocar profundidade por falsa estabilidade. Você resolveria os "travamentos" mas geraria petições de qualidade inferior.
✅ Caminho certo
Corrigir os 4 bugs de payload é trabalho de 1 sessão. São mudanças pontuais em 2 arquivos:

1. supabase/functions/olivia-gerar/index.ts — adicionar 2 campos no payload
2. olivia-pipeline-v3.mjs linha ~S1 — injetar ctx.texto_ocr
3. olivia-pipeline-v3.mjs linha ~S1.5 — corrigir campo V4

Depois dessas correções, o mesmo pipeline que "não funciona direito" vai gerar petições com contexto completo — análise de provas, instruções do advogado, respostas das perguntas. Tudo chegando ao Claude.
