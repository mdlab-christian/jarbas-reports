🎯 1. Contexto — Por Que Isso Existe
O chat v2 não substitui o gerador. É o plano B quando o gerador trava — e gradualmente pode se tornar o plano A.
✅
Visão correta: Pelo Telegram, Christian já consegue mandar direto para a OlivIA e ela gera petições, faz análises, tudo funciona. O problema é que esse poder não está disponível para os usuários do MdFlow dentro do frontend. O Chat v2 é exatamente isso: uma interface de chat que dá ao usuário do MdFlow o mesmo poder que o Christian tem pelo Telegram — com o contexto do processo carregado.

💬 2. Fluxo Conversacional — Como o Usuário Vai Usar
Do início ao download da petição — tudo via chat. Inspirado no que já funciona pelo Telegram.
🗺️

Fluxo Principal — Gerar Petição via Chat

1

Criação do Chat — Processo Obrigatório (default) ou Livre

Botão "Nova Conversa" abre modal com duas opções: Com Processo (padrão, botão em destaque) e Chat Livre (opção secundária, menor). Para "Com Processo", usuário busca pelo CNJ usando o SearchBoxCNJ que já existe. OlivIA carrega dados do processo do Supabase e abre o chat já contextualizado.

Reutiliza SearchBoxCNJ existente Chat livre = opção secundária
── OlivIA carregou: Processo 0012345-67.2024.8.21.0001 ── Banco Bradesco vs. João Silva · Negativa de Crédito · TJRS OlivIA: Processo carregado ✓ O que você quer fazer? 📊 Analisar provas ⚖️ Gerar petição 📁 Upload de documentos
2

Upload de Documentos (quando necessário)

Botão 📎 no composer, ou arrastar arquivo. Upload vai direto para Supabase bucket olivia-docs (mesmo bucket que o gerador usa, mesmo código do UploadDrawer.tsx). OlivIA roda classifica-docs e OCR automaticamente e confirma o que recebeu.

Bucket olivia-docs — já existe Reutiliza lógica do UploadDrawer Max 10MB por arquivo
Usuário: [📎 Contrato_Bradesco.pdf] OlivIA: Documento recebido ✓ 📄 Contrato_Bradesco.pdf (2.3 MB) Classificação: Contrato de empréstimo OCR: texto extraído (4 páginas) Pronto para usar na geração.
3

Intent Detection — OlivIA Detecta o Que o Usuário Quer

Usuário escreve em linguagem natural. O agente v4 classifica a intenção antes de responder: GERA_PETICAO, ANALISE_PROVAS, BUSCA_JURIS, RESUMO, LIVRE. Se a intenção for geração de petição e falta informação (tipo, modelo), OlivIA pergunta via action cards — não via form.

Novo: intent router no agente v4 Action cards para inputs estruturados
Usuário: gera a réplica OlivIA: Certo. Qual tipo de petição? 📝 Réplica padrão 📝 Réplica com análise de provas 📝 Impugnação à contestação
4

Triagem Conversacional — OlivIA Pergunta O Que Precisa

Em vez de form com 10 campos, OlivIA faz as perguntas de triagem relevantes uma a uma via chat. Usa as mesmas perguntas do gerador (PerguntaDinamica, PerguntaSimNao), mas em formato conversacional. Usuário responde em texto ou clicando em action cards.

= Steps 2-3 do gerador, conversacional Mesmo JSON de triagem do backend
OlivIA: O banco apresentou extrato de operações? ✓ Sim✗ Não Usuário: não apresentou OlivIA: Houve impugnação à autenticidade do contrato? ✓ Sim✗ Não
5

Geração com Progresso em Tempo Real

OlivIA inicia a geração e insere mensagens de progresso no Supabase (role: 'system') que chegam via Realtime já existente. Frontend renderiza como progress bubbles diferentes dos bubbles de chat. Score automático + revisor se <7. Usuário acompanha sem travar a UI.

= Steps 4-5 do gerador Realtime já existe — só precisa do frontend Anti-fingerprint seed CNJ mantido
OlivIA: Iniciando geração da réplica... ▸ Carregando contexto do processo ▸ Buscando teses jurídicas via RAG [ChromaDB] ▸ Gerando seção: Dos Fatos (1/6)... ▸ Gerando seção: Do Direito (2/6)... ▸ Score: 8.4/10 — aprovado ✓
6

Download Inline — Action Card com Arquivos

Ao concluir, OlivIA posta uma mensagem com action card de download. Links diretos do Supabase Storage (URLs públicas temporárias). Usuário clica e baixa. Também pode pedir "manda o relatório" para baixar o relatório de análise.

= Step 6 do gerador Supabase Storage signed URLs
OlivIA: Réplica gerada com sucesso ✓ (score 8.4) ⬇ Baixar DOCX ⬇ Baixar PDF 📊 Ver relatório ✏️ Solicitar ajuste

Schema Supabase — Mudanças Mínimas

-- 1. olivia_threads: adicionar chat_state (estado conversacional)
ALTER TABLE olivia_threads
  ADD COLUMN IF NOT EXISTS chat_state jsonb DEFAULT '{}';
-- { processo_loaded: bool, triagem_done: bool, docs: [{id, nome, classificacao}], pending_action: string }

-- 2. olivia_messages: action_cards já existe — confirmar tipo
-- action_cards: [{ type: 'choice'|'download'|'progress', id, label, value, url, style }]

-- 3. olivia_chat_docs: docs enviados pelo chat (separado de processos_documentos)
CREATE TABLE IF NOT EXISTS olivia_chat_docs (
  id              uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  thread_id       uuid REFERENCES olivia_threads(id) ON DELETE CASCADE,
  organizacao_id  uuid NOT NULL,
  processo_id     uuid REFERENCES processos(id),
  filename        text NOT NULL,
  storage_path    text NOT NULL,     -- bucket olivia-docs
  mime_type       text,
  tamanho_bytes  bigint,
  classificacao   text,             -- prova, contrato, laudo, outros
  ocr_text        text,
  created_at      timestamptz DEFAULT now()
);
ALTER TABLE olivia_chat_docs ENABLE ROW LEVEL SECURITY;
💡 Nota: Os docs enviados pelo chat ficam em olivia_chat_docs (separado de processos_documentos). Assim não polui o histórico de documentos do processo, mas ainda usa o mesmo bucket olivia-docs. Se o usuário quiser mover um doc para o processo, OlivIA pode fazer isso via action card.

📋 4. Plano de Implementação — 4 Fases
Cada fase entrega valor sozinha. Nenhuma quebra o gerador atual.
FASE 1 Frontend Redesign — MVP ~2-3h CC · Frontend only
Entrega: Chat com processo obrigatório (padrão), ProcessoContextHeader, Markdown rendering, composer melhorado. Zero mudança no backend.
  • Reconstruir OliviaChatPage.tsx — modal de novo chat com "Com Processo" como default visual (botão maior, destaque). "Chat Livre" como opção secundária menor.
  • ProcessoContextHeader — strip no topo do chat mostrando: número CNJ, nome do cliente, tipo de réu (empresa/órgão). Clicável para abrir drawer do processo.
  • Markdown rendering — trocar <p className="whitespace-pre-wrap"> por ReactMarkdown (já importado no ChatDrawerOlivia, só trazer aqui).
  • Quick suggestions dinâmicas — se tem processo vinculado: "Analisar provas", "Gerar réplica", "Upload de documentos". Se chat livre: "Resumir jurisprudência", "Fazer pergunta".
  • Composer melhorado — auto-resize textarea, Shift+Enter para nova linha, Enter para enviar, contador de chars, placeholder contextual.
FASE 2 Upload de Documentos Inline ~2-3h CC · Frontend + Backend mínimo
Entrega: Usuário pode enviar PDFs e documentos diretamente no chat. OlivIA confirma recebimento e classifica.
  • ComposerWithUpload — botão 📎 no composer, drag&drop, preview do arquivo antes de enviar. Reutilizar lógica de upload do UploadDrawer (bucket olivia-docs, mesmo path pattern).
  • Criar tabela olivia_chat_docs — migration simples, 8 campos, RLS básico.
  • Pill de documento na conversa — quando usuário manda arquivo, aparece como message especial (não como texto), mostrando nome do arquivo, tamanho, ícone de tipo.
  • Backend OlivIA: processar upload — quando recebe docs no payload, lê os docs do Supabase Storage, roda classifica-docs + OCR, atualiza olivia_chat_docs, confirma via mensagem no chat.
  • Passar docs no payload do webhook — ao enviar mensagem, incluir docs com os IDs dos docs da thread atual no payload olivia/chat.
FASE 3 Intent Router + Action Cards + Geração ~4-5h CC · Backend OlivIA + Frontend
Entrega: OlivIA detecta "gera a réplica" e conduz todo o fluxo via chat. Triagem conversacional, geração com progresso, resultado estruturado.
  • Intent Router no agente v4 — antes de gerar texto livre, classificar: GERA_PETICAO, ANALISE_PROVAS, BUSCA_JURIS, RESUMO, TRIAGEM_RESP, ACTION_RESP, LIVRE. Cada intent → skill específica.
  • Triagem conversacional — GERA_PETICAO sem tipo/modelo → OlivIA responde com action card de tipo de petição. Após escolha, roda skill_olivia_gerar_triagem, serve as perguntas uma a uma como mensagens.
  • Progresso via Realtime — durante geração, inserir rows em olivia_messages com role: 'system' e content de progresso. Frontend renderiza como ProgressBubble (visual diferente do chat bubble).
  • ActionCardRenderer no frontend — detectar action_cards[] nas mensagens. Renderizar como botões abaixo do bubble. Clique → envia action_response no próximo payload.
  • Chat State persistido — OlivIA atualiza olivia_threads.chat_state a cada step: triagem completada, modelo selecionado, geração iniciada. Assim retomada funciona mesmo se o usuário fechar o browser.
FASE 4 Download Inline + Polish Final ~2h CC · Frontend + UX
Entrega: Petição gerada → action card de download direto no chat. Áudio, polish de animações, histórico de docs visível na sidebar.
  • DownloadCard — action card especial ao final da geração: [⬇ DOCX] [⬇ PDF] [📊 Relatório] [✏️ Pedir ajuste]. URLs do Supabase Storage (signed URL ou public, dependendo do bucket config).
  • Gravação de áudio — botão 🎤 no composer. Reutilizar hook useAudioRecorder que já existe no projeto. Transcreve com Whisper antes de enviar.
  • Docs sidebar — abinha lateral opcional mostrando docs enviados na thread atual com classificação. Clique para remover ou reenviar para análise.
  • Animações de loading — durante geração, mostrar indicador visual diferente do "OlivIA está pensando..." atual. Algo que comunique que é um processo longo (barra de progresso, pulsing com texto "Gerando seção 3/6...").
🧠 5. Minhas Considerações — O Que Pode Dar Errado
Pontos que precisam de atenção antes ou durante a implementação.
⚠️

Riscos Técnicos

R1

Timeout do webhook — o maior risco

Geração de petição leva 3-8 minutos. O frontend tem timeout de 60s no callWebhook. A solução é resposta imediata 202 ("iniciando...") e tudo via Realtime. O backend OlivIA precisa responder imediatamente ao webhook e depois trabalhar async. Verificar se o agente v4 já faz isso ou se bloqueia até concluir.

R2

Estado de triagem entre mensagens

Quando OlivIA faz uma pergunta de triagem e aguarda resposta, o estado precisa ser mantido. Se o usuário mandar outra coisa ("na verdade cancela") o intent router tem que entender o contexto. Usar chat_state.pending_action para isso — OlivIA sabe que estava esperando uma resposta específica.

R3

Context flooding no agente v4

Threads longas com muitas mensagens vão estourar o contexto do Claude. Precisar de limite de histórico injetado — últimas 20 mensagens ou sumarização automática. Implementar na Fase 3 junto com o intent router.

💡

Oportunidades Não Óbvias

O1

Chat como feedback loop do gerador

Quando a petição gerada pelo modo gerador está ruim, o usuário hoje tem que recomeçar do zero. Com o chat, ele pode pedir "ajusta a seção Do Direito" ou "adiciona a tese X" diretamente. O chat v2 pode ser o modo de refinamento do gerador, não só o fallback.

O2

Compartilhar thread entre usuários

Uma thread de chat vinculada a um processo pode ser compartilhada entre usuários do mesmo escritório. O advogado começa a triagem, o assistente sobe os docs, a OlivIA gera. Fluxo colaborativo sem custo adicional.

O3

Processar múltiplas petições via chat

Depois que o processo está carregado, o usuário pode pedir "gera a réplica" → download → "agora gera as alegações finais" → download. Sequencial, sem sair do chat. O estado da triagem é reaproveitado quando pertinente.
