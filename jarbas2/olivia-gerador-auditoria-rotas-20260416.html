Correção: Consolidar tudo em agente/ (canônico), remover 01-identidade/, recriar como symlink no Pro + garantir que o Obsidian Sync respeite via .obsidianignore.

2. Disparo (step Gerar)

// Frontend faz INSERT em peticoes_oliviav3:
const { data } = await supabase
  .from('peticoes_oliviav3')
  .insert({
    status: 'aguardando_analise',
    modo: 'peticao',
    contexto: { processo_id, cliente_id, causa_pedir, provas, ... },
    organizacao_id: orgId,
  })
  .select('id')
  .single();

// Subscreve Realtime para receber updates:
supabase.channel(`peticao-${data.id}`)
  .on('postgres_changes', { event: 'UPDATE', ... }, onUpdate)

3. Poller — Mac Mini detecta

/Users/olivia/olivia-server/poller.mjs (LaunchAgent, ciclo 5s)
  └─ SELECT id FROM peticoes_oliviav3 WHERE status='aguardando_analise' LIMIT 1;
  └─ Se encontrou: POST http://localhost:19100/run { peticao_id, modo: 'peticao' }

4. Servidor gateway (porta 19100)

server.mjs
  switch(modo) {
    case 'peticao':    import('./pipeline/core.mjs').then(m => m.run(id))
    case 'vibelaw':    import('/Users/Shared/olivia-skills/olivia-pipeline-v3.mjs')  ← LEGADO
    case 'relatorio':  import('/Users/Shared/olivia-skills/olivia-pipeline-v3.mjs')  ← LEGADO
    case 'treinamento': import('./pipeline/treinamento.mjs')
  }

5. Pipeline v4 — modo petição

pipeline/core.mjs (orquestrador)
  ├─ 5a. playbook-loader.mjs    → carrega playbook correto do vault
  │       (réplica-bancario, réplica-concessionária, replica-generica, etc)
  ├─ 5b. analise.mjs            → Claude Opus analisa contexto + provas + juris
  │       UPDATE peticao SET status='analisando', analise={...}
  ├─ 5c. safety-guard.mjs       → validação anti-erros (ASSINATURA_RE, temRodape, cidade)
  ├─ 5d. geracao.mjs            → GPT-4o gera texto completo + HTML renderizado
  │       UPDATE peticao SET status='gerando', texto_html='...'
  ├─ 5e. criar-doc.mjs          → cria docx/pdf, detecta juris (W11 D01 árbitro)
  │       UPDATE peticao SET status='finalizado', arquivo_url='...'
  └─ 5f. aprendizado.mjs        → registra uso de playbook/juris para W9 training

6. Frontend recebe via Realtime

OliviaGeradorV3 escuta UPDATE em peticoes_oliviav3:
  - status='analisando'   → mostra step "Analisando..."
  - status='gerando'      → mostra step "Gerando texto..."
  - status='finalizado'   → mostra arquivo + botão de download
  - status='erro'         → mostra mensagem + retry

Decisão pendente do CTO bloqueia W12

◆ D-W12-01 — Qual nome canônico manter: agente vs 01-identidade?

A pipeline lê agente/ hoje (via playbook-loader.mjs e imports correlatos). Mas o vault foi originalmente organizado com 01-identidade/ (numeração). O symlink no Mini hoje é 01-identidade → agente, indicando agente como real.

Opção A — Manter "agente/" como real (status quo do Mini)

Vantagens: zero mudança no código, já é como o Mini está hoje. Desvantagem: quebra convenção de numeração do vault.
Ação: no Pro, deletar 01-identidade/ (real) e recriar como symlink → agente. .obsidianignore ignora 01-identidade/.

Opção B — Migrar tudo para "01-identidade/" como real

Vantagens: mantém convenção 01/02/03/04. Desvantagem: precisa refatorar pipeline (todos os paths que hoje apontam agente/).
Estimativa: +3 arquivos pipeline editados, +1 commit extra, +30min de trabalho.

Opção C — Manter ambos como reais e sincronizar via script

Vantagens: flexibilidade. Desvantagem: complexidade operacional alta, risco de divergência contínua.
Não recomendado.

Recomendação: Opção A. Menor blast radius, preserva código atual, resolve o problema de sync.

◆ D-W12-02 — Mesma decisão para playbooks vs 04-playbooks

Pipeline lê de 04-playbooks/ hoje. Mini tem symlink playbooks → 04-playbooks.

Opção A — Manter "04-playbooks/" como real

Simétrico à Opção A de D-W12-01. Recomendado.
