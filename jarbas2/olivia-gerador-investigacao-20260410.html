Pull-Based Architecture

Zero webhooks diretos. O Mac Mini faz polling no banco a cada 5 segundos, buscando registros em olivia_historico com status='PROCESSANDO'. Tres pollers independentes (analise, geracao, criar_doc) com cutoff de 30 minutos para evitar loops. O banco funciona como Single Source of Truth — se o server cair, basta reiniciar e ele retoma de onde parou.

Beneficios do Design
  • ✓ Resiliente a falhas de rede (sem websocket obrigatorio)
  • ✓ Estado persistido (banco = SSOT, nao memory)
  • ✓ Retry natural (registro fica PROCESSANDO ate ser concluido)
  • ✓ Desacoplamento total (frontend, EFs e server independentes)
  • ⚠ Latencia max 5s entre trigger e deteccao (aceitavel)
  • ⚠ Ideal futuro: webhook push + poller como fallback

Step-by-Step do Gerador

4 Steps
1

Identificacao (UI Pura)

User configura processo, tipo de peticao, modelo, versao, advogado, documentos e instrucoes especiais. Nenhuma Edge Function e chamada neste step.

  • Componente: Step1IdentificacaoExpress.tsx
  • Pre-fetch via RPCs: fn_olivia_get_model_skeleton + fn_olivia_get_tags_com_blocos
  • Validacao: processo + tipoPeticaoId + modeloId obrigatorios
  • Acao: Botao "Iniciar Analise" navega para Step 2 via goToNextStep()
  • Zero Edge Functions chamadas neste step — tudo local no frontend
goToNextStep()
2

Topicos + Analise (5-15 min)

Analise profunda do caso pelo motor IA. Mac Mini processa em background, frontend atualiza via Realtime.

  • Mount: Se historicoId=null, chama EF olivia-topicos
  • EF olivia-topicos: INSERT olivia_historico (status=PROCESSANDO, pct=1) → retorna historico_id
  • Mac Mini detecta (poller 5s) → runAnalise():
    • 1. Carrega vault: SOUL.md + STYLE.md + ERRORS.md + analise.md + GLOSSARY.md
    • 2. fetchContextoAnalise: processo, modelo, documentos, movimentacoes (paralelo)
    • 3. fetchTagsComBlocos: tags + blocos por categoria
    • 4. loadPlaybooks: playbooks on-demand por tipo_peticao + categoria_empresa
    • 5. callClaude (Sonnet/Opus): analisa provas, resolve tags, gera topicos
    • 6. Persiste: analise_estruturada, topicos_gerados, tag_answers
  • Frontend via Realtime: mostra progresso, topicos, analise
  • Botao "Aprovar e Redigir": chama EF olivia-gerar → monta peticao_referencia (modelo + secoes + blocos + juris) → define etapa_atual='geracao', pct=1
EF olivia-gerar → etapa_atual='geracao'
3

Editor VibeLaw (10-30 min)

Redacao da peticao topico a topico. Opus 4.6 SEMPRE, independente do tier. Editor TipTap com slash commands.

  • Mac Mini detecta etapa_atual='geracao' → handleGeracao()
  • Opus 4.6 SEMPRE (independente do tier) — gera topico a topico com blocos + juris
  • Persiste: peticao_html + texto_gerado
  • Frontend: TipTap carrega peticao_html para edicao
  • VibeLaw: 5 slash commands interativos:
    • /reescrever — Reescreve trecho selecionado
    • /expandir — Expande argumentacao
    • /simplificar — Simplifica linguagem
    • /formalizar — Torna mais formal/juridico
    • /jurisprudencia — Adiciona juris relevante
  • VibeLaw EF: olivia-vibelaw (SSE streaming, Sonnet 4.6, timeout 45s)
  • Autosave: Debounce 15s → UPDATE peticao_html
  • Botao "Concluir": forceSave() → navega para Step 4
forceSave() → Step 4
4

Conclusao

Geracao de documentos finais (DOCX/PDF), relatorio de analise, score e feedback.

  • EF olivia-criar-doc: Proxy Mac Mini → DOCX (Calibri/Georgia por advogado) + PDF (Playwright)
  • EF olivia-relatorio: Gera HTML report com 10 secoes (Sonnet 4.6)
  • Downloads: DOCX/PDF via URLs assinadas (30min expiry)
  • Score + Feedback + Atividade: Registrado no banco
  • P0-A Relatorio retorna 500/502 — bloqueado
  • P0-D PDF Playwright crash (SIGTRAP) — bloqueado

Consideracoes do CTO (Jarbas)

10 avaliacoes
✔
1. Arquitetura pull-based
EXCELENTE — Elimina problemas de rede, webhooks flakey, timeouts de cloud functions. Banco como SSOT garante que nada se perde. Padrao industrial solido para workloads async pesados.
✔
2. Opus 4.6 SEMPRE na geracao
CORRETO — Qualidade justifica o custo ($0.97/peticao e barato para documento juridico). Nao economizar no motor que produz o produto final. Sonnet para analise, Opus para redacao — divisao perfeita.
⚠
3. Vault como cerebro
BOM mas MEMORY.md vazio e um gap — O vault esta muito bem estruturado (SOUL.md, prompts, playbooks). Porem, sem feedback loop automatico, a OlivIA nao aprende com cada peticao. P1-C precisa ser prioridade.
✔
4. 15 playbooks por segmento
SOLIDO — Cobertura de 5 segmentos (financeiro, telefonia, energia, varejo, marketplace) + analise de provas completa com tripe (relacao juridica + contratacao + origem debito). Playbook-raiz com fallback inteligente.
⚠
5. Rate limiter in-memory
ACEITAVEL para escala atual — 5 concurrent / 50 por hora e suficiente para ~5.800 processos. Se reiniciar server, reset do counter. Para multi-tenant real, migrar para Redis ou PostgreSQL advisory locks.
✖
6. PDF via Playwright
FRAGIL — Playwright headless e pesado para server (chromium inteiro). SIGTRAP mostra instabilidade. WeasyPrint seria mais leve, deterministico e sem dependencia de browser. Recomendacao forte de migracao.
✔
7. olivia_historico como state machine
BEM DESENHADO — Status claro (PROCESSANDO, CONCLUIDO, ERRO), etapa_atual granular (analise, geracao, criar_doc), progresso_pct + progresso_msg para UI. Cobre todos os cenarios de lifecycle.
✔
8. Tags + Blocos
SISTEMA INTELIGENTE — 183 tags auto-resolviveis + 1258 blocos reutilizaveis permite que a IA tenha contexto rico sem precisar de prompts enormes. Auto-resolucao com confianca e diferencial competitivo.
⚠
9. Frontend multi-tab
DESNECESSARIO para agora — Complexidade prematura. Mas nao atrapalha o fluxo. Quando escalar para equipe de 5+ advogados, vai ser util. Por ora, nao investir mais tempo nisso.
⚠
10. Poller 5s
FUNCIONAL — Ideal seria webhook push + poller como fallback (best of both worlds). Mas 5s de latencia e aceitavel para fluxo de peticoes que leva 8-42 minutos. Nao e prioridade otimizar.
