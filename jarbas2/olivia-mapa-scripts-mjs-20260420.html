Fluxo End-to-End — 4 Steps

Do clique do advogado ate os downloads
Step 1 Selecao e Configuracao
Frontend
  • Advogado seleciona processo, tipo de peticao, modelo de IA, advogado responsavel
  • Faz upload de documentos (OCR via olivia-extrair-texto)
  • Clica "Avancar" — dispara olivia-topicos EF
  • Exibe spinner aguardando Step 2 via Realtime
Backend (Mac Mini)
  • olivia-topicos EF insere row, retorna historico_id
  • server.mjs polling 5s detecta novo row (etapa='topicos')
  • Inicia analise.mjs — aproximadamente 2 minutos
  • UPDATE historico: topicos_gerados, analise_estruturada, status=CONCLUIDO
Step 2 Revisao de Topicos e Aprovacao
Frontend
  • Realtime dispara — exibe topicos_gerados e analise compacta
  • Advogado revisa topicos, responde perguntas interativas
  • Seleciona fidelidade (alta / equilibrada / livre)
  • Clica "Aprovar e Redigir" — dispara olivia-gerar EF
Backend (Mac Mini)
  • olivia-gerar EF monta peticao_referencia, UPDATE etapa='geracao'
  • server.mjs detecta, inicia geracao.mjs
  • Stages 1-9: contexto → RAG → juris → placeholders → style → Claude pMap
  • Stage 7 Boris — INATIVO (ENABLE_BORIS=false)
  • UPDATE historico: peticao_html, score, tokens, custo
Tempo: ~13 min (IA_ASSISTIDA / Replica, p95: 26min)
Step 3 Edicao com TipTap + VibeLaw
Frontend
  • Realtime recebe peticao_html — exibe no TipTap editor
  • Advogado edita livremente (opcional)
  • VibeLaw sidebar disponivel para edicao conversacional
  • Clica "Proximo" — transiciona para Step 4
Backend (Mac Mini)
  • criar-doc.mjs inicia em background automaticamente
  • DOCX via docx@9.6.1 — funcionando
  • PDF via LibreOffice headless — BUG: falha silenciosamente
  • Google Doc via Drive API — funcionando
  • INSERT olivia_documentos_gerados, UPDATE urls no historico
DOCX e GDoc: ~30s | PDF: 0 gerados (LibreOffice quebrado)
Step 4 Download e Feedback
Frontend
  • Realtime recebe docx_url e google_doc_url
  • Botoes DOCX / PDF (desabilitado) / GDoc habilitados
  • Advogado baixa e fornece feedback (nota 1-5 + comentario)
Backend (Mac Mini)
  • Rota /olivia/feedback-step4 recebe nota + comentario
  • INSERT olivia_feedbacks — Camada 3 do treinamento
  • Sem retreino automatico por enquanto

analise.mjs — taxa de erro ~55% (ragMeta / TAG undefined)

Correcao em andamento hoje. Verificar declaracao de escopo de ragMeta e TAG — mover para escopo correto dentro do pipeline de analise. Adicionar tests minimos de tipo antes de usar.

Verificar se soffice esta instalado e acessivel no PATH para user olivia. Alternativa mais confiavel para ambiente daemon: substituir LibreOffice por Puppeteer headless Chrome para geracao de PDF.

P1
Daemon v4.0 diverge do repo Pro (MD5 diferente)
server.mjs rodando no Mac Mini (PID 59171) tem MD5 diferente do arquivo no repo do MacBook Pro. Deploy e manual via rsync sem pipeline automatizado. A rota /olivia/feedback-step4 existe no Pro mas nao no Mini. A cada atualizacao no Pro, o Mini fica para tras ate um sync manual ser feito.
Criar script de deploy automatizado: Pro commit → n8n webhook → SSH Mini → git pull + launchctl kickstart daemon. Ou hook de pre-commit que verifica se rsync foi feito.
#4
P1
aprendizado.mjs sem time budget — ciclo pode durar 11 horas
O modo 'aprendizado' nao tem limite de tempo configurado. Media de 3.3 horas, maximo documentado de 11 horas. Durante todo este periodo o slot de pipeline fica ocupado no rate limiter (5 pipelines/org). Um unico ciclo de aprendizado pode bloquear toda a capacidade de geracao de uma org por horas.
Adicionar APRENDIZADO_TIMEOUT_MS = 1_800_000 (30 min) com AbortController no fetch do Claude. Analise retrospectiva deveria ser job assincrono com fila dedicada e prioridade baixa, fora do rate limiter principal.
#5
P1
Boris inativo — review automatico de qualidade nao acontece
ENABLE_BORIS=false e o padrao desde o deploy. Stage 7 de geracao.mjs e completamente pulado. O score de qualidade exibido ao advogado e heuristico, nao real. Peticoes com problemas juridicos nao sao sinalizadas. Boris foi especificado para usar Sonnet o4-mini como revisor independente e persistir em olivia_boris_outputs.
Ativar ENABLE_BORIS=true em modo shadow: Boris roda mas nao bloqueia geracao nem altera o resultado. Score Boris salvo em olivia_boris_outputs para comparacao com heuristico. Rollout progressivo: 10% → 50% → 100%.
#6
P2
DNS flaky — ENOTFOUND intermitente nos pollers
Erro ENOTFOUND qdivfairxhdihaqqypgb.supabase.co aparece de forma intermitente no loop de polling de server.mjs. Quando ocorre, o polling falha silenciosamente — rows ficam presos em etapa='topicos' ou 'geracao' indefinidamente sem aviso ao advogado. Sem retry com backoff implementado.
Adicionar retry com exponential backoff (3 tentativas: 5s → 15s → 45s) no loop de polling. Alert via Telegram se polling falhar por mais de 2 minutos consecutivos. DNS resolve hardcoded (IP direto) como fallback de emergencia.
#7
P2
Bug insight.trim em /olivia/treinar — tipo inconsistente no payload
Rota /olivia/treinar lanca insight.trim is not a function quando o campo insight do payload nao e string (objeto ou null). treinamento.mjs assume string sem validar o tipo. Treinamentos enviados via Telegram falham silenciosamente sem feedback ao advogado.
Substituir por String(insight ?? '').trim() ou adicionar validacao de schema (zod) no handler da rota. Log explicito com o payload recebido quando o tipo for inesperado.
