Vantagens e Desvantagens em Detalhe

S-04
✅ Vantagens — Requisição Única
  • 01 Custo -70% de input tokens Elimina contexto duplicado por seção. Em vez de enviar o processo 6 vezes, Claude recebe tudo uma vez: ~35K vs 150K de input.
  • 02 Coerência total da petição Claude escreve o documento sabendo de todas as seções simultaneamente. Elimina "quebras de voz" visíveis entre seções geradas em paralelo por workers independentes.
  • 03 Sem JSON fragmentation Parse de 1 resposta bem estruturada ao invés de N JSONs independentes com regex fallback. Menos pontos de falha de parsing.
  • 04 Zero overhead de pMapConcurrent Elimina latência de scheduling de workers paralelos, conexões HTTP múltiplas e race conditions entre seções.
  • 05 Menos chamadas API = menos pontos de falha 6 chamadas × falha de rede = 6 riscos. Uma chamada: 1 risco. Rate limiting também reduzido 6x.
  • 06 Petição mais fluida estilisticamente Claude usa introduções, transições e conclusão coesas, sem repetições entre seções que eram geradas sem conhecimento umas das outras.
❌ Riscos e Desvantagens
  • 01 Output token limit 15 páginas HTML ≈ 25-30K tokens. Requer Extended Output (beta Anthropic). Se truncar aos 28K, perde a petição completa.
  • 02 Sem recuperação parcial No sistema atual, se a seção 3 falha, as outras 5 sobrevivem. No modo único, qualquer falha exige retry completo da petição inteira.
  • 03 Latência de first token maior Com 28K de output, o usuário espera mais para ver o primeiro resultado. No modo paralelo, a UI pode mostrar progresso seção a seção.
  • 04 Attention dilution Claude com 35K de input pode ter atenção distribuída de forma menos focada por seção do que uma chamada dedicada a cada tópico.
  • 05 Debugging mais difícil Quando a petição tem problema (score baixo, tópico raso), não é possível isolar qual seção falhou. No modo atual, a seção problemática é identificável.
  • 06 Modo REFUTAÇÃO precisa ser reescrito A injeção dinâmica de tópico REFUTAÇÃO em réplicas (buildPromptSecao específico) precisa de novo formato de prompt para o modo único.

Plano de Implementação — Dual-Mode com Feature Flag

S-05
🔒 Princípio fundamental: O sistema atual é INTOCÁVEL. Criamos um novo path paralelo ativado por feature flag. Rollback instantâneo via variável de ambiente. Zero risco para petições em produção durante os testes.
Feature Flag — Controle de Modo
# .env — controle do modo de geração OLIVIA_MODO_GERACAO=per_section # atual (padrão, default) OLIVIA_MODO_GERACAO=single_request # novo (ativar por org para teste) # Exemplo de uso no geracao.mjs: const modoUnico = process.env.OLIVIA_MODO_GERACAO === 'single_request'; if (modoUnico) { return await gerarPeticaoUnica(topicos, contexto); } else { return await gerarPorTopico(topicos, contexto); // path atual intocado }
FASE 1 Infraestrutura Base ⏱ 1-2 dias
  • Criar buildPromptCompleto() em geracao.mjs ao lado de buildPromptSecao() — sem modificar o existente
  • Definir schema de output: { sections: [{titulo, conteudo_html, score, resumo}], peticao_html, score_geral }
  • Adicionar feature flag OLIVIA_MODO_GERACAO no .env e .env.example
  • Criar arquivo geracao-single.mjs com stub de função para não bloquear merge
  • Adicionar header Extended Output na instância Anthropic do modo único: anthropic-beta: output-128k-2025-02-19
FASE 2 Implementação do Novo Path ⏱ 2-3 dias
  • Implementar gerarPeticaoUnica(topicos, contexto) com max_tokens=32000 e stream=true
  • Criar parseSecaoUnica(response) — parser para o novo schema com array de sections
  • Retry logic específica: detectar truncamento quando output_tokens >= max_tokens * 0.95
  • Adaptar progressUpdate — no modo único usar streaming de tokens como indicador de progresso
  • Reescrever injeção de tópico REFUTAÇÃO no prompt único para réplicas
  • Fallback automático: se modo único falha 2x, chamar gerarPorTopico() automaticamente
FASE 3 Testes A/B Controlados ⏱ 1-2 semanas
  • Ativar via OLIVIA_MODO_GERACAO=single_request SOMENTE para org Midas em ambiente dev
  • Comparar métricas: score médio, tempo total, custo por petição, taxa de truncamento, taxa de erro JSON
  • Threshold de aprovação: score médio ≥ 7.6 E truncamento < 5% E JSON error < 2%
  • Logar ambos os modos na tabela olivia_geracoes com coluna modo_geracao para comparação estatística
  • Testar por tipo: manifestação (1-2 seções), réplica (6 seções), apelação (8+ seções)
FASE 4 Rollout Gradual ⏱ conforme aprovação FASE 3
  • Se FASE 3 aprovado: tornar single_request padrão para tipos simples primeiro — manifestação, emenda inicial, alvará
  • Após 2 semanas estável: ativar para réplica e contra-razões (maior benefício por número de seções)
  • Por último: apelação — mais seções, maior risco de truncamento, migrar somente após validação completa
  • Manter per_section como fallback automático permanente — nunca remover o path atual do código
  • Atualizar progressUpdate na UI do OlivIA para refletir o novo modelo de progresso

Aprovado
