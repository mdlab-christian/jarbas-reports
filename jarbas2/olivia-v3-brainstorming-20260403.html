1. Sinopse do Projeto

O que estamos reconstruindo

O gerador de peticoes OlivIA e a funcionalidade central do MdFlow. Estamos reconstruindo 3 camadas simultaneamente:


Mecanismo
  • Migrar de OpenCloud para scripts/skills Claude Code
  • Sistema de agentes no Mac Mini (user Olivia)
  • Analise automatica de processos via EPROC
  • Cerebro via RAG + Obsidian
  • Personalizacao por advogado
  • Modelo normal (Sonnet 4.6) e avancado (Opus 4.6)
Frontend (Modo Express)
  • Step 1: Identificacao (lapidar)
  • Step 2: Triagem/Topicos (recriar do zero)
  • Step 3: VibeLaw Editor (recriar do zero)
  • Step 4: Conclusao (recriar do zero)
  • Sistema de abas (ate 5 simultaneas)
  • Relatorio HTML dinamico
Playbooks (/playbooks)
  • Recriar pagina inteira do zero
  • Abas: Tipos, Modelos, Blocos, Tags, Juris
  • Sistema de treinamento (Telegram + UI)
  • Busca de jurisprudencias via browser
  • Style guides por advogado
  • Aba de Reportagens
Por que esse projeto e diferente

Christian escreveu 4 documentos detalhados a mao antes de usar IA. O desafio nao e falta de planejamento — e garantir que a IA siga o plano 100% sem alucinar, sem inventar, sem pular detalhes. Cada campo, cada botao, cada mecanismo esta especificado. A solucao: documentacao exaustiva + gestao de contexto por terminais + loop de qualidade automatizado.

3. Framework Multi-Terminal

Arquitetura de 6 Terminais

Cada terminal do Claude Code opera com contexto isolado. A comunicacao entre eles e feita via arquivos no Obsidian (jarbas-vault).

4. Sistema de Waves

Conceito de Wave

Uma Wave e um conjunto de tarefas paralelas que podem ser executadas ao mesmo tempo em diferentes terminais. Cada wave tem pre-requisitos, entregas e criterios de aprovacao.

Wave 0 — Organizacao & Pesquisa AGORA
Terminais: T1 (Gerente)
Entregas: HTML brainstorming, prompts Lovable/OliviaMDBot, estrutura Obsidian, docs organizados
Criterio: Christian valida HTML + recebe feedback externo
Wave 1 — Lapidacao do Projeto
Terminais: T1 + ate 3 terminais de pesquisa
Tarefas: Deep research por camada (backend, frontend, mecanismo), mockups HTML, mapeamento completo de tabelas, definicao de editor de texto, fluxos em diagrama, prompts pre-definidos
Criterio: Specs lapidados aprovados pelo Christian
Wave 2 — Backend Foundation
Terminais: T2 (Backend)
Tarefas: Migrations SQL, novas tabelas, RPCs, RLS policies, indices, seeds, verificacao schema
Criterio: Todas tabelas existem com campos corretos, RPCs funcionais, RLS testada
Wave 3 — Frontend Build
Terminais: T3 (Steps 1-2) + T4 (Steps 3-4) + T5 (Playbooks)
Tarefas: Implementar todos os componentes conforme spec lapidado
Pre-requisito: Wave 2 concluida
Criterio: Todas as telas renderizam sem erro, componentes conectados ao backend
Wave 4 — Mecanismo & Integracao
Terminais: T2 (scripts/skills) + T3 ou T4 (integracao frontend-mecanismo)
Tarefas: Skills de geracao, analise EPROC, RAG, prompts IA, webhooks n8n, conexao frontend<->mecanismo
Criterio: Fluxo completo funciona end-to-end (gerar 1 peticao real)
Wave 5 — QA Loop (repete ate 100%)
Terminais: T6 (QA) + terminais de correcao conforme necessidade
Tarefas: Testes E2E por browser, relatorios HTML, correcao de bugs, lapidacao UI/UX, performance
Criterio: 50 peticoes geradas sem falha, todos os checks do checklist master passam

5. Gestao de Contexto Anti-Alucinacao

O Problema
  • Projeto tem +30.000 palavras de documentacao
  • Context window do Claude Code tem limite (~200K tokens)
  • Multiplos terminais NAO compartilham contexto
  • IA alucina quando tem contexto demais ou de menos
  • Historicamente, projetos grandes falham por drift do spec
A Solucao
  • Contexto on-demand: Cada terminal le APENAS os arquivos que precisa
  • Spec como SSOT: Obsidian e a verdade, nao a memoria da IA
  • /clear agressivo: A cada 50% da context window
  • Prompts atomicos: Cada prompt faz UMA coisa, referenciando o spec
  • Verificacao cruzada: T6 (QA) valida contra o spec original
Template de Prompt Anti-Alucinacao
TAREFA: [descricao especifica] SPEC: Leia ~/jarbas-vault/02_projetos/olivia-v3/03_spec-lapidado/[ARQUIVO].md REGRAS: - NAO invente campos, tabelas ou funcionalidades que NAO estao no spec - Se encontrar ambiguidade, PARE e escreva em DECISOES.md - Ao terminar, atualize STATUS.md com o que foi feito - Se context > 50%, faca /clear e recomece referenciando o spec CHECKLIST de validacao: [lista especifica para a tarefa]

O que o E2E DEVE verificar (nao so acessar)
  • Geracao de peticao: A peticao gerada esta correta juridicamente? Tem todos os topicos? Jurisprudencias corretas?
  • Relatorio HTML: Todas as sections existem? Dados estao corretos? Design system ok?
  • Step 2 Topicos: Os topicos gerados pela IA sao relevantes para o caso? Drag-and-drop funciona?
  • Step 3 Editor: O texto aparece formatado (nao markdown)? VibeLaw edita corretamente?
  • Step 4: PDF baixa? Google Docs abre? Relatorio final esta completo?
  • Multiplas abas: 5 peticoes simultaneas funcionam sem conflito?
  • Jurisprudencias: Sistema usa juris do estado correto? Respeita regra por advogado?
  • RAG: Respostas estao usando o treinamento da Olivia?

8. Checklist Master de Validacao

Frontend — Design System v5
  • Zero hex hardcoded (somente CSS variables/tokens Tailwind)
  • Icones: apenas lucide-react
  • Fontes: Manrope H1/H2, Inter corpo, JetBrains Mono IDs
  • Glassmorphism apenas em cards/modais/popovers
  • Arquivos TSX maximo 500 linhas
  • Dark mode Warm Charcoal (#12100e background)
  • Primary color via var(--primary) nunca #20c997
  • Responsividade: 375/640/768/1024/1440px
  • z-index scale respeitado
Frontend — Codigo
  • TypeScript estrito (zero any)
  • staleTime: 30_000 em TODOS os useQuery
  • .is('deleted_at', null) em TODAS as queries
  • .eq('organizacao_id', orgId) em TODAS as queries
  • Zero console.log em producao
  • Nomenclatura: snake_case banco, camelCase JS, PascalCase componentes
  • Hooks: use{Conceito}.ts
  • Valores monetarios em centavos (INT8)
  • Datas: date-fns + ptBR locale
  • Action-Driven UI (switch response.action)
Backend — Supabase
  • organizacao_id + RLS em TODA tabela nova
  • TEXT CHECK (nunca ENUM nativo PostgreSQL)
  • Soft delete (deleted_at) obrigatorio
  • RETURNING id em todo INSERT
  • Trigger updated_at em toda tabela
  • Indices: organizacao_id + WHERE deleted_at IS NULL
  • JSONB atomico via jsonb_set (nunca UPDATE direto)
  • RPCs: SECURITY DEFINER, RETURNS JSONB
  • public.get_org_id() para tenant isolation
  • Unaccent + lower para buscas
Mecanismo — Geracao de Peticoes
  • Jurisprudencias do estado correto (RS->TJRS, SC->TJSC, SP->TJSP)
  • Minimo 1 jurisprudencia STJ
  • Style guide do advogado aplicado
  • Enderecamento personalizado por advogado
  • Rodape personalizado por advogado
  • Peticao de referencia (Smart Blocks) gerada antes da IA
  • Revisao por IA com nota minima 8/10
  • Relatorio HTML atualizado por step
  • Playbook correto para categoria de empresa
  • Contexto on-demand (nao sobrecarregar)
  • Rate limit: 10 geracoes/hora por usuario
  • Modelo normal (Sonnet 4.6) vs avancado (Opus 4.6)
Mecanismo — Analise de Processo (EPROC)
  • Acessa EPROC via Playwright (Mac Mini, user Olivia)
  • Le arvore de eventos do processo
  • Identifica documentos relevantes por tipo de peticao
  • Baixa e analisa documentos (OCR + Vision)
  • Categoriza provas automaticamente
  • Delays anti-bot: 800-2500ms entre acoes
  • TOTP para autenticacao 2FA
  • Nao rodar 2 sessoes simultaneas mesma OAB
Pagina /playbooks
  • Abas: Tipos Peticao, Modelos, Blocos, Tags, Juris, Playbooks, Gerador, Reportagens, Style Guides
  • Cada aba com CRUD completo
  • Sistema funciona para Smart Blocks E Express
  • Alimenta RAG, tabelas backend E Obsidian
  • Busca jurisprudencias via browser (tribunais)
  • Formatacao preservada na peticao final
  • Interface non-tech para equipe juridica
  • Excluir abas Flow e Aprovacoes

12. Riscos & Mitigacoes

R1: Perda de Contexto entre Terminais

Risco: Terminais divergem do spec porque nao compartilham memoria

Mitigacao: Obsidian como SSOT. Cada prompt refere arquivo especifico. QA valida contra spec.

R2: Alucinacao da IA

Risco: IA inventa campos, funcionalidades ou comportamentos

Mitigacao: Prompts atomicos com "NAO invente". Checklist de validacao pos-implementacao. /clear agressivo.

R3: Editor de Texto Complexo

Risco: VibeLaw (AI editing) pode ser muito complexo de implementar

Mitigacao: Editor base funciona independente do VibeLaw. Implementar editor primeiro, VibeLaw depois.

R4: EPROC Automation Instavel

Risco: Scraping do EPROC pode falhar (captcha, mudancas de layout)

Mitigacao: Reutilizar mecanismo comprovado do Controller. Fallback: analise manual continua funcionando.

R5: Context Window Overflow

Risco: Specs muito grandes para caber na context window

Mitigacao: Dividir specs por camada (<25K tokens cada). /clear + referencia ao arquivo.

R6: Performance com 5 Abas Simultaneas

Risco: 5 editors + 5 chats + 5 realtime subscriptions = lento

Mitigacao: Lazy loading, aba inativa suspende subscriptions, Zustand com sessionStorage.
