O Que AJUSTARIA no Projeto

7 ajustes ▼

1 Issues atomicas em vez de waves

O conceito de "waves" com terminais paralelos e bom para sequenciamento macro, mas dentro de cada wave, devemos usar issues atomicas (1 comportamento = 1 issue). Cada terminal recebe 1 issue por vez, nao a wave inteira. Isso evita context window overflow e garante foco absoluto.

2 Prototipos primeiro

Seguindo Deborah: fazer frontend visual ANTES de plugar backend. Step 1 visual → Step 2 visual → Step 3 visual → Step 4 visual → DEPOIS conectar backend. Permite validacao visual com Christian antes de investir em logica complexa.

3 Relatorio HTML como produto separado

D5 confirmou: relatorio NAO fica inline expandido. E um "super relatorio" HTML acessavel via botao. Isso simplifica MUITO o Step 2 — so mostra resumo compacto (score + pilares + alertas criticos). Gerar 1x e salvar em Storage.

4 Custo vai subir com zero-Haiku

De R$0.05-0.30 para R$0.50-1.50 por peticao. Para ~80-100 peticoes/dia = ~R$40-150/dia (~R$900-3.400/mes). Precisa monitorar. Mas Christian priorizou qualidade sobre custo (D9).

5 Edge Functions vs SSH direto

D1 diz "Edge Functions delegam para scripts via SSH". Mas SSH para Mac Mini e ponto de falha. Alternativa: Edge Functions chamam Claude API direto (sem Mac Mini no loop critico). Mac Mini so para: EPROC (Playwright), vault access, PDF generation.

6 VibeLaw com Sonnet 4.6

Slash commands no editor vao ter ~1-2s de latencia (Sonnet). Aceitavel para /reescrever secao, mas pode ser lento para /sugerir palavra. Considerar: Sonnet para operacoes pesadas, cache de sugestoes para leves.

7 Codex review model

Christian quer "o mais avancado do Codex" (D3). Verificar qual e o modelo mais recente disponivel via API OpenAI. Candidatos: o3, o4-mini-high, codex-mini. Definir exatamente qual usar antes de Wave 1.

RISCOS Nao Cobertos

6 riscos ▼

R1 Latencia do pipeline completo

Hoje com Haiku: 2-5 min por peticao. Com Sonnet 4.6 em TODOS os steps (S0b, S1, S3, S4, S6, S9), pode ir para 5-10 min. Operadores acostumados com 2-3 min podem reclamar. Sem benchmark real ainda.

R2 Sem fallback de modelo

Se a API da Anthropic cair, todo o pipeline OlivIA para. Nao ha fallback definido para OpenAI/Gemini. Para ~80-100 peticoes/dia, 1 hora de downtime = ~5-10 peticoes perdidas.

R3 SmartBlocks ceiling de 8K chars

Peticoes complexas (execucao fiscal, acao coletiva) podem exceder o limite de 8.000 caracteres por bloco. Estrategia de chunking nao definida. 1.244 blocos existentes no banco.

R4 RAG com 404 intermitente

fn_rag_search retorna PGRST202 as vezes (funcao nao encontrada). Fallback por tags existe mas nao e ideal — perde contexto semantico. Nao investigado a fundo ainda.

R5 Apenas 3 juizes mapeados

So 3 juizes tem perfil no sistema de potencialmente 50+ juizes ativos nos tribunais TJRS, TJSC e TJSP. Estrategia generica para juizes nao mapeados precisa ser robusta o suficiente para nao degradar qualidade.

R6 olivia_agente_contexto vazia

Tabela de memoria processual (olivia_agente_contexto) nunca foi populada. Se o pipeline depende dela para contexto, vai falhar silenciosamente. Decidir: popular ou remover do schema.

Recomendacoes do Jarbas

6 sugestoes ▼

Sugestoes baseadas na experiencia com os outros projetos do MdFlow. Nenhuma e obrigatoria — Christian decide.

1 Pipeline critico sem Mac Mini

Para geracao de peticoes, Edge Functions devem chamar Claude API direto (sem SSH para Mac Mini). Mac Mini so para: Playwright/EPROC, PDF generation, vault access. Isso elimina SSH como ponto de falha no caminho critico. Se a Edge Function faz POST para api.anthropic.com, o Mac Mini pode estar offline sem impactar geracao.

2 Cache de SmartBlocks

Os 1.244 blocos do banco nao mudam frequentemente. Cachear em Edge Function memory ou KV store (Supabase cache / Deno KV). Evita query ao Supabase em cada geracao. Invalidar cache so quando operador edita bloco na pagina /playbooks. Economia de ~200ms por geracao.

3 Relatorio como HTML estatico

Em vez de gerar o relatorio HTML dinamicamente toda vez que o operador clica "Ver Relatorio", gerar 1x apos a analise (Step S2-S3) e salvar em Supabase Storage bucket olivia-reports. Botao no Step 2 so abre a URL. Latencia zero no acesso ao relatorio.

4 Issues com pre/pos condicoes

Cada issue deve ter: PRE (o que ja deve existir antes de implementar), POS (o que validar apos implementar). Isso permite que qualquer terminal pegue qualquer issue sem depender de contexto da conversa. Terminal pode /clear e retomar de onde parou.

5 Prototipos em branch

Frontend visual em branch separada (olivia-v3-proto) para Christian validar mockups antes de implementar logica. Merge apos aprovacao. Prototipos usam dados mockados — sem Supabase, sem Edge Functions. Previne retrabalho de logica em UI nao aprovada.

6 Monitoring desde o dia 1

Adicionar metricas desde o inicio: tempo de geracao por step, score medio do revisor, custo por peticao (tokens * preco), taxa de aprovacao (score >= 7). Dashboard simples no /lolito ou tabela olivia_metrics. Sem metricas, nao sabemos se esta melhor ou pior que v2.
