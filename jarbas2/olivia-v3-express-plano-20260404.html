Seção 1 — Decisão Estratégica

Por que Express First?

Funcionalidade P0

O gerador de petições é o coração do MdFlow. Sem ele, nada funciona. Antes de construir Chat e Análise, o Express precisa estar estável, testado e confiável em produção.

Escopo controlável

Express first = 4 steps bem definidos, sem complexidade de Chat e Análise simultaneamente. Cada issue é atômica, testável e entregável numa única sessão.

Validação de mecanismo

Express conecta diretamente com o pipeline-v3.mjs no Mac Mini. Se funcionar end-to-end, temos confiança arquitetural para expandir para os outros modos.

Frontend limpo

Apagar Chat e Análise reduz complexidade em 32%. Mais fácil de manter, debugar e testar. Zero dead code, zero imports quebrados, zero componentes orfãos.

VEREDICTO: PROJETO NOVO via /projeto
  • Projeto anterior tinha 76 issues incluindo mecanismo e backend — ambos já feitos
  • Spec novo deve refletir: somente Express, somente 4 steps, somente frontend
  • Spec limpo = issues atômicas sem bagagem antiga nem issues resolvidas contaminando o board
  • Usar skill /projeto para: spec → break → plan → execute (anti-vibe-coding)
  • Cada issue = 1 comportamento ou 1 componente. Commit após cada uma.

Seção 6 — Fluxo Completo do Express na Prática

Entrada
Usuário abre /olivia
OliviaGeradorLanding renderiza landing simples com histórico recente e botão "Nova Petição Express". Sem abas, sem complexidade.
Step 1
OliviaGeradorExpress renderiza Step 1
Usuário seleciona: processo via CNJ search, tipo de petição, modelo, qualidade IA. Faz upload de documentos de suporte (PDFs). Edge Fn olivia-extrair-texto extrai texto dos PDFs. Usuário clica "Analisar".
SearchBoxProcesso → SeletorTipoPeticao → SeletorModelo → QualidadeIaSelector → UploadDrawer
Transição 1→2 (crítica)
handleStep1Next() executa com await
RPC fn_olivia_iniciar_geracao() retorna historicoId. Webhook POST /olivia/gerar fase=1 é disparado. SOMENTE APÓS historicoId confirmado no banco o setStepIndex(1) executa. Mac Mini: S1 fetchContexto + S2 RAG + S3 gera tópicos → UPDATE status='AGUARDA_RESPOSTAS'.
await fn_olivia_iniciar_geracao() → historicoId await confirmHistoricoId(historicoId) // query banco webhook(fase=1, historicoId) // fire setStepIndex(1) // só aqui
Step 2
Tópicos + Análise (com historicoId garantido)
Realtime subscription aguarda olivia_historico.status='AGUARDA_RESPOSTAS'. Carrega tópicos e análise via useTopicosV3. Usuário: drag-and-drop para reordenar tópicos, responde perguntas, toggle on/off cada tópico. Usuário clica "Gerar Petição".
Realtime: olivia_historico WHERE id = historicoId → on('AGUARDA_RESPOSTAS'): carrega tópicos + análise → TopicosPanel com DnD sortable → handleStep2Next(topicosAprovados)
Transição 2→3
handleStep2Next() dispara fase 2
Webhook POST /olivia/gerar fase=2 com tópicos aprovados e instruções extras. setStepIndex(2) imediato + GenerationProgressPanel mostra progresso. Mac Mini: S4 gerarPeticao + S5 processarHtml + S6–S8 loopRevisao + S9 persistir.
webhook(fase=2, { historicoId, topicosAprovados, instrucoes }) setStepIndex(2) // imediato — GenerationProgressPanel Mac Mini → UPDATE progresso_pct, progresso_msg Mac Mini → UPDATE status='CONCLUIDO', peticao_html=...
Step 3
Editor TipTap recebe petição gerada
Realtime detecta status='CONCLUIDO'. Carrega peticao_html do banco no editor TipTap. Usuário edita manualmente, formata, revisa. Autosave a cada 30s via RPC. Usuário clica "Finalizar".
Realtime → on('CONCLUIDO'): setContent(peticao_html) useInterval(30_000, () => autosave(historicoId, editor.getHTML())) → ExpressStep4Rascunho + TipTap + GenerationProgressPanel

Conclusão + Downloads
Grid de download: PDF, DOCX, Google Docs. FeedbackSection: advogado avalia qualidade da petição (1–5 estrelas + comentário). Feedback aciona fn_olivia_registrar_feedback → retroalimenta playbook (Stage 9). Botão "Nova Petição" reinicia o fluxo.
DownloadCardsGrid → olivia-criar-doc (Edge Fn) → Supabase Storage URL FeedbackSection → fn_olivia_registrar_feedback(historicoId, nota, comentario) → retroalimenta vault/playbook para próximas gerações

Seção 8 — Como Usar o /projeto (Anti-vibe-coding)

/projeto = Spec → Break → Plan → Execute. Cada fase tem um /clear entre elas. Nenhuma issue começa sem spec aprovado. Nenhum código sem plan aprovado.

01

/projeto spec

Christian escreve o spec completo da /olivia Express baseado neste HTML + vault. Descreve cada step, cada componente, cada integração, cada critério de aceite por issue. Este documento é o ponto de partida.

02

/projeto break

Quebrar o spec em ~26 issues atômicas. Cada issue = 1 comportamento ou 1 componente. Pré-condições e pós-condições por issue. Ordem: limpeza → mecanismo → wizard → step1 → step2 → step3 → step4 → EF → QA.

03

/projeto plan

Para cada issue, Claude Code pesquisa o codebase + docs antes de implementar. Evita inventar soluções que já existem. Referencia arquivos concretos com linha exata. Apresenta 2 opções quando há decisão arquitetural.

04

/projeto execute

Implementação issue por issue com commit após cada uma. 3–4 issues por sessão. Code review após cada grupo. Um terminal por camada (front, back, EF). Sem acumular débito entre sessões.
