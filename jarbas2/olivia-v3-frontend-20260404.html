🔒 Segurança — Vulnerabilidades P0
P0-A OLIVIA_TOKEN exposto no bundle frontend
3 arquivos fazem fetch() direto para o servidor OlivIA com token no header Authorization. O token fica visível no JavaScript bundle da aplicação (qualquer usuário pode acessar).
ArquivoLinhaEndpointFix
useExportarDocumento.ts 44 OLIVIA_BASE_URL/olivia/criar-doc → functions.invoke('olivia-criar-doc')
UploadDrawer.tsx 224 OLIVIA_BASE_URL/olivia/extrair-texto → functions.invoke('olivia-extrair-texto')
OliviaAnaliseUnificada.tsx 32 OLIVIA_BASE_URL/olivia/analisar → deletar tryOlivia() por completo
P0-B Fire-and-forget com token — UploadDrawer:224
Além de expor o token, a chamada para extrair-texto é fire-and-forget (.catch(() => {})). Erro silencioso: usuário não sabe se o PDF foi indexado. Docs sem extração = IA sem contexto = petição mais fraca.
// ANTES — P0: fetch(`${OLIVIA_BASE_URL}/olivia/extrair-texto`, { headers: { 'Authorization': `Bearer ${OLIVIA_TOKEN}` }, body: JSON.stringify({ documento_id, arquivo_url }), }).catch(() => {}); // fire-and-forget // DEPOIS — V3: const { error } = await supabase.functions.invoke('olivia-extrair-texto', { body: { documento_id, arquivo_url } }); if (error) toast.warning('Extração falhou — IA terá menos contexto');
P0-C dangerouslySetInnerHTML — OliviaRelatorioPage:159
Fallback de renderização renderiza peticao_html diretamente no DOM sem sanitização. Risco XSS se o HTML gerado contiver scripts maliciosos. Fix trivial: substituir por iframe sandboxed.
// ANTES — XSS: <div dangerouslySetInnerHTML={{ __html: dados.peticao_html }} /> // DEPOIS — seguro: <iframe srcDoc={dados.peticao_html} sandbox="allow-same-origin" title="Petição" className="flex-1 w-full border-0" />
