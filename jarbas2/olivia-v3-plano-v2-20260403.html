Decisoes Confirmadas

▼
D1 Arquitetura 100% Scripts + Edge Functions (ZERO n8n) CONFIRMADO

Toda a pipeline de geracao de peticoes roda via scripts Node.js no Mac Mini (user: olivia) + Edge Functions Supabase. O n8n NAO participa do fluxo OlivIA V3. Motivacao: controle total, sem intermediarios, latencia minima, logs atomicos.

D2 Editor TipTap para edicao de peticoes CONFIRMADO

O editor de peticoes usa TipTap (ProseMirror). Ja instalado no projeto. Suporte a extensoes custom para VibeLaw (slash commands, formatacao juridica, navegacao por secoes). Substitui textarea simples e permite output HTML nativo.

D3 Code Review automatizado via OpenAI o4-mini CONFIRMADO

Cada commit no ciclo de build passa por review automatizado usando o4-mini (custo baixo, alta capacidade de analise de codigo). Classifica achados em P0 (bloqueante), P1 (importante), P2 (melhoria). P0s devem ser corrigidos antes de merge.

D4 Chat OlivIA — Modelo IA: Hibrido Haiku/Sonnet DECIDIR

O chat da OlivIA (VibeLaw slash commands + reescrita) precisa de um modelo IA. Tres opcoes analisadas:

ModeloCusto / 1M tokensLatenciaQualidadeUso Ideal
Haiku 4.5 $0.25 / $1.25 ~200ms Bom para slash commands VibeLaw, classificacao, quick edits
Sonnet 4 $3 / $15 ~1s Melhor para reescrita Geracao de peticao, reescrita longa
Hibrido Variavel Variavel Melhor de ambos Haiku VibeLaw + Sonnet geracao
Recomendacao: Hibrido — Haiku 4.5 para VibeLaw slash commands (rapido, barato) + Sonnet 4 para geracao/reescrita de peticao (qualidade juridica). Routing automatico por tipo de request.
D5 Relatorio de Analise — Posicionamento na UI DECIDIR

O relatorio de analise de provas (output do Step S3) precisa de um local na UI. Tres opcoes:

OpcaoDescricaoProsContras
A. Embutido Step 2 Aparece inline no wizard UX fluida, sem navegacao extra Step 2 fica mais pesado
B. Pagina separada Rota /olivia/relatorio/:id Compartilhavel, bookmarkable Quebra fluxo do wizard
C. Modal/Drawer Sheet lateral ou modal Rapido, nao sai do wizard Espaco limitado, scroll extenso
Recomendacao: Opcao A — Embutido no Step 2. O usuario veria topicos + relatorio de analise na mesma tela, com secoes colapsaveis. Mantem o fluxo natural do wizard sem navegacao extra.
D6 Tab System via Zustand Store — MANTER CONFIRMADO

O sistema de tabs (abas de peticao) usando Zustand store (usePeticaoStore.ts) permanece. Funciona bem, mantem estado entre steps, permite multiplas peticoes simultaneas. Nao ha razao para migrar para outro state manager.

Conhecimento Critico — 6 Regras Inviolaveis

▼
1. Venire Check
Verificar se o credito do cliente ja foi regularizado ANTES de gerar a peticao. Se o Serasa/SPC ja removeu a negativacao, a peticao perde o objeto. Pipeline S3 deve consultar status atual do credito.
2. Filtro de Jurisprudencia por Estado
RS usa jurisprudencia do TJRS. SC usa TJSC. SP usa TJSP. NUNCA misturar. A tabela olivia_jurisprudencias precisa do campo estado (UF) para filtrar corretamente. Migration pendente.
3. Travessao PROIBIDO
O caractere travessao (—) e proibido em QUALQUER peticao, para ambos advogados. O pipeline S6 e S9 devem detectar e substituir automaticamente por virgula ou ponto. Regex: /[\u2013\u2014]/g
4. Condensacao de Topicos
Maximo 6-8 topicos por peticao. Se o Step S4 gerar mais, deve condensar automaticamente. Topicos muito granulares enfraquecem a argumentacao. Regra: se > 8, merge os 2 mais relacionados ate caber.
5. Sumula 385
Se o cliente tem MULTIPLAS restricoes ativas (nao so a do caso), a Sumula 385 do STJ pode impedir dano moral. O Step S3 deve detectar multiplas negativacoes e alertar o advogado ANTES de gerar a peticao.
6. Jurisprudencia Validada
Todo numero de jurisprudencia citado na peticao DEVE existir no banco (olivia_jurisprudencias). O Step S9 (revisor) verifica cada citacao. Se inventar numero, score automatico = 0.
