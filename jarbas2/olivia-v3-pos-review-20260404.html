6. A Abordagem por Issues Funcionou?

Pros
  • Rastreabilidade: Cada commit com ISS-XXX permite auditar o que foi feito, quando, e por que
  • Paralelismo: 6 terminais (T-CTRL, T-FRONT, T-BACK, T-MECH, T-PLAY, T-QA) sem conflitos de arquivo
  • Atomicidade: Cada issue = 1 comportamento = 1 commit. Facil reverter se necessario
  • Progresso visivel: Em qualquer momento, sabemos quantas issues estao feitas vs pendentes
  • Context window: Issues atomicas evitam overflow de contexto no Claude Code
  • Sprint boundaries: Validacao a cada sprint evita acumulo de bugs
Contras (para melhorar)
  • Sprint 9 (QA) sem rigor: Issues de teste ficaram "NAO VERIFICAVEL" — faltou criterio de aceite automatizado
  • code-review.mjs nunca criado: O loop de qualidade de build nao existiu na pratica
  • 4 ideias aprovadas nao feitas: ISS-029/031/032/033 foram planejadas mas nao executadas — faltou prioridade vs escopo

Recomendacao para proximos projetos:

Manter issues atomicas (funcionou). Criar script de code review ANTES de comecar (Wave 0). Sprint de QA precisa de criterios automatizados (E2E scripts, nao "verificacao manual"). Ideias aprovadas que nao cabem no escopo devem ser movidas para backlog explicitamente, nao deixar "NAO FEITO".
