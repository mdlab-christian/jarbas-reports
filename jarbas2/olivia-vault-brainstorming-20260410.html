5. Sistema On-Demand — Agente Recebe o que Precisa

Principio de ouro
O agente nunca recebe TUDO — a janela de contexto é limitada. Recebe exatamente o que precisa para cada etapa. Vault menor e mais focado = contexto de maior qualidade = petições melhores. Mais contexto irrelevante = pior performance.
1
Step 1 — Identificação do processo
Zero vault. Apenas UI e banco. Coleta dados: número CNJ, réu, tipo de negativação, datas. Nenhum arquivo do vault é carregado nesta etapa.
0 arquivos
2
Step 2 — Análise jurídica do caso
Vault carrega: SOUL + STYLE + ERRORS + analise.md + playbooks filtrados por réu/tipo + GLOSSARY + TESES relevantes. RAG busca jurisprudências por similaridade com os fatos do caso.
~8 arquivos
3
Step 3 — Geração da petição
Vault carrega: SOUL + STYLE + geracao.md + playbooks + feedbacks few-shot (modo avançado: 3-5 petições aprovadas similares) + DISTINCOES-ADVOGADOS. RAG busca blocos por categoria.
~10 arquivos
V
Modo VibeLaw — revisão de estilo
Vault carrega: SOUL + vibelaw.md. Mínimo absoluto — apenas identidade e prompt específico do modo. Velocidade máxima para revisões rápidas.
~2 arquivos
R
Modo Relatório — síntese executiva
Vault carrega: SOUL + relatorio.md + dados do processo do banco. Sem playbooks nem few-shot. Foco em dados estruturados e formatação executiva.
~2 arquivos

9. Nota Crítica do Avaliador

Honestidade técnica
Por que 8.5 e nao 10/10
Blocos e jurisprudências ficam no Supabase. Boa decisão arquitetural, mas limita o vault. O agente depende de conectividade e do pgvector para acessar 2.611 itens. Se o RAG falhar, o agente perde acesso ao conteúdo mais rico.

MEMORY.md manual, não automático. O agente não escreve na própria memória. Alguém precisa popular manualmente. Problema real a resolver no futuro com edge functions ou pipeline de pós-geração.

50 petições few-shot são poucas. Para calibrar bem um LLM no domínio de negativação indevida, o ideal seria 200-500 exemplos com variação por réu, valor, tribunal e juiz.

Google Embeddings ainda não implementados. P0 crítico independente do vault. O RAG atual usa embeddings genéricos. Embeddings específicos para jurídico brasileiro melhorariam significativamente a qualidade das buscas semânticas.
Por que o plano é o correto
Sem vault rico, o agente é cego. Pode ter o melhor RAG do mundo, mas sem as regras da equipe, o estilo correto, as teses vencedoras e os feedbacks anteriores, vai gerar petições genéricas que não refletem o padrão do escritório.

O ativo mais valioso está esperando no recuperacao-vault. O conhecimento dos advogados Leonardo, Stefany e Heloísa é único, não reproduzível por nenhum modelo de IA. Isso é o diferencial competitivo real do sistema.

Com vault rico, o agente opera com autonomia real. Contexto correto no momento certo = petições calibradas para o réu, tribunal e advogado específicos. A diferença entre 7/10 e 9/10 está inteiramente no contexto injetado.
Dependencia critica — independente do vault
Google Embeddings (ou embeddings jurídicos especializados) é P0 independente desta reestruturação. O vault pode estar perfeito, mas se o RAG retornar jurisprudências irrelevantes por embeddings genéricos, o agente usará contexto errado. Esta é a próxima fronteira técnica após o vault estar estruturado.
