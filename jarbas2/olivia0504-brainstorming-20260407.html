Step 1 — Identificação: Issues

Upload de provas, instruções, seleção de processo
🔴 Provas não persistem ao mudar de aba
P0 · Bug
Ao fazer upload de documentos no Step 1 e mudar para outra aba (tab do wizard), os arquivos enviados desaparecem. O usuário perde todas as provas ao voltar para o tab.
Causa raiz: sessionStorage serializa o estado Zustand (usePeticaoStore), mas objetos File do JavaScript não são serializáveis. A lista de documentos armazena referências a File objects que são perdidas na serialização. Solução: armazenar apenas os metadados (nome, URL no bucket, tipo) e não os File objects em si — o upload para olivia-docs já acontece antes de mudar de aba.
🟡 Instruções adicionais não chegam na Edge Function
P1 · Bug
O campo de instruções adicionais (observações do advogado, chips preset "Incluir Súmula 385", etc.) é preenchido no Step 1, mas pode não estar sendo passado corretamente no body da chamada a olivia-topicos.
Causa raiz: Verificar no StepIdentificacao.tsx se o campo `instrucoes` do wizardState está incluído no body da invocação da EF olivia-topicos. Também verificar se a EF repassa esse campo para o Mac Mini e se o pipeline S3 (buildUserPromptV3) usa as instruções no prompt.
🟡 Petição simples sem análise de processo: fluxo undefined
P1 · Feature
Quando o usuário seleciona um tipo de petição simples (emenda, manifestação) sem habilitar análise de processo e sem enviar provas, o sistema não tem um fluxo claro. O que acontece no Step 2? Tópicos genéricos? Perguntas baseadas no modelo?
Causa raiz: O projeto original especificou dois caminhos: com análise (tópicos por IA baseados em provas) e sem análise (perguntas baseadas no modelo de petição e backend). O segundo caminho (petição simples) não está mapeado/implementado. Precisamos de um fluxo específico: Step 2 simplificado sem tópicos de análise de provas, apenas perguntas do modelo.
ℹ️ Feedback de OCR ausente para o usuário
P2 · UX
Após upload, os documentos ficam em status pendente_ocr. O usuário não sabe que o OCR ainda não foi executado. Só acontece ao clicar "Iniciar Análise". Se o OCR falhar, não há feedback claro.
Causa raiz: Arquitetura deliberada (OCR deferido para economizar processamento), mas falta comunicação visual. Adicionar badge/tooltip informando que "Os documentos serão processados ao iniciar a análise".

Step 2 — Tópicos e Análise: Issues

Perguntas, score, chat, relatório, drawer
🔴 Tópicos não exibem perguntas
P0 · Bug Crítico
Cada tópico deveria ter perguntas geradas por IA (sim/não, input numérico, select, texto livre). A estrutura PerguntaTopicoV3 existe no hook useTopicosV3, mas não há componente de UI para renderizar e responder essas perguntas dentro de cada card de tópico.
Causa raiz: O hook armazena as perguntas via `updatePerguntas()` mas o componente TopicoItemV3 não as renderiza. O agente de IA retorna perguntas no JSON de tópicos, mas o frontend ignora esse campo. Precisa criar componente PerguntasTopicoUI que renderize dinamicamente baseado no `tipo` da pergunta (boolean → toggle, numero → number input, texto → textarea, select → combobox).
🔴 "VibeLaw está digitando" aparece 2x
P0 · Bug
Ao enviar mensagem no chat do modo OlivIA (Step 2), o indicador "VibeLaw está digitando" aparece duplicado na interface.
Causa raiz: Provavelmente há dois listeners/states sendo atualizados simultaneamente no OliviaChatStep2: um no componente pai e outro no filho. Verificar se o estado `isTyping` está sendo gerenciado em dois lugares distintos ou se há dois componentes de chat sendo renderizados simultaneamente.
🔴 Botão de baixar relatório HTML carregando infinito
P0 · Bug
O botão de baixar relatório HTML no Step 2 fica em estado de loading indefinido e nunca gera o arquivo.
Causa raiz: O hook useOliviaExportV3 invoca a EF olivia-criar-doc com formato 'relatorio_html', mas a EF pode estar: (a) com timeout, (b) com erro silencioso, ou (c) o Mac Mini não implementou geração de HTML de relatório completo ainda. O DownloadCardsGrid tem fallback window.open() mas pode ser bloqueado por popup blocker. Investigar logs da EF olivia-criar-doc no Supabase.
🔴 Score representa nota da defesa, não nossa chance de ganhar
P0 · Lógica
O score exibido no Step 2 (e Step 4) está avaliando a qualidade/solidez da defesa da empresa ré. Deveria ser o contrário: nossa probabilidade de êxito baseada na análise das provas, jurisprudência e comportamento do juiz.
Causa raiz: O prompt do quality reviewer (Mac Mini S6) está configurado para avaliar a "nota da defesa" em vez de "nossa chance de ganhar". Precisa corrigir o sistema prompt: "Avalie a probabilidade de êxito do autor (MP Advocacia) neste processo, considerando: força probatória das provas, jurisprudência favorável, comportamento histórico do juiz, e solidez jurídica dos pedidos. Score 0-10 onde 10 = vitória quase certa."
🟡 Modelo de petição não considerado na análise de tópicos
P1 · Bug
O modelo de petição selecionado no Step 1 (ex: réplica, apelação, agravo interno) determina quais tópicos e perguntas devem ser gerados. Atualmente o sistema parece gerar tópicos genéricos independente do modelo.
Causa raiz: O modelo_skeleton é buscado via RPC fn_olivia_get_model_skeleton e passado para olivia-topicos, mas o Mac Mini S2 (RAG + tópicos) pode não estar usando as tags e seções do modelo como guia para gerar os tópicos relevantes. Verificar se o buildUserPromptV3 inclui o modelo_skeleton no contexto.
🟡 Drawer lateral sem drag-resize
P1 · UX
O drawer do chat OlivIA no Step 2 tem largura fixa de 30% da tela. O usuário não consegue ajustar o tamanho arrastando a borda para ver mais tópicos ou mais chat.
Solução: Adicionar uma borda drag-handle entre o painel de tópicos e o chat drawer. Implementar com MouseEvent onMouseDown → onMouseMove no document. Armazenar a largura no localStorage para persistir entre sessões.
3️⃣

Step 3 — Editor VibeLaw: Issues

Revisor, modelo, sumário, resize, polish geral
🔴 Revisor gera petição do zero em vez de corrigir pontos
P0 · Lógica Crítica
Ao avançar do Step 2 para o Step 3, o sistema (revisor de IA) deveria analisar a petição gerada e fazer correções pontuais nos trechos que precisam melhorar, preservando o restante. Atualmente o revisor está gerando uma nova petição do zero, ignorando a petição de referência.
Causa raiz: O pipeline S6 (quality review) calcula o score mas não implementa a lógica de "patch" — identificar seções específicas que precisam melhoria e reescrevê-las. Precisa ser reformulado: (1) revisor analisa a petição gerada e produz um JSON de patches [{secao, problema, sugestao}], (2) o sistema aplica apenas os patches nas seções problemáticas, (3) avança para o Step 3 com a petição corrigida (não nova).
🔴 Tipo de petição não considerado na geração (réplica vs inicial)
P0 · Bug Crítico
Independente do tipo de petição selecionado (réplica, apelação, agravo interno, petição inicial), o sistema sempre gera como se fosse uma petição inicial. A estrutura, tom e conteúdo específico de cada tipo são ignorados.
Causa raiz: O buildSystemPromptV3 no Mac Mini não recebe ou não usa o tipo_peticao_id para customizar as instruções de geração. O modelo_skeleton contém as seções corretas por tipo, mas o prompt do agente não instrui o Claude a seguir essa estrutura. Corrigir: incluir no system prompt as instruções específicas do tipo de petição + usar as seções do modelo_skeleton como outline obrigatório.
🟡 Sumário não conectado ao seletor de tópicos do VibeLaw
P1 · Feature
O SectionNavV3 (sidebar de navegação) lista os headings H1/H2/H3/H4 da petição. O VibeLaw tem um seletor de tópicos. Esses dois não estão conectados — o usuário deveria poder clicar em um tópico no sumário e o VibeLaw já selecionar automaticamente aquele tópico para edição.
Solução: Adicionar ao SectionNavV3 a capacidade de emitir um evento ao clicar em um heading que propague para o VibeLawChatPanel definindo o tópico ativo. Mapear headings do editor com os tópicos (por slug/texto do título).
🟡 Drawer VibeLaw sem drag-resize
P1 · UX
O drawer do VibeLaw tem largura fixa de 320px. Em telas menores isso toma muito espaço do editor. Em telas grandes, o usuário pode querer mais espaço para o chat.
Solução: Mesmo padrão do Step 2 — borda drag-handle entre editor e chat. Mínimo 240px, máximo 50% da tela. Persistir no localStorage.
🟡 VibeLaw precisa lapidação completa de UX
P1 · UX
O chat do VibeLaw é funcional mas precisa de vários refinamentos: (a) indicadores visuais de qual trecho está sendo editado, (b) highlight verde temporário quando IA aplica mudança, (c) modo fila de prompts mais claro, (d) histórico de patches com revert mais intuitivo, (e) sugestões de prompt inspiradas no Lovable.
Itens para lapidar: (1) Streaming visual mais fluido — atualmente pode parecer que nada aconteceu. (2) Diff highlight: colorir verde por 3s o trecho recém editado pela IA. (3) Chips de sugestão na área de input (ex: "Fortalecer o argumento do dano moral", "Adicionar jurisprudência do STJ"). (4) Fila de prompts: progress visual tipo Lovable. (5) Aba histórico: mostrar antes/depois com botão de revert mais visível.
ℹ️ diffPatches ainda é stub — sem highlight visual
P2 · Bug
O useVibeLaw retorna diffPatches: [] fixo. A EF olivia-vibelaw retorna patches, mas o frontend não os aplica visualmente. O usuário não sabe o que a IA mudou.
Causa raiz: A integração entre a resposta da EF (patches) e o componente DiffHighlighter está incompleta. Implementar: parse da resposta SSE → extrair patches → aplicar highlight temporário (3s, verde) via TipTap Decoration API.

Step 4 — Conclusão: Issues

Downloads, escritório, atividade, formatação
🔴 Botão gerar relatório e baixar PDF não funcionam
P0 · Bug Crítico
Os botões de download PDF e de gerar/baixar relatório HTML ficam em estado de loading e nunca completam. Isso impossibilita o uso real do sistema.
Causa raiz: Investigar useOliviaExportV3 — possíveis causas: (1) EF olivia-criar-doc com timeout (Supabase EF timeout = 60s, geração de PDF via Puppeteer pode demorar mais), (2) Mac Mini não está acessível para geração do documento, (3) A URL do documento não está sendo salva corretamente em olivia_historico após geração, (4) Parsing da resposta da EF com erro silencioso. Verificar logs no Supabase dashboard da EF olivia-criar-doc.
🔴 Seletor de escritório advocacia da empresa ré não existe
P0 · Feature Faltando
O Step 4 deveria ter uma searchbox para pesquisar e vincular o escritório de advocacia que representa a empresa ré. Essa funcionalidade foi especificada no projeto original mas não foi criada.
Solução: Criar componente EscritorioSearchbox que use a RPC search_escritorios (já existe!). Ao não encontrar, mostrar botão "Novo escritório" que abre modal igual ao da página /empresas aba "escritórios". Vinculação não deve ser obrigatória. Salvar o escritorio_id em olivia_historico (verificar se campo existe, senão criar migration).
🔴 Botão "criar atividade" com modal errado
P0 · Bug
O botão de criar atividade no Step 4 abre um modal diferente do padrão da página /atividades. O modal atual deve ser excluído e substituído pelo modal correto de /atividades, com pre-fill automático dos dados do processo (numero_cnj, processo_id, tipo, advogado).
Solução: Localizar o componente modal de criar atividade na página /atividades (provavelmente em src/components/atividades/). Importá-lo no Step 4, remover o modal atual. Ao abrir, passar os dados do processo via props (processo_id, numero_cnj, advogado_id) para pré-preenchimento automático.
🔴 Formatação da petição no Step 4 diverge do Step 3
P0 · Bug
No Step 3, a petição editada no TipTap tem formatação rica (negrito no nome do cliente, headings, etc.). Quando a petição é "gerada" para o Step 4, a formatação é diferente — a fonte é Times New Roman fixa e o negrito some.
Causa raiz: A petição do Step 4 é gerada pelo Mac Mini com formatação hardcoded (Times New Roman, estrutura fixa). A petição editada no TipTap (Step 3) tem seu HTML salvo em olivia_historico.peticao_html, mas ao gerar o DOCX/PDF no Step 4, o sistema usa o template antigo em vez do HTML editado pelo usuário. Solução: garantir que o DOCX/PDF gerado use o peticao_html do historico (que reflete as edições do Step 3) e não um template separado.

Arquitetura: Como a IA Funciona

Fluxo de dados do prompt até a petição gerada
📤 Como as provas chegam até o Claude
1
Usuário faz upload no Step 1 → olivia-docs bucket (Supabase Storage)
2
Ao clicar "Iniciar Análise" → EF olivia-topicos invocada com doc_urls[]
3
Mac Mini S0: valida histórico → S1: busca URLs dos docs no banco
4
EF olivia-extrair-texto: OCR via Claude Vision em cada PDF/imagem → texto extraído
5
S2: RAG pgvector busca blocos relevantes ao contexto das provas (768-dim embedding)
6
S3: texto das provas + blocos RAG + contexto do processo montam o prompt do Claude
7
Claude Sonnet/Opus analisa e retorna JSON de tópicos → salvo em olivia_historico.topicos_json
📝 Como o score de chance de ganho deveria funcionar
1
Após análise de provas, revisor avalia o caso com o4-mini ou Sonnet
2
Critérios: força probatória (30%), jurisprudência favorável (25%), comportamento juiz (25%), solidez pedidos (20%)
3
Score 0-10 onde 10 = vitória praticamente garantida (nossa chance, não nota da defesa)
4
Jurimetria: buscar histórico de processos contra a mesma empresa e no mesmo juiz
5
Retorno: {score, breakdown, alertas[], recomendacoes[]}
🔴 Atualmente: score avalia qualidade da defesa da empresa ré. Precisa inverter a perspectiva.
⚡ Como o RAG seleciona blocos de texto
1
Query = resumo do caso (empresa, tipo de restrição, tipo de petição)
2
fn_rag_search_v2: gera embedding 768-dim da query via OpenAI text-embedding-3-small
3
Busca cosine similarity em rag_chunks.embedding_v2 → top-8 mais similares (threshold 0.45)
4
Filtro adicional: prioriza blocos do mesmo estado do processo (TJRS/TJSC/TJSP)
5
Blocos selecionados incluídos no prompt do Claude como "exemplos aprovados" e "argumentos testados"
