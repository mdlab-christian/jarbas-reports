8 Decisoes que o CTO Precisa Tomar

1
Mac Mini vs Alternativa — Onde roda a geracao pesada?
O Mac Mini M4 roda server.mjs na porta 19100 via Tailscale. Funciona bem mas o deploy e manual e a observabilidade e ruim. A alternativa seria mover para n8n ou cloud function sem timeout.
OPCAO A — Manter Mac Mini
+ Hardware dedicado, zero custo adicional, sem limite de tempo para geracao, Playwright disponivel para EPROC futuro

- Deploy via SSH manual, risco de queda sem auto-restart robusto, sem observabilidade nativa
OPCAO B — Migrar para servico cloud
+ Deploy automatizado, observabilidade nativa, escalabilidade

- Custo recorrente elevado para geracao longa (Railway, Fly.io, Modal), sem Playwright local, migracao complexa, timeout ainda pode ser problema
⭐ RECOMENDADO A — Manter Mac Mini, melhorar deploy com launchd robusto e health check melhorado. Custo zero, sem risco de migracao.
2
Consolidar server.mjs — Remover rotas legadas?
server.mjs tem 3 rotas redundantes: /olivia/gerar, /olivia/pipeline-v3 e /olivia/pipeline-v4. As duas primeiras sao legadas e podem ter comportamentos ligeiramente diferentes do v4, causando inconsistencias silenciosas.
OPCAO A — Remover rotas legadas agora
+ Codigo limpo, sem ambiguidade, uma unica fonte de verdade para o pipeline

- Risco se alguma EF ou codigo legado ainda chamar /gerar ou /pipeline-v3 (precisa auditoria antes)
OPCAO B — Manter com alias deprecation log
+ Seguranca — rotas legadas retornam 410 Gone com log de quem chamou

- Codigo mais verboso, problema pode ser mascarado por semanas antes de ser notado
🔀 HIBRIDO — Auditar todas as EFs e frontend para chamadas a /gerar e /pipeline-v3. Se zero chamadas encontradas: remover. Se encontradas: alias com log por 2 semanas, depois remover.
3
OCR — Edge Function ou Mac Mini?
Atualmente o OCR via Claude Vision e feito dentro da olivia-extrair-texto (EF), chamada internamente pela olivia-topicos. Para muitos documentos, isso pode exceder o timeout de 150s do Supabase.
OPCAO A — Mover OCR para Mac Mini
+ Sem timeout, Mac Mini aguenta N documentos, permite progresso granular (doc 1/5 processado), consistencia — toda IA pesada no mesmo lugar

- Maior latencia inicial (Tailscale), EF precisa esperar callback do Mac Mini ou usar polling
OPCAO B — Manter OCR na EF com limite de docs
+ Sem mudancas arquiteturais, resposta mais rapida para poucos docs

- Timeout em casos reais com 3+ documentos longos, experiencia inconsistente para o usuario, problema que vai reaparecer
⭐ RECOMENDADO A — Mover OCR para Mac Mini. EF apenas envia lista de URLs dos docs e aguarda callback via polling do banco. Resolve timeout definitivamente e unifica IA no Mac Mini.
4
Smart Blocks — JSON referencia + IA ou Mecanico puro como fallback?
O Smart Blocks original usava tags+blocos fixos (mecanico puro — zero alucinacao mas rigido). A proposta atual usa peticao_referencia JSON como andaime estruturado para guiar a IA com mais flexibilidade.
OPCAO A — JSON referencia + IA com juris vinculadas
+ Flexivel, peticoes com linguagem mais natural, cada tag tem juris verificadas no banco — sem alucinacao de precedentes

- Mais complexo de implementar, ainda ha risco residual se prompt nao for robusto o suficiente
OPCAO B — Smart Blocks mecanico puro como fallback
+ Zero alucinacao garantida, texto 100% controlado, confiavel para peticoes criticas

- Rigido demais, texto robotico, dificulta customizacao por tipo de peca, nao escala para novos tipos
🔀 HIBRIDO — JSON referencia + IA (principal) com Smart Blocks mecanico como fallback de seguranca. Se revisor pos-geracao detectar juris nao verificadas, exibir aviso e oferecer versao mecanica.
5
EPROC Integration — Implementar agora ou adiar?
A integracao EPROC permitiria enviar peticoes automaticamente apos geracao. Requer sub-agente Playwright na Cecilia ou no Mac Mini. Alta complexidade e fora do escopo de lapidacao do mecanismo.
OPCAO A — Implementar no olivia0804
+ Feature de alto valor, elimina passo manual do advogado

- Escopo explode, risco de atrasar toda a lapidacao, EPROC instavel (2FA, captchas), Cecilia ja tem 11 skills ocupadas
OPCAO B — Adiar para olivia0905
+ Foco total na lapidacao, sem risco de scope creep, EPROC pode ser projeto separado com PRD proprio

- Feature valiosa adiada mais um ciclo
⭐ RECOMENDADO B — Adiar para olivia0905. Adicionar campo eproc_numero_protocolo no banco agora (sem logica). Manter stub no Step 4 com "Em breve".
6
n8n no Fluxo — Usar para notificacoes/retry ou manter tudo em EF+Mac Mini?
n8n ja existe na infra (Railway). Poderia ser usado para notificacoes de conclusao via Telegram, retry de falhas do Mac Mini e webhook chains. Mas adiciona uma camada extra de complexidade.
OPCAO A — Usar n8n para notificacoes + retry
+ Retry automatico com backoff, Telegram nativo, observabilidade visual, sem codigo extra nas EFs

- Uma camada a mais para debugar, webhook chain pode falhar silenciosamente se n8n estiver fora
OPCAO B — Manter tudo em EF + Mac Mini
+ Simples, menos pontos de falha, stack uniforme

- Retry manual nas EFs, notificacoes precisam de codigo customizado, sem observabilidade visual
⭐ RECOMENDADO A — Usar n8n APENAS para notificacoes Telegram de conclusao e retry de falhas criticas do Mac Mini. Nao usar n8n no fluxo critico de geracao.
7
Retry Robusto — Re-chamar EF inteira ou reusar historico_id?
Atualmente, um erro no Step 2 (Mac Mini lento ou fora) leva o usuario de volta ao Step 1, perdendo todo o estado. Isso e frustrante e inaceitavel para o advogado.
OPCAO A — Retry com historico_id existente
+ Usuario nao perde estado, EF apenas rechama Mac Mini com historico_id ja criado, banco continua como source of truth

- Requer que Mac Mini seja idempotente (se historico_id existe, retoma de onde parou — nao reprocessa OCR)
OPCAO B — Re-chamar EF inteira (novo historico)
+ Simples de implementar, nao requer idempotencia

- Perde topicos aprovados, OCR refeito, usuario frustra, historicos orfaos no banco
⭐ RECOMENDADO A — Retry com historico_id. Criar RPC supabase fn_olivia_retry(historico_id) que rechama Mac Mini no status correto. Mac Mini deve verificar status antes de reprocessar.
8
HTML Report — EF separada (olivia-relatorio) ou dentro do Mac Mini?
Uma funcionalidade desejada e gerar um HTML report da peticao (com raciocinio, tags usadas, juris, style guide aplicado). Pode ser gerado no Mac Mini ao final do pipeline ou por uma EF separada.
OPCAO A — EF separada (olivia-relatorio)
+ Chamada on-demand, usuario pode pedir o relatorio quando quiser sem regerar a peticao, EF leve (so monta HTML a partir do banco)

- EF adicional para manter, precisa ter todos os dados no banco previamente
OPCAO B — Dentro do pipeline Mac Mini
+ Gerado automaticamente ao final, zero chamada extra

- Aumenta tempo do pipeline principal, HTML gerado mesmo quando usuario nao precisa, acoplado ao Mac Mini
⭐ RECOMENDADO A — EF leve olivia-relatorio chamada on-demand no Step 4. Dados ja estao no banco (historico_id). EF monta e salva o HTML no storage, retorna URL.

10 Regras Maximas para olivia0804

1
Todo estado no banco — olivia_historico e a unica source of truth. Nenhum estado critico vive apenas na memoria do Mac Mini ou em variaveis de session no frontend.
2
Payload minimo — EF envia ao Mac Mini apenas {historico_id, organizacao_id, modo}. Mac Mini busca tudo do banco. Zero duplicacao de dados no payload.
3
Retry sem recriar — Qualquer reprocessamento usa o historico_id existente. Mac Mini verifica status e retoma de onde parou. Nunca criar novo historico para retry.
4
Smart Blocks obrigatorio — peticao_referencia sempre gerada antes do modo gerar. Nenhuma geracao direta sem andaime estruturado.
5
Jurisprudencias verificadas — Todo precedente citado deve vir do banco (olivia_tag_jurisprudencias). Revisor pos-geracao valida. Se juris nao verificadas encontradas: flag no banco + aviso no frontend.
6
Style guide sempre lido — jarbas_style_guide do advogado e carregado no inicio de todo pipeline de geracao Mac Mini. Sem excecoes.
7
Sem rotas legadas — server.mjs consolida em rota unica. Auditar e remover /olivia/gerar e /olivia/pipeline-v3 antes do deploy.
8
OCR no Mac Mini — OCR pesado (Claude Vision) sai das EFs para o Mac Mini. EF apenas dispara e aguarda polling. Elimina timeout de 150s para casos reais.
9
Progresso granular — Mac Mini atualiza olivia_historico.status a cada fase: processando_ocr → gerando_topicos → revisando → gerando_peticao → revisando_peticao → concluido. Frontend exibe estado em tempo real via Supabase Realtime.
10
Obsidian de problemas — Qualquer issue de producao identificado e logado em /olivia-problemas no Obsidian com template padrao (data, descricao, impacto, resolucao, li-cao). Webhook do banco automatiza a criacao.
