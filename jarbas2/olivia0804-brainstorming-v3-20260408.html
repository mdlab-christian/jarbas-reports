Arquitetura Revisada (com consideracoes do CTO)

Principio: Frontend → EF → Backend ← Frontend
// Fluxo garantido: nada se perde Frontend dispara Edge Function (HTTP POST) → EF valida + persiste estado no Supabase (ANTES de processar) → EF despacha para Mac Mini (fire-and-forget) → Mac Mini le tudo do banco (DB-driven) → Mac Mini processa + atualiza banco (progresso + resultado) → Frontend puxa do banco (Realtime subscription) → ZERO perda de estado em qualquer falha
Decisao: Super Agente Unico (recomendado)

Por que um agente e melhor que 10:
• Sonnet 4.6 tem 200K tokens de contexto — 21K de input e trivial
• 1 chamada API vs 10+ = 3x mais barato ($0.10 vs $0.35)
• Narrativa mais coesa (ve o todo, nao fragmentos)
• Sem overhead de sincronizacao/merge
• Tempo similar: 30-60s (unico) vs 30-45s (paralelo + merge)
• Debug mais facil (1 prompt vs 10+)

Mas: manter infraestrutura multi-agent como fallback para peticoes muito longas (>15 topicos).

Decisao: Sem Agente Revisor

O sintetizador JA revisa:
• Recebe blocos de referencia + peticao gerada
• Compara contra template do modelo + style guide
• Corrige inconsistencias, melhora coesao
• Garante formatacao HTML correta

Economia: ~$0.08/peticao (era $0.05 por revisao × 1-3 tentativas)
Risco: Sem safety net. Mitigacao: score do advogado no Step 4 alimenta melhoria continua.

Sistema Legado (n8n + GAS) — como funcionava
n8n recebia via webhook:
1. processo_id + modelo.id → buscava dados (processo, advogado, cliente, empresa)
2. Montava campos_substituicao: 30+ campos formatados ({{advogado_nome}}, {{numero_processo}}, etc.)
3. Enviava payload pro Google Apps Script com: modelo.google_doc_id + blocos[{ordem, placeholder, google_doc_id}] + jurisprudencias[{placeholder, google_doc_id}]

GAS fazia substituicao em 2 passadas:
Passada 1: Para cada bloco (ordem DESC), encontrava {{placeholder}} no Google Doc template, deletava o placeholder, copiava o conteudo inteiro do bloco (preservando formatacao: paragrafos, tabelas, imagens, negrito).
Passada 2: Para cada jurisprudencia, mesma logica — encontrava {{juris_placeholder}}, substituia pelo texto formatado.

Insight chave: O sistema legado era 100% mecanico. Nenhuma IA. As tags determinavam EXATAMENTE quais blocos usar. O novo sistema replica isso, mas a IA auto-resolve as tags quando possivel.

VibeLaw Revisado: Edicao por Topico

Mudanca fundamental: VibeLaw edita TOPICO inteiro
Antes: Usuario seleciona trecho de texto → VibeLaw altera so o trecho
Agora: Usuario seleciona TOPICO no TOC → VibeLaw altera a SECAO inteira

Mecanismo:
1. TOC (SectionNavV3) extrai headings H2 do editor → cada H2 = 1 topico
2. Usuario clica no topico no TOC → seta sectionTarget = heading text
3. VibeLaw envia comando + secao inteira para olivia-vibelaw EF

Decisoes Pendentes (v3)

ND9
Super agente unico OU multi-agent por topico?
A) Super agente unico (recomendado)
1 chamada Claude, 3x mais barato, narrativa mais coesa, debug facil. Sonnet 4.6 com 200K context handles 21K input tranquilo.
B) Multi-agent (1 por topico)
10+ chamadas paralelas. Mais rapido (paralelo). Melhor para peticoes muito longas. 3x mais caro.
ND10
Blocos no payload: titulo + preview OU conteudo completo?
A) Conteudo completo (recomendado)
IA ve o bloco inteiro pra decidir se e adequado. Mais tokens, mas melhor resolucao. ~500-2000 chars por bloco × ~5-10 blocos por tag × ~8 tags = ~40K-160K chars. Cabe no context.
B) Titulo + preview (120 chars)
Mais leve. Menos tokens. Risco: titulo pode ser ambiguo, IA resolve errado.
ND11
Sintetizador como agente separado OU integrado no Gerador?
A) Agente separado
2 chamadas Claude. Gerador gera, Sintetizador refina. Maior qualidade. +$0.05-0.10 por peticao.
B) Integrado no Gerador (recomendado)
1 chamada Claude com instrucoes de coesao no prompt. Mais rapido, mais barato. Se qualidade insuficiente, adiciona sintetizador depois.
ND12
Google Embeddings: text-embedding-004 ou gemini-embedding?
A) text-embedding-004 (recomendado)
768-dim. Mesmo tamanho que OpenAI v2 atual. Migracao mais simples (mesma coluna). Proven, estavel.
B) gemini-embedding-exp
Mais novo. Potencialmente melhor. Experimental — pode mudar. Dimensoes podem ser diferentes.
