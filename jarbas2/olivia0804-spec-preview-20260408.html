Super Agente OlivIA — Arquitetura

Modo ANALISE

Sonnet 4.6 (padrao) / Opus 4.6 (avancado)

Step 2 — Analisa provas, resolve tags, gera topicos

  • Recebe: docs (document blocks), modelo_skeleton, tags+blocos, playbooks
  • Resolve tags automaticamente das provas
  • Gera perguntas para tags incertas
  • Output: JSON {analise, topicos[], tag_answers{}, perguntas[]}

Modo GERACAO

Opus 4.6 SEMPRE

Step 2→3 — Gera peticao topico a topico

  • Recebe: topicos aprovados, blocos resolvidos, juris com assunto
  • Gera cada topico como secao independente
  • Auto-revisa coerencia e completude
  • Output: JSON {topicos_gerados[], peticao_html, score}

Modo VIBELAW

Sonnet 4.6 (rapido)

Step 3 — Edita topico INTEIRO via slash command

  • 5 comandos: /reescrever /expandir /simplificar /formalizar /jurisprudencia
  • Recebe topico inteiro (H2 ate proximo H2)
  • Reescreve mantendo argumentos centrais
  • Output: JSON {topico_html_atualizado, explicacao}

Modo RELATORIO

Sonnet 4.6

Step 4 — Gera HTML report com 10 secoes

  • Recebe: historico completo, analise, peticao final
  • 10 secoes: resumo, provas, topicos, juris, score, etc.
  • Output: JSON {relatorio_html}
Thread_id Resolvido: olivia_historico.id = Thread

Anthropic API nao tem thread_id, mas nos nao precisamos. O banco E a memoria.
Cada call: EF monta contexto → le olivia_historico (state machine) → injeta no prompt → Claude API (stateless) → persiste resultado no DB → proximo step le do DB.
O agente nunca "lembra" entre calls. Recebe tudo que precisa via payload.

Playbooks On-Demand

Conceito: Zero Hardcode, Zero Deploy

Playbooks sao .md no Obsidian vault. Um registry.json mapeia caso → playbooks. Pipeline le do disco antes de cada geracao. Equipe edita, salva, proxima peticao ja usa.

Fluxo Tecnico
  1. Pipeline recebe caso (tipo_peticao, empresa_categoria)
  2. Le registry.json do vault
  3. Carrega playbooks "always" + matched rules
  4. Se modo avancado: carrega training examples
  5. Injeta no user message do prompt
Fluxo da Equipe
  1. Advogado abre playbook no Obsidian
  2. Edita regras, estrategias, exemplos
  3. Salva (Cmd+S)
  4. Proxima peticao gerada ja usa versao nova
  5. Zero restart. Zero deploy. Zero config.
Exemplo: playbook_registry.json
{ "always": ["playbook-regras-gerais.md", "playbook-formatacao.md"], "rules": [ { "match": {"tipo_peticao": "peticao_inicial", "empresa_categoria": "banco"}, "playbooks": ["playbook-negativa-banco.md", "playbook-dano-moral.md"] }, { "match": {"tipo_peticao": "replica"}, "playbooks": ["playbook-replica.md"] }, { "match": {"tipo_peticao": "apelacao"}, "playbooks": ["playbook-apelacao.md"] } ], "training": { "modo_avancado": true, "source": "~/olivia-vault/06_feedbacks-equipe/aprovados/", "max_examples": 3 } }

Workflow Completo Step-by-Step

S1

Step 1: Pure UI — Zero API calls

Zustand state only. User configura: processo, tipo peticao, modelo, versao, advogado, qualidade (padrao/avancado), documentos, instrucoes adicionais.
Pre-fetch: modelo_skeleton + tags_com_blocos via RPCs no mount.

S2

Step 2: Analise + Topicos — EF olivia-topicos + olivia-gerar

Mount (historicoId=null): EF olivia-topicos cria olivia_historico (PENDENTE) → persiste modelo_versao_id, modelo_skeleton → fire-and-forget Mac Mini (modo=analise).
Mac Mini: le historico do DB, busca docs do storage, carrega playbooks on-demand, chama Super Agente (modo=analise), persiste analise_estruturada + topicos + tag_answers.
Frontend: Realtime subscription → progresso_pct → mostra topicos + analise. DnD reorder. Perguntas de tags incertas.
"Aprovar": EF olivia-gerar valida topicos, monta peticao_referencia {modelo, secoes, blocos resolvidos, juris com assunto}, fire-and-forget Mac Mini (modo=geracao).

S3

Step 3: Editor + VibeLaw — EF olivia-vibelaw

Mac Mini (geracao): Opus 4.6 gera peticao topico a topico → JSON {topicos_gerados[]} → monta peticao_html → persiste no DB.
Frontend: TipTap editor carrega peticao_html. SectionNav mapeia topicos (headings H2). VibeLaw: user seleciona topico inteiro → slash command → EF olivia-vibelaw → topico reescrito (SSE streaming).
Autosave: 15s debounce → UPDATE peticao_html. forceSave() obrigatorio antes de Step 4.

S4

Step 4: Conclusao — EF olivia-relatorio + olivia-criar-doc

EF olivia-relatorio: Super Agente (modo=relatorio) → HTML report 10 secoes.
EF olivia-criar-doc: Mac Mini → DOCX (Calibri/Georgia per advogado) + PDF.
Score + feedback + registro atividade.

Riscos e Mitigacoes

System prompt muito grande
Com playbooks + vault + instrucoes, pode passar 10K tokens. Mitigacao: Playbooks no user message (nao system). Carregar so os relevantes (max 3). Teses so como fallback.
Opus 4.6 custo alto na geracao
~$0.50-1.00 por peticao. Mitigacao: Aceitar. Custo marginal vs valor (peticao bem feita = caso ganho = cliente satisfeito).
Re-embed obrigatorio (Google Embeddings)
Vector spaces incompativeis. Mitigacao: Nova coluna (nao substituir). Batch em background. ~$3-5 total. Switchover gradual.
VibeLaw reescrita potencialmente destrutiva
Editar topico inteiro pode perder detalhes. Mitigacao: Manter versao anterior (undo). Mostrar diff antes de aplicar. 1 topico por vez.
JSON parsing fragil (output do agente)
Claude pode retornar JSON malformado. Mitigacao: Schema validation no Mac Mini. Fallback: pedir correcao (1 retry). Sanitizar HTML (DOMPurify).
Escopo grande (50 tasks)
Risco de perder foco. Mitigacao: Wave por wave. Review a cada task. Investigacao ANTES de execucao. Vault como memoria persistente.
