Cron cecilia-captura-4xdia — falhando em todo disparo Jarbas 2.0

Verificar payload do cron e corrigir delivery.mode Adicionar: delivery: { mode: "announce", channel: "telegram" } Ou desabilitar se funcionalidade for coberta pelo Jarbas Mini.
⚡ Impacto: elimina 4 falhas/dia + possível custo de tokens por init

Jarbas Mini — 116 sessions acumuladas, idleReset de 30 dias Jarbas Mini

⚠️ P1 — Alta Prioridade (maior impacto de custo)
💓
Jarbas Mini — Heartbeat a cada 6h com Sonnet + 80k tokens Jarbas Mini
Este é provavelmente o maior gasto da rede. O Jarbas Mini acorda 4x/dia, carrega 80k tokens de contexto com Sonnet 4.6 e roda o HEARTBEAT.md. As verificações são simples (pinga grupos, checa tarefas presas) — isso não justifica Sonnet 4.6 com 80k de contexto 4x/dia.

Opção A: Reduzir para 12h ou 24h (Jarbas Mini é legado — não está em uso ativo)
Opção B: Adicionar um modelo específico para o heartbeat (Haiku)
Opção C: Reduzir contextTokens de 80k para 20k para o heartbeat
No openclaw.json do Jarbas Mini, alterar: agents.defaults.heartbeat.every: "24h" # de 6h para 24h agents.defaults.contextTokens: 30000 # de 80k para 30k agents.defaults.session.reset.idleMinutes: 1440 # 1 dia, não 30
💰 Economia estimada: ~60-75% do custo do Jarbas Mini (maior item)

Jarbas Mini — contextTokens de 80k é muito alto para agente legado Jarbas Mini
O Jarbas Mini é um agente legado com função limitada: rodar crons de monitoramento e responder em alguns grupos Telegram. Com contextTokens de 80k, cada interação carrega um sistema de cache muito maior do que o necessário.
Comparação: Jarbas 2.0 usa 50k e faz muito mais. Jarbas Mini com 20-30k seria suficiente para suas funções atuais.
agents.defaults.contextTokens: 20000 # suficiente para Jarbas Mini legado agents.defaults.compaction.mode: "auto" # ativar compaction automática
💰 Economia: reduz custo de cache e tokens processados em ~60%

🔧 agent-cost-monitor — Skill de Monitoramento de Custos

Skill que roda todo dia às 22h, coleta dados de custo de todos os agentes da rede, identifica anomalias e gera relatório com sugestões de otimização.

📋 Responsabilidades

  • Coletar sessões do dia de cada agente (Jarbas 2.0, Jarbas Mini, OlivIA, Cecília, Beto)
  • Estimar tokens consumidos por agente (sessões + crons + heartbeats)
  • Identificar anomalias: sessão grande, heartbeat muito frequente, cron falhando
  • Comparar com baseline (semana anterior) e calcular delta
  • Gerar HTML com breakdown completo + sugestões de otimização
  • Publicar em GitHub Pages + mandar link via Telegram
  • Se gasto > threshold ($30/dia) → alerta urgente

📁 Estrutura da Skill

~/.openclaw/workspace-christian/skills/agent-cost-monitor/ SKILL.md scripts/ collect-agent-stats.sh ← coleta sessões e configs de todos os agentes estimate-costs.py ← estima custo por modelo e tokens detect-anomalies.py ← detecta padrões anômalos generate-report.py ← gera HTML de relatório references/ pricing.json ← tabela de preços por modelo baseline.json ← baseline semanal para comparação

📊 Métricas Coletadas por Agente

  • Sessions do dia: count, total linhas, maior sessão
  • Crons executados: count ok / count error / modelos usados
  • Heartbeats: frequência, modelo, estimativa de tokens
  • contextTokens config: verificar se está calibrado
  • Estimativa de custo: baseado em tokens processados × preço do modelo

🚨 Alertas Automáticos

  • Sessão única com mais de 500k tokens → alerta amarelo
  • Cron falhando há mais de 1 dia → alerta vermelho
  • Heartbeat em Sonnet com frequência < 4h → alerta laranja
  • Gasto estimado do dia > $30 → alerta vermelho urgente
  • Gasto > 150% do baseline semanal → alerta laranja

💡 Sugestões de Otimização Automática

A skill analisa padrões e sugere ações concretas, como: "Jarbas Mini heartbeat 6h → reduzir para 24h = economia estimada $X/mês". Não executa automaticamente — sempre apresenta para aprovação do Christian.
