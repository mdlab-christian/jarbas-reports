Diretiva CTO — PROMPT-ORIGINAL #18

"Mudança radical: hoje gera tópico por tópico, sequencial. Quero:
— Primeiro gera o esqueleto, inspirado no modelo
— Ao invés de fazer por tópico, fazer por seções (junção de tópicos). Cada tag uma seção
— Ao invés de sequencial, fazer paralelo (Promise.all)
— Agrupar tópicos coerentes em seção. Ideal: gerar toda a petição em uma seção
— Final: passada de costura — ajusta transições, remove repetição, normaliza numeração"

Detalhamento técnico — TG-05 (Opção B)

Algoritmo de Agrupamento

Regex aplicada sobre o título do tópico. Regra de fallback: qualquer tópico sem match cai em fundamentacao.

// groupBySections(topicos): Section[] const REGRAS = [ { nome: 'preliminares', re: /PRELIMINAR|ILEGITIMID|COMPETÊNCIA |NULIDADE/i }, { nome: 'fundamentacao', re: /FUNDAMENT|MÉRITO|DANO|RESPONSABILID |DIREITO|PROVA|NEGATIVAÇ/i }, { nome: 'pedidos', re: /PEDIDO|REQUER|TUTELA|LIMINAR/i } ] // Default: fundamentacao // Cada grupo → 1 call Claude
Paralelismo + Costura

pMap com concurrency 3 roda os grupos simultâneos. Passada de costura final com Haiku (~10s, barato).

// Paralelo const secoes = await pMap( groups, sec => gerarSecao(sec, context), { concurrency: 3 } ) // Passada de costura (Haiku) const final = await passadaCostura(secoes) // Instrução: ajustar transições abruptas, // repetição de fatos, numeração, // tom uniforme. NÃO reescrever conteúdo. // Logging [G7-TG05] secao=fundamentacao topicos=4
Estimativa por execução típica (6 tópicos): 3 grupos x ~25s paralelos + 10s costura = ~70s total. Tokens: +15% por contexto maior por seção. Custo: ~$0.45 (+18% vs TG-04).

Perguntas pendentes para o CTO

Q-01
Agrupamento por regex vs. Claude: a regra de agrupamento por regex de título é suficiente, ou prefere que Claude faça o agrupamento automaticamente (mais flexível, mas custa +1 call por geração)?
Regex: rápido, zero custo extra, determinístico. Claude-agrupador: flexível para tópicos com títulos incomuns, +~$0.02 e +~5s por geração.
Q-02
Modelo da costura: Haiku para a passada de costura final (barato, ~10s) ou Sonnet (melhor qualidade de ajuste de transições, ~+$0.05)?
Haiku recomendado: a costura não reescreve conteúdo, apenas ajusta transições e numeração — tarefa simples que Haiku resolve bem.
Q-03
TG-06 shadow mode: quer ativar TG-06 em shadow mode paralelo ao TG-05 desde o início, ou aguardar TG-05 estável antes de testar TG-06?
Rodar ambos em shadow ao mesmo tempo gera dados mais rápido, mas duplica o custo do shadow mode (~3 chamadas por geração vs. 2).
Q-04
Score floor para promoção: se TG-05 der score médio abaixo de 8.0 no shadow mode, rollback imediato para TG-04 como padrão definitivo?
Proposta: gate de promoção = TG-05 score >= TG-04 score em 10 gerações consecutivas. Se não atingir, manter TG-04 e investigar regressão antes de nova tentativa.
