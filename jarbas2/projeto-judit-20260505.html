Decisões do CTO (D1–D12)

Seção 02
12 decisões necessárias antes do Auto-Mode. Opção recomendada indicada com borda verde e tag. Após aprovação: registrar D1=X, D2=Y etc.
D1 — Tracking vs Polling por Demanda

Como monitorar processos: tracking Judit ou polling próprio?

Judit tem 2 formas: (a) POST /tracking cria monitor contínuo verificado diariamente, mas entrega é por EMAIL sem webhook; (b) Nós fazemos GET /responses com nosso próprio scheduler 3x/dia.
AUsar tracking da Judit (POST tracking.prod.judit.io) + polling /responses para buscar novidades
BIgnorar tracking, fazer POST /requests → poll → GET /responses direto por CNJ quando quisermosRecomendado
CHíbrido — tracking para prioritários, polling direto para bootstrap
Rec. B — tracking não tem callback_url (só email), precisamos fazer polling de qualquer forma. Tracking é custo extra sem ganho real.
D2 — Distribuição por lote: quantos CNJs por ciclo?

Tamanho do lote de polling por ciclo (7.658 processos ativos)

Rate limit: 500 req/min. Ciclo 3x/dia. Com 500 por ciclo e 3 ciclos/dia = 1.500 checados/dia, cobertura total em ~5 dias.
A500 por ciclo (mais antigos sem monitoramento primeiro) — 500 req / 500 req_min = 1 min de execuçãoRecomendado
B200 por ciclo (mais conservador, 600/dia, cobertura em ~13 dias)
CTodos os 7.658 em ciclo único (pode exceder rate limit)
Rec. A — cobertura completa em ~5 dias, sem risco de throttling.
D3 — Campo judit_last_step_id: onde guardar?

Armazenamento do step_id para detecção de novidades

Precisamos saber se o processo teve movimentação nova comparando step_id anterior vs atual a cada ciclo.
AGuardar em processos.judit_last_step_id (nova coluna via migration) — O(1) lookup, sem JOINRecomendado
BGuardar em cecilia_movimentacoes_brutas (buscar mais recente por CNJ — requer ORDER BY + LIMIT)
CTabela separada judit_tracking_state
Rec. A — Migration mínima, leitura O(1), mais simples de manter.
D4 — Processos FINALIZADO na Judit: ação automática?

O que fazer quando Judit retorna status='FINALIZADO'

response_data.status pode indicar processo encerrado. Alterar status automaticamente é crítico — pode afetar ~5.800 processos do Midas.
AUPDATE processos SET status='concluido' automaticamente
BSó registrar em judit_phase + log, sem alterar status (requer confirmação humana via /controller)Recomendado
CCriar atividade no MdFlow alertando advogado para conferir
Rec. B — Status de processo é crítico. Confirmação humana por ora. Auto-update pode vir em v2 após validação.
D5 — Worker C: remover navegação EPROC completamente?

Destino do Playwright no Worker C

Worker C hoje usa Playwright para navegar EPROC e baixar documento. Com Judit, documento vem via last_step.content (texto limpo, não HTML).
ARemover navegação EPROC completamente — SKIP_EPROC_NAVIGATION=true permanenteRecomendado
BManter como fallback opcional (se last_step.content vazio → tenta EPROC)
CRemover Playwright mas manter adapters de navegação (código morto)
Rec. A — Playwright era o gargalo e fonte de falhas. last_step.content está sempre preenchido. Limpeza é saúde do código.
D6 — Download de attachments (PDFs): quando baixar?

Estratégia para PDFs via GET /attachments/attachment-access

Endpoint retornou 404 em teste com step sem PDF. Precisa de confirmação com equipe Judit antes de implementar.
AImplementar já no Worker C — se attachments[] não vazio → baixar PDF e usar em vez de last_step.content
BImplementar depois (v2), por ora usar sempre last_step.content — suficiente para análise IARecomendado
CNunca baixar PDFs, texto sempre suficiente
Rec. B — Endpoint 404 em teste, precisa confirmação Judit. last_step.content suficiente. Attachments = feature v2.
D7 — Migrate cecilia_jobs: adicionar RLS? P0

RLS em cecilia_jobs (atualmente ausente — P0)

cecilia_jobs não tem RLS. Tabela interna do pipeline, não exposta ao frontend. Workers B/C usam service_role.
AAdicionar RLS modelo A (organizacao_id = get_org_id()) + índice partial — custo baixo, previne bugs multi-tenantRecomendado
BDeixar sem RLS (tabela interna, acesso via service_role nos workers)
CAdicionar RLS mas só para leitura (SELECT)
Rec. A — Workers usam service_role, não são impactados. RLS é custo mínimo de implementação.
D8 — Tracking delivery: como saber quais processos movimentaram hoje?

Estratégia de descoberta de novidades em lote

Judit não tem callback webhook. /responses/find-many sem UUID retorna erro. Precisamos de estratégia de polling eficiente.
A+BFila própria por judit_monitored_at ASC NULLS FIRST + confirmar com Judit endpoint batch antes de fechar v1Recomendado
BSó confirmar com Judit endpoint batch "movimentações hoje" — aguardar resposta antes de implementar
CPOST /requests em paralelo para todos os 500 CNJs do lote sem esperar resposta Judit
Rec. A+B — Usar A como estratégia principal. Perguntar B à equipe Judit antes de fechar v1.
D9 — Renomear user Mac Mini de "cecilia" para "Cecilia - Server"?

Quando fazer o rename do user UID 506

User atual UID 506 é "cecilia". LaunchAgents, scripts, paths todos referenciam esse user. Renomeação afeta tudo simultaneamente.
ARenomear agora junto com a implementação do ingestor
BRenomear depois — operação de alto risco, planejar separado com checklist completoRecomendado
CCriar user novo "cecilia-server" (UID 507), migrar gradualmente
Rec. B — Não misturar com implementação do ingestor. Planejar separado com checklist.
D10 — Bootstrap: registrar os 200 CNJs de teste agora?

Sequência correta para o bootstrap dos 200 CNJs selecionados

Selecionamos os 200 CNJs mais recentes (TJRS: 137, TJSC: 26, TJSP: 19). Precisamos fazer POST /requests para cada um e comparar last_step_id.
ACriar script de bootstrap agora e rodar nos 200 imediatamente
B+CMigrations primeiro → testar 5 CNJs manualmente → rodar os 200 (sequência que evita retrabalho)Recomendado
CTestar manualmente com 5 CNJs primeiro, depois escalar para os 200
Rec. B+C — Primeiro migrations (judit_last_step_id em processos), depois 5 manuais para validar, depois os 200.
D11 — PB6 (navegação EPROC): arquivar ou deletar?

Destino do playbook PB6-navegacao-eproc-v14.md

PB6 em cecilia-vault é o playbook de navegação EPROC. Com Judit, é obsoleto para o Worker C.
AArquivar PB6 (renomear para PB6-OBSOLETO-v14.md) + criar PB6-v2 com instruções de uso de dados JuditRecomendado
BDeletar PB6 permanentemente
CManter PB6 como fallback (Worker C continua tentando EPROC se Judit falhar)
Rec. A — Arquivar, não deletar. PB6-v2 documenta uso de last_step.content e payload_bruto para Worker C.
D12 — lawyers[] vazio no TJSC: impede detecção de polo?

Estratégia para detectarPolo() quando lawyers[] retorna vazio

No teste com TJSC, lawyers[] retornou vazio para todas as partes. Pode ser limitação do plano de teste, do TJSC ou da API key.
C+AConfirmar com Judit primeiro + implementar fallback por nome do cliente (buscar em processos) como segurançaRecomendado
BAceitar polo='desconhecido' para TJSC (26 processos) por ora, investigar depois
CConfirmar com Judit se lawyers[] é limitação do plano antes de implementar
Rec. C+A — Confirmar com Judit primeiro. Implementar fallback por nome como segurança independente da resposta.

ALTER TABLE processos ADD COLUMN IF NOT EXISTS judit_last_step_id TEXT, ADD COLUMN IF NOT EXISTS judit_monitored_at TIMESTAMPTZ, ADD COLUMN IF NOT EXISTS judit_phase TEXT, ADD COLUMN IF NOT EXISTS judit_judge TEXT; CREATE INDEX IF NOT EXISTS idx_processos_judit_monitored ON processos(judit_monitored_at NULLS FIRST) WHERE deleted_at IS NULL AND status = 'ativo';

ALTER TABLE cecilia_movimentacoes_brutas ADD COLUMN IF NOT EXISTS judit_step_id TEXT, ADD COLUMN IF NOT EXISTS judit_request_id TEXT, ADD COLUMN IF NOT EXISTS fonte TEXT DEFAULT 'eproc'; CREATE UNIQUE INDEX IF NOT EXISTS idx_cmb_judit_step_id ON cecilia_movimentacoes_brutas(judit_step_id) WHERE judit_step_id IS NOT NULL;
