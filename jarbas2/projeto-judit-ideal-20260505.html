Decisao de Arquitetura: Mac Mini Scripts (nao n8n)

Por que NAO n8n
  • ✗ Polling assíncrono (POST → wait → GET) é antinatural em nos n8n
  • ✗ n8n Railway = servico externo adicional como ponto de falha
  • ✗ Code Nodes perdidos no painel, difíceis de versionar
  • ✗ n8n é ideal para multi-servico orquestrado — aqui o fluxo e linear
Por que Mac Mini Scripts
  • ✓ Workers B e C já rodam no Mac Mini — ingestor fica junto
  • ✓ Scripts versionados no git, nao Code Nodes perdidos
  • ✓ Menos pontos de falha (offline tolerante)
  • ✓ LaunchAgent nativo macOS — mesmo padrao da Cecília
✅
Conclusao: novo script worker-judit-ingestor.mjs no Mac Mini, user cecilia, LaunchAgent com.cecilia.worker-judit.
