Problema Resolvido

A Cecilia capturava movimentacoes pelo EPROC usando a sessao do advogado, que so enxerga o polo do advogado. O polo adversario era invisivel, causando 68.2% de falsos "movimento_em_duvida".

⛔
Root causeSessao EPROC do advogado so exibe eventos do polo dele. Intimacoes do polo adversario nao aparecem, tornando a analise de contexto impossivel.
Solucao

A API Judit acessa o tribunal diretamente (nao como advogado), obtendo dados completos de ambos os polos, com campos semanticos estruturados em JSON.

✅
O que mudaWorker A (scraping) substituido. Workers B/C mantidos sem alteracao. Frontend /controller sem nenhuma mudanca. Schema intimacoes intacto.

Worker Judit Ingestor — Pseudocodigo Completo

Baseado em worker-a/index.mjs + scrape-eproc.mjs. Substitui completamente o Worker A. Sem browser, sem TOTP, sem Playwright.

// worker-judit-ingestor.mjs
// Baseado em: worker-a/index.mjs + scrape-eproc.mjs
// Substitui: browser-pool + circuit-breaker + adapters EPROC

import { supabase, ORG_ID } from '../../lib/supabase-client.mjs';
const JUDIT_KEY   = process.env.JUDIT_API_KEY;
const BASE        = 'https://requests.prod.judit.io';
const NOSSAS_OABS = ['RS116571', 'RS131163', 'SC074025', 'SC075466', 'SP535101'];

// 1. Buscar CNJs para monitorar (menos recentemente consultados primeiro)
const { data: processos } = await supabase
  .from('processos')
  .select('id, numero_cnj, judit_last_step_id')
  .eq('status', 'ativo')
  .is('deleted_at', null)
  .not('numero_cnj', 'is', null)
  .order('judit_monitored_at', { ascending: true, nullsFirst: true })
  .limit(500);

// 2. Processar em lotes de 50 (rate limit: ~50 req/s)
for (const lote of chunks(processos, 50)) {
  await Promise.allSettled(lote.map(async (proc) => {

    // Criar request na Judit
    const { request_id } = await juditPost('/requests', {
      search: { search_type: 'lawsuit_cnj', search_key: proc.numero_cnj }
    });

    // Poll ate completed (max 30s, intervals de 2s)
    const status = await pollUntilCompleted(request_id, 30_000);
    if (status !== 'completed') return; // timeout ou failed: pular

    // Buscar resultado fresco (cached:false = dado direto do tribunal)
    const responses = await juditGet(`/responses?page=1&request_id=${request_id}`);
    const fresh = responses.page_data?.find(r => !r.cached);
    if (!fresh) return;

    const rd   = fresh.response_data;
    const step = rd.last_step;
    if (!step) return;

    // Sem novidade — so atualizar timestamp de verificacao
    if (step.step_id === proc.judit_last_step_id) {
      await supabase.from('processos')
        .update({ judit_monitored_at: new Date() })
        .eq('id', proc.id);
      return;
    }

    // ★ NOVA MOVIMENTACAO — inserir na fila de analise ★
    const { data: mov } = await supabase
      .from('cecilia_movimentacoes_brutas')
      .insert({
        organizacao_id:    ORG_ID,
        processo_cnj:      rd.code,
        processo_id:       proc.id,
        evento_titulo:     step.step_type,        // semantico agora
        evento_data:       step.step_date,
        documento_raw:     step.content,          // texto limpo do evento
        evento_polo:       detectarPolo(rd.parties),
        tribunal:          rd.tribunal_acronym,
        judit_step_id:     step.step_id,
        judit_request_id:  request_id,
        fonte:             'judit',
        payload_bruto:     JSON.stringify({
          judge: rd.judge, phase: rd.phase, status: rd.status,
          amount: rd.amount, parties: rd.parties, attachments: rd.attachments,
        }),
        detectado_em: new Date(),
      })
      .select('id').single();

    // Criar job para Worker B processar
    await supabase.from('cecilia_jobs').insert({
      fase: 'B', status: 'pending', priority: 5,
      payload: { movimentacao_ids: [mov.id] },
    });

    // Marcar processo como verificado com o novo step_id
    await supabase.from('processos').update({
      judit_last_step_id: step.step_id,
      judit_monitored_at: new Date(),
    }).eq('id', proc.id);

  }));

  await sleep(1_000); // 1s entre lotes — respeita ~50 req/s
}

// Detecta em qual polo o escritorio esta via OAB dos advogados
function detectarPolo(parties) {
  const nosso = parties?.find(p =>
    p.lawyers?.some(l => NOSSAS_OABS.includes(l.oab))
  );
  if (!nosso) return 'desconhecido'; // fallback se lawyers[] vazio
  return nosso.side === 'Active' ? 'autor' : 'reu';
}

Migrations Supabase

Apenas additive — zero breaking changes. Nenhuma coluna existente e alterada ou removida.

-- Migration: 20260505_judit_integration
-- Todos os campos sao additive. Nenhum campo existente e alterado.

-- 1. Rastrear origem Judit nas movimentacoes brutas
ALTER TABLE cecilia_movimentacoes_brutas
  ADD COLUMN IF NOT EXISTS judit_step_id     TEXT,
  ADD COLUMN IF NOT EXISTS judit_request_id  TEXT,
  ADD COLUMN IF NOT EXISTS fonte             TEXT DEFAULT 'eproc';

-- Garante unicidade: cada step_id da Judit inserido apenas uma vez
CREATE UNIQUE INDEX IF NOT EXISTS idx_cmb_judit_step_id
  ON cecilia_movimentacoes_brutas(judit_step_id)
  WHERE judit_step_id IS NOT NULL;

-- 2. Rastrear estado do monitoramento por processo
ALTER TABLE processos
  ADD COLUMN IF NOT EXISTS judit_last_step_id  TEXT,
  ADD COLUMN IF NOT EXISTS judit_monitored_at  TIMESTAMPTZ;

-- Indice para rotacao fair: menos recentemente verificados primeiro
-- Permite ORDER BY judit_monitored_at NULLS FIRST no ingestor
CREATE INDEX IF NOT EXISTS idx_processos_judit_mon
  ON processos(judit_monitored_at NULLS FIRST)
  WHERE deleted_at IS NULL
    AND status = 'ativo';

Incertezas — Confirmar com Equipe Judit

01
Busca em Lote por Movimentacoes do Dia
Existe endpoint para buscar todos os processos que tiveram movimentacao em uma data especifica (ex: hoje)? Ou precisamos consultar processo a processo via /responses de cada CNJ individualmente? Um endpoint de batch reduziria de 8.485 chamadas para potencialmente 1-10 por rodada.

distribution_date_gte em Trackings Antigos
Nos testes notamos que novos trackings tem distribution_date_gte automatico. Para os 13.498 CNJs ja monitorados historicamente, como garantir que o ingestor receba todo o historico e nao so as novas movimentacoes a partir de hoje?
03
Endpoint de Attachments — Plano e Parametros
O endpoint GET /attachments/attachment-access retornou 404 no teste com step sem documento. Em qual plano o acesso a attachments esta disponivel? Quais step_types geram attachments na pratica (sentenca, acordao, outros)?
04
Campo lawyers[] Vazio — Plano ou Comportamento Padrao?
No processo Danilo x Santander, o array parties[0].lawyers veio vazio. Isso e limitacao do plano de teste, ou alguns processos realmente nao tem OAB cadastrada? A deteccao de polo via detectarPolo() depende deste campo.
