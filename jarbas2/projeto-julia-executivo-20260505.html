Sumário Executivo

5 destaques que MAIS importam
⚡ Veredito R8 (Revisão Arquitetura — GATE)

⚠️ Pode seguir com ajustes. A arquitetura aguenta o escopo MVP (500 leads/dia). Backend já está 80% pronto. Stack runtime canônica e gratuita. State machine validada em campo. 4 P0 destrancáveis em 1-2 dias úteis. Nenhum risco força replanejar.

🟢 Backend Supabase 80% pronto

R7 mostra: 4 das 6 tabelas julia_* já existem em produção. 10 RPCs fn_julia_* ativas. empresas.processavel já existe (915 empresas, 429 TRUE). Faltam apenas 5 tabelas + 3 colunas em leads + 5 RPCs. Migration: ~30s a 1min.

🟢 6 RPCs Sprint 2 já em produção (R4)

fn_julia_processa_inbound, fn_julia_registra_envio, fn_julia_registra_inbound, fn_julia_metrics, fn_julia_followup_candidatos, fn_julia_marcar_optout. State machine 12 fases + system prompt v2 (8322 chars) validados em campo Wati.

🔴 4 P0 destrancáveis em Fase 0 (~1 dia)
  1. Janela 24h Meta API — registrar 6 templates HSM (~6h Meta)
  2. Credenciais hardcoded n8n julia-beta — rotacionar Anthropic agora
  3. Schema drift vendedor_atribuido → na verdade é responsavel_id
  4. Cutover Wati → mdzap sem playbook — usar shadow → canary → full + 30d standby
🟡 Schema drift crítico

Briefing canoniza vendedor_atribuido. R3 mostra que essa coluna NÃO existe — frontend usa atendente_id (FK users). R7 propõe reusar responsavel_id (Opção A, zero migration). Sem decisão CTO, frontend e Julia escrevem em campos diferentes. Ver D-arq-15.

🟡 Janela 24h Meta API (P0 sem solução)

Toda mensagem outbound após 24h sem inbound do lead requer template HSM aprovado pela Meta. Sem templates aprovados = follow-up de procuração morre silenciosamente. ~6h trabalho Meta + 15min/template × 6 templates. Ver D-arq-13.

🟢 Custo IA -53% com prompt caching

System prompt v2 (8322 chars) × 500 chamadas/dia = 4.1M tokens cached/dia. Anthropic cache: 90% desconto leitura. Custo real: ~$130/mês ao invés de $270. Whisper.cpp local Mac Mini → -$27/mês. Total $290 → $130 efetivo.

O que a Julia FAZ — 13 capacidades MVP

Pipeline lead-to-cash automatizado

Pipeline completo do lead, do inbound mdzap até a entrega na coluna aguardando_procuracoes com follow-up automático. Decisões A1-M5 do briefing aprovadas pelo CTO.

1
Recebe inbound mdzap
Edge Function mdzap-webhook → Cloudflare Tunnel → Mac Mini Express. Número 5551816855848 (sdr_leads), Julia escuta com ai_enabled=TRUE em julia_config.
2
Identifica CPF na mensagem
Regex parser extrai CPF do texto/áudio. Se não tiver, Julia pede educadamente.
3
Consulta APIFULL (POST /api/scpc-boavista)
Reaproveita parser do n8n julia-beta (R4 §3.4) — 6 tipos de restrição. Token APIFULL movido para .env, não hardcoded.
4
Extrai PEND_FINANCEIRAS
Foco do procedimento (Decisão C1). Outras categorias (CCF, protestos, falências) ficam apenas na classificação interna alta/média/baixa.
5
Categoriza empresas
Cartão, banco, telefonia, etc. Reusa julia_categoria_mapping (já populada). Lookup contra empresas.categoria (já existe, 486/915 preenchidas).
6
Classifica processáveis vs não-processáveis
Cruza com empresas.processavel (BOOLEAN, já existe — 429 TRUE / 486 FALSE). 100% das empresas já classificadas (R7 §2.2).
7
CRIA lead na tabela leads
Via RPC nova fn_julia_criar_lead (R7 §6.1) — idempotente por whatsapp + organizacao_id. Preenche nome, cpf, whatsapp, vendedor (responsavel_id), endereço completo do APIFULL, coluna_crm = 'novos', memoria_julia = {}.
8
Gera PDF de auditoria
SOMENTE pend_financeiras (Decisão C1, C2). UI bonita. LibreOffice headless ou Playwright. Storage bucket auditorias-pdf.
9
Envia PDF + texto + lista 2 docs
PDF como anexo Meta Cloud API. Pede RG/CNH + comprovante residência na mesma mensagem (Decisão C4). Persona "Vinicius da Midas" (não revela ser IA).
10
Move 'novos → em_atendimento'
Via RPC nova fn_julia_mover_coluna (R7 §6.2). Registra leads_interacoes audit. Trigger trg_lead_historico_update já existente popula leads.historico JSONB.
11
Follow-up estados crm
Em em_atendimento, aguardando_documentos, aguardando_procuracoes. pg_cron 3h chama fn_julia_followup_candidatos (já existe). 3 tentativas: leve, gratuidade, último contato.
12
Persona dinâmica (vendedor_slug)
Julia se passa pelo vendedor vinculado ao lead (Decisão F1-F5). Hoje: Vinicius. Amanhã: multi-vendedor (Gustavo, etc) desde dia 1. Persona em julia-vault/05_personas/vendedores/{slug}.md.
13
Audio handling (Whisper) + escala humana 6 triggers
Recebe áudio (transcreve com Whisper.cpp local — D-arq-5=b), NUNCA responde em áudio. Cache em julia_audio_transcricoes. Escala automática se: (1) 3 falhas consecutivas, (2) APIFULL erro, (3) lead pediu humano, (4) Haiku detecta sentiment irritado, (5) fora scope, (6) palavra-chave perigosa.
