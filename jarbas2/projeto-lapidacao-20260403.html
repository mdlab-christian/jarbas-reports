Plano de Lapidacao — Item por Item

Manter

Fluxo 4 fases: spec → break → plan → execute

O core do workflow esta correto. E exatamente o que funciona no video e no SDD existente.

Manter

Armazenamento em .projeto/ (arquivos locais)

Zero dependencia de banco. Portavel. Funciona offline. Superior ao gerenciador-projetos nesse aspecto.

Manter

Anti-hallucination rules + "O que NAO Fazer"

Essencial. O Christian ja sofreu com IA mexendo onde nao devia. As regras anti-alucinacao sao inviolaveis.

Manter

/projeto status + /projeto reset

Comandos de consulta e limpeza. Simples, uteis, sem mudanca necessaria.

Ajustar

M1: /projeto spec — absorver CLAUDE.md + docs existentes

Problema: A spec v1 detecta stack genericamente. Mas o Christian SEMPRE tem CLAUDE.md com 700+ linhas de convencoes.
Melhoria: Na fase spec, LER o CLAUDE.md inteiro e EXTRAIR: arquitetura, design system, convencoes de nomenclatura, anti-patterns, stack exato. Se o projeto tem docs/ com specs existentes, ABSORVER e referenciar em vez de reescrever do zero. O template spec.md ganha secao "Convencoes Absorvidas" que lista as regras do CLAUDE.md que se aplicam.

Ajustar

M2: /projeto break — categorias por camada (backend/frontend/n8n/infra)

Problema: As categorias v1 sao genericas (infra/ui/functional/polish). Mas 95% dos projetos do Christian seguem a arquitetura 3+1 pilares: Frontend, Backend (Supabase), Automacoes (n8n), JARBAS (opcional).
Melhoria: Categorias alinham com as camadas reais: backend (migrations, RPCs, RLS), frontend (componentes, hooks, pages), n8n (webhooks, cenarios), infra (tipos, configs). Ordem de execucao: backend → frontend → n8n → polish. Isso espelha a Secao 12 (6 fases) do CLAUDE.md.

Ajustar

M3: /projeto execute — commit automatico + push em main

Problema: O execute v1 nao commita. Mas o Christian SEMPRE commita apos cada tarefa e faz push direto em main (regra do CLAUDE.md secao 24).
Melhoria: Apos execucao bem-sucedida: git add . → git commit -m "tipo(escopo): descricao" → git push origin main. Tipo do commit inferido da categoria da tarefa (fix, feat, refactor, chore). Escopo do commit = pagina/modulo afetado.

Ajustar

M4: /projeto plan — integrar skills especializadas por camada

Problema: O plan v1 so busca no codebase. Mas o Christian tem skills especializadas por camada: /mdflow-backend, /mdflow-frontend, /mdflow-rpc, /mdflow-migration, /mdflow-n8n.
Melhoria: O plano referencia qual skill usar na execucao. Se a tarefa e backend → plano diz "usar /mdflow-backend" ou "usar /mdflow-migration". Se frontend → "seguir /mdflow-frontend". O execute entao carrega a skill relevante automaticamente. Mas isso so se aplica quando o projeto tem skills disponiveis — se nao tem, segue o plan generico.

Novo

A1: /projeto html — gerar HTML visual do estado do projeto

Motivo: O Christian revisa via HTML publicado no GitHub Pages, nao lendo .md no terminal. O gerenciador-projetos ja tinha esse comando e funcionava bem.
Implementacao: Gera HTML com identidade visual Jarbas 2.0 mostrando: pipeline visual, progresso por camada, lista de tarefas com status, spec resumida, links para plans/logs. Publica em ~/jarbas-reports/jarbas2/projeto-{nome}-YYYYMMDD.html.

Novo

A2: /projeto next — atalho para proxima tarefa disponivel

Motivo: 80% do uso sera: "qual a proxima?" → plan → execute. O /projeto next analisa dependencias, encontra a proxima tarefa pendente com deps concluidas, e roda plan+execute em sequencia.
Implementacao: Le tasks.json, encontra proxima pendente com deps OK, pergunta "Planejo e executo T0X?" — se sim, roda plan e depois execute na mesma sessao (para tarefas S/M). Para tarefas L, recomenda /clear entre plan e execute.

Novo

A3: /projeto validate — validacao pos-execucao

Motivo: O Christian valida com browser (Playwright), queries Supabase, e screenshots. O gerenciador-projetos tinha tipo_validacao (vision/e2e/n8n-review/manual) que funcionava bem.
Implementacao: Apos execute, rodar validacao automatica: build check obrigatorio, query Supabase se tarefa backend, screenshot via agent-browser se tarefa frontend. Marcar validated=true no tasks.json. Se falhar → status="failed", gerar fix task automaticamente.

Remover

R1: Principio "Zero dependencias externas"

Motivo: Era bom para ser generico, mas o Christian TEM Supabase MCP, n8n MCP, agent-browser, 21st.dev. Impedir o uso dessas ferramentas e um handicap desnecessario.
Substituir por: "Usar ferramentas disponiveis no ambiente. Se MCP Supabase existe → verificar schema real. Se agent-browser existe → validar visualmente. Nao EXIGIR nenhuma, mas USAR se presente."

Remover

R2: Regra "NUNCA referenciar ferramentas especificas (Supabase, n8n)"

Motivo: Se o CLAUDE.md do projeto menciona Supabase e n8n → a skill DEVE referenciar. Ignorar o contexto do projeto para ser "generico" eh contra-produtivo.
Substituir por: "Adaptar ao contexto detectado. Se CLAUDE.md menciona Supabase → usar SQL MCP para verificar schema. Se menciona n8n → verificar webhooks. Se nao menciona nada → ser generico."
