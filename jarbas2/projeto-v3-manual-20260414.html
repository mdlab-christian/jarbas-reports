O que e e por que usar

§0

O /projeto e uma skill do Claude Code que transforma uma ideia em codigo profissional seguindo um workflow estruturado de 5 fases. Sem vibe-coding. Sem "vai implementando e ve o que da".

Por que usar
  • Context window e finito — cada fase roda numa sessao limpa com /clear entre elas.
  • Comunicacao entre fases = arquivos em disco (nunca memoria volatil da sessao).
  • 1 tarefa = 1 unidade atomica — sem escopo rastejante.
  • Explora o codigo existente antes de planejar — zero alucinacao sobre o que ja existe.
  • Absorve CLAUDE.md automaticamente: design system, stack, anti-patterns, autonomia.
  • Funciona em qualquer projeto: generico se nao tiver CLAUDE.md, rico se tiver.
Filosofia de sessoes
# /clear entre fases mantem o context window limpo = melhor output /projeto spec --> /clear --> /projeto break --> /clear --> loop: /projeto plan + execute + verify

Guia Passo a Passo — Do Zero ao Codigo

§3
F1

spec — Especificacao

/projeto meuapp spec "landing page com formulario de contato"
  • 1Detecta a stack automaticamente: package.json, pyproject.toml, Cargo.toml, etc.
  • 2Le o CLAUDE.md inteiro e absorve: design system, stack, anti-patterns, regras de autonomia.
  • 3Absorve docs existentes na pasta docs/ — nenhum contexto perdido.
  • 4Se Supabase MCP conectado: verifica schema das tabelas relevantes antes de escrever qualquer coisa.
  • 5Se a descricao for curta, pergunta: objetivo, paginas, comportamentos, dependencias externas.
  • 6Gera spec.md com a secao ## Flows — crucial para dependencias automaticas no break.
  • 7Inicializa projeto.json com metadata da stack e fase atual.
O que sao Flows (secao ## Flows no spec.md)

Flows capturam os workflows do usuario conectando operacoes em sequencia. O break usa esses Flows para determinar depends_on automaticamente — sem chutes.

## Flows ### Flow: Enviar formulario de contato Usuario preenche e envia o formulario, recebe confirmacao por email 1. visualizar formulario -- landing-page componente-formulario 2. preencher campos -- landing-page B01-validar-campos 3. enviar -- landing-page B02-submeter-formulario 4. receber confirmacao -- landing-page B03-mostrar-sucesso
Output gerado
.projeto/meuapp/spec.md .projeto/meuapp/projeto.json
Proximo passo
Revisar spec.md /clear → /projeto meuapp break
F2

break — Decomposicao em Tarefas

/projeto meuapp break   ·   /projeto meuapp break --sync
  • 1Le spec.md inteiro — nenhum contexto assumido de sessoes anteriores.
  • 2Extrai os Flows → determina depends_on entre tarefas automaticamente.
  • 3Gera tarefas por camada na ordem obrigatoria: backend → frontend → n8n → polish.
  • 4Cada tarefa tem: id, title, category, complexity (S/M/L), depends_on, files, skill_hint, acceptance_criteria.
  • 5Gera tasks.md (legivel) + tasks.json (machine-readable).
  • 6Cria arquivo auto-contido tasks/T01.md para cada tarefa com AC embutido.
  • 7Com --sync: cria GitHub Issue para cada tarefa via gh issue create e salva o github_id.
Regras de granularidade
S
1–2 arquivos
M
3–5 arquivos
L
5+ → dividir

1 tarefa = 1 comportamento (nao 1 pagina inteira). Backend sempre antes do frontend que o consome.

Output gerado
tasks.md + tasks.json tasks/T01.md, T02.md...
Proximo passo
Revisar tarefas /clear → /projeto meuapp plan T01
F3

plan — Planejamento Tecnico

/projeto meuapp plan T01
  • 1Le tasks/T01.md — tarefa e acceptance criteria.
  • 2Le secao relevante do spec.md.
  • 3Verifica dependencias — avisa se alguma nao foi concluida antes de continuar.
  • 4Exploracao READ-ONLY (obrigatoria): navega pelos arquivos da tarefa, le codigo existente, busca patterns similares ja implementados. Proibido modificar qualquer arquivo nesta etapa.
  • 5Se Supabase MCP disponivel: verifica se tabela/RPC ja existe antes de planejar migration.
  • 6Codex pre-review: envia arquivos ao o4-mini com 8 checks: TypeScript any, hex hardcoded, console.log, arquivos >500 linhas, organizacao_id, deleted_at, codigo Frankenstein, impacto em outras areas.
  • 7Atualiza tasks/T01.md in-place com plano completo: Codex Pre-Review + Exploracao + Spec Funcional + Spec Tecnica + O que NAO fazer + Commit previsto.
📄
O arquivo tasks/T01.md se torna auto-contido apos o plan. O execute so precisa ler esse unico arquivo — nenhum contexto adicional necessario.
Output gerado
tasks/T01.md (atualizado) plans/task-T01.md (backup)
Proximo passo
Revisar plano /clear → /projeto meuapp execute T01
F4

execute — Implementacao

/projeto meuapp execute T01   ·   /projeto meuapp execute T01 --worktree
  • 1git pull --rebase obrigatorio — garante codigo atualizado em ambiente multi-maquina.
  • 2Le tasks/T01.md — documento auto-contido com tudo que precisa.
  • 3Carrega a skill referenciada em skill_hint se existir.
  • 4Executa passo a passo da Especificacao Tecnica sem desviar do plano.
  • 5Build check apos cada arquivo (npm run build) — falha cedo, corrige cedo.
  • 6Se backend + Supabase MCP: aplica migration via MCP (nao via SQL direto).
  • 7Codex post-review: mesmo checklist do plan. Verifica que nenhum anti-pattern foi introduzido.
  • 8Gera logs/task-T01-result.md com o que foi feito e arquivos modificados.
  • 9Atualiza tasks/T01.md, tasks.md, tasks.json e projeto.json com status completed.
  • 10Commit + push: tipo(escopo): descricao da tarefa. Se github_id: comenta na Issue.
Com --worktree
  • Cria .worktrees/meuapp-T01/ com branch propria para a tarefa.
  • Executa tudo no worktree isolado — main branch fica intocada durante o execute.
  • Ao finalizar: merge em main, apaga branch e worktree automaticamente.
Output gerado
codigo implementado commit + logs/task-T01-result.md
Proximo passo
Sem /clear necessario /projeto meuapp verify T01
F5

verify — Verificacao por Acceptance Criteria

/projeto meuapp verify T01
  • 1Le Acceptance Criteria de tasks/T01.md.
  • 2Verifica se playwright-cli esta disponivel: npx @playwright/cli@latest --version.
  • 3Verifica se dev server esta rodando em localhost (porta 5173 ou configurada).
  • 4Para cada Acceptance Criterion: abre browser → snapshot → interacao → snapshot → verifica resultado esperado.
  • 5Exibe relatorio com PASS/FAIL e evidencia de cada criterio.
  • 6Atualiza tasks/T01.md com status dos criterios.
  • 7Se github_id existe e todos PASS: gh issue close automaticamente.
[PASS] Formulario renderiza com 3 campos obrigatorios [PASS] Botao desabilitado quando campos vazios [FAIL] Email de confirmacao — esperado: toast verde, encontrado: nada Resultado: 2/3 criterios passaram.
ℹ️
Se playwright-cli nao estiver disponivel: cai no validate padrao (build + schema + RLS).

Comandos Especiais

§4
/projeto ajuda — Help no terminal

Exibe a tabela completa de todos os comandos formatada no terminal. Use quando esquecer qualquer comando. Nao abre browser, roda direto no Claude Code.

/projeto <nome> next — Atalho do dia a dia

Encontra a proxima tarefa com dependencias satisfeitas e executa plan + execute automaticamente.

  • Para tarefas S e M: faz plan + execute na mesma sessao.
  • Para tarefas L: executa o plan e recomenda /clear antes do execute.
  • Prioridade automatica: backend → frontend → n8n → polish.
/projeto <nome> style — Atualizar tema (tweakcn)

Para projetos React/shadcn/ui: aplica design tokens do tweakcn.com no globals.css com merge inteligente.

  1. Acesse tweakcn.com, configure seu tema, copie o CSS gerado.
  2. Rode /projeto meuapp style.
  3. Cole o CSS quando solicitado.
  4. Merge respeita a ordem: imports → dark → @theme → :root → .dark → @layer base.

Estrutura de Arquivos

§5
.projeto/ meuapp/ ├─ projeto.json metadata: stack, fase atual, progresso X/Y ├─ spec.md especificacao completa com secao ## Flows ├─ tasks.md lista legivel com status e Acceptance Criteria ├─ tasks.json machine-readable: todos os campos de cada tarefa ├─ tasks/ │ ├─ T01.md arquivo auto-contido: │ │ desc + AC + plano embutido │ │ (Functional Spec + Technical Spec │ │ + Codex review + exploracao) │ └─ T02.md ├─ plans/ backup dos planos (compatibilidade legado) │ └─ task-T01.md └─ logs/ └─ task-T01-result.md o que foi feito, arquivos modificados, Codex post .worktrees/ git worktrees por tarefa (apenas com --worktree) meuapp-T01/ branch isolada durante o execute
💡
Todos os arquivos em .projeto/ devem ser commitados junto com o codigo. Eles sao a memoria duravel do projeto — necessarios para continuar em outra sessao ou outra maquina.

Multi-projeto — Trabalhando em Paralelo

§6

A skill suporta multiplos projetos simultaneos em terminais diferentes. Cada projeto tem seu estado isolado em .projeto/<nome>/.

Terminal 1
$ /projeto olivia execute T05
Terminal 2
$ /projeto controller plan T03
Terminal 3
$ /projeto financeiro break
Regras
  • Nome SEMPRE obrigatorio quando ha 2+ projetos ativos.
  • Pode abreviar: olivia funciona para olivia-0504 se for o unico com esse prefixo.
  • Se omitir nome: a skill para e lista os projetos disponiveis — nao adivinha.
Multi-maquina — Pro + Air + Mini
  • git pull --rebase roda automaticamente antes de spec e execute.
  • Se conflito detectado: PARA e reporta. Nunca continua com codigo desatualizado.
  • /projeto pull — atalho manual para sync quando necessario.
✅

Boas Praticas

§7

DO — Faca

  • Use /clear entre fases — context window limpo = output melhor.
  • Revise a spec.md antes do break — especialmente a secao ## Flows.
  • Revise o plano em tasks/T01.md antes do execute.
  • Para tarefas L: divida antes de continuar.
  • Se Codex pre-review retornar "REFATORAR PRIMEIRO": respeite antes de executar.
  • Use /projeto ajuda quando esquecer um comando.

DON'T — Evite

  • Nao pule o plan — o execute precisa do plano em tasks/T01.md.
  • Nao implemente alem do plano — sem "melhorias de passagem".
  • Nao ignore dependencias — se T03 depende de T01, conclua T01 primeiro.
  • Nao edite tasks/T01.md manualmente apos o plan.
  • Nao use git push --force — sempre commit normal.
⚠️
Regra de ouro: Se a spec esta ambigua, PARE e revise. A skill nao adivinha o comportamento — e nem deve.

Flags Disponiveis

§8
FlagUsado comO que faz
--sync break Cria GitHub Issue para cada tarefa via gh issue create. Salva github_id no arquivo da tarefa. Requer gh auth login configurado.
--worktree execute Executa em .worktrees/<nome>-<id>/ com branch propria. Merge automatico em main ao finalizar. Apaga branch e worktree. Util para isolamento total entre tarefas paralelas.
