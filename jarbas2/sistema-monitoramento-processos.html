2.5 Limitações da Arquitetura Atual

❌ Ponto Único de Falha

Tudo depende de 1 Mac Mini M4 rodando Playwright. Se ele travar, reiniciar ou perder conexão — zero captura naquele ciclo.

❌ Não Escalável

Adicionar nova OAB exige configuração manual complexa. Suportar outros escritórios seria impossível com a arquitetura atual.

❌ Sem API Pública

O MdFlow não tem como consumir os dados de monitoramento via API. Tudo é integração direta com Supabase — frágil e sem contrato.

❌ Cobertura Incompleta

Das 8 OABs, algumas ainda não passaram por teste E2E completo em todos os tribunais. Sistema parcialmente validado.

Nossas Necessidades

O que precisamos de uma API profissional de monitoramento processual

🎯
Regra máxima: O sistema de monitoramento precisa ter cobertura de 100% das intimações endereçadas às OABs monitoradas. Nenhuma intimação pode passar despercebida. Uma intimação perdida = prazo perdido = perda do processo.

📨 1. Monitoramento de Intimações

Capturar 100% das intimações enviadas para as OABs cadastradas, de todas as fontes:

  • EPROC dos tribunais monitorados — via login com OAB
  • DJEN — Diário de Justiça Eletrônico Nacional (obrigatório desde mai/2025)
  • DJEs estaduais — publicações nos diários eletrônicos
  • PJe (futura expansão)

⏰ 2. Cálculo Automático de Prazos

Para cada intimação capturada, o sistema deve:

  • Identificar o tipo de ato processual
  • Calcular o prazo em dias úteis (não corridos)
  • Considerar feriados nacionais e estaduais
  • Alertar quando prazo < 7 dias (urgente) e < 3 dias (crítico)
  • Respeitar as novas regras do DJEN (D+3 úteis)

📄 3. Download de Documentos

Para alimentar a OlivIA (nossa IA jurídica) com contexto completo:

  • Conteúdo completo de decisões e despachos
  • PDFs das peças processuais
  • Laudos periciais quando houver
  • Formato: PDF e/ou HTML extraído
  • Armazenamento: S3/R2 com URL de acesso

🤖 4. Análise por IA

Processamento automático de cada intimação:

  • Classificação de urgência (A-G)
  • Tipo de ação necessária (réplica, recurso, etc.)
  • Resumo executivo (3-5 linhas)
  • Prazo calculado + data de vencimento
  • Confiança da análise (0-100%)

🔌 5. API de Integração

O MdFlow precisa consumir os dados via API REST padronizada:

  • Autenticação JWT multi-tenant
  • Webhooks para notificação em tempo real
  • Endpoints documentados com OpenAPI/Swagger
  • Rate limits claros e previsíveis
  • SLA definido (>99,5% uptime)

📈 6. Escalabilidade

A API deve suportar crescimento:

  • 20-50 escritórios de advocacia
  • 100-500 OABs monitoradas
  • Multi-tenant com isolamento completo de dados
  • Configurável por cliente (frequência, filtros)
  • Dashboard de status por escritório

O Que o Desenvolvedor Precisa Saber

Problemas técnicos reais que descobrimos — cada um com evidências de produção

🔴
Leia com atenção: Esta seção contém as descobertas mais valiosas deste documento. São meses de tentativas, falhas e soluções. Cada item abaixo foi descoberto com código real em produção, não em teoria.

4.1 Por Que Monitorar EPROC É Difícil

Não existe API pública oficial para o EPROC. O sistema eletrônico dos tribunais brasileiros foi construído para uso humano via browser, não para integração com sistemas externos. Qualquer automação é feita de forma não-oficial, contra mecanismos de proteção implementados pelos próprios tribunais.

O EPROC usa Keycloak SSO (Single Sign-On) como autenticação — o mesmo sistema usado por grandes empresas de tecnologia — com autenticação em 2 fatores (TOTP). Isso já elimina qualquer solução simples de scraping.

Ao contrário de monitorar um site comercial, monitorar o EPROC envolve: autenticação com credenciais de advogado OAB + senha + código 2FA que muda a cada 30 segundos + seleção de perfil + navegação contra CSRF dinâmico em todas as URLs.

4.2 CSRF Hash — O Mecanismo Mais Crítico

⚠️ O EPROC Assina Cada URL com um Hash Único de Sessão

Esta foi a descoberta mais importante e mais difícil de detectar. O EPROC gera um hash CSRF único para CADA URL interna, vinculado à sessão ativa. Qualquer URL construída manualmente é rejeitada com "Link sem assinatura válida".

✅ Solução: Sempre extrair URLs diretamente do DOM após navegar para cada página. Nunca construir URLs com parâmetros fixos. O hash muda a cada sessão e não pode ser reutilizado. 
// ❌ FALHA: URL construída manualmente
page.goto('https://eproc.tjrs.jus.br/eprocV2/controlador.php?acao=relatorio_processo_procurador_listar')
// Resultado: "Link sem assinatura válida" → redirect para homepage

// ✅ FUNCIONA: URL extraída do DOM
const link = await page.$eval('a[href*="relatorio_processo_procurador_listar"]', el => el.href);
// link = 'https://eproc.tjrs.jus.br/eprocV2/controlador.php?acao=relatorio_processo_procurador_listar&hash=a3f8d2e1c0b9...'
await page.goto(link);  // Funciona ✅

4.3 TOTP 2FA — Autenticação de Dois Fatores

O EPROC usa RFC 6238 TOTP (Time-based One-Time Password) para segundo fator de autenticação. O código de 6 dígitos muda a cada 30 segundos e é gerado a partir de um segredo base32 fornecido no momento de ativação do 2FA.

O que isso significa para a API: Cada escritório precisa fornecer o segredo TOTP de cada OAB no momento do cadastro. Esses segredos devem ser armazenados com criptografia AES-256 no banco de dados da API.

// Implementação RFC 6238 sem dependências externas (zero deps)
function generateTOTP(secret_base32) {
  const secret = base32Decode(secret_base32);  // ex: 'HEZDAMZQGRRDQZBRMY4W...'
  const counter = Math.floor(Date.now() / 1000 / 30);  // janela de 30 segundos
  const hmac = createHmac('sha1', secret).update(toBuffer(counter)).digest();
  const offset = hmac[19] & 0xf;
  const otp = (
    (hmac[offset] & 0x7f) << 24 |
    (hmac[offset+1] & 0xff) << 16 |
    (hmac[offset+2] & 0xff) << 8  |
    (hmac[offset+3] & 0xff)
  ) % 1000000;
  return otp.toString().padStart(6, '0');
}

// CRÍTICO: aguardar janela segura (≥5s restantes)
const ms_restantes = 30000 - (Date.now() % 30000);
if (ms_restantes < 5000) {
  await sleep(ms_restantes + 100);  // aguardar nova janela
}
const code = generateTOTP(secret);

4.4 Mecanismos Anti-Bot Descobertos

Todos os itens abaixo foram descobertos com evidências reais de bloqueio em produção:

🤖 Sessões Simultâneas = Bloqueio

Fazer login com a mesma OAB em dois browsers ao mesmo tempo resulta em bloqueio imediato da sessão. O tribunal detecta e invalida ambas as sessões.

✅ Logins obrigatoriamente SERIAIS. Nunca paralelos para mesma OAB. Aguardar ciclo anterior completar antes de iniciar novo.

🔑 Hash Único por Processo

Cada vez que você navega para um processo específico, o EPROC gera um sessionHash único. Reutilizar o hash de uma navegação anterior resulta em acesso negado.

✅ Sempre navegar via links do DOM da sessão atual. Nunca salvar e reutilizar hashes de navegações anteriores.

⏱️ Timeout de 14 Minutos

A sessão EPROC expira após ~14 minutos de inatividade. Processos extensos ou lotes grandes causam expiração no meio da execução.

✅ Implementar renovação preventiva de sessão a cada 10-12 minutos. Salvar estado KC (session cache) para relogin rápido.

⌨️ React Inputs não Respondem a page.fill()

Os campos de login e senha são componentes React. O método page.fill() preenche visualmente mas o React não dispara eventos — ao submeter, o campo aparece vazio para o servidor.

✅ Usar keyboard.insertText() que simula digitação real, disparando todos os eventos React necessários.

📋 Delays Obrigatórios Entre Processos

Navegar entre processos muito rapidamente ativa rate limiting ou bloqueio temporário. Comportamento de bot detectável pela velocidade de navegação.

✅ Delays aleatórios de 800-2500ms entre cada processo. 5 minutos entre OABs diferentes. Máximo 1 sessão ativa por OAB.

📊 "Painel do Advogado" vs "Últimas Movimentações"

Usamos por meses o "Painel do Advogado" pensando que era a fonte correta. Ele retorna apenas ~22 intimações formais de meses atrás. As movimentações recentes ficam em outra seção.

✅ Sempre usar "Últimas Movimentações" (não o Painel). A URL interna contém o parâmetro ord_ultimas_movimentacoes=S.

4.5 Por Que HTTP Puro Não Funciona

Tentamos fazer requisições HTTP diretamente com cookies de sessão válidos exportados de uma sessão real. Falhou por 3 razões independentes:

  1. CSRF Hash: mesmo com cookies válidos, qualquer URL de ação retorna "Link sem assinatura" porque o hash é validado contra o estado JavaScript da sessão, não apenas os cookies.
  2. Keycloak State: o SSO mantém estado interno JavaScript. Requisições HTTP puras não reconstituem este estado, mesmo enviando todos os cookies corretos.
  3. Token Nonce: o fluxo OAuth2 do Keycloak usa parâmetros nonce e state únicos por sessão, processados pelo JavaScript da página durante o redirect. HTTP puro quebra este fluxo.

Conclusão definitiva: O EPROC requer um browser completo com motor JavaScript ativo. Não há atalho.

4.7 DJEN — Mudança Crítica de Maio 2025

⚡ Atenção: Nova Realidade de Intimações no Brasil

A partir de 16 de maio de 2025, o CNJ determinou que todas as intimações nos sistemas EPROC passem a ser veiculadas exclusivamente pelo DJEN — Diário de Justiça Eletrônico Nacional. Este é um ponto que muitos sistemas de monitoramento ainda não tratam corretamente.

Novo Ciclo de Prazos (Obrigatório DJEN):

D0
Tribunal envia ao DJEN
→
D1
DJEN disponibiliza
→
D2
DJEN publica
→
D3
Prazo começa a contar

Impacto: Escritórios que monitoram apenas o painel do EPROC podem estar calculando o início do prazo no dia errado. A intimação vai para o DJEN, e o prazo começa D+3 úteis após o envio — não no dia do envio. A API precisa monitorar AMBAS as fontes.

DJEN disponível em: https://comunica.pje.jus.br/

4.8 Datajud CNJ — Útil mas Limitado

✅ O Que Oferece

  • API pública e gratuita do CNJ
  • Cobertura de TODOS os tribunais do Brasil
  • Metadados da capa processual (partes, classe, datas)
  • Histórico de movimentações processuais
  • Ótimo para jurimetria e análise histórica

❌ O Que NÃO Oferece

  • Documentos completos (sem acesso a PDFs)
  • Tempo real — delay médio de 18,8 dias
  • Webhooks (só polling)
  • Intimações com prazo urgente
  • Conteúdo de decisões e despachos
⚠️
Datajud para monitoramento de prazos = inútil. Um delay de 18,8 dias significa que quando o Datajud te informa sobre uma intimação, o prazo provavelmente já está vencido ou prestes a vencer. Use Datajud apenas para enriquecimento histórico e jurimetria.

4.9 Como Grandes Players Realmente Acessam os Dados

Investigamos como empresas como Judit e JusBrasil conseguem monitorar 90+ tribunais com latência baixa. A resposta é uma combinação de:

  1. Acordos formais com tribunais — Alguns TJs fornecem feeds de dados diretamente para plataformas homologadas pelo CNJ. Requer aprovação judicial e processo burocrático (meses).
  2. Crawlers Playwright em escala — A mesma técnica que usamos, mas com dezenas de instâncias em cloud (AWS/GCP), IPs rotativos e gerenciamento de sessões em Redis distribuído.
  3. Processamento de Diários Eletrônicos — Download e parse dos DJEs estaduais (PDFs publicados diariamente). Mais lento mas cobre casos onde o login não é possível.
  4. DJEN Nacional — Feed do Domicílio Judicial Eletrônico Nacional para intimações formais (desde mai/2025).
  5. Datajud CNJ — Para metadados históricos e reconciliação.

Nossa vantagem: Para EPROC dos tribunais cobertos, a abordagem via Playwright com login OAB é a única que fornece acesso completo incluindo documentos, em tempo real. Já temos esse código funcionando.

6.4 Requisitos Técnicos Críticos

⚙️ Workers (Crítico)

  • 1 instância Playwright Chrome por OAB — nunca compartilhar sessão
  • Logins SERIAIS entre OABs da mesma organização
  • Circuit breaker: desativar OAB após 5 falhas consecutivas
  • Session cache: salvar estado Keycloak por 25 min (evitar re-login)
  • Delays humanizados: 800-2500ms entre processos
  • 5 minutos mínimo entre ciclos da mesma OAB
  • Auto-healing: screenshot + Vision IA para estados inesperados

🔐 Segurança (Crítico)

  • TOTP secrets cifrados com AES-256 em repouso
  • Senhas nunca em plaintext — cifradas ou hashed
  • Multi-tenant: cada cliente só vê seus próprios dados (RLS)
  • JWT com expiração curta (1h) + refresh tokens
  • LGPD: logs sem CPF/dados pessoais, retenção 90 dias
  • API Keys com escopos (leitura, escrita, monitoramento)

📊 Disponibilidade

  • 99,5% uptime mínimo por mês
  • Alerta via Telegram se OAB falhar 2+ ciclos consecutivos
  • Health check endpoint para monitoring externo (Uptime Robot)
  • Logs estruturados (JSON) para debugging
  • Retry com backoff exponencial para falhas temporárias
  • Rollback automático se nova versão causar regressão

💾 Banco de Dados

  • PostgreSQL com multi-tenancy via organizacao_id
  • Deduplicação: signature_hash (SHA256 de CNJ+evento+data)
  • Soft delete obrigatório (deleted_at)
  • Índices em: organizacao_id, oab, cnj, data_intimacao, urgencia
  • Documentos: S3/R2/MinIO com URL com expiração
  • Audit log para todas as operações sensíveis
