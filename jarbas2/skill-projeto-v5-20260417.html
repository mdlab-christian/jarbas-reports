1. Por que este plano

O que funcionou em Olivia 1604
  • 103/103 tasks executadas em 6 waves, G1-G6 todos verdes
  • $29 reais vs $500 de budget (margem 17x)
  • Autônomo noturno: Pro liga, dorme, acorda com merge na main
  • Estrutura de pastas clara + PROMPT-ORIGINAL preservado
  • DECISOES-PENDENTES em formato contexto + A/B/C + recomendação + por quê
  • SUBPROJETOS com prompts copy-paste self-contained
  • AUTO MODE com retry loop + deep review + gate check + merge
O que falta generalizar
  • Workflow ficou amarrado ao projeto Olivia
  • Scripts orquestrar/check-gate/gerar-prompts não são templates
  • Não existe comando dedicado a cada fase
  • Sem runbook único para AUTO MODE (readme espalhado)
  • Lições não voltam para knowledge base do vault
  • Paralelismo hardcoded em 14 workers quebra em projetos pequenos
🎯 Objetivo
Transformar o ciclo que funcionou em Olivia em um template reutilizável da skill /projeto. CTO pensa em 4 fases, executa com 4 comandos, reutiliza infra provada, não re-inventa orquestração a cada projeto novo.

manual operacional

CL-1..N por wave + CLF-1..N final
🪵

logs por task
🏁

done/failed/blocked por task
🎬

FINAL-REPORT, ATTENTION, DECISIONS, timeline, waves/
Pipeline orquestrar.sh auto
preflight → cria tag baseline → loop(wave):
  LAUNCH (spawn workers em worktrees via claude -p headless)
  WAIT poll 60s
  CLASSIFY tasks (done/failed/blocked)
  RETRY LOOP max 3 (prompt enriquecido com stderr)
  DEEP REVIEW Opus
  BUDGET CHECK
  GATE CHECK (G4 CORE sagrado = HALT obrigatório)
  MERGE --no-ff → push → cleanup worktrees
FINAL-REPORT.md sempre gerado
🛑 Condições de HALT
Gate G4 vermelho · merge conflict · >30% fail rate · budget >limite · timeout 6h por wave · review=abort
Flags
--dry-run  simula sem executar
--from WX  retoma a partir de wave X
--budget-usd 500  limite de custo
--max-retries 3  tentativas por task
--fail-pct 0.30  fail rate máximo
--poll-sec 60  intervalo de polling
--wave-timeout 21600  timeout em segundos
--max-parallel 8  workers por wave

Faz pós-mortem do AUTO-RUN, roda deep review (grep hardcodes, spec drift, arquivos >500L, E2E pending), gera plano de correções com N terminais paralelos e, se tudo verde, sugere tag GA.
⛔ Pré-requisito

Artefatos gerados

🔍

👋

Workflow
  1. Lê AUTO-RUN-*/FINAL-REPORT.md + ATTENTION.md
  2. Deep review via Task Explore: grep hardcodes, spec drift, arquivos >500L, E2E pending
  3. Gera post-mortem.md com métricas
  4. Gera 05-review-pos-auto.html via html-jarbas (seções: Resumo AUTO, Bugs P0/P1, Gaps, Drifts vs spec, Correções pendentes)
  5. Sugere CORRECOES-MANUAL-*.md com prompts copy-paste pra N terminais
  6. Atualiza PROMPT-ONBOARDING-TERMINAIS.md
  7. Se tudo verde: sugere tag GA

4. Padrão INVIOLÁVEL de apresentar decisões

Regra de ouro
Toda decisão (em DECISOES-PENDENTES.md ou em qualquer interação com CTO) segue o formato contexto + opções + recomendação + por quê. Violar esse padrão é violação do princípio P-01 da skill. 
### D-XX-YY — Título da decisão

**Subprojeto:** NN (se aplicável)
**Contexto:** (1-3 frases explicando o trade-off)

- **Opção A:** descrição
- **Opção B:** descrição ✅ **RECOMENDO**
- **Opção C:** descrição

**Por que B:** justificativa clara com referência a princípio/doc/custo/risco.
Princípio #1
Formato de decisão é inviolável. Sem recomendação explícita, sem decisão.
Princípio #2
Gate G4 CORE é sagrado. Vermelho = HALT obrigatório, sem exceção.
Princípio #3
Sempre criar tag baseline antes de AUTO mode. Rollback é git revert.

5. Decisões pendentes CTO

8 decisões · responder antes do AUTO
D-V5-01 Nome do comando da Fase 1
Precisamos de um comando projeto-level para iniciar. Já existe /projeto:plan task-level estabelecido — confundir ou não os dois?
AManter /projeto:plan e reutilizá-lo para projeto-level E task-level (flag --task T01)
BCriar novo /projeto:init para projeto-level, manter /projeto:plan só para task-levelRECOMENDO
CRenomear atual plan para plan-task e criar plan novo
Por que B: zero breakage do workflow atual. init é semanticamente claro (Fase 1 = iniciar projeto). plan continua específico de task. Devs descobrem init no /projeto:ajuda.
D-V5-02 Paralelismo máximo do AUTO mode
Olivia usou 14 workers em paralelo, mas isso assumiu Mac Pro M-series com RAM sobrando. Projeto menor quebra.
AHardcoded em 14 workers/wave (como Olivia 1604)
BConfigurável via flag --max-parallel N com default 8RECOMENDO
CAuto-detecta CPU (N/2)
Por que B: default 8 é seguro para a maior parte dos projetos e das máquinas. Flag permite tuning pra cima em projetos grandes (14) ou pra baixo em máquinas saturadas (4).
D-V5-03 Onde vivem templates reutilizáveis
Templates de scripts/pastas/prompts podem morar na skill local ou no vault sincronizado.
A~/.claude/skills/projeto/templates/ (dentro da skill)
B~/jarbas-vault/04_claude-code/templates/projeto/ (Obsidian-synced)
CAmbos, com skill symlinkando para vaultRECOMENDO
Por que C: vault é o cérebro (sync entre Macs via Obsidian Sync). Skill lê via symlink, edições propagam automaticamente. Opção A isola; B fica descoberto; C combina ambos.
D-V5-04 Deep review engine no AUTO mode
Deep review por wave consome token pesado. Opus 4.7 pega drift sutil; Sonnet 4.6 é barato mas pode deixar passar.
AClaude Opus 4.7 (Olivia usou, caro ~$3/wave)
BClaude Sonnet 4.6 (barato, talvez não pegue drift sutil)
CConfigurável via flag --review-model, default Opus 4.7RECOMENDO
Por que C: review é o único momento onde Opus vale a pena. Default Opus preserva qualidade Olivia. Flag permite downgrade pra Sonnet em projetos menores e baratos.
D-V5-05 Integração com sync-macs
Pro tem Opus e é MASTER. Air já está mapeado como worker. Mini é server.
AAUTO mode só no Pro (worktrees locais)
BAUTO mode pode spawnar workers no Air via SSHRECOMENDO
CAUTO mode usa Mac Mini como worker pool
Por que B: Air como worker expande paralelismo sem comprometer o MASTER. SSH já mapeado (ssh air). Mini é server (OlivIA+Cecília), não quer interferir. Implementação inicial pode ser flag opt-in --use-air.
D-V5-06 Gates configuráveis por projeto
Olivia tinha G1-G6 fixos, mas cada projeto tem validações próprias.
AGates fixos G1..G6 como Olivia 1604
BGates definidos no tasks.json do projeto, /projeto:init gera template de 3 gates defaultRECOMENDO
CSem gates, só checa build+tests
Por que B: template de 3 gates (build / tests / e2e) é baseline universal. tasks.json declara quais checks compõem cada gate. Projetos complexos podem adicionar mais gates; simples ficam com 3.
D-V5-07 Pré-req git para projeto novo
AUTO mode pode corromper trabalho se main estiver suja ou sem contexto.
AQualquer repo git serve
BSó rodar em repos com CLAUDE.md local + main branch limpoRECOMENDO
CPermissão explícita do CTO por projeto
Por que B: CLAUDE.md garante contexto absorvido. Main limpo garante baseline. Sem isso, /projeto:automode pode corromper trabalho. Check automático antes de iniciar qualquer fase.
D-V5-08 Ciclo de aprendizado entre projetos
Cada projeto gera lições. Onde elas devem viver pra reaproveitamento?
ACada projeto isolado
B/projeto:review escreve lições em ~/jarbas-vault/04_claude-code/licoes-projetos.mdRECOMENDO
CLições em self-improving/domains/
Por que B: lições de projetos são reference material (acessível a todos os terminais via vault sincronizado). self-improving é de regras globais; licoes-projetos.md é knowledge base específica.
