TREINAMENTOS TELEGRAM — protagonistas, rich
PRIMARY · alimenta RAG

Path canonical: ~/Documents/olivia-vault/memoria/treinamentos/telegram/YYYY-MM-DD/NNN-{slug}.md

Namespace "telegram": representa o sistema legado (não apenas o canal físico). MdFlow, MdOs e OpenClaw convergem para cá.

Identificação: YYYY-MM-DD-NNN com numeração 3 dígitos sequencial por dia.

Slug: derivado do título pela IA classifier, kebab-case, máx 60 chars.

Seções obrigatórias do arquivo MD
  • ## Contexto — situação, processo de origem, razão do treinamento
  • ## Treinamento (literal) — texto exato enviado pelo advogado/CTO
  • ## Classificação IA — tipo + categoria + subcategoria + score
  • ## Conteúdo Inserido — insights extraídos pela Sonnet 4.6
  • ## Vinculações — links Obsidian [[playbook--X]]
  • ## RAG Queries sugeridas — 3-7 queries para busca semântica

SECONDARY · NÃO-RAG

Modo: append-only — novos feedbacks acrescentados ao final do arquivo do dia.

Feedback Step 4 JAMAIS é promovido para treinamento sem passagem explícita pelo pipeline /treinar. Se um feedback contém insight de regra de negócio, o advogado deve criar um treinamento Telegram separado.

#!/usr/bin/env node // ~/Documents/olivia-vault/_scripts/migracao-136.mjs // Fase 5 - migra feedbacks/log/* -> memoria/treinamentos/telegram/YYYY-MM-DD/ import fs from 'node:fs/promises'; import path from 'node:path'; import matter from 'gray-matter'; import { slugify } from './lib/slug.mjs'; import { classify } from './lib/classifier.mjs'; // Sonnet 4.6 const VAULT = process.env.OLIVIA_VAULT; const SRC = path.join(VAULT, 'memoria/feedbacks/log'); const DST = path.join(VAULT, 'memoria/treinamentos/telegram'); const logs = (await fs.readdir(SRC)).filter(f => f.endsWith('.md')); const counters = {}; for (const file of logs) { const raw = await fs.readFile(path.join(SRC, file), 'utf8'); const { data: oldFm, content } = matter(raw); if (oldFm.status === 'removido') continue; const data = oldFm.data || extractDate(file); counters[data] = (counters[data] || 0) + 1; const nnn = String(counters[data]).padStart(3, '0'); const enriched = await classify(content, oldFm); const slug = slugify(enriched.titulo).slice(0, 60); const newFm = { id: `${data}-${nnn}`, data, hora: oldFm.hora || '00:00:00', autor: oldFm.autor || 'christian', fonte: oldFm.fonte || 'telegram', tipo: enriched.tipo, categoria: enriched.categoria, subcategoria: enriched.subcategoria || null, empresa_slug: enriched.empresa_slug || null, processo_ref: oldFm.processo_ref || null, advogado: oldFm.advogado || null, score_confianca: enriched.score, tags: enriched.tags, playbook_refs: enriched.playbook_refs || [], status: 'ativo' }; const newContent = matter.stringify(content, newFm); const targetDir = path.join(DST, data); await fs.mkdir(targetDir, { recursive: true }); const targetFile = path.join(targetDir, `${nnn}-${slug}.md`); await fs.writeFile(targetFile, newContent, 'utf8'); console.log(`OK ${file} -> ${data}/${nnn}-${slug}.md`); } console.log(`Migracao concluida: ${logs.length} logs processados`);

Lint automático — validação pré-absorção

Executado no pipeline /treinar entre safety-guard e classifier. Qualquer falha → fila pendente_aprovacao com notificação Telegram ao CTO.

1
Frontmatter schema JSON válido

Valida contra JSON Schema Draft-7. Chaves obrigatórias: id, data, hora, autor, fonte, tipo, categoria, status. Falha → pendente_aprovacao.

2
tipo ∈ 10 canonical (9 + decisao_favoravel)

Enum estrito. Qualquer outro → rejeitar.

3
categoria NOT NULL

Obrigatório ∈ {bancario, telecom, cessao, energia, lojas, orgao_restritivo, geral}. Fallback "geral" apenas quando score < 0.6.

4
conteúdo ≥ 100 caracteres

Seção "Treinamento (literal)" + "Conteúdo Inserido" somados ≥ 100 chars. Abaixo = provável teste → reject.

5
Sem duplicata semântica (threshold 0.85)

Embedding comparado com últimos 30 dias. Similaridade > 0.85 → conflict-detector decide (merge | reject | flag).

6
autor ∈ chat_ids autorizados

Telegram: christian, heloisa, leonardo, stefany. MdFlow/MdOs: JWT user.role ∈ {advogado_senior, admin}. Não autorizado → 403.

Mitigação de riscos — R12 + R17

R12 — Treinamentos não consolidados ALTA severidade

Risco: treinamentos fragmentados entre canais (MdFlow/MdOs/Telegram), sem visão unificada, alguns absorvidos sem lint, outros perdidos.

Mitigações ativas
  • ✓ Consolidação automática: 3 triggers → 1 vault path
  • ✓ Cron diário 02:00: rebuild RAG + dashboard
  • ✓ Auditoria semanal: alerta se < 5 treinamentos em dia útil
  • ✓ Métrica KPI: % vinculados a playbook — alvo > 70%
  • ✓ Lint bloqueante: falhas vão para fila, nunca se perdem
R17 — Vault dessync Mac Mini ↔ Pro MÉDIA severidade

Risco: edições concorrentes entre Mac Mini (master) e MacBook Pro (auditor) causando perda de dados ou merge conflicts silenciosos via Obsidian Sync.

Mitigações ativas
  • ✓ Regra imutável: Mac Mini = único master writer em memoria/treinamentos/
