Problemas Identificados
🧭
Sem orientação de progresso
Usuário não sabe quantas tags faltam, quais são obrigatórias ou se pode avançar. Nenhum indicador de "onde estou" no fluxo.
🖱️
Fluxo não guiado (discovery problem)
O usuário precisa clicar em cada tag à esquerda para descobrir os blocos disponíveis. Não há sequência natural de preenchimento.
📐
Orientação espacial difícil
Em modelos com 12+ tags e 50 blocos por tag, o usuário se perde entre as colunas. Carga cognitiva alta sem recompensa visual.
📱
Mobile degradado
2 colunas em tela estreita obrigam scroll horizontal ou colapso que elimina o contexto da tag selecionada.
Dados do Sistema
Tags por modelo
5 a 15 tags únicas, geradas pelo skeleton do modelo
Blocos por tag
Até 50 blocos por tag (média ~12 em produção)
Campos auto-preenchidos
44 campos vindos do processo (CNJ, partes, advogados, datas)
Tags obrigatórias
Definidas no schema do modelo — bloqueiam geração se vazias
Estado persistido
blocosSelecionados + camposPreenchidos + overridesManuais
Oportunidade: O fluxo sequencial é o comportamento esperado. O usuário naturalmente quer preencher uma tag de cada vez. O modo lista força descoberta manual onde deveria haver guia.

Plano de Implementação

4.1 — Arquivos a criar / modificar
Step2WizardMode.tsx NOVO
Componente principal do wizard. Recebe as mesmas props de Step2TagsBlocos. Gerencia currentTagIndex internamente. Renderiza TagProgressBar + Step2TagCard + BlocoGrid + WizardNav. Quando currentTagIndex >= placeholders.length, renderiza RevisionScreen.
step1: Step1Data sessionId: string blocosSelecionados: Record<string, string> camposPreenchidos: CampoData[] overridesManuais: Record<string, string> onChange: (tag, blocoId) => void onNext: () => void onBack: () => void
Step2TagCard.tsx NOVO
Card grande com stripe teal à esquerda. Exibe: "Tópico N de M — Nome da Tag", badge Obrigatório/Opcional, hint descritivo e código da tag em JetBrains Mono.
placeholder: PlaceholderItem totalTags: number currentIndex: number isRequired: boolean
TagProgressBar.tsx NOVO
Linha horizontal de pills. 4 estados: done (verde+check), current (teal+glow), warn (laranja+!), pending (cinza). Label com nome curto abaixo. Clicável para salto direto em qualquer posição.
placeholders: PlaceholderItem[] currentIndex: number blocosSelecionados: Record<string, string> onJump: (index: number) => void
Step2TagsBlocos.tsx MODIFICAR
Adicionar prop modo: 'lista' | 'wizard' com default 'wizard'. Adicionar seletor de modo (toggle pill) no topo do componente. Quando modo='wizard': renderiza Step2WizardMode. Quando modo='lista': renderiza layout atual sem nenhuma alteração. O estado do modo pode ser persistido em localStorage por usuário.
modo?: 'lista' | 'wizard' ← nova onModoChange?: (m: 'lista' | 'wizard') => void ← nova
BlocoCard.tsx REUSAR
Funciona perfeitamente para o wizard. Zero alteração necessária.
CamposRevisaoAccordion REUSAR
Usado no passo final sem modificação. Mesmas props.
handleSelecionarBloco REUSAR
Lógica de seleção idêntica — apenas quem chama muda.

4.3 — Ordem das Tags
Manter a ordem retornada pelo skeleton do modelo (não reordenar obrigatórias primeiro). O skeleton já retorna na sequência lógica do template. Reordenar quebraria a coerência textual da petição.

Se usuário pular tag opcional (card "Pular"): setCurrentTagIndex(i + 1) sem validar.
Se tag obrigatória: mostra toast de aviso mas permite avançar. Bloqueio real só no botão "Concluir Seleção".
4.4 — Persistência e Retorno ao Step 2
currentTagIndex é estado de UI puro — não persiste no banco via sessionId.

Ao voltar de Step 3 para Step 2: reinicia no índice 0. Alternativa: iniciar na primeira tag sem bloco selecionado (placeholders.findIndex(p => !blocosSelecionados[p.placeholder])) — mais amigável para edição.

Os blocos selecionados já persistem via sessionId — nenhuma regressão.
