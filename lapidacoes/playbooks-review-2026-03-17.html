🔴 P0 — Bloqueadores (3)
P0
TabPlaybooks: selectColumns com joins não garantidos podem crashar
TabPlaybooks.tsx → useServerPagination
O parâmetro selectColumns: '*, olivia_tipos_peticao(id, nome), empresas_categorias(id, nome), documentos_tipos(id, nome)' faz joins que assumem foreign keys válidas. Se qualquer linha tiver FKs inválidas ou NULL, o response pode omitir itens silenciosamente ou lançar erro de PostgREST. A tabela olivia_playbooks pode não ter todos esses vínculos obrigatórios.
✏️ Fix
Usar selectColumns: '*, olivia_tipos_peticao!left(id, nome), empresas_categorias!left(id, nome), documentos_tipos!left(id, nome)' (LEFT JOIN explícito para evitar exclusão de rows sem FK).
P0
ModalBloco wizard: tipo do bloco NÃO é persistido no banco
ModalBloco.tsx → handleSubmit → useBlocos mutation
O wizard Step 1 coleta tipo (ex: fundamento, conclusao), mas o handleSubmit NÃO inclui tipo no payload enviado ao hook criarBloco/atualizarBloco. O campo tipo é coletado mas descartado silenciosamente. Resultado: todos os blocos ficam sem tipo no banco.
✏️ Fix — ModalBloco.tsx handleSubmit
const payload = {
  titulo: titulo.trim(),
  conteudo: conteudo.trim(),
  tipo,                          // ← ADICIONAR ESTA LINHA
  tag_ids: tagIds,
  categoria_empresa_id: categoriaEmpresaId || null,
  status: 'draft' as const,
  arquivo_original_path: uploadedPath,
  juris_ids: selectedJurisIds.length > 0 ? selectedJurisIds : null,
};

Verificar também se useBlocos / olivia_blocos_texto tem coluna tipo — se não tiver, precisa de migration.

P0
TabModelosPeticao: filtro de categoria é client-side sobre 1 página
TabModelosPeticao.tsx → displayModelos useMemo
A query de modelos usa pageSize: 25 com filtros server-side para status e tipo_peticao_id, mas o filtro de categoria_tipo (que vem da tabela joinada olivia_tipos_peticao) é aplicado client-side depois de receber apenas 25 registros. Se a organização tiver >25 modelos de outras categorias, os modelos da categoria selecionada podem não aparecer.
✏️ Fix
Mover filterCategoria para o servidor via filters, adicionando uma query especial que filtre pela categoria da tabela join. Alternativa: aumentar pageSize quando filterCategoria !== 'todas' ou usar uma view no Supabase que flattene a categoria.

🟡 P1 — Importantes (5)
P1
ModalBloco Step 3: WizardStepper entra antes do registry carregar
ModalBloco.tsx → useEffect step 3
Quando o usuário avança para o Step 3, o useEffect que detecta placeholders roda imediatamente. Porém a query olivia_placeholders_registry só é ativada (enabled: step === 3) nesse momento. O useEffect de detecção também depende de registry, mas o estado valid é calculado com registry.length > 0 ? !!reg : null. O problema: a query pode demorar 300-500ms, causando uma janela onde todos os badges mostram "verificando" indefinidamente se a query falhar silenciosamente.
✏️ Fix
Adicionar isLoading: loadingRegistry ao useQuery e mostrar um indicador de loading na tabela enquanto loadingRegistry é true. Também adicionar fallback: se registry falhar, marcar valid: false em vez de null.
P1
ModalPlaybook: campo "Instruções" sem limite de caracteres visível
ModalPlaybook.tsx → instrucoes Textarea
O campo instrucoes é um Textarea sem maxLength e sem contador de chars. System prompts longos demais podem causar timeout na API da IA ou rejeição silenciosa. A spec não define limite, mas é boa prática mostrar contador e alertar quando ultrapassar 8000 chars (context window do agente).
✏️ Fix
Adicionar contador de chars abaixo do textarea: {instrucoes.length} chars com estilo vermelho quando > 8000. Warning: "Instruções muito longas podem reduzir precisão do agente."
P1
TabPlaybooks: upload de vídeo não exibe progresso real
TabPlaybooks.tsx → handleVideoUpload
O upload envia o FormData via fetch simples sem XHR com onprogress. Para arquivos de vídeo grandes (até 500MB), o usuário vê apenas "Enviando..." sem nenhuma indicação de progresso. Isso é especialmente ruim em conexões lentas — parece travado.
✏️ Fix
Usar XMLHttpRequest com upload.onprogress em vez de fetch, ou mostrar um progress HTML com percentual calculado pelo e.loaded / e.total. Alternativamente: estimar o progresso pelo tamanho do arquivo com um timer (gambiarra, mas melhor que nada).
P1
ModalBloco Step 4 (juris): títulos das selecionadas ficam como UUID quando jurisList não carregou
ModalBloco.tsx → selectedJurisTitles
Em modo edição (isEdit=true), os selectedJurisIds são pré-populados com os IDs do banco. Mas a query jurisList só carrega quando step === 4. Se o usuário vai direto ao Step 5 (Revisão) sem passar pelo Step 4, os chips de jurisprudências no Step 5 exibirão apenas os UUIDs brutos: {j.titulo} = id (fallback do código: { id, titulo: id }).
✏️ Fix
Alterar a query de jurisList para enabled: !!organizacao_id && (step === 4 || (isEdit && selectedJurisIds.length > 0)), garantindo que ao editar, a lista carregue antes do Step 5.
P1
PlaybooksKPICards: "Agentes" conta ai_agents_config mas aba não existe na página
PlaybooksKPICards.tsx + index.tsx (tabs)
O KPI card "Agentes" exibe count de ai_agents_config, mas não há aba "Agentes" na lista de TABS da página. Isso cria inconsistência: o usuário vê o número mas não sabe onde gerenciar. Ou a aba foi removida e o KPI ficou órfão, ou faltou implementar a aba.
✏️ Fix
Opção A: Remover o KPI "Agentes" do card se não há aba correspondente. Opção B: Adicionar link "ver agentes" no KPI card apontando para a página de configuração de agentes (/configuracoes/agentes ou similar). A opção A é mais rápida.
🔵 P2 — Melhorias (6)
P2
Sheet preview de Bloco exibe tag_ids como UUIDs raw
TabBlocosTexto.tsx → Sheet previewBloco
No Sheet de preview, os chips de tags fazem tagMap[tagId]?.codigo ?? '?' — correto no card da lista. Mas no SheetHeader, o loop é: previewBloco.tag_ids.slice(0,4).map(tagId => <span key={tagId}>{tagId}</span>), usando o UUID diretamente. O tagMap existe (importado do hook) mas não é usado no Sheet.
✏️ Fix
Mudar para {tagMap[tagId]?.codigo ?? tagId.slice(0,8)} no SheetHeader também.
P2
ModalNovoModelo: edit mode não mostra informação de versões existentes
ModalNovoModelo.tsx
Quando editando um modelo, o modal não exibe quantas versões existem, qual é a ativa, ou uma lista de versões. Para um usuário que quer entender o estado do modelo antes de editar, isso é uma gap. A spec menciona ModalModeloDetalhes para isso, mas o botão de editar no ModeloCard vai direto para o ModalNovoModelo sem passar pelo detalhes.
✏️ Sugestão
Adicionar linha informativa no topo do modal em edit mode: "X versões cadastradas · Versão ativa: vY · Status: Z". Dados disponíveis via versoesByGrupo[modelo.id] já no TabModelosPeticao.
P2
TabPlaybooks: sem filtro por agente_key na lista
TabPlaybooks.tsx
AGENT_KEYS define 6 agentes (olivia_peticao, olivia_revisor, etc.) mas não há filtro na lista por agente. Com muitos playbooks, não dá para ver "apenas os playbooks do olivia_peticao" sem buscar manualmente.
✏️ Sugestão
Adicionar Select de filtro por agent_key ao lado do filtro de escopo existente.
P2
ModalPlaybook: exemplos como JSON manual é friction alto
ModalPlaybook.tsx
O campo "Exemplos" pede JSON array escrito manualmente: [{"input": "...", "output": "..."}]. Isso é difícil de usar e propenso a erros de sintaxe. O campo valida, mas a experiência de criação é ruim.
✏️ Sugestão
Substituir por um mini-builder de pares: botão "Adicionar exemplo" → dois campos (Input / Output) → serializado como JSON internamente. Manter a textarea JSON como fallback avançado.
P2
PlaybooksPage: sem loading state no mobile select de abas
src/pages/playbooks/index.tsx
O Select mobile (sm:hidden) troca de aba instantaneamente, mas o conteúdo pode demorar para carregar (queries sendo feitas). Não há indicador visual de que a aba está carregando no mobile. No desktop os tabs têm loading por ErrorBoundary, mas no mobile não.
✏️ Fix
Adicionar Suspense wrapper no content area ou reutilizar o skeleton global de loading.
P2
KPI card "Modelos" conta olivia_modelos_peticao (versões), não modelos_grupo
PlaybooksKPICards.tsx
O KPI card "Modelos" conta registros em olivia_modelos_peticao, que são versões individuais de modelos. Se um modelo tem 3 versões, o KPI conta 3. O que faz mais sentido semânticamente é contar olivia_modelos_grupo (modelos únicos). A label "Modelos" confunde.
✏️ Fix
Mudar a query para olivia_modelos_grupo ou renomear o label para "Versões de Modelos".
