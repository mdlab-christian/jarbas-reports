🌐 O que é a JARBAS Network

A JARBAS Network é uma rede de agentes OpenClaw que trabalham em conjunto para o ecossistema MdLab/MdFlow. Cada agente roda num Mac (Mini ou Air), tem um bot Telegram próprio, uma porta OpenClaw dedicada e um conjunto de skills.

⚡ Regra Absoluta de Execução

Cada agente executa tudo diretamente. Claude Code = recurso auxiliar ocasional para paralelismo real ou quando o dono pedir explicitamente. Agentes não ficam esperando delegação — agem na hora.

ℹ️ Para criar um novo agente: decida o perfil (JARBAS-like = automação/backend; JANA-like = frontend/dev), a máquina (Mac Mini ou MacBook Air), e o foco de skills. O resto deste manual guia o processo completo.

Problemas do JARBAS (e Como Resolver)
⚠️ Estes são os problemas reais documentados em produção. Todo novo agente deve nacer já com os fixes aplicados.
🧠 Quebra de Contexto (Context Overflow)

Workspace context > 20KB causava flood de mensagens no gateway. O sistema travava e enviava mensagens repetidas para o Telegram.

Manter MEMORY.md < 6.000 chars. SOUL.md < 3.000 chars. AGENTS.md modular. Nunca incluir logs brutos em arquivos do workspace. Usar memory/YYYY-MM-DD.md para notas diárias.

👻 Alucinações de "Missão Ativa"

O agente "lembrava" de tarefas de sessões anteriores que não existiam mais, criando missões fantasmas e actions desnecessárias.

AGENTS.md com regra explícita: "não inferir tarefas de chats anteriores". HEARTBEAT.md limpo e com instruções precisas. Usar checkpoints reais em memory/group-[chat_id].md.

🔔 Notificações no Canal Errado

Notificações automáticas apareciam no DM do Christian em vez do grupo de notificações dedicado. Vazamento de informação técnica em conversa pessoal.

Regra D-012 explícita no MEMORY.md: "DM do Christian = conversa pessoal APENAS. Tudo vai para -5251265489." Cada agente tem um grupo de notificações configurado.

🔄 MEMORY.md Desatualizado

Depois de restart o agente ficava desorientado — não sabia o estado atual do sistema, skills registradas, decisões ativas. Precisava de re-briefing manual a cada sessão.

MEMORY.md deve ser atualizado toda vez que uma decisão importante é tomada. Estrutura: Decisões Ativas (D-001...), Estado do Sistema, Pendências P1. Manter vivo e atual é parte do trabalho do agente.

🌐 Navegador na Máquina Errada

JARBAS abria o navegador no MacBook Air (JANA) quando devia usar o Mac Mini, e vice-versa. Causava comportamento inesperado e falhas de automação.

TOOLS.md com regra explícita por máquina: JARBAS → target="host" (Mac Mini). JANA → target="host" (MacBook Air). NUNCA redirecionar para a máquina parceira.

📦 Skills Legadas vs OpenClaw Skills

Confusão entre skills Node.js (.mjs) legadas do JARBAS e OpenClaw Skills (SKILL.md). O agente tentava usar skills legadas que não existiam no novo contexto.

Separar claramente: OpenClaw Skills = ~/.openclaw/workspace/skills/ (SKILL.md). Skills legadas = ~/jarbas/src/skills/ (apenas para JARBAS clones). Novo agente usa APENAS OpenClaw Skills.

🚫 Delegação Excessiva para Claude Code

O agente delegava para Claude Code tarefas que ele mesmo podia fazer em segundos, gerando latência desnecessária e custos extras.

SOUL.md com "REGRA ABSOLUTA DE EXECUÇÃO": agente executa tudo diretamente. CC = paralelismo real ou revisão explicitamente pedida. Nunca delegar por insegurança.

🧩 ChromaDB RAG Morta

A skill de RAG não funcionava porque o ChromaDB não estava populado. O agente respondia como se o RAG existisse quando na verdade era stub.

Para novo agente: NÃO incluir skill de RAG/ChromaDB se não estiver populada. Usar Obsidian como cérebro documental (confiável, sem dependência de infra). RAG = fase 2.

🔐 Vazamento de Dados em Grupos

MEMORY.md (com dados pessoais do Christian) era carregada em grupos, expondo informações que não deveriam estar lá.

AGENTS.md com "MEMORY.md Security Rule": APENAS em sessão principal (DM direto). NUNCA em grupos. Dados sensíveis = exclusivos da sessão com o dono.

⏱️ Tarefas Presas (Immune System)

Tasks com status='processando' ficavam presas indefinidamente sem notificação, acumulando lixo no banco.

Heartbeat verifica jarbas_tarefas com status='processando' há > 15 min → marca como erro e notifica. Implementar immune_system equivalente em cada novo agente.

Arquitetura de Arquivos Base
📁 Estrutura Obrigatória do Workspace
~/.openclaw/workspace/
├── SOUL.md          # Quem é o agente (personalidade, missão, comandos)
├── AGENTS.md        # Instruções de sessão, skills, segurança
├── MEMORY.md        # Memória de longo prazo (< 6KB!)
├── IDENTITY.md      # Vibe, estilo de resposta
├── USER.md          # Quem é o dono (perfil, stack, estilo)
├── TOOLS.md         # Configurações específicas (SSH, browsers, contas)
├── HEARTBEAT.md     # Verificações periódicas automáticas
└── memory/
    ├── YYYY-MM-DD.md    # Log diário de eventos
    └── group-[id].md    # Contexto por grupo Telegram
⚠️ Regra dos 8KB (Critical)

O workspace context é injetado em CADA mensagem. Se ultrapassar ~20KB total, o gateway começa a travar e enviar mensagens duplicadas. Cada arquivo deve ser enxuto:

ArquivoLimite máximoMotivo
MEMORY.md6.000 charsCarregado em toda sessão principal
SOUL.md3.000 charsInjetado em toda mensagem
AGENTS.md4.000 charsContexto de sessão
HEARTBEAT.md2.000 charsPoll periódico
TOOLS.md2.000 charsConfig técnica
Workspace total< 20KBLimite absoluto
🔵 Separação de Memória

MEMORY.md → Decisões ativas, estado do sistema, lições consolidadas. Lida APENAS no DM com o dono.


memory/YYYY-MM-DD.md → Log bruto do dia. O agente escreve aqui. Não fica no workspace context automaticamente.


memory/group-[id].md → Contexto de cada grupo. Lido apenas quando o agente responde naquele grupo.

🔴 O que NÃO incluir no workspace
  • ❌ Logs brutos de execução
  • ❌ Credenciais completas (só aliases)
  • ❌ Dumps de banco de dados
  • ❌ HTMLs gerados
  • ❌ Código de skills completo
  • ❌ Histórico de conversas
  • ❌ Arquivos > 3KB individuais

Obsidian — Cérebro Obrigatório de Todo Agente
🧠 Regra Fundamental: Todo Agente Usa Obsidian

O vault Obsidian é o cérebro documental de qualquer agente da rede. Substitui RAG/ChromaDB que requer infra. É confiável, sincronizado via Obsidian Cloud e disponível em todas as máquinas.

📂 Vault do JARBAS
~/obsidian-olivia/

Knowledge base jurídica. Perfis de empresas rés, teses, jurisprudências. Lido antes de gerar petições via OlivIA.

📂 MDLab Vault (compartilhado)
Sincronizado via Obsidian Cloud

Disponível em Mac Mini e MacBook Air. Christian operando no MacBook Air. Sync automático via plano Obsidian pago.

⚙️ Como Configurar Obsidian no Novo Agente

Identificar o vault relevante

Definir qual vault o agente precisa: jurídico (~/obsidian-olivia/), MDLab, ou criar vault específico para o domínio do novo agente.

Documentar no SOUL.md e TOOLS.md

# No SOUL.md:
## Vault Obsidian
`~/obsidian-[nome]/` — descrição do conteúdo
**Quando ler:** listar os triggers específicos
**Como ler:** tool `read` com caminho `~/obsidian-[nome]/[arquivo]`

Configurar leitura contextual

O agente deve ler o vault ANTES de responder perguntas específicas do domínio. Ex: antes de gerar petição → ler perfil da empresa ré. Nunca responder sem consultar o vault quando relevante.

Ativar skill kb-collector

A skill kb-collector permite ao agente salvar novos conteúdos no vault automaticamente (URLs, PDFs, vídeos). Essencial para o vault ficar vivo.

Sync entre máquinas

Via Obsidian Sync Cloud — não precisa Git. Disponível automaticamente em todas as máquinas com o vault. Verificar se o vault está sincronizado antes de usar.

✅ Best practice: O agente deve sempre mencionar "consultei o vault" quando usar conhecimento do Obsidian. Isso cria transparência e facilita auditoria.

Variáveis de Ambiente (.env)
🔐 Nunca commitar o .env. Nunca incluir valores reais no HTML. Use aliases e nomes de variável — o agente que receber este manual vai preencher os valores do seu próprio .env.
🔑 Variáveis Obrigatórias (todo agente)
# IA
ANTHROPIC_API_KEY=sk-ant-...          # Claude
OPENAI_API_KEY=sk-...                  # GPT (opcional)
GEMINI_API_KEY=...                     # Gemini (opcional)

# Telegram
TELEGRAM_BOT_TOKEN=...                 # Token do bot (@BotFather)
TELEGRAM_ALLOWED_CHATS=123456789       # IDs separados por vírgula
TELEGRAM_EXTRA_GROUPS=-GROUP_ID,...    # Grupos adicionais

# OpenClaw
OPENCLAW_PORT=18789                    # Porta única por agente
OPENCLAW_SECRET=...                    # Segredo da instância

# Notificações
NOTIFICATIONS_GROUP=-5251265489        # Grupo de alertas
🏗️ Variáveis por Perfil
# JARBAS-like (automação jurídica)
SUPABASE_URL=https://[PROJECT].supabase.co
SUPABASE_KEY=eyJ...                    # service_role (NÃO anon)
SUPABASE_ACCESS_TOKEN=sbp_...          # Para DDL/Management API
N8N_API_KEY=...                        # Bearer token n8n
N8N_URL=https://primary-production-....up.railway.app
GITHUB_TOKEN=ghp_...                   # Para git push Pages

# JANA-like (frontend/dev)
SUPABASE_URL=https://[PROJECT].supabase.co
SUPABASE_KEY=eyJ...                    # service_role
GITHUB_TOKEN=ghp_...
TWENTY_FIRST_API_KEY=...               # 21st.dev components

# Obsidian (todo agente)
OBSIDIAN_VAULT_PATH=~/obsidian-[nome]/ # Caminho local do vault

# ElevenLabs (opcional — TTS)
ELEVENLABS_API_KEY=...
ELEVENLABS_VOICE_ID=xNGAXaCH8MaasNuo7Hr7  # Alterar por agente
⚠️ Atenção: SUPABASE_KEY ≠ SUPABASE_SERVICE_KEY

A variável correta é SUPABASE_KEY com o valor service_role. Usar SUPABASE_SERVICE_KEY como nome quebra as skills. Verificar isso primeiro quando uma skill de Supabase falhar.

Passo a Passo: Criar um Novo Agente
ℹ️ Informe ao agente que vai executar este passo a passo: nome do agente, perfil (JARBAS-like ou JANA-like), máquina (Mac Mini ou MacBook Air), usuário macOS (admin ou normal), e token do bot Telegram.

Criar Bot no Telegram

Abrir @BotFather → /newbot → definir nome e username → guardar o token. Anotar o ID do bot após enviar a primeira mensagem.

# Verificar que o bot está ativo
curl https://api.telegram.org/bot[TOKEN]/getMe

Criar Usuário macOS (se necessário)

# Usuário admin
sudo dscl . create /Users/[nome]
sudo dscl . create /Users/[nome] UserShell /bin/zsh
sudo dscl . create /Users/[nome] RealName "[Nome do Agente]"
sudo dscl . create /Users/[nome] UniqueID [UID]
sudo dscl . create /Users/[nome] PrimaryGroupID 80  # admin
sudo dscl . create /Users/[nome] NFSHomeDirectory /Users/[nome]
sudo createhomedir -c -u [nome]

# Usuário normal (sem sudo)
# substituir PrimaryGroupID para 20 (staff)

Instalar OpenClaw

# Como o novo usuário (su - [nome])
npm install -g openclaw

# Verificar instalação
openclaw --version
openclaw status

Configurar .env

mkdir -p ~/.openclaw
cat > ~/.openclaw/.env << 'EOF'
ANTHROPIC_API_KEY=sk-ant-[SEU_TOKEN]
TELEGRAM_BOT_TOKEN=[TOKEN_DO_BOT]
TELEGRAM_ALLOWED_CHATS=[ID_DO_DONO]
OPENCLAW_PORT=18[XX]  # porta única
OBSIDIAN_VAULT_PATH=~/obsidian-[nome]/
# ...demais variáveis conforme perfil
EOF
chmod 600 ~/.openclaw/.env

Criar Workspace Base

mkdir -p ~/.openclaw/workspace/memory
mkdir -p ~/.openclaw/workspace/skills

# Criar os 6 arquivos base (ver Seções 9-11 para templates)

Configurar PM2 (serviço contínuo)

# ecosystem.config.js
module.exports = {
  apps: [{
    name: '[nome-agente]',
    script: 'openclaw',
    args: 'gateway start',
    cwd: '/Users/[usuario]',
    env_file: '/Users/[usuario]/.openclaw/.env',
    watch: false,
    autorestart: true,
    max_restarts: 10,
    restart_delay: 3000,
    log_file: '/Users/[usuario]/.openclaw/logs/gateway.log',
    error_file: '/Users/[usuario]/.openclaw/logs/error.log',
  }]
}

# Iniciar
pm2 start ecosystem.config.js
pm2 save
pm2 startup  # configurar auto-start no boot
