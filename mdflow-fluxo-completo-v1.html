📋 O que a distribuição deve fazer
1
Pegar lead em Triagem
SELECT leads WHERE coluna_crm = 'triagem'
2
Para cada restrição do lead
SELECT restricoes WHERE lead_id = X
3
Criar processo
INSERT processos (cliente_id, empresa_id, restricao_id, vara, cidade, data_distribuicao)
4
Atualizar restrição
UPDATE restricoes SET processo_id = novo_processo_id
5
Criar/vincular cliente
Se não existe: INSERT clientes (lead_id FK) | Senão: vincular
6
Arquivar lead
UPDATE leads SET arquivado = true, arquivado_motivo = 'convertido'
🔴 Problemas Identificados
RPC distribuir_lead_completo — NÃO EXISTE (E22)
A operação de distribuição precisa ser atômica. Sem a RPC, qualquer falha no meio do processo deixa dados inconsistentes (processo criado mas lead não arquivado, ou restrição sem processo_id).
Campos obrigatórios do processo podem vir vazios
vara_distribuicao, cidade_distribuicao, data_distribuicao são informados no drawer de preparação (DistribuicaoDrawerPreparacao.tsx). Se o drawer não forçar preenchimento, o processo é criado incompleto.
Link restrição → processo
restricoes.processo_id deve ser populado após distribuição. processos.restricao_id também. Se ambos não forem atualizados atomicamente, o vínculo fica quebrado.

⚠️ FK bidirecional pode causar deadlock na criação
restricoes.processo_id referencia processos.id E processos.restricao_id referencia restricoes.id. Na distribuição, precisamos criar o processo SEM restricao_id primeiro, atualizar a restrição com processo_id, depois atualizar o processo com restricao_id. A RPC deve lidar com isso atomicamente.
