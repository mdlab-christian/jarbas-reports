🎯 Resumo Executivo

✅ O que a API entrega bem

  • Processos completos — dados, fases, datas, CNJ, honorários
  • Movimentações processuais — TRIBUNAL + MANUAL, histórico completo
  • Publicações por processo — intimações capturadas automaticamente
  • Clientes/Contatos — CPF/CNPJ, dados completos, origem
  • Tarefas por processo — histórico de atividades
  • Financeiro — honorários, receitas, despesas
  • Filtros avançados com paginação
  • API bem documentada (Fevereiro 2026)

❌ O que a API NÃO entrega

  • Download de documentos/arquivos — CRÍTICO, não existe endpoint
  • "Todas as publicações" — só consulta por lawsuit_id, sem paginação global
  • Audiências explícitas — não tem endpoint dedicado
  • Resultados de acordos — não documentado
  • Sem webhook/push — só pull
  • Acesso requer aprovação formal da Advbox
⚠️ Ponto crítico de atenção: A API não tem endpoint de documentos. Para o objetivo de "baixar todos os documentos antes de cancelar o plano", a API sozinha não resolve. Isso vai precisar de uma estratégia alternativa (scraping da plataforma web ou exportação manual).

🔴 Documentos — O Problema Crítico

🔴 BLOQUEADOR: A API Advbox não possui nenhum endpoint de download de arquivos/documentos. Os 19 endpoints documentados cobrem processos, movimentações, publicações, tarefas, transações e configurações — mas nenhum expõe os documentos salvos nos processos.

Alternativas para recuperar os documentos

  • Opção A (recomendada): Contatar o suporte da Advbox e solicitar um export de dados completo antes do cancelamento — muitos SaaS oferecem isso por compliance com LGPD
  • Opção B: Scraping automatizado da plataforma web via Playwright — riscado mas funcional se o volume for gerenciável
  • Opção C: Download manual seletivo — apenas documentos críticos (petições iniciais, sentenças, acordos)
  • Verificar: Se há endpoints não documentados na API (a doc é v1.0 de Fev/2026 — pode ter algo no código-fonte da plataforma web)
💡 Estratégia: Antes de cancelar o plano, abrir ticket com a Advbox pedindo export completo (documentos + dados). É um direito do cliente sob LGPD (portabilidade de dados). Isso resolve sem precisar de scraping.

📢 Validação de Intimações — Como Fazer

✅ Viável. Mas exige estratégia para contornar a limitação de "só por processo".

Abordagem 1 — via /last_movements ⭐ Recomendada

Endpoint retorna movimentações de TODOS os processos, ordenadas por data.

  • GET /last_movements com date_start + date_end (hoje)
  • Filtra origin=TRIBUNAL para capturar apenas publicações oficiais
  • Compara com o que chegou no MdFlow via EPROC no mesmo período
  • Diferença = intimações que estão numa mas não na outra
  • Rate limit: 30 GET/min é suficiente para monitoramento diário

Abordagem 2 — via /publications/{lawsuit_id}

Mais detalhado por processo, mas caro em requests.

  • GET /publications/{id} para cada processo
  • 5.800 processos × 1 req = 5.800 calls
  • Com 30/min → ~3,2 horas para varrer tudo
  • Viável como verificação pontual, não como monitoramento contínuo
  • Faz sentido para auditoria completa antes de cancelar o plano

⚖️ Puxar Decisões/Despachos/Sentenças

✅ Viável via movimentações. As decisões chegam como movimentações do tipo TRIBUNAL.

Fluxo de extração de decisões

  1. GET /last_movements com filtro de data e/ou process_number
  2. Filtrar origin=TRIBUNAL (movimentações capturadas automaticamente)
  3. Keywords para identificar decisões: "sentença", "acórdão", "decisão", "despacho", "julgamento"
  4. Cruzar com processos via lawsuit_id → saber qual cliente foi beneficiado
  5. Integrar ao histórico de decisões no MdFlow
💡 Isso espelha exatamente o que a página "Decisões" do MdFlow precisa. O JARBAS consegue monitorar isso periodicamente via /last_movements sem precisar varrer processo a processo.

💡 Estratégia de importação em massa: Para importar tudo de uma vez (migração), rodar o script de madrugada em 3 passos: (1) Lista completa de processos, (2) detalhes + movimentações em paralelo com delay de 2s entre calls, (3) clientes + financeiro. Estimativa total: 8-12 horas com rate limit seguro.

🗺️ Plano de Ação Recomendado

Fase 1 — Obtenção de Acesso (antes de qualquer código)

  • Verificar se a conta atual do Advbox já tem ou pode obter API key
  • A API é exclusiva para "parceiros e integradores aprovados" — contactar suporte
  • Solicitar também export completo de dados (LGPD) incluindo documentos
  • Obter API key de teste para validar endpoints

Fase 2 — Importação de Dados (com API key em mãos)

  1. GET /settings → mapear todos os IDs (fases, tipos, usuários)
  2. GET /customers → importar todos os clientes para MdFlow
  3. GET /lawsuits paginado → importar lista completa de processos
  4. GET /lawsuits/{id} em lote → enriquecer com dados financeiros
  5. GET /movements/{id} em lote → histórico completo de movimentações
  6. GET /publications/{id} em lote → histórico de intimações
  7. GET /transactions → importar dados financeiros históricos

Fase 3 — Validação de Intimações (monitoramento paralelo)

  • Por N dias antes do cancelamento, rodar comparação diária
  • GET /last_movements (date_start=hoje) → intimações Advbox
  • Comparar com EPROC do JARBAS (mesmo período)
  • Delta = processos com monitoramento diferente → investigar
  • Quando delta consistente = zero por 7+ dias → pode cancelar com segurança

Fase 4 — Documentos (problema em aberto)

  • Opção A (primeiro tentar): Pedir export via suporte/LGPD
  • Opção B (se negar): Playwright para scraping da plataforma web
  • Priorizar apenas documentos críticos: petições iniciais, sentenças, acordos
  • Processar em lote por cliente, não por processo

🏁 Veredicto Final

7.5/10
API Viável — com ressalva crítica nos documentos
A API Advbox é bem documentada e cobre 85% das necessidades de migração. O único bloqueador real é a ausência de endpoint para documentos — que precisa ser resolvido por outras vias (export LGPD ou scraping web) antes do cancelamento.

✅ Decisão: Usar a API para...

  • Importação completa de processos → MdFlow
  • Validação comparativa de intimações (período de transição)
  • Migração de clientes e dados financeiros
  • Extração de decisões via movimentações TRIBUNAL
  • Histórico de tarefas por processo

⚠️ Não resolver via API...

  • Download de documentos → resolver via suporte/LGPD primeiro
  • Monitoramento contínuo pós-migração → JARBAS + EPROC são superiores
  • Audiências → extrair via movimentações (workaround)
