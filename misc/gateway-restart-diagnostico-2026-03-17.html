🔍 Causa Raiz — O que está acontecendo

Diagnóstico: O gateway OpenClaw tem um memory leak estrutural. Ele cresce para ~900MB em intervalos de 10–60 minutos independentemente do uso. O heartbeat detecta e reinicia — mas o gateway volta a crescer rapidamente, causando o ciclo. O PM2 tem 38 restarts porque reinicia automaticamente quando o processo crasha.

A cadeia de causas identificada (da mais impactante para a menos):

CRÍTICO — Causa #1

Memory Leak no Gateway OpenClaw (Node.js)

O processo openclaw-gateway cresce continuamente na memória, atingindo 900–1015MB em intervalos regulares (10–60min). Após reiniciar, começa do zero (~500MB) e cresce novamente. Isso é um memory leak no próprio gateway, não nos scripts JARBAS. Prova: o gateway reinicia e em ~10min já está acima de 700MB novamente. O comportamento não muda com ou sem uso ativo do usuário.

CRÍTICO — Causa #2

82MB de sessões em disco são carregadas em memória pelo gateway

O gateway mantém sessões JSONL em memória para roteamento. Com 69 sessões pesando 82MB em disco (maior: 565KB, sessão principal: ~142KB), o gateway infla o heap ao carregar essas sessões. O contextTokens: 200000 no config também força manter contexto enorme por sessão. Solução imediata: limpar sessões antigas + reduzir retenção.

CRÍTICO — Causa #3

Cooldown de restart quebrado — cascata de restarts às 19:30–20:07

Entre 19:30 e 20:07, o heartbeat reiniciou o gateway 10 vezes em 37 minutos. O problema: o cooldown de 20min não funcionou porque o gateway morria antes de o processo filho do restart registrar o timestamp. A lógica atual usa execSync('openclaw gateway restart') mas não aguarda confirmação que o restart completou antes de salvar o cooldown file — então o próximo ciclo do heartbeat (60s depois) vê o gateway ainda alto em memória e reinicia novamente.

ALTO — Causa #4

Bug no session-cleaner-cron: "Cannot access 'result' before initialization"

O cron de limpeza de sessões falha silenciosamente desde as 17:09 de hoje. O erro Cannot access 'result' before initialization indica que o cleanSessions() está retornando undefined ou tendo erro assíncrono que não retorna o objeto esperado. Com o cleaner quebrado, as sessões crescem e o gateway absorve mais memória. Resultado: o mecanismo de limpeza preventiva que deveria reduzir a memória está inativo há ~8h.

MÉDIO — Causa #5

contextTokens: 200000 + compaction: off

O openclaw.json define contextTokens: 200000 (200K tokens) e contextPruning: { mode: "off" }. Isso significa que o gateway tenta manter o contexto máximo de cada sessão ativo na memória. Com múltiplas sessões (heartbeats, grupos, DM), o consumo se multiplica. O compaction.memoryFlush.enabled: false também desativa a liberação proativa de memória.

🛠️ Plano de Correção — 4 Ações

Fix #1 — Corrigir bug do session-cleaner-cron PENDENTE IMPL.

O erro Cannot access 'result' before initialization ocorre porque cleanSessions() pode retornar sem objeto ou lançar erro assíncrono mal tratado. Fix: adicionar fallback no retorno e garantir que o cron usa result?.cleaned ?? 0.

// start.mjs linha ~1737 — FIX do cron
setInterval(async () => {
  try {
    const result = await cleanSessions({ notificar: false, verbose: false });
    const cleaned = result?.cleaned ?? 0;
    const savedKB = result?.savedKB ?? 0;
    if (cleaned > 0) {
      console.log(`[session-cleaner-cron] ${cleaned} sessões limpas, ${savedKB}KB liberados`);
    }
  } catch (err) {
    console.error('[session-cleaner-cron] Erro:', err.message);
  }
}, 30 * 60 * 1000);

Fix #2 — Corrigir cascata de restarts (cooldown race condition) PENDENTE IMPL.

O cooldown deve ser salvo antes do restart, não depois. Além disso, o heartbeat deve pular a verificação de memória por 3 ciclos (3min) após disparar um restart, dando tempo ao gateway de subir e estabilizar.

// heartbeat.mjs — Fix M7 cooldown
if (rssMB > M7_THRESHOLD_MB) {
  const agora = Date.now();
  let lastRestart = 0;
  try { lastRestart = parseInt(rfM7(M7_COOLDOWN_FILE, 'utf8') || '0'); } catch {}
  
  if (agora - lastRestart < M7_COOLDOWN_MS) {
    // cooldown ativo — skip
  } else {
    // CRÍTICO: salvar cooldown ANTES do restart
    wfM7(M7_COOLDOWN_FILE, String(agora));
    await new Promise(r => setTimeout(r, 500)); // flush
    execM7('openclaw gateway restart', { timeout: 15000 });
  }
}

Fix #3 — Limpar sessões antigas do disco agora PENDENTE IMPL.

Deletar sessões inativas com mais de 7 dias e comprimir as sessões grandes (565KB → remover thinking blocks). Isso reduz a memória do gateway na próxima inicialização em ~200-300MB estimados.

# Comando a executar agora
find ~/.openclaw/agents/main/sessions -name "*.jsonl" \
  -not -newer ~/.openclaw/agents/main/sessions/62708d61*.jsonl \
  -mtime +7 -delete

# Forçar limpeza de thinking blocks agora
openclaw agent session clean --all --dry-run  # verificar
openclaw agent session clean --all            # executar

Fix #4 — Aumentar threshold de restart para 1200MB RECOMENDAR

Com 16GB de RAM, reiniciar em 900MB é muito conservador. O Mac Mini tem sobra. Aumentar para 1200MB dá mais tempo entre restarts e reduz o impacto nas conversas ativas. O leak continuará existindo (é do gateway), mas os restarts serão menos frequentes enquanto o problema upstream não é corrigido.

// heartbeat.mjs linha ~352
const M7_THRESHOLD_MB = 1200; // era 900 — Mac Mini tem 16GB
const M7_COOLDOWN_MS = 30 * 60 * 1000; // era 20min → 30min
