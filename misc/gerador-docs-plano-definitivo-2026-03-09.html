Gerador de Comprovante de Restrição — Spec Completa
skill_gerar_comprovante.mjs — A criar (P0)
Pipeline de geração
1. Fetch: processos JOIN restricoes JOIN clientes JOIN advogados
2. Fetch auditoria: auditorias WHERE cliente_id ORDER BY created_at DESC LIMIT 1
3. Build HTML com dados bureau (Serasa/BV/SPC/Quod/BCB)
4. Playwright: chromium.launch() → page.setContent() → page.pdf()
5. Upload: Supabase Storage bucket 'comprovantes'
6. INSERT processos_documentos (status=disponivel)
7. UPDATE processos.comprovante_restricao_*
Versão por advogado
Sem tabela nova. Template HTML recebe dinamicamente:
— advogado.nome (ex: Heloisa Miglioranza Schmitt)
— advogado.oab + advogado.oab_uf
— Assinatura na base do PDF com dados do advogado do processo
— Bureau branding: Serasa=vermelho, BV=verde, SPC=azul, Quod=roxo, BCB=navy
— 5 variantes de template (1 por advogado_id) são geradas dinamicamente
Conexão com Auditorias
O campo auditorias.resultado_json contém todos os bureaus consultados, valores individuais, datas e status de cada restrição — muito mais rico que os campos flat processos.restricao_* (que têm apenas 1 restrição).

Query: SELECT * FROM auditorias WHERE cliente_id = $1 ORDER BY created_at DESC LIMIT 1
Fallback: se não há auditoria, usa campos flat de processos.
Impacto: comprovante pode mostrar TODAS as restrições do cliente (ex: BV + SPC juntas) no mesmo PDF.
