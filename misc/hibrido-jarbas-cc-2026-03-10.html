🔬 JARBAS vs Claude Code — Onde Cada Um Ganha

🦞 JARBAS (OpenClaw Sonnet)

  • Acesso nativo a tools (sem subprocess) RÁPIDO
  • Contexto do Telegram (sabe o que Christian pediu) CONTEXTO
  • Pode perguntar ao Christian mid-task INTERATIVO
  • Browser, imagens, múltiplas tools simultâneas TOOLS
  • Análise de código quando sabe a estrutura ANÁLISE
  • Geração de Markdown/HTML (sem subprocess) GERAÇÃO
  • Pressão de contexto (histórico do chat) LIMITE
  • Loop longo com muitos arquivos CANSATIVO
vs

🤖 Claude Code (subprocess)

  • Exploração autônoma de codebase desconhecido EXPLORAÇÃO
  • Loop iterativo com tsc + fix + commit ITERATIVO
  • Muitos arquivos lidos/escritos em sequência VOLUME
  • Execução de shell em loop (tests, builds) SHELL
  • Isolamento — não afeta contexto do JARBAS ISOLADO
  • Overhead de spawn (5–10s por chamada) LENTO
  • Sem contexto de quem pediu ou por quê SEM CTX
  • Não pode interagir com Christian mid-task CEGO

Ambos usam o mesmo modelo base (Claude Sonnet-4-6). A diferença é o overhead operacional, não a inteligência.

🎯 Como Orquestrar em Missões Complexas

JARBAS é o Cérebro

✦ Recebe o pedido de Christian
✦ Lê o contexto (arquivos, DB, histórico)
✦ Cria o plano de execução
✦ Decide o que vai para CC e o que faz direto
✦ Monitora CC (watchdog)
✦ Revisa o resultado do CC
✦ Notifica Christian com resumo

CC é o Executor Pesado

✦ Recebe instrução específica do JARBAS
✦ Escreve código (muitos arquivos)
✦ Roda tsc + fix em loop até 0 erros
✦ Faz commit com mensagem padronizada
✦ Retorna resultado para JARBAS avaliar
✦ NÃO decide estratégia — só executa
✦ NÃO fala com Christian — JARBAS faz isso

Regra de decisão: se a tarefa precisa de < 20 operações de arquivo → JARBAS direto. Se precisa de > 20 ou loop iterativo → CC. JARBAS sempre sabe mais sobre o contexto — CC sabe mais sobre execução bruta.

🛠️ Plano de Implementação — O que Mudar no JARBAS

1

Flag modo_hibrido na lapidação

Adicionar ao payload da tarefa. Quando ativo, substitui CC nas fases de análise por chamadas diretas via OpenClaw tools. runCCPlan() → analisarCodigo() (JARBAS lê arquivos + raciocina). gerarPRD() → JARBAS escreve direto. Estimativa: ~2h de refactor.

2

Classifier de complexidade de tarefa

Antes de qualquer runClaudeCode(), verificar: quantos arquivos vão ser lidos/escritos? Usa regex simples no prompt para estimar volume. Se baixo → JARBAS direto. Se alto → CC. Implementável em skill_claude_code.mjs como middleware.

3

Bridge de contexto JARBAS → CC

Quando JARBAS delega para CC, passar o contexto relevante como briefing compacto (já existe em ~/jarbas/.claude/bridge-to-jarbas.mjs). CC executa com contexto, retorna resultado, JARBAS revisa. Nenhuma decisão estratégica no CC.

4

JARBAS Review pós-CC

Após CC terminar a implementação, JARBAS lê o diff gerado (git diff HEAD~1) e faz o code review final. Sem subprocess — JARBAS lê o texto do diff com exec e raciocina. Se nota < 9, gera instrução específica e manda CC corrigir só o ponto fraco.

5

Telemetria de tempo por fase

Logar timestamp de início/fim de cada fase (JARBAS vs CC). Após 5 lapidações, gerar relatório de onde o tempo vai. Usar esse dado para refinar o classifier de complexidade continuamente.

✅ Veredicto — Vale muito a pena

O JARBAS atual já é o cérebro da operação. O CC funciona como um executor burro (muito bom nisso, mas burro). O problema é que hoje o JARBAS delega trabalho de análise e escrita pro CC — que é lento por overhead de subprocess — quando poderia fazer direto.

Implementando o modo híbrido: lapidações passam de ~90min para ~50min. Missões complexas ficam mais inteligentes porque JARBAS mantém contexto de ponta a ponta. CC vira um "subcontratado de obra" — entra só pra fazer trabalho braçal pesado.

Prioridade: implementar os passos 1 e 2 primeiro (modo_hibrido + classifier). O ganho é imediato na próxima lapidação.
