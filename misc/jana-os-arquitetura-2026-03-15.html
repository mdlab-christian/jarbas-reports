🔍 Diagnóstico: O Problema Real

Antes de propor solução, entender o problema com precisão.

⚠️ O Telegram não trava — o JARBAS trava

O Telegram pode receber e entregar mensagens de múltiplos grupos simultaneamente sem problema. O gargalo é o JARBAS processando tarefas pesadas inline — quando ele está lapidando a /olivia, o thread principal fica ocupado e as mensagens dos outros grupos ficam na fila esperando. Com dois computadores, a solução não é duplicar o receptor de mensagens — é separar quem recebe de quem executa.

💡 O que já existe no Mac Mini que resolve isso

O JARBAS já tem task-consumer.mjs + tabela jarbas_tarefas no Supabase. A infraestrutura de fila já existe. O problema é que tarefas como lapidação ainda rodam inline em vez de serem enfileiradas. O Mac Mini precisa virar o executor da fila — e o JARBAS no Air precisa ser o enfileirador.

🏷️ Renomear Mac Mini para JANA — Vale a Pena?

⚠️ Risco Real: Renomear quebra coisas

O Mac Mini tem 54 skills, paths hardcoded, usuário jarbas, variáveis de ambiente com JARBAS_, nome do serviço OpenClaw, nome no SOUL.md, nome no Supabase. Renomear tudo é uma migração de 2-4 horas com risco de quebrar coisas em produção — EPROC, crons, n8n webhooks, paths de arquivo.

✅ Solução: Jana é uma identidade lógica, não física

O Mac Mini continua se chamando JARBAS nos arquivos, usuário, serviços. A "JANA" é apenas como você e o JARBAS se referem a ele logicamente. Um arquivo JANA.md no workspace define o papel. Nenhum risco de quebra, zero migração.

🔗 Skills Compartilhadas — Como Funciona

Você perguntou se os dois agentes podem compartilhar as mesmas skills. Sim — via Supabase como broker central.

Supabase = Sistema Nervoso Central

  • Tabela jarbas_tarefas é a fila compartilhada — qualquer máquina pode enfileirar, qualquer máquina pode executar
  • Cada tarefa tem campo preferred_host (macmini | macbook | any)
  • Skills idênticas nos dois computadores (via rsync) — mesmas capacidades
  • Resultados e notificações passam pelo Supabase → JARBAS entrega pro Telegram

↔️ Bidirecional: Mac Mini também pode jogar pro Air?

💡 Pode, mas não é necessário agora

Como o Air nem sempre está ligado (você leva pra fora), tarefas do Mini enfileiradas pro Air podem ficar presas. O mais seguro: Mini sempre executa tudo, Air executa quando está ligado e há sobrecarga. Você pode rotear explicitamente com no air: [tarefa] quando quiser usar o Air.

💡 Recomendação: um bot agora, dois depois

Implementar a fila de offload resolve 90% do problema de travamento. Segundo bot é fase 2 — só faz sentido quando você quer conversar diretamente com a Jana ou quando a fila não for suficiente para a carga.

📋 Plano de Execução — O que Fazer

P1
Completar setup do MacBook Air
rsync do Mini → Air · ~20 min · Etapa 4 do plano anterior

Rodar no Mac Mini para espelhar tudo no Air:

rsync -avz ~/jarbas/ jarbas@100.101.208.82:~/jarbas/
rsync -avz ~/.openclaw/ jarbas@100.101.208.82:~/.openclaw/
rsync -avz ~/.claude/ jarbas@100.101.208.82:~/.claude/
rsync -avz ~/.ssh/ jarbas@100.101.208.82:~/.ssh/
Depois no Air: instalar dependências com cd ~/jarbas && npm install
P1
Adicionar campo preferred_host na jarbas_tarefas
Migration Supabase · ~5 min
ALTER TABLE jarbas_tarefas 
ADD COLUMN IF NOT EXISTS preferred_host TEXT DEFAULT 'macmini',
ADD COLUMN IF NOT EXISTS executing_host TEXT,
ADD COLUMN IF NOT EXISTS host_claimed_at TIMESTAMPTZ;
Isso permite roteamento por máquina. JANA só pega tarefas com preferred_host = 'macmini' ou 'any'.
P1
task-consumer.mjs — filtrar por host
Editar arquivo existente · ~15 min

Cada máquina só pega tarefas destinadas a ela:

// No task-consumer.mjs, na query de busca de tarefas:
const HOST = process.env.JARBAS_HOST || 'macmini'; // 'macbook' no Air

const { data: tarefas } = await supabase
  .from('jarbas_tarefas')
  .select('*')
  .eq('status', 'pendente')
  .in('preferred_host', [HOST, 'any'])
  .is('executing_host', null)
  .order('created_at', { ascending: true })
  .limit(3);
P2
Skill de delegação no JARBAS Air
Nova skill · ~30 min

JARBAS detecta tarefa pesada e enfileira no Mini automaticamente:

// skill_delegar_jana.mjs
// Tarefas que sempre vão pro Mini:
const HEAVY_TASKS = ['lapidar_pagina', 'olivia_gerar_peticao', 
                     'analise_juridica', 'analisar_video', 'e2e_vision'];

export async function delegarParaJana({ skill, payload, chatId }) {
  return supabase.from('jarbas_tarefas').insert({
    skill,
    payload: { ...payload, chat_id: chatId },
    preferred_host: 'macmini',
    status: 'pendente'
  });
}
P2
OpenClaw no Air com identidade separada
Config · ~10 min

Air recebe mensagens Telegram, Mini processa fila. Ambos com mesmo bot token por enquanto:

# .env no MacBook Air:
JARBAS_HOST=macbook
JARBAS_ROLE=orchestrator

# .env no Mac Mini:
JARBAS_HOST=macmini
JARBAS_ROLE=worker
⚠️ Por enquanto o Air fica como PRIMARY no Telegram. Mini fica em standby para Telegram mas ativo para processar a fila. Quando Air desliga, Mini assume o Telegram automaticamente.
P3
