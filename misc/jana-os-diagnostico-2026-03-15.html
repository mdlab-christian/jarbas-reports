Ambas as máquinas estão online e funcionais. O problema é que 100% das tarefas são criadas com preferred_host = 'any', transformando a distribuição numa corrida entre as duas máquinas — e o Mini, por ter mais workers ativos, ganha a maioria. A lógica do dispatcher existe no código mas não está sendo aplicada na criação das tarefas.

🔴 Problema Identificado

🐛 Root Cause: preferred_host = 'any' em 100% das tarefas

A função SQL jarbas_consume_tarefa_host busca tarefas com:

WHERE status = 'pendente' AND (preferred_host = 'any' OR preferred_host = p_host) ORDER BY prioridade DESC, created_at ASC FOR UPDATE SKIP LOCKED

Com 100% das tarefas como 'any', ambas as máquinas competem por todas as tarefas. O Mini ganha mais porque tem workers que acordam mais cedo (processo mais antigo, poll mais frequente).

O dispatcher em Node.js existe e está correto — mas a lógica nunca é aplicada porque as tarefas chegam ao banco já com preferred_host = 'any', contornando toda a lógica inteligente.

🔍 Cadeia do Problema
1

Tarefa é criada — preferred_host não é setado

Tarefas como cerebro_documental, captura_intimacoes, analise_intimacoes_bulk são criadas diretamente via Supabase insert SEM passar pela função criarTarefa() do dispatcher, ou com preferred_host: 'any' explícito.

2

Banco recebe preferred_host = 'any'

O campo fica como 'any' — padrão do insert. A lógica de roteamento (MINI_ALWAYS, AIR_PREFERRED, EPROC_SHARED) nunca é consultada.

3

SQL retorna a tarefa pra quem pegar primeiro

Com FOR UPDATE SKIP LOCKED, a primeira máquina que fizer poll pega a tarefa. O Mini tem workers mais estáveis (processo rodando há 16h vs 30min do Air), então ganha mais corridas.

4

Resultado: Mini sobrecarregado, Air ocioso

Mini com 494MB de RAM livre executando 71% das tarefas. Air com 1.97GB livre e CPU a 1% executando apenas 28% — e nesse momento 0 workers ativos.

✅ Solução Recomendada

✅ Fix em 2 frentes: SQL + Node.js

1

Fix SQL — Lógica de roteamento dentro do banco

Atualizar a função jarbas_consume_tarefa_host para considerar o tipo da tarefa quando preferred_host = 'any'. Isso garante roteamento mesmo se o Node não setar o campo.

-- Lógica adicional na WHERE clause: AND ( preferred_host = p_host OR (preferred_host = 'any' AND ( -- Mini pega: rotinas fixas dele (p_host = 'macmini' AND tipo IN ( 'cerebro_documental', 'captura_intimacoes', 'importacao_diaria', 'session_cleaner', 'immune_system' )) OR -- Air pega: IA pura e lapidação (p_host = 'macbook' AND tipo IN ( 'lapidar_pagina', 'claude_code', 'analise_juridica', 'relatorio_html', 'analisar_video', 'jurimetria' )) OR -- Qualquer um: resto (load balancing) tipo NOT IN ('cerebro_documental', 'captura_intimacoes', ...) )) )
2

Fix Node.js — Garantir que criarTarefa() sempre seta preferred_host

Auditar todos os pontos de criação de tarefa e garantir que usam criarTarefa() do dispatcher, que já tem a lógica correta. Ou adicionar um hook pre-insert no Supabase.

// task-dispatcher.mjs — já correto, mas precisa ser chamado sempre export async function criarTarefa(tarefa, opts = {}) { // Computar preferred_host se não veio explícito const preferred_host = await dispatchHost(tarefa.tipo, opts); // Garantir que tarefa.preferred_host não sobrescreva se for 'any' const final_host = (tarefa.preferred_host && tarefa.preferred_host !== 'any') ? tarefa.preferred_host : preferred_host; await supabase.from('jarbas_tarefas') .insert({ ...tarefa, preferred_host: final_host }); }
3

Bonus — Balanceamento por RAM quando preferred_host = 'any'

Para tarefas genuinamente flexíveis, o dispatcher já considera carga via heartbeat. Melhorar para também considerar RAM livre:

// getCargaAtual() — adicionar ram_free_mb ao heartbeat check const miniScore = carga.macmini.ativos / carga.macmini.max + (1 - carga.macmini.ram_mb / 16384); const airScore = carga.macbook.ativos / carga.macbook.max + (1 - carga.macbook.ram_mb / 16384); return miniScore <= airScore ? 'macmini' : 'macbook';
