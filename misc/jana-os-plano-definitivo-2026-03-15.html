🔬 Diagnóstico Real (após investigação do código)

⚠️ Bug confirmado: task-consumer roda com concurrency=1

O arquivo task-consumer.mjs linha 326 diz literalmente: "Loop sequencial iniciado (concurrency=1)". Uma tarefa por vez. Se você tem uma lapidação rodando, nenhuma outra tarefa da fila executa até ela terminar. Isso explica tudo.

⚠️ OpenClaw é o canal primário — não o JARBAS Node.js

O poller Telegram foi desativado ("OpenClaw é o canal primário" — linha 1479 do start.mjs). O OpenClaw gerencia as sessões por grupo. Quando processa uma resposta longa, aquela sessão fica bloqueada. Grupos diferentes deveriam ser paralelos, mas tarefas pesadas que rodam inline na sessão do OpenClaw travam tudo.

✅ O que já existe e funciona

  • 54 skills registradas e funcionando
  • Fila jarbas_tarefas no Supabase com RPC jarbas_consume_tarefa
  • Orphan recovery automático (tarefas travadas > 10min são resetadas)
  • Reagendamento automático com backoff exponencial
  • Tailscale conectando Mac Mini ↔ MacBook Air ✅

Conclusão do diagnóstico: A infraestrutura está quase pronta. Precisam de 3 mudanças cirúrgicas: (1) concurrency no task-consumer, (2) filtro de host na fila, (3) task-consumer rodando no Air também.

🔧 O Que Precisa Mudar (3 mudanças cirúrgicas)

1
task-consumer: concurrency=1 → 3
Mac Mini + MacBook Air · ~30 min · P1 — resolve o travamento

Mudar de loop sequencial para pool de workers paralelos. Cada máquina processa até 3 tarefas simultâneas.

// task-consumer.mjs — substituir runTaskLoop() por:

const MAX_CONCURRENT = 3; // ajustável por máquina
const HOST = process.env.JARBAS_HOST || 'macmini'; // 'macbook' no Air

async function runWorkerPool() {
  console.log(`[task-consumer] 🚀 Pool paralelo — concurrency=${MAX_CONCURRENT} host=${HOST}`);
  
  // Lança N workers independentes
  const workers = Array.from({ length: MAX_CONCURRENT }, (_, i) => 
    runSingleWorker(i + 1)
  );
  await Promise.all(workers); // nunca retorna
}

async function runSingleWorker(id) {
  while (true) {
    try {
      // Pega tarefa destinada a este host (ou 'any')
      const { data } = await supabase.rpc('jarbas_consume_tarefa_host', {
        p_organizacao_id: ORG_ID,
        p_instance_id: `${INSTANCE_ID}-w${id}`,
        p_host: HOST
      });

      const tarefa = Array.isArray(data) ? data[0] : data;
      if (!tarefa?.id) {
        await sleep(POLL_INTERVAL);
        continue;
      }

      console.log(`[worker-${id}] Executando: ${tarefa.tipo} (${tarefa.id.slice(0,8)})`);
      await executarTarefa(tarefa);
      await sleep(INTER_TASK_DELAY);
    } catch (e) {
      console.error(`[worker-${id}] Erro:`, e.message);
      await sleep(POLL_INTERVAL * 2);
    }
  }
}
A RPC atual jarbas_consume_tarefa precisa de uma variante com filtro de host — ver próxima mudança.
2
Supabase: adicionar preferred_host + nova RPC
Migration + função SQL · ~15 min · P1 — habilita roteamento
-- Migration:
ALTER TABLE jarbas_tarefas 
ADD COLUMN IF NOT EXISTS preferred_host TEXT DEFAULT 'any';
-- Valores: 'any' | 'macmini' | 'macbook'

-- Nova RPC com filtro de host:
CREATE OR REPLACE FUNCTION jarbas_consume_tarefa_host(
  p_organizacao_id UUID, p_instance_id TEXT, p_host TEXT
) RETURNS SETOF jarbas_tarefas AS $$
DECLARE v_tarefa jarbas_tarefas;
BEGIN
  SELECT * INTO v_tarefa FROM jarbas_tarefas
  WHERE organizacao_id = p_organizacao_id
    AND status = 'pendente'
    AND (preferred_host = 'any' OR preferred_host = p_host)
    AND (scheduled_at IS NULL OR scheduled_at <= NOW())
  ORDER BY prioridade DESC, created_at ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED;
  
  IF v_tarefa.id IS NULL THEN RETURN; END IF;
  
  UPDATE jarbas_tarefas SET
    status = 'em_andamento',
    iniciado_em = NOW(),
    instance_id = p_instance_id,
    executing_host = p_host
  WHERE id = v_tarefa.id;
  
  v_tarefa.status := 'em_andamento';
  RETURN NEXT v_tarefa;
END;
$$ LANGUAGE plpgsql;

🌐 Tunnel Cloudflare — Compartilhar ou Separar?

O Mac Mini usa um Named Tunnel do Cloudflare (token fixo no LaunchAgent). Isso é diferente de um Quick Tunnel temporário — tem URL fixa e é associado a uma conta Cloudflare.

⚠️ NÃO use o mesmo tunnel nos dois computadores

Um Named Tunnel do Cloudflare só pode ter uma origem ativa por vez. Se você rodar o mesmo token no Air e no Mini simultaneamente, eles vão brigar pela conexão — um derruba o outro. A URL pública ficaria instável.

✅ Solução correta: cada máquina tem seu próprio tunnel

O tunnel do Mac Mini (JANA) fica como está — URL pública fixa para webhooks do n8n, EPROC, Lovable. O Air precisa do seu próprio tunnel separado, com URL diferente. Como o Air é o orquestrador (não o servidor de webhooks), ele pode até não precisar de tunnel — depende do uso.
