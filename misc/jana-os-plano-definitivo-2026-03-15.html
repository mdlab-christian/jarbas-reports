❓ Suas Perguntas — Respostas Diretas

🤖 Posso continuar usando o mesmo bot do Telegram independente do computador ligado?
Sim, sem nenhuma mudança. O bot Telegram é identificado pelo token — não pelo computador. Se os dois computadores usam o mesmo token, ambos recebem as mesmas mensagens. Com a arquitetura correta (Primary/Standby), um responde e o outro fica em espera. Se o Primary cair, o Standby assume automaticamente. Para você, nada muda — mesmo bot, mesma conversa.
💻 Mando do MacBook Pro pessoal — como funciona a distribuição?
Você não precisa saber qual máquina está ligada. Você manda mensagem pro bot — o OpenClaw Primary responde. Ele enfileira no Supabase. Os workers das duas máquinas consomem a fila automaticamente — quem estiver livre pega. Totalmente transparente para você.

👤 Preciso criar usuário separado no OpenClaw para cada máquina?
Não. O OpenClaw usa o mesmo workspace (~/.openclaw/) e a mesma conta. O que diferencia as máquinas é o JARBAS_HOST no .env — não o usuário. Uma conta, dois computadores, cada um sabe seu papel.
🔑 A API key do Claude pode ser a mesma nos dois computadores?
Sim, mesma key nos dois. A API Anthropic não limita por número de máquinas — limita por tokens/minuto e tokens/dia. Com dois computadores usando a mesma key, você dobra o uso simultâneo mas os limites são compartilhados. Na prática, com o volume atual do escritório Midas, você está longe do rate limit. Se um dia atingir, aí cria uma segunda key.

🔬 Diagnóstico do Código (investigado)

⚠️ Bug confirmado: task-consumer concurrency=1

Linha 326 do task-consumer.mjs diz literalmente: "Loop sequencial iniciado (concurrency=1)". Uma tarefa por vez. Lapidação rodando? Nenhuma outra executa até terminar. Este é o gargalo principal.

💡 Página /jarbas já tem infraestrutura de Mission Control

Investigei o código. A página já tem: aba Mission Control (tab 'jarbas'), aba Orchestrator, aba Monitor. Já busca da jarbas_tarefas e jarbas_execucoes. O problema: falta o campo preferred_host / executing_host na tabela — sem isso, não dá pra mostrar qual máquina está executando o quê. Com a migration, o Mission Control já existente vira funcional de verdade.

⚡ Upgrade JARBAS — Multi-tarefa Real

⚠️ Claude Code hoje: processo sequencial único

A skill_claude_code.mjs usa claude --print — spawn de processo filho descartável. Com task-consumer concurrency=1, só um processo Claude Code roda por vez. Com o upgrade para 3 workers paralelos, precisamos garantir que cada worker inicia seu próprio processo isolado.

U1
Task-consumer: concurrency=1 → 3 workers paralelos P1 · resolve o gargalo
// task-consumer.mjs — substituir runTaskLoop() sequencial por pool paralelo:
const MAX_CONCURRENT = 3;
const HOST = process.env.JARBAS_HOST || 'macmini';

async function runWorkerPool() {
  console.log(`[task-consumer] 🚀 ${MAX_CONCURRENT} workers · host=${HOST}`);
  await Promise.all(
    Array.from({ length: MAX_CONCURRENT }, (_, i) => runWorker(i + 1))
  );
}

async function runWorker(id) {
  while (true) {
    const { data } = await supabase.rpc('jarbas_consume_tarefa_host', {
      p_organizacao_id: ORG_ID,
      p_instance_id: `${INSTANCE_ID}-w${id}`,
      p_host: HOST
    });
    const tarefa = Array.isArray(data) ? data[0] : data;
    if (!tarefa?.id) { await sleep(POLL_INTERVAL); continue; }
    await executarTarefa(tarefa); // processo Claude Code isolado por worker
    await sleep(INTER_TASK_DELAY);
  }
}

🎯 Vai funcionar na prática?

Sim. A arquitetura é sólida porque usa o Supabase como broker central — as máquinas não precisam se conhecer diretamente, só consomem a mesma fila. O bug principal (concurrency=1) é uma linha de configuração. O MacBook Pro pessoal não muda nada — você continua mandando pro mesmo bot. A única complexidade real é garantir que apenas um OpenClaw responde o Telegram por vez — e o failover automático já é comportamento padrão do Telegram (o token funciona onde o processo estiver rodando).
