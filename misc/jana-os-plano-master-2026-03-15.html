❓ Perguntas Respondidas (todas)

🔴 O Telegram vai continuar travando?
Não, após o fix. O travamento tem duas causas:

1. task-consumer concurrency=1 → tarefas pesadas bloqueiam a fila inteira. Fix: pool de 3 workers paralelos (30min de código).
2. Skills pesadas rodando inline no OpenClaw → sessão do Telegram fica ocupada durante a execução. Fix: detectar automaticamente e enfileirar em vez de executar inline.

Após os dois fixes: JARBAS responde qualquer mensagem em <2 segundos. Tarefa pesada vai pra fila, resultado chega quando terminar. Telegram nunca trava.
🤖 A distribuição de tarefas vai ser inteligente?
Sim, automática. Lógica simples que funciona na prática:

— Cada worker pega a próxima tarefa disponível da fila (SELECT FOR UPDATE SKIP LOCKED)
— Se Air está ligado com 3 workers + Mini tem 3 workers = 6 executores competindo pela fila
— Tarefa com preferred_host=macmini só Mini pega. any = primeiro que estiver livre
— Sem scheduler central, sem coordinator — o Supabase resolve o locking automaticamente
— Se Air desligar: Mini pega tudo. Se Mini travar: Air pega tudo. Failover automático.
💻 JARBAS vai ter acesso aos arquivos do MacBook Air?
Sim, via SSH pelo Tailscale. Do Mini, acessa qualquer arquivo do Air:
ssh jarbas@100.101.208.82 "cat ~/qualquer/arquivo"

Posso criar uma skill skill_acesso_air.mjs que abstrai isso — você diz "lê o arquivo X do Air" e o JARBAS busca via SSH. O Air acessa o Mini da mesma forma.
💪 MacBook Air vai ser capaz de comandar todo o JARBAS?
Sim, completamente. Air terá todas as 54 skills, mesmo .env, mesmo acesso Supabase, mesmo acesso GitHub/n8n. Mais: acesso SSH direto ao Mini via Tailscale. Air pode executar skills localmente OU disparar skills no Mini via SSH.
🌐 Tenho uma rede de computadores trabalhando?
Sim, exatamente isso. Supabase = sistema nervoso central. Tailscale = rede privada segura. Os dois computadores são nós da mesma rede, compartilham a mesma fila, reportam heartbeat no mesmo banco. É um cluster de 2 nós com broker central.
✈️ Se Air estiver fora, sem problemas?
Zero problema. Mac Mini roda 24/7, tem todas as skills, responde Telegram sozinho. Os workers do Air simplesmente ficam ausentes da fila — Mini absorve tudo. Nenhuma tarefa perde, nenhum cron para.
🗂️ Qual o papel do Mac Mini nesse novo modelo?
Servidor permanente e executor principal.
— Sempre responde o Telegram (não tem Primary/Standby — Mini é o ponto fixo)
— Roda todos os crons (heartbeat, briefing, monitor n8n, importação noturna)
— 3 workers paralelos consumindo fila
— Air é o executor extra que aumenta capacidade quando ligado

Mudança em relação ao plano anterior: tirei Primary/Standby. Mini sempre responde Telegram. Mais simples, mais confiável.

🗄️ Mudanças no Supabase

Migration necessária (jarbas_tarefas):

-- Adicionar campos de roteamento dual machine
ALTER TABLE jarbas_tarefas
  ADD COLUMN IF NOT EXISTS preferred_host TEXT DEFAULT 'any',
  ADD COLUMN IF NOT EXISTS executing_host TEXT;
-- Valores: 'any' | 'macmini' | 'macbook'
-- executing_host: preenchido pelo worker quando pega a tarefa

-- Índice para performance do worker pool
CREATE INDEX IF NOT EXISTS idx_jarbas_tarefas_host_status
  ON jarbas_tarefas (preferred_host, status, prioridade DESC, created_at ASC)
  WHERE status = 'pendente';

Nova RPC para workers com filtro de host:

CREATE OR REPLACE FUNCTION jarbas_consume_tarefa_host(
  p_organizacao_id UUID,
  p_instance_id TEXT,
  p_host TEXT  -- 'macmini' ou 'macbook'
) RETURNS SETOF jarbas_tarefas LANGUAGE plpgsql AS $$
DECLARE v jarbas_tarefas;
BEGIN
  SELECT * INTO v FROM jarbas_tarefas
  WHERE organizacao_id = p_organizacao_id
    AND status = 'pendente'
    AND (preferred_host = 'any' OR preferred_host = p_host)
    AND (scheduled_at IS NULL OR scheduled_at <= NOW())
  ORDER BY prioridade DESC, priority_order ASC, created_at ASC
  LIMIT 1 FOR UPDATE SKIP LOCKED;

  IF v.id IS NULL THEN RETURN; END IF;

  UPDATE jarbas_tarefas SET
    status = 'em_andamento',
    iniciado_em = NOW(),
    instance_id = p_instance_id,
    executor = p_host,
    executing_host = p_host
  WHERE id = v.id;

  v.status := 'em_andamento';
  RETURN NEXT v;
END; $$;

🔍 Deep Review — Ponto a Ponto

✅
Arquitetura Supabase como broker central — Sólido. SKIP LOCKED garante que dois workers não pegam a mesma tarefa. Comprovado em produção por milhares de sistemas de fila. Funciona.
✅
Failover automático — Sólido. Air desliga = workers somem = Mini absorve a carga. Sem coordinator, sem heartbeat entre máquinas. Simples e confiável.
✅
Telegram sempre no Mini — Sólido. Mini nunca desliga, então Telegram nunca cai. Elimina completamente a complexidade de Primary/Standby.
✅
rsync para espelhar Air — Sólido. Tailscale já conectado, SSH funcional. 15 minutos de setup inicial, aliases para sync futuro.
⚠️
Tarefas inline → assíncronas — Trabalho real necessário. Precisa identificar cada skill que ainda roda inline e refatorar. Não é difícil, mas é ~2h de trabalho sistemático. Sem isso, o Telegram ainda pode travar para tarefas muito longas.
⚠️
Mission Control funcional — Depende da migration do Supabase. Sem executing_host na tabela, não dá para mostrar qual máquina está executando. Com a migration, o componente React existente precisa de atualização significativa — não é trivial, mas é claro.
✅
Mesma API key Claude nos dois — Funciona. Rate limit é compartilhado mas volume atual está longe do limite.
✅
Tunnel Cloudflare só no Mini — Correto. Air não precisa receber webhooks externos. Webhooks do n8n/EPROC continuam chegando no Mini como hoje.
✅
Gustavo usa Mini via SSH — Zero conflito. Workers dele não interferem com workers do JARBAS — cada um tem seu próprio processo.
⚠️
Sessão OpenClaw no Air — Atenção: não ligar o OpenClaw com Telegram ativo no Air. Se ligar, dois processos vão receber mensagens e dividir aleatoriamente. Air pode ter OpenClaw para desenvolvimento local, mas sem responder o bot Telegram.
