Problemas identificados e correções
⚠️
jarbas-reports não sincronizado entre Mini e Air
Mini tem 90 HTMLs. Air tem versão desatualizada e não há sync automático. Se a Jana gerar um relatório HTML, ele fica no Air mas o link GitHub Pages só funciona se for commitado. Correção: adicionar rsync de jarbas-reports no script sync-to-jana.sh (ou melhor: Jana commita direto no GitHub como o Mini faz).
⚠️
Air não tem Tailscale instalado
O Mini encontra o Air pelo IP local (192.168.15.81). Se você levar o Air pra outro lugar (fora de casa), o Mini perde o contato. Correção: instalar Tailscale no Air → garante conexão de qualquer lugar pelo IP 100.x.x.x.
⚠️
Arquivos que você salva no Air não são visíveis no JARBAS automaticamente
Se você salvar algo em ~/Desktop no Air e mandar pro JARBAS "está em ~/Desktop/arquivo.txt", o JARBAS vai ler via SSH. Funciona, mas só se ambos estiverem na mesma rede. Melhor abordagem: usar ~/jarbas/workspace/ como pasta compartilhada — qualquer arquivo ali fica acessível das duas máquinas via git ou rsync.
✅
Telegram: JARBAS no Mini, Jana no Air — abordagem correta
Dois bots separados evitam conflito de polling. O Mini é o servidor 24/7 e deve ser o canal principal. Jana é acionada diretamente quando você quer rodar algo no Air especificamente. Sem problema aqui.
✅
Segurança: SSH com chave, sem senha, loopback
OpenClaw gateway em loopback only. SSH com chave ed25519. Nenhuma porta exposta externamente. Token de auth no gateway. Seguro.
Quando usar JARBAS vs Jana

🌸 Use o JARBAS (90% do tempo)

  • Qualquer coisa com EPROC, OlivIA, processos
  • Monitoramento, heartbeat, n8n
  • Consultas rápidas, status, memória
  • Quando não importa onde roda — dispatcher decide
  • Quando o Air estiver desligado ou em uso pessoal
  • Comandos urgentes — Mini está sempre online

🦞 Use a Jana quando quiser:

  • Rodar algo pesado SEM travar o JARBAS
  • Lapidação longa enquanto JARBAS processa EPROC
  • Executar no contexto do Air (arquivos locais do seu Mac)
  • Segunda instância de Claude Code em paralelo
  • Tarefa que claramente não precisa do Mini (só IA pura)
  • Testar algo novo sem afetar o servidor principal
💡 Regra prática: Manda tudo pro JARBAS. O dispatcher encaminha automaticamente pro Air quando faz sentido. Só use Jana diretamente quando quiser controle explícito sobre onde roda.

✅ É viável controlar a fila de JARBAS e Jana pelo MdFlow?
Sim. A tabela jarbas_tarefas já tem todas as colunas necessárias (tipo, status, preferred_host, executing_host, prioridade). Uma interface React com Supabase Realtime consegue exibir e manipular a fila em tempo real, incluindo criar tarefas manualmente, cancelar, redirecionar entre Mini e Air, e ver o log de execução. Nenhuma nova API é necessária.
