Seção 2 — Resposta direta
📸 Prints vs Vídeo para Revisão de Páginas
Qual é a forma certa de documentar o estado de cada página antes de lapidar

✅ Recomendado: Screenshots Automáticos

JARBAS tira prints automaticamente antes e depois de cada lapidação usando Playwright ou screencapture nativo. Vantagens:

  • Automático — zero trabalho manual do Christian
  • Comparação visual antes/depois (diff visual)
  • Armazenado em Supabase Storage + local
  • Leve: 200-500KB por print
  • Pode fazer multiple breakpoints (desktop + mobile)
  • JARBAS analisa o print com Vision para verificar erros visuais
  • Link direto no briefing: "veja como ficou"

❌ Por que NÃO usar vídeo

  • Arquivo enorme: 50-200MB por página
  • JARBAS não consegue "ver" vídeo com Vision API
  • Lento de gravar + processar + enviar
  • Difícil de comparar before/after
  • Ocupa mais disco (já temos 13GB de PDFs vindOs)

🤖 Como JARBAS vai fazer automaticamente

// No início de cada lapidação (antes de tocar no código): 1. screencapture -w /tmp/lapida-before-[rota]-$(date +%s).png → envia para Supabase Storage → analisa com Vision: "descreva problemas visuais visíveis" → salva em screenshot_before na jarbas_tarefas // Após conclusão (nota ≥ 9, 3 loops feitos): 2. screencapture -w /tmp/lapida-after-[rota]-$(date +%s).png → envia para Supabase Storage → analisa com Vision: "a página está funcional? problemas visuais?" → salva em screenshot_after na jarbas_tarefas → gera link para o Christian ver o resultado

A coluna screenshot_before e screenshot_after já existem na jarbas_tarefas desde a migração de hoje.

Seção 3 — Nova Feature
🧠 Detecção de Overflow de Contexto
Como saber quando a conversa está perdendo contexto — e o que fazer

📊 Como funciona o contexto no Claude Sonnet

Claude Sonnet 4.6 tem 200k tokens de contexto. Em conversa típica com você:

Tipo de mensagemTokens estimados
Áudio transcrito (1min)~150-300 tokens
Resposta JARBAS (média)~500-1000 tokens
HTML inline (evitado agora)~5.000-30.000 tokens
System prompt + SOUL + MEMORY~8.000 tokens fixos
Conversa de 50 turnos normal~30.000-50.000 tokens
Sessão pesada (código + HTMLs)~100.000+ tokens

⚠️ O Problema Real

Quando o contexto está em 80%+, o Claude começa a:

  • Esquecer decisões tomadas no início da conversa
  • Repetir sugestões já descartadas
  • Perder referências a arquivos e commits antigos
  • Tornar-se mais lento (mais tokens = mais custo + latência)
  • Ter probabilidade de alucinação mais alta

Na sessão de hoje, depois do compaction, a conversa ficou muito mais rápida e precisa — é exatamente o efeito de limpar o contexto.

💡 Solução: skill_contexto_monitor

Contexto leve (<50k tokens): Conversa normal~25%
✅ Safe
Contexto médio (80-140k tokens): Começa a esquece~60%
⚠️ Atenção
Contexto pesado (160k+ tokens): Degradação real~85%
🔴 Limpar agora

🛠️ Como Implementar — Heurística Simples

JARBAS não consegue ver o count de tokens diretamente, mas pode usar heurísticas confiáveis:

// skill_contexto_monitor.mjs // Heurística: conta turnos + estima tokens por tipo const LIMITES = { turnos_aviso: 30, // Avisar após 30 turnos turnos_critico: 50, // Crítico após 50 turnos peso_audio: 300, // tokens/mensagem de áudio peso_texto: 500, // tokens/mensagem de texto peso_html_jarbas: 3000, // tokens/resposta com HTML peso_codigo: 2000, // tokens/resposta com código sistema: 8000, // tokens fixos (system prompt) limite_total: 200000, // Claude Sonnet 200k aviso_em: 0.75, // Avisar em 75% }; // JARBAS conta os turnos internamente via heartbeat // Quando atingir threshold → manda mensagem para Christian: const msg = ` ⚠️ *Contexto em ~80%* — esta conversa está longa. Opções: A) Compactar agora (salvo resumo + nova sessão) B) Continuar (próximos 10-15 turnos ok) C) Encerrar e começar nova conversa O que prefere? `; // Christian responde "A" → JARBAS salva resumo em memory/YYYY-MM-DD.md // e sugere começar nova conversa
