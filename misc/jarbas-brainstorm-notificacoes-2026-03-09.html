Seção 1 — Problema Principal
🔕 Redesign Completo de Notificações
Você está sendo spammado. Vamos consertar cada fonte de ruído.

🚨 O Problema Atual

JARBAS dispara notificações para cada R$0,50 gasto, cada variação de RAM, cada task da fila. Isso é ruído. Você precisa de signal, não noise. A regra de ouro: notificação boa é aquela que você não pode ignorar. Tudo abaixo desse limiar é silêncio.

Matriz de Notificações — Antes vs. Depois

Tipo Antes (problema) Depois (correto) Lógica
💰 Custo A cada R$0,50 gasto R$200+ por evento | R$500+ no dia Você gasta ~R$200-500/dia. Alertar cada centavo é spam.
🖥️ RAM Percentual errado, spammando ≥75% por >5min → aviso | ≥88% → crítico | só 1 alerta a cada 30min Pico de RAM é normal. Só importa se sustentado.
⚙️ Tasks da fila Notificação a cada subtarefa Só quando tarefa PRINCIPAL completa ou falha. Silêncio no meio. "Step 3 de 7 concluído" é ruído. "Lapidação /acordos concluída ✅" é signal.
🔄 n8n Monitor Alerta para qualquer workflow inativo Só 7 workflows críticos. Erro = alerta. Os outros: só no relatório noturno. 187 workflows inativos são arquivados, não quebrados. Não alertar.
💓 Heartbeat Às vezes vaza mensagem no chat HEARTBEAT_OK = silêncio total. Só mensagem se tiver problema real. Heartbeat é interno. Christian nunca deve ver HEARTBEAT_OK.
🏥 JARBAS Node.js Sem monitoramento = crash silencioso Se cair → alerta imediato + LaunchAgent reinicia automaticamente O crash de domingo não foi notificado. Isso não pode acontecer.
⚖️ EPROC Resultado de cada processo individualmente Só batch summary: "20 processos ✅ | 3 ⚠️ com erro" + lista dos com erro 80-100 distribuições/dia = 80-100 notificações se individual. Não.
📋 Relatórios diários Às vezes inconsistente ou com timing errado 06h | 13h | 23h — fixo. Relatório 23h = DUPLO (escritório + JARBAS saúde) Manter. Esses são valiosos — contexto que você quer receber.
🎯 Intimações novas — Prazo ≤ 2 dias úteis → alerta imediato. Resto → briefing 06h Prazo curto = urgência real. O resto pode esperar o briefing.

Semáforo de Saúde — Versão Correta

🟢 VERDE — silêncioRAM <75% | CPU <300% | claude_procs <3 | JARBAS rodando | n8n OK
🟡 AMARELO — aviso (1x, não repete antes de 30min)RAM 75-87% sustentado | claude_procs 3-4 | algum workflow crítico falhando
🔴 VERMELHO — alerta imediato + repetição a cada 10minRAM ≥88% | JARBAS Node.js offline | 2+ workflows críticos falhando | Custo ≥R$200 em uma sessão

🔄 Auto-Resurrect — JARBAS nunca fica offline

P0 — Fazer agora

O crash de domingo foi fatal porque não havia auto-restart. LaunchAgent + KeepAlive:true resolve 95% dos casos. Para os 5% restantes (erro de inicialização, .env corrompido), um segundo nível: cron a cada 5min verifica se o processo Node está rodando, se não está, tenta reiniciar e avisa Christian.

  • LaunchAgent com KeepAlive:true — reinicia imediatamente após crash
  • Cron de watchdog (bash script rodando no crontab do launchd) — backup do LaunchAgent
  • Se 3 tentativas falharem → alerta DM + log do erro para diagnóstico
  • Tempo de implementação: 20min

🧠 Smart Routing v2 — JARBAS sabe para onde mandar cada coisa

P0 — Core da visão

Hoje o routing é: texto livre → Claude, /comando → skill. Falta a camada de inteligência que detecta intenção e escolhe o agente certo automaticamente. Um áudio sobre lapidação vai para o Lapidador. Um áudio sobre processo vai para OlivIA. Sem precisar de prefixo.

  • Classifier Haiku (barato, rápido) analisa a intenção da mensagem
  • Categorias: DEV | JURIDICO | ESCRITORIO | PESSOAL | ADMIN
  • Cada categoria tem roteamento padrão para o agente correto
  • Christian pode override: "cc: [tarefa]" força Claude Code, "olivia: [tarefa]" força OlivIA

🎯 Modo Missão — execução paralela inteligente

P1 — Alinha com visão

Christian quer tudo feito ao mesmo tempo, não sequencial. O Mission Orchestrator atual é primitivo. Proposta: quando Christian manda uma ideia complexa, JARBAS automaticamente decompõe em tarefas paralelas, usa o limite de RAM disponível para calcular quantas pode rodar ao mesmo tempo, e despacha tudo.

  • Analisa ideia com Haiku (barato) → gera DAG de dependências
  • Calcula concorrência segura baseado em RAM disponível
  • Despacha tarefas independentes em paralelo, sequencia dependentes
  • Dashboard no Telegram: "/missao" mostra progresso em tempo real
  • Christian pode pausar/cancelar tarefas individuais via resposta

🔮 JARBAS Jurimetrista — aprende com cada processo

A visão: quando OlivIA está gerando uma petição contra o Banco do Brasil na Comarca de Canoas, JARBAS automaticamente injeta: "taxa de sucesso 68%, juíza X é favorável a tutelas antecipadas, usar argumento Y que funcionou em 12 casos similares".

🗂️ Grupos Telegram com Contexto por Página

P1 — QOL importante

A confusão atual entre grupos é por contexto compartilhado. A solução elegante: cada grupo Telegram corresponde a um "contexto de trabalho" com estado persistente. JARBAS sabe que no grupo /olivia o contexto é sobre a sprint OlivIA, no /dev é sobre lapidações.

  • Cada grupo tem um "contexto ativo" armazenado no Supabase
  • Mensagens de um grupo não vazam para o contexto de outro
  • "/contexto" mostra qual contexto está ativo no grupo
  • "/reset-contexto" limpa o estado do grupo atual

Seção 5 — Otimização Financeira
💰 Otimização de Custos de IA
Você gasta bem. Mas há desperdícios identificáveis.

🔴 Desperdícios Identificados

  • Usar Sonnet para tasks que Haiku resolve (routing, status, ping)
  • Context longo desnecessário em skills simples
  • MEMORY.md acima de 4000 chars → mais tokens por request
  • Claude Code sem cache → repaga contexto todo request
  • Múltiplas lapidações simultâneas com Sonnet (4+ = ~R$200/rodada)

🟡 Otimizações Rápidas

  • Mover routing/classify para Haiku (10x mais barato)
  • Cache de prompts longos recorrentes (SOUL.md, MEMORY.md)
  • Batch de tasks similares (ex: 20 análises jurídicas → 1 request multi)
  • MEMORY.md slim: manter <3000 chars, resto em arquivos referenciados
  • Limitar lapidações simultâneas: máximo 2 Sonnet + 2 Haiku
