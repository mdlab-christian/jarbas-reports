🔴 Problemas Identificados
P0

Playwright derruba o processo inteiro quando browser crasha

cerebro_documental e captura_intimacoes usam Playwright intensivamente. Quando o browser fecha inesperadamente (Target crashed, page.waitForTimeout: Target page closed), o erro sobe sem ser capturado e mata o worker pool → JARBAS reinicia.

Fix aplicado hoje (parcial): guard uncaughtException + worker pool auto-restart em 5s. Mas o root cause (browser pool sem health check) ainda existe.
P0

Processos zumbi após restart

Quando o PM2 reinicia o JARBAS, o processo velho não morre — fica como zumbi com PPID=1. Dois processos rodando simultâneos causam conflito de porta, duplicação de polling e memory overhead (~400MB desnecessário).

Fix aplicado hoje (parcial): lsof com path completo /usr/sbin/lsof. Mas falta script de startup que mata processos node com mesmo argv.
P0

OpenClaw gateway cresce sem limite (904MB hoje)

O gateway acumula sessões em memória sem eviction. Sessões de grupos de lapidação (dceaf9b2: 801KB) ficam ativas indefinidamente. Com muitas sessões simultâneas, o processo consome >900MB e fica com 119% CPU.

Fix sugerido: Cron a cada 30min que reinicia o gateway se memória >600MB — e limpa sessões >400KB que não tiveram atividade nas últimas 2h.
P1

enriquecer_processo com 59% de cancelamentos (111 de 188)

Quase 2/3 das tarefas de enriquecimento são canceladas. Esse é o maior desperdício de recursos — Playwright abrindo browser e navegando por processos que acabam cancelados antes de concluir.

Investigar: timeout muito curto? Fila sendo esvaziada por restart? Tasks duplicadas?
P1

5 captura_intimacoes + 2 cerebro_documental presas em "pendente"

Tasks criadas às 19:14 ainda pendentes às 18:58. O task-consumer não está pegando — provável crash do worker no momento da criação.

Fix imediato: immune_system deveria pegar essas tasks, mas parece não estar rodando.
P1

Contexto de grupos perdido na auto-limpeza

Auto-compact estava atrapando contexto dos grupos a cada 10 turnos. A conversa da OlivIA foi perdida hoje cedo. Desativado hoje mas o problema de contexto pesado ainda existe.

Melhor abordagem: salvar contexto estruturado automaticamente a cada N turnos (sem apagar histórico), e reinjetar no próximo restart.
P2

Sem health check automático do próprio JARBAS

Quando o JARBAS trava, Christian descobre por comportamento anômalo no chat. Não há monitoramento proativo que detecte crash, processo zumbi ou memória alta e notifique antes de virar problema.

P2

Browser pool sem reconexão inteligente

Quando um contexto Playwright morre no meio de uma task, o browser-pool ainda mantém a entrada inválida. A próxima task tenta reusar e crasha de novo.

💡 Melhorias Sugeridas — Baseadas no Uso Real
🛡️

M1 — Browser Pool com Health Check + Auto-Recovery

Antes de reusar uma sessão Playwright do pool, verificar se o contexto ainda está vivo (browser.isConnected()). Se morto, invalidar e criar novo. Elimina a maioria dos crashes de cerebro_documental e captura_intimacoes.

⏱ Esforço médio (2-3h) · Impacto: elimina crashes diários de Playwright
🧹

M2 — Startup Killer: mata processos node zumbis do JARBAS

No início do start.mjs, matar todos os processos node com o mesmo script (/jarbas/src/start.mjs) exceto o PID atual. Resolve o problema de dois processos rodando simultaneamente após restart.

⚡ Esforço baixo (30min) · Impacto: elimina duplicação de processo e conflito de porta
📊

M3 — Self-Monitor no Heartbeat com Alertas Inteligentes

A cada heartbeat, verificar: (1) processo zumbi JARBAS, (2) OpenClaw >600MB → reiniciar gateway, (3) tasks presas >30min → reprocessar, (4) taxa de erro das últimas 10 tasks >30% → alertar Christian. Hoje esse trabalho é manual.

⏱ Esforço médio (2h) · Impacto: detecta problemas antes de virar crash
🔍

M4 — Investigar e corrigir enriquecer_processo (59% cancelados)

111 tasks canceladas em 188 totais é um desperdício enorme de Playwright. Diagnóstico: verificar se o timeout de 90min está sendo respeitado, se há tasks duplicadas sendo criadas, ou se o cancelamento é forçado por restart.

⚡ Esforço baixo (1h diagnóstico) · Impacto: reduz carga Playwright e aumenta dados enriquecidos
🧠

M5 — Contexto de Grupo Persistente Automático

Salvar checkpoint de contexto estruturado a cada 5 turnos por grupo (sem apagar histórico). Ao reiniciar, reinjetar apenas o checkpoint — não o histórico completo. Resolve perda de contexto sem inflar o token count.

⏱ Esforço médio (2h) · Impacto: nunca mais perder contexto de lapidação em andamento
⚡

M6 — Fila de Prioridade para Tasks

Hoje todas as tasks têm a mesma prioridade. Sugestão: HIGH (distribuir, captura_intimacoes), MEDIUM (cerebro_documental, importar), LOW (enriquecer, analisar_ia). Workers reservados por tier: 2 HIGH, 2 MEDIUM, 1 LOW. Evita que cerebro_documental pesado bloqueie captura urgente.

🔧 Esforço alto (4-6h) · Impacto: responsividade muito melhor em picos de carga
🔄

M7 — Gateway Auto-Restart quando >600MB

Cron a cada 15min: verificar memória do processo openclaw-gateway. Se >600MB, reiniciar gateway e notificar Christian. Hoje chegou a 904MB com 119% CPU antes de alguém perceber.

⚡ Esforço baixo (30min) · Impacto: elimina degradação gradual do gateway
🎯

M8 — MODO ARCHON (Opus 4.6) — ainda não implementado

Está em P1 desde 12/03. Triggers: archon: [tarefa], lapida /pagina archon. Para páginas complexas como OlivIA C2b (10-12h de trabalho), Opus 4.6 faria diferença real.

⏱ Esforço médio (2h) · Impacto: qualidade 30-40% maior em lapidações complexas
🗄️

M9 — ChromaDB/RAG populado para consultas processuais

Ainda não populado. Com 5.800 processos importados + cerebro_documental processando documentos, o RAG seria extremamente valioso para: "qual juiz costuma deferir em casos de negativação por banco X?", "qual o histórico deste cliente?", etc.

🔧 Esforço alto (1 dia) · Impacto: novo nível de inteligência jurídica
📱

M10 — Dashboard de Status em Tempo Real

Hoje o /status é textual e você precisa pedir. Um HTML auto-atualizado (GitHub Pages + fetch a cada 30s) mostrando: processos PM2, tasks em fila, últimos erros, memória OpenClaw, EPROC status. Acesso rápido sem precisar perguntar.

⏱ Esforço médio (3h) · Impacto: visibilidade total sem fricção
