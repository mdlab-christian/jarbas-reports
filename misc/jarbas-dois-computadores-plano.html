Ganhos imediatos

O que muda
com dois computadores.

Não é só "dobrar capacidade" — é resolver os problemas reais que você tem hoje.

Problema hojeCom dois computadores
JARBAS travando quando roda EPROC + lapidação + chat ao mesmo tempo Mac Mini roda EPROC e n8n sozinho. MacBook livre para você.
Mac Mini vai ser usado pela equipe jurídica — risco de conflito Mac Mini dedicado para operação. MacBook dedicado para você.
Briefings automáticos dependem do Mac Mini estar disponível Mac Mini roda tudo 24/7. Seu MacBook só recebe as notificações.
Análises pesadas de vídeo travam o chat MacBook Prime delega análise de vídeo para Mac Mini processar.
Quando você está viajando, EPROC não monitora Mac Mini roda independente de onde você está.
Claude Code + JARBAS ao mesmo tempo = lentidão Cada instância roda na sua própria máquina, sem concorrer.

💡 O ganho real não é processamento — é isolamento

Você não precisa do dobro de RAM para ter o dobro de capacidade. O que você precisa é isolar o que é crítico (EPROC, n8n, OlivIA) do que é interativo (seu chat pessoal, planejamento, desenvolvimento). Com dois computadores, uma tarefa pesada no Mac Mini não afeta sua conversa no MacBook. É o mesmo princípio de microsserviços: separar responsabilidades.

Arquitetura de delegação

Como os dois
se comunicam.

JARBAS Prime no MacBook sabe que pode delegar para o Mac Mini. O mecanismo é simples: uma API interna que o Mac Mini expõe via túnel seguro.

🔗

Opção A — Delegação via SSH + arquivo de tarefa

JARBAS Prime escreve uma tarefa em `~/jarbas/queue/task.json` e envia via SSH para o Mac Mini. O JARBAS Ops monitora a pasta, executa, devolve o resultado em arquivo. Simples, sem dependência externa.

  • ✅ Sem porta exposta na internet
  • ✅ Usa infraestrutura que já existe (Cloudflare tunnel)
  • ✅ 0 latência — direto por SSH
  • ⚠️ Delegação é assíncrona (não tempo real)
Recomendado para fase 1
🌐

Opção B — API REST interna (JARBAS Bridge)

Mac Mini expõe uma API Express interna (`localhost:3099`) via Cloudflare Tunnel. JARBAS Prime faz POST com a tarefa e recebe um job ID. Mac Mini processa e notifica via Telegram quando pronto.

  • ✅ Delegação síncrona e assíncrona
  • ✅ JARBAS Prime pode verificar status em tempo real
  • ✅ Base para a rede de Mac Minis futura
  • ⚠️ Precisa de 2–3 dias para implementar

Sincronização e memória

Como os dois
ficam alinhados.

O maior desafio de dois agentes é manter consistência de contexto. A solução usa o que já existe: Supabase + GitHub.

🧠

Memória via Supabase

Ambos JARBAS leem e escrevem na mesma tabela `jarbas_memoria` no Supabase. O que um aprende, o outro sabe. Decisões, contexto de projetos, estado de missões — tudo compartilhado em tempo real.

Já funciona hoje
📁

Workspace via GitHub

O repositório `jarbas` (SOUL.md, skills, MEMORY.md) fica no GitHub. Ambas as máquinas sincronizam com `git pull` antes de executar tarefas. Configuração muda num lugar, propaga para os dois.

Já funciona hoje
🔄

Estado de missão via arquivo

Missões longas (lapidação, importação em lote) salvam estado em `memory/project-*.md` que é commitado e disponível para ambas as máquinas. Se o MacBook cai, o Mac Mini pode retomar de onde parou.

A pergunta principal

Mesmo sistema ou
sistemas separados?

Você perguntou exatamente a coisa certa. A resposta depende do que você quer agora versus no futuro.

✅

Recomendação: mesmo código, instâncias separadas

Ambas as máquinas rodam o mesmo repositório jarbas/. Mesmas skills, mesmo SOUL.md, mesmo conhecimento. A diferença está no SOUL.md de cada uma — Prime tem papel de orquestrador e assistente pessoal, Ops tem papel de executar operações jurídicas.

  • ✅ Uma atualização de skill vai para os dois via git pull
  • ✅ Mesmo MEMORY.md (Supabase ou GitHub sync)
  • ✅ Mais fácil de manter
  • ✅ Escalável para N máquinas com o mesmo padrão
Esse é o caminho
❌

Por que não sistemas completamente diferentes

Separar tudo parece mais limpo, mas na prática cria o maior problema que você pode ter: o JARBAS de um computador não sabe o que o outro fez. Decisões, contexto, memória — tudo duplicado e dessincronizado.

  • ❌ Sincronização manual constante
  • ❌ Atualizar skill em dois lugares diferentes
  • ❌ Memória fragmentada
  • ❌ Não escala para a rede futura

Passo a passo

Quando o MacBook chegar —
o que fazer primeiro.

1

Instalar o ambiente base no MacBook

Homebrew → Node.js → OpenClaw (npm install -g openclaw) → Claude Code. Clonar o repositório jarbas do GitHub. Copiar o arquivo .env do Mac Mini (mesmas credenciais). Configurar OpenClaw com o mesmo Telegram bot (ou um bot separado — a definir).

⏱️ 30–60 min
2

Criar o SOUL.md do JARBAS Prime

Criar SOUL-prime.md com a identidade do MacBook: orquestrador, assistente pessoal, não faz EPROC diretamente. Ele sabe que tem um Mac Mini disponível e sabe delegar. Incluir a variável de ambiente JARBAS_ROLE=prime para o agente saber quem ele é.

⏱️ 20 min
3

Testar comunicação MacBook ↔ Mac Mini

Via Cloudflare Tunnel (que o Mac Mini já tem): SSH do MacBook para o Mac Mini. Teste simples: JARBAS Prime manda ssh jarbas-ops "node ~/jarbas/src/start.mjs --task ping" e verifica se recebe resposta. Isso confirma que a delegação básica funciona.

⏱️ 30 min
4

Configurar sync de memória entre as duas máquinas

Criar um cron no Mac Mini que commita e faz push do MEMORY.md a cada 30 minutos. MacBook faz git pull antes de iniciar qualquer sessão. A memória de longo prazo fica sincronizada via GitHub. Simples e confiável.

⏱️ 15 min
5

Implementar skill_bridge.mjs (delegação inteligente)

A skill que faz o JARBAS Prime delegar automaticamente para o Mac Mini. Regras de decisão: tarefa EPROC? → Mac Mini. Uso de CPU > 80%? → Mac Mini. Tarefa > 10min estimados? → Mac Mini. Respostas curtas/chat? → Fica no MacBook.

⏱️ 1–2 dias
