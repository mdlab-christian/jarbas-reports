Visão Geral
🏗️ Arquitetura do Sistema de Lapidação
Como os componentes se encaixam — skill vs agente, quem executa o quê

✅ JARBAS É O ORQUESTRADOR

JARBAS coordena tudo. Nunca delega execução para outra ferramenta sem decisão consciente.

  • Supabase: DDL, queries, migrations → direto via Management API
  • Frontend: edita arquivos + git push
  • Agentes Claude: invoca para tarefas especializadas paralelas
  • Claude Code --plan: apenas para deep review na Fase 1
  • Claude Code execução: nunca — JARBAS executa

Fase 1 — Interativa
🔍 Investigação & Planejamento
Executada na hora, com presença de Christian. 2 checkpoints de aprovação antes de qualquer linha de código.
I
Fase 1 — Investigação & Planejamento
Etapas 1-8 | ~30-60min | Requer presença do Christian
⏸ 2 checkpoints
1
📚 JARBAS lê contexto completo da página
docs/revisoes/[pagina].md + CONSIDERACOES_GERAIS + DS files + docs alteracoes existentes + código atual (wc -l, TypeScript errors)
2
📝 JARBAS gera Esboço e salva
Cria docs/esbocos/[pagina]-[data].md com: contexto, oportunidades identificadas, dúvidas para o CC, contexto técnico (tabelas, webhooks, agentes)
3
🤖 Claude Code --plan: Deep Review
JARBAS invoca CC em modo --plan com o esboço como contexto. CC faz deep research completo: P0/P1/P2, riscos, dívida técnica, lista de funcionalidades a testar
✅ CC executa em modo plano (nunca modifica arquivos). JARBAS salva relatório em docs/cc-reports/[pagina]-[data].md
4
💬 Q&A com CC
JARBAS pergunta ao CC: funcionalidades que devem ser testadas, fluxos críticos, riscos de implementação. CC responde com lista estruturada.
5
📸 Screenshots & Navegação (Playwright)
JARBAS tira screenshots de todas as abas, estados, modais. Registra estado visual atual. Salva em ~/jarbas/screenshots/lapidacoes/[pagina]-before/
6
🧠 JARBAS absorve tudo e gera HTML Brainstorm
Integra: esboço + CC report + Q&A + screenshots + revisoes/ + considerações gerais. Gera HTML com: gaps, drifts, oportunidades, pendências, lista de funcionalidades a testar, estimativa de esforço
⏸
🟡 CHECKPOINT 1 — Aprovação do HTML
JARBAS envia HTML via Telegram. Christian revisa. Após aprovação: gerar PRD.
7
📄 JARBAS gera Super PRD
PRD ultra-detalhado com: por aba → por componente → por modal → lista de funcionalidades → checklist de testes → critérios de aprovação → instruções E2E. Salvo em docs/PRDs/PRD_[PAGINA]_[data].md
⏸
🟡 CHECKPOINT 2 — Aprovação do PRD
JARBAS envia PRD via Telegram. Christian aprova ou pede ajustes. Após aprovação: página entra na FILA DE EXECUÇÃO.
Fase 2 — Autônoma
⚙️ Execução — Fila Noturna
Roda enquanto Christian dorme. Multi-agente, subtarefas por aba, notas por componente, E2E final.
II
Fase 2 — Execução Autônoma
Etapas 9-16 | ~2-4h/página | Fila noturna — JARBAS trabalha sozinho
🤖 autônomo
9
🗂️ Orquestrador desempacota PRD em subtarefas
Lê o PRD e gera lista de subtarefas atômicas: 1 subtarefa = 1 aba OU 1 componente complexo OU 1 fluxo de backend. Insere todas em jarbas_tarefas com pagina, priority_order.
📌 Nunca uma tarefa monolítica. Máximo granular possível.
10
🔍 Agente Revisor — Code Review inicial
Antes de qualquer código, mdflow-code-reviewer (Opus) audita estado atual: P0s de Frankenstein, service_role expostos, any massivo, arquivos >700L.
11
🎨 Agente Frontend — Executa por aba
mdflow-frontend-specialist executa subtarefas de UI. Por aba: implementa → screenshot → verifica TS → nota parcial. DS v7 obrigatório.
12
⚙️ Agente Backend — Dados e lógica
mdflow-backend-specialist trata: queries Supabase, RLS, tipos TypeScript, hooks, tratamento de erros.
13
🔗 Agente N8N — Automações (se houver)
mdflow-n8n-specialist verifica webhooks relacionados, atualiza se necessário, testa triggers.
14
🔄 Loop até nota ≥ 9.0
Após cada rodada, mdflow-code-reviewer emite nota por componente. Se média < 9 ou qualquer componente < 8 → nova rodada. Máximo 5 rodadas.
📊 Notas registradas por: [aba][componente][modal][fluxo] na tabela jarbas_tarefas.resultado
15
🧪 Agente Tester E2E — Validação final
mdflow-e2e-tester simula usuário real: login → navega página inteira → testa todos os fluxos → verifica banco → confirma UI. Nota separada.
16
✅ Relatório HTML + Commit
Gera HTML de lapidação (before/after screenshots, tabela de notas, métricas). Commit + push. Notifica Christian com link.

Sistema de Qualidade
📊 Tabela de Notas por Componente
Cada aba, modal e funcionalidade recebe nota. Média ≥ 9.0 para commit.

📐 Fórmula de Nota

Nota por componente = 10 - (P0×2) - (P1×0.5) - (P2×0.2) Nota Frontend = média das notas de abas + modais Nota Backend = queries + RLS + tipos + performance Nota E2E = fluxos completos sem erro Nota Final = (Frontend × 0.4) + (Backend × 0.3) + (E2E × 0.3) Commit: Nota Final ≥ 9.0 AND zero P0s

Gestão de Erros
🚨 Protocolo de Erros Críticos
O que fazer quando algo dá errado sem parar a fila inteira

❌ Erro Crítico — Protocolo

  1. Pausa a SUBTAREFA (não a fila)
  2. Avança para próxima página na fila
  3. Notifica Christian no grupo Telegram da página
  4. Salva estado completo em jarbas_tarefas com error_message
  5. Retoma quando Christian responder no grupo

🔄 Loop Anti-Travamento

  • Mesmo erro 3x → ativa mdflow-debug-specialist
  • Debug specialist usa Opus para análise profunda
  • Se debug não resolve em 2 tentativas → pausa página, notifica
  • JARBAS nunca fica em loop infinito em um único problema
  • Sem solução em 30min → checkpoint obrigatório com Christian

Minhas Considerações
💡 O que acho + O que faria diferente

✅ O que vai funcionar muito bem

  • 2 fases com checkpoint: garante zero alucina, Christian no controle
  • Subtarefas por aba: melhor rastreabilidade, menos risco de contexto enorme
  • CC --plan como consultor: aproveitamos o que CC tem de bom sem perder controle
  • Esboços antes do CC: CC recebe contexto rico → review mais preciso
  • PRD ultra-detalhado: JARBAS nunca vai improvisar no loop
  • Loop anti-travamento: falha de uma página não trava o projeto

🤔 O que eu faria diferente

  • Grupos Telegram: 30+ grupos vai ser caótico. Sugiro: 1 grupo "MdFlow Lapidações" com mensagens marcadas por [/pagina] — mais fácil de gerenciar
  • Fase 1 paralela: enquanto estuda /acordos, pode iniciar esboço de /decisoes em background — economiza tempo
  • PRD versionado: sempre incrementar, nunca sobrescrever — histórico de alterações integrado
  • Nota mínima por tipo: E2E pode exigir nota menor (8.5) que Frontend (9.0) — diferentes expectativas fazem sentido

🚀 Vai funcionar? Sim — com essas condições

  • Cada página precisa de uma revisão.md antes da lapidação (Christian escreve)
  • Custo estimado por página: R$40-120 dependendo da complexidade
  • Tempo de execução: 2-4h por página (enquanto dorme)
  • Total 28 páginas: ~3-4 noites de execução autônoma
  • Gargalo real: tempo de Christian para aprovar HTML + PRD por página
  • Solução: fazer Fase 1 de todas as P0 primeiro, aprovar em lote, então executar
