🔍 1. Diagnóstico — O que está acontecendo

🔴 Ciclo de reinicializações em cascata (root cause primário)

O gateway OpenClaw cresce de memória continuamente até atingir o threshold (~900–1200MB). O heartbeat detecta isso e reinicia. Mas o processo não é instantâneo — antes do restart terminar, o heartbeat detecta memória alta novamente e tenta reiniciar de novo. Resultado: 22 restarts em menos de 24h.

19:30 — Gateway em 716MB → restart #1
19:30:57 — Gateway em 717MB → restart #2 (apenas 20s depois!)
19:31:08 — Gateway em 745MB → restart #3 (apenas 11s depois!)
20:06–20:07 — 6 restarts em 2 minutos (cascata total)
22:10–22:36 — Cooldown funcionou, mas gateway voltou a subir

🟠 Memory leak persistente no OpenClaw gateway

O gateway está com contextPruning: off e 43 sessões acumuladas (80MB total). A cada nova conversa, o contexto cresce mas nunca é podado. O gateway carrega todas as sessões ativas na memória → leak gradual e previsível.

🐛 2. Root Causes identificados (7 problemas)

RC-01 — session-cleaner TDZ bug (recorrente)

CRÍTICO

O erro Cannot access 'result' before initialization apareceu 17+ vezes desde ontem. Isso significa que a skill cleanSessions() tem uma variável result usada antes de ser inicializada internamente. O fix do import fora do setInterval foi feito corretamente, mas o bug está dentro da função.

// Dentro de skill_session_cleaner.mjs, provavelmente:
const result = await someAsyncFunc();  // ← se isso lança, result não é inicializado
doSomething(result);  // ← TDZ aqui

⚠️ Como o cleaner está falhando silenciosamente, as sessões pesadas NÃO estão sendo limpas → gateway acumula memória

RC-02 — Threshold de restart muito baixo (já ajustado mas insuficiente)

ALTO

O threshold foi aumentado de 600→900→1200MB, mas o Mac Mini M4 tem 16GB. O problema real não é o threshold — é que a memória continua subindo porque o cleaner não funciona (RC-01). Mesmo com threshold em 1200MB, o gateway chega lá em algumas horas.

RC-03 — contextPruning OFF + 200k tokens

ALTO

Sem pruning automático, cada sessão cresce indefinidamente. Com contextTokens: 200000 (200k) e 43 sessões abertas, o potencial teórico é enorme. O OpenClaw carrega contexto de sessões ativas na RAM → crescimento linear com o uso.

RC-04 — Race condition no restart loop

ALTO

O cooldown de 30min é salvo em /tmp/jarbas-gw-restart-cooldown ANTES do restart, mas quando o restart acontece e o processo recomeça, a RAM não cai imediatamente. O heartbeat roda 20–30s depois e vê memória ainda alta → tenta reiniciar de novo. O cooldown evita isso, mas havia casos onde o arquivo /tmp era limpo no boot.

RC-05 — schema mismatch: intimacoes.processo_cnj

MÉDIO

Skills skill-verificar-prazos e prazo-guardian estão consultando a coluna processo_cnj na tabela intimacoes, mas essa coluna não existe. Causa erros a cada execução (7h30, 8h, 12h) e pode estar travando workers.

ERROR: column intimacoes.processo_cnj does not exist

RC-06 — Cérebro Documental crashando Playwright

MÉDIO

O cron-cerebro roda a cada 10min para 3 OABs (RS116571, SC074025, SP535101). Quando o gateway está com memória alta, o Playwright crashava durante page.evaluate. Isso é sintoma, não causa — mas contribui para o consumo de RAM.

❌ Target crashed
❌ Execution context was destroyed, most likely because of a navigation

RC-07 — CEREBRO_INTERVAL de 10min é muito agressivo

BAIXO

O Cérebro Documental cria tarefas para 3 OABs a cada 10min. Cada tarefa abre um browser Chromium. Com múltiplas OABs e intervalo curto, acumula browsers e memória rapidamente, especialmente se uma tarefa anterior ainda está rodando.

⚖️ 3. JARBAS vs JANA — Por que ela não reinicia

A JANA não reinicia pelos seguintes motivos (inferidos da arquitetura):

AspectoJARBAS (instável)JANA (estável)
Carga de trabalho EPROC + OlivIA + Cérebro + Crons pesados Frontend only (lapidações, revisões)
Playwright/Browser Múltiplos browsers 24/7 (EPROC + Cérebro) Ocasional ou nenhum
Workers simultâneos 4 workers + crons independentes Menor concorrência
Sessões OpenClaw 43 sessões, 80MB, sem pruning Menos sessões ativas
Heartbeat aggressivo Reinicia gateway automaticamente Sem auto-restart do gateway
Conclusão JARBAS faz muito mais — o problema não é fraqueza, é falta de otimização de recursos

🔧 4. Plano de Correção — 12 Fixes Priorizados

⚡ IMEDIATO

FIX-01 — Corrigir TDZ em skill_session_cleaner.mjs

Este é o fix mais urgente. Com o cleaner quebrando a cada 30min, as sessões nunca são limpas e o gateway acumula memória indefinidamente.

Ação: Adicionar wrapper try/catch robusto com inicialização antecipada de result.

skill_session_cleaner.mjs — correção do TDZ
- const result = await someRiskyFunction();
- doSomething(result);
+ let result = { cleaned: 0, savedKB: 0 }; // inicializar antes
+ result = await someRiskyFunction();
+ doSomething(result);

📍 Arquivo: ~/jarbas/src/skills/skill_session_cleaner.mjs

⚡ IMEDIATO

FIX-02 — Habilitar contextPruning no openclaw.json

Com pruning desligado, cada sessão cresce indefinidamente. Habilitar o pruning automático vai reduzir drasticamente o uso de RAM do gateway.

~/.openclaw/openclaw.json — contextPruning
- "contextPruning": {
- "mode": "off",
+ "contextPruning": {
+ "mode": "auto",
"keepLastAssistants": 3,
}
⚡ IMEDIATO

FIX-03 — Reduzir contextTokens de 200k para 100k

200k tokens por sessão é excessivo para o JARBAS. A maioria das conversas usa menos de 50k. Reduzir para 100k diminui o uso de RAM pela metade sem impactar funcionalidade.

~/.openclaw/openclaw.json — contextTokens
- "contextTokens": 200000,
+ "contextTokens": 100000,
⚡ IMEDIATO

FIX-04 — Habilitar memoryFlush no openclaw.json

O flush de memória estava desabilitado. Habilitar força o garbage collector a liberar memória periodicamente.

- "enabled": false
+ "memoryFlush": {
+ "enabled": true
}

FIX-07 — Aumentar CEREBRO_INTERVAL de 10 para 30 min

O Cérebro roda a cada 10min para 3 OABs. Cada OAB = 1 browser. 10min é muito agressivo e acumula processos Chromium quando um ciclo ainda está rodando.

~/jarbas/.env

+ CEREBRO_INTERVAL=30

Isso reduz de 144 ciclos/dia para 48 ciclos/dia — ainda suficiente para capturar novos documentos.

3 sessões Claude concorrentes consomem muita RAM. O heartbeat já detectou múltiplos processos Claude simultâneos como risco de OOM. Reduzir para 2 como proteção.
