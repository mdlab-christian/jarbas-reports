Componente 3 — ClienteDrawerTabDocumentos
ClienteDrawerTabDocumentos.tsx
~70 linhas · usado no ClienteDrawer do CRM · mostra documentos do cliente
P0 conceitual
Problema crítico de arquitetura
❌
Lê tabela ERRADA: Usa cliente.documentos (campo JSON legado) e lead.documentos — NÃO usa a tabela processos_documentos. O sistema novo de docs não aparece aqui.
❌
Sem processo associado: O ClienteDrawer não tem contexto de processo_id. Não dá para gerar docs sem saber qual processo.
⚠️
Sem botão "Gerar": Tab puramente visualização de docs legados (JSON). Não tem ação de geração.
✅
Drive link ok: Mostra pasta Drive do cliente quando existe.
Decisão arquitetural necessária
O ClienteDrawer recebe cliente ou lead, não um processo. Um cliente pode ter N processos.

Opção A (recomendada): Tab mostra lista de processos do cliente. Para cada processo: mini-card com status dos 4 docs + botão "Gerar" por tipo. Seleciona processo → gera doc.

Opção B (simples): Tab mantém docs legados + adiciona seção "Documentos por Processo" buscando processos_documentos WHERE cliente_id.

Opção C: Remover tab Documentos do ClienteDrawer. Deixar docs só no ProcessoDrawer.

⚠️ Decisão específica: ModalGerarPeticao vs Tab Documentos
A distribuição tem dois pontos de geração de petição:
1. TabPreparacao: ModalGerarPeticao (modal dedicado com UX rico — progress, confetti)
2. TabAguardando: ProcessoDrawer → Tab Documentos (futura, com botão Gerar)

Recomendação: Manter ambos. TabPreparacao mantém o ModalGerarPeticao (UX rica para primeiro geração). TabAguardando usa ProcessoDrawer Tab Documentos (para regeneração e outros docs). JARBAS deve atualizar ambos: processos_documentos + processos.peticao_status.
