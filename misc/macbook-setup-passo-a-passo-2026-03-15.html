⚠️ Ponto Crítico: Dois OpenClaw NÃO podem responder ao mesmo tempo

Se ambas as máquinas tiverem OpenClaw ativo respondendo o mesmo bot Telegram, as duas vão receber e responder cada mensagem — duplicando respostas e criando conflito de estado. A solução: apenas uma máquina é PRIMARY por vez. O Mac Mini fica em standby quando o Air está ligado.

💡 Sobre "offload automático quando ocupado"

O plano original de rotear tarefas automaticamente para o Mac Mini quando o Air estiver ocupado é arquiteturalmente complexo — requer um scheduler, health checks e coordination layer. Por ora, a abordagem prática é: você roteia explicitamente com no mac mini: [tarefa] ou o Air delega via SSH. Automático pode ser fase 2.

4 — Mac Mini configura o MacBook Air
⏱ ~10 min · AGORA

Roda estes comandos no Mac Mini. Ele vai copiar tudo para o Air via Tailscale/SSH automaticamente.

4a. Sincronizar skills e scripts do JARBAS:

rsync -avz --progress ~/jarbas/ jarbas@100.101.208.82:~/jarbas/

4b. Sincronizar workspace OpenClaw (SOUL.md, MEMORY.md, skills):

rsync -avz --progress ~/.openclaw/ jarbas@100.101.208.82:~/.openclaw/

4c. Sincronizar Claude Code (agents, settings, permissões):

rsync -avz --progress ~/.claude/ jarbas@100.101.208.82:~/.claude/

4d. Copiar chaves SSH do GitHub:

rsync -avz --progress ~/.ssh/ jarbas@100.101.208.82:~/.ssh/
# No MacBook Air depois: chmod 600 ~/.ssh/id_* && chmod 700 ~/.ssh

4e. Ajustar identidade no .env do Air:

# Rodar no MacBook Air após o rsync:
sed -i '' 's/JARBAS_HOST=.*/JARBAS_HOST=macbook/' ~/jarbas/.env
# Adicionar se não existir:
echo "JARBAS_ROLE=interactive" >> ~/jarbas/.env
💡 O rsync copia tudo mas preserva arquivos que o Air já tem se você usar --ignore-existing. Por padrão vai sobrescrever — que é o que queremos aqui (espelho do Mini).

6 — Configurar modo PRIMARY/STANDBY
⏱ ~5 min

Garantir que apenas um OpenClaw responde o Telegram por vez.

No Mac Mini — pausar o gateway quando Air estiver ativo:

# Adicionar ao .env do Mac Mini:
OPENCLAW_TELEGRAM_STANDBY=true   # não responde, só processa crons

No MacBook Air — é o primary:

# Adicionar ao .env do Air:
OPENCLAW_TELEGRAM_PRIMARY=true

Alias útil no Mac Mini para alternar:

# No .zshrc do Mac Mini:
alias jarbas-primary="openclaw gateway stop && echo 'Mini em standby'"
alias jarbas-standby="openclaw gateway start --standby && echo 'Mini voltou'"
⚠️ Enquanto esse mecanismo não estiver implementado no OpenClaw, a regra prática é: Mac Mini sempre ligado para crons, Air responde Telegram quando você está usando. Se quiser desligar o Air, o Mini retoma automaticamente o Telegram.

✅ Vai funcionar bem? Sim, com essa arquitetura

A divisão Primary/Standby resolve o conflito do Telegram. O rsync garante paridade de configuração. Acesso total Air→Mini via SSH já está funcionando. O único ponto que precisa de cuidado manual por ora é garantir que não sobe dois OpenClaw respondendo Telegram ao mesmo tempo — mas isso é fácil de controlar com os aliases acima.

🔮 Fase 2 — Automação de Offload (futuro)

Quando quiser o offload automático real:

  • Task queue no Supabase (jarbas_tarefas) com campo preferred_host
  • Health check periódico: cada máquina reporta CPU/RAM para o Supabase
  • Scheduler: se Air CPU > 80%, roteia próxima tarefa para o Mini
  • Ambos consultam a mesma fila — quem estiver livre pega a tarefa
Isso já é metade do caminho para o Jana OS — sistema distribuído real.
