O que é o Modo Agente?

O Modo Agente é o motor principal de geração de petições do OlivIA. Diferente do modo Express (que depende do N8N) e do Smart Blocks (blocos manuais), o Modo Agente opera 100% via JARBAS, sem depender de workflows externos.

Objetivo
→ Advogado seleciona processo + modelo
→ JARBAS pré-responde triagem com dados do banco
→ Advogado confirma/ajusta em segundos
→ JARBAS gera petição com score ≥ 9/10
→ Google Doc + DOCX salvos automaticamente
Diferenciais vs outros modos
✦ Express: N8N depende de webhook externo
✦ Smart Blocks: montagem manual por blocos
✦ VibeLaw: studio livre de refinamento
✦ Agente: JARBAS faz tudo, advogado aprova

Enqueue → Poll (frontend)

1. Frontend chama enqueueTask('olivia_gerar_peticao', payload) 2. INSERT em jarbas_tarefas (status=pendente) 3. Poll a cada 3s via progRef — lê meta.progresso_percent 4. Quando status=concluida → extrai resultado 5. Fallback N8N se JARBAS offline (isJarbasOnline check)

JARBAS processa (backend)

1. task-consumer pega tarefa pendente 2. Chama skillOliviaGerarPeticao(tarefa) 3. Atualiza meta.progresso_percent em 5 etapas 4. Loop gera+revisa (max 3x, meta score ≥9) 5. POST Apps Script → google_doc_url 6. POST docx-converter → pdf_url 7. UPDATE olivia_historico + status=concluida

⚠️ Gaps de Integração Pendentes

GAP #1 — Frontend não passa chat_historico no payload
O hook useOliviaJarbas.gerarPeticao() não tem parâmetro chatHistorico. O campo foi implementado na skill mas não chega via frontend.
→ FIX: Adicionar chat_historico?: Array<{role:string;content:string}> nos params do hook e passar no enqueueTask.
GAP #2 — instrucoes (livre) não mapeia para instrucoes_ia no payload
O frontend passa instrucoes: instrucoes (campo livre do advogado), mas a skill lê instrucoes_ia como campo específico separado de instrucoes_adicionais. As instruções do advogado chegam como instrucoes_adicionais, não como instrucoes_ia (que é para instruções do modelo).
→ FIX: Comportamento está correto — instrucoes_ia = do banco/modelo, instrucoes_adicionais = do advogado. Somente documentar para evitar confusão futura.
GAP #3 — outline[] vs estrutura[] — nomenclatura inconsistente
Frontend envia outline: [] (sempre vazio, hardcoded na linha 273 do OliviaGeradorJarbas). A skill espera estrutura[]. Os campos têm nomes diferentes — o frontend nunca popula.
→ FIX: Ou renomear estrutura→outline na skill, ou mapear no enqueueTask: estrutura: params.outline ?? [].
GAP #4 — Progresso visual: mensagens não refletem etapa real
A função getGerandoMsg(prog) no hook usa thresholds fixos para mensagens ("Buscando jurisprudências no RAG...", etc.) mas a skill atualiza meta.etapa com texto real ("Montando contexto...", "Redigindo petição..."). O frontend ignora meta.etapa e usa só o percent.
→ FIX: No polling do frontend, ler meta.etapa quando disponível e usar como mensagem em vez do map estático.
GAP #5 — pdf_url não aparece na UI
A skill agora retorna pdf_url no resultado da tarefa. O hook useOliviaJarbas extrai google_doc_url do resultado mas não extrai pdf_url. O botão "Baixar PDF" no StepConclusaoJarbas só aparece se status.pdf_url estiver preenchido — que fica null.
→ FIX: Adicionar pdf_url: resultado.pdf_url as string | undefined no setStatus após pollTask.
GAP #6 — Google Doc URL não exposta via hook
O hook já extrai google_doc_url mas não coloca em GeracaoStatus. A interface GeracaoStatus tem o campo declarado mas o setStatus após pollTask inclui o campo. Verificado: está OK — linha 352 do hook inclui google_doc_url. Não é gap.
