🧠 3. JARBAS — Arquitetura Interna
O que é o JARBAS

JARBAS é um processo Node.js ESM que roda como broker central de automação. Não é um chatbot simples — é um sistema com fila de tarefas, workers especializados, acesso Playwright ao EPROC, análise jurídica com IA e capacidade de editar o próprio código via Claude Code.

Ele está construído em camadas:

1
Telegram Poller (telegram-poller.mjs)
Long-polling da API Telegram. Recebe mensagens de Christian em DM, grupos de páginas (lapidação) e grupos de projetos. Faz smart routing: texto livre → resposta inline; verbos de ação → OpenClaw; /skill → skill direta; cc: → Claude Code.
2
Task Consumer (task-consumer.mjs)
Consome fila Supabase (jarbas_tarefas). Processa tarefas assíncronas: EPROC, análise de vídeos, geração de petições (OlivIA), lapidações, relatórios. Suporta até 3 workers paralelos. Cada tarefa tem tipo, payload, status e chat_id para notificação.
3
Skills System (54 skills em ~/jarbas/src/skills/)
Módulos especializados exportando uma função principal. Cada skill tem acesso ao Supabase, filesystem, Claude API, n8n, EPROC. Skills inline respondem diretamente; skills de fila são agendadas no Supabase. Registro em start.mjs via registerSkill().
4
OpenClaw Gateway (ws://127.0.0.1:18789)
Wrapper sobre Claude Code que mantém sessões com contexto, executa ferramentas (read/write/edit/exec/browser), e persiste o workspace em ~/.openclaw/workspace/. JARBAS usa via skill_claude_code.mjs — envia prompt, recebe resultado + artifacts.
5
Agentes Especializados (30 em ~/.claude/agents/)
System prompts especializados para Claude Code: mdflow-lapidador, mdflow-frontend-specialist, mdflow-debug-specialist, mdflow-e2e-tester, mission_control, olivia-controller, mdflow-estrategista, entre outros. Cada agente tem contexto, ferramentas e comportamento definidos.
6
Crons e Heartbeat
cron-monitorar.mjs: captura intimações EPROC às 6h/13h/19h. heartbeat.mjs: ping Supabase a cada 60s (dead man's switch). cron-cerebro.mjs: importa processos continuamente. Relatorios diários às 23h. Monitor n8n a cada hora.
🦞 4. Jana OS — O Sistema de Rescue

"Jana OS" é o apelido que demos para o MacBook Air no papel de sistema de rescue do JARBAS. O nome veio de forma orgânica — a máquina tomou personalidade própria.

O problema que resolvemos: se o Mac Mini travar (RAM, processo morto, gateway offline), como Christian reinicia o JARBAS remotamente via Telegram? A resposta foi o Jana OS: o Air faz SSH no Mini pelo Tailscale e executa comandos de emergência.

Fluxo de Rescue
Christian envia: /rescue full Telegram → JARBAS.Mini recebe? ├── SIM (JARBAS vivo) → executa rescue direto no Mini └── NÃO (JARBAS morto) → ??? # Problema: se JARBAS está morto, como executar o rescue? # Solução atual: Jana OS no Air tem um processo separado de recepção # (ou Christian precisa ter acesso SSH direto nesse caso extremo) # Fluxo quando JARBAS está vivo: Telegram → JARBAS.Mini → skill_acesso_remoto (rescue) → SSH jarbas@Air.Tailscale → jarbas-rescue.sh full → kill-9 claude/chromium → restart gateway → restart JARBAS → Notifica Telegram: "✅ JARBAS reiniciado" # Acesso bidirecional implementado: Mini → Air: ler arquivos, listar diretório, executar comandos Air → Mini: rescue (kill/restart/gateway/logs/status)
⚠️ Lacuna conhecida: Se o JARBAS estiver completamente morto (sem resposta no Telegram), o fluxo /rescue não funciona porque depende do JARBAS para receber o comando. Isso ainda não tem solução automatizada — Christian precisa acessar via SSH direto nesse caso extremo. É um ponto de melhoria que ainda não endereçamos.
🎯 5. Mission Control — Lapidação Autônoma

Mission Control é o sistema de lapidação autônoma do MdFlow. O MdFlow tem 38+ páginas React que precisam ser revisadas, corrigidas e polidas iterativamente. Em vez de fazer isso manualmente, criamos um fluxo de fila autônomo.

Conceito: Christian define a fila de páginas, dispara /fila-ativar, e o JARBAS processa automaticamente: investiga a página, planeja, implementa, testa com screenshot E2E Vision, avalia com nota 1-10, refaz até nota ≥ 9, passa para a próxima.

# Ciclo de uma lapidação [Supabase: mdflow_lapidacao_fila] status: 'pendente' → 'processando' → 'concluída' Fila ativa (3 páginas paralelas máx): /controller → worker-1 /olivia → worker-2 /acordos → worker-3 Por página (sequencial dentro do worker): 1. INVESTIGATION ├── Screenshot da página (Playwright) ├── Análise de código (~/beta-mdflow/src/pages/) ├── Leitura do PRD/spec se existir └── Diagnóstico: bugs P0/P1/P2 identificados 2. PLANNING └── Plano estruturado: o que mudar, como, por quê 3. IMPLEMENTATION ├── Claude Code (mdflow-lapidador agent) ├── Edit/write arquivos TypeScript/TSX ├── Fix TypeScript errors em loop (tsc) └── Commit git ao final 4. E2E VISION REVIEW ├── Screenshot nova (dev server local :5173) ├── Análise visual com Claude Vision └── Nota 1-10 por critério (funcionalidade, UX, visual) 5. PASS/FAIL ├── Nota ≥ 9 → ✅ próxima página ├── Nota < 9 → 🔄 nova iteração (loop até 9) └── Após 3 loops sem progresso → escalate para Christian 6. REPORT └── HTML publicado no GitHub Pages

Isso separa contexto por página e evita que uma lapidação longa polua a DM principal.

🤔 10. Pedido de Avaliação — O Que Queremos Saber
Questão 1 — Arquitetura Geral
Essa arquitetura faz sentido como está? Mac Mini como hub central, Jana OS como rescue via Tailscale, tudo passando por Telegram. O que você mudaria?
Contexto: o sistema tem que funcionar 24/7 sem intervenção humana para EPROC e automações jurídicas. Christian é o único que opera isso e não quer ter que abrir terminal regularmente.
Questão 2 — Divisão JARBAS vs Claude Code
A divisão atual (JARBAS faz tudo direto, CC só para lapidações complexas) é a correta? Ou você vê cenários onde essa fronteira está errada?
A regra atual: JARBAS nunca delega para CC o que pode fazer com write/edit/exec. CC só para exploração ampla de codebase ou loops iterativos com contexto persistente.
Questão 3 — Jana OS — Robustez
O modelo de rescue atual (Air faz SSH no Mini via Tailscale) é robusto o suficiente? Existe um cenário onde tanto JARBAS está morto quanto o canal de rescue não funciona?
Lacuna conhecida: se JARBAS está completamente morto, /rescue não chega. Precisamos de um mecanismo de bootstrap independente no Air que monitore o Mini proativamente.
Questão 4 — Mission Control e Fila de Lapidação
O fluxo de lapidação autônoma (INV → PLAN → IMPL → E2E Vision → nota ≥ 9 → próxima) faz sentido? Você vê problemas de design nessa abordagem?
O maior risco é loop infinito em páginas com bugs estruturais que Claude Code não consegue resolver em 3 tentativas. Temos escalate, mas a definição de "progresso real" ainda é vaga.
Questão 5 — OpenClaw como Middleware
Usar OpenClaw como wrapper do Claude Code faz sentido aqui? Ou seria mais direto usar claude --print --dangerously-skip-permissions diretamente para lapidações?
OpenClaw gerencia sessões, contexto, workspace unificado, e tem compaction automático. O tradeoff é latência de inicialização de sessão vs controle direto do subprocess.
Questão 6 — Custo
Com ~R$1.700/dia nos picos de lapidação intensa e ~R$500/dia de média, esse modelo é sustentável para um escritório de advocacia de médio porte? O que você cortaria sem perder utilidade?
Claude Max (CC ilimitado) = $100/mês fixo. API Anthropic (JARBAS + análises) = ~R$300–1.700/dia variável. Gemini (análise vídeo/docs bulk) = barato. OpenAI = pouco usado.
Questão 7 — Avaliação Geral
De 0 a 10, como você avalia essa arquitetura no geral? O que te impressionou (positivo)? O que te preocupou (negativo)?
Sem filtros. Christian quer feedback honesto, não elogios. Se tem problema de design, pode falar.
