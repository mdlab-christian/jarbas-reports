Avaliação Crítica — Positivos e Negativos
Aspecto Positivo ✅ Negativo ❌
Arquitetura Dual-engine (JARBAS + N8N fallback) é robusto. Tunnel permanente resolve instabilidade de ngrok. 2 engines = 2 codebase de lógica para manter. Risco de divergência de comportamento.
UX Geral Landing com 4 KPIs (30d), recovery tabs para rascunhos em andamento, animações com Framer Motion. 4 modos com nomes abstratos. Usuário não sabe qual escolher. Falta diferenciação clara de resultado esperado.
Smart Blocks Mais estável. Dá controle total ao usuário. Bom para advogados que preferem compor manualmente. Manual demais — sem sugestão de blocos por IA. Não usa o corpus jurídico do escritório.
Express Triagem com perguntas pré-respondidas por IA (com confidence score) é excelente UX. Bug crítico parado a feature. N8N como motor primário = latência alta (~15-30s) + frágil.
Modo Agente JARBAS como motor = acesso direto ao RAG, blocos, jurisprudências, playbooks internos. Steps insuficientes. Sem estrutura/outline antes de gerar. Sem feedback rico durante geração.
VibeLaw Editor generativo com patches aplicáveis é diferencial competitivo real. Split-view elegante. Começa vazio. Deveria chegar com petição pronta. Primeira tela não coleta contexto suficiente.
Infra Rate limit ativo. Recovery de rascunhos automático. jarbas_instances com heartbeat. Sem métricas de qualidade das petições geradas. Sem A/B entre JARBAS e N8N.

Brainstorm — Novo Modo JARBAS (O Modo Principal)

🎯 Conceito

O Modo JARBAS é a fusão de Express + Modo Agente + VibeLaw num único fluxo premium. É o modo mais avançado — usa N8N como referência de cenários mas executa tudo internamente no JARBAS. A petição chega pronta no editor, não em branco. O usuário pode refinar via chat generativo, mas o trabalho pesado já foi feito.

📋

Step 1 — Identificação Completa

Processo + tipo de petição + modelo base (selecionável da biblioteca) + advogado responsável. Diferencial: usuário vê preview do modelo antes de selecionar.

Novo
📤

Step 2 — Upload de Provas

Upload de documentos (comprovantes, negativações, contratos). JARBAS lê e extrai dados relevantes automaticamente. Integrado com os documentos já existentes no processo.

Novo
🎯

Step 3 — Triagem Aprimorada

Igual ao Express, mas com mais perguntas e contexto. JARBAS usa os documentos do Step 2 para pré-responder automaticamente. Jurimetria integrada (taxa de sucesso + réu típico).

Melhorado
🗂️

Step 4 — Outline / Estrutura

JARBAS propõe estrutura da petição: seções, blocos de texto candidatos, jurisprudências relevantes. Usuário pode reorganizar antes de gerar. Visual drag-and-drop.

Novo
⚙️

Step 5 — Geração com RAG

JARBAS gera internamente usando: corpus do escritório + blocos curados + jurisprudências + provas enviadas + style guide. Score de qualidade exibido em tempo real. Target: 9+/10.

Core
✏️

Step 6 — VibeLaw Editor

Petição já pronta abre no VibeLaw. Chat IA para refinamentos. Patches aplicáveis. Preview real-time. Exporta para Google Doc / PDF quando aprovado.

Integrado

Flow Técnico Proposto

1
Frontend → POST jarbas.jarbas-mdlab.com/olivia/gerar-modo-jarbas com { processo_id, modelo_id, documentos, respostas_triagem, outline }
2
JARBAS cria tarefa na jarbas_tarefas (tipo: OLIVIA_MODO_JARBAS) + retorna task_id imediatamente
3
skill_olivia_modo_jarbas.mjs executa: lê processo + documentos → RAG blocos → RAG jurisprudências → Claude gera rascunho → aplica style guide → score de qualidade
4
Frontend faz polling via jarbas_tarefas com Supabase Realtime → transições de estado em tempo real
✓
Resultado → petição salva em olivia_historico + abre VibeLaw com documento pré-carregado + score visível
