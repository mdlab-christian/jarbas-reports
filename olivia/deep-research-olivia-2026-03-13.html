🚨
P0 — Conteúdo dos blocos de texto está vazio (44-64 chars)
Confirmado via Supabase: olivia_blocos_texto.conteudo contém apenas títulos (44-64 chars), não o texto jurídico completo. O conteúdo real está nos Google Docs de origem (google_doc_id) mas não foi importado. O gerador monta petições sem substância real — este é o bug raiz de 80% dos problemas de qualidade.
🚨
P0 — olivia_versoes com 0 registros (FK constraint quebrando geração)
A tabela olivia_versoes está completamente vazia. O frontend tenta criar registros com FK para versão, causando constraint violation em olivia_historico. O código contém workaround (versao_id removido), mas o constraint ainda existe em alguns paths.
🚨
P0 — RLS em jarbas_tarefas bloqueando Modo Agente
Modo Agente usa supabase.from('jarbas_tarefas').insert() com usuário autenticado. A política RLS está bloqueando este insert: "new row violates row-level security policy". Resultado: triagem e geração via JARBAS completamente inoperantes.
🚨
P0 — RAG não populado (indexar-mdflow.py nunca rodou)
ChromaDB existe em ~/.jarbas/rag/mdflow/ mas as collections processos_jarbas e intimacoes_jarbas estão vazias. Olivia e Clayton dependem do RAG para contextualizar petições e gerar triagem inteligente. Sem RAG, geram petições genéricas.
⚠️
P1 — instrucoes_ia vazia em todos os modelos
Campo instrucoes_ia tem 0 chars em todos os modelos testados. Este campo alimenta diretamente o prompt dos agentes N8N. Sem ele, os agentes operam sem instruções específicas por tipo de petição.

HTML como formato no VibeLaw — Análise
✅
VibeLaw é o modo mais adequado para HTML nativo
Proposta implementação: VibeLaw armazena a petição como HTML em olivia_vibelaw_threads.conteudo_html. O preview à direita renderiza o HTML diretamente. O agente Olivia faz patches cirúrgicos via seletores CSS/ID em seções específicas. Exemplo: <section id="qualificacao">...</section>.

Benefícios: formatação rica (endentação jurídica, espaçamento 1.5, negritos), renderização direta, JARBAS patcheia por seção sem reescrever o documento todo, exportação HTML→DOCX via pandoc ou biblioteca existente.

JUIZ...</header> <section id="fatos">DOS FATOS...</section> <section id="direito">DO DIREITO...</section> <section id="jurisprudencia">DA JURISPRUDÊNCIA...</section> <footer id="pedidos">DOS PEDIDOS...</footer> </article>

🎯 Conclusão do Deep Research

O sistema OlivIA tem arquitetura sólida e código bem estruturado. Os problemas são de dados e infra, não de lógica. O bug mais crítico é trivial de resolver: o conteúdo dos blocos de texto nunca foi importado dos Google Docs — o campo existe, o google_doc_id existe, falta só o script de importação. O segundo maior bloqueador é o RAG vazio — ChromaDB e script prontos, nunca rodados.

A boa notícia: com Fase 0 + Fase 1 resolvidas (2 dias), o Smart Blocks e o Express já funcionam com conteúdo real. Modo Agente funciona após fix de RLS. O VibeLaw precisa de um fix de contexto específico.
