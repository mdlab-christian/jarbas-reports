❌ Diagnóstico: Problema é multicamada, mas tem solução direta
O gerador de petições tem infraestrutura quase completa — o n8n está ativo, o JARBAS webhook está rodando, a triagem IA funciona (testei ao vivo: retornou 7 perguntas contextuais em 2s). O que está quebrando são 3 camadas específicas:

1. Frontend — seletor de processo não vincula ao banco corretamente, blocos repetidos (query sem filtro de tag), botões disabled por ausência de processo_id no estado do wizard.
2. Backend JARBAS — webhook-main.mjs (servidor que serve jarbas.jarbas-mdlab.com) tem as rotas certas, mas 8 rotas que o frontend chama estão ausentes nesse servidor. Existe um segundo servidor (start.mjs) com outras rotas, mas as rotas de quality-check, vibelaw-chat, vibelaw-gerar-inicial, upload-documento, documentos-processo e analise-decisao não existem em lugar nenhum.
3. Banco de dados — FK constraint e RLS bloqueiam inserção em olivia_historico e jarbas_tarefas.

A boa notícia: não precisa de v2 frontend. Os bugs são cirúrgicos e localizados. Estimativa de resolução: 6–8h de execução focada por modo.

8 bugs · fonte: vídeo + código + banco ao vivo
ID Camada Sev Descrição & Causa Raiz Fix
B1 DB P0 FK olivia_historico bloqueia geração
Constraint olivia_historico_modelo_id_fkey — INSERT falha se modelo_id não existe em olivia_modelos_peticao. O wizard não garante que o modelo selecionado tem ID válido.		 Garantir que Step 1 salva modelo_id de registro real antes de avançar. Auto-resolver via active_version_id já tem código (ce5721a) mas não cobre todos os casos.
B2 DB P0 RLS jarbas_tarefas bloqueia Modo Agente
Policy RLS ativa (relrowsecurity=true) — usuário autenticado não tem permissão de INSERT. Erro: "new row violates row-level security policy".		 Adicionar policy INSERT para authenticated ou usar service_role no client de jarbas_tarefas.
B3 Frontend P0 Seletor de processo não retorna resultados
Vídeo (00:38): processo selecionado com sucesso apenas digitando o número CNJ. Mas o E2E não conseguiu selecionar automaticamente — seletor usa componente customizado sem data-testid padrão. Funcionalmente: seletor FUNCIONA se o usuário digitar CNJ correto.		 Melhorar UX: exibir nome_cliente, empresa e advogado (OAB) na lista. Adicionar data-testid para E2E. Busca por nome + CNJ.
B4 n8n P0 Loading Express trava em 5% / "Failed to fetch"
Vídeo (03:10): "Failed to fetch". Causa: /olivia/quality-check não existe no JARBAS. Após geração do DOCX, o frontend tenta chamar quality-check e falha silenciosamente, ficando preso.		 Implementar rota /olivia/quality-check no webhook-main.mjs. Temporariamente: fazer quality-check retornar score fixo 7.5 para desbloquear o fluxo.
B5 Backend P0 6 rotas do VibeLaw/upload/quality inexistentes
Frontend chama: vibelaw-chat, vibelaw-gerar-inicial, upload-documento, documentos-processo, quality-check, analise-decisao. Nenhuma está registrada no webhook-main.mjs.		 Implementar as 6 rotas faltantes no webhook-main.mjs. Cada uma é relativamente simples — a maioria enfileira tarefas ou chama Supabase diretamente.
B6 Frontend P1 Blocos repetidos no Smart Blocks Step 2
Vídeo (00:50): "tá aparecendo repetido aqui os blocos de texto". Query retorna todos os blocos sem filtrar por tag do modelo selecionado. SecaoBlocos não filtra por tag_id.		 Query deve filtrar: WHERE tag_id IN (tags do modelo selecionado) com DISTINCT por bloco_id.
B7 Frontend P1 VibeLaw chat responde JSON puro
Vídeo (06:40): "Nenhuma petição carregada ainda", "Não tenho acesso ao número do processo". Frontend não parseia o response do n8n olivia/chat webhook. Exibe JSON raw ao invés do texto.		 Verificar parseamento do response em useVibeLaw.ts. O n8n retorna objeto, frontend espera string.
B8 Frontend P1 Download DOCX quebrado no Smart Blocks conclusão
Vídeo (02:15): botão de download clicado mas nada acontece. ManualStep4Conclusao tem fix commitado (4833e6d) mas ainda falha — URL do DOCX pode estar null ou o STORAGE_URL não está apontando corretamente.
