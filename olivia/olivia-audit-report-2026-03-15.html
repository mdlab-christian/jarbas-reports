🚨 Script de migração para categoria_empresa_id — pronto para executar

A spec já fornece o SQL exato para popular os 169 modelos por inferência de nome (~60–70% de cobertura). Os restantes ficam com NULL (aparecem para todos os processos — comportamento correto). Exemplo de execução imediata:

-- Inferência por nome — cobertura ~65%
UPDATE olivia_modelos_grupo mg
SET categoria_empresa_id = ec.id
FROM empresas_categorias ec
WHERE mg.deleted_at IS NULL
  AND mg.categoria_empresa_id IS NULL
  AND (
    (ec.nome = 'Mercado Livre' AND mg.nome ILIKE '%mercado livre%')
    OR (ec.nome = 'Instituições Financeiras' AND (mg.nome ILIKE '%banco%' OR mg.nome ILIKE '%BV%' OR mg.nome ILIKE '%itaú%' OR mg.nome ILIKE '%caixa%' OR mg.nome ILIKE '%nubank%' OR mg.nome ILIKE '%financ%'))
    OR (ec.nome = 'Telefonia e Internet' AND (mg.nome ILIKE '%claro%' OR mg.nome ILIKE '%vivo%' OR mg.nome ILIKE '%tim%' OR mg.nome ILIKE '%internet%' OR mg.nome ILIKE '%telefon%'))
    OR (ec.nome = 'Órgão Restritivo' AND (mg.nome ILIKE '%SPC%' OR mg.nome ILIKE '%SERASA%' OR mg.nome ILIKE '%SCPC%' OR mg.nome ILIKE '%boa vista%'))
    OR (ec.nome = 'Lojas' AND (mg.nome ILIKE '%loja%' OR mg.nome ILIKE '%magazine%'))
    OR (ec.nome = 'Energia Elétrica' AND (mg.nome ILIKE '%CEEE%' OR mg.nome ILIKE '%CPFL%' OR mg.nome ILIKE '%energia%'))
  );
-- Depois confirmar contagem

6 Problemas Críticos — Análise Técnica

🔴 CRÍTICO-1: categoria_empresa_id NULL — P0 Bloqueante

Causa Raiz Confirmada

169 modelos publicados, todos com categoria_empresa_id = NULL. Verificado via Supabase REST com Prefer: count=exact → content-range: 0-168/169.

O código do frontend está correto em todos os modos (filtra por categoria). O problema é exclusivamente de dados. Impacto: todos os 169 modelos aparecem para todos os processos — sem discriminação por tipo de empresa.

Fix: Executar o SQL de inferência por nome (ver Seção 4). Cobertura esperada: ~65%. Os ~35% restantes (modelos genéricos, sem indicação no nome) ficam com NULL e continuam aparecendo para todos — comportamento correto conforme spec.

🔴 CRÍTICO-2: olivia-template.docx ausente — Impacto em todos os modos

Causa Raiz Confirmada

DEC-11 da spec autoriza pandoc com template de referência: --reference-doc=/Users/jarbas/jarbas/assets/olivia-template.docx. O arquivo não existe. Pandoc usa formatação default.

Sintoma observado: 2/4 gerações SUCESSO têm pdf_url = NULL. Hipótese: modelos com conteudo_template vazio geram HTML vazio → pandoc produz DOCX malformado → upload falha silenciosamente → pdf_url permanece NULL.

Fix: (1) Criar olivia-template.docx mínimo em ~/jarbas/assets/. (2) Adicionar validação no docx-converter: se html_content vazio, retornar erro 400 antes de chamar pandoc.

🔴 CRÍTICO-3: BUG-CC-01 — Warning falso positivo permanente

1 linha de fix — alto impacto

useStepIdentificacaoData.ts linha 152: .select('id, nome, descricao, ordem, config')

Falta conteudo_template no SELECT. Resultado: sel.conteudo_template é sempre undefined → condição !sel.conteudo_template sempre true → warning amarelo "Template sem conteúdo" aparece para 100% dos modelos, inclusive os 276 que têm conteúdo real.

Fix (1 linha): .select('id, nome, descricao, ordem, config, conteudo_template')

🔴 CRÍTICO-4: VibeLawEditorPanel stub

Componente incompleto — 21 linhas

VibeLawEditorPanel.tsx tem apenas 21 linhas. Para um editor de petição jurídica, seria esperado um editor rich text (TipTap, Slate, ou similar) com formatação, heading levels, etc. Verificar se OliviaVibelawStudio.tsx está usando outro componente para edição ou se este painel realmente precisa ser expandido.
