Por que o CC declarou 9.1/10 e não funciona?

O Claude Code testou a camada técnica superficial: verificou que rotas existem, que migrations foram aplicadas, que TypeScript compila, que N8N retorna HTTP 200. Não testou com dados reais.

Os problemas são estruturais e só aparecem ao usar o sistema com processos reais — filtros que dependem de dados nunca preenchidos, queries com JOINs errados, backend com campos hardcoded errados.

ℹ️ Por que só aparece "v1.0" no seletor de versão?
O hook busca: SELECT * FROM olivia_versoes WHERE modelo_id = :grupo_id
Mas o FK real é: olivia_versoes.modelo_id → olivia_modelos_peticao.id
Resultado: nenhuma versão encontrada → o frontend exibe "v1.0" como fallback hardcoded. Para corrigir: buscar versões via olivia_modelos_peticao WHERE modelo_grupo_id = :grupo_id, depois pegar os olivia_versoes WHERE modelo_id IN (ids das peticoes do grupo).
Como popular categoria_empresa_id nos modelos (script de migração)

Os modelos têm o nome da categoria no próprio nome (ex: "[BV] ...", "[Telefonia]", "Mercado Livre"). É possível inferir automaticamente com ~90% de precisão:

-- Estratégia de inferência por nome
UPDATE olivia_modelos_grupo mg
SET categoria_empresa_id = ec.id
FROM empresas_categorias ec
WHERE mg.deleted_at IS NULL
  AND mg.categoria_empresa_id IS NULL
  AND (
    (ec.nome = 'Mercado Livre'            AND mg.nome ILIKE '%mercado livre%')
    OR (ec.nome = 'Instituições Financeiras' AND (mg.nome ILIKE '%banco%' OR mg.nome ILIKE '%financ%' OR mg.nome ILIKE '%cartão%' OR mg.nome ILIKE '%BV%' OR mg.nome ILIKE '%itaú%' OR mg.nome ILIKE '%caixa%' OR mg.nome ILIKE '%nubank%'))
    OR (ec.nome = 'Telefonia e Internet'  AND (mg.nome ILIKE '%telefon%' OR mg.nome ILIKE '%claro%' OR mg.nome ILIKE '%vivo%' OR mg.nome ILIKE '%tim%' OR mg.nome ILIKE '%oi%' OR mg.nome ILIKE '%internet%'))
    OR (ec.nome = 'Órgão Restritivo'      AND (mg.nome ILIKE '%SPC%' OR mg.nome ILIKE '%SERASA%' OR mg.nome ILIKE '%SCPC%' OR mg.nome ILIKE '%BOA VISTA%' OR mg.nome ILIKE '%boa vista%'))
    OR (ec.nome = 'Lojas'                 AND (mg.nome ILIKE '%loja%' OR mg.nome ILIKE '%magazine%' OR mg.nome ILIKE '%casas bah%'))
    OR (ec.nome = 'Energia Elétrica'      AND (mg.nome ILIKE '%energia%' OR mg.nome ILIKE '%CEEE%' OR mg.nome ILIKE '%COPEL%' OR mg.nome ILIKE '%CELESC%' OR mg.nome ILIKE '%CPFL%'))
  );

-- Depois: revisar manualmente os que ficaram NULL
⚠️ Decisão de Christian necessária
A migração automática cobre ~60-70% dos casos. Os modelos genéricos (sem indicação de empresa no nome) devem ficar com categoria_empresa_id = NULL para aparecer em todos os processos (comportamento atual do hook, que trata NULL como "genérico"). Confirmar se essa é a regra desejada.

Fluxo correto Express/Agente (o que Christian explicou no áudio)
// Conceito fundamental:
// Tags = perguntas | Blocos de texto = respostas
// Express = Smart Blocks com respostas pré-preenchidas por IA
// Agente = Express com respostas geradas com mais contexto

SMART BLOCKS:
  Usuário seleciona manualmente quais blocos de texto usar
  → Os blocos já são as "respostas" das tags
  → Monta a petição com os blocos selecionados

EXPRESS:
  IA recebe as tags (perguntas) do modelo
  IA analisa provas do processo
  IA responde as tags com os blocos de texto mais adequados
  → Só pergunta ao usuário o que ficou em dúvida ou não tem resposta automática
  → Resultado final: petição como se tivesse sido gerada no Smart Blocks

AGENTE:
  Igual ao Express, mas com análise mais profunda
  Usa todas as informações do processo no backend
  IA tem contexto completo para responder mais tags automaticamente

D-1: Smart Blocks — Google Drive vs Bucket Storage?
Bloqueante para FIX-4

Contexto: O sistema antigo (Google Drive + Google Apps Script) funcionava para Smart Blocks. A migração para Supabase Storage está causando problemas. Você mencionou que pode ser melhor voltar para Google Drive se não conseguirmos fazer funcionar no Storage.

OpçãoPrósContrasRecomendação
A) Manter Storage Já integrado, sem dependência Google, mais rápido, controle total Problemas de formatação com pandoc; DOCX pode sair diferente do Google Docs
B) Voltar Google Drive Já funcionava, formatação garantida, Google Apps Script maduro Dependência externa, mais complexo, mais lento, tokens Google Auth
C) Híbrido: Storage para armazenar, Google Docs para converter Melhor dos dois mundos — armazenamento no Supabase, conversão pelo Google Complexidade extra ✅ JARBAS recomenda esta opção
ℹ️ Recomendação JARBAS
Tentar opção A primeiro com uma correção específica: configurar pandoc com referência a um template DOCX de qualidade. Se o resultado ainda estiver corrompido após 1 tentativa, partir direto para C.
D-2: Migração de categorias — automática ou manual?
Bloqueante para FIX-0

167 modelos sem categoria. Posso inferir ~60-70% pelo nome (ex: "[BV]" → Órgão Restritivo, "[Mercado Livre]" → Mercado Livre). Os ~40% restantes seriam genéricos (aparecem para todos).

Opções:

  • A) Migração automática por nome + revisão posterior — Executo agora, você revisa a lista de modelos sem categoria depois
  • B) Você exporta a lista, classifica, eu aplico — Mais preciso, mas demora
  • C) Manter todos como genéricos (sem categoria) — Todos aparecem para todos, sem filtro. Mais simples, menos preciso.
ℹ️ Recomendação JARBAS
Opção A. Rodar o script agora com log do que foi atualizado. Você valida depois. Não bloqueia a execução dos outros fixes.
