Root Cause #1 — SmartBlocks completamente quebrado no banco novo

No sistema antigo: olivia_modelos_peticao.tags_ids ARRAY vinculava cada modelo às suas tags. No sistema novo, essa vinculação foi movida para olivia_modelos_grupo.tags_ids — mas está NULL em 100% dos grupos. Ou seja: os dados (tags + blocos) existem no banco, mas o wizard não consegue saber quais tags pertencem a qual modelo. É por isso que o ManualStep2Conteudo mostra TODAS as 183 tags da org, causando as 72 seções absurdas no vídeo.

⚠️

Root Cause #2 — Mudança de schema: tag_id → tag_ids

Antigo: olivia_blocos_texto.tag_id UUID (FK singular, 1 tag por bloco). Novo: olivia_blocos_texto.tag_ids ARRAY (múltiplas tags por bloco). O hook useBlocosPorTag do sistema antigo fazia .eq('tag_id', tagId) — isso não funciona mais. O novo precisa fazer .contains('tag_ids', [tagId]).

⚠️

Root Cause #3 — Sistema de versões: texto "1/2/3" → integer + bloco_grupo_id

Antigo: versao TEXT ('1','2','3') e o usuário escolhia a versão no VersaoSelector (hardcoded 1/2/3). Novo: version INTEGER + bloco_grupo_id UUID para agrupar versões. O VersaoSelector antigo (hardcoded) é incompatível com a nova estrutura. O que Christian quer é migrar para Style Guides por forma de escrita — que é o caminho certo.

🚨

Root Cause #4 — VibeLaw retorna JSON cru para o usuário

O vídeo da equipe mostra claramente: o VibeLaw retorna {"response":"...","action_cards":[...]} como JSON cru no chat. O frontend não está parseando a resposta estruturada — está renderizando o JSON diretamente. O backend retorna o formato correto, o problema é 100% frontend (parser ausente no VibeLaw chat renderer).

Brainstorming — Ideias para Discutir

Sugestões além do óbvio, para levar a OlivIA para o próximo nível

💡 SmartBlocks 2.0 — Sub-wizard por seção

  • Em vez de 72 tags lineares, agrupar em seções do documento (já existe secao_id)
  • Mostrar accordion por seção (Fatos, Direito, Pedidos...) — usuário expande só a seção que quer editar
  • Tags opcionais ficam colapsadas por padrão, só as obrigatórias abertas
  • Progress indicator por seção (não total linear) — reduz sensação de fadiga

🔀 Modo Híbrido: SmartBlocks + IA

  • O usuário seleciona blocos manualmente, mas pode pedir à IA: "sugira o melhor bloco para este caso"
  • IA lê o processo e pré-seleciona os blocos mais adequados — usuário só confirma ou troca
  • Elimina a fadiga de escolha sem tirar o controle do advogado
  • Diferencial: SmartBlocks com inteligência embutida, sem precisar do Express

🎨 Style Guides visuais

  • Em vez de "Versão 1/2/3", mostrar cards com preview de estilo: "Formal RS", "Técnica Concisa SP", "Persuasiva SC"
  • Cada style guide tem exemplo de parágrafo de abertura para o usuário comparar
  • Pode ser vinculado ao perfil do advogado (Dr. Christian usa estilo X por padrão)
  • Anti-fingerprint automático: alterna micro-variações dentro do style guide escolhido

📊 Score em tempo real no Modo Agente

  • Enquanto a petição vai sendo gerada, mostrar o score parcial subindo em tempo real
  • Se score ficar abaixo de 7, mostrar card de alerta com o motivo antes de concluir
  • Revisor automático com 3 pontos de melhoria específicos (não genérico)
  • "Regenerar só a seção X" — sem precisar refazer a petição inteira

⚡ Express Turbo — Volume em lote

  • Selecionar múltiplos processos da lista → gerar todas as petições de uma vez
  • Fila assíncrona: notifica quando cada uma ficar pronta
  • Para processos do mesmo tipo/empresa: reutiliza contexto da triagem, só muda o processo
  • Ideal para grupos de 20-50 processos iguais (muito comum no Midas)

🎭 VibeLaw Premium — Editor rico

  • Adicionar toolbar WYSIWYG no painel de preview (negrito, itálico, lista, cabeçalho)
  • Ao selecionar texto no preview, menu flutuante: "Reescrever", "Expandir", "Resumir", "Deixar mais formal"
  • Histórico de versões do documento (undo/redo via VibeLaw)
  • Comparação lado-a-lado: versão IA original vs versão editada pelo advogado

🔍 Playbooks como memória jurídica

  • Cada playbook deveria incluir histórico de resultados: quando usado, score médio, vitórias
  • OlivIA aprende automaticamente: após sentença favorável, atualiza o playbook da empresa/tipo
  • Jurisprudências vinculadas automaticamente: quando JARBAS baixa nova decisão favorável → propõe adicionar ao playbook
  • Base de conhecimento que cresce com o escritório

🚦 Validação de petição antes de protocolar

  • Antes de protocolar no EPROC: checklist automático de consistência
  • Validar: data, número CNJ, partes corretas, valor da causa, tipo de recurso compatível com fase
  • Alertar se a petição menciona empresa errada (frequente em copia-cola)
  • Score mínimo configurável por tipo de peça antes de liberar protocolo
