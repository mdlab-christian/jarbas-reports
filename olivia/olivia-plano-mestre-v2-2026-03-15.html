3 Arquitetura Definitiva — Princípios
🏗️ Princípio de Independência

Smart Blocks: Funciona sem N8N e sem JARBAS. Puro frontend + Supabase + pandoc (JARBAS para DOCX).
Express: Funciona sem JARBAS. 100% N8N. Se JARBAS cair, Express continua funcionando.
Agente: 100% JARBAS. Evolução do Express — mesmo resultado, mais capacidades.
VibeLaw: JARBAS SSE + N8N apply-patch. Construir por último.

✅ Princípio do Relatório HTML

Relatório HTML = exclusivo do Modo Agente (JARBAS tem capacidade de gerar HTML rico). No Express e Smart Blocks: N8N retorna JSON com campo relatorio (texto estruturado, não HTML). Frontend exibe o relatório no Step 4/5. Nunca chamar JARBAS para relatório nos modos N8N.

⚠️ Princípio da Triagem

Perguntas de triagem são sempre geradas por IA com base na análise de provas. Nunca fixas. O workflow OLIVIA-ANALISE (TupOpRfZXkTHzzEuBOk7f) gera a análise → o resultado da análise alimenta o gerador de perguntas (seção A do rFxJtaKqBm5YM4dBWtz3S). A triagem não pode iniciar antes da análise estar completa.

Smart Blocks como Fundação do Sistema

🧱 Tags = Perguntas / Blocos = Respostas

Smart Blocks: Usuário seleciona manualmente os blocos para cada tag → petição montada com os blocos selecionados.
Express: IA responde as tags/escolhe os blocos automaticamente com base na análise de provas + triagem respondida → passa a "petição Smart Blocks resultante" + "melhore isso com o style guide" para o agente redator.
Agente: Mesmo Express, mas com mais capacidades (JARBAS tem acesso a RAG, histórico, jurisprudências próprias).
Conceito chave: O Express SEMPRE gera uma petição Smart Blocks internamente primeiro — isso é o mecanismo de redundância e qualidade.

5 D2 — Migração de Categorias (backend antigo)
✅ Viável e altamente preciso: 328/329 modelos com categorias no DB antigo

O banco antigo hzqoxkmxvwxejjanstdh tem olivia_modelos_peticao.categorias_empresa_ids (array) preenchido em 328/329 modelos. O banco novo tem os mesmos 329 modelos (mesma estrutura de nome). A migração via mapeamento por nome de modelo tem cobertura ~95% — muito melhor que os 65% da inferência por nome da categoria.

Tabelas de Importação a Criar (nome sugerido)

TabelaPropósitoCampos principais
importacao_olivia_categorias Mapeamento 1:1 entre categorias do banco antigo e novo id_antigo, id_novo, nome_antigo, nome_novo, mapeado_em
importacao_olivia_modelos Log de migração: qual modelo antigo virou qual grupo novo + categorias aplicadas id_modelo_antigo, id_modelo_novo, nome_modelo, categorias_antigas[], categoria_nova_id, match_tipo (nome|manual), migrado_em, status

Estratégia de Migração

1
Criar mapeamento de categorias
Buscar empresas_categorias dos dois bancos → mapear por nome (ex: "Órgão restritivo" → "Órgão Restritivo"). Inserir em importacao_olivia_categorias. ~10 registros.
2
Match modelos antigo → novo por nome
Para cada olivia_modelos_peticao do banco antigo com categorias_empresa_ids preenchido: buscar olivia_modelos_grupo no banco novo com mesmo nome → pegar primeira categoria do array (modelo novo tem 1 categoria, não array) → mapear ID via tabela do passo 1.
3
Aplicar UPDATE no banco novo
UPDATE olivia_modelos_grupo SET categoria_empresa_id = [mapeado] WHERE id = [match]. Log em importacao_olivia_modelos com status=migrado.
4
Verificar modelos sem match
Modelos sem correspondência por nome → listar em importacao_olivia_modelos com status=sem_match → revisão manual posterior. Ficam com categoria NULL (aparecem para todos — comportamento correto).
ℹ️ Por que array → single category

O banco antigo usava categorias_empresa_ids[] (múltiplas categorias por modelo). O banco novo usa categoria_empresa_id (uma categoria). Para modelos com múltiplas categorias: usar a primeira da array como categoria principal. Na prática, a maioria dos modelos tem 1-2 categorias e a primeira já é a correta.

Gaps Novos — Não Capturados pelo JARBAS

🔴
GAP-CC-01 · pdf_url nunca populado no fallback Smart Blocks
SmartBlocks fallback usa downloadDocxReal local — não chama JARBAS /olivia/docx-converter. Historico salvo sem pdf_url. Critério "pdf_url 100% populado" nunca será atingido sem este fix.
Fix: substituir downloadDocxReal por POST /olivia/docx-converter + UPDATE historico com pdf_url retornado.
🔴
GAP-CC-02 · SmartBlocks: zero recovery de rascunho
OliviaGeradorSmartBlocks.tsx usa useState local — não lê recoveryRunId da URL. Se usuário sair e voltar com o link de recuperação, o wizard começa do zero. Inconsistente com Express que tem recovery completo via useWizardTabs.
Fix: implementar leitura de recoveryRunId + restaurar state a partir de olivia_historico.meta.wizard_state (padrão idêntico ao Express).
🔴
GAP-CC-03 · SmartBlocks: sem dirty-state protection
Express tem isDirty + ConfirmDialog antes de navegar. SmartBlocks não tem — usuário perde blocos selecionados silenciosamente ao clicar Cancelar ou navegar via breadcrumb.
Fix: adicionar isDirty + ConfirmDialog (já existe o componente shared em /components/shared/confirm_dialog).
🔴
GAP-CC-04 · SmartBlocks: modo='MANUAL' no fallback insert
Linha 246 em OliviaGeradorSmartBlocks.tsx: modo: 'MANUAL' deveria ser 'SMART_BLOCKS'. Recovery routing funciona por coincidência (ambos → rota smart-blocks) mas KPIs e analytics ficam errados. Estatísticas de uso do modo nunca refletem Smart Blocks.
Fix: trocar para modo: 'MANUAL' → 'SMART_BLOCKS' na linha 246. 1 linha.
⚠️
GAP-CC-05 · Express: step "estrutura" não documentado no plano
OliviaGeradorIA.tsx tem 5 steps: identificacao → triagem → estrutura (outline ≥2 seções) → revisao → conclusao. O plano v2.0 descreve o Express com 6 etapas (processo, provas, triagem, geração, revisão, DOCX) sem mencionar "estrutura". Qual é o papel do step de estrutura no novo fluxo? Ele sobrevive ou é removido?
Precisa de decisão antes de C2: manter StepEstrutura (configura outline para o Redator) ou remover e deixar N8N decidir a estrutura automaticamente?
⚠️
GAP-CC-06 · conteudo_template: confirmar onde vive no schema novo
BUG-CC-01 do plano diz "adicionar conteudo_template ao SELECT em useStepIdentificacaoData.ts". Mas revisando: nem o select de olivia_modelos_grupo (L126–138) nem o de olivia_versoes (L147–162) incluem conteudo_template. Precisa confirmar: o campo está em olivia_versoes.conteudo_template ou em outro lugar? Verificar schema antes de aplicar BUG-CC-01.
Rodar: SELECT column_name FROM information_schema.columns WHERE table_name IN ('olivia_versoes','olivia_modelos_grupo') AND column_name LIKE '%template%' no Supabase MCP.
⚠️
GAP-CC-07 · Rate limit aplicado ao SmartBlocks sem IA
SmartBlocks fallback (concatenação local) não usa tokens de IA mas passa pela verificação check_olivia_rate_limit. Se usuário atingiu o limite por usar Express, SmartBlocks também é bloqueado — mesmo sem chamar Claude. Semanticamente incorreto.
Decisão: remover rate limit do SmartBlocks inteiramente (é modo offline/manual) OU aplicar rate limit apenas quando webhook N8N é chamado. Recomendação: remover da tela de geração do SmartBlocks.
⚠️
GAP-CC-08 · VibeLaw: criarGoogleDoc vs DOCX baixável
VibeLawEditorPanel chama vl.criarGoogleDoc para exportação. O critério do plano é "DOCX baixável". Google Doc e DOCX local são fluxos diferentes. Se o usuário não tem acesso ao Google Drive da conta da organização, o botão "Abrir Doc" falha silenciosamente.
Verificar em C4: se criarGoogleDoc retorna URL pública ou requer OAuth. Adicionar fallback via JARBAS docx-converter se URL não for acessível.
ℹ️
GAP-CC-09 · dados_escavador no WizardState — campo não utilizado
WizardState em useWizardTabs.ts tem campo dados_escavador: Record<string, any> | null. Não foi referenciado em nenhum step revisado. Está sendo serializado no meta JSONB de cada olivia_historico. Se não há integração com Escavador planejada no v2.0, remover para reduzir payload.
P2 — não bloqueia nada, mas limpa dívida técnica.
