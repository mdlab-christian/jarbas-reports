4 Smart Blocks — Arquitetura da Solução

Problema raiz: falta a tabela de vinculação modelo↔tags

O banco tem olivia_blocos_texto.tag_ids[] mas não tem como saber quais tags pertencem a qual modelo. Resultado: todos os blocos aparecem para qualquer petição.

olivia_modelos_peticao (atual)
id · uuid
tipo_peticao_id · uuid
nome · text
conteudo_template · text
instrucoes_ia · text
placeholders · jsonb
❌ SEM vinculação de tags
❌ SEM vinculação de blocos
olivia_modelo_tags (NOVA — proposta)
id · uuid
modelo_id · uuid → olivia_modelos_peticao
tag_id · uuid → olivia_tags
obrigatoria · boolean (default false)
ordem · integer
organizacao_id · uuid
✅ índice: (modelo_id, tag_id)

Fluxo correto do Smart Blocks pós-fix

// Step 1: usuário seleciona Tipo de Petição olivia_tipos_peticao.id ──► filtra olivia_modelos_grupo por tipo_peticao_id + filtra por categoria_empresa_id do processo // Step 2: usuário seleciona Modelo olivia_modelos_grupo.active_version_id ──► olivia_modelos_peticao.id └──► busca olivia_modelo_tags WHERE modelo_id = X └──► agrupa tags por olivia_tags_secao // Step 3: monta seções de blocos Para cada secao: └──► busca olivia_blocos_texto WHERE tag_ids && [tags_do_modelo] └──► exibe apenas blocos relevantes ao modelo // Step 4: triagem Perguntas geradas pelas tags do modelo ──► fn_olivia_get_triagem_by_grupo(modelo_id) // Auto-inclusão de jurisprudências olivia_jurisprudencias_assunto WHERE assunto = tipo_peticao └──► filtra por UF do processo (RS/SC/SP) └──► inclui automaticamente na geração (sem step manual)

Migration necessária

CREATE TABLE olivia_modelo_tags (
  id               UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  organizacao_id   UUID NOT NULL REFERENCES organizacoes(id),
  modelo_id        UUID NOT NULL REFERENCES olivia_modelos_peticao(id) ON DELETE CASCADE,
  tag_id           UUID NOT NULL REFERENCES olivia_tags(id) ON DELETE CASCADE,
  obrigatoria      BOOLEAN DEFAULT FALSE,
  ordem            INTEGER DEFAULT 0,
  created_at       TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(organizacao_id, modelo_id, tag_id)
);

-- Índices de performance
CREATE INDEX idx_modelo_tags_modelo ON olivia_modelo_tags(modelo_id);
CREATE INDEX idx_modelo_tags_tag ON olivia_modelo_tags(tag_id);

-- RLS
ALTER TABLE olivia_modelo_tags ENABLE ROW LEVEL SECURITY;
CREATE POLICY org_isolation ON olivia_modelo_tags
  USING (organizacao_id = current_setting('app.organizacao_id')::UUID);

PRD como fonte de verdade

Cada item do checklist é uma tarefa atômica com: arquivo a editar, migration a aplicar, teste a rodar. JARBAS nunca "improvisa" — segue o PRD.

🔧

Implementação direta

JARBAS edita os arquivos .tsx/.ts diretamente, aplica migrations via Supabase Management API, faz commit e push. Sem Claude Code como intermediário.

🧪

Teste end-to-end obrigatório

Não marca como concluído sem testar. Para cada modo: gera uma petição real de teste com processo existente → verifica se DOCX abre → verifica qualidade mínima do texto.

📢

Máx 5 min sem update no Telegram

Se a tarefa demorar: envia "🔄 ainda em: [o que está fazendo]". Ao concluir: "✅ [item] — próximo: [item]". Ao amanhecer: relatório HTML completo com status de tudo.

🛡️

Anti-loop obrigatório

Se mesmo erro 3x seguidas: para, marca como BLOQUEADO, notifica com diagnóstico completo. Não desperdiça tokens tentando a mesma abordagem.

9 Minhas Ideias — Para validar

Estas são sugestões que vão além do que foi pedido. Incluo para enriquecer a discussão:

💡 Ideia JARBAS #1 — Smart Score de Completude

Indicador de qualidade do pacote de petição antes de gerar

Antes de gerar, exibir um "Score de Completude": Documentos do processo: 4/5 ✅ · Blocos obrigatórios: 3/3 ✅ · Provas anexadas: ✅ · Contestação: ❌ (opcional). Score ≥ 80% = verde, gera com segurança. Isso previne petições incompletas.

💡 Ideia JARBAS #2 — Modo Comparação de Modelos

Ver 2 modelos side-by-side antes de escolher

Na tela de seleção de modelo, botão "Comparar" → mostra dois modelos lado a lado: estrutura, blocos vinculados, taxa de uso no escritório, última vez que foi gerada petição com ele. Facilita escolha do advogado.

💡 Ideia JARBAS #3 — Auto-sugestão de Modo

JARBAS sugere qual modo usar baseado no processo

Quando o advogado seleciona um processo, antes de escolher o modo, JARBAS exibe: "Para este processo, recomendo Modo Agente — temos 8 decisões analisadas e 3 provas salvas." Ou "Recomendo Smart Blocks — sem provas salvas, controle manual é melhor."

💡 Ideia JARBAS #4 — Histórico por Tipo de Petição

Na landing, mostrar distribuição de petições por tipo (últimos 30d)

Cards de KPI por tipo: "Réplica: 47 este mês | Apelação: 23 | Manifestação: 18". Ajuda o coordenador a visualizar volume de trabalho e detectar gargalos.

💡 Ideia JARBAS #5 — Revisão Automática de Qualidade

Score jurídico automático após geração (já existe na skill)

A skill skill_olivia_gerar_peticao.mjs já tem lógica de "Score < 7 → auto-regenerar". Expor esse score na UI para o advogado: "Qualidade: 8.4/10 — Coerência jurídica: Alta · Completude: Alta · Originalidade: Média". Com botão "Regenerar" se score < 8.

💡 Ideia JARBAS #6 — Biblioteca de Petições Aprovadas

Marcar petições boas como "referência" para treinar os modelos

Na conclusão, botão "Marcar como referência" → petição vai para biblioteca. Quando JARBAS gerar próxima petição do mesmo tipo/modelo, usa as referências como few-shot. Com o tempo, a qualidade aumenta automaticamente.

💡 Ideia JARBAS #7 — Detecção de Tese Repetida (Anti-fingerprint)

Avisar se a petição ficou muito parecida com uma anterior do mesmo réu

Já temos anti-fingerprint por seed no JARBAS. Adicionar: ao gerar petição para mesmo réu/tipo, comparar com últimas 3 petições e alertar se similaridade > 80%. Isso evita que o réu detecte padrões nas petições do escritório.

💡 Ideia JARBAS #8 — Protocolo Direto pós-geração

Botão "Protocolar no EPROC" na tela de conclusão

Já temos skill_olivia_eproc_protocolar.mjs. Na tela de conclusão do wizard, adicionar botão "Protocolar no EPROC" que dispara a skill com o DOCX gerado + processo selecionado. Loop completo: gera → protocola → movimentação registrada.

💡 Ideia JARBAS #9 — Modo Express com Sugestão de Tipo

IA sugere o tipo de petição baseado nos documentos enviados

No Express, após upload das provas, antes de avançar: IA analisa os docs e sugere "Com base nos documentos, parece ser uma Réplica. Confirma?" Se usuário não sabe qual tipo escolher, a IA guia. Reduz erro de seleção.
