📋 Sumário Executivo
🐛

Bug Crítico — Corrigido

A triagem perguntava valor da restrição e órgão negativante ao usuário, sendo que esses dados já existem em processos.restricao_valor_new e restricao_orgao_nome. O payload do webhook não incluía esses campos. Fix aplicado.

✅

Pontos Fortes

Arquitetura de wizard bem estruturada. Autosave a cada 30s funcional. Fallbacks robustos em todos os steps. Rate limit em dois pontos. Suporte a múltiplas tabs simultâneas.

⚠️

Oportunidades

Backend é muito mais rico do que o que é enviado para a IA. Contexto do processo pode ser enriquecido em vários pontos. Lógica de geração duplicada entre StepConclusao e useGerarPeticao.

Análise Geral

O Modo Express da OlivIA é um fluxo sólido e bem pensado. O wizard de 5 steps cobre o ciclo completo de geração de petições com tratamento de erro em cada etapa, autosave funcional e suporte a múltiplas sessões paralelas. O principal problema identificado — e corrigido — era que a IA não recebia dados do backend que já existem (valor da restrição, órgão negativante), forçando o usuário a preencher manualmente o que o sistema já sabe. Além disso, há oportunidade significativa de enriquecer o contexto enviado para a IA com dados já disponíveis na tabela processos: valor da causa, grau, tribunal, dados de restrição completos.

🗺️ Mapa do Fluxo E2E
1
StepIdentificacao — Seleção do Processo
👤 Usuário: Busca e seleciona o processo, escolhe tipo de petição, modelo, advogado
⚙️ Sistema: Seta dados_processo com todos os dados do processo no estado do wizard. Opcionalmente enriquece via webhook escavador (fire-and-forget)
DadosProcesso inclui: id, numero_cnj, objeto, tipo_processo, grau, valor_causa, restricao_orgao_nome, restricao_valor_new, clientes, empresas, advogados, tribunal, vara
2
StepTriagem — Perguntas IA
👤 Usuário: Responde perguntas geradas pela IA sobre o caso específico
⚙️ Sistema: Chama JARBAS /olivia/triagem-questions com processo_context. Aguarda análise de provas (se em andamento, máx 30s). Pré-preenche respostas quando possível. Fallback para fn_olivia_get_triagem_by_grupo se webhook falhar
⚡ Fix aplicado: restricao_valor + restricao_orgao_nome agora incluídos no payload. Pre-fill automático para perguntas sobre valor/órgão de restrição
3
StepEstrutura — Outline + Disparo
👤 Usuário: Revisa estrutura da petição (seções), adiciona instruções adicionais (texto ou voz), clica "Gerar Petição"
⚙️ Sistema: Verifica rate limit, chama useGerarPeticao.gerarPeticao() → webhook mdflow/olivia/gerar-doc. Seta status_geracao = EM_PROGRESSO. Avança para StepRevisao
Outline padrão: 6 seções. Outline recursal: 6 seções diferentes. Toggle para pular revisão automática. Suporte a microfone
4
StepRevisao — Polling + Editor
👤 Usuário: Aguarda geração, depois revisa/edita a minuta no editor (Preview ou Editar). Define escritório contraparte e chance de sucesso
⚙️ Sistema: Polling a cada 3s em olivia_historico. Progress steps visuais. Autosave a cada 30s via RPC olivia_autosave_wizard. Ao concluir, preenche minuta_editada
Auto-trigger de geração se usuário chegar no step com status 'pendente' e sem texto gerado
5
StepConclusao — Resultado Final
👤 Usuário: Copia, baixa DOCX, abre Google Doc, gera relatório HTML do processo
⚙️ Sistema: Busca métricas (score_revisor, modelo_usado, tokens, custo_usd, tempo). Exibe resultado. Disponibiliza download DOCX real (docx@8)
Estado 'pre_geracao' disponível para reprocessar. Integração com Google Drive para URL do documento
🔧 Bug Corrigido
✅ CORRIGIDO — 13/03/2026 🔴 Crítico

Contexto de restrição não enviado para IA de triagem

O payload enviado para o webhook olivia/triagem-questions não incluía restricao_valor_new e restricao_orgao_nome no processo_context. Como resultado, a IA de triagem não sabia o valor da restrição nem o órgão negativante (SPC, Serasa, etc.), gerando perguntas ao usuário para dados que já existem no banco. Em um escritório especializado em negativação indevida, isso impacta praticamente 100% das petições geradas.

StepTriagem.tsx — processo_context.processo

processo: { id: dp.id ?? processoId, numero_cnj: dp.numero_cnj ?? null, objeto: dp.objeto ?? null, grau: dp.grau ?? null, valor_causa: dp.valor_causa ?? null, empresa_nome: dp.empresas?.nome ?? dp.empresa_nome ?? null, + restricao_valor: dp.restricao_valor_new ?? dp.restricao_valor ?? null, + restricao_orgao_nome: dp.restricao_orgao_nome ?? null, + restricao_contrato: dp.restricao_contrato ?? null, },

TriagemCampos.tsx — getPreFill() e ContextHint

+ // Pre-fill para perguntas sobre valor da restrição + if ((t.includes('valor') && (t.includes('restrição') || t.includes('negativação'))) && dp.restricao_valor_new) { + return dp.restricao_valor_new ?? dp.restricao_valor; + } + // Pre-fill para perguntas sobre órgão negativante + if ((t.includes('órgão') || t.includes('credor')) && dp.restricao_orgao_nome) { + return dp.restricao_orgao_nome; + }

💡 Recomendações Priorizadas
1

Criar buildFullProcessoContext() centralizado

Criar uma função utilitária que extrai todos os campos disponíveis do DadosProcesso: restricao_data, restricao_contrato, advogado completo (OAB/UF), tribunal, vara, comarca, dados do escavador. Usar em todos os webhooks: triagem, geração, análise. O backend é muito mais rico do que o que chega para a IA.

Impacto Alto — Qualidade das petições geradas
2

Refatorar StepConclusao para usar useGerarPeticao

A lógica de iniciarGeracao() em StepConclusao (250+ linhas) duplica o que useGerarPeticao já faz, mas com payload diferente e sem buildProcessoContext(). Extrair para o hook centralizado elimina inconsistência e facilita manutenção futura.

Impacto Alto — Manutenibilidade
3

Adicionar valor da restrição no Resumo do Caso (StepEstrutura)

O Resumo do Caso no StepEstrutura mostra processo, tipo de petição, cliente e empresa. Para escritório de negativação indevida, o campo mais crítico é o valor da restrição e o órgão (SPC/Serasa). Adicionar esses dados no card de resumo para conferência antes do disparo.

Impacto Médio — UX de conferência
4

Indicador visual para aguardarAnaliseProvas

Durante o carregamento da triagem, o sistema pode aguardar até 30s a análise de provas em background. O usuário vê apenas um spinner genérico. Exibir mensagem específica: "Aguardando análise das provas enviadas..." com progresso visual.

Impacto Médio — UX de espera
5

Expor seletor de qualidade_ia no StepEstrutura

A qualidade_ia está no estado (padrão: 'padrao') mas nunca é exposta ao usuário no fluxo Express. Para casos mais complexos ou alto valor, o usuário deveria poder selecionar "Pensamento Estendido". Adicionar toggle simples nas configurações do StepEstrutura.

Impacto Médio — Qualidade para casos complexos
6

Validação mínima em StepIdentificacao antes de avançar

Verificar se o processo selecionado tem os dados mínimos necessários: empresa ré, cliente, tipo do processo. Se faltar campo crítico, exibir alerta (não bloqueante) antes de avançar: "Processo sem órgão negativante cadastrado — triagem pode gerar mais perguntas".

Impacto Baixo — Prevenção de surpresas
7

Expandir ContextHint para mais campos

Adicionar hints contextuais para: valor da causa, grau do processo, tribunal, comarca. O usuário vê confirmação ao lado de cada pergunta dos dados já cadastrados, aumentando confiança na resposta pré-preenchida.

Impacto Baixo — UX de confiança
8

Toast informativo no fallback DOCX → HTML

Quando o download DOCX cai no fallback HTML-wrapped, o usuário não é avisado. Adicionar toast: "Baixando em formato compatível com Word (pode haver diferenças visuais)".

Impacto Baixo — Clareza
