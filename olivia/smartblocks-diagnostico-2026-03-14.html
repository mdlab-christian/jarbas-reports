🐛 Os 3 bugs raiz que impedem o funcionamento

🔴 BUG-1 CRÍTICO — Query com limit(200) + filtro client-side

Onde: SmartBlocksStep2Blocos.tsx, query olivia_blocos_texto

Problema: A query busca até 200 blocos e depois filtra por tag_ids no client (JS). Mas o banco tem 1.244 blocos. Os 1.044 não buscados nunca entram no filtro — os blocos corretos do modelo ficam de fora.

Sintoma observado: "Nenhum bloco vinculado a este modelo" ou lista vazia mesmo com dados corretos no banco.

busca limit(200) → filtra JS client-side → 1044 blocos ignorados

Fix: Usar .overlaps('tag_ids', modeloTagsIds) — filtro server-side no PostgREST. Retorna APENAS os blocos com intersecção nas tags, sem limit.

.overlaps('tag_ids', tags) → filtro no Supabase → só blocos do modelo 
// ANTES (bugado — client-side)
const allBlocos = await query.limit(200); // perde 1044 blocos
return allBlocos.filter(b => b.tag_ids.some(t => modeloTagsIds.includes(t)));

// DEPOIS (correto — server-side)
const { data } = await supabase
  .from('olivia_blocos_texto')
  .select('id, titulo, conteudo, tag_ids')
  .eq('organizacao_id', organizacao_id)
  .is('deleted_at', null)
  .eq('status', 'published')
  .overlaps('tag_ids', modeloTagsIds); // ← PostgREST array overlap
🔴 BUG-2 CRÍTICO — Query desabilitada sem tags resolvidas

Onde: SmartBlocksStep2Blocos.tsx, enabled da query de blocos

Problema: enabled: !!organizacao_id && (modeloTagsIds !== undefined)

modeloTagsIds é calculado de junctionTagIds e modeloGrupo?.tags_ids. Se a junction table retorna array vazio (possível durante loading), modeloTagsIds fica null e a query de blocos nunca dispara (porque null !== undefined é true, mas a lógica depende de dados assíncronos resolvendo antes).

Fix: enabled deve aguardar o modelo ser carregado E as tags estarem disponíveis:

enabled: !!organizacao_id && !!state.modelo_grupo_id && modeloTagsIds !== null
⚠️ BUG-3 UX — populate de blocos no render com setTimeout

Onde: SmartBlocksStep2Blocos.tsx, lógica needsPopulate

Problema: setState dentro do render com setTimeout(0) é anti-pattern React — pode causar loop de render, inconsistência de estado e bugs difíceis de debugar. Especialmente problemático quando o modelo muda (re-render infinito possível).

Fix: Mover para useEffect com dependência em blocosDb e state.modelo_grupo_id. Reset explícito dos blocos quando modelo muda (já feito no Step1, mas o useEffect precisa reagir).

useEffect(() => {
  if (!blocosDb.length) return;
  setState(prev => ({
    ...prev,
    blocos: blocosDb.map(b => ({
      id: b.id,
      titulo: b.titulo,
      conteudo: b.conteudo,
      ativo: true,
      tag_nome: b.tag_ids?.[0] ? tagsMap[b.tag_ids[0]] : undefined,
    })),
  }));
}, [blocosDb]); // dispara só quando dados do banco mudam

❓ Decisão necessária — Intervenção humana

⚠️ UX do Step 2: DnD flat list vs Tag-wizard (como o Manual)?

O OliviaGeradorManual (ManualStep2) funciona assim:

  • Cada tag é um passo do sub-wizard
  • Ex: "BV - E-mail" → mostra os 4 blocos da tag BV-Email → usuário seleciona qual quer
  • Avança para próxima tag, e assim por diante
  • Ao final, os blocos selecionados formam a petição

O Smart Blocks (C2a — o que foi implementado) funciona assim:

  • Lista plana com todos os blocos do modelo
  • Usuário marca/desmarca e reordena via DnD
  • Não tem navegação por tag-pergunta

Christian, qual é o comportamento correto para Smart Blocks?

A) Tag-wizard + DnD final

Igual ao Manual: pergunta por pergunta (tag), seleciona blocos em cada tag, depois tela de reordenação DnD antes de gerar.

B) Lista DnD direta (atual)

Mostra todos os blocos do modelo em lista única, agrupados por tag (header de seção), o usuário marca/reordena livremente.

Se for A, o Smart Blocks vira essencialmente o mesmo que o Manual mas com DnD ao final — pode fazer sentido se o objetivo é reordenar depois de selecionar. Se for B, é diferente e mais rápido.
