🚨 Suspeita de causa raiz — Smart Blocks não filtra blocos corretamente
ManualStep2Conteudo.tsx (499 linhas) é onde os blocos são exibidos e filtrados por tags. Se o query usa status='active' mas os blocos estão todos como 'published', TODOS os 893 blocos aparecem sem filtro. A "mistura" de blocos irrelevantes que o Christian viu no vídeo provavelmente vem disso.
⚠️ callWebhook → N8N (não JARBAS diretamente)
A função callWebhook() em webhook.ts aponta para VITE_N8N_WEBHOOK_URL (n8n). Apenas chamadas com fetch(JARBAS_BASE_URL + ...) vão direto ao JARBAS. Isso significa que toda a geração passa pelo n8n primeiro, mesmo no Modo Agente — o n8n chama o JARBAS internamente.

A política RLS da tabela jarbas_tarefas bloqueia o insert quando chamado pelo Modo Agente. 2
🔴 VibeLaw Chat → JSON bruto
O workflow n8n DnecdvxoRKyMjZMD (olivia/vibelaw-chat) retorna o JSON completo da resposta da IA em vez de extrair só o .text ou .content. Isso requer editar o nó de resposta no n8n. Christian: abrir o workflow no n8n e verificar o último nó antes do Respond — precisa extrair o campo correto.
3
🔴 FK constraint olivia_historico_modelo_id_fkey
O histórico está sendo criado com um modelo_id que não existe ou é inválido (talvez usando o modelo_grupo_id onde deveria ser o versao_id atual do grupo). Decisão necessária: qual ID enviar — o do grupo (168 registros) ou o da versão publicada (329 registros)?
4
🟡 Triagem sem contexto de provas
O /olivia/triagem-questions no JARBAS precisa de uma análise prévia em olivia_analises para gerar perguntas contextuais. O Express deveria: (1) fazer upload das provas, (2) disparar ANALISE-V2, (3) aguardar conclusão, (4) só então carregar a triagem. O flow atual pula o passo 2-3. Decisão: triagem sempre espera análise? Ou tem timeout com fallback para perguntas genéricas?
5
🟡 Loading de 5% sem realtime
O Express mostra uma barra de progresso que não avança porque não há comunicação do n8n para o frontend durante a execução. O realtime deveria ser via polling de olivia_historico.status a cada 2s.
