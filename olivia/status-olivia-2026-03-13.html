🧱 Smart Blocks — Blocos repetidos e irrelevantes
P0  ManualStep2Conteudo.tsx · Afeta: Smart Blocks 100%
🔍 Causa raiz identificada
O filtro de blocos funciona em 2 etapas: (1) buscar tag_id da tabela olivia_modelo_tags para o modelo selecionado, (2) filtrar blocos cujos tag_ids contenham qualquer tag do modelo (.some()). O problema: cada modelo tem em média ~50 tags (16.668 vínculos ÷ 329 modelos). Com o filtro .some(), um bloco passa se tiver qualquer uma das 50 tags — o que faz a maioria dos 893 blocos aparecer.

Além disso, o componente divide os blocos por tag em steps separados (TagSteps). Um bloco com 3 tags aparece em 3 steps diferentes — o usuário vê o mesmo bloco repetido.
💻 Código problemático
// ManualStep2Conteudo.tsx — filtro atual (PROBLEMA) return allBlocos.filter(b => (b.tag_ids ?? []).some(tid => modeloTagsIds.includes(tid)) ); // → com 50 tags/modelo, a maioria dos blocos passa // → mesmo bloco aparece em múltiplos TagSteps

// Query para buscar só tags obrigatórias: supabase.from('olivia_modelo_tags') .select('tag_id') .eq('modelo_id', modeloGrupo.active_version_id) .eq('organizacao_id', organizacao_id) .eq('obrigatoria', true) // ← ADD ISSO

🤖 Modo Agente — RLS bloqueando insert em jarbas_tarefas
P0  useOliviaJarbas.ts · Afeta: Modo Agente 100%
🔍 Causa raiz identificada
O frontend (useOliviaJarbas.ts linha 95) insere diretamente em jarbas_tarefas usando o cliente Supabase do usuário (JWT do usuário logado, role authenticated). A policy RLS jarbas_tarefas_insert_authenticated permite INSERT para authenticated, mas a policy jarbas_tarefas_insert_org verifica org_id() que lê auth.jwt() ->> 'organizacao_id'. Se esse campo não está no JWT do usuário, a policy retorna NULL e o insert falha.

Nota: o JARBAS (webhook-main.mjs) usa service_role e bypassa RLS — o problema é exclusivamente do insert feito pelo frontend.
💻 Código problemático
// useOliviaJarbas.ts linha 95 — INSERT via client do usuário (PROBLEMA) const { data, error } = await supabase // ← JWT do usuário, não service_role .from('jarbas_tarefas') .insert({ organizacao_id, // ← se org_id() não confirma isso, RLS bloqueia tipo, payload, status: 'pendente', ... });
// RLS atual — jarbas_tarefas_insert_org: roles: {public}, cmd: INSERT, qual: null ← sem restrição? // jarbas_tarefas_insert_authenticated: roles: {authenticated}, cmd: INSERT ← mas org_id() pode falhar
🤔 Decisão necessária
⚠️ 2 opções — Christian decide
OPÇÃO A — Rota tudo via JARBAS (sem insert direto no frontend)
Frontend chama fetch(JARBAS_BASE_URL + '/olivia/gerar-modo-agente', ...) diretamente em vez de inserir no banco. JARBAS usa service_role e enfileira. Mais seguro, mais correto arquiteturalmente. Requer trocar a lógica do useOliviaJarbas de insert direto para HTTP call.
OPÇÃO B — Criar RPC no Supabase (Edge Function) que bypassa RLS RECOMENDO
Criar função rpc_olivia_enqueue_task(tipo, payload) com SECURITY DEFINER (roda como service_role internamente). Frontend chama supabase.rpc('rpc_olivia_enqueue_task', {...}). Mais simples, mantém a arquitetura atual, 10 min de implementação.
OPÇÃO C — Corrigir a RLS policy para usar users.organizacao_id
Alterar a policy para usar (organizacao_id IN (SELECT organizacao_id FROM users WHERE id = auth.uid())). Resolve o problema da query org_id() que pode falhar. Mais simples mas mais arriscado — alterar RLS em produção.
