3 VibeLaw 2.0 — Proposta Ideal

VibeLaw: Vibe Coding para Advogados

O advogado descreve o que quer em linguagem natural. O JARBAS gera, refina e entrega. Não é um wizard, não são templates — é uma conversa que vira petição. JARBAS como runtime, streaming real, output profissional.

Conceito central

VibeLaw é o modo mais avançado da OlivIA. Enquanto Smart Blocks usa templates e Express usa wizard IA, VibeLaw é conversacional: o advogado "conversa" com a Olivia e a petição emerge iterativamente, como pair programming mas para peças jurídicas.

User Flow proposto

1

Entrada no modo

Advogado seleciona um processo → VibeLaw abre com contexto pré-carregado: partes, tipo de ação, histórico, restrições, decisões anteriores. Olivia apresenta: "Olá! Processo BV/SPC carregado. O que você precisa gerar hoje?"

2

Request inicial

Advogado digita: "Gera uma réplica focando nos extratos bancários e na ausência de notificação prévia". JARBAS processa: RAG no corpus jurídico + contexto do processo + olivia_juridico agent.

3

Geração com streaming

Petição aparece na tela em tempo real (streaming via JARBAS tunnel). Advogado vê o texto sendo gerado. Pode interromper se necessário.

4

Refinamento iterativo (patches)

Advogado solicita ajustes: "Encurta os fundamentos jurídicos", "Adiciona jurisprudência do STJ", "Torna o tom mais incisivo". JARBAS aplica e retorna versão atualizada. Thread mantém contexto completo.

5

Revisão de qualidade

Botão "Revisar qualidade" aciona o agente revisor (treinado com vídeos do Leonardo). Retorna score + observações específicas antes de exportar.

6

Export + Protocolo

Aprovada → geração de DOCX + PDF + HTML. Opção de protocolar via EPROC diretamente (skill_olivia_eproc_protocolar). Thread salva em olivia_threads para histórico.

4 Decisões Ainda Abertas

Estas decisões precisam ser respondidas antes ou durante a SPEC. Cada uma tem impacto de implementação.

⚡

DEC-A1 · VibeLaw: Streaming ou Polling?

Como o frontend recebe a resposta do JARBAS em tempo real

A) SSE (Server-Sent Events) via JARBAS tunnel RECOMENDADO
Frontend faz GET para https://jarbas.jarbas-mdlab.com/vibelaw/stream · Texto chega token por token · UX moderna
B) Polling olivia_messages a cada 2s
Mais simples de implementar · Sem streaming real · Carga no DB · UX inferior
C) WebSocket dedicado
Bidirecional real · Mais complexo · Overkill para o caso de uso
💡 SSE é o padrão de facto para chat AI (ChatGPT, Claude usam SSE). O JARBAS tunnel já existe. Implementação: ~1 dia. Polling funciona mas a UX degradada vai incomodar o Leonardo e a Heloísa na prática diária.
🧵

DEC-A2 · VibeLaw: Thread por documento ou por processo?

Define como o histórico da conversa é organizado no banco

A) Uma thread por documento gerado RECOMENDADO
Cada petição tem sua própria conversa · Histórico limpo · Fácil de retomar · Lista de "drafts" por processo
B) Uma thread por processo (múltiplas petições na mesma conversa)
Contexto acumulado · Thread fica longa e confusa · Rolling window de 10 msgs fica inadequado
💡 Thread por documento = melhor UX e melhor gerenciamento de contexto. O advogado pode ver todos os drafts de um processo como uma lista, cada um com sua história. A rolling window de 10 msgs faz sentido por documento, não por processo inteiro.
📐

DEC-A3 · VibeLaw: Layout da interface

Como o usuário vai ver o chat + a petição ao mesmo tempo

A) Split view fixo: Chat (40%) | Preview petição (60%) RECOMENDADO
Vê chat e petição ao mesmo tempo · Natural para advogado que vai lendo enquanto refina · Mobile: tabs
B) Chat no centro, petição em modal/drawer lateral
Foco no chat · Petição secundária · Menos imersivo
C) Chat e preview em tabs separadas
Mais simples de implementar · Perde o contexto visual simultâneo
💡 Split view é o padrão de ferramentas de "vibe coding" (Cursor, Lovable). O advogado precisa ver a petição sendo construída enquanto conversa — é o coração da proposta de valor do VibeLaw.
🔄

DEC-A4 · Apply Patch: regerar tudo ou edição cirúrgica?

Quando o advogado pede uma mudança, o JARBAS regenera a petição inteira ou edita só a parte relevante?

A) Regenerar a petição completa a cada patch RECOMENDADO
Mais simples · Sem risco de inconsistência entre partes · JARBAS tem contexto completo · Qualidade homogênea
B) Edição cirúrgica: identificar e substituir só o trecho
Mais rápido · Mais difícil de implementar corretamente · Risco de petição inconsistente · Complexidade alta
💡 Regeneração completa é o que funciona melhor na prática (é o que modelos de linguagem fazem bem). Petições jurídicas têm interdependências entre seções — uma mudança no fundamento afeta a conclusão. Cirurgical editing parece bom na teoria mas é arriscado para peças jurídicas formais. Custo extra de tokens é aceitável dado o valor.
📦

DEC-A5 · Tags e blocos faltantes no import

75 tags ativas vs 183 no DB antigo (108 faltando). 893 blocos vs 1244 (351 faltando). Importar do antigo ou ignorar?

A) Importar do DB antigo (hzqoxkmxvwxejjanstdh) na Fase 1 RECOMENDADO
108 tags + 351 blocos recuperados · Smart Blocks fica completo · 1 script de import · Baixo risco
B) Ignorar — os 75 ativos são suficientes
Mais simples · Mas perde metade do conteúdo pago que já existia · Smart Blocks fica incompleto
💡 Importar é trivial (1 script) e recupera conteúdo valioso que já foi produzido. Ignorar seria desperdiçar trabalho passado. Prioridade: Fase 1 junto com reimport dos blocos atuais.
🔒

DEC-A6 · ProcessoSelector — melhorias de UX

Durante o planejamento foi mencionado melhorar a busca e o painel de info do processo. Entra no escopo desta SPEC?

A) Incluir como parte da Fase 3 (Front Core) RECOMENDADO
Busca por CNJ + cliente + empresa ré · Painel tipo Empresa com advogado + empresa + valor/data · Melhora toda a UX dos 4 modos
B) Deixar para SPEC separada de UX
SPEC menor agora · Mas ProcessoSelector atual é limitante para o Leonardo na prática diária
💡 ProcessoSelector melhorado beneficia todos os 4 modos e é relativamente rápido de implementar. Sem ele, o advogado perde tempo buscando o processo certo antes de gerar. Incluir na Fase 3.
