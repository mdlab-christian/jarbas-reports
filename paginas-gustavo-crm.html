4. Bugs Identificados (todos confirmados por Gustavo)
1
Gerar Contrato de Honorários → retorna erro (código alfanumérico)
No modal do lead LEAD FIX3 (id: 140eec64-4ef6-4097-ac07-e51cd4ccffec, coluna_crm: triagem), ao clicar em Gerar Contrato, retorna notificação de erro com código tipo "1A2B3C". Provavelmente o webhook n8n /mdflow/doc/gerar-contrato-honorarios está recebendo processo_id mas neste estágio não há processo criado ainda — apenas lead_id.
Webhook: POST /webhook/mdflow/doc/gerar-contrato-honorarios | Passa: processo_id (inexistente neste estágio)
2
Gerar Procuração → mesmo erro de código alfanumérico
Mesmo problema do bug 1. Webhook /mdflow/doc/gerar-procuracao recebe processo_id que não existe. Deve receber lead_id + restricao_id para funcionar no CRM antes da triagem.
Webhook: POST /webhook/mdflow/doc/gerar-procuracao | Solução: aceitar lead_id + restricao_id como alternativa
3
Botão "Abrir" não funciona após geração bem-sucedida
Diagnóstico no código: useGerarDocumento.ts usa data.data?.url ?? data.data?.google_doc_url para montar o toast "Abrir". Se o n8n não retorna esses campos, o botão não aparece ou não abre. Nenhum estágio testado com sucesso ainda — o n8n provavelmente não está retornando url ou google_doc_url na resposta.
Fix: garantir que n8n retorna { success: true, data: { url: "https://drive.google.com/..." } }
4
Painel de documentos não disponível em "Em Atendimento"
DocumentosPanel com origin="crm" usa processoId. No estágio "Em Atendimento" não há processo — só lead. O hook useDocumentosProcesso filtra por processo_id. Precisa aceitar lead_id para funcionar no CRM.
useDocumentosProcesso → aceitar lead_id | processos_documentos → ter coluna lead_id
5
Botão "Nova Auditoria" com visual errado e posição errada
Atualmente em CrmFilterBar.tsx com variant="outline" e posicionado à direita. Gustavo quer: cor verde (igual ao "Novo Lead") e posicionado à esquerda do botão "Novo Lead".
CrmFilterBar.tsx → variant="default" + className="gap-1.5" | CrmPage.tsx → reordenar botões
6
Modo Lista não mostra todas as informações (só Kanban mostra)
CrmListView.tsx mostra poucas colunas. Deve mostrar as mesmas informações do Kanban: empresa(s) da restrição, valor estimado, status, origem, data, responsável.
CrmListView.tsx → adicionar colunas: restricoes, valor_estimado, origem, responsavel
7
Drawer Distribuição não exibe número do contrato da restrição
O campo restricao_contrato já existe em ProcessoDistribuicao (tipo TypeScript confirmado) mas não está sendo renderizado no drawer/modal de preparação. Deve aparecer nos dados da restrição ao lado de empresa, valor, órgão e data.
DistribuicaoDrawerPreparacao.tsx → adicionar restricao_contrato na exibição
8
Ao cadastrar novo lead, não aparece na coluna "Novos" imediatamente
UI não atualiza em tempo real ao criar lead. Cache do React Query precisa ser invalidado imediatamente após criação para o card aparecer.
ModalNovoLead → invalidateQueries(['crm-leads-kanban']) após sucesso

8. Distribuição — Core do Fluxo
Como funciona: Após triagem aprovada (rpc_crm_aprovar_triagem), o lead é convertido e cada restrição vira 1 processo. Esses processos vão para Aguardando Preparação na página de Distribuição. Lá, os documentos já carregados se tornam os documentos de cada processo.

⚠️ Não existe aba/coluna "Distribuição" no CRM. Existe um botão "Enviar para Distribuição" dentro do modal do lead (na etapa de Triagem). Após clicar, o lead: (1) é convertido em processos, (2) é movido para arquivados no CRM com motivo "cliente".

Fluxo da Triagem → Distribuição

1
Triagem: rpc_crm_aprovar_triagem
Cria cliente no Supabase (ou vincula existente). Cria 1 processo por restrição selecionada. Retorna processos_ids[].
2
Processos → Aguardando Preparação (Distribuição)
Os processos criados entram na página Distribuição como "Aguardando Preparação". Cada processo tem sua restrição correspondente.
3
Modal Drawer da Distribuição
Mostra todos os dados do lead/processo. Documentos do lead já propagados. Documentos individuais da restrição visíveis. Botão de distribuir para o tribunal.
4
Distribuição no Tribunal (página Distribuição)
Na página de Distribuição existe o botão "Efetuar Distribuição" (ModalEfetuarDistribuicao). Após efetuar, o processo sai do CRM completamente. No CRM, o lead vai para arquivados com motivo "cliente".

O que aparece no Modal/Drawer de Distribuição

👤
Dados completos do cliente/lead
Nome, CPF, endereço, contato, advogado responsável
🏢
Dados da restrição (1 por processo)
Empresa, valor, órgão, data, número do contrato (restricao_contrato — campo já existe no tipo ProcessoDistribuicao mas não está sendo exibido no drawer). Cada processo = 1 restrição específica.
📄
Documentos do processo (organizados por restrição)
Docs comuns (RG, comprovante...) + procuração específica + comprovante da restrição. Todos já carregados do CRM.
⚖️
Botão de Distribuir
Protocola o processo no tribunal (TJRS/TJSC/TJSP). Muda status para "distribuído".
⚠️ O que atualiza ao distribuir: Lead → página Clientes (se ainda não existia). Processo → status "aguardando_distribuicao" → "distribuido". Aparece no monitoramento EPROC.
