🎯 Visão Geral e Rationale
Por que começar pelo TJSC?

Santa Catarina é o laboratório ideal: menor volume (583 processos), maior cobertura de análise IA atual (77%), 2 OABs validadas e funcionando. Qualquer bug encontrado aqui afeta no máximo 583 processos — e o fix vai beneficiar TJRS (3.849) e TJSP (1.503) depois.

🤖 Conceito do Loop de Auto-Melhoria

JARBAS não é só executor — é o engenheiro do próprio sistema. Durante a análise dos 583 processos, cada erro encontrado dispara um ciclo de investigação → diagnóstico → implementação de fix → retry. Ao terminar o TJSC, o pipeline está maduro o suficiente para rodar TJRS e TJSP sem supervisão.

🤖 Como o Loop de Auto-Melhoria Funciona

Detecção de Padrão de Erro

JARBAS mantém contador de erros por tipo durante a execução. Quando o mesmo tipo de erro aparece 3+ vezes:

  • Agrupa os casos com stack trace + screenshots
  • Analisa o padrão com Claude (o próprio JARBAS)
  • Identifica se é: seletor CSS, timeout, sessão expirada, estrutura HTML diferente

Implementação Automática do Fix

JARBAS usa self_update para corrigir o próprio código:

  • Edita o arquivo da skill com o fix identificado
  • Valida syntax com node --check
  • Faz commit com mensagem descritiva
  • Reinicia JARBAS para recarregar a skill
  • Re-enfileira os processos que haviam falhado

Guardrails de Segurança

O auto-fix tem limites para não quebrar o sistema:

  • Só edita arquivos de skill (nunca task-consumer ou start.mjs)
  • Sempre valida syntax antes de commit
  • Máximo 3 auto-fixes por sessão (evita loop infinito)
  • Christian é notificado de cada fix aplicado
  • Se o fix falhar → pausa e pede intervenção humana

Conhecimento Acumulado

Cada fix é documentado para não repetir erros:

  • Salvo em memory/YYYY-MM-DD.md e JARBAS_KNOWLEDGE.md
  • Padrões recorrentes viram regras permanentes no código
  • Ao escalar TJRS/TJSP, o pipeline já incorporou os aprendizados do TJSC
  • Relatório de auto-melhoria gerado ao final de cada fase

🔧 Gaps Conhecidos que o Auto-Loop Vai Resolver Durante TJSC
  • processo_documentos insert falhando silenciosamente — o loop vai detectar que o bucket tem arquivos mas o banco não tem registros, investigar e corrigir
  • Documentos TJSP com onclick/iframe — se aparecerem durante TJSC (não esperado), o Vision fallback + auto-fix vai tratar
  • Rate limits do EPROC por OAB — se uma OAB sofrer bloqueio temporário, o loop detecta e faz cooldown automático
  • PDFs corrompidos ou vazios — fallback para movimentações do banco, registrado como fonte = 'movimentacoes'
  • Qualquer novo seletor CSS do EPROC — Vision fallback captura o que o DOM não pega, e o fix vira regra permanente
